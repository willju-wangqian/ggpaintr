Level 2 UI placement regression REGRESSION

Repo: ggpaintr-rewrite · branch core-rewrite-impl · recorded 2026-05-11 · author note for follow-up work

The core rewrite preserves Level 1 (turn-key apps) and Level 3 (extract plot/code/error from a live instance, custom render path) but drops the Level 2 capability to position the control UI and the plot/error/code UI independently inside an existing Shiny layout. In the rewrite, embedding ggpaintr into another app is all-or-nothing: ptr_module_ui() emits a complete page; there is no public API to get "just the controls" and "just the outputs" as separate UI fragments.

What Level 2 was in legacy ggpaintr

Legacy exposed two routes. The fine-grained route gave full layout freedom:

Legacy exportRoleIn rewrite?
ptr_input_ui(ids, ui_text, ns)Returns the control fragment only (uiOutput(control_panel) + draw actionButton)removed
ptr_output_ui(ids, ns)Returns the output fragment only (plotOutput, error uiOutput, code verbatimTextOutput)removed
ptr_build_ids(...)Lets the embedder choose the DOM ids for control panel / draw button / plot / error / coderemoved
ptr_server_state(...)Builds the reactive statekept
ptr_setup_controls(input, output, state)Wires the control widgets without binding outputsremoved (now internal ptr_setup_*, not exported, and not separable)
ptr_register_draw(input, state)Wires the draw buttonremoved (folded into ptr_setup_*)
ptr_register_plot / _error / _code(output, state)Bind each output individuallykept & exported

Because the control fragment and the output fragment were separate return values with caller-chosen ids, a legacy user could write:

ui <- fluidPage(
  titlePanel("My dashboard"),
  fluidRow(
    column(3, ptr_input_ui()),                # controls here
    column(6, ptr_output_ui()),               # plot/error/code here
    column(3, my_own_sidebar_widgets())       # and my own stuff alongside
  )
)
server <- function(input, output, session) {
  st <- ptr_server_state("ggplot(data = mtcars, aes(x = var, y = var)) + geom_point()")
  ptr_setup_controls(input, output, st)
  ptr_register_draw(input, st)
  ptr_register_plot(output, st); ptr_register_error(output, st); ptr_register_code(output, st)
}

What Level 2 is in the rewrite today

Only the module pair is public, and ptr_module_ui() returns a finished fluidPage:

# R/paintr-app.R — ptr_build_app_ui(), called by ptr_module_ui()
shiny::fluidPage(
  shiny::titlePanel(...),
  shiny::sidebarLayout(
    do.call(shiny::sidebarPanel, sidebar_children),     # layer picker + hidden tabset + "Update plot"
    shiny::mainPanel(
      shiny::plotOutput(ns("ptr_plot")),
      shiny::uiOutput(ns("ptr_error")),
      shiny::verbatimTextOutput(ns("ptr_code"))
    )
  )
)
  • The control widgets and the plot/error/code outputs are welded into one sidebarLayout. The embedder gets one opaque block; they cannot put the controls in their app's own sidebar and the plot in their own main panel.
  • Output ids are hard-coded: ptr_plot, ptr_error, ptr_code (plus ptr_layer_select, ptr_layer_tabset, ptr_update_plot), namespaced under NS(id). No ids = argument anywhere.
  • ptr_server() bundles output registration into the wiring step — it calls ptr_setup_producer_inputs / _pipelines / _stage_enabled / _runtime / _consumer_uis / _layer_picker and ptr_register_plot / _error / _code. There is no exported entry point that wires the controls but leaves output placement to the caller.
  • ptr_register_plot/_error/_code(output, state) are exported and do let you re-bind an output (last assignment wins, so a custom renderPlotly can override ptr_plot) — that is the Level 3 story and it still works. But it does not restore layout freedom, because the only thing that emits the control widgets is ptr_build_app_ui(), which also emits the outputs. build_ui_for() is exported but driving it correctly (picker + hidden tabset + ids matching ptr_setup_layer_picker's expectations) amounts to re-implementing ptr_build_app_ui().
Net effect: a Level 2 user can drop ggpaintr in as a self-contained panel, but cannot do what the legacy README/extensibility vignette advertised — "decide where to put the control UI and where to put the plot/error/code UI" without touching the host app.

Suggested fixes (pick one or combine)

  1. Split the UI builder. Factor ptr_build_app_ui() into ptr_controls_ui(tree_or_formula, ns, ui_text, checkbox_defaults) and ptr_outputs_ui(ns), with ptr_module_ui() kept as the thin combiner (back-compat). Embedders place the two fragments wherever they want; the namespace argument keeps ids consistent with ptr_module_server().
  2. Re-introduce configurable ids. Bring back something like ptr_build_ids(plot = "ptr_plot", error = "ptr_error", code = "ptr_code", layer_select = ..., update = ...) and thread it through ptr_server() / ptr_register_*() so the hard-coded names aren't a hidden contract.
  3. Expose a controls-only setup. Export ptr_setup_controls(state, input, output, session) that runs the internal ptr_setup_* group without the ptr_register_* calls; have ptr_server() call it plus the registers. Then an embedder can wire controls and choose which outputs to bind / where.

Option 1 + 3 together most closely restores the legacy Level 2 ergonomics; option 2 is the smaller, lower-risk first step.

Related: restore the headless path as internal-only TODO

The rewrite dropped the legacy headless chain (ptr_parse_formula() → ptr_exec() → ptr_extract_*() on a detached runtime result; the rewrite's ptr_extract_*() now take a live ptr_state). Decision: add it back, but unexported / internal only. Rationale — these benefits are real and don't need a running Shiny session:

  • Batch report generation — knit N plots from a parameter grid in an .Rmd without starting a server.
  • Runtime regression tests — assert on runtime$ok / stage / code_text without shiny::testServer() scaffolding.

The "give me the generated ggplot code, I don't want the app" use case is judged rare and is not a reason on its own — but it falls out of the same internals for free.

Shape:

  • Internal helpers (no @export, not in NAMESPACE): a parse step (ptr_translate() already exists internally) and a headless exec step that returns the detached runtime list (ok, stage, error, plot, code_text, …) — i.e. factor the body of ptr_setup_runtime() so the pure computation is callable without reactives.
  • Keep it wrapped: reuse it inside ptr_setup_runtime() and inside the test/report helpers; do not let the detached-result API leak into the public surface. If a public need appears later, export then — not now.

Implementation plan

Two independent parts. Part A restores the Level-2 split-UI capability; Part B re-adds the headless runtime as internal-only. Do Part A first (no behavior change, low risk), then Part B (pure refactor + new internal helper). All code below is illustrative — match the surrounding style and the exact current call sequences; do not invent new behavior.

Part A — Split-UI for Level 2

A1. Factor ptr_build_app_ui() into internal panel builders (no behavior change)

In R/paintr-app.R, pull the two halves of the current ptr_build_app_ui() body out into unexported helpers, and rewrite ptr_build_app_ui() as the combiner. The rendered output must be byte-for-byte identical so existing UI snapshot tests stay green.

# R/paintr-app.R  (internal)

# Sidebar contents: shared section (optional) + layer picker + hidden tabset + Update button.
# Returns a list of tags (so callers can do.call(sidebarPanel, .) or wrap in tagList()).
ptr_controls_panel <- function(tree, ui_text = NULL, checkbox_defaults = NULL,
                               ns = shiny::NS(NULL), render_shared_section = FALSE) {
  shell_copy   <- layer_panel_default_shell_copy(ui_text)
  layer_names  <- vapply(tree$layers, function(l) l$name, character(1))

  panels <- lapply(tree$layers, function(layer) {
    build_ui_for(layer, ui_text = ui_text, ns_fn = ns,
                 checkbox_defaults = checkbox_defaults, shell_copy = shell_copy)
  })
  picker <- shinyWidgets::pickerInput(
    inputId = ns("ptr_layer_select"),
    label   = shell_copy$layer_picker_label %||% "Layer",
    choices = layer_names,
    selected = if (length(layer_names)) layer_names[1L] else NULL
  )
  hidden_tabset <- do.call(shiny::tabsetPanel,
    c(list(id = ns("ptr_layer_tabset"), type = "hidden"), panels))

  shared_section <- if (isTRUE(render_shared_section)) {
    entries <- collect_shared_placeholders(tree)
    widgets <- drop_null(lapply(entries, function(e) build_ui_for(e$node, ui_text = ui_text, ns_fn = ns)))
    if (length(widgets)) shiny::wellPanel(do.call(shiny::tagList, widgets)) else NULL
  } else NULL

  drop_null(list(
    shared_section, picker, hidden_tabset,
    shiny::actionButton(ns("ptr_update_plot"), label = shell_copy$update_plot_label %||% "Update plot")
  ))
}

# Output contents: plot + error + code, in that DOM order. ids are the contract
# ptr_register_plot()/_error()/_code() write to.
ptr_outputs_panel <- function(ns = shiny::NS(NULL)) {
  shiny::tagList(
    shiny::plotOutput(ns("ptr_plot")),
    shiny::uiOutput(ns("ptr_error")),
    shiny::verbatimTextOutput(ns("ptr_code"))
  )
}

# Combiner — unchanged public behavior; ptr_module_ui() / ptr_app_components() keep calling this.
ptr_build_app_ui <- function(tree, ui_text = NULL, checkbox_defaults = NULL,
                             ns = shiny::NS(NULL), render_shared_section = FALSE) {
  shiny::fluidPage(
    shiny::titlePanel(ptr_resolve_ui_text("title", ui_text = ui_text)$label %||% ""),
    shiny::sidebarLayout(
      do.call(shiny::sidebarPanel,
              ptr_controls_panel(tree, ui_text, checkbox_defaults, ns, render_shared_section)),
      shiny::mainPanel(ptr_outputs_panel(ns))
    )
  )
}

A2. New exported Level-2 split API

Two thin wrappers, also in R/paintr-app.R, next to ptr_module_ui(). They impose no layout — the embedder drops the fragments wherever they want. Keep the signature shape consistent with ptr_module_ui(id, formula, ...): id first-or-named, formula-based (parsing the formula twice — once here, once in ptr_module_server() — is already what ptr_module_ui()+ptr_module_server() do today, so it's fine).

# R/paintr-app.R  (exported)

#' Control widgets for an embedded ggpaintr formula
#'
#' UI fragment containing only the generated controls (layer picker, per-layer
#' parameter panels, the "Update plot" button). Place it anywhere in your app's
#' layout; pair with [ptr_outputs_ui()] for the plot/error/code panes and
#' [ptr_module_server()] for the server logic. `id` must match the `id` passed
#' to [ptr_module_server()] / [ptr_outputs_ui()].
#' @return A [shiny::tagList()].
#' @export
ptr_controls_ui <- function(id, formula, ui_text = NULL, checkbox_defaults = NULL,
                            expr_check = TRUE) {
  tree <- ptr_translate(formula, expr_check = expr_check)
  shiny::tagList(!!!ptr_controls_panel(tree, ui_text, checkbox_defaults,
                                       ns = shiny::NS(id), render_shared_section = FALSE))
}

#' Plot / error / code panes for an embedded ggpaintr formula
#'
#' UI fragment containing only the plot, inline error, and generated-code
#' outputs. Pair with [ptr_controls_ui()] and [ptr_module_server()]. `id` must
#' match the `id` passed to those.
#' @return A [shiny::tagList()].
#' @export
ptr_outputs_ui <- function(id) {
  ptr_outputs_panel(shiny::NS(id))
}

(!!! splice needs rlang, already a dependency; or use do.call(shiny::tagList, ptr_controls_panel(...)).)

A3. Server side — nothing to change, one thing to verify

ptr_module_server(id, formula, ...) already binds purely by id (it forwards to ptr_server(), which reads/writes input[[ns("ptr_layer_select")]], output[[ns("ptr_plot")]], etc.). It does not care whether the UI came from ptr_module_ui() or from ptr_controls_ui()+ptr_outputs_ui() — provided the same id is used everywhere. Verify with a test (A5). The control-side ids (ptr_layer_select, ptr_layer_tabset, ptr_update_plot, the producer-input ids, the consumer uiOutput ids from build_ui_for()) come from ptr_controls_panel(); the output-side ids (ptr_plot/ptr_error/ptr_code) come from ptr_outputs_panel() — both already match what the server expects, because they're the same builders the combiner uses.

Part B — Internal headless runtime

Read before you start Part B — two facts verified against the current source; the pseudo-code below is illustrative and got some defaults wrong.
  1. Exact signatures of the lifted helpers (do not "tidy" the defaults — pass through verbatim):
    • ptr_complete_expr_safe(node, snapshot = list(), shared_bindings = list(), eval_env = parent.frame(), safe_to_remove = NULL, is_standalone = NULL, upstream_cols = list()) — note the extra is_standalone arg, safe_to_remove defaults NULL (not character()), upstream_cols defaults list() (not NULL). Defined in R/paintr-safe.R.
    • ptr_assemble_plot_safe(result, expr_check = TRUE) — passthrough when !result$ok. ptr_validate_plot_render_safe(result) — single arg. Both in R/paintr-safe.R.
    • runtime_upstream_cols(state, snapshot = list()) → wraps runtime_upstream_data(state, snapshot). This pair is deeply state-coupled and reactive (reads state$tree(), state$stage_enabled(), state$shared_resolutions, state$resolved_data[[..]](), state$upstream_cache, state$expr_check, …; runs under shiny::isolate()). Do NOT move them and do NOT call them from ptr_exec_headless(). Keep upstream_cols a plain input parameter of ptr_exec_headless(): the reactive caller (ptr_setup_runtime) keeps computing it via runtime_upstream_cols(state, snapshot) and passes the result in; the headless caller (ptr_run_formula) computes its own via a tiny new pure helper that calls the already-pure ptr_resolve_upstream(value, snapshot, shared_bindings, eval_env, cache, expr_check, stage_enabled) directly on each data-consumer node's $upstream — no state, no isolate. In headless mode it's fine for that helper to ignore the bare-data-source-layer shortcut and always resolve from $upstream.
  2. inject_resolved_data() has exactly one caller — ptr_setup_runtime() at R/paintr-server.R:464. Nothing else in R/ touches it, so the split is safe and local. Its body is trivial: for each is_ptr_layer layer whose name is a key in the resolved-data map with a non-null frame, layer$data_arg <- ptr_literal(df). So: new inject_resolved_data_list(pruned, list_of_frames) does the work; existing inject_resolved_data(pruned, state) becomes a one-liner — inject_resolved_data_list(pruned, lapply(state$resolved_data, function(rv) rv())).

B1. Extract the pure computation from ptr_setup_runtime()

The body of ptr_setup_runtime()'s shiny::isolate({...}) block is already a pure function of (tree, input snapshot, stage-enabled map, resolved-data map, extras, eval env, safe-to-remove, expr_check). Lift it verbatim into an unexported helper; have the observer build the inputs and call it. Do not change the call sequence or the result shape — the runtime list keys (ok, stage, error, plot, code_text, pruned, …) and the order ptr_complete_expr_safe → inject_resolved_data → ptr_assemble_plot_safe → (+ extras) → ptr_validate_plot_render_safe must be preserved exactly.

# R/paintr-server.R (or new R/paintr-headless.R), internal — NOT exported

# `resolved_data`: plain named list (layer name -> data.frame or NULL), already
#   de-reactived by the caller. `upstream_cols`: as today (caller passes the
#   value it already computes). `stage_enabled`: named list (stage id -> logical).
ptr_exec_headless <- function(tree, snapshot,
                              shared_bindings = list(),
                              eval_env       = parent.frame(),
                              safe_to_remove = character(),
                              expr_check     = TRUE,
                              extras         = list(),
                              stage_enabled  = NULL,
                              resolved_data  = list(),
                              upstream_cols  = NULL) {
  if (!is.null(stage_enabled)) tree <- disable_walk(tree, stage_enabled)
  res <- ptr_complete_expr_safe(
    tree, snapshot = snapshot, shared_bindings = shared_bindings,
    eval_env = eval_env, safe_to_remove = safe_to_remove, upstream_cols = upstream_cols
  )
  res$pruned <- inject_resolved_data_list(res$pruned, resolved_data)   # see note
  res <- ptr_assemble_plot_safe(res, expr_check = expr_check)
  if (isTRUE(res$ok) && length(extras)) res$plot <- Reduce(`+`, extras, res$plot)
  ptr_validate_plot_render_safe(res)
}

Note on inject_resolved_data: the current helper is inject_resolved_data(pruned, state) and reads state$resolved_data[[layer$name]](). Split it: keep a inject_resolved_data_list(pruned, list_of_frames) that does the actual substitution from a plain list, and make the existing inject_resolved_data(pruned, state) a one-liner that builds lapply(state$resolved_data, function(rv) rv()) and calls the list version. ptr_setup_runtime()'s observer then becomes:

shiny::observe({
  if (!triggered()) return(invisible())
  shiny::isolate({
    spec     <- state$input_spec
    snapshot <- stats::setNames(lapply(spec$input_id, function(id) input[[ns(id)]]), spec$input_id)
    upstream_cols <- runtime_upstream_cols(state, snapshot)
    res <- ptr_exec_headless(
      tree           = state$tree(),
      snapshot       = snapshot,
      shared_bindings= state$shared_bindings,
      eval_env       = state$eval_env,
      safe_to_remove = state$safe_to_remove,
      expr_check     = state$expr_check,
      extras         = state$extras(),
      stage_enabled  = state$stage_enabled(),
      resolved_data  = lapply(state$resolved_data, function(rv) rv()),
      upstream_cols  = upstream_cols
    )
    state$runtime(res)
  })
})

(Behavior identical — the only change is that the computation now lives in a callable function. Existing runtime / testServer tests are the regression guard.)

B2. Internal convenience wrapper for the common case

# internal — NOT exported
# inputs: named by RAW input_id (the `input_id` column of ptr_runtime_input_spec()),
#   not namespaced. Anything not supplied falls back to a sensible default
#   (layer_checkbox -> TRUE, stage_enabled -> TRUE, others -> NULL).
ptr_run_formula <- function(formula, inputs = list(), envir = parent.frame(),
                            expr_check = TRUE, safe_to_remove = character(),
                            checkbox_defaults = NULL) {
  tree     <- ptr_translate(formula, expr_check = expr_check)
  spec     <- ptr_runtime_input_spec(tree)
  snapshot <- ptr_default_snapshot(spec, tree, checkbox_defaults)   # small new internal helper
  snapshot[names(inputs)] <- inputs
  upstream_cols <- ptr_headless_upstream_cols(tree, snapshot)        # static analog of runtime_upstream_cols
  ptr_exec_headless(tree, snapshot, eval_env = envir,
                    safe_to_remove = validate_safe_to_remove(safe_to_remove),
                    expr_check = expr_check, upstream_cols = upstream_cols)
}

ptr_default_snapshot(): iterate spec, set role == "layer_checkbox" and role == "stage_enabled" rows to TRUE (respecting checkbox_defaults for the former via the existing ptr_resolve_checkbox_defaults() path), everything else NULL. ptr_headless_upstream_cols(): for a bare-data-source layer whose value is a symbol resolvable in envir, return its column names; otherwise NULL — i.e. the no-Shiny analog of what runtime_upstream_cols() derives from live inputs. Keep it minimal; data-consumer (var) widgets in headless mode just need the upstream columns so completion can validate.
