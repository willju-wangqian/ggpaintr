PLAN-01 (PLAN-03 rerun) — Super-app 2a: upload + registry + mid-pipeline

Source: ADR 0013 §"App 2a — Upload + registry + mid-pipeline" + D1/D2/D4/D5/D6/D7/D9/D10 (App-2a row) · Depends on: none (PLAN-01 anchors satisfied externally at 6ee63d7; Finding 1 fixed at 6ee63d7; Finding 2 fixed at 04b2fc2) · Parallel group: G0 (standalone, no in-bundle siblings) · Status: draft

Context

ADR 0013 §App-2a locks the slot picks: ptr_app() · ppUpload mid-pipeline head (ppUpload(df_main) |> dplyr::filter(...) |> dplyr::mutate(...) |> ggplot(...) + ...) · G1 string capture · window code panel · default theme. Two custom placeholders register locally in app.R: a value placeholder ppPower (numericInput, resolve_expr = function(v) v^2, validate_input for [0,1]) and a consumer placeholder ppMultiVar (picker multiple, resolve_expr returns either rlang::missing_arg(), a bare symbol, or rlang::call2("interaction", !!!syms)).

The roster (per ADR §App-2a): ppUpload×2 (both with positional bareword companion-name args per ADR-0010, e.g. ppUpload(df_main) / ppUpload(df_aux)) · ppVar×6 → 5 widgets after shared "grp" collapse · ppNum×2 · ppText×3 · ppExpr×2 · ppPower×1 · ppMultiVar×1. Total widgets ≈ 13. Two sample CSV files ship sibling to app.R: sample_main.csv and sample_aux.csv (both mtcars subsets with mpg / wt / cyl / am / hp columns).

Unique pressure axes (ADR §App-2a): (i) mid-pipeline ppUpload + dplyr chain (project memory pipeline-head-data-source — May 2026 fix); (ii) multi-data-source via layer data = using geom_smooth(data = ppUpload(df_aux), aes(...), ...); (iii) ADR-0010 positional companion-name on two ppUploads in one formula; (iv) custom consumer returning interaction(...) for non-scalar; (v) D9 validate_input inline error on ppPower; (vi) shared= confined to root-data scope.

Governing principle

ADR 0013 commits the suite to propagation assertions: every placeholder gets a literally-unique sentinel that round-trips under final + preserve. App 2a's additional load: the propagation contract MUST hold through a multi-data-source formula. The two ppUploads' positional companion names (per ADR-0010) MUST round-trip in preserve mode as ppUpload(df_main) and ppUpload(df_aux), NOT collapsed and NOT swapped. ppVar pickers inside the layer-data aes() MUST populate from df_aux's columns, NOT from df_main — proving the project memory pipeline-head-data-source fix holds end-to-end at the picker level, not just at the parser level.

The custom consumer's resolve_expr for non-scalar returns is the most fragile path: the final-mode renderer must emit a synthetic call interaction(cyl, am) rather than the keyword ppMultiVar(...). If the propagation assertion fails on the synthesized-call shape, that is a product-level integration regression that no unit test would have caught; soft "the code is non-empty" wording would hide exactly this. STOP-and-escalate, do not narrow.

Wait: resolve_expr = function(v) v^2 rewrites the value, so the final-mode output is 0.42^2 (a call expression, not a number) — that is the exact sentinel propagation contract for a value placeholder with a non-identity resolve_expr. In preserve mode, expect_sentinel_in_code(app, code_output_id, "0.42", "ppPower\\(([^)]*)\\)", "preserve").
  • Sentinel propagation — ppMultiVar custom consumer non-scalar return. The test sets the ppMultiVar input (a multi-select picker) to c("cyl", "am") via set_sentinel(app, <id>, c("cyl", "am")), draws, and asserts final-mode expect_sentinel_in_code(app, code_output_id, "interaction(cyl, am)", "aes\\([^)]*group\\s*=\\s*([^,)]*)", "final"). In preserve mode, expect_sentinel_in_code(app, code_output_id, "ppMultiVar(c('cyl', 'am'))", "ppMultiVar\\(([^)]*)\\)", "preserve") (or the project's specific preserve serialization for consumer placeholders with vector input — see render_placeholder_preserved at R/paintr-render.R:535).
  • ppUpload positional companion-name round-trip — both names retained. After both uploads, the test toggles to preserve mode and asserts the rendered code contains BOTH literal substrings ppUpload(df_main) AND ppUpload(df_aux) (bareword form per R/paintr-render.R:534's bareword_kw dispatch for ppUpload). The two strings are distinct; collapsing them or swapping the companion names fails this expectation.
  • Layer-data ppVar picker populates from df_aux's columns, not df_main's. After both uploads, the test asserts expect_picker_populated(app, <ppVar id inside geom_smooth's data = ppUpload(df_aux) aes>, "mpg") succeeds AND expect_picker_populated(app, <same id>, "cyl") FAILS — because cyl is in df_main but NOT in df_aux (sample_aux.csv has only mpg/wt). The negative assertion uses an explicit expect_failure() wrapper or asserts the picker's HTML does not contain the string "cyl".
  • D9 validate_input — inline error appears, runtime does not crash. The test sets ppPower to 1.5 (out of range), draws, and asserts: (a) an inline error block appears next to the widget — verified via app$get_html(<widget container selector>) containing the literal string "must be in [0,1]"; (b) the host output retains its previous valid plot (no plot-error class added, expect_no_plot_error(app) still passes); (c) the previously-rendered code text is preserved (sentinel from the prior successful Draw is still present).
  • Shared "grp" confined to root-data scope (no cross-scope contamination). The test sets "shared_grp" to a df_main column choice (e.g. "cyl"), draws, asserts in final mode that cyl appears as bare symbol in both aes(... color = cyl ...) AND facet_wrap(vars(cyl)). The geom_smooth's separate aes() (which reads df_aux) does NOT acquire a color = cyl — verified by expect_sentinel_nowhere(app, code_output_id, "geom_smooth(... color = cyl") via a tight regex.
  • B3 toggle differential. Toggle preserve↔final once. Final-mode strips all ppX( wrappers; preserve-mode retains them. Both ppUpload(<name>) wrappers (bareword form) are present in preserve and ABSENT in final (where they expand to the auto-resolved bare symbol or call).
  • Authoritative gate returns FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + 1) with the browser stack available.

Definition of Done

Authoritative gate (copy verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + 1). Baseline at this branch's HEAD 6ee63d7 = PASS 2384 (recorded in the entry-point handoff). After this plan: PASS ≥ 2385.

Proxy traps — do not be fooled (verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate. Never claim the suite green, merge a branch, mark a step done, or clear context off the skipping/erroring harness.

This plan adds 1 new test_that() call at tests/testthat/test-super-pressure.R (in the super-2a anchor region). The gate must show PASS count rise by ≥ 1 with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline.
