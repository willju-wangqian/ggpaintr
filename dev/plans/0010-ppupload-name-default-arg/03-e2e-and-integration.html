PLAN-03 — adr10-ppupload-name e2e fixture + auto-resolve integration tests

Source: ADR 0010 (Decision §2, §4 + Clarifications C2, C3, C5) · Depends on: PLAN-01, PLAN-02 · Parallel group: G2 · Status: draft

Context

ADR 0010 Decision §2 (final-mode substitution emits the user name) and §4 (outside-app evaluation) describe end-to-end behavior whose unit-level pieces are owned by PLAN-01 and PLAN-02. This plan binds those pieces with two integration surfaces:

  1. An R-level test file that exercises ptr_app_components end-to-end against a small in-memory data.frame in the caller env — asserting that ppUpload(penguins) |> ... in the formula causes the live-eval AST emitted from substitute_walk to evaluate against the caller-env penguins via the existing state$eval_env parent chain (Clarification C2). No new server-side wiring is required by this plan; the plan verifies that what is already wired works under the seeded-companion change PLAN-01 introduces.
  2. A browser-level shinytest2 fixture under tests/testthat/fixtures/vignette-apps/adr10-ppupload-name/ following the app-dir + pkgload::load_all pattern documented in CLAUDE.md's "Browser e2e (shinytest2) — hard-won gotchas" block, asserting (a) the companion textInput renders seeded with "penguins", (b) the code panel renders the formula with bareword name in both preserve and final modes, (c) the rendered plot appears (auto-resolve works against the caller env), (d) editing the companion input to "iris" causes preserve mode to round-trip ppUpload(iris).

Implementation notes — discovering the upload companion-id and code-panel-id

The BDD scenarios below use the literal placeholders <upload-companion-id> and <code-panel-id>. The implementer resolves these at test-write time as follows:

  1. Upload companion-id: obtain the translated tree and harvest the node's companion_id slot directly. 
    formula <- "ppUpload(penguins) |> ..."
tree    <- ggpaintr:::ptr_translate(formula)   # internal — `:::` not `::`
# Use the existing tree walker (R/paintr-walk.R:16) to collect ppUpload nodes:
found <- list()
ggpaintr:::ptr_walk(tree, visit = function(n) {
  if (ggpaintr:::is_ptr_ph_data_source(n) && identical(n$keyword, "ppUpload")) {
    found[[length(found) + 1L]] <<- n
  }
  n
})
companion_id <- found[[1L]]$companion_id   # e.g. "<layer>-up<n>_name"
    The companion-id is derived as paste0(node$id, "_name") per ptr_upload_name_id (R/paintr-upload.R:7–9) registered as companion_id_fn in the upload_fn registry block (R/paintr-builtins.R:327–333); deriving from node$id after the fact is equivalent and simpler if traversal is awkward.
  2. Code-panel-id: reuse the selector pattern from existing browser tests. Read tests/testthat/test-e2e-vignette-examples-shinytest2.R for the canonical code-panel selector used by boot_vignette_app()-driven assertions; the same selector applies here because adr10-ppupload-name uses the same ptr_app() entry-point and therefore the same default UI scaffold.
  3. Mode toggle: the preserve-mode / final-mode selector lives in the project's default UI under a known input id — find it the same way (existing browser tests already drive it). Do not hardcode a guess.

If any of these discovery steps reveals that the symbol or selector pattern has changed, that is a regression against PLAN-01 / PLAN-02 (not this plan's scope to fix). Surface it in the audit report per the Out-of-scope handling section.

Definition of Done

Authoritative gate (copied verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N where N = baseline_PASS_at_branch_point + N_new_tests_in_this_plan. Record both numbers in the plan's worktree first commit message. Where "SKIP 0" requires the canonical shinytest2 + chromote + Chrome stack — on the maintainer's dev box that's true; on a CI box missing Chrome a clean all-skip on the browser file alone is acceptable but must be reported, not hidden.

Proxy traps — do not be fooled (copied verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate. Never claim the suite green, merge a branch, mark a step done, or clear context off the skipping/erroring harness.

This plan adds approximately 7 new tests (3 R-level + 4 browser-level). The gate must show PASS rise by ≥ 7 with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline (modulo the documented browser-stack-missing case).
