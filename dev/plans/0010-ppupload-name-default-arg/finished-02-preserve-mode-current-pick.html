PLAN-02 — Preserve-mode current_pick for ppUpload reads the companion-id snapshot

Source: ADR 0010 (Decision §3 + Clarification C1) · Depends on: none · Parallel group: G1 · Status: draft

Context

ADR 0010 Decision §3 says "render_placeholder_preserved dispatches ppUpload through format_placeholder_value with bareword formatting, identical to ppVar. ppUpload(penguins) in the formula round-trips back to ppUpload(penguins) in the code panel." Clarification C1 documents the half of this that is already done: format_placeholder_value (R/paintr-render.R:391–412) already lists "ppUpload" in bareword_kw. What is not yet wired up is stamp_current_pick_walk.ptr_ph_data_source (R/paintr-render.R:358–364), which today reads snapshot[[node$id]] — the fileInput's value (a four-column data.frame describing the upload, or NULL when no file was chosen) — and uses that as node$current_pick. For ppUpload nodes, the meaningful "current pick" is instead the user-entered display name living at snapshot[[node$companion_id]].

This plan rewires stamp_current_pick_walk.ptr_ph_data_source so that when node$companion_id is set, the companion-id snapshot value drives node$current_pick. With PLAN-01 seeding that companion textInput from node$default, the end-to-end round trip ppUpload(penguins) → preserve-mode ppUpload(penguins) works without further changes to render_placeholder_preserved or format_placeholder_value. Companion-less data-source placeholders (registered via ptr_define_placeholder_source() without companion_id_fn) keep the existing snapshot[[node$id]] path.

Success Criteria

  • stamp_current_pick_walk.ptr_ph_data_source in R/paintr-render.R reads from snapshot[[node$companion_id]] when node$companion_id is non-NULL and non-empty; otherwise falls back to the existing snapshot[[node$id]] path. The "is this a real pick" guard (.snapshot_value_is_set(node, val)) still applies on the chosen value.
  • render_placeholder_preserved on a ptr_ph_data_source node with keyword="ppUpload", companion_id="up1_name", and current_pick="penguins" (set by the rewired stamp pass) returns the character scalar "ppUpload(penguins)". (format_placeholder_value's existing bareword_kw dispatch handles the formatting — no render.R change downstream of the stamp pass.)
  • render_placeholder_preserved on a ptr_ph_data_source node with keyword="ppUpload" and current_pick=NULL (no companion value typed, no default) returns "ppUpload()" — legacy behavior preserved.
  • render_placeholder_preserved on a ptr_ph_data_source node with keyword="ppUpload" and current_pick="" (companion textInput was rendered but emptied) returns "ppUpload()". Empty-string is treated by .snapshot_value_is_set as "not set" so current_pick stays NULL — verify the guard does this for character scalars; if it does not, this plan does NOT modify .snapshot_value_is_set — instead the test asserts whatever the existing guard actually does, and the BDD scenario records that ground truth.
  • A companion-less data-source registry entry (e.g. a fabricated test placeholder registered via ptr_define_placeholder_source(keyword="ppTestSrc", build_ui=..., resolve_data=...) with no companion_id_fn) still stamps current_pick from snapshot[[node$id]], preserving the existing path for non-upload sources. Asserted by direct call to the rewired stamp function with a synthetic node.
  • Authoritative gate returns FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + N_new).

Constraints

  • Files this plan owns:
    • R/paintr-render.R — modify only stamp_current_pick_walk.ptr_ph_data_source (lines 358–364). No other functions in the file may be touched. In particular: format_placeholder_value (391–412) is already correct for ppUpload per Clarification C1; render_placeholder_preserved (372–385) handles all keywords uniformly and needs no change; render_walk.ptr_ph_data_source (254–259) routes through render_placeholder_preserved in preserve mode and needs no change.
    • tests/testthat/test-ppupload-preserve-mode.R (new file).
  • Files this plan must NOT touch: R/paintr-builtins.R (owned by PLAN-01), R/paintr-substitute.R, R/paintr-nodes.R, R/paintr-translate.R, R/paintr-default-args.R, NAMESPACE, man/, any vignette, any fixture under tests/testthat/fixtures/vignette-apps/.
  • The rewired stamp pass must remain order-independent w.r.t. PLAN-01 — its tests construct synthetic ptr_ph_data_source nodes directly and supply explicit snapshot lists, so they verify the stamp logic without depending on PLAN-01's UI seeding or node$default.
  • Honor project conventions: %||% for null coalesce, no stop()/warning(), snake_case, 2-space indent.
  • No scope creep into .snapshot_value_is_set — if a test scenario depends on its behavior, encode the actual current behavior rather than changing the helper.

BDD scenarios

Feature: stamp_current_pick_walk reads the companion-id snapshot for upload-style data sources

Scenario: Companion-id snapshot value populates current_pick
  Given a ptr_ph_data_source node N with keyword="ppUpload",
        id="up1", companion_id="up1_name", current_pick=NULL
  And   a snapshot list with snapshot[["up1_name"]] = "penguins"
        and snapshot[["up1"]] absent or NULL
  When  I call ggpaintr:::stamp_current_pick_walk(N, snapshot)
  Then  the returned node's current_pick equals the character scalar "penguins"

Scenario: Empty companion-id value leaves current_pick NULL
  Given the same N
  And   snapshot[["up1_name"]] = ""
  When  I call stamp_current_pick_walk(N, snapshot)
  Then  the returned node's current_pick is NULL
        (the existing .snapshot_value_is_set guard treats "" as not-set)

Scenario: NULL companion-id value leaves current_pick NULL
  Given the same N
  And   snapshot is list() with no "up1_name" entry
  When  I call stamp_current_pick_walk(N, snapshot)
  Then  the returned node's current_pick is NULL

Scenario: Companion-less source still uses node$id path
  Given a ptr_ph_data_source node M with keyword="ppTestSrc",
        id="src1", companion_id=NULL, current_pick=NULL
  And   snapshot[["src1"]] = "iris"
  When  I call stamp_current_pick_walk(M, snapshot)
  Then  the returned node's current_pick equals "iris"

Scenario: render_placeholder_preserved emits ppUpload(name) bareword
  Given a ptr_ph_data_source node with keyword="ppUpload" and current_pick="penguins"
  When  I call ggpaintr:::render_placeholder_preserved(node)
  Then  the returned string equals "ppUpload(penguins)"

Scenario: render_placeholder_preserved emits ppUpload() when no current_pick
  Given a ptr_ph_data_source node with keyword="ppUpload" and current_pick=NULL
  When  I call render_placeholder_preserved(node)
  Then  the returned string equals "ppUpload()"

Scenario: End-to-end round-trip through ptr_translate + ptr_render
  Given the formula string "ppUpload() |> ggplot(aes(x = ppVar)) + geom_point()"
  And   a snapshot whose upload-node companion-id maps to "penguins"
        and whose ppVar input maps to "mpg"
  When  I call ggpaintr::ptr_translate(formula),
        then ggpaintr:::stamp_current_pick_walk(root, snapshot),
        then ggpaintr::ptr_render(root, preserve_placeholders = TRUE)
  Then  the rendered text contains the literal substring "ppUpload(penguins)"
  And   does NOT contain the literal substring "ppUpload()"
