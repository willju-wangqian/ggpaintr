PLAN-01 — ppUpload registry default_arg + companion-input seeding + outside-app identity

Source: ADR 0010 (Decision §1, §4 + Clarifications C1, C4, C5, C6) · Depends on: none · Parallel group: G1 · Status: draft

Context

ADR 0010 Decision §1 says "ppUpload's entry gains default_arg = ptr_default_symbol_or_string(), the same validator ppVar uses." Clarification C1 establishes that the display-name mechanism is the existing companion textInput (R/paintr-builtins.R:218–240) — seeding its value from node$default is the work, not inventing a parallel mechanism. Clarification C4 changes the top-level ppUpload from function(...) rlang::abort(...) to function(x, ...) if (missing(x)) abort() else x so the arg-bearing form is identity outside ptr_app. Clarifications C5/C6 lock the legacy and storage-shape behaviors that this plan must honor.

This plan groups every change to R/paintr-builtins.R into one indivisible unit so PLAN-02 (which touches R/paintr-render.R) and PLAN-03 (which adds an e2e fixture and integration tests) can run with zero file overlap.

Success Criteria

  • The upload_fn registry block in ptr_register_builtins() (R/paintr-builtins.R:327–333) carries default_arg = ptr_default_symbol_or_string() as a sixth argument to ptr_define_placeholder_source(). Verified by inspecting the registry entry post-load (ggpaintr:::ptr_registry_lookup("ppUpload")$default_arg is a function — not NULL).
  • ptr_translate("ppUpload(penguins) |> ggplot(mtcars, aes(x = mpg)) + geom_point()") produces a tree whose ptr_ph_data_source node for the upload has node$default == "penguins" (single character scalar). Mirrored by ptr_translate("ppUpload(\"penguins\") |> ...") producing the same node$default value (symbol-or-string accepted, returned as character scalar).
  • ptr_translate("ppUpload() |> ggplot(...)") and ptr_translate("ppUpload |> ggplot(...)") still produce node$default == NULL (legacy behavior strictly preserved — C5).
  • ptr_translate("ppUpload(1 + 2) |> ggplot(...)") aborts with a validator error (ptr_default_symbol_or_string() rejects non-symbol/non-string ASTs). The error class is "rlang_error".
  • ptr_builtin_upload_build_ui(node) with node$default = "penguins" returns a shiny.tag.list whose embedded textInput at node$companion_id carries value = "penguins". With node$default = NULL, that textInput carries value = "" (legacy default). Asserted by walking the rendered tag tree and reading the value attribute of the <input type="text"> element.
  • Top-level ggpaintr::ppUpload changes signature from function(...) to function(x, ...); ppUpload(penguins) in a plain-R session with penguins <- data.frame(a = 1) in scope returns that data.frame unchanged (identity). ppUpload() still aborts with a message containing the substring "only meaningful inside" (verified via expect_error(..., "only meaningful inside")). ppUpload("penguins") returns the string "penguins" verbatim (C4 — string passthrough).
  • The roxygen block at R/paintr-builtins.R:257–289 (@name pp_placeholders) is updated to reflect the new ppUpload(x) identity semantics: the prose stops claiming "ppUpload() aborts when called outside ptr_app()" without qualification and instead specifies that the no-arg form aborts. The @return tag covers both branches. devtools::document() regenerates man/pp_placeholders.Rd consistently (regeneration is the integration step's responsibility per the project's "Generated files" rule — this plan flags drift, does not regenerate).
  • Authoritative gate (Definition of Done) returns FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + N_new) with the new tests this plan adds in tests/testthat/test-ppupload-name-arg.R.

Constraints

  • Files this plan owns:
    • R/paintr-builtins.R — modify only: (a) the upload_fn block (lines 327–333), (b) ptr_builtin_upload_build_ui (lines 218–240; change the value = "" literal in the shiny::textInput call), (c) the top-level ppUpload function (lines 365–367), (d) the roxygen prose around @name pp_placeholders (lines 257–289) and the immediate @return/@param rows describing ppUpload. No other functions in the file may be touched.
    • tests/testthat/test-ppupload-name-arg.R (new file).
  • Files this plan must NOT touch: R/paintr-render.R (owned by PLAN-02 — stamp + preserve-mode changes), R/paintr-substitute.R (Clarification C1 establishes this needs no change), R/paintr-nodes.R (constructor already has default slot), R/paintr-translate.R (parser already routes through entry$default_arg), R/paintr-default-args.R (validator already exists), NAMESPACE, man/, any vignette under vignettes/, any fixture under tests/testthat/fixtures/vignette-apps/.
  • No new exports. The registry slot and UI seeding are internal; the top-level ppUpload stays @export as today (signature changes, export tag unchanged).
  • No scope creep into auto-resolve plumbing — per Clarification C2, the eval-env parent chain in R/paintr-server.R:207 already delivers auto-resolve once the companion textInput is seeded; no server-side change is needed and this plan must not introduce one.
  • Honor project conventions: snake_case, 2-space indent; use rlang::abort() not stop() in the new no-arg guard; use %||% (already in the file) for the node$default %||% "" coalesce.

BDD scenarios

Feature: ppUpload(name) parses, seeds the companion text input, and is identity outside ptr_app

Scenario: Bare symbol arg stamps node$default
  Given the formula string "ppUpload(penguins) |> ggplot(aes(x = ppVar)) + geom_point()"
  When  I call ggpaintr::ptr_translate(formula)
  Then  the resulting ptr_root tree contains exactly one ptr_ph_data_source node
  And   that node$default equals the character scalar "penguins"
  And   that node$keyword equals "ppUpload"

Scenario: String arg stamps node$default identically
  Given the formula string "ppUpload(\"penguins\") |> ggplot(aes(x = ppVar)) + geom_point()"
  When  I call ggpaintr::ptr_translate(formula)
  Then  the ptr_ph_data_source node$default equals "penguins"

Scenario: Legacy ppUpload() leaves node$default NULL
  Given the formula string "ppUpload() |> ggplot(aes(x = ppVar)) + geom_point()"
  When  I call ggpaintr::ptr_translate(formula)
  Then  the ptr_ph_data_source node$default is NULL

Scenario: Legacy bareword ppUpload leaves node$default NULL
  Given the formula string "ppUpload |> ggplot(aes(x = ppVar)) + geom_point()"
  When  I call ggpaintr::ptr_translate(formula)
  Then  the ptr_ph_data_source node$default is NULL

Scenario: Non-symbol-or-string positional arg aborts
  Given the formula string "ppUpload(1 + 2) |> ggplot(aes(x = ppVar)) + geom_point()"
  When  I call ggpaintr::ptr_translate(formula)
  Then  the call aborts with class "rlang_error"

Scenario: build_ui seeds the companion textInput value from node$default
  Given a ptr_ph_data_source node with keyword="ppUpload",
        id="up1", companion_id="up1_name", default="penguins"
  When  I call ggpaintr:::ptr_builtin_upload_build_ui(node)
  Then  the returned shiny.tag.list contains a textInput at inputId "up1_name"
  And   that textInput's value attribute equals "penguins"

Scenario: build_ui leaves the companion textInput empty when default is NULL
  Given a ptr_ph_data_source node with keyword="ppUpload",
        id="up1", companion_id="up1_name", default=NULL
  When  I call ggpaintr:::ptr_builtin_upload_build_ui(node)
  Then  the textInput at "up1_name" has value ""

Scenario: ppUpload(x) outside app is identity
  Given penguins is bound to data.frame(a = 1L) in the current frame
  When  I call ggpaintr::ppUpload(penguins)
  Then  the returned value is identical (base::identical) to penguins

Scenario: ppUpload("string") outside app returns the string verbatim
  When  I call ggpaintr::ppUpload("penguins")
  Then  the returned value is the character scalar "penguins"

Scenario: ppUpload() (no arg) outside app still aborts
  When  I call ggpaintr::ppUpload()
  Then  the call aborts with a message containing "only meaningful inside"

Definition of Done

Authoritative gate (copied verbatim from CLAUDE.md's "Authoritative gate (Definition of Done)" block):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N where N = baseline_PASS_at_branch_point + N_new_tests_in_this_plan. Record both numbers in the plan's worktree first commit message.

Proxy traps — do not be fooled (copied verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate. Never claim the suite green, merge a branch, mark a step done, or clear context off the skipping/erroring harness.

This plan adds approximately 10 new tests across the BDD scenarios above, all in tests/testthat/test-ppupload-name-arg.R. The gate must show PASS rise by ≥ 10 with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline.
