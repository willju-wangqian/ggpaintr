PLAN-05 — ptr_spec emission in preserve-mode code panel

Source: ADR 0012 (§2 row "ptr_spec", §3.5 "ptr_spec construction and shape", §4 S5) · Depends on: 04 · Parallel group: G4 · Status: draft

Context

ADR 0012 §3.5: "Construction is a one-pass diff: ptr_spec = snapshot ∖ registry_defaults, keyed by fully-qualified Shiny input id. The result is added to state as state$spec (reactive over the runtime snapshot)." The preserve-mode code panel emits a single self-contained call that includes the sparse spec.

This plan adds the construction + emission. Apply (replay) is plan 06's job.

Success Criteria

  • A new internal helper ptr_spec_from_snapshot(snapshot, registry_defaults) in R/paintr-server.R (or a new R/paintr-spec.R if the function grows past ~30 lines) returns a flat named list of fully-qualified input ids → values, omitting entries whose value identical()-matches the registry default for that id. Returns an empty named list when every value is at default.
  • ptr_state exposes state$spec as a reactive over state$runtime()'s frozen snapshot (the res$snapshot field already populated by the ADR-0009 preserve-mode work). Empty spec returned when state$runtime() is NULL (pre-first-Update).
  • A new internal helper ptr_spec_combine(specs) in the same file takes a list of per-plot spec lists (already namespaced by each engine) and returns one flat named list — the union of entries across plots, keyed by the fully-qualified namespaced id. If two specs ever supply the same fully-qualified id (should not happen if grid namespacing is correct), abort with rlang::abort() naming the colliding id; do not silently choose. Used by ptr_app_grid in plan 06 to produce a single grid-wide spec from the per-plot state$spec() reactives.
  • A new internal helper format_spec_for_panel(spec) renders the spec list as R source code text (using deparse + light indent), emitting nothing (empty string) when spec is empty.
  • ptr_register_code's preserve branch (in ptr_register_code in R/paintr-server.R) emits the existing formula text exactly as today, then — if and only if spec is non-empty — appends a blank line followed by a ptr_spec <- list(...) assignment beneath it. Shape: 
    <formula text>

ptr_spec <- list(
  `<id1>` = <value1>,
  `<id2>` = <value2>
)
    When spec is empty, the output is bit-identical to today's preserve-mode output (just the formula text). The formula is never wrapped in ptr_app(...) by this plan — the user assembles the formula + spec into a ptr_app(formula, spec = ptr_spec) call themselves when they want to recreate the app.
  • tests/testthat/test-ptr-spec-emission.R exists with at least:
    1. Sparse encoding: a snapshot with a single entry at default produces an empty spec.
    2. Sparse encoding: a snapshot with one default + one non-default entry produces a one-entry spec.
    3. Full-default short-circuit: format_spec_for_panel(empty_list) returns "".
    4. Round-trip readability: format_spec_for_panel(non_empty_spec) is parseable R that rlang::parse_expr() accepts.
    5. Preserve panel integration: when ptr_register_code's preserve mode is exercised via shiny::testServer() against a state with one non-default snapshot entry, the rendered text contains the formula text on top AND the ptr_spec <- list(…) assignment below it, separated by a blank line. The formula text alone is bit-identical to today's preserve output.
  • Authoritative gate (Definition of Done): FAIL 0 / ERROR 0 / SKIP 0 / PASS rises by ≥ 5 (new tests), no existing tests regress.

Constraints

  • Files this plan owns: R/paintr-server.R (add helpers + extend ptr_register_code preserve branch + add state$spec reactive to ptr_init_state), tests/testthat/test-ptr-spec-emission.R (new).
  • Files this plan must NOT touch: R/paintr-translate.R, R/paintr-prune.R, R/paintr-resolve.R, R/paintr-safe.R, R/paintr-app.R, any vignette, any fixture, NAMESPACE, man/. The spec= argument plumbing through wrappers belongs to plan 06.
  • Default-comparison: identical-only (base::identical()). Do not implement type-coerced equality (e.g., "TRUE" vs TRUE) — Shiny inputs preserve type, and identical-only avoids surprise drops.
  • Registry default lookup: use the existing placeholder registry's default field. If a registry entry has no documented default, treat the snapshot value as non-default (emit it). Do not invent defaults.
  • Quote backtick-fragile ids in the panel output (every id may contain underscores and digits; backtick-quoting is safest and matches the ADR 0012 §3.5 example).
  • No reactive watchers, no observer side-effects. state$spec is a pull-style reactive over state$runtime().

BDD scenarios

Feature: Preserve-mode code panel emits a sparse ptr_spec alongside the formula

Scenario: Empty spec collapses to today's preserve output
  Given a state whose runtime snapshot is empty (or every value matches its registry default)
  When  the preserve-mode code panel renders
  Then  the rendered text is bit-identical to today's preserve-mode output for the same formula
  And   does NOT contain the substring "ptr_spec"

Scenario: One non-default entry shows in spec
  Given a snapshot with `ggplot_2_stage_enabled = FALSE` (default is TRUE) and everything else at default
  When  the preserve-mode code panel renders
  Then  the rendered text contains the formula text first
  And   contains the substring "ptr_spec <- list("
  And   contains the substring "`ggplot_2_stage_enabled` = FALSE"
  And   does NOT contain entries for ids whose values are at default
  And   does NOT wrap the formula in a ptr_app(...) call

Scenario: Pure-function sparseness
  Given a snapshot list and a registry-defaults list with one identical entry and one differing entry
  When  I call ggpaintr:::ptr_spec_from_snapshot(snapshot, defaults)
  Then  the result has length 1
  And   the named id is the one whose value differed from default

Scenario: Format is parseable R
  Given a non-empty spec list
  When  I call ggpaintr:::format_spec_for_panel(spec) and pass the result through rlang::parse_expr()
  Then  parsing succeeds (no error)
  And   the parsed expression is a `list(...)` call with the same named entries

Scenario: state$spec reacts to runtime
  Given a state with no runtime result yet (state$runtime() is NULL)
  When  I read state$spec()
  Then  the result is an empty named list (no error, no warning)

Scenario: ptr_spec_combine unions per-plot specs
  Given two per-plot specs:
        spec_p1 = list(`p1-ggplot_1_1_ppVar_NA` = "x")
        spec_p2 = list(`p2-ggplot_2_stage_enabled` = FALSE)
  When  I call ggpaintr:::ptr_spec_combine(list(spec_p1, spec_p2))
  Then  the returned list has length 2
  And   contains both entries with their original namespaced ids unchanged

Scenario: ptr_spec_combine errors on colliding ids
  Given two per-plot specs whose ids accidentally collide (e.g., both contain `p1-foo`)
  When  I call ggpaintr:::ptr_spec_combine(list(spec_a, spec_b))
  Then  the call errors with class "rlang_error"
  And   the error message names the colliding id

Proxy traps — do not be fooled (copied verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard. If a check run shows the e2e file ERRORing, that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests. That alone is the gate.

This plan adds ~5 new tests in one file. The gate must show PASS rise by that count, FAIL/ERROR still 0, SKIP unchanged.
