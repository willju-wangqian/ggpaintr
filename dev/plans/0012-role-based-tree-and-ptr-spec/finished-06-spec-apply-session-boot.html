PLAN-06 — spec= apply at session boot (round-trip closes the loop)

Source: ADR 0012 (§2 rows "spec= plumbing", "Apply timing of spec="; §3.6 "spec= apply at session boot"; §4 S6) · Depends on: 05 · Parallel group: G5 · Status: draft

Context

ADR 0012 §3.6 lays out the boot-time apply: per-entry namespace check, silent drop of unknown ids with an aggregated cli::cli_inform, updateXyzInput at session boot via session$onFlushed, drop with cli::cli_warn on invalid choice. §2 row "spec= plumbing" locks the single engine + four wrappers contract.

Verified (ADR 0012 §2): ptr_app / ptr_app_bslib / ptr_app_grid / ptr_server all reach ptr_server_internal. The spec= argument lives in the engine; wrappers thread it through.

Success Criteria

  • ptr_server_internal gains a spec = NULL argument. When non-NULL, it must be a named list of fully-qualified Shiny input id → value.
  • A new internal helper apply_spec_at_boot(spec, session, ns_prefix) in R/paintr-server.R implements the four-step protocol from ADR 0012 §3.6:
    1. Filter spec entries whose id starts with ns_prefix.
    2. Among those, drop entries whose stripped id is unknown to this engine instance's registry / has no bound widget; collect the dropped ids into a vector emitted as a single cli::cli_inform at the end.
    3. For each surviving entry, dispatch to the appropriate updateXyzInput based on the widget type registered for that id. The dispatch table is keyed by the placeholder hook's widget kind (existing registry field; no new registry shape).
    4. If the value is not a valid choice for the widget (e.g., not in the picker's current choices), drop with a single cli::cli_warn naming the id and the invalid value.
    All updateXyzInput calls happen inside session$onFlushed(once = TRUE) so widgets exist when touched.
  • ptr_app, ptr_app_bslib, ptr_app_grid, and ptr_server each gain a spec = NULL argument with identical semantics, threaded through to ptr_server_internal. For ptr_app_grid, the same flat spec= list is passed to every per-plot ptr_server_internal call; each engine instance filters by its own namespace prefix.
  • A new e2e fixture tests/testthat/fixtures/vignette-apps/adr12-spec-roundtrip/app.R exists. The app boots ptr_app(formula, spec = list(...)) with a precooked spec, asserts that on session start the widgets named in the spec carry the spec values (rather than defaults).
  • tests/testthat/test-adr12-spec-roundtrip.R exists, gated like the other browser tests. The test:
    1. Boots the fixture under AppDriver$new(...).
    2. Asserts that for at least one stage-checkbox id in the spec, the rendered widget value matches the spec's value (not the default).
    3. Asserts that for at least one var-picker id in the spec, the rendered selected option matches the spec's value.
    4. Asserts that an unknown id in the spec is silently dropped (no app crash, no error visible to the user).
  • Authoritative gate (Definition of Done): FAIL 0 / ERROR 0 / SKIP 0 / PASS rises by ≥ N_new_assertions.

Constraints

  • Files this plan owns: R/paintr-server.R (new apply_spec_at_boot helper + new spec= arg on ptr_server_internal), R/paintr-app.R (new spec= arg on ptr_app, ptr_app_bslib, ptr_app_grid, ptr_server plus ptr_make_app_server / ptr_app_grid_components as needed for threading), tests/testthat/fixtures/vignette-apps/adr12-spec-roundtrip/app.R (new), tests/testthat/test-adr12-spec-roundtrip.R (new).
  • Files this plan must NOT touch: R/paintr-translate.R, R/paintr-prune.R, R/paintr-resolve.R, R/paintr-safe.R, any other vignette or fixture, NAMESPACE, man/ (roxygen entries for the new spec= arg ARE in scope — they live in the same source files this plan owns; document the new arg in the existing function roxygen blocks).
  • One-shot apply only: no observe() or observeEvent() watching the spec, no re-application after boot. After the onFlushed handler fires, the spec is no longer consulted.
  • Unknown-id behavior: silent drop with one aggregated cli::cli_inform at the end of the apply pass listing all dropped ids (not a per-id message). Invalid-value behavior: cli::cli_warn per affected id.
  • The spec= argument default is NULL for backward compatibility — existing callers see no behavior change.
  • For ptr_app_grid: pass the same flat spec to every per-plot engine; each engine filters by namespace prefix. Do not pre-partition the spec at the grid level.
  • For ptr_app_grid OUTPUT: expose a grid-level combined spec by calling ptr_spec_combine() (the helper from plan 05) over the list of per-plot state$spec() reactives. The combined spec is the single flat namespaced list a user can paste back into ptr_app_grid(spec = ...) for round-trip. Wire this into whatever grid-level accessor exists (today there is no grid-level code panel; minimum bar is that the combiner is callable from the grid's returned components — extend the return shape if needed).
  • Document each new spec= arg with one roxygen line per function: @param spec An optional named list of fully-qualified Shiny input id → value, used to override widget defaults at session boot. See ADR 0012.

BDD scenarios

Feature: spec= replays user picks at session boot

Scenario: Stage checkbox honored at boot
  Given the fixture app booted with spec = list(`ggplot_2_stage_enabled` = FALSE)
  When  the app boots (session$onFlushed fires once)
  Then  the rendered stage checkbox for the filter stage is unchecked

Scenario: Var picker honored at boot
  Given the fixture app booted with spec = list(`ggplot_1_1_ppVar_NA` = "bill_depth_mm")
        and the upstream data has columns including "bill_depth_mm"
  When  the app boots
  Then  the rendered ppVar picker shows "bill_depth_mm" as the selected option

Scenario: Unknown id silently dropped
  Given the fixture app booted with spec = list(`no_such_widget_id_99` = "x")
  When  the app boots
  Then  the app does not crash
  And   no rendered error is visible to the user
  And   a single cli::cli_inform message lists the dropped id (visible in app log / test stderr)

Scenario: spec=NULL preserves today's behavior
  Given an existing fixture app boot path that does not pass spec=
  When  the app boots
  Then  every widget shows its registry default (identical to today's behavior — no regression)

Scenario: Grid: same flat spec across plots
  Given a ptr_app_grid call with two plots (p1, p2) and spec containing entries with both `p1-` and `p2-` prefixes
  When  the grid boots
  Then  p1's widgets reflect spec entries whose ids start with `p1-`
  And   p2's widgets reflect spec entries whose ids start with `p2-`

Definition of Done

Authoritative gate (copied verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected: FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline_after_plan_05 + N_new). The browser e2e tests run (not silently SKIP) under NOT_CRAN=true.

Proxy traps — do not be fooled (copied verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard. If a check run shows the e2e file ERRORing, that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests. That alone is the gate.

This plan adds ~4 browser-driven assertions in test-adr12-spec-roundtrip.R plus likely 2–3 unit tests covering apply_spec_at_boot's pure-function pieces (namespace filtering, unknown-id collection). Gate must show PASS rise by that total, FAIL/ERROR still 0, SKIP unchanged.
