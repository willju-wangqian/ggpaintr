PLAN-02 — resugar_pipeline lift + terminal grounding

Source: ADR 0012 (§2 "Lift rule", §3.1 "The lift transform", §3.2 "Terminal-grounding check", §4 S2) · Depends on: 01 · Parallel group: G2 · Status: draft

Context

ADR 0012 §1.1–§1.2 documents the root divergence: a formula written with |> or as nested calls produces a data_arg of class ptr_call, while the semantically-identical %>% form produces a ptr_pipeline with explicit $stages. The role-based redesign collapses both surface forms to one canonical ptr_pipeline by structural lift.

The lift rule is terminal-grounded: a nested-call chain becomes a pipeline iff its innermost first-arg is a known data source. Unknown terminal → quiet failure, stays a ptr_call (ADR 0012 §2 row "Lift rule", §3.2).

Success Criteria

  • resugar_pipeline(expr) (internal, in R/paintr-translate.R) implements the engine sketch from ADR 0012 §3.1: walks a nested-call chain by descending into cur[[2L]], collecting stages into a list whose head is the deepest first-arg and whose tail is the user-order stages.
  • ptr_classify_calls(root) (body filled in this plan; slot reserved by plan 01) walks each layer's data_arg; when data_arg is a ptr_call whose underlying R expression has a nested call as its first arg AND the deepest first-arg grounds to a known data source, rewrites data_arg to the ptr_pipeline shape (same constructor that the existing %>% path uses via translate_pipeline / collect_pipe_stages).
  • Terminal grounding rule (closes ADR §5 open question 1, verified 2026-05-21): The lift fires iff the deepest first-arg of the chain — the value left in cur after resugar_pipeline's loop terminates — is not itself a call. I.e., it must be a symbol, a literal, a ptr_ph_data_source, or any other non-call AST node. If the deepest first-arg IS a call (e.g., make_data(), read_csv("foo.csv"), data.frame(x = 1)), the chain stays as a plain ptr_call — quiet failure. Rationale: today's ptr_classify_data only tags ptr_ph_data_source nodes (verified by reading R/paintr-classify.R — no is.symbol, no inherits…data.frame, no eval-env probe), so the original ADR §3.2 three-category rule would only fire for placeholder sources. The non-call rule is purely structural, requires no eval-env lookup, and naturally captures all three intended cases (placeholder, bare data-name symbol, literal value) while filtering out opaque-call terminals.
  • The post-classify tree for "ppUpload |> filter(ppVar > ppNum) |> mutate(new_var = ppVar + 1) |> ggplot(aes(ppVar, ppVar)) + geom_point()" is structurally identical() to the post-classify tree for the same formula with %>% substituted for |> — i.e., the subbed-1 vs subbed-2 divergence from ADR 0012 §1.1 is closed. Compare on tree$layers[[1]]$data_arg: both must be ptr_pipeline with the same $stages shape and the same $op.
  • ptr_translate output for plain (non-pipeable) calls — e.g., "ggplot(penguins, aes(bill_length_mm)) + geom_point()" — is bit-identical to the plan-01 baseline (no lift fires; the ggplot() call's first arg penguins has no nested call inside, so the loop terminates immediately).
  • Unknown-terminal case: "some_user_helper() |> ggplot(aes(x)) + geom_point()" stays as ptr_call data_arg, no lift, no warning. (Quiet failure per ADR 0012 §3.2.)
  • Authoritative gate (see Definition of Done) shows FAIL 0 / ERROR 0 / SKIP 0 / PASS rising by ≥ N_new_tests with all existing tests still passing.

Constraints

  • Files this plan must NOT touch: R/paintr-prune.R (the elide logic is unchanged — explicitly per ADR 0012 §3.3), R/paintr-server.R, R/paintr-app.R, R/paintr-resolve.R, R/paintr-safe.R, any vignette, any fixture under tests/testthat/fixtures/vignette-apps/, NAMESPACE, man/. The bug-3b regression test and the runtime-fast-path deletion belong to plans 03 and 04 respectively.
  • Reuse the existing ptr_pipeline constructor and the existing translate_pipeline / translate_node machinery; do not introduce a parallel pipeline-construction path. The lift's job is to identify a pipeline structure and feed the stages into the same constructor the %>% path uses.
  • Honor the + fence (ADR 0012 §3.4): the lift pass walks data_arg subtrees only; never descends into the layer's children (aes / mapping / aesthetic args), never crosses a layer boundary.
  • No exported functions; no NAMESPACE changes; no @export roxygen tags.
  • snake_case, 2-space indent, rlang::abort() for errors, assertthat::assert_that() for argument validation.

BDD scenarios

Feature: Lift nested-call data args into a canonical ptr_pipeline when the chain grounds to a known data source

Scenario: |> input now produces the same tree shape as %>% input
  Given two formulas that differ only in pipe syntax:
        f_native  = "ppUpload |> filter(ppVar > ppNum) |> mutate(new_var = ppVar + 1) |> ggplot(aes(ppVar, ppVar)) + geom_point()"
        f_magrittr = "ppUpload %>% filter(ppVar > ppNum) %>% mutate(new_var = ppVar + 1) %>% ggplot(aes(ppVar, ppVar)) + geom_point()"
  When  I run ptr_translate(f_native) and ptr_translate(f_magrittr)
  Then  both trees have layer[1]$data_arg of class "ptr_pipeline"
  And   both pipelines have the same number of $stages
  And   the stage names (filter / mutate) in order match across the two trees

Scenario: Pure-function lift on a plain R expression
  Given the R expression `quote(mutate(filter(penguins, x > 1), y = z + 1))`
  When  I call ggpaintr:::resugar_pipeline(expr)
  Then  the returned list has length 3
  And   the first element is the symbol `penguins` (the source)
  And   the second element is the call `filter(x > 1)` (filter stage, source removed)
  And   the third element is the call `mutate(y = z + 1)` (mutate stage, source removed)

Scenario: No lift when there is no nested-call chain
  Given the formula "ggplot(penguins, aes(bill_length_mm)) + geom_point()"
  When  I run ptr_translate(formula)
  Then  layer[1]$data_arg is of class "ptr_literal" (or whatever ptr_classify_data tagged the bare `penguins` as)
  And   no ptr_pipeline node was constructed

Scenario: Bare data-name symbol grounds the lift
  Given the formula "penguins |> filter(bill_length_mm > 40) |> ggplot(aes(bill_length_mm)) + geom_point()"
        (`penguins` is a bare symbol — not a placeholder, not a call)
  When  I run ptr_translate(formula)
  Then  layer[1]$data_arg is of class "ptr_pipeline"
  And   the pipeline's first stage corresponds to the `penguins` symbol

Scenario: Quiet failure when terminal is an opaque CALL
  Given the formula "make_data() |> filter(x > 1) |> ggplot(aes(x)) + geom_point()"
        (the deepest first-arg `make_data()` is itself a call)
  When  I run ptr_translate(formula)
  Then  layer[1]$data_arg is of class "ptr_call" (not ptr_pipeline)
  And   no warning, no error, no cli message is emitted

Scenario: Quiet failure when terminal is read_csv("foo.csv")
  Given the formula "read_csv(\"foo.csv\") |> filter(x > 1) |> ggplot(aes(x)) + geom_point()"
  When  I run ptr_translate(formula)
  Then  layer[1]$data_arg is of class "ptr_call" (no lift — opaque call terminal)

Scenario: + fence holds
  Given the formula "ggplot(aes(x = filter(penguins, y > 1))) + geom_point()" (a nested call inside aes(), not the data arg)
  When  I run ptr_translate(formula)
  Then  the aes() child is unchanged — no lift was applied inside the layer's children subtree

Proxy traps — do not be fooled (copied verbatim from CLAUDE.md):

The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard. If a check run shows the e2e file ERRORing, that guard is missing/broken — a harness defect, not a product failure, and never the gate.
