ggpaintr Extensibility — Technical Reference

A developer reference for embedding, extending, and integrating ggpaintr. Covers the L1/L2/L3 architecture, complete working examples, and placeholder design rationale. Every claim is grounded in source code read from R/, fixture apps in tests/testthat/fixtures/vignette-apps/, and the authoritative domain document CONTEXT.md.

1. L1 / L2 / L3 Entry-Point Architecture

Overview and motivation

ggpaintr converts a ggplot-like formula string into a running Shiny app. The formula may contain placeholder tokens (var, text, num, expr, upload) that become Shiny input widgets; substitution happens at render time. This single transformation pipeline is exposed at three levels of abstraction.

The layered design solves a fundamental tension: a one-line entry point that handles 80 % of use cases must coexist with fine-grained control for the remaining 20 %. A flat API forces every embedder to know about Shiny module namespacing, asset injection, reactive wiring, and custom renderer lifecycle — details that only matter when the defaults are insufficient. A tiered API lets each level absorb exactly the complexity required.

LevelEntry pointsWhat the caller controlsWhat ggpaintr controls
L1ptr_app(), ptr_app_grid()Formula string, minor style overridesEverything: layout, reactive wiring, rendering, assets
L2ptr_module_ui() / ptr_module_server(), shared trioHost layout, host navigation, placement of the module blockDefault layout inside the block, reactive wiring, rendering, assets
L3ptr_ui_* bare pieces, combinators, ptr_server()Every piece of markup and placement, custom renderer off state$runtime()Reactive wiring only (producer inputs, pipeline, runtime observer, output registration)

L1 is not a "simplified L2" — it internally composes the same bare pieces an L3 user assembles by hand (ptr_app_components() builds a tree, calls ptr_build_app_ui(), and wires ptr_make_app_server() R/paintr-app.R: ptr_app_components()). The three levels are three views of the same runtime, not three separate implementations.

Parse layer: ptr_translate()

All three levels share a single formula-to-tree transformation. ptr_translate(formula, expr_check = TRUE) R/paintr-translate.R: ptr_translate() performs these steps in order:

  1. Pre-process native pipe (|>) to make R's parser happy.
  2. Parse the string with rlang::parse_exprs(). A formula must be a single expression; multiple top-level expressions are rejected.
  3. Split on + operators to enumerate layers.
  4. Translate each layer expression into a typed AST node tree (ptr_root → ptr_layers → ptr_calls → ptr_ph_* leaves and ptr_literal/ptr_closure nodes).
  5. Deduplicate layer names (repeated geom_point becomes geom_point, geom_point-2).
  6. Run the safety walker (ptr_validate_tree_safety()) if expr_check = TRUE — this recursively checks every AST node, including expr placeholder bodies, against the denylist.
  7. Annotate: assign stable input ids (ptr_assign_ids()), classify data-arg nodes (ptr_classify_data()), bind shared-key metadata (ptr_shared_bind()).

The resulting ptr_root object is the single source of truth for both the UI builders (which read placeholder node ids to know what inputIds to emit) and the runtime (which reads the same ids to pull from input$). Ids are deterministic from the formula, so the same formula always produces the same widget ids.

L1 — All-in-one

L1 turns a formula string into a fully self-contained shiny.appobj in one call. Two entry points:

ptr_app( formula, envir = parent.frame(), ui_text = NULL, checkbox_defaults = NULL, expr_check = TRUE, safe_to_remove = character(), css = NULL ) → shiny.appobj # R/paintr-app.R: ptr_app() ptr_app_grid( plots, # named or unnamed list of formula strings shared_ui = list(), envir = parent.frame(), ui_text = NULL, draw_all_label = "Draw all", expr_check = TRUE, css = NULL ) → shiny.appobj # R/paintr-app.R: ptr_app_grid()

ptr_app() delegates immediately to ptr_app_components(), which calls ptr_translate(), then ptr_build_app_ui() and ptr_make_app_server(), and returns a list list(ui, server). The returned objects are passed straight to shiny::shinyApp().

ptr_app_grid() is not simply N × ptr_app() — it builds a shared coordinator internally (ptr_app_grid_components()), wires a top-level Draw all button whose click propagates to every module's runtime observer via a draw_trigger reactive, and places a standalone shared panel above the grid when any formula uses shared = "..." annotations.

L2 — Self-contained embed

L2 is "plug ggpaintr into my own Shiny app, accept the default layout." The module pair is the only default-layout block at L2; there is no controls-here/plot-there split at this level (that is L3).

ptr_module_ui( id, formula, ui_text = NULL, checkbox_defaults = NULL, expr_check = TRUE, css = NULL, shared = NULL # ptr_shared_spec from ptr_shared(), or NULL ) → shiny.tag # R/paintr-app.R: ptr_module_ui() ptr_module_server( id, formula, envir = parent.frame(), ..., # forwarded to ptr_init_state() shared_state = NULL # ptr_shared_state from ptr_shared_server() ) → ptr_state invisibly # R/paintr-app.R: ptr_module_server()

ptr_module_ui() is implemented by composing the L3 bare pieces — it internally calls ptr_ui_controls(), ptr_ui_plot(), ptr_ui_error(), ptr_ui_code(), and the two combinators, wrapping them in fluidPage > div.ptr-app > sidebarLayout R/paintr-app.R: ptr_module_ui() lines 649–688. It emits the asset bundle once via ptr_ui_assets(); htmlDependency deduplicates it page-wide so N modules load assets once.

ptr_module_server() wraps shiny::moduleServer(id, ...). Inside, it calls ptr_server() with the module session's auto-namespace plumbing (ns = session$ns, server_ns = shiny::NS(NULL)). It returns the ptr_state so the caller can react to it (e.g. call ptr_gg_extra()).

Shared trio (multi-instance only):

ptr_shared( formulas, # list of formula strings shared_ui = list(), # map: key → function(id) → shiny.tag ui_text = NULL, expr_check = TRUE, draw_all_label = "Draw all" ) → ptr_shared_spec # R/paintr-shared-coordinator.R: ptr_shared() ptr_shared_panel( obj, # ptr_shared_spec css = NULL ) → shiny.tag | NULL # R/paintr-shared-coordinator.R: ptr_shared_panel() ptr_shared_server( obj, # ptr_shared_spec envir = parent.frame(), shared = list(), # reactive overrides for panel keys draw_trigger = NULL ) → ptr_shared_state # R/paintr-shared-ui.R: ptr_shared_server()

ptr_shared() is a pure, non-reactive computation. It runs ptr_translate() on every formula, collects all shared = "..." annotations, and applies the partition rule: keys referenced by exactly one formula are formula-local (inline shared section inside that module's control panel); keys referenced by two or more formulas are panel keys (standalone shared panel, global ids). The resulting ptr_shared_spec is the single source of truth — obj$panel_keys and obj$local_keys_by_formula are read by the UI and server independently, ensuring they can never disagree.

ptr_shared_panel() renders only obj$panel_keys. It returns NULL when the panel body is empty (valid configuration — see below). It must be placed exactly once per page. The panel uses global, un-namespaced ids (shared_<key>, ptr_shared_draw_all, ptr_shared_errors); placing it more than once would produce duplicate ids.

ptr_shared_server() must be called at the top level of the server function, never inside moduleServer(). It reads input$shared_<key> at the top-level namespace (modules cannot reach those slots through their namespaced sessions). It returns a ptr_shared_state bundle, which is threaded into each ptr_module_server(shared_state = ...).

Single-instance invariant: ptr_shared() / ptr_shared_panel() / ptr_shared_server() are strictly multi-instance API. One ggpaintr instance should never use them — every shared key is formula-local by definition, auto-renders inline, and requires no coordinator. ptr_module_server() given a shared = "..." formula without a coordinator wiring will abort() with a clear message.

L3 — Own the layout and the render path

L3 gives the embedder control over every piece of markup and — critically — the ability to replace ggpaintr's built-in renderPlot() with any renderer (Plotly, ggiraph, custom). The enabling mechanism is the return value of ptr_server(): a ptr_state object that exposes a reactive state$runtime() from which the caller reads the computed ggplot object.

ptr_server( input, output, session, formula, envir = parent.frame(), ..., # forwarded to ptr_init_state() shared_state = NULL ) → ptr_state # R/paintr-server.R: ptr_server()

ptr_server() is the L3 server entry point. Inside it calls ptr_init_state() to allocate reactive slots, then runs a sequence of setup functions in order:

  1. ptr_setup_producer_inputs() — builds debounced reactives for every placeholder's input slot.
  2. ptr_setup_pipelines() — wires upload / data-source observers.
  3. ptr_setup_stage_enabled() — toggles for per-stage pipeline enable/disable checkboxes.
  4. ptr_setup_shared_stage_enabled() — mirrors shared stage state into the module.
  5. ptr_setup_runtime() — the main observer: fires on Update plot click or draw_trigger, calls ptr_exec_headless(), stores the result in state$runtime.
  6. ptr_setup_consumer_uis() — renderUI observers that rebuild var column pickers when upstream data changes.
  7. ptr_setup_layer_picker() / ptr_setup_layer_panel_classes() — layer navigation tab wiring.
  8. ptr_register_plot() / ptr_register_error() / ptr_register_code() — bind the three canonical outputs to output$ptr_plot, output$ptr_error, output$ptr_code. These are always registered; if no UI piece is present for a given id, the binding is inert (Shiny simply never requests it).

The L3 bare pieces emit only their widgets and carry no self-wrap, no asset bundle, and no theme scope:

Combinator nesting: combinators take already-built pieces as arguments and return a new tag tree. They are nestable: the output block used internally by ptr_app() and ptr_module_ui() is exactly:

ptr_ui_toggle_code(
  ptr_ui_inline_error(ptr_ui_plot(id), ptr_ui_error(id)),
  ptr_ui_code(id)
)

Naming convention

The ptr_<x> vs ptr_ui_<x> distinction encodes level ownership (locked in CONTEXT.md, enforced by ADR 0005):

  • ptr_<x> = L2 self-contained: owns its own .ptr-app scope and asset bundle. Drop it into a host page with no scaffolding. Examples: ptr_module_ui(), ptr_shared_panel().
  • ptr_ui_<x> = L3 bare: emits only its widgets. The L3 user supplies their own Shiny shell (or uses ptr_ui_page()). Examples: ptr_ui_controls(), ptr_ui_plot(), ptr_ui_inline_error().

The exception is the module pair (ptr_module_ui / ptr_module_server), which use the ptr_ prefix for historical reasons and are the L2 sealed-bundle exception documented in ADR 0004.

The ptr_state object

ptr_init_state() R/paintr-server.R: ptr_init_state() allocates all reactive slots for one formula instance. The object is a named list of class ptr_state. The slots relevant to L3 integrators:

SlotTypeDescription
runtimereactiveVal(NULL)Set by ptr_setup_runtime() on every draw. Read as state$runtime() inside a reactive context. The value is a list with slots $ok (logical), $plot (ggplot2 object), $code_text (character), $error (character | NULL).
extrasreactiveVal(list())ggplot2 layer objects appended by ptr_gg_extra(). Folded into the plot at render time.
eval_envenvironmentChild of the caller's envir. Pipeline-head data sources resolve here.

Two reading paths off ptr_state:

ContextHow to readEstablishes reactive dependency?

Never mix paths in a render block. Calling ptr_extract_plot(state) inside renderPlotly({...}) silently breaks reactivity — the block fires once on mount and never again. Use state$runtime()$plot inside reactive contexts, ptr_extract_*() outside.

2. Complete Working Examples

All examples are runnable after library(ggpaintr). The fixture apps in tests/testthat/fixtures/vignette-apps/ are verbatim-equivalent to the corresponding vignette chunks and are used by the shinytest2 browser test suite.

L1 — Single plot: ptr_app()

The minimal entry point. The formula uses three var placeholders (x-axis, y-axis, color), a text placeholder for the chart title, and a num placeholder for point size. The user gets a sidebar with three column pickers, a text field, a number input, and an Update plot button, plus a code pane showing the generated call.

library(ggpaintr)

ptr_app(
  "ggplot(iris, aes(x = var, y = var, color = var)) +
     geom_point(size = num) +
     labs(title = text)"
)

Why this formula? iris is in scope without any setup — it is an R base dataset, not an upload. The three var tokens resolve against iris's columns at runtime; the text token allows an arbitrary chart title without restricting input to a fixed list; the num token accepts a numeric size value directly rather than forcing a string-parse. See § 3 for full justification of each placeholder type.

Source: tests/testthat/fixtures/vignette-apps/app-basic/app.R, vignette chunk app-basic.

L1 — Pipeline formula

ggpaintr supports data-arg pipelines — a chain of |> or %>% stages before the final ggplot() call. Each stage that contains a placeholder gets its own widget row under the layer's Data subtab, plus a per-stage enable/disable toggle.

library(ggpaintr)

ptr_app(
  "mpg |>
     dplyr::filter(displ > num) |>
     dplyr::group_by(class) |>
     dplyr::filter(dplyr::n() > num) |>
     dplyr::ungroup() |>
     ggplot(aes(var, var, color = class)) +
     geom_point(alpha = num)"
)

What happens under the hood: ptr_translate() walks the pipeline chain and creates a ptr_pipeline node attached to the ggplot layer's data argument. Each pipeline stage containing a placeholder becomes a per-stage observer in ptr_setup_pipelines(). Column pickers for var downstream of a stage lazily re-resolve against the current filtered data — changing the displacement threshold immediately refreshes the picker options. The generated code panel shows the fully resolved pipeline.

Note: anonymous lambda functions as pipeline stages (df |> (\(x) x[x$cyl > 4, ])()) are not supported — the runtime cannot assign a stable Shiny id to a nameless stage. Lift lambdas to named helpers in the calling environment.

L1 — Multiple plots: ptr_app_grid()

Two plots with a shared x-axis column picker at the top. Annotating both var tokens with shared = 'metric' and supplying shared_ui promotes the picker to a single widget that drives both tiles when the user clicks Draw all.

library(ggpaintr)

ptr_app_grid(
  plots = list(
    "ggplot(iris, aes(x = var(shared = 'metric'), y = Sepal.Length, fill = Species)) +
       geom_boxplot()",
    "ggplot(iris, aes(x = var(shared = 'metric'), y = Sepal.Width, fill = Species)) +
       geom_violin()"
  ),
  shared_ui = list(metric = function(id) selectInput(id, "Metric", names(iris)))
)

The 'metric' key appears in both formulas → the partition rule sends it to the standalone shared panel at the top of the grid. Each plot still has its own Update plot button; the Draw all button fires both simultaneously. Internally, ptr_app_grid_components() builds a ptr_shared_spec, calls ptr_shared_server() at the top level, and wires the resulting bundle into two ptr_module_server() calls — the same machinery the L2 shared trio exposes.

Source: tests/testthat/fixtures/vignette-apps/app-grid-shared-added/app.R, vignette chunk app-grid-shared-added.

L2 — Single module embed

The host app has its own navigation and layout; one ggpaintr block lives as a section inside it. ptr_module_ui() and ptr_module_server() handle namespacing automatically — all ggpaintr-generated widget ids are prefixed p-.

library(ggpaintr)
library(shiny)

formula <- "ggplot(iris, aes(var, var, color = var)) + geom_point()"

ui <- fluidPage(
  titlePanel("My host app"),
  # The host app could have its own sidebar, tabs, or other panels here.
  # ptr_module_ui owns its own .ptr-app scope + asset bundle.
  ptr_module_ui("p", formula)
)
server <- function(input, output, session) {
  # Returns the ptr_state invisibly; capture if you need ptr_gg_extra() etc.
  ptr_module_server("p", formula)
}

shinyApp(ui, server)

Why use L2 here instead of L1? The host app owns the page shell — titlePanel(), navigation, or other panels above/below the ggpaintr block. L1 ptr_app() builds a complete shinyApp that occupies the entire page. L2 drops a self-contained module into a larger app without interfering with the host's layout or its own Shiny inputs.

Source: tests/testthat/fixtures/vignette-apps/module-app/app.R, vignette chunk module-app.

L2 — Multiple linked modules

Two ggpaintr modules, each with a different formula (boxplot vs. violin), sharing a single x-axis column picker rendered in the shared panel above the modules. Changing the picker and clicking Draw all redraws both.

library(ggpaintr)
library(shiny)

plots <- list(
  "ggplot(iris, aes(x = var(shared = 'metric'), y = Sepal.Length, fill = Species)) +
     geom_boxplot()",
  "ggplot(iris, aes(x = var(shared = 'metric'), y = Sepal.Width,  fill = Species)) +
     geom_violin()"
)

# Step 1: build the coordinator once from the full formula list.
# 'metric' is referenced in both formulas → panel key → standalone panel.
obj <- ptr_shared(
  formulas = plots,
  shared_ui = list(metric = function(id) selectInput(id, "Metric", names(iris)))
)

ui <- fluidPage(
  titlePanel("My host app"),
  # Step 2: the one standalone shared panel (Draw all + metric picker).
  # Place it exactly once. Returns NULL when panel_keys is empty.
  ptr_shared_panel(obj),
  fluidRow(
    # Step 3: pass obj to ptr_module_ui so it excludes panel keys from the
    # inline shared section (they are already in the panel above).
    column(6, ptr_module_ui("plot_1", plots[[1]], shared = obj)),
    column(6, ptr_module_ui("plot_2", plots[[2]], shared = obj))
  )
)
server <- function(input, output, session) {
  # Step 4: call ptr_shared_server() ONCE at the top level — never inside
  # moduleServer(). It reads input$shared_metric at the top-level namespace.
  sh <- ptr_shared_server(obj)
  # Step 5: thread the bundle into each module.
  ptr_module_server("plot_1", plots[[1]], shared_state = sh)
  ptr_module_server("plot_2", plots[[2]], shared_state = sh)
}

shinyApp(ui, server)

Namespace isolation in action: each module uses a unique id ("plot_1", "plot_2"). ggpaintr-generated ids inside module 1 are plot_1-ptr_plot, plot_1-ggplot_1_1_var_NA, etc. Module 2's ids are plot_2-ptr_plot, plot_2-ggplot_1_1_var_NA, etc. They coexist without collision. The shared panel's ids (shared_metric, ptr_shared_draw_all) live at the top-level namespace, outside any module scope.

Source: tests/testthat/fixtures/vignette-apps/l2-shared/app.R, vignette chunk l2-shared.

L2 — Partition in action: mixed local and panel keys

This example demonstrates how the partition rule separates formula-local keys from panel keys. ax1 appears only in plot 1, ax2 only in plot 2, and sz appears in both — so only sz reaches the standalone panel; the axis pickers remain inline in each module's own control panel.

library(ggpaintr)
library(shiny)

plots <- list(
  "ggplot(iris, aes(x = var(shared = 'ax1'), y = var(shared = 'ax1'),
                    color = Species)) + geom_point(size = num(shared = 'sz'))",
  "ggplot(iris, aes(x = var(shared = 'ax2'), y = Sepal.Width,
                    color = Species)) + geom_point(size = num(shared = 'sz'))"
)

obj <- ptr_shared(formulas = plots)
# obj$panel_keys        → "sz"   (cross-formula → standalone panel)
# obj$local_keys_by_formula[[1]] → "ax1" (only plot 1 → plot 1's inline section)
# obj$local_keys_by_formula[[2]] → "ax2" (only plot 2 → plot 2's inline section)

ui <- fluidPage(
  ptr_shared_panel(obj),  # renders sz only; ax1, ax2 stay inline
  fluidRow(
    column(6, ptr_module_ui("plot_1", plots[[1]], shared = obj)),  # ax1 inline
    column(6, ptr_module_ui("plot_2", plots[[2]], shared = obj))   # ax2 inline
  )
)
server <- function(input, output, session) {
  sh <- ptr_shared_server(obj)
  ptr_module_server("plot_1", plots[[1]], shared_state = sh)
  ptr_module_server("plot_2", plots[[2]], shared_state = sh)
}

shinyApp(ui, server)

What the partition rule enforces: ownership is exclusive. The sz widget lives exactly once (in the shared panel), and its reactive value is read from the top-level input$shared_sz by ptr_shared_server(). The modules never emit their own sz widgets — ptr_module_ui(shared = obj) excludes obj$panel_keys from each module's inline section. Double-binding (the host panel also binding formula-local keys, or vice versa) is explicitly prevented in ptr_shared_server() by the panel_consumer_keys = intersect(consumer_keys, panel_keys) guard.

Source: tests/testthat/fixtures/vignette-apps/l2-shared-partition/, vignette chunk l2-shared-partition.

L3 — Hand-laid-out bare pieces

Every pane placed individually. Header, controls, plot, error, and code are in separate regions. ptr_ui_page() provides the Bootstrap shell and the single .ptr-app scope.

library(ggpaintr)
library(shiny)

formula <- "ggplot(iris, aes(x = var, y = var, color = var)) + geom_point()"

ui <- ptr_ui_page(            # Bootstrap page + single .ptr-app scope + asset bundle
  ptr_ui_header("Iris explorer"),
  fluidRow(
    column(4, ptr_ui_controls(formula = formula)),
    column(8, ptr_ui_plot())   # bare plot card; no error slot
  ),
  ptr_ui_error(),               # error banner in its own row
  ptr_ui_code()                  # always-visible code card
)
server <- function(input, output, session) {
  # ptr_server() registers ptr_plot, ptr_error, ptr_code.
  # A piece you don't place is simply inert — its output binding fires into nothing.
  ptr_server(input, output, session, formula)
}

shinyApp(ui, server)

Why this is L3, not L2: the caller owns the page markup — error is below the plot instead of inline; code is always visible instead of behind a toggle. ptr_module_ui() cannot produce this layout. The server side is identical to L2; the difference is entirely in the UI composition.

What ptr_ui_page() does (and what to do when it's not enough): it expands to page(ptr_ui_assets(css), tags$div(class = "ptr-app", ...)). For navbarPage or bslib roots, inline this expansion by hand: add ptr_ui_assets() once inside the tab and wrap pieces in tags$div(class = "ptr-app", ...). ptr_ui_assets() is htmlDependency-backed and self-deduplicating, so repeating it across tabs is harmless.

Source: tests/testthat/fixtures/vignette-apps/l3-pieces/app.R, vignette chunk l3-pieces.

L3 — Combinator recipe: slide-out code + inline error

The same layout as ptr_app() / ptr_module_ui() internally, but placed inside a custom host layout. The combinator call is byte-for-byte what those L2 functions produce, so behavior and performance match exactly.

library(ggpaintr)
library(shiny)

formula <- "ggplot(iris, aes(x = var, y = var, color = var)) + geom_point()"

ui <- ptr_ui_page(
  ptr_ui_header("Iris explorer"),
  sidebarLayout(
    sidebarPanel(ptr_ui_controls(formula = formula)),
    mainPanel(
      ptr_ui_toggle_code(              # </> slide-out toggle ...
        ptr_ui_inline_error(
          ptr_ui_plot(),               # ... around plot + inline error
          ptr_ui_error()
        ),
        ptr_ui_code()                   # ... wrapped as the slide-out window
      )
    )
  )
)
# server unchanged from l3-pieces
server <- function(input, output, session) {
  ptr_server(input, output, session, formula)
}

shinyApp(ui, server)

How the combinators work: ptr_ui_inline_error(plot, error) mutates the plot card's tag tree in-place — it appends the error slot to plot$children[[2]] (the card body) so errors render inline below the plot image. ptr_ui_toggle_code(plotish, code) injects a </> button into the card head and wraps the whole thing in a div.ptr-output alongside a .ptr-code-window container. The bundled JavaScript in inst/www/ggpaintr-ui.js drives the slide-out animation purely DOM-locally off the CSS classes — no Shiny round-trip.

L3 — Custom Plotly renderer off state$runtime()

This is the canonical L3 render-path pattern. The default ptr_ui_plot() / renderPlot() pair is not placed — with no UI slot for ptr_plot, the binding registered by ptr_register_plot() is inert. Instead, a plotlyOutput() is placed at a custom id and bound to output$custom_plot inside a moduleServer() that also holds the ptr_server() call.

library(ggpaintr)
library(shiny)
library(plotly)

formula <- "ggplot(iris, aes(x = var, y = var, color = var)) + geom_point()"

ui <- ptr_ui_page(
  fluidRow(
    column(5,
      # ptr_ui_controls scoped to "plot1" namespace
      ptr_ui_controls("plot1", formula)
    ),
    column(7,
      # Custom output at the module-namespaced id NS("plot1")("custom_plot")
      plotlyOutput(NS("plot1")("custom_plot"), height = "500px") |>
        ptr_ui_toggle_code(ptr_ui_code("plot1"))  # code pane still works
    )
  )
)

server <- function(input, output, session) {
  # Wrap ptr_server() in your own moduleServer so namespacing is automatic
  # and plain-Shiny-idiomatic. No internal namespace plumbing needed.
  moduleServer("plot1", function(input, output, session) {
    # ptr_server() wires all reactive plumbing and registers ptr_plot
    # (inert — no ptr_ui_plot in the UI), ptr_error, ptr_code.
    state <- ptr_server(input, output, session, formula)

    # Custom renderer reads the computed ggplot from state$runtime().
    output$custom_plot <- renderPlotly({
      res <- state$runtime()        # reactive dependency → re-runs on draw
      req(isTRUE(res$ok), res$plot) # quiet between draws and on error
      ggplotly(res$plot)
    })
  })
}

shinyApp(ui, server)

Why this is definitionally L3: the caller owns the render path. ptr_server() computes the ggplot2 object and stores it in state$runtime()$plot; using that value to drive a custom renderer (Plotly, ggiraph, any host) is owning the render path — the enabling return value lives at the L2 server boundary (ptr_server() returning state), but using it to replace a pane is by definition L3.

The req(isTRUE(res$ok), res$plot) guard: on the first reactive flush before any draw click, state$runtime() returns NULL. After a failed draw, it returns a list with ok = FALSE. Without the guard, Plotly's renderer throws a null-dereference. With it, the output stays quiet until a successful draw.

L3 — Programmatic layer injection: ptr_gg_extra()

ptr_gg_extra(state, ...) R/paintr-server.R: ptr_gg_extra() evaluates ggplot2 expressions and folds them into the rendered plot at the next draw cycle. This allows the host app to add layers, scales, or themes programmatically — for example, toggling a log scale in response to a button press. The code panel includes the injected expression in its output.

library(ggpaintr)
library(shiny)

formula <- "ggplot(mtcars, aes(x = mpg, y = hp)) + geom_point()"

ui <- fluidPage(
  actionButton("add_log", "Add log scale"),
  ptr_module_ui("p", formula)
)
server <- function(input, output, session) {
  # Capture the state returned by ptr_module_server.
  state <- ptr_module_server("p", formula)
  observeEvent(input$add_log, {
    # Inject a scale; replaces any previous extras on this state.
    # The runtime folds it in on the next Update plot click.
    ptr_gg_extra(state, ggplot2::scale_x_log10())
  })
}

shinyApp(ui, server)

Semantics: each ptr_gg_extra() call replaces all previously captured extras. Eval errors leave existing extras intact (atomic update). Extras are suppressed when the runtime reports a failure. ptr_extract_code(state) includes extras expressions in the returned string.

Shared placeholders — single instance: inline section

One ggpaintr instance with one key used twice. Both x and y axes are driven by the same column picker. The widget renders automatically in the inline shared section inside the control panel. No coordinator, no panel, no shared trio.

library(ggpaintr)

# One instance, one key used twice: widget auto-renders inline. No coordinator.
ptr_app(
  "ggplot(iris, aes(x = var(shared = 'col'), y = var(shared = 'col'),
                    color = Species)) + geom_point()"
)

What shared = 'col' does here: both var occurrences are annotated with the same key. ptr_shared_bind() (inside ptr_translate()) links them to a single canonical input id shared_col. The UI builder renders exactly one widget for that key in the shared section; both aes slots read from it at render time. This is the simplest use of shared placeholders and requires no multi-instance machinery.

Shared placeholders — multiple instances: cross-formula panel

This is the same as the L2 multi-module example (§ 2.5), repeated here in the shared-placeholder context to make the mechanic explicit.

The key architectural point: var(shared = 'metric') tells the formula parser that this var token shares a common reactive value with every other occurrence of 'metric' in the formula set. When the coordinator finds 'metric' in both formulas, it assigns it to the shared panel. The panel renders one selectInput (supplied via shared_ui); ptr_shared_server() builds one reactive input$shared_metric; each module reads it from the ptr_shared_state bundle. The two plots never read each other's state directly — all cross-plot coordination flows through the top-level session's input namespace.

3. Placeholder Justification

Each placeholder type corresponds to a specific data-analysis decision and a specific Shiny input widget. Choosing the wrong type produces either the wrong widget (the user can enter invalid values) or unnecessary restriction (the user cannot enter valid values). This section justifies every choice made in the examples above.

var — column picker (data-aware selectInput)

What it is: a selectInput whose choices are dynamically populated from the columns of the upstream data at the point where the placeholder appears in the formula. Implemented in ptr_builtin_var_build_ui() / ptr_builtin_var_resolve_expr() R/paintr-builtins.R. The widget re-resolves its choices whenever upstream data changes (e.g. a filter stage changes the available columns, or an upload lands).

Role in the registry: var is a data consumer placeholder — it is registered via ptr_define_placeholder_consumer() and has a resolve_expr hook that converts the user's string selection to an unquoted symbol (e.g. "Sepal.Length" → Sepal.Length) for insertion into the ggplot2 aes call.

Why var and not text for aes mappings: text would allow the user to type any arbitrary string, including column names that do not exist, syntactically invalid R identifiers, or expressions that would silently misbehave. var restricts choices to actual columns of the upstream data, giving the analyst immediate visual feedback (the picker is pre-populated with valid options) and preventing the most common class of runtime errors ("object 'x' not found in data").

Uses in the examples:

  • aes(x = var, y = var) — axis column selection. The analyst wants to choose which numeric column maps to each axis; only actual data columns are valid.
  • aes(color = var) — color group selection. Same rationale; the palette is computed from a column's levels.
  • aes(x = var(shared = 'metric')) — shared axis selection. Same widget, but driven by the shared section/panel so it synchronizes across multiple plots or multiple occurrences in the same formula.

When not to use var: if the valid values are not data columns (e.g. a color name, a geom parameter that takes a string constant), use text or num. Using var against non-data positions would populate the picker with column names that are meaningless in that argument position.

text — free-text input

What it is: a plain textInput that accepts any string. Implemented in ptr_builtin_text_build_ui() / ptr_builtin_text_resolve_expr() R/paintr-builtins.R. The resolved expression is a string literal — "My title" is substituted literally into the generated ggplot2 call.

Uses in the examples:

  • labs(title = text) — chart title. The analyst wants to type an arbitrary label; there is no restricted vocabulary. Using var here would populate the picker with column names (wrong domain); using num would force a number (wrong type). text is the only placeholder that accepts an arbitrary string.

Empty-string behavior: when text resolves to an empty string, the argument is treated as missing. If that leaves labs() with no arguments, and labs is in ggpaintr's curated cleanup list, the entire labs() call is dropped from the generated code. This avoids labs(title = "") appearing in the code pane.

When not to use text: for arbitrary R expressions (formulas, function calls), use expr. For numeric values, use num — text would require the user to type a number as a string, bypassing the numeric validation a numericInput provides.

num — numeric input

What it is: a numericInput that accepts a number. Implemented in ptr_builtin_num_build_ui() / ptr_builtin_num_resolve_expr() R/paintr-builtins.R. The resolved expression is a numeric literal.

Uses in the examples:

  • geom_point(size = num) — point size. A continuous numeric value; only a number is valid here. Using text would allow "large" or "5px" — strings that ggplot2 would either reject or interpret unexpectedly. Using var would offer column names instead of numeric values.
  • dplyr::filter(displ > num) — filter threshold. The analyst sets a numeric cutoff; only a number makes sense. num ensures Shiny renders a numeric spinner with up/down arrows and validates the value as numeric before passing it to the pipeline.
  • geom_point(alpha = num) — transparency level. A continuous value in [0, 1]; same rationale.
  • num(shared = 'sz') — shared point size. Same widget type, driven by the shared section so it synchronizes across two geom_point() calls in different formulas.

When not to use num: for column selections, use var. For arbitrary expressions, use expr. For string labels, use text. num is exclusively for scalar numeric arguments where the analyst sets a threshold, size, or parameter value.

expr — code editor (validated against a denylist)

What it is: a code-editor widget (ACE editor via shinyAce) that accepts an arbitrary R expression. The expression is parsed and checked by the safety walker before being substituted into the formula. Implemented in ptr_builtin_expr_build_ui() / ptr_builtin_expr_resolve_expr() R/paintr-builtins.R.

Why expr exists: some ggplot2 arguments accept complex expressions — facet_wrap(~var + var2), scale_color_manual(values = c("red", "blue")), theme customizations. These cannot be expressed with var, text, or num because they are not a single column name, a string literal, or a scalar number. expr gives the analyst an escape hatch for arbitrary ggplot2 calls while still running the safety walker to prevent injection of dangerous code.

Uses in the examples:

  • facet_wrap(expr) in the formula tour example — the analyst types a faceting formula (e.g. ~Species or ~Species + cut). The argument domain is arbitrary R; only expr can accommodate this.

Safety model: the safety walker performs a recursive AST descent on the user-supplied expression, checking every node against the denylist (~151 entries) and structural rules. The denylist covers known-dangerous patterns but is not the primary safety mechanism; the walker's recursive inspection of all AST nodes (including lambda bodies and string literals) is the primary gate. See vignette("ggpaintr-safety") for the full model.

Empty-expression behavior: when expr is empty, the argument resolves as missing. If that leaves facet_wrap() with no arguments and facet_wrap is in the cleanup list, the call is dropped entirely. This is why the cleanup list matters — without it, facet_wrap() with no arguments would render as an error.

When not to use expr: use var for column selections (narrows the domain and prevents injection), text for string labels, num for numeric scalars. expr should only be used when the other three types are genuinely insufficient, because it widens the trust surface.

upload — file picker (returns a data frame)

What it is: a file-upload widget (fileInput) that accepts CSV, TSV, Excel, or Parquet files and resolves to a data frame. Implemented in ptr_builtin_upload_build_ui() / ptr_builtin_upload_resolve_data() R/paintr-builtins.R. It is a data source placeholder — registered via ptr_define_placeholder_source() — meaning its resolved value is a data frame that feeds downstream var column pickers.

Role in the data flow: when upload appears as the data argument of a ggplot() call or as the head of a pipeline (upload |> dplyr::filter(...) |> ggplot(...)), ptr_setup_pipelines() wires an observer that resolves the uploaded file to a data frame, stores it in the state, and triggers downstream var pickers to re-resolve their column choices against the new data. Column names are normalized via ptr_normalize_column_names() automatically on upload.

Why upload and not a static dataset reference: when the analyst's data is not in the R session at app-build time — it is in a user's local file system — upload provides the only path to get it into the plot. Using a static ggplot(my_data, ...) would hardcode the dataset; upload makes the data itself interactive.

Uses in the examples:

  • ggplot(upload, aes(var, var)) — the entire data source comes from the user's uploaded file. Every downstream var picker refreshes its choices when a file is uploaded. This pattern allows the same app to be used with any CSV the analyst provides.
  • upload |> dplyr::filter(...) |> ggplot(...) — the upload feeds a pipeline. The filter threshold is set by a num placeholder; the var pickers downstream resolve against the filtered columns. This is the most compositionally powerful use of upload.

When not to use upload: if the data is already in the R session (e.g. iris, mtcars), use a bare symbol reference in the formula. upload is for analyst-supplied external data. For a curated set of built-in datasets, register a custom data-source placeholder (see below).

Custom placeholders

ggpaintr's placeholder registry is open to extension. Three registration functions cover the three roles a placeholder can play:

Registration functionRoleResolved value
ptr_define_placeholder_value(keyword, build_ui, resolve_expr, copy_defaults)Value — a scalar expression substituted into the callAny R expression (returned by resolve_expr(value, node, ...))
ptr_define_placeholder_consumer(keyword, build_ui, resolve_expr, resolve_cols, copy_defaults)Data consumer — a column or set of columns from upstream dataSymbol or expression referencing upstream columns
ptr_define_placeholder_source(keyword, build_ui, resolve_data, copy_defaults)Data source — a data frame that feeds downstream pickersA data.frame

The source placeholder is the most powerful extension point. The fixture app at tests/testthat/fixtures/vignette-apps/source-dataset/app.R shows a custom dataset placeholder that presents a dropdown of built-in R datasets and resolves to the selected data frame, replacing upload with a curated menu:

library(ggpaintr)

# Register a custom data-source placeholder named "dataset".
ptr_define_placeholder_source(
  keyword       = "dataset",
  build_ui      = function(node, label = NULL, ...) {
    selectInput(node$id, label = label %||% "Built-in dataset",
                choices = c("iris", "mtcars", "diamonds", "economics"))
  },
  resolve_data  = function(value, node, ...) {
    if (is.null(value) || !nzchar(value)) return(NULL)
    switch(value,
           iris      = iris,
           mtcars    = mtcars,
           diamonds  = ggplot2::diamonds,
           economics = ggplot2::economics,
           NULL)
  },
  copy_defaults = list(label = "Built-in dataset for {param}")
)

# Use the new keyword in a formula just like the built-in "upload".
ptr_app("ggplot(dataset, aes(var, var)) + geom_point()")

The build_ui hook receives the ptr_ph_data_source node (with its stable node$id) and returns any Shiny tag. The resolve_data hook converts the widget's value to a data frame. Downstream var pickers automatically re-resolve against the returned frame whenever the selection changes — the same mechanism as upload.
