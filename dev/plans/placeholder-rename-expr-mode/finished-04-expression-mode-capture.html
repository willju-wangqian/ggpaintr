PLAN-04 — Expression-mode capture at ptr_app / ptr_server

Owning ADR: dev/adr/0009-pp-placeholders-and-expression-input.html · Branch: adr9/04-expr-mode · Parallelism: PARALLEL with PLAN-01, 02, 03, 05 · Depends on: nothing · Date: 2026-05-20

What this plan does (self-contained)

Adds an alternate input mode to the public surface: ptr_app() and ptr_server() may now be called with an unquoted expression instead of a string. The captured expression is deparsed to a single string at the public boundary and fed into the existing ptr_translate() pipeline. The string-mode path is unchanged. After this plan, both calls below produce equivalent apps:

ptr_app("ggplot(mtcars, aes(x = var, y = var)) + geom_point()")       # string mode (existing)
ptr_app(ggplot(mtcars, aes(x = var, y = var)) + geom_point())          # expression mode (new)

The new expression-mode path uses the existing keyword vocabulary (var/num/text/expr/upload). The keyword rename to ppX is PLAN-08. The two changes are independent — the expression-mode dispatch is purely an entry-point capture, with no dependency on what the formula's tokens are called.

Context from ADR 0009 (only the bits this plan touches)

Per ADR 0009 §2 ("Two input modes — string and unquoted expression"):

  • String mode (existing): the argument is a character scalar. Routed through preprocess_native_pipe() (which substitutes |> for a sentinel before parse) and then rlang::parse_exprs(). This stays.
  • Expression mode (new): the argument is captured unevaluated. R's parser has already desugared |> at parse time, so the pipe surface is lost in expression mode (calls become nested function calls). Users who want |> preserved must stay in string mode or use %>%.

Both modes share the same downstream typed-AST pipeline (ptr_translate() step 4 onward).

Dispatch sketch

ptr_app <- function(formula, envir = parent.frame(), ...) {
  formula_str <- ptr_capture_formula(rlang::enexpr(formula), envir)
  parts <- ptr_app_components(formula_str, envir = envir, ...)
  shiny::shinyApp(ui = parts$ui, server = parts$server)
}

# NOTE (ADR 0009 cross-cutting): use rlang::enexpr() not substitute().
# enexpr() processes `!!` splicing automatically and is the rlang-idiom
# equivalent of substitute() for argument capture at the public boundary.
ptr_capture_formula <- function(formula_captured, envir) {
  # (a) caller passed a string literal -> enexpr returns the character scalar.
  if (rlang::is_string(formula_captured)) {
    return(formula_captured)
  }
  # (b) caller passed a symbol referencing a variable. Resolve once in envir.
  if (rlang::is_symbol(formula_captured)) {
    resolved <- tryCatch(rlang::eval_bare(formula_captured, envir),
                         error = function(e) NULL)
    if (rlang::is_string(resolved)) return(resolved)
    if (is.call(resolved) || rlang::is_symbol(resolved)) {
      formula_captured <- resolved
      # fall through to wrapper-unwrap + deparse
    } else {
      rlang::abort(paste0(
        "`formula` must be a single string or an unquoted ggplot expression; ",
        "the variable `", rlang::as_string(formula_captured),
        "` did not resolve to either."
      ))
    }
  }
  # (c) unwrap one layer of rlang::expr() / rlang::quo() / quote() / bquote()
  #     wrappers (per ADR 0009 §"Edge-case resolutions" - E1.a).
  if (rlang::is_call(formula_captured)) {
    head <- formula_captured[[1L]]
    head_name <- if (rlang::is_symbol(head)) {
      rlang::as_string(head)
    } else if (rlang::is_call(head) && length(head) == 3L &&
               rlang::as_string(head[[1L]]) %in% c("::", ":::")) {
      rlang::as_string(head[[3L]])
    } else ""
    if (head_name %in% c("expr", "quo", "quote", "bquote") &&
        length(formula_captured) >= 2L) {
      formula_captured <- formula_captured[[2L]]
    }
  }
  # (d) deparse the language to a single string via rlang::expr_text().
  if (is.call(formula_captured) || rlang::is_symbol(formula_captured)) {
    return(rlang::expr_text(formula_captured, width = 500L))
  }
  rlang::abort(paste0(
    "`formula` must be a single string or an unquoted ggplot expression; ",
    "got ", typeof(formula_captured), "."
  ))
}

Why rlang::enexpr() over substitute() (ADR 0009 cross-cutting + E1):

  • Drop-in replacement for substitute() in the common case (character / symbol / language argument).
  • Processes !! splicing: col <- quote(mpg); ptr_app(ggplot(aes(x = ppVar(!!col)))) works (the !!col splices to mpg). With substitute(), !!col would parse as !(!col) and break silently.
  • Idiomatic across the rlang ecosystem (dplyr, ggplot2, rlang itself), so users coming from those packages get expected behavior.

Wrapper unwrap (E1.a): if the captured expression's root is one of rlang::expr(...), rlang::quo(...), quote(...), bquote(...), unwrap exactly once. Users who pre-quote an expression and pass it via ptr_app(rlang::expr(ggplot(...))) get the natural behavior; deeper-nested wrapping is not unwrapped (rare and unambiguous user intent).

Why capture without evaluating: if R evaluates ggplot(aes(x = var, y = var)) before ptr_app sees it, the call fails (var is not a value in the caller's frame). Capturing via rlang::enexpr(formula) at the top of ptr_app gives the unevaluated AST instead, sidestepping evaluation entirely. The dispatch helper above branches on what the captured object is (string → use; symbol → resolve in envir; wrapped language → unwrap once; language → deparse) and produces a string for the existing translate pipeline.

String-mode is unchanged: when the user passes a character literal, enexpr() returns it verbatim (it never wraps strings in language objects). The dispatch's first branch rlang::is_string returns it directly.

Symbol-resolution covers the f <- "..."; ptr_app(f) case: enexpr returns the symbol f; the dispatch's second branch evaluates it in envir, gets back the string. (A single symbol cannot itself be the new "unquoted ggplot expression" surface — no calls, no aes, no + — so re-resolving is safe.)

Success Criteria

  1. ptr_app("ggplot(...)") continues to produce the identical app as today.
  2. ptr_app(ggplot(mtcars, aes(x = var, y = var)) + geom_point()) produces an app whose translated AST is equivalent to the string-mode call (verified by comparing ptr_translate()'s output classes and the per-layer call shape).
  3. ptr_server accepts both modes identically.
  4. ptr_app_components accepts both modes (the dispatch is centralized in ptr_capture_formula).
  5. ptr_app(f) where f <- "ggplot(...)" resolves the symbol in the caller's frame and behaves as if the string was inlined.
  6. ptr_app(NULL), ptr_app(42), ptr_app(c("a","b")) all abort with a clear message naming the accepted input shapes.
  7. The native pipe (|>) in string mode still round-trips through preprocess_native_pipe() (existing behaviour).
  8. The native pipe in expression mode is desugared by R's parser before capture; the deparsed string no longer contains |>. This is a documented limitation (mentioned in the public-facing docs of ptr_app).
  9. The authoritative gate reports 0/0/0 PASS N ≥ baseline + the new tests in test-expression-mode.R.

Constraints

  • No keyword changes. This plan uses the existing keyword vocabulary (var/num/text/expr/upload). The rename to ppX is PLAN-08 and is independent.
  • No translate-time changes. ptr_translate() is unchanged. The new dispatch terminates at the public-API boundary by converting language → string.
  • No registry changes. R/paintr-registry.R is not touched.
  • No node changes. R/paintr-nodes.R is not touched.
  • The captured environment for symbol-resolution defaults to parent.frame() via the existing envir arg. Do not invent a new env-capture mechanism.
  • Roxygen. Update ptr_app's and ptr_server's roxygen @param formula to document both modes; add an @examples block demonstrating the new surface.
