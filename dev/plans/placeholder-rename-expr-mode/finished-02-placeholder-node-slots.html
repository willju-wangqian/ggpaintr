PLAN-02 — Placeholder node slots (default, named_args)

Owning ADR: dev/adr/0009-pp-placeholders-and-expression-input.html · Branch: adr9/02-node-slots · Parallelism: PARALLEL with PLAN-01, 03, 04, 05 · Depends on: nothing · Date: 2026-05-20

What this plan does (self-contained)

Adds two additive slots — default and named_args — to the three placeholder node constructors in R/paintr-nodes.R. The slots default to NULL and list() respectively, so every existing call site continues to produce identical node structures. PLAN-06 (parser) populates the slots; PLAN-07 (UI) and PLAN-08 (built-ins) consume them.

This plan does not change any externally observable behaviour. It is data-shape groundwork: after merging, the node objects carry the new slots but the rest of the codebase still ignores them.

Context from ADR 0009 (only the bits this plan touches)

Per ADR 0009 §4 ("Registry-declared argument schema"), placeholder nodes need to carry two new pieces of validated data:

  • node$default — the canonical value returned by the placeholder definition's default_arg validator. NULL when the placeholder call had no positional argument or the registry entry had no default_arg declared.
  • node$named_args — a named list of canonical values, one entry per validated named argument declared in the registry entry's named_args. Empty list when none.

The existing node$shared slot (a single character string or NULL) is unchanged. Reserved name "shared" continues to live on its own slot, not under named_args.

Construction shape (target)

ptr_ph_value <- function(id = NA_character_, keyword, param = NA_character_,
                         expr, shared = NULL,
                         default = NULL, named_args = list()) {
  new_ptr_node(
    class = "ptr_ph_value",
    id = id, keyword = keyword, param = param, expr = expr,
    shared = shared,
    default = default, named_args = named_args
  )
}

Identical pattern for the consumer and source variants (each preserves its own existing extra slot — upstream and companion_id respectively).

Success Criteria

  1. The three constructors accept default and named_args parameters with the documented defaults.
  2. A node built with no default/named_args arguments has node$default == NULL and identical(node$named_args, list()).
  3. A node built with default = "mpg" exposes node$default == "mpg"; identical contract for arbitrary values.
  4. A node built with named_args = list(step = 5) exposes node$named_args$step == 5.
  5. The S3 class membership of each constructor's output is unchanged (is_ptr_ph_value(), etc., still return TRUE).
  6. Every existing caller of the three constructors continues to compile and run; no caller in R/paintr-*.R needs to pass the new arguments (the defaults cover the no-arg case).
  7. The authoritative gate reports 0/0/0 with at least the baseline PASS count plus the new tests.

Constraints

  • No call-site changes. Do not edit any caller of the three constructors. They keep working because the new parameters have defaults.
  • No registry changes. R/paintr-registry.R is not touched.
  • No parser changes. R/paintr-translate.R is not touched.
  • No validation. The constructors store whatever they are given; the registry/parser is responsible for shape-checking before the constructor is called.
  • S3 class strings unchanged. Do not rename the class strings ptr_ph_value, ptr_ph_data_consumer, ptr_ph_data_source; many downstream files dispatch on them.
  • Roxygen. Each constructor's roxygen block (or the file's header comment) gets a one-line note on the two new fields. Do not invent ceremonial multi-paragraph docs.
  • NOT_CRAN gate only. See proxy-trap list below.
  • Surface-not-absorb. BDD scenarios

Feature: placeholder node constructors carry default and named_args slots

  Scenario: default slot defaults to NULL
    Given a call to ptr_ph_value(keyword = "ppNum", expr = quote(ppNum()))
    Then the resulting node$default is NULL

  Scenario: default slot stores its argument verbatim
    Given a call to ptr_ph_value(keyword = "ppNum", expr = quote(ppNum(5)), default = 5)
    Then the resulting node$default equals 5

  Scenario: named_args slot defaults to empty list
    Given a call to ptr_ph_value(keyword = "ppNum", expr = quote(ppNum()))
    Then identical(node$named_args, list()) is TRUE

  Scenario: named_args slot stores a named list
    Given a call to ptr_ph_value(keyword = "ppRange", expr = quote(ppRange(c(0,1), step = 0.5)),
                                  named_args = list(step = 0.5))
    Then node$named_args$step equals 0.5

  Scenario: existing fields and class membership unchanged
    Given a node built without the new parameters
    Then is_ptr_ph_value(node) is TRUE
    And node$keyword, node$param, node$expr, node$shared are populated identically to the baseline

  Scenario: ptr_ph_data_consumer keeps its upstream slot
    Given a call to ptr_ph_data_consumer(keyword = "ppVar", expr = quote(ppVar()), upstream = some_node)
    Then node$upstream is unchanged
    And node$default is NULL by default

  Scenario: ptr_ph_data_source keeps its companion_id slot
    Given a call to ptr_ph_data_source(keyword = "ppUpload", expr = quote(ppUpload()), companion_id = "x_name")
    Then node$companion_id is unchanged
    And node$default is NULL by default

Plus: devtools::check(document = FALSE, manual = FALSE, args = c("--as-cran","--no-manual")) → 0 errors / 0 warnings.

Proxy traps:

  • Browser e2e tests rely on placeholder nodes being constructable with the documented shape. A passing test_dir with NOT_CRAN unset would skip the e2e file and miss regressions there — run with NOT_CRAN=true.
  • devtools::check --as-cran e2e SKIP is expected and correct; ERROR there would indicate a deeper regression and is never the gate.

Scope guard

In scope

  • Adding two parameters to each of the three placeholder node constructors.
  • Tests covering the new slots' presence, defaults, and pass-through behaviour.

Out of scope

  • Anything outside R/paintr-nodes.R and the new test file.
  • Validating shape of default or named_args arguments at construction (validation lives in the registry/parser layer).
  • Updating S3 dispatch methods (render_walk.ptr_ph_*, etc.) — they are tolerant of extra list fields and ignore the new ones.
  • Anything touching the parser or registry — PLAN-03 / PLAN-06.
