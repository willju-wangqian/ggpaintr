Phasing

Phase 1 (parallel) — plans 01, 02, 03, 04, 05. All branch off vignette-review HEAD. None changes externally observable behaviour. Suite stays green throughout. Merge order is free.

Phase 2 (sequential) — plan 06 lands after 01+02+03 are merged; plan 07 lands after 02+06 are merged. Plans 04 and 05 can still be in flight in parallel.

Phase 3 (atomic) — plan 08 lands last and flips every formula in the codebase from var/num/text/expr/upload to ppVar/ppNum/ppText/ppExpr/ppUpload in a single merge. Vignette, test, fixture, README, and built-in changes are bundled because the authoritative gate (NOT_CRAN=true devtools::test(), 0/0/0) would otherwise go red between merges.

Authoritative gate (shared across every plan)

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected: FAIL 0 / ERROR 0 / SKIP 0 / PASS N. devtools::test() is equivalent (sets NOT_CRAN=true itself).

Proxy traps — never accept the suite as green from any of these:

  • Rscript -e 'testthat::test_dir()/test_file()' without NOT_CRAN: skip_on_cran() fires and every shinytest2 browser test silently SKIPs.
  • devtools::check(... --as-cran): the e2e browser file cleanly SKIPs via the source-root guard in boot_vignette_app(); an ERROR there is a harness defect, never the product gate.
  • Only NOT_CRAN=true / devtools::test() actually runs the browser tests.

Per-plan worktree handling

Each plan owns its own worktree branch (column 6 above). Worktrees are pre-created off vignette-review HEAD (after the plans-and-ADR-fix commit). A fresh Claude session launched in a worktree should:

  1. Read its own plan file at dev/plans/placeholder-rename-expr-mode/<NN-…>.html (self-contained spec).
  2. Cross-reference ADR 0009 (dev/adr/0009-pp-placeholders-and-expression-input.html) only when a plan claim is ambiguous and the original decision needs verification.
  3. Run the authoritative gate before claiming done.
  4. For SEQUENTIAL plans (06, 07, 08): rebase onto the post-merge base before starting if upstream dependencies have already landed.

The user merges plans one at a time and independently verifies each merge before launching the next dependent plan.
