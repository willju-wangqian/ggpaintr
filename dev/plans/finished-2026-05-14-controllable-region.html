Plan: generalize disable/greyout into a controllable-region abstraction

Date: 2026-05-14 (visual-cue paragraph updated 2026-05-15) · Branch hint: feat/controllable-region · Audit reference: dev/audit/2026-05-14-compatibility.html §3.3

2026-05-15 update — visual cue. The "disabled" affordance no longer uses opacity: 0.5 on .ptr-stage-disabled. Opacity on an ancestor of a picker creates a stacking context that clips the picker's .dropdown-menu behind later stage blocks. The cue is now the universal combo: strikethrough on the verb code label, muted text colour on labels, dashed light-grey .ptr-stage-fields border, faint background tint. Selectors live in ptr_layer_assets() (R/paintr-build-ui.R). Custom L3 widgets dim "for free" because the CSS hangs off .ptr-stage-disabled .ptr-stage-head/.ptr-stage-fields/label — not off any widget-internal class. This plan inherits that contract: the new region abstraction must keep emitting exactly .ptr-stage + .ptr-stage-head + .ptr-stage-fields, and the toggled class stays .ptr-stage-disabled.

1. Problem statement

The "checkbox above a region greys out the region and removes its contribution at eval time" affordance is implemented three times in source and only works for one node kind:

  • UI duplication. The ptr-stage / ptr-stage-head / ptr-stage-fields wrapper plus its checkboxInput() is built inline in build_pipeline_stage_ui() (R/paintr-build-ui.R:185–250), wrap_shared_widgets_with_stage_blocks() (R/paintr-build-ui.R:429–472), and ptr_shared_ui() (R/paintr-shared-ui.R:170–296). Three copies of the same DOM shape and the same ns_fn(paste0(sid, "_stage_block")) id pattern.
  • Runtime duplication. Two parallel observer setups: ptr_setup_stage_enabled() (per-pipeline-stage checkbox) and ptr_setup_shared_stage_enabled() (shared-orphan checkbox), both writing into the same state$stage_enabled() dict.
  • Closed to other node kinds. disable_walk.ptr_call collapses a disabled call to its data arg; disable_walk.default is a pass-through. There is no way to mark a ptr_ph_value, a ptr_ph_data_consumer, or a ptr_layer sub-region as "disabled when its checkbox is off" without forking this code.

2. Goal

Introduce a controllable-region abstraction that owns the UI shape (head + fields wrapper, checkbox, dom id convention) and a continuation helper for the bare-fields case. The three current UI emitters become one-line callers.

Scope reduction (2026-05-15). The original plan also unified the observer wiring and the tree-rewrite predicate. After re-examining the three observer sites we found they are not symmetric: (a) per-module pipeline stage = local-checkbox → local state$stage_enabled, (b) cross-module shared stage = local-checkbox → shared reactive + shared-panel DOM class, (c) module-side shared-orphan mirror = external reactive → local state$stage_enabled (no checkbox, no DOM). Forcing all three through one bind_controllable_region(predicate=…) obscures the semantic differences and risks subtle reactive-timing changes (an observeEvent(input[[id]], …) will not re-fire when an external reactive in predicate invalidates). The UI-shape unification stands; the observer/tree-rewrite unification is moved to §10 Follow-ups.

Future "grey out an arbitrary placeholder", "grey out a layer's controls subtab", or "grey out a custom L3 region" become new callers of controllable_region() — they still wire their own observer in whichever shape matches.

4. Constraints

  • Single-PR scope. No layout-shell refactor, no upload-reader registry, no node-taxonomy changes — those are separate items in the audit's recommendation list.
  • No public API surface change. All new symbols are internal (no @export, no ptr_ prefix unless absolutely required by S3 dispatch).
  • Backward-compatible DOM ids. Existing tests and any user-supplied CSS selectors target #<sid>_stage_block, .ptr-stage, .ptr-stage-head, .ptr-stage-fields, .ptr-stage-disabled. Keep these exact strings.
  • Backward-compatible state shape. state$stage_enabled() stays a named-by-stage-id list of logicals. The abstraction layers on top; it does not replace.
  • Backward-compatible custom message. Continue using session$sendCustomMessage("ptr_set_class", ...); do not introduce a second message type.
  • No stacking-context creators on popover ancestors. The disabled visual must use colour/border/background only. No opacity, filter, or transform on .ptr-stage, .ptr-stage-head, .ptr-stage-fields, or any wrapper the abstraction emits — picker .dropdown-menus are descendants and would otherwise be clipped behind sibling regions. (Layer-level .ptr-layer-disabled keeps opacity: 0.5; pointer-events: none because its semantic is "whole layer off" and it never wraps a live popover.)
  • No new runtime dependencies. shiny + bslib (already optional) only.
  • No behavior change in disable semantics. Pipeline stage k≥2 collapses out, stage k=1 falls through to call walker — preserve disable_walk.ptr_pipeline (R/paintr-disable.R:115–133) exactly. The "AND across shared keys cohabiting one orphan stage" semantic in ptr_setup_shared_stage_enabled() must stay.
  • S3 dispatch stays the runtime extension mechanism. Do not introduce a parallel registry for region kinds; reuse the node-class S3 protocol.
  • One walk method per node kind. Do not split disable_walk into multiple generics. If a region cannot fit, it is a separate plan.
  • Internal-only abstraction. Do not promise a "register your own region" public API in this PR. We earn that by using the abstraction internally first.

5.3 Sketch — the two helpers

# R/paintr-region.R

# Full region: head checkbox + fields wrapper. `ns_fn` is the *UI* ns
# (i.e. the same `ns_fn` the caller would otherwise have applied
# inline); it wraps both the checkbox inputId and the block DOM id, so
# the observer factory the caller wires up can read `input[[ns(id)]]`
# and target `#(_stage_block)` via `ptr_set_class`.
controllable_region <- function(region_id, head_label, body,
                                ns_fn = identity, default_on = TRUE) {
  shiny::div(
    class = "ptr-stage",
    id = ns_fn(paste0(region_id, "_stage_block")),
    shiny::div(
      class = "ptr-stage-head",
      shiny::checkboxInput(
        inputId = ns_fn(region_id),
        label = head_label, value = default_on, width = "auto"
      )
    ),
    shiny::div(class = "ptr-stage-fields", body)
  )
}

# Continuation: a bare `.ptr-stage-fields` block, no head. Used when a
# region's body is split across non-contiguous emit sites (today: the
# `seen` / `emitted` guard in the two build-ui call sites for shared
# placeholder interleave).
controllable_region_continuation <- function(body) {
  shiny::div(class = "ptr-stage-fields", body)
}

Observer wiring is intentionally not part of the abstraction in this PR. Callers continue to use ptr_setup_stage_enabled, ptr_setup_shared_stage_enabled, and the in-line observer in ptr_shared_server; the helpers above just guarantee they all target the same DOM id and class. The region_id the caller passes in is the same string those observers already use, so nothing on the runtime side changes.

5.4 Migration of the three sites

  1. build_pipeline_stage_ui (R/paintr-build-ui.R): the inline shiny::div(class = "ptr-stage", id = ns_fn(paste0(sid, "_stage_block")), …) block becomes controllable_region(sid, head_label, fields, ns_fn = ns_fn). The seen branch becomes controllable_region_continuation(fields). Net diff: ~25 lines deleted.
  2. wrap_shared_widgets_with_stage_blocks (R/paintr-build-ui.R): same replacement, with the orphan-stage's verb-derived label and the same module ns_fn. The emitted branch becomes controllable_region_continuation(w).
  3. ptr_shared_ui (R/paintr-shared-ui.R): the inline shared-panel block becomes controllable_region(shared_stage_input_id(k), head_label, w, ns_fn = identity). (Shared panel ids are not ns'd; passing ns_fn = identity makes that explicit.)

5.5 What stays unchanged

  • ptr_setup_stage_enabled, ptr_setup_shared_stage_enabled, and the shared-panel CSS-class observer inside ptr_shared_server — names, bodies, and reactive shapes preserved.
  • disable_walk.ptr_call, disable_walk.ptr_pipeline, is_stage_disabled — untouched.
  • CSS classes, custom message name, DOM id format, state$stage_enabled() shape.

6. Step-by-step build sequence

  1. Land R/paintr-region.R with controllable_region() + controllable_region_continuation() + roxygen, no callers yet. Tests in test-controllable-region.R render each helper, snapshot the resulting tag tree, and assert: (a) the outer div has class ptr-stage and id <ns>(<id>_stage_block), (b) the checkbox inputId is <ns>(<id>), (c) the continuation helper emits exactly one .ptr-stage-fields div with no head.
  2. Migrate build_pipeline_stage_ui (largest, two emit branches). Run full devtools::test(); spot-check stage-disable manually with the shared-panel app from the recent bug report.
  3. Migrate wrap_shared_widgets_with_stage_blocks. Run tests.
  4. Migrate ptr_shared_ui. Run tests; manual check that the picker dropdown inside a disabled shared stage still stacks above sibling stages (SC7 regression guard).
  5. Add the third region-kind test in test-controllable-region.R: call controllable_region() with a fabricated region_id not used by any built-in path, assert the tag tree matches the contract. (No shipped UI; this only proves the helper is generic.)
  6. Write dev/adr/0003-controllable-region.html: status, context, decision (UI-only unification this PR; observers deferred — and why), consequences. Cross-link from the audit recommendation.

7. Risks and mitigations

RiskMitigation
The two emit branches (full region vs. continuation) in build_pipeline_stage_ui / wrap_shared_widgets_with_stage_blocks drift apart — a future change touches one helper but not the other, and the continuation div ends up missing a class or id that the full one carries.Test in test-controllable-region.R asserts that controllable_region_continuation(body)$attribs$class equals "ptr-stage-fields" exactly, and that the full helper's inner fields div has the same class. Class drift becomes a test failure.
DOM id collisions if a future region kind reuses a region_id that overlaps a stage_id.Document that region_id shares the state$stage_enabled() namespace; future callers must pick non-colliding ids (current callers already do).
The CSS class flip uses add = !val via custom message; if a region's body is rebuilt mid-session, the class state can drift.Out of scope for this PR — same risk exists today. Note in the ADR; if it bites, fix in a follow-up by re-asserting class on rebuild.
A reviewer may want the abstraction to also cover layer-level checkboxes (ptr-layer-content / ptr-layer-disabled).Explicitly out of scope (see Constraints). Layer checkboxes have a different semantic (default-active resolution, layer-name dedup) — fold in a later PR if the abstraction proves itself.

  • Public extension API for "register your own region kind."
  • Per-placeholder greyout as a user-facing feature (the third test only exercises the abstraction; it doesn't ship UI).
  • Replacing the ptr_set_class custom message with a different mechanism.
  • Touching paintr-app.R's shell layout.

9. Done definition

SC1, SC3–SC8 green (SC2 was dropped — see §5.1); ADR merged; recommendation #1 in dev/audit/2026-05-14-compatibility.html marked partially closed (UI half landed; observer half tracked in §10). 10. Follow-ups (deferred from this PR)

  1. Observer/state unification. Decide whether the three observer shapes can collapse into a small set of named binders (e.g. bind_region_to_checkbox and bind_region_to_reactive) without obscuring the reactive-timing differences. Open question: does the module-side shared-orphan mirror need to live in this set at all, or is it better named as mirror_shared_region_state? Decide after the UI helper has been in use for a release.
  2. region_is_disabled predicate. Introduce only if a second caller of the disable check appears (today there is exactly one — is_stage_disabled inside disable_walk). Premature otherwise.
  3. Layer-level checkbox unification. build_ui_for.ptr_layer's active_input_id path uses a different semantic (default-active resolution, layer-name dedup, opacity: 0.5; pointer-events: none). Fold in only if it passes the same "is this really the same affordance?" test.
  4. Public "register your own region kind" API. Only after at least one in-repo caller demonstrates value beyond the three current ones.
