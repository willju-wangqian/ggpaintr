open API 1. Redundancy: ptr_extract_plot() / _error() vs. state$runtime()$plot / $error

Surfaced by: use-cases vignette L3 drafting (commit c7da43d). The plan framed L3 as "extract accessors inside renderPlotly()", but the accessors wrap shiny::isolate(), so used inside a reactive renderer they silently break reactivity. The canonical pattern (verified against gallery §5.1) is state$runtime()$plot directly. That leaves the extract accessors redundant for plot / error.

Verified bodies (R/paintr-server.R:1241-1251):

AccessorBodyPure wrapper?
ptr_extract_plot(state) shiny::isolate(state$runtime())$plot Yes — 1-liner.
ptr_extract_error(state) shiny::isolate(state$runtime())$error Yes — 1-liner.
ptr_extract_code(state) shiny::isolate(format_code_with_extras(state$runtime(), state$extras_exprs())) No — folds ptr_gg_extra() extras into the code text the same way the bundled output$ptr_code does. Re-implementing this on the host side duplicates format_code_with_extras().

Origin: Per the vignette plan's verification trail, testServer was the intended L3 use-case for these accessors ("use them to snapshot the runtime under shiny::testServer"). That justification is gone: the plan refuted "extract accessors render under testServer" and banned testServer from vignettes. The internal-only headless runtime also no longer needs the public accessors. Plot / error accessors are left without a primary use-case; the code accessor still earns its keep.

Options:

OptionActionCost / risk
A — Un-export plot / error, keep code Remove @export from ptr_extract_plot and ptr_extract_error. Keep ptr_extract_code. Use-cases L3 teaches state$runtime()$plot / $error directly. Cleanest. Honest about which accessor pulls its weight. Breaking for any external caller relying on the two un-exports — likely none in practice (the rewrite branch is unreleased). Vignette L3 already frames it correctly; minor edit only.
B — Drop shiny::isolate() from the accessors Rewrite ptr_extract_plot / _error to return state$runtime()$plot / $error without isolation. Then they take the reactive dep when used inside a render block (matching user expectation from the function names); callers who want a snapshot wrap externally with shiny::isolate(). Behavior change, not API change. Universal pattern, names earn their keep. Requires updating tests that rely on the isolate-by-default contract. Slightly more invasive than (A).
C — Status quo Leave the accessors exported with current isolation. Use-cases L3 frames them as "non-reactive snapshot accessors" (current state). Permanent mild API redundancy. Documentation has to keep explaining when each pattern applies. No code change.

Recommendation: Option A. It matches the api-audit memory's un-exports direction, the use-cases vignette only needs a small edit (drop the "ptr_extract_error(state) is the non-reactive counterpart" lines), and the one non-redundant accessor (ptr_extract_code) is kept for the case it uniquely covers.

Touchpoints if A is chosen:

  • R/paintr-server.R — strip @export from ptr_extract_plot / _error; regenerate NAMESPACE via devtools::document().
  • man/ptr_extract.Rd — split or trim so the un-exported names disappear from the rendered docs (or move them to @noRd).
  • vignettes/ggpaintr-use-cases.Rmd — drop the two sentences naming the un-exported accessors in the L3 section.
  • _pkgdown.yml — remove ptr_extract_plot / ptr_extract_error from the "Advanced embedders" section.
  • API-audit checklist (dev/plans/api-audit.md) — once merged, list the un-exports under "tiers / un-exports".

open scope 2. inst/llm/primer.md is from a previous API generation

Surfaced by: step 8 pkgdown rebuild (commit pending) of the vignette-rewrite cycle. Building the site succeeded, but ggpaintr-llm.html still references ptr_build_ids() in its topic-catalog table — and the underlying inst/llm/primer.md references many other API names that no longer exist.

Dead names surveyed in inst/llm/primer.md:

  • ptr_input_ui(), ptr_output_ui() — verified absent from NAMESPACE. Replaced by ptr_controls_ui() / ptr_outputs_ui().
  • ptr_server_state(formula, ...) — verified absent. Replaced by ptr_init_state(formula, ...).
  • ptr_setup_controls(input, output, ptr_state) — verified absent. Equivalent wiring is now folded into ptr_server() and ptr_module_server().
  • ptr_build_ids(...) — verified absent. No replacement; the "rename the five configurable top-level ids" feature no longer exists in the post-rewrite API.
  • "Four binders" framing — current API has three binders (ptr_register_plot / _error / _code). The fourth (ptr_register_draw) was removed.
  • Headless workflow (level3_headless topic) — per the vignette plan's locked decisions, headless is internal-only and must not appear in any public-facing primer or vignette.

What the primer is and why this matters: inst/llm/primer.md + inst/llm/topics/*.md are the documentation bundle ptr_llm_register() exposes to ellmer-backed coding assistants. When an LLM calls llm_primer() at runtime, it gets this text verbatim. A stale primer means assistants hallucinate the previous API generation, which is worse than no primer at all.

Out of scope for step 8: The vignette rewrite plan said "ggpaintr-llm: kept, unchanged. No changes." That decision scoped the vignette body; it did not survey the underlying inst/llm/ bundle. The bundle's staleness pre-dates this cycle.

Scope of the fix when prioritised:

  • Rewrite inst/llm/primer.md against the post-rewrite API surface — 3-level model (L1 turn-key, L2 embed, L3 custom rendering via state$runtime() + extract accessors).
  • Audit inst/llm/topics/*.md: delete or rewrite topic files for dead features (level3_headless, level2_custom_ids, anything referencing the four binders or ptr_input_ui / ptr_output_ui).
  • Refresh the topic catalog table in vignettes/ggpaintr-llm.Rmd to match the rewritten topics — the table appears to be hand-maintained, not generated.
  • Cross-link to vignette("ggpaintr-use-cases") from the primer rather than re-teaching the same material; same for safety.
  • Add a regression test that grep-scans inst/llm/primer.md + each topic file for any name that is not in NAMESPACE (catches future drift).

Recommendation: File as a tracked task before the API freeze. This is a substantial standalone rewrite, not a quick fix; folding it into the current vignette commit chain would balloon scope. The vignette rewrite plan locked "ggpaintr-llm unchanged"; that decision needs a follow-up "but the underlying primer bundle does need rewriting" decision once the user has bandwidth.

open API 3. Route title= through ui_text$shell$title$label

Surfaced by: api-audit follow-through (2026-05-14). The grill-with-docs audit on 2026-05-12 recommended trimming title= from ptr_app_bslib() and ptr_app_grid() so a single copy channel (ui_text$shell$title$label) owns the app title across every entry point. Other audit items have landed (commit 37e8768); this one is still pending.

Current state:

  • ptr_app_bslib(title = NULL, ...) — keeps a separate title= arg and consults ui_text$shell$title$label, with an @section Precedence documenting that title= wins. Two ways to set the same thing.
  • ptr_app_grid(title = "ggpaintr grid", ...) — separate arg, no ui_text fallback path documented.
  • ptr_app() — already only honours ui_text$shell$title$label; no title= arg. This is the target shape.

Proposed change: remove title= from both ptr_app_bslib() and ptr_app_grid(); route the bslib navbar brand and the grid page header through ui_text$shell$title$label unconditionally. Delete the @section Precedence block on ptr_app_bslib() (it describes a fallback path that no longer exists).

Recommendation: Take the trim. Implements the audit decision (ADR dev/adr/0002-public-api-tiering-and-trims.md) and removes one of the two remaining shell-chrome arg redundancies.

Touchpoints if approved:

  • R/paintr-app-bslib.R — drop the title= param, delete the @section Precedence block, read the brand from the resolved ui_text.
  • R/paintr-app.R (ptr_app_grid) — drop the title= param, read the page header from ui_text$shell$title$label.
  • Regenerate man/ptr_app_bslib.Rd + man/ptr_app_grid.Rd via devtools::document().
  • Search-and-update any examples / tests that pass title =.

open API 4. Route draw_all_label= through ui_text$shell$draw_all_button$label

Surfaced by: api-audit follow-through (2026-05-14). Same audit cycle as decision #3; same pattern — a chrome-string arg duplicates a channel ui_text already owns.

Current state: ptr_app_grid(draw_all_label = "Draw all", ...) keeps the "Draw all" button label as a positional-ish arg, while ptr_default_ui_text()$shell$draw_all_button$label already holds the same string as a copy default. No precedence is documented; whichever the implementation reads first wins, which is a bug magnet.

Proposed change: drop draw_all_label= from ptr_app_grid(); the button label flows entirely through ui_text$shell$draw_all_button$label, with the default coming from ptr_default_ui_text().

Recommendation: Take the trim. Bundles cleanly with decision #3 (one focused commit removes both chrome-string redundancies).

Touchpoints if approved:

  • R/paintr-app.R (ptr_app_grid + ptr_app_grid_components) — drop the param and any internal threading.
  • Regenerate man/ptr_app_grid.Rd.
  • Tests that pass draw_all_label =.

open API 5. Custom-layout wrappers need a documented output-id contract and an exported assets helper

Surfaced by: grilling session 2026-05-15 on the "ptr_app_bslib() as wrapper demo" reframing. To reposition ptr_app_bslib() as a demonstration that wrappers are easy to write with the public API, the function will be rewritten on top of ptr_controls_ui() + ptr_outputs_ui() + ptr_server() (decision: Q7 option A, single-card output layout). That rewrite uncovered two real gaps that limit any wrapper author who wants a layout finer-grained than ptr_outputs_ui()'s bundled arrangement.

Gap 1 — output ids are an implicit contract. ptr_register_plot() / _code() / _error() (and ptr_server() which calls them) write to namespaced output slots with fixed ids "ptr_plot" / "ptr_code" / "ptr_error" (verified R/paintr-server.R:1225,1242,1294). Nothing in the exported documentation tells a wrapper author "place a plotOutput(ns("ptr_plot")) in your layout and ptr_server() will fill it." The contract is only discoverable by reading ptr_outputs_panel() source.

Gap 2 — ptr_assets() is unexported. ptr_outputs_ui() wraps its contents in <div class="ptr-app"> and injects ptr_assets(css = ...) so the bundled stylesheet attaches. A wrapper author who hand-places the three outputs in their own layout (e.g. plot in one bslib::card, code in another) needs that wrapper and asset linkage themselves. Currently they would have to call ggpaintr:::ptr_assets() and hand-write <div class="ptr-app"> — both undocumented, both unexported. So splitting plot and code into separate cards on the public API is currently impossible without ggpaintr::: namespacing.

Gap 3 — single-formula var(shared = "...") coordination has no public wrapper path. When one formula declares the same shared key on multiple placeholders (the canonical ptr_app() shared widget case), ptr_app() uses the internal ptr_make_app_server helper to manually build the shared reactives and call ptr_server(..., auto_bind_shared = TRUE). ptr_module_server() — the public wrapper-friendly path — aborts with a clear pre-flight error unless the caller supplies a pre-built ptr_shared_state. ptr_shared_ui() / ptr_shared_server() exist for the multi-formula case but require explicit shared_ui = builders and at least one declared shared key — they do not act as a no-op for a single formula. Discovered after the bslib demo rewrite landed; the shipping ptr_app_bslib() documents the limitation.

Why this is bigger than the bslib demo: any L2 embedder who wants the bundled CSS but a custom layout hits gap 2; any L2 embedder who wants to place plot / code / error individually hits gap 1; any wrapper that wants ptr_app()-style single-formula shared coordination hits gap 3. All three silently push users toward ::: calls in production code.

Options:

OptionActionCost / risk
A — Export ptr_assets(), document the output-id contract Add @export to ptr_assets(). Add a "Output ids" section to ?ptr_server / ?ptr_register_plot documenting the three fixed ids and the namespace convention. No new functions. Cheapest. One un-export reversed, one documentation block added. Output-id contract becomes a public commitment — renaming the slots later becomes a breaking change.
B — Export ptr_assets(); add named output primitives Add ptr_plot_output(id = NULL) / ptr_code_output(id = NULL) / ptr_error_output(id = NULL) as thin exported wrappers around shiny::plotOutput() etc. with the correct ids baked in. Wrapper authors call those instead of memorising the id strings. Three new exports + an export. Larger surface; hides the id contract behind helpers (which is also a long-term commitment, just a friendlier one).
C — Status quo, weaken the wrapper-demo pitch Leave the API as-is. Customization vignette says "you can write a wrapper as long as you accept the bundled output layout from ptr_outputs_ui(); finer-grained layouts need internal helpers and are on the roadmap." No code change. The wrapper-demo pitch is honest but partial; future users wanting split layouts have no public path.

Recommendation: Defer to a follow-up cycle and ship the bslib demo with option C framing in the customization vignette. The current reframing work doesn't require the API change — the simpler single-card layout (Q7 option A) is genuinely a better demo. When the API is revisited (api-audit cycle), option A is the lowest-cost durable fix; option B is only worth it if a second motivating use case appears.

Touchpoints if A is later approved:

  • R/paintr-assets.R (or wherever ptr_assets lives) — add @export + roxygen docs.
  • R/paintr-server.R — extend @section on ptr_register_* / ptr_server documenting the three fixed output ids.
  • Regenerate NAMESPACE + man/.
  • vignettes/ggpaintr-customization.Rmd — revise the wrapper demo section to show a split-card layout as a second example.
