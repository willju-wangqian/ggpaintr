PLAN-04 — e2e selection round-trip via input-layer injection

Source: ADR 0028 (Decision 8 tier 3; worked examples "becomes possible #1/#2", "rejected by design #2/#3") · Depends on: 03 (and transitively 01, 02) · Parallel group: G4 · Status: draft

Context

ADR 0028 Decision 8 tier 3: "e2e selection round-trip by injecting the event at the Shiny input layer: plotly's R-side event_data() reads input[["plotly_selected-<source>"]] (the .clientValue- prefix is the client→server id, stripped before the R read — verified against the installed plotly source), settable via app$set_inputs(..., allow_no_input_binding_ = TRUE) with a crafted key payload — asserting table fill, flag-mode redraw, deselect-clears, redraw-resets." Probe re-verified 2026-06-10 against the installed plotly: event_data() builds eventID <- paste(event, source, sep = "-") and reads session$rootScope()$input[[eventID]] — so for PLAN-03's explicit source = "ptr_e2e" the injected ids are plotly_selected-ptr_e2e / plotly_deselect-ptr_e2e, at the ROOT scope (no module prefix). This plan also pins worked example "rejected by design #3" (draw_trigger = sel silently never redraws — the click-counter contract, clicked() at R/paintr-server.R:1860, consumed at :1868) with a second, default-gated fixture, mirroring the discriminating-pair pattern of tests/testthat/test-runtime-gate.R.

Governing principle

Selection is injected at the Shiny input layer — plotly's own R-side contract — never simulated by mouse-drag or JS emit. The input id format is an internal plotly contract: re-verify it against the installed plotly source before committing and pin the verification with a comment; if it proves unstable across plotly versions, the ADR's sanctioned fallback is to downgrade this tier to driving the selection reactive directly — recorded, not hidden. And draw_trigger stays a click-counter (numeric scalar ≥ 1): the rejected ptr_draw_trigger(rx) adapter stays rejected; live mode (gate_draw = FALSE) is the one supported route to per-brush redraws. A "fix" that makes draw_trigger fire on a data-frame reactive to get this plan green would break the batch-on-click promise everywhere — an obvious ADR violation.

Success Criteria

  • New round-trip tests appended to tests/testthat/test-plotly-e2e.R against PLAN-03's adr28-plotly-linked fixture, injecting a crafted selection payload (points carrying key values, shaped per plotly's parsed-event contract — shape verified against the installed plotly source at implementation time, pinned with a comment citing plotly:::event_data's eventID <- paste(event, source, sep = "-") line) via app$set_inputs(`plotly_selected-ptr_e2e` = <payload>, allow_no_input_binding_ = TRUE, wait_ = FALSE), asserting all four ADR observables:
    • table fill — after injecting keys c(2, 5), the rows-mode table's HTML gains exactly 2 data rows (zero before);
    • flag-mode redraw — instance 2's plot HTML changes from its pre-injection snapshot (the poll_html() changed-detector pattern, tests/testthat/helper-vignette-apps.R:86), i.e. the brush drove a redraw of the selection-fed formula with zero trigger wiring;
    • deselect clears — injecting plotly_deselect-ptr_e2e returns the table to zero data rows;
    • redraw resets — after re-selecting, changing an instance-1 ppVar picker (a new draw in live mode → new keys) returns the table to zero data rows: the old selection is NOT carried over.
  • A second fixture tests/testthat/fixtures/vignette-apps/adr28-plotly-trigger-misuse/app.R under the DEFAULT gate_draw = TRUE wiring state2 <- ptr_server(f2, "p2", draw_trigger = sel) (the selection reactive itself as trigger — ADR rejected-by-design #3), plus a test asserting: after instance 1 draws (Update click) and a selection payload is injected, instance 2's plot HTML is identical to its pre-injection snapshot — it silently never redraws, because clicked() (R/paintr-server.R:1860) rejects a data-frame-carrying reactive. The live-mode round-trip test above is the positive control proving this no-change assertion has teeth (same injection, HTML does change), mirroring test-runtime-gate.R's discriminating pair.
  • Verified by Rscript -e 'devtools::test(filter="plotly-e2e", reporter="progress", stop_on_failure=FALSE)' → FAIL 0 / ERROR 0, tests RUN on the dev machine.

Definition of Done

Authoritative gate: see target repo's CLAUDE.md §"Authoritative gate" — Rscript -e 'devtools::test(reporter="progress", stop_on_failure=FALSE)' at FAIL 0 / ERROR 0 / SKIP 0. The proxy-trap list lives there and applies to every plan.

Plan-specific gate risks:

  • The injected input id (plotly_selected-ptr_e2e) and payload shape are an internal plotly contract — a green run after a silent downgrade to "drive the reactive directly" without recording the downgrade is a hidden proxy; the downgrade is sanctioned but MUST be reported.
  • Inner-loop test_dir without NOT_CRAN SKIPs all of this plan's tests; only devtools::test() runs them.
  • The identical-HTML (no-redraw) assertion is only trustworthy alongside its positive control; if the control scenario is removed or skipped, the misuse test is vacuous — auditor must see both RUN.

This plan adds ≥ 5 new e2e tests at tests/testthat/test-plotly-e2e.R. The gate must show PASS count rise accordingly with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline.
