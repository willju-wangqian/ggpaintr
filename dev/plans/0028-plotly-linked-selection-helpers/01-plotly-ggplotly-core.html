PLAN-01 — R/paintr-plotly.R cores + ptr_ggplotly()

Source: ADR 0028 (Decisions 1, 2, 5, 6, 7; testing tier 1) · Depends on: none · Parallel group: G1 · Status: draft

Context

ADR 0028 Decision 1: "Scope: thin helper layer, nothing deeper. New file R/paintr-plotly.R; plotly stays in Suggests (already there — DESCRIPTION:56) with a call-time guard, mirroring the optional-bslib pattern of ptr_app_bslib()" (which guards via requireNamespace() + rlang::abort(), R/paintr-app-bslib.R:62-66; v1 may use rlang::check_installed() — rlang is in Imports). This plan ships the first of the two helpers, ptr_ggplotly(state, ..., source = NULL) (Decision 2), the key-channel ownership rules it enforces (Decision 5: plotly key aesthetic + .ptr_row), and the per-session session$userData store both helpers share (Decision 6). The selection helper itself is PLAN-02. The runnable reference for the boilerplate being internalized is dev/scripts/plotly_linked_select.R (the renderPlotly block: key tagging, ggplotly(source=), dragmode = "select", event_register("plotly_selected"/"plotly_deselect")).

Governing principle

The plotly interop is a thin, opt-in helper layer on top of the locked L3 model (ADR 0006): custom render extracts from state; core learns nothing about plotly or selection. That is why the signature is state-first — ptr_ggplotly() owns the req() pre-draw guard and derives the source id from the instance's namespace, so the two helpers coordinate without the user wiring anything — and why the return value is a plain plotly object, keeping the user's plotly verbs composable after it. Row keys are minted per draw, on the widget copy only: the key must physically travel inside the widget to the browser and back; the user's drawn data is never mutated. Do not add a renderer = "plotly" backend, do not touch core runtime files, do not make keys persistent across draws.

Success Criteria

  • ptr_ggplotly(state, ..., source = NULL) is a new exported function in R/paintr-plotly.R with roxygen @export and full-loop @examples (ADR Decision 7). Called inside plotly::renderPlotly() on a drawn instance, it returns an object inheriting class "plotly" (not a ggpaintr-wrapped class), verified by devtools::test(filter = "plotly-helpers").
  • Each helper piece is factored as a pure core + thin reactive shell (ADR Decision 8, a locked design constraint): the key-minting/widget-build logic is callable on a plain ggplot object with no Shiny session, and the tier-1 unit tests in tests/testthat/test-plotly-helpers.R exercise the cores directly (via ggpaintr:::), guarded by skip_if_not_installed("plotly").
  • Key minting: for a drawn plot over a 32-row data frame with an identity-stat point layer, the widget copy's data gains a .ptr_row column equal to seq_len(32) and the plot maps the plotly key aesthetic to it; the data frame held by the state's runtime is NOT mutated (no .ptr_row on state$runtime()$plot$data after the widget builds).
  • Key-channel ownership: a formula plot that already maps aes(key = gear) has its key mapping replaced by the minted .ptr_row, with a cli warning emitted on the first build for the instance and NOT on the second build (warn-once contract, ADR Decision 5); verified by expect_warning() + a second call wrapped in expect_no_warning() (or expect_silent()).
  • Collision overwrite: a drawn data frame that already contains a .ptr_row column (reserved name, e.g. riding in from a chained instance) is silently overwritten — no warning, no error, and the minted keys are seq_len(nrow) regardless of the stale values.
  • Stat-layer limitation (ADR Decision 3, accepted): for a plot whose only layer is geom_smooth() (non-identity stat), the built widget carries no minted keys on that layer's traces — brushing it can never produce keyed events.
  • Guard: with plotly unavailable (forced via testthat::local_mocked_bindings() on the installed-check), calling ptr_ggplotly() aborts with a message naming plotly, matching the optional-dependency pattern of ptr_app_bslib() (R/paintr-app-bslib.R:62-66) or rlang::check_installed().
  • Source id + store: with source = NULL the source string is derived from the instance's namespace (via the state handle's server_ns_fn field, R/paintr-server.R:297), so two instances ("p1", "p2") derive distinct strings; the derived source string and the per-draw snapshot of the drawn data are recorded in a per-session store hung on session$userData, keyed by instance id (ADR Decision 6) — never in a package-global environment.
  • Widget configuration matches the demo's boilerplate: the returned widget has dragmode = "select" set and the plotly_selected + plotly_deselect events registered; ... is forwarded to plotly::ggplotly() (e.g. tooltip = "all" reaches it).

Interface Contract

ptr_ggplotly(state, ..., source = NULL) (exported)

  • Signature: ptr_ggplotly(state, ..., source = NULL) — state: a ptr_state handle as returned by ptr_server() (required); ...: forwarded verbatim to plotly::ggplotly() (e.g. tooltip = "all"); source: character scalar or NULL (default — derive from the instance namespace). Returns a plotly htmlwidget object.
  • Input domain: inside the circle — a ptr_state whose state$runtime()$plot is a drawn ggplot (call from inside plotly::renderPlotly()); source a non-empty character scalar or NULL. Outside — a pre-draw state (runtime NULL / not ok): the internal req() guard raises Shiny's silent pre-draw condition, exactly like the demo's req(p); plotly not installed: hard abort (below).
  • Output spec: an object inheriting class "plotly" (plain — no ggpaintr subclass), with dragmode = "select" set, plotly_selected + plotly_deselect events registered, and every identity-stat point trace carrying the minted .ptr_row keys (1..nrow(drawn data)).
  • Error modes: plotly missing → rlang abort naming plotly (the ptr_app_bslib() guard pattern / rlang::check_installed()); non-ptr_state state → abort; pre-draw → silent req() condition (not an error).
  • Side effects: records the instance's source string and a per-draw snapshot of the drawn data in the per-session store on session$userData (keyed by instance id); emits the one-time key-override warning when the formula mapped its own aes(key=). Never mutates state$runtime()$plot$data.
  • Worked example: with state1 drawn over mtcars, w <- ptr_ggplotly(state1, tooltip = "all") → inherits(w, "plotly") is TRUE and plotly::layout(w, legend = list(orientation = "h")) succeeds, still inheriting "plotly" (ADR worked example "becomes possible #3").

plotly_mint_keys(plot, key_warned = FALSE) (internal pure core — the tier-1 unit-test seam)

  • Signature: plotly_mint_keys(plot, key_warned = FALSE) — plot: a built ggplot object; key_warned: logical scalar, whether this instance has already warned. Returns list(plot = <widget-ready copy>, key_overridden = <logical>).
  • Input domain: inside — any ggplot whose data is a data frame (including one already carrying a stale .ptr_row column, and one mapping its own aes(key=)). Outside — a non-ggplot: abort.
  • Output spec: the returned plot's data is a copy with .ptr_row = seq_len(nrow(data)) (silently overwriting any existing .ptr_row) and the plotly key aesthetic mapped to it; the INPUT plot's data is unchanged. key_overridden is TRUE iff the incoming plot mapped a foreign key aesthetic.
  • Error modes: none beyond the non-ggplot abort; collisions are silent by contract (ADR Decision 5).
  • Side effects: a cli warning mentioning key iff (foreign key mapped) AND (key_warned is FALSE) — the shell passes the stored per-instance flag to get warn-once.
  • Worked example: r <- plotly_mint_keys(ggplot(mtcars, aes(wt, mpg, key = gear)) + geom_point()) → warns once, r$key_overridden is TRUE, r$plot$data$.ptr_row is 1:32; re-run with key_warned = TRUE → no warning, same data.

plotly_installed() (internal — the guard's mock target)

  • Signature/spec: plotly_installed() → logical scalar; a trivial wrapper around the installed-check (requireNamespace("plotly", quietly = TRUE)) existing so the not-installed branch is interceptable via testthat::local_mocked_bindings(plotly_installed = function(...) FALSE) on the ggpaintr namespace — a namespaced rlang::check_installed() call alone is not mockable that way. The guard in ptr_ggplotly() consults it and aborts (message naming plotly) when FALSE.
  • Side effects: none (pure read of library state).
  • Worked example: with the mock active, ptr_ggplotly(state) → abort whose message contains "plotly"; unmocked on the dev machine, plotly_installed() → TRUE.

plotly_source_id(state) + plotly_store(session) (internal)

  • Signature/spec: plotly_source_id(state) → deterministic non-empty character scalar derived from the state's namespace function (server_ns_fn field, R/paintr-server.R:297); distinct for instances "p1" vs "p2". plotly_store(session) → an environment hung on session$userData (created on first call, reused after), whose per-instance entries hold the source string, the per-draw snapshot, and (from PLAN-02) the selection state.
  • Error modes / side effects: plotly_store() creates the store on first access (idempotent); nothing is ever written to a package-global environment (ADR Decision 6).
  • Worked example: two states built with ns = shiny::NS("p1") / NS("p2") → plotly_source_id() values are non-equal character scalars.
