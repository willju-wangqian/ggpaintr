Plan: L3 piece model — single shell, bare pieces, asset dedup

Date: 2026-05-16 · Supersedes-in-part: dev/plans/2026-05-15-css-unification.html §2 (assets-internal lock) · ADR: dev/adr/0004-l3-ui-piece-model.html · Builds on committed work 822ba35, 486984f, 8b04975

One-line summary. The L3 story is "own every piece of ggpaintr's public UI." Pieces stay bare; a single exported shell ptr_ui_page(..., page = shiny::fluidPage, css = NULL) owns the Bootstrap page, the one .ptr-app scope, and the (now htmlDependency-deduped) assets. Un-export the three ptr_register_* functions (dead post-rewrite surface). All renames are deferred to the API-audit pass. Naming contracts and the css-unification supersession are recorded in ADR 0004.

Reconciliation alert — read before implementing. Two earlier-session commits (822ba35) regressed an already-implemented, decision-backed design:
  • css-unification §2 locked "assets bundle stays internal; users never inject assets manually." Commit 822ba35 added export(ptr_ui_assets) and renamed the internal emitter ptr_ui_assets → core_assets_tags. The export is a real new need (L3 hand-composition for non-fluidPage roots) but it contradicts a locked decision and therefore requires explicit supersession via ADR 0004 — not a silent change.
  • ?ptr_css is a deliberate roxygen-only topic (R/paintr-css-doc.R, @name ptr_css, @keywords internal; man/ptr_css.Rd exists). An earlier grill conclusion to "fix the 7 dangling [ptr_css()] links" was wrong and is retracted. The links are valid; they stay. This is exactly the "don't fix the missing function back into the namespace" hazard ADR 0002 records.
Net: this plan keeps ptr_ui_assets() exported but records it in ADR 0004 as a conscious supersession; keeps core_assets_tags as the internal name (the css-unification plan's §3.1 prose is stale and gets a superseded banner, not a rewrite); and never touches the [ptr_css()] links.

1. Problem statement

The L3 surface shipped in 822ba35 redefined L3 as "one exported builder per UI piece" with bare pieces the user wraps by hand. Field use exposed three defects and one dead-surface finding:

  • Footgun A — forgotten Bootstrap page. Bare pieces placed under a bare shiny::tags$div() root never load Bootstrap; shinyWidgets::pickerInput() (layer selector) and the Bootstrap grid render unstyled. Fixed in docs (486984f) but the root cause is that the user must remember fluidPage().
  • Footgun B — hand-written .ptr-app + assets ritual. Non-obvious, easy to omit; also drove the collapsed-<details> confusion (8b04975).
  • Defect C — per-piece self-wrap is unsafe. ggpaintr.css:7 sets .ptr-app { min-height: 100vh; background: … } and there is no .ptr-app .ptr-app neutralization rule (the comment in ptr_controls_ui claiming otherwise is factually false). If every piece self-wrapped, the canonical L3 layout (controls / plot / error / code as vertical siblings) would stack N×100vh of empty background. A descendant selector cannot neutralize siblings.
  • Dead surface D — ptr_register_plot/error/code. Exported, but ptr_server() is their only caller and calls all three unconditionally; each reads state$runtime(), which is populated only by the internal ptr_setup_runtime() (called only by ptr_server()). Without ptr_server() they render nothing forever. The L2 "Splitting outputs further" subsection documents a use case that is impossible post-rewrite.

2.1 Locked naming contracts (honored by the audit; recorded in ADR 0004)

  • ptr_module_* is a deliberate naming exception. ptr_module_ui/ptr_module_server remain a sealed two-function bundle namespace, not renamed into the ptr_ui_* / ptr_* grammar. Documented as an intentional exception so it does not read as an oversight.
  • ptr_ui_outputs ≡ ptr_ui_plot_code(error = TRUE). Post-audit, ptr_ui_plot_code() defaults error = FALSE (pure L3 building block); ptr_ui_outputs() is the discoverable "ggpaintr's default output UI" alias with error nested. Different default ⇒ different default DOM ⇒ not a silent twin. Recorded so future-us does not "dedupe" the alias away.
  • Server family after audit: ptr_server (generic), ptr_shared_resolve (renamed from ptr_shared_server), ptr_module_server (bundle exception). UI family: ptr_ui_*.

3. Target architecture

3.1 The shell

# R/paintr-app.R (new exported function)

#' Page shell for hand-composed ggpaintr UIs
#'
#' Wraps composed L3 pieces in a Bootstrap page + the single `.ptr-app`
#' theme scope + the (deduped) asset bundle. The only thing an L3 user
#' must remember.
#'
#' @param ... UI children (pieces, layout, your own widgets).
#' @param page A Bootstrap-3 page builder whose `...` are tag children:
#'   `shiny::fluidPage` (default), `fixedPage`, `fillPage`,
#'   `bootstrapPage`, `basicPage`. NOT `navbarPage` (needs a positional
#'   `title` + `tabPanel` children) and NOT bslib/BS5 pages (the bundled
#'   CSS is Bootstrap-3-scoped — see [ptr_app_bslib()]). For those roots,
#'   compose by hand: see the decomposition example in
#'   `vignette("ggpaintr-use-cases")`.
#' @param css Optional character vector of extra stylesheet paths,
#'   linked after `ggpaintr.css`. See [ptr_css()].
#' @seealso [ptr_ui_plot()], [ptr_ui_controls()], [ptr_server()], [ptr_css()]
#' @export
ptr_ui_page <- function(..., page = shiny::fluidPage, css = NULL) {
  assertthat::assert_that(is.function(page))
  page(
    ptr_assets(css = css),                       # internal bundle (htmlDependency)
    shiny::tags$div(class = "ptr-app", ...)
  )
}

Exactly one .ptr-app, exactly one Bootstrap page, assets deduped to one <head> injection by htmltools. ptr_assets() stays internal (css-unification name); it is reached by users only through a css= portal (now including ptr_ui_page) or the exported ptr_ui_assets() escape hatch.

3.2 Pieces revert to bare

822ba35 shipped the pieces already bare (no self-wrap). No structural change is needed here — this plan only confirms bare as the locked model and removes the now-misleading roxygen note added in 486984f that tells users to hand-write fluidPage(ptr_ui_assets(), div(class="ptr-app", …)) (replace with "compose inside ptr_ui_page(); for non-fluidPage roots see the decomposition recipe").

3.3 Assets as htmlDependency

# R/paintr-build-ui.R
core_assets_dep <- function() {
  htmltools::htmlDependency(
    name = "ggpaintr", version = utils::packageVersion("ggpaintr"),
    src = c(file = system.file("www", package = "ggpaintr")),
    stylesheet = "ggpaintr.css",
    script = "ggpaintr-ui.js",   # extracted from the inline ptr_ui_js() string
    all_files = FALSE
  )
}
  • Move the inline ptr_ui_js() string + the ptr_set_class handler into a shipped inst/www/ggpaintr-ui.js (or keep inline via htmltools::htmlDependency(head=) if avoiding a new file is preferred — decide in Step 2).
  • ptr_assets(css) returns tagList(layer_dep_or_tags, core_assets_dep(), ptr_user_css_assets(css)). Layer assets (.ptr-stage-* inline CSS + ptr_set_class JS) also become a dependency so the ptr_app_bslib bypass still works (it includes only the layer dep, never core_assets_dep).
  • Delete the JS self-registration guards (__ptr_ui_assets_registered, __ptr_set_class_registered) — htmltools now guarantees single injection. Keep a one-line comment recording why the guards were removed.

5. Acceptance criteria

  1. ptr_ui_page() exported; emits exactly one .ptr-app + one Bootstrap container; page= swap + css= + non-function-rejection tests pass.
  2. A page nesting ≥2 shells injects exactly one ggpaintr.css + one script (htmlDependency dedup).
  3. ptr_register_plot/error/code absent from NAMESPACE; L2 "Splitting outputs further" gone; pkgdown has no missing-topic for them.
  4. [ptr_css()] links intact; ?ptr_css still renders; no roxygen cross-ref WARNING.
  5. Bundled ptr_app()/ptr_module_ui() <body> tree unchanged; <head> assets deduped. Full suite green; check 0E/0W.
