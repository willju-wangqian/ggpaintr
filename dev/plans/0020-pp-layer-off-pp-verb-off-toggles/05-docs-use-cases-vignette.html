PLAN-05 — use-cases vignette section for ppLayerOff / ppVerbOff

Source: ADR 0020 (Out-of-scope §4, Risk §3 — ppVerbOff data-arg position) · Depends on: 04 · Parallel group: G4 · Status: draft

Context

Plan 01 ships the keywords ppLayerOff + ppVerbOff as exported functions with roxygen man pages (man/ppLayerOff.Rd, man/ppVerbOff.Rd) including the ppVerbOff data-arg-position note. Plan 04 deletes the old checkbox_defaults = documentation and migrates existing vignette mentions to ppLayerOff. Neither plan adds a new narrative section explaining when and how a user reaches for these keywords.

This plan adds that narrative — one cohesive section in vignettes/ggpaintr-use-cases.Rmd covering: the formula-as-source-of-truth motivation, the layer-toggle use case (ppLayerOff), the stage-toggle use case (ppVerbOff), the runtime symmetry with naked-ggplot, the spec = precedence ordering, and the data-arg-position assumption for ppVerbOff (the headline pitfall ADR 0020 Risk §3 calls out). Examples in the vignette must round-trip cleanly through the standard build (devtools::document() + the use-cases vignette knit pass).

Governing principle

The vignette is the only place a user learns these keywords exist before they need them. ADR 0020's machinery is only as useful as the user's awareness of it. The use-cases vignette is the existing home for "here is how you express X in a formula" material, and the two new keywords belong in that frame, not in a separate doc. The vignette section MUST:

  • Lead with the motivation (formula = single source of truth for boot state) before any syntax, so the reader understands why they would reach for these over spec =.
  • Show the runtime-symmetry contract — a naked-R example that renders without ggpaintr loaded, then the same expression inside ptr_app.
  • Document the ppVerbOff data-arg-position assumption explicitly with a "what won't work" counter-example (a verb that takes its data argument elsewhere — flagged by ADR Risk §3).
  • Show the spec = precedence (a spec = entry overrides the hide =-derived default at flush time), so users know how round-trip restore composes with formula-side toggles.

What this plan does not do: re-explain the deleted checkbox_defaults = (Plan 04 already removed it; the vignette is post-deprecation). Nor does it add a new vignette file; new files mean extra _pkgdown.yml wiring and rebuild surface area for marginal benefit.

Definition of Done

Authoritative gate (copy verbatim from CLAUDE.md's "Authoritative gate" block):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N, with PASS rising by ≥ 4 over the post-Plan-04 baseline (the new test-pp-off-vignette.R).

Proxy traps — do not be fooled:

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does not make skip_on_cran() fire. Browser fixtures cleanly SKIP via the source-root guard. If a check run shows the e2e file ERRORing, that's a harness defect, not the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests. That alone is the gate.
  • Vignette-specific proxy: a passing gate does NOT imply the vignette knits. Run the project's vignette build (devtools::build_vignettes() or source("dev/build-pkgdown.R"); build_pkgdown_clean()) and inspect the rendered output before claiming SC7. The gate cannot catch a broken vignette.

Additionally, run the project's vignette build at least once and confirm vignettes/ggpaintr-use-cases.html renders cleanly with the new section visible.
