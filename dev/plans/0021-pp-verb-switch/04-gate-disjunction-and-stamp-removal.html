PLAN-04 — gate disjunction + delete stamp_default_stage_enabled_ids

Source: ADR 0021 · Depends on: 03 · Parallel group: G4 · Status: draft

Context

From the ADR's Decision section:

Stage-id gate. Extend is_data_chain_call in R/paintr-disable.R:20-22 from one predicate to a disjunction:

is_data_chain_call <- function(node) {
  is_ptr_call(node) &&
    (walk_has_placeholder(node) || isTRUE(node$has_user_control))
}

Single source of truth for "should this stage have a UI checkbox?" Three triggers (placeholder presence, explicit has_user_control from ppVerbSwitch) collapse to one gate.

Delete stamp_default_stage_enabled_ids. The Plan-01 second-pass workaround (R/paintr-translate.R:131-153 + the call site at line 117) is no longer needed once the gate is extended: every stage that gets has_user_control = TRUE from ppVerbSwitch already gets a stage_id from assign_stage_ids via the extended gate. The lone caller at line 117 is removed in the same change.

Two coordinated edits: extend the gate predicate in R/paintr-disable.R, and delete the redundant second-pass walk + its call site in R/paintr-translate.R. Both happen together because deleting the workaround without extending the gate would regress: a ppVerbOff(verb, TRUE)-wrapped stage on this branch would lose its stage_id (until plan 06 removes ppVerbOff; we still need that stage_id during the intermediate state).

Governing principle

Single mental model for "should this stage have a UI checkbox?" The ADR's most important architectural commitment is collapsing three previously distinct triggers — (a) presence of a placeholder in the subtree, (b) a post-translate stamping pass that retroactively assigned stage_id to default_stage_enabled = FALSE nodes, (c) the proposed sibling-node shape — into a single predicate that is the only place in the codebase that decides "this stage is data-chain-eligible". Two triggers (placeholder OR has_user_control) instead of two passes (assign + retroactively-stamp). This plan MUST land the gate extension and the deletion in one atomic change. Extending the gate without deleting the second pass leaves a duplicate code path that future maintainers will mis-extend; deleting the second pass without extending the gate regresses ppVerbOff (still present on this branch) from having a UI checkbox boot-OFF.

A second invariant the plan must preserve: stage_id is a UI-routing key, not execution-shape. The three guards in is_stage_disabled at R/paintr-disable.R:149-154 (non-node → FALSE, NULL stage_id → FALSE, anything-but-literal-FALSE stage_enabled[[sid]] → FALSE) ensure that an "unwired" stage_id (stamped but with no bound reactive) is runtime-equivalent to no stage_id at all. This plan adds stage_id stamping for has_user_control = TRUE nodes; it MUST NOT change is_stage_disabled's guards, MUST NOT add a fourth guard that special-cases has_user_control, and MUST NOT route has_user_control through some new disable code path. The whole point of the corrected classification is that stage_id's only effect is to enable a UI-driven drop; has_user_control piggy-backs on the existing routing, it does not introduce a new one.

Success Criteria

  • SC-1. After translation of "ggplot(mtcars |> ppVerbSwitch(mutate(mpg = mpg + 100), TRUE))", the carrier ptr_call at the pipeline-stage position has a non-NULL stage_id string of the form "<layer_name>_<path>_stage_enabled". Verified by an assertion in tests/testthat/test-pp-verb-switch-gate.R.
  • SC-2. The same translation with switch_on = FALSE also yields a non-NULL stage_id on the carrier — i.e. the gate extension does not depend on switch_on's value. Verified by an assertion in the same file.
  • SC-3. A bare verb stage (no ppVerbSwitch wrap, no placeholder) does NOT get a stage_id — the gate's two disjuncts both return FALSE. Translation of "ggplot(mtcars |> mutate(mpg = mpg + 100))" yields a carrier ptr_call with is.null(stage_id) == TRUE. Verified by an assertion in the same file.
  • SC-4. A placeholder-containing stage (the existing trigger) still gets a stage_id. Translation of "ggplot(mtcars |> mutate(x = ppVar(col)))" yields a carrier ptr_call with a non-NULL stage_id. Verified by an assertion in the same file.
  • SC-5. The helper stamp_default_stage_enabled_ids is absent from R/paintr-translate.R after this plan merges. Verified by expect_false(exists("stamp_default_stage_enabled_ids", envir = asNamespace("ggpaintr"), inherits = FALSE)) in the same file.
  • SC-6. ppVerbOff(verb, hide = TRUE) still boots OFF on this branch — the gate extension covers has_user_control from ppVerbSwitch, but the existing ppVerbOff path stamps default_stage_enabled = FALSE AND was previously rescued by stamp_default_stage_enabled_ids; with that helper removed, ppVerbOff(verb, TRUE) must still yield a stage_id. This SC drives one of two acceptable implementations:
    • Option A (preferred, matches ADR): in unwrap_pp_verb_off_stage at R/paintr-translate.R:833-854, stamp has_user_control <- TRUE on the resulting ptr_call (in addition to the existing default_stage_enabled <- !isTRUE(hide)). The gate then picks it up via the same disjunction.
    • Option B (acceptable if A breaks unrelated tests): leave a narrowly-scoped variant of the second-pass helper that only fires on ppVerbOff-stamped default_stage_enabled = FALSE nodes, rename it to clarify its purpose, and delete it together with ppVerbOff in plan 06.
    Either way SC-6 is verified by a tree-side assertion that, after ptr_translate_formula("ggplot(mtcars |> ppVerbOff(mutate(mpg = mpg + 100), TRUE))") and the subsequent assign_stage_ids pass, the carrier ptr_call at the pipeline-stage position has both identical(node$default_stage_enabled, FALSE) AND a non-NULL character node$stage_id. The tree-side check is the load-bearing one: default_stage_enabled = FALSE is what downstream readers (snapshot, UI, server spec-apply) consume to boot the checkbox OFF. (We do NOT assert against ptr_runtime_input_spec() — its output schema at R/paintr-input-spec.R:6-9 is input_id, role, layer_name, keyword, param_key, source_id, shared; there is no default column.) Verified in the same test file.
  • SC-7. Structural-equality holds at the disable-walk layer: after assign_stage_ids stamps stage_id on the ppVerbSwitch-carrier, the tree still compares equal under ptr_tree_structural_equal to the bare-verb tree (the comparator excludes stage_id by plan 01). Verified by a regression assertion in the same file: tree_A = translate("ppVerbSwitch(mutate(x = 1), FALSE)"), tree_B = translate("mutate(x = 1)"), expect_true(ptr_tree_structural_equal(tree_A, tree_B)).
  • SC-8. The existing tests/testthat/test-pp-off-readers.R and tests/testthat/test-pp-off-e2e.R files (which assert ppVerbOff behaviour) PASS unchanged. (If SC-6 Option A is taken, the snapshot/UI behaviour is byte-identical to pre-plan; if Option B is taken, the carrier may differ but the tests assert at the snapshot/UI layer where the value is the same.)
  • SC-9. The DoD command returns FAIL 0 / ERROR 0 / SKIP 0 / PASS N with N rising by 7 new test_that blocks (SC-1..SC-7; SC-8 is regression coverage, SC-9 is the gate itself).

Definition of Done

Authoritative gate (copy verbatim from project CLAUDE.md "Authoritative gate" block):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N. Baseline N = plan-03's post-merge N; expected N after = baseline + 7.

Proxy traps — do not be fooled:

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire. The browser fixtures instead cleanly SKIP via the source-root guard. If a check run shows the e2e file ERRORing, that guard is a harness defect, not a product failure.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests. That alone is the gate.
  • Project-specific proxy: testing the gate via a single targeted testthat::test_file() can silently skip the e2e file that exercises ppVerbOff(hide=TRUE)-boots-off behaviour (SC-6 regression coverage); always run the full suite.

This plan adds 7 new test_that blocks at tests/testthat/test-pp-verb-switch-gate.R (SC-1..SC-7). The gate must show PASS count rise by exactly 7 with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline.
