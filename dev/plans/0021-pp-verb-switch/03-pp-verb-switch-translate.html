PLAN-03 — ppVerbSwitch translate-time validators + unwrap + wiring

Source: ADR 0021 · Depends on: 01, 02 · Parallel group: G3 · Status: draft

Context

Translate-time wiring for ppVerbSwitch in R/paintr-translate.R. From the ADR's Decision section:

A new helper unwrap_pp_verb_switch_stage mirrors unwrap_pp_verb_off_stage: validates that switch_on is a literal scalar non-NA logical and label is a literal scalar character or NULL (both via new helpers validate_pp_verb_switch_switch_on and validate_pp_verb_switch_label, aborting with class "ptr_translate_error"); translates the inner verb call via translate_node; stamps three fields on the resulting ptr_call: has_user_control <- TRUE, default_stage_enabled <- switch_on, stage_label <- label.

Used outside a pipeline-stage position → translate-time error via the existing ptr_assert_no_surviving_structural_wrappers at R/paintr-translate.R:161-191. Extend the wrapper-name list from c("ppLayerOff", "ppVerbOff") to c("ppLayerOff", "ppVerbOff", "ppVerbSwitch") and update the error-message branch to mention ppVerbSwitch's valid position.

The wiring site is the existing build_pipeline_from_lift branch at R/paintr-translate.R:234 that already dispatches on is_structural_keyword_call(st$call, "ppVerbOff"). This plan adds a sibling ppVerbSwitch branch — it does NOT replace the ppVerbOff branch (that removal is plan 06).

Governing principle

ADR 0012 commits the project to a canonical typed tree: every surface form of the same semantic content MUST translate to the same typed-tree node. ppVerbSwitch(.data, verb, switch_on = TRUE) at translate time MUST produce the same ptr_call structurally (per the comparator with the new exclusion list) as bare verb; ppVerbSwitch(.data, verb, switch_on = FALSE) MUST produce a ptr_call that compares equal to one produced by the pre-removal ppVerbOff(.data, verb, hide = TRUE) path. The mechanism is "translate the inner verb call via translate_node (so any placeholders inside it resolve normally), then stamp UI-metadata onto the resulting node" — never construct a new ptr_call by hand. The wrapper itself never appears in the tree (ADR 0012 §1: the tree is semantic, not syntactic — the wrapping syntax is metadata that the comparator ignores). Translate-time validation of switch_on / label as literals only is the same source-of-truth rule that applies to hide: variables and non-literal expressions are disallowed in the slot because the boot-state metadata is consumed at translate time and the formula is the only declaration site. An implementer who relaxes the literal requirement to "evaluate the expression in the caller frame" turns the formula into one of N possible source-of-truth declarations depending on enclosing scope and breaks every downstream snapshot/spec contract.

Success Criteria

  • SC-1. ptr_translate_formula("ggplot(mtcars |> ppVerbSwitch(mutate(mpg = mpg + 100), TRUE))") returns a typed tree where the carrier ptr_call at the corresponding pipeline-stage position has has_user_control == TRUE, default_stage_enabled == TRUE, and stage_label is NULL. Verified by direct tree-walk assertions in tests/testthat/test-pp-verb-switch-translate.R.
  • SC-2. The same translation with switch_on = FALSE yields has_user_control == TRUE, default_stage_enabled == FALSE, stage_label NULL. Verified by an assertion in the same file.
  • SC-3. ppVerbSwitch(verb, switch_on = TRUE, label = "Transform x") stamps stage_label == "Transform x" on the carrier ptr_call. Verified by an assertion in the same file.
  • SC-4. Structural-equality invariant: ptr_tree_structural_equal(tree_A, tree_B) returns TRUE when tree_A is the translation of "ggplot(mtcars |> ppVerbSwitch(mutate(mpg = mpg + 100), FALSE))" and tree_B is the translation of "ggplot(mtcars |> mutate(mpg = mpg + 100))". Verified by an assertion in the same file. (This is plan-03's contribution to closing ADR-0020's dropped Plan-03 SC7 — the comparator exclusion came from plan 01; the structural identity at translate time comes from this plan.)
  • SC-5. Literal-only switch_on: ptr_translate_formula("ggplot(mtcars |> ppVerbSwitch(mutate(x = 1), some_var))") aborts with class "ptr_translate_error" and a message containing the substring `switch_on` (backticks included). Verified by expect_error(..., class = "ptr_translate_error") + expect_match on the message in the same file.
  • SC-6. Literal-only label: ptr_translate_formula("ggplot(mtcars |> ppVerbSwitch(mutate(x = 1), TRUE, my_label_var))") aborts with class "ptr_translate_error" and a message containing the substring `label`. Verified by expect_error + expect_match in the same file.
  • SC-7. Non-call verb_expr: ptr_translate_formula("ggplot(mtcars |> ppVerbSwitch(42, TRUE))") aborts with class "ptr_translate_error" and a message containing verb call and the keyword name ppVerbSwitch. Verified by expect_error in the same file.
  • SC-8. Wrong-position guard: ptr_translate_formula("ggplot(mtcars, aes(x = ppVerbSwitch(mutate(x = 1), TRUE)))") aborts with class "ptr_translate_error" and a message that mentions ppVerbSwitch AND the valid-position guidance (data-argument pipeline or equivalent — the message MUST contain ppVerbSwitch by name). Verified by expect_error + expect_match in the same file.
  • SC-9. Composability with placeholders: ptr_translate_formula("ggplot(mtcars |> ppVerbSwitch(mutate(x = ppVar(col)), TRUE, \"Transform x\"))") succeeds, the carrier ptr_call has has_user_control == TRUE AND stage_label == "Transform x", AND the inner ptr_ph_value for ppVar(col) is present unchanged. Verified by tree-walk assertions in the same file.
  • SC-10. ppVerbOff translation continues to work unchanged on this branch (plan 06 has not merged). Verified by re-running tests/testthat/test-pp-off-translate.R with zero regressions.
  • SC-11. The DoD command returns FAIL 0 / ERROR 0 / SKIP 0 / PASS N with N rising by 9 new test_that blocks (SC-1..SC-9; SC-10 is regression coverage, SC-11 is the gate itself).

Constraints

  • Files this plan must NOT touch: R/paintr-builtins.R (plan 02 owns the keyword), R/paintr-nodes.R (plan 01 owns the slots), R/paintr-disable.R (plan 04), R/paintr-build-ui.R (plan 05), every vignette, every other test file, every fixture. Do NOT remove or modify unwrap_pp_verb_off_stage / validate_pp_off_hide / the existing ppVerbOff branches — they coexist on this branch.
  • Do NOT touch the stamp_default_stage_enabled_ids call at R/paintr-translate.R:117 or the function body at lines 130-152 — plan 04 owns that deletion. The ppVerbSwitch-stamped ptr_call's eligibility for stage_id assignment relies on plan 04's gate-extension; on this branch, ppVerbSwitch-with-no-placeholder will fall through plan-04's would-be path and may be picked up by stamp_default_stage_enabled_ids only when switch_on = FALSE (because that helper only stamps on isFALSE(default_stage_enabled)) — that intermediate behavior is intentional and the SC list explicitly covers only switch_on = FALSE for assertions about stage_id presence on this branch. The full gate-disjunction story is plan 04's.
  • Naming: the new helpers MUST be named unwrap_pp_verb_switch_stage, validate_pp_verb_switch_switch_on, validate_pp_verb_switch_label. Snake_case, internal (unexported). The ADR names them; do not abbreviate.
  • Error class: every translate-time abort from the new validators MUST be class "ptr_translate_error" via the existing abort_translate() helper at R/paintr-translate.R:754-756. Do not introduce a new error class.
  • Wrapper-list extension: in ptr_assert_no_surviving_structural_wrappers at R/paintr-translate.R:161-191, the keyword-name guard at the existing line that reads if (nm %in% c("ppLayerOff", "ppVerbOff")) MUST be extended to c("ppLayerOff", "ppVerbOff", "ppVerbSwitch"), and the if/else error-message dispatch MUST be extended to a third branch that names ppVerbSwitch. The two existing branches stay byte-for-byte.
  • SCOPE NARROWING FORBIDDEN. If implementing SC-4 reveals that the existing build_pipeline_from_lift path produces a carrier ptr_call with some additional slot not yet in the comparator-exclusion list, STOP and escalate — do not adapt the assertion to "ignore that field too", do not add ad-hoc setdiff() calls in tests. The exclusion list lives in ptr_tree_structural_equal (plan 01) and is the only place that classification can be edited.
  • Honor project conventions: cli not used for translate-time errors (the file uses rlang::abort via abort_translate), backticks around argument-name in messages, mirror the validate_pp_off_hide message shape verbatim where applicable so reviewers can diff-compare.

Definition of Done

Authoritative gate (copy verbatim from project CLAUDE.md "Authoritative gate" block):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N. Baseline N = plan-02's post-merge N; expected N after = baseline + 9.

Proxy traps — do not be fooled:

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard. If a check run shows the e2e file ERRORing, that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests. That alone is the gate.
  • Project-specific proxy: a test_file() harness can split the placeholder registry across package and namespace envs and report a stale view of ptr_registry_keywords() — use devtools::test() not test_file() for any registry assertion. (Memory feedback-harness-test_file-vs-devtools-test.)

This plan adds 9 new test_that blocks at tests/testthat/test-pp-verb-switch-translate.R (SC-1..SC-9). The gate must show PASS count rise by exactly 9 with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline.
