PLAN-05 — UI emission of stage_label

Source: ADR 0021 · Depends on: 03 · Parallel group: G4 · Status: draft

Context

From the ADR's Decision section:

UI emission. Thread stage_label through the same entries carrier in R/paintr-build-ui.R that already threads default_stage_enabled (L242 read, L334 entry creation, L451 second entry creation). build_pipeline_stage_ui (defined L190-250) at L234-237 chooses between user-label and auto-label:

head_label <- if (!is.null(entries[[i]]$stage_label)) {
  entries[[i]]$stage_label              # user-provided plain text
} else if (has_verb) {
  shiny::tags$code(paste0(verb, "()"))  # existing auto-label
} else NULL

Mirrored in wrap_shared_widgets_with_stage_blocks (defined L492-528) at L518-520.

This plan owns only the threading and the label choice in two UI builders. The stamping of stage_label on the typed-tree carrier is plan 03's. The carrier-to-entries-list packing is the existing logic in R/paintr-build-ui.R — this plan extends it by one additional packed field.

Governing principle

The UI emitter is a pure consumer of the typed tree's UI metadata — it never derives metadata from scratch and never overrides what the typed tree declared. stage_label on a carrier ptr_call is the user's source-of-truth declaration of the stage-block head label; the emitter's job is to splice that string into controllable_region(head_label = ...), falling back to the auto-derived verb() code form only when the user did not declare a label. The fall-back ordering is non-negotiable: user-declared first, auto-derived second, NULL third. An emitter that reverses the precedence — auto-label first with a user-label "override" applied afterward — invites side-channel state that breaks the source-of-truth contract. An emitter that "improves" the label by wrapping the user's plain-text string in shiny::tags$code(...) breaks the user's intent: plain text was chosen because the user wanted plain text. Pass it through. The per-placeholder " in mutate()" suffix (the existing label-suffix rule inside build_pipeline_stage_ui's build_ph closure at L196-199) is intentionally untouched by this plan: the ADR's "Out of scope" §4 explicitly defers the question of whether the stage-level stage_label should compose with or replace the placeholder-suffix, with a decision to override only the stage-block head label.

Success Criteria

  • SC-1. Per-layer stage-block head label uses user stage_label when set: launching ptr_app("ggplot(mtcars |> ppVerbSwitch(mutate(x = ppNum), TRUE, \"Transform x\"), aes(x = mpg, y = wt)) + geom_point()") renders a stage-block whose head label has visible text Transform x (plain text, not wrapped in <code>). Verified by an HTML-snapshot test in tests/testthat/test-pp-verb-switch-ui.R using shiny::testServer or a static-render assertion against build_pipeline_stage_ui()'s output.
  • SC-2. Fallback to auto-label when stage_label is NULL: launching the same formula with stage_label omitted (i.e. ppVerbSwitch(mutate(x = ppNum), TRUE)) renders a stage-block whose head label is <code>mutate()</code> (i.e. shiny::tags$code("mutate()")). Verified by an HTML-snapshot assertion in the same file.
  • SC-3. Bare-pipeline (no carrier) renders NULL head label: a stage block built from entries with no stage_id AND no verb fingerprint AND no stage_label renders head_label = NULL, matching pre-plan behavior. Verified by a unit-level assertion in the same file.
  • SC-4. Shared-stage variant honours stage_label: a formula that places a shared-bound placeholder inside a ppVerbSwitch(verb, TRUE, "Custom") wrapper emits the shared-stage head label as Custom (plain text). Verified by an HTML-snapshot assertion in the same file targeting wrap_shared_widgets_with_stage_blocks()'s output.
  • SC-5. Per-placeholder label-suffix rule unchanged: a placeholder rendered inside a ppVerbSwitch(mutate(x = ppNum), TRUE, "Custom") wrapper still shows its widget label with the auto-derived " in mutate()" suffix (or its existing pre-plan rendering, whichever applied). The stage-level label override does NOT bleed into per-placeholder labels. Verified by a regression assertion in the same file.
  • SC-6. ppVerbOff-stamped stage still renders without label override (the wrapper carries no stage_label; the emitter falls back to auto-label). Verified by re-running the existing tests/testthat/test-pp-off-readers.R and observing zero regressions in HTML structure assertions there.
  • SC-7. Entries-list invariant: every entry in the list passed to build_pipeline_stage_ui and wrap_shared_widgets_with_stage_blocks after this plan has a stage_label field (NULL or a length-1 character) — i.e. the carrier-to-entries packing in the two entry-creation sites (per the ADR, L334 + L451) reads $stage_label from the carrier ptr_call and packs it into the entry. Verified by an assertion that an entry built from a stamped carrier has identical(entry$stage_label, "X").
  • SC-8. The DoD command returns FAIL 0 / ERROR 0 / SKIP 0 / PASS N with N rising by 7 new test_that blocks (SC-1..SC-7).

If the count of such sites differs from "two" (e.g. there's a third site the ADR missed), STOP and escalate — do not invent a third packing site, and do not skip one.
  • The user-label shiny::tags$code() wrap is FORBIDDEN. Per the ADR: entries[[i]]$stage_label is "user-provided plain text" and goes through controllable_region(head_label = ...) unchanged.
  • SCOPE NARROWING FORBIDDEN. If implementing SC-5 reveals that the per-placeholder suffix needs to suppress when stage_label is set, STOP and escalate — that is exactly the ADR-named out-of-scope question, and the decision is to keep them independent for this plan. Do not narrow SC-5 to "tested only when stage_label is NULL", do not add a suppression branch, do not add a unit test that asserts suppression.
  • Honor project conventions: 2-space indent, %||% idiom (the file already uses it; preserve), shiny::tags$code() only on the auto-label branch.

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N. Baseline N = plan-03's post-merge N (this plan is in parallel group G4 with plan-04; both branch from the same orchestrator HEAD); expected N after = baseline + 7.

Proxy traps — do not be fooled:

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire. The browser fixtures instead cleanly SKIP via the source-root guard. If a check run shows the e2e file ERRORing, that guard is a harness defect, not a product failure.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests. That alone is the gate.
  • Project-specific proxy: app$get_values() in shinytest2 snapshots EVERY output and 500s on pre-draw shiny.silent.error states; use app$get_html("#id") / app$get_value(output="id") targeted assertions instead. (CLAUDE.md "Browser e2e gotchas".)

This plan adds 7 new test_that blocks at tests/testthat/test-pp-verb-switch-ui.R (SC-1..SC-7). The gate must show PASS count rise by exactly 7 with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline.
