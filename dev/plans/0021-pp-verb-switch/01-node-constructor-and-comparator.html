PLAN-01 — ptr_call node slots + comparator exclusions

Source: ADR 0021 · Depends on: none · Parallel group: G1 · Status: draft

Context

ADR 0021 introduces a new typed-tree carrier shape and corrects the structural-equality comparator. From the ADR's Decision section:

Node-constructor change. Add two new slots to ptr_call in R/paintr-nodes.R: has_user_control = FALSE and stage_label = NULL. The existing default_stage_enabled = TRUE slot is unchanged.

Comparator exclusion list. Extend ptr_tree_structural_equal at R/paintr-nodes.R:142-143 to include stage_id, has_user_control, and stage_label.

This plan owns those two edits and only those two edits — no callers, no readers, no UI emission. Downstream plans (02–06) will stamp has_user_control = TRUE and stage_label = "…" on the new carriers; without the slots existing first, those writes are silently absorbed into list(...) on a node that was not declared to carry them, and downstream readers cannot ask isTRUE(node$has_user_control) with confidence about presence vs. NULL.

Governing principle

ADR 0012 commits the project to a semantic tree, not a syntactic one: surface-form variants (|>, %>%, nested calls; ppVerbSwitch(verb, switch_on = FALSE) vs bare verb) MUST collapse to one canonical structural shape. ADR 0021 corrects a misclassification in ADR 0020: stage_id is a UI-routing key, not execution-shape — a stage with a stamped stage_id but no bound reactive behaves identically at runtime to a stage with no stage_id at all (verified via the three early-return guards in is_stage_disabled at R/paintr-disable.R:149-154: non-node → FALSE, NULL stage_id → FALSE, anything-but-literal-FALSE stage_enabled[[sid]] → FALSE). Therefore stage_id, plus the new sibling UI-metadata fields has_user_control and stage_label, all belong in the same comparator-exclusion category as default_active and default_stage_enabled: they are metadata the UI layer reads, not load-bearing pieces of the execution AST. An implementer who keeps any of these three names inside the comparator walk would be re-asserting the rejected ADR-0020 framing and re-breaking the verb-side structural-equality invariant; do not narrow this list to "just stage_id" or "just the new ones" — the three names move together. Source code, not intuition, governs the classification: if you are tempted to keep one of them in, re-read is_stage_disabled and confirm the three guards are unchanged by this plan.

Success Criteria

  • SC-1. ptr_call("foo", list()) (default args; fun = "foo", args = list(), expr = NULL) returns a ptr_call whose has_user_control slot is identical to FALSE and whose stage_label slot is identical to NULL. Verified by an assertion identical(ptr_call("foo", list())$has_user_control, FALSE) and is.null(ptr_call("foo", list())$stage_label) in tests/testthat/test-pp-verb-switch-node.R. Implementation note: the current ptr_call(fun, args, expr, default_stage_enabled = TRUE) signature (R/paintr-nodes.R:40) requires expr; implementer adds a default expr = NULL alongside the two new slots so the two-arg form in this SC is callable — this is the minimum signature change implied by the new SCs, NOT a license to add other defaults.
  • SC-2. ptr_call("foo", list(), has_user_control = TRUE, stage_label = "X")$has_user_control is TRUE and the same node's stage_label is "X". Verified by a constructor round-trip test in the same file.
  • SC-3. Two ptr_call nodes that differ only on stage_id, has_user_control, or stage_label (singly or in combination) compare equal under ptr_tree_structural_equal. Verified by three discrete assertions in the same file (one per excluded slot) using fresh ptr_call nodes built side by side.
  • SC-4. The existing exclusion-list semantics for op, expr, default_active, default_stage_enabled are preserved unchanged. Verified by re-running tests/testthat/test-pp-off-comparator.R and observing every test_that block PASSes with zero regressions in PASS count. The SC1 source-code lockdown block (lines 21-37) is re-anchored to the new 7-name literal as part of this plan (see Constraints carve-out); no other block in that file is edited.
  • SC-5. The DoD command in §"Definition of Done" returns FAIL 0 / ERROR 0 / SKIP 0 / PASS N with N rising by exactly 3 new test_that blocks (one for SC-1, one for SC-2, one for SC-3 written with three expect_true assertions inside it) from the baseline.

Constraints

  • Files this plan owns:
    • R/paintr-nodes.R — full ownership.
    • tests/testthat/test-pp-verb-switch-node.R (NEW) — full ownership.
    • tests/testthat/test-pp-off-comparator.R — scoped ownership: lines 21-37 only (the SC1 source-code lockdown test_that block). Every other test_that block in that file (SC2 through SC8) is OFF-LIMITS. The carve-out exists because that file was added by ADR-0020 PLAN-03 as a source-code lockdown guard over the comparator's exclusion list: it greps R/paintr-nodes.R for the literal 4-name c(...) argument. SC-3 of this plan changes that exact literal from 4 names to 7, which mechanically invalidates the lockdown regex. The lockdown must be re-aimed at the new 7-name literal in the same pass — it is the same change in two places, not separate work. Without this carve-out, plan 01 cannot satisfy SC-3 and DoD simultaneously; with it, the lockdown's purpose (catch accidental drift of the canonical exclusion list) is preserved, just re-anchored on the post-plan reality.
  • Files this plan must NOT touch: every other R/paintr-*.R; every other test file (including the non-SC1 blocks of test-pp-off-comparator.R); vignettes; fixtures; NAMESPACE; man/; DESCRIPTION.
  • No new is_* predicate. No reclassification of the existing slots. No roxygen2 on the constructor (it is internal — the file's existing convention is plain inline comments).
  • Comments are ALLOWED to update the comment block at R/paintr-nodes.R:32-39 (the default_stage_enabled note) to mention the new sibling fields, but the comment must not duplicate the ADR — it is a cross-reference, not a re-derivation.
  • SCOPE NARROWING FORBIDDEN. If implementing SC-3 reveals that some existing comparator caller (e.g. a snapshot equality check) relied on stage_id being part of the structural identity, STOP and escalate. Do not narrow SC-3 to "exclude only the two new fields", do not write a test that asserts the narrowed behaviour, do not justify the hedge inline. The governing principle's classification of stage_id as UI-metadata is not negotiable at the implementer layer.
  • The lockdown-regex update is the only permitted edit inside test-pp-off-comparator.R. The new regex MUST list all 7 excluded names in the same prefix order ("op", "expr", "default_active", "default_stage_enabled", "stage_id", "has_user_control", "stage_label") — it must not be loosened to "any list containing these names" because loosening removes the lockdown's purpose. A short inline comment above the updated expect_match calls should cite ADR-0021 PLAN-01 as the reason for the change.
  • Honor project conventions: tidyverse style, snake_case, 2-space indent, internal helpers unexported, rlang::abort for any abort (none expected in this plan), setdiff() already in use — keep that idiom.

Definition of Done

Authoritative gate (copy verbatim from project CLAUDE.md "Authoritative gate" block):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N. Baseline N before this plan: record at worktree creation; expected N after = baseline + 3.

Proxy traps — do not be fooled:

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate. Never claim the suite green, merge a branch, mark a step done, or clear context off the skipping/erroring harness.

This plan adds 3 new test_that blocks at tests/testthat/test-pp-verb-switch-node.R. The gate must show PASS count rise by exactly 3 with FAIL/ERROR still 0 and SKIP unchanged from the pre-plan baseline.
