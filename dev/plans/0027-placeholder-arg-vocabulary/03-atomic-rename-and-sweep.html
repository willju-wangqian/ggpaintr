PLAN-03 — atomic argument rename + default wiring + call-site sweep

Source: ADR 0027 (D2, D5, D4 default-wiring, D6 regression) · Depends on: 01, 02 · Parallel group: G2 · Status: draft

Context

ADR 0027 D2 renames the four public constructor arguments — positional_arg → parse_positional_arg, named_args → parse_named_args, validate_input → validate_session_input, runtime → embellish_eval — across ptr_define_placeholder_value/_consumer/_source and the internal ptr_define_core. Internal registry-record keys (default_arg, copy_defaults, the stored named_args/validate_input/runtime slots) stay; only the public argument names change. D5 mandates this be atomic: the signature change, the default wiring (embellish_eval defaults to embellish_identity() from PLAN-01), the removal of the ptr_arg_numeric_vector alias (PLAN-02 retained it), the DESCRIPTION version bump, and every in-repo call-site update land in one merge so the gate is green after it. D6 keeps the one-positional cap unchanged and pins the two rejection regressions.

Governing principle

The placeholder's parenthesized arguments are interpreted by one coherent family of slots, and the rename makes that family legible: two slots parse the formula's default args into the widget seed (parse_positional_arg, parse_named_args), one validates the live session value (validate_session_input), one defines the plain-R evaluation of the embellished expression (embellish_eval). The rename is a hard break taken deliberately (ADR D5): argument names cannot be soft-deprecated, so it must be atomic — a half-applied rename leaves the package unloadable or the suite red. Do NOT leave any old name reachable "for compatibility"; the ADR rejects that. Apply the rename to every caller in the same change, including the built-in registrations, or the package will not load.

Success Criteria

  • The three constructors accept parse_positional_arg, parse_named_args, validate_session_input (value/consumer only), embellish_eval; passing the old names (positional_arg= etc.) raises an "unused argument" error. Verified by devtools::test(filter="rewrite-registry|registry-slots|extract-placeholder-args").
  • embellish_eval defaults to embellish_identity() for value/consumer roles (a registered placeholder with no embellish_eval has the identity runtime in its registry entry).
  • ptr_arg_numeric_vector no longer exists (removed from source + NAMESPACE); every former caller uses ptr_arg_numeric(vector = TRUE).
  • Built-in placeholders still register (package loads; ppText/ppNum/ppExpr/ppVar resolve) after their R/paintr-builtins.R call sites move to the new names.
  • ADR 0027 worked example "becomes possible #2": a value placeholder registered with a custom embellish_eval = function(x, ...) as.character(x) evaluates its call to a string in plain R. ADR 0027 worked example "rejected by design #1": colvars(mpg, hp) (two positional args) still aborts at translate with "accepts at most one positional argument".
  • DESCRIPTION Version bumped from 0.10.0. Whole-suite gate green: FAIL 0 / ERROR 0 / SKIP 0.

Interface Contract

The changed entry points are the three public constructors. Only the four argument names change; positional order of the leading required args (keyword, build_ui, resolve_expr/resolve_data) and all internal registry-record keys (default_arg, named_args, validate_input, runtime, copy_defaults) are unchanged.

ptr_define_placeholder_value(...) and ptr_define_placeholder_consumer(...)

  • Signature (renamed args): ..., validate_session_input = NULL, parse_positional_arg = NULL, parse_named_args = list(), embellish_eval = NULL, ui_text_defaults = .... (_consumer's build_ui additionally takes cols/data — unchanged.) Returns the runtime callable (invisibly registers the entry).
  • Input domain: inside — the four renamed args supplied by their new names. Outside — any of the old names (positional_arg=, named_args=, validate_input=, runtime=).
  • Output spec: the registry entry is stored under keyword with entry$default_arg = the value passed to parse_positional_arg, entry$named_args = parse_named_args, entry$validate_input = validate_session_input, entry$runtime = embellish_eval (or, when embellish_eval = NULL, the identity wired from embellish_identity()).
  • Error modes: passing an old argument name raises R's "unused argument ( = …)" error (the params no longer exist). Other validation (keyword shadow, hook arity) is unchanged.
  • Side effects: registers/overwrites the keyword in .ptr_registry (same as before the rename).
  • Worked example: ptr_define_placeholder_value(keyword="t", build_ui=function(node,...) NULL, resolve_expr=function(value,node,...) value, parse_positional_arg = ptr_arg_numeric()) registers; ptr_registry_lookup("t")$default_arg(quote(5)) → 5. Calling the same with positional_arg = ptr_arg_numeric() → "unused argument" error. A registration with no embellish_eval → entry$runtime(5L) returns 5L.

ptr_define_placeholder_source(...)

  • Signature (renamed args): ..., resolve_data, resolve_expr = NULL, parse_positional_arg = NULL, parse_named_args = list(), embellish_eval = NULL, shortcut = FALSE, ui_text_defaults = .... Source has no validate_session_input (it had no validate_input — unchanged). The source-role embellish_eval default remains the abort closure ("only meaningful inside ptr_app()"), NOT identity.
  • Input domain / Error modes / Side effects: as for value/consumer; old names → "unused argument".
  • Worked example: a source registered with parse_positional_arg = ptr_arg_string() registers and its entry$default_arg(quote("x")) → "x".

DESCRIPTION

  • Change: Version: 0.10.0 → a higher version (minor bump, e.g. 0.11.0) reflecting the breaking rename. Observable: read.dcf("DESCRIPTION")[, "Version"] is no longer "0.10.0".

Definition of Done

Authoritative gate: CLAUDE.md §"Authoritative gate" — Rscript -e 'devtools::test(reporter="progress", stop_on_failure=FALSE)' → FAIL 0 / ERROR 0 / SKIP 0. This is a shared-runtime + codegen-adjacent change touching the registry constructors and built-ins; per .claude/rules/testing.md the blast radius is wide — run the FULL suite (including e2e), not a filtered subset, before declaring done.

Plan-specific gate risks:

  • The fixture app.R/reference.R files are exercised only by the full shinytest2 e2e suite (test-e2e-vignette-examples-shinytest2.R); a missed fixture call-site renames cleanly compiles but the browser boot errors. The e2e file must RUN (NOT_CRAN via devtools::test()), not SKIP — a clean all-skip would hide an un-swept fixture.
  • Registry-env split: custom-placeholder tests must run under devtools::test(), not raw test_dir (per CLAUDE.md proxy trap).

This plan changes existing tests (renames within them) and adds ≥ 2 regression tests (worked examples #2 positive, #1 negative). Net PASS count must not drop; FAIL/ERROR 0; SKIP unchanged (0 under the gate).
