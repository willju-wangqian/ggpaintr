PLAN-01 — partition_view: one origin for shared-key ownership

Source: ADR 0026 rank 11 (partition-view seam, promoted from ADR 0017 C1) · Depends on: none · Parallel group: G1 · Status: draft

Context

ggpaintr answers one yes/no question for every shared key and every UI surface: "does THIS surface render and bind THIS shared key?" Today that single question is answered by three hand-derived complement sets plus one UI boolean, kept consistent by hand. When they drift, a shared widget renders but never binds (or binds twice) — the shipped 2026-05-22 partition-binder bug (project memory project-shared-section-binder) was exactly this.

The load-bearing quantity is host_owned_keys: the binder ptr_bind_local_shared_consumers() binds setdiff(names(resolutions), host_owned_keys) (R/paintr-server.R:3376), so host_owned_keys is the complement — "keys this surface must NOT bind because something else owns them." It is derived three different ways, and the UI side carries a parallel boolean render_shared_section that must agree:

Surfacehost_owned_keys (the complement)Net: bindsrender_shared_sectionSource (verified 2026-05-30)

The ADR's "All related files" list cited shared-ui.R:388 (the file has since drifted to :416) and omitted the embed-instance derivation at app.R:1164; that third site was confirmed by re-grep for this plan (the re-grep the ADR boundary note flagged as required before locking the interface). render_shared_section = TRUE appears at exactly one site (app.R:169); every other call passes FALSE (:403 default, :760, :964) — so it is a pure function of surface role.

Governing principle

One question, one origin. A single internal, pure, non-reactive partition_view becomes the only place that maps a surface's role to its ownership answers. Every call site constructs the view from the partition facts it already holds and queries it; no call site re-encodes the complement or the boolean. A new shared-key surface gets a query, not a "compute it correctly" step to get wrong. The change is behavior-preserving by construction: each accessor returns exactly the literal the corresponding site passes today, which the unit tests pin by enumerating every role.

Interface Contract (frozen — item 10)

New internal file R/paintr-partition-view.R (unexported; matches the paintr-*.R source pattern; isolates the seam for unit testing). Three internal functions:

# Internal. Pure, non-reactive. The single origin for shared-key ownership.
# role:          one of "single", "panel_host", "instance".
# panel_keys:    shared keys owned by the standalone panel (>=2 formulas).
#                character(0) for "single". For "instance" this is the set the
#                coordinator handed in (names(dots$shared)).
# consumer_keys: all shared consumer/value keys reachable from this surface's
#                tree(s). Consulted ONLY for role "panel_host".
new_partition_view <- function(role,
                              panel_keys = character(),
                              consumer_keys = character()) -> <ptr_partition_view>

# The complement set the binder must NOT bind (today's `host_owned_keys`).
#   single     -> character(0)
#   panel_host -> setdiff(consumer_keys, panel_keys)
#   instance   -> panel_keys
partition_host_owned_keys(view) -> character vector

# Whether this surface renders its inline formula-local shared SECTION
# (today's `render_shared_section`). TRUE iff role == "single".
partition_renders_section(view) -> logical(1)

Validation: role is checked with match.arg(role, c("single","panel_host","instance")) (rejects any other value); panel_keys / consumer_keys are coerced/asserted to be character vectors via assertthat::assert_that(is.character(...)). The class tag ptr_partition_view exists so accessors can assert_that(inherits(view, "ptr_partition_view")) and fail fast on a raw list.

Success Criteria

  • new_partition_view("single") yields a view for which partition_host_owned_keys(view) is character(0) and partition_renders_section(view) is TRUE. Verified by devtools::test(filter = "partition-view") case "single instance owns all keys and renders its section".
  • new_partition_view("panel_host", panel_keys = c("b"), consumer_keys = c("a","b")) yields partition_host_owned_keys(view) == "a" (the formula-local complement) and partition_renders_section(view) == FALSE. The expected value "a" is derived independently from the partition rule (CONTEXT.md: a key in exactly one formula is formula-local; a ∉ panel ⇒ formula-local ⇒ not bound by the panel host), not by running the code. Verified by test-partition-view.R case "panel host binds only panel keys".
  • new_partition_view("instance", panel_keys = c("g","h")) yields partition_host_owned_keys(view) equal to c("g","h") (the embed instance defers panel keys to the host) and partition_renders_section(view) == FALSE. Verified by test-partition-view.R case "embed instance defers panel keys to host".
  • new_partition_view("bogus") errors with a message naming the allowed roles. Verified by an expect_error in test-partition-view.R (negative space).
  • The three rewired call sites pass partition_host_owned_keys(view) in place of their inline expression, and the resulting value is byte-identical to today's: a grep for the retired literals — setdiff(consumer_keys, panel_keys) in R/paintr-shared-ui.R, host_owned_keys = character(0) and host_owned_keys <- names(dots$shared in R/paintr-app.R — returns zero matches after the change. Verified by grep -n over those two files.
  • The single render_shared_section = TRUE literal at R/paintr-app.R:169 is replaced by partition_renders_section(view) for a "single" view, and a unit test asserts partition_renders_section(new_partition_view("single")) is TRUE while it is FALSE for both other roles — pinning the UI/server agreement that drift used to break.
  • Behavioral invariant preserved (regression target): a shared key in exactly one formula renders+binds in that formula's shared section; a shared key in ≥2 formulas renders+binds in the one shared panel — unchanged before/after. Verified by the e2e grid/shared cluster passing (Definition of Done).
  • Full authoritative gate (below) shows FAIL 0 / ERROR 0 / SKIP 0 with PASS rising by the number of new test-partition-view.R cases and zero new failures in pre-existing tests; devtools::check(--as-cran) clean.

Constraints

  • Files this plan owns:
    • R/paintr-partition-view.R — new. The three internal functions + the ptr_partition_view constructor. No @export (internal seam).
    • R/paintr-app.R — two edits only: (a) at the single-instance binder call (~:337-339) construct view <- new_partition_view("single") and pass host_owned_keys = partition_host_owned_keys(view); (b) at the embed-instance binder call (~:1164 / :1214) construct view <- new_partition_view("instance", panel_keys = names(dots$shared %||% list())) and pass host_owned_keys = partition_host_owned_keys(view). The render_shared_section = TRUE at :169 becomes partition_renders_section(new_partition_view("single")).
    • R/paintr-shared-ui.R — one edit: at the panel-host binder call (~:416) construct view <- new_partition_view("panel_host", panel_keys = panel_keys, consumer_keys = consumer_keys) and pass host_owned_keys = partition_host_owned_keys(view).
    • tests/testthat/test-partition-view.R — new. The unit cases enumerated in Success Criteria + the BDD scenarios below. These are pure-function tests (no app boot, no NOT_CRAN).
    • NEWS.md — one bullet under the appropriate heading: internal refactor, no user-visible change ("Unified shared-key ownership behind an internal partition_view; no behavior change").
  • Files this plan must NOT touch:
    • R/paintr-shared-coordinator.R — the partition algorithm (shared_partition(), panel_keys <- all_keys[formula_counts >= 2L] at :42) is already centralised and correct. This plan changes the consumption side only.
    • R/paintr-server.R — the binder ptr_bind_local_shared_consumers() keeps its host_owned_keys parameter and its setdiff(names(resolutions), host_owned_keys) body verbatim. The view feeds that parameter; it does not change the binder.
    • The source-ownership / panel_sources path (ptr_setup_panel_sources(), R/paintr-shared-ui.R:485+). Panel-owned sources already derive ownership from the same panel_keys and are delivered via the panel_sources reactive list, not via a hand-wired complement — there is no separate source-ownership channel to absorb. Leave it byte-stable (see Out of scope).
    • Any exported function signature or behavior — ptr_app / ptr_shared / ptr_shared_panel / ptr_shared_server / ptr_server are frozen (CONTEXT.md locked).
    • Generated man/*.Rd / NAMESPACE — no exported surface changes, so devtools::document() is a no-op here; do not hand-edit.
  • Behavior must be byte-stable: every accessor returns exactly the value the rewired site passes today. This is provable, not aspirational — the unit tests enumerate all three roles against the literals currently in source.
  • No public API change. No new exported symbol. No reactive state inside the view (it is a pure value object; ADR 0006's state = NULL host scope is unaffected because the view never touches state).
  • Do not fold in validate_partition_invariant() (the ADR's optional merge from the dropped partition-rule-ownership-verification candidate). It is a separable follow-up; including it would break this plan's independence and behavior-preservation guarantee. See Out of scope.

BDD

Feature: partition_view is the single origin for shared-key ownership

  Scenario: single instance owns every shared key and renders its section
    Given a single-instance surface with no coordinator
    When  a partition_view is built with role "single"
    Then  partition_host_owned_keys(view) is character(0)
    And   partition_renders_section(view) is TRUE

  Scenario: panel host binds only the cross-formula (panel) keys
    Given shared consumer keys {a, b} where only b is referenced by >=2 formulas
    When  a partition_view is built with role "panel_host", panel_keys {b}, consumer_keys {a, b}
    Then  partition_host_owned_keys(view) is {a}
    And   partition_renders_section(view) is FALSE

  Scenario: embed instance defers the panel keys to the host
    Given an embedded instance whose coordinator owns panel keys {g, h}
    When  a partition_view is built with role "instance", panel_keys {g, h}
    Then  partition_host_owned_keys(view) is {g, h}
    And   partition_renders_section(view) is FALSE

  Scenario: an unknown surface role is rejected at construction
    Given a caller passing role "bogus"
    When  new_partition_view("bogus") is evaluated
    Then  it errors with a message listing the allowed roles single, panel_host, instance

  Scenario: a formula-local var(shared="g") in a grid where g is also cross-formula still binds
    Given a grid with formulas f1 = ppVar(shared="a") and f2 = ppVar(shared="a") + ppVar(shared="g") and a separate formula using g
    When  the app boots and the panel host and each instance build their partition_view
    Then  g renders+binds exactly once in the standalone shared panel
    And   the formula-local var(shared="a") picker in each instance is populated, not blank

Definition of Done

Authoritative full-suite gate (CLAUDE.md), run serially — devtools::test() sets NOT_CRAN=true itself so the browser/e2e tests RUN (0 SKIP):

Rscript -e 'devtools::test(reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / WARN 0 / ERROR 0 / SKIP 0 / PASS ≥ 3439 + (new test-partition-view.R cases). The count is not the gate; 0/0/0 is. The blast-radius e2e cluster that must be green (run via devtools::test(filter="...") so NOT_CRAN is set): l2-shared, l2-shared-partition, grid-shared-partition, app-grid-shared-added — this is a shared-runtime change, so the full e2e suite is the merge gate, not a leaf subset.

Plus the package check, expected clean (by-design NOTEs only — CRAN "new submission", .git, top-level CONTEXT.md):

Rscript -e 'devtools::check(document = FALSE, manual = FALSE, args = c("--as-cran","--no-manual"))'

Named proxy traps (do not be fooled):

  • A raw Rscript -e 'testthat::test_dir()' with no NOT_CRAN makes skip_on_cran() fire → every shinytest2 browser test silently SKIPs. "PASS, SKIP n" off that is the inner-loop check, not the gate — the grid/shared e2e fixtures are precisely the ones that would skip, and they are this change's regression target.
  • Under TESTTHAT_PARALLEL=TRUE, testthat::test_dir() defaults to load_package="none" so workers don't load ggpaintr → use devtools::test(). Parallel also exposes boot-tail seeding/binding flakes; a residual flake under parallel load must be re-confirmed serially before being called a regression.
  • devtools::check(--as-cran) does NOT make skip_on_cran() fire (devtools sets NOT_CRAN=true for the check subprocess); the e2e file instead cleanly SKIPs via the source-root guard. A check run showing the e2e file ERRORing ("DESCRIPTION not found") is a harness defect, never a product failure and never the gate.

Out of scope (explicit)

  • Source-ownership refactor. panel_sources / ptr_setup_panel_sources() stay byte-stable. Source ownership already flows from panel_keys; there is no second hand-derived source complement to centralise. A future plan could add a partition_owns_source(view, sid) accessor, but not here.
  • validate_partition_invariant() fail-fast at ptr_shared() construction. The ADR floats merging the dropped partition-rule-ownership-verification candidate; deferred to a separable follow-up plan so this one stays a pure behavior-preserving seam.
  • The render_shared_section = FALSE default sites (app.R:403/760/964). They already evaluate to the correct non-"single" answer; this plan retires only the one explicit TRUE literal (the asymmetric one). Threading a view object through the entire UI-assembly chain is unnecessary surface for a behavior-frozen refactor.
  • Any change to shared_partition(), the binder, the coordinator object fields, or CONTEXT.md/ADR text.

Handling new findings: if implementation surfaces anything outside this plan's scope — a fourth ownership derivation, a related drift in render_shared_section, a panel_sources irregularity, a stale test — surface it as a tracked follow-up (a note in this plan dir or a .scratch/ issue), never silently absorb it into this change or paper over it. A new finding is a reason to stop and record, not to scope-creep.

Independence

Any implementer / agent / self-report status block touching this plan — "done", "tests pass", "scope clean", "byte-stable confirmed" — is a claim to verify, never proof. It is the reporter's belief produced by the same fallible process, evidence to prioritise what to check, never a substitute for checking. The integrator independently re-runs the Definition-of-Done command above and reads its actual output (FAIL 0 / ERROR 0 / SKIP 0) before trusting any status, and does not treat a sub-agent's or prior turn's "green" as the gate. The byte-stable claim in particular is verified by the post-change grep for the retired literals returning zero matches, run by the integrator, not by accepting the implementer's word.

Dependencies & independence

No dependencies. Single self-contained plan (parallel group G1, no siblings). The new file + four edits + new test file land together; nothing else in the tree references partition_view until this plan creates it. The three rewired sites are independent of each other (different surfaces) and each is a literal-for-call substitution with an identical return value.
