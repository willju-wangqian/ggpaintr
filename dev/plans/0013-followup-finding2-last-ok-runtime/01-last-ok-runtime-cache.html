PLAN-01 (Follow-up Finding 2) — last_ok_runtime cache for validate_input retain-on-error

Source: ADR 0013 follow-up · Depends on: PLAN-07 (already merged at d17ad3d on post-add-expr) · Parallel group: G0 (standalone, no in-bundle siblings) · Status: draft

Context

Two plans in the ADR-0013 super-app pressure-test suite block on identical product behavior:

  • PLAN-03 D9 (dev/plans/0013-super-app-pressure-tests/03-app-2a-upload-registry.html): asserts "code panel must still contain `0.42^2` from the prior successful draw" after setting ppPower to 1.5 (out of [0,1]). Current product clears it.
  • PLAN-06 I5 (c) (dev/plans/0013-super-app-pressure-tests/06-app-4-user-css-safety-adversarial.html): asserts the prior G5-draw's qsec sentinel survives in the code panel after setting ppColor to "notahex". Same product clears it.

Current control flow (verified against source at 4a23d39):

  1. R/paintr-substitute.R:114-119 — value-role substitute_walk.ptr_ph_value calls entry$validate_input(value, hook_ctx). On non-TRUE / non-NULL return, rlang::abort() with Invalid value for placeholder \`<kw>\`: <msg>. (Mirror at :160-165 for consumer-role.)
  2. R/paintr-safe.R:55-65 — ptr_complete_expr_safe's tryCatch(error = ...) branch returns list(ok = FALSE, stage = "complete", code_text = "", pruned = NULL, plot = NULL, error = "Input error: ...", condition = e). code_text = "" is hard-coded.
  3. R/paintr-server.R:2029-2034 — format_code_with_extras: base <- if (is.null(res)) "" else (res$code_text %||% ""); if (is.null(res) || !isTRUE(res$ok) || length(extras_exprs) == 0L) return(base). Emits the empty string to the code output.
  4. R/paintr-server.R:1981-1991 — ptr_register_plot: if (is.null(res) || !isTRUE(res$ok) || is.null(res$plot)) { graphics::plot.new(); return(invisible(NULL)) }. The plot also blanks (the comment "matches legacy graphics::plot.new()" makes this intentional). The PLAN-03/06 tests pass expect_no_plot_error(app) only because plot.new() doesn't add a shiny-output-error class — the prior plot is NOT visually preserved.

Both plans' BDD Then clauses — "the rendered code panel still contains the sentinel from the prior successful draw" and "the host output retains its prior successful plot" — codify a retain-on-error UX that the product currently does not deliver.

Governing principle

Transient input errors must NOT discard the rendered context the user was about to compare against. Text editors don't blank on a syntax error; REPLs don't blank on a type error; drawing apps don't blank on one bad parameter. The error pane surfaces the diagnostic; the code panel and plot panel retain the last successful output until the user fixes the input and re-draws.

The fix introduces a single last_ok_runtime reactiveVal on ptr_state, populated by a one-line observer on state$runtime()'s ok-branch, and read as a fallback in ptr_register_code and ptr_register_plot when the current runtime is not-ok. No changes to ptr_substitute, no changes to validate_input's contract, no changes to the error-pane rendering.

Success Criteria

  • state$last_ok_runtime exists on ptr_state. Added to ptr_init_state at R/paintr-server.R:86 as a shiny::reactiveVal(NULL). Added to ptr_validate_state's needed field list at R/paintr-server.R:404. A unit test (tests/testthat/test-validate-input-retain.R) constructs a ptr_state via ptr_init_state() and asserts shiny::is.reactivevalues(.) || shiny::is.reactive(state$last_ok_runtime) AND shiny::isolate(state$last_ok_runtime()) is NULL on initialization.
  • A dedicated registration helper ptr_register_last_ok_cache(output, state) is added next to the existing ptr_register_* family in R/paintr-server.R and invoked from the server-setup site at R/paintr-server.R:750-752 (alongside ptr_register_plot, ptr_register_error, ptr_register_code). The helper registers a shiny::observe() that reads state$runtime() and, when isTRUE(res$ok), writes state$last_ok_runtime(res). The helper takes output only to match the existing register-family signature; it does not write to output (no output[[…]] <-).
  • ptr_register_code's final-mode branch falls back to last_ok_runtime on !isTRUE(res$ok). The change is local to R/paintr-server.R:2080: 
    # OLD
format_code_with_extras(state$runtime(), state$extras_exprs())

# NEW
res <- state$runtime()
last <- state$last_ok_runtime()
chosen <- if (is.null(res) || isTRUE(res$ok) || is.null(last)) res else last
format_code_with_extras(chosen, state$extras_exprs())
    Preserve-mode (R/paintr-server.R:2057-2076, the identical(mode, "preserve") branch) ALSO falls back to last_ok_runtime's snapshot when the current runtime is not-ok and a prior ok-result exists. If no prior ok-result exists, preserve-mode renders the current state as today.
  • ptr_register_plot falls back to last_ok_runtime$plot on !isTRUE(res$ok) || is.null(res$plot) when a prior ok-result exists. Edit at R/paintr-server.R:1984-1990:

if (!is.null(res) && isTRUE(res$ok) && !is.null(res$plot)) {

if (!is.null(last) && !is.null(last$plot)) {
  return(last$plot)
}
graphics::plot.new()
invisible(NULL)
  • New test file tests/testthat/test-validate-input-retain.R drives the cache + fallback via shiny::testServer. Asserts at least the following expectations (each is a separate expect_*, so PASS rises by ≥ 8):
    1. Initial state: state$last_ok_runtime() is NULL on session boot.
    2. First successful draw populates the cache: after a Draw click that produces res$ok == TRUE, state$last_ok_runtime() equals that res (or has the same code_text + plot at minimum).
    3. validate_input failure leaves the cache UNCHANGED: drive an out-of-range value through a placeholder whose validate_input returns a non-TRUE string; state$runtime() has ok = FALSE; state$last_ok_runtime() still equals the prior successful res.
    4. Code-panel renderer returns the cached code_text on fail: read the output reactive at the code-panel id and assert identical(., format_code_with_extras(prior_ok_res, state$extras_exprs())) — NOT "".
    5. Plot-panel reactive returns the cached plot on fail: read via the documented session$getReturned() or equivalent testServer hook; assert !is.null(.) when a prior ok-result exists. (If testServer can't surface the rendered plot object, replace this expectation with a direct read of state$last_ok_runtime()$plot after a fail and assert it equals the prior res$plot.)
    6. Recovery: fix the input to a valid value, draw again; state$runtime() ok = TRUE; state$last_ok_runtime() updates to the NEW res; the prior cached res is replaced.
    7. Consecutive fails preserve the cache: drive two consecutive validate_input failures from a known-good baseline; assert state$last_ok_runtime() is unchanged across both fails.
    8. No prior ok-result + immediate fail: a session that boots and immediately drives a validate_input failure (no successful intermediate draw) leaves state$last_ok_runtime() as NULL; the code-panel renderer emits "" (same as today's behavior); the plot-panel renderer blanks the device (same as today).
    Tests use a minimal in-test placeholder registration: define a custom ppRetainTest value placeholder with validate_input = function(v, ctx) if (is.numeric(v) && v >= 0 && v <= 1) TRUE else "must be in [0,1]" (or reuse PLAN-07's existing test fixture in test-validate-input-ctx.R if a registered placeholder already exists there). Clean up via ptr_clear_placeholder() in withr::defer.

BDD scenarios

Feature: validate_input failure retains last successful code + plot until recovery

Scenario: Initial boot — cache is empty
  Given a ptr_state freshly constructed by ptr_init_state()
  When no draw has been triggered yet
  Then state$last_ok_runtime() is NULL
  And the code panel reactive returns ""
  And the plot panel reactive blanks the device

Scenario: First successful draw populates the cache
  Given a ptr_state with a registered draw trigger
  When the user clicks Draw with all valid inputs
  And state$runtime() returns ok = TRUE
  Then state$last_ok_runtime() equals that runtime result (same code_text + plot)

Scenario: validate_input failure preserves the cache
  Given a prior successful draw with a known code_text and plot in the cache
  When the user sets a placeholder input to a value whose validate_input returns the string "must be in [0,1]"
  And clicks Draw
  Then state$runtime() returns ok = FALSE with error "Input error: Invalid value for placeholder ..."
  And state$last_ok_runtime() is UNCHANGED from before the failed draw
  And the code panel reactive returns the prior code_text (NOT "")
  And the plot panel reactive returns the prior plot object (NOT a blank device)
  And the error panel reactive returns a non-NULL renderUI showing the validator message

Scenario: Recovery — cache replaces on next successful draw
  Given a failed-state app whose cache holds a prior successful result
  When the user fixes the input to a valid value
  And clicks Draw
  And state$runtime() returns ok = TRUE
  Then state$last_ok_runtime() is REPLACED with the new ok-result
  And the code panel reactive reflects the NEW code_text (no leftover from the cache)

Scenario: Consecutive failures preserve the cache
  Given a prior successful draw cached
  When the user drives two consecutive validate_input failures (different invalid inputs)
  Then state$last_ok_runtime() is unchanged across both fails
  And both fails surface their respective error messages in the error pane

Scenario: No prior ok-result + immediate failure
  Given a freshly booted ptr_state with no prior successful draw
  When the user immediately drives a validate_input failure on the first Draw click
  Then state$last_ok_runtime() remains NULL
  And the code panel reactive returns "" (same as today)
  And the plot panel reactive blanks the device (same as today)
  And the error panel reactive shows the validator message

Scenario: Preserve-mode toggle during a failed state
  Given a prior successful draw cached, current runtime ok = FALSE
  When the user toggles code mode to "preserve"
  Then the preserve-mode renderer reads from state$last_ok_runtime()$snapshot (not the current failed runtime)
  And the rendered code contains the prior draw's preserve-mode round-trip

Definition of Done

Authoritative gate (verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / WARN 0 / SKIP 0 / PASS (baseline + N) where N ≥ 8 (the new test_that expectations from test-validate-input-retain.R). Baseline at post-add-expr HEAD 4a23d39 is PASS 2352. Target after this plan: PASS ≥ 2360.

Proxy traps — do not be fooled (verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard. If a check run shows the e2e file ERRORing, that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate.
  • Plan-specific proxy trap: a green gate from running only test-validate-input-retain.R in isolation is NOT the gate. The full-suite gate is required because the new behavior changes the contract of state$runtime()'s downstream consumers (ptr_register_plot, ptr_register_code), and any test elsewhere in the suite that implicitly asserted the OLD empty-on-fail behavior would silently regress.
  • Plan-specific proxy trap: a passing state$last_ok_runtime() reactiveVal probe at orchestrator-side (e.g. Rscript -e 'state <- ptr_init_state(...); shiny::isolate(state$last_ok_runtime())') is NOT the gate. The gate is the full testthat::test_dir; the orchestrator-side probe only confirms the field exists, not that the fallback wiring works end-to-end.

They are L3 read accessors with documented "current runtime" semantics; changing them is a separate question. Surface as a follow-up note in the report.
  3. Document the narrowing in a code comment citing this plan. Do NOT eliminate the cache entirely as a memory-pressure mitigation.
  4. An issue with the preserve-mode toggle reactive ordering (the toggle reads state$runtime()$snapshot; the fallback reads last_ok_runtime$snapshot — if these diverge mid-flush) — surface it explicitly; do not paper over with a shiny::isolate.
