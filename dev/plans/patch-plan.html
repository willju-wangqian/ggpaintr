Phase 1 — Widget copy fidelity

Root cause (audit V1/V2, items 2.5 / 2.13 / 3.10): build_ui_for resolves the full ui_text copy list but forwards only label to the placeholder build_ui hooks, and the builtin hooks emit bare widgets. So help, placeholder, empty_text copy entries — and the upload accept filter and upload_file/upload_name copy keys — are silently dropped. ptr_ui_text_leaf_fields() still advertises help/placeholder/empty_text, so users can set them with no error and no effect.

#PriChangeFilesRisk
1.1P0Pass the resolved help / placeholder / empty_text copy (in addition to label) into builtin build_ui hooks. Cleanest: pass the whole resolved copy list as a named arg (e.g. copy = copy) alongside the existing label =, so future fields don't need re-plumbing. Update invoke_build_ui, build_ui_for.ptr_ph_value, build_ui_for.ptr_ph_data_source, and the consumer path's server-side render call.R/paintr-build-ui.R, R/paintr-server.R (ptr_setup_consumer_uis path)M
1.2P0Re-introduce a ptr_attach_help(ui, help) internal helper (legacy paintr-ui.R): when help is a non-empty string, wrap the control in shiny::tagList(ui, shiny::helpText(help)). Apply in every builtin build_ui hook (text / num / expr / var / upload).R/paintr-builtins.R (or a small new R/paintr-help.R)L
1.3P1text builtin: pass placeholder = copy$placeholder to shiny::textInput(); restore copy default placeholder = "Plain text - quotes are added automatically" in ptr_register_builtins().R/paintr-builtins.RL
1.4P1expr builtin: pass placeholder = copy$placeholder to shiny::textAreaInput(); restore an analogous copy default. (Keep the multi-line textAreaInput — intentional change.)R/paintr-builtins.RL
1.5P1var builtin (consumer): use noneSelectedText = copy$empty_text %||% picker_label; restore copy default empty_text = "Choose one column"; restore the legacy default label text "Choose a column for {param}" (currently "Pick a column for {param}"). Thread copy/empty_text from the server-side consumer render (it goes through ptr_setup_consumer_uis, not build_ui_for directly — confirm that path resolves & forwards copy).R/paintr-builtins.R, R/paintr-server.RM
1.6P0upload builtin (source): add accept = c(".csv", ".tsv", ".rds", ".xlsx", ".xls", ".json") to shiny::fileInput().R/paintr-builtins.RL
1.7P1Restore the upload_file / upload_name copy components: re-add them to the upload copy group in ptr_ui_text_component_paths() / ptr_default_ui_text() and to the allowed_* sets in ptr_validate_ui_text(). Drive the fileInput label/help from upload_file and the companion textInput label/placeholder/help from upload_name (instead of the hard-coded "Upload data" / "Dataset name"). Wire via build_ui_for.ptr_ph_data_source resolving both components and passing them to the hook.R/paintr-copy.R, R/paintr-build-ui.R, R/paintr-builtins.RM

Phase 3 — Layer-disabled visual cue

Audit V3 / item 3.3: the ptr-layer-disabled class is applied only at UI-construction time (build_ui_for.ptr_layer: if (default_on) "ptr-layer-content" else "... ptr-layer-disabled"). When the user toggles a layer's include-checkbox at runtime the layer is correctly dropped from the plot, but the panel is not greyed out. Legacy toggled the class via a ptr_set_class custom-message handler + a style tag.

#PriChangeFilesRisk
3.1P1Re-add the JS plumbing: a Shiny.addCustomMessageHandler("ptr_set_class", …) (or use shinyjs::toggleClass if shinyjs is already a dep — it is not in DESCRIPTION, so prefer the hand-rolled handler) plus a <style> tag defining .ptr-layer-disabled { opacity: .5; pointer-events: none; } (port legacy ptr_layer_switcher_style_tag()). Inject both into the app head — in ptr_build_app_ui(), ptr_app_bslib(), and ptr_module_ui().R/paintr-app.R, R/paintr-app-bslib.R, R/paintr-build-ui.R (helper)M
3.2P1Add a server observer: for each layer with an include-checkbox, observeEvent(input[[layer_checkbox_id]], session$sendCustomMessage("ptr_set_class", list(id = ns(layer_content_id), class = "ptr-layer-disabled", add = !isTRUE(input[[...]])))). Put it next to ptr_setup_layer_picker (e.g. a new ptr_setup_layer_panel_classes(state, input, session)), called from ptr_server. Requires a stable, namespaced layer-content-div id — expose the id-builder used in build_ui_for.ptr_layer (e.g. layer_content_id(layer_name)) so both UI and server agree.R/paintr-server.R, R/paintr-build-ui.R, R/paintr-ids.RM

Phase 5 — Definition / robustness regressions

#PriChangeFilesRisk
5.1P2Audit item 3.10: validate_copy_defaults() currently only checks "named list of length-1 strings" — a typo'd copy field (e.g. tooltip) is silently accepted. Restore the legacy whitelist check: reject any name not in ptr_ui_text_leaf_fields() (which after 1.7 also covers the upload-specific keys, or handle those by keyword). Mirrors legacy ptr_validate_placeholder.R/paintr-registry.R, R/paintr-copy.RL
5.2P2Audit item 3.11: re-add a lightweight internal ptr_validate_state(state) — assert the required reactives/fields exist and are of the right kind (tree, runtime, extras/extras_exprs, eval_env, …) — and call it at the top of ptr_register_plot / ptr_register_error / ptr_register_code (and any other public function that takes state). Restores the legacy guard for embedders passing a malformed state.R/paintr-server.RL
5.3P3Audit item 3.6 (ADVISORY): the substitute-stage safety re-check of resolver output hardcodes expr_check = TRUE, so a user's custom denylist/allowlist or expr_check = FALSE is ignored there (the eval-stage check still honors it, so this is "stricter than asked", not unsafe). Optional fix: thread an expr_check parameter through ptr_substitute() → validate_resolve_expr_return() so the substitute-stage check uses the user's setting too. Skip if you're happy with "always strict on synthesized code".R/paintr-substitute.R, R/paintr-server.RM
5.4P3Audit item 2.12: expr parse-error message wording regressed. Optional: on parse_exprs failure, wrap the parser's conditionMessage into the abort (legacy: "expr placeholder: could not parse input as R expression: <msg>") and use legacy multi-expression wording ("input must contain exactly one expression, but N were found.").R/paintr-builtins.RL

Sequencing, risks & verification

Suggested order

  1. Phase 1 (widget copy) — biggest user-facing fidelity gain; touches paintr-builtins.R + paintr-build-ui.R + paintr-copy.R. Do 1.1+1.2 first (the plumbing), then 1.3–1.7 (per-widget) build on it.
  2. Phase 2 (error/render) — small, independent.
  3. Phase 5.1, 5.2 (whitelist, validate_state) — small, independent; do alongside Phase 1.
  4. Phase 4 (verb-suffix labels) — independent; needs the legacy verb helper ported.
  5. Phase 3 (layer-disabled cue) — highest-risk (new JS + new server observer + a stable layer-content id contract between UI and server); do last among the code phases.
  6. Phase 6 — write each paired test as part of its phase (TDD-friendly); the 6b standalone tests can land any time.
  7. Phase 7 — separate decisions.

Risks & watch-outs

  • Consumer (var) copy path. var widgets are server-rendered via ptr_setup_consumer_uis, not the static build_ui_for call. Items 1.1 and 1.5 must thread the resolved copy through that server path too — confirm it has access to ui_text / layer_name / param there.
  • Layer-content id contract (Phase 3). The UI builds the layer panel div with some id; the server observer must use the same namespaced id. Extract a single layer_content_id() helper and use it on both sides — don't duplicate the string.
  • No NAMESPACE change expected. All new helpers (ptr_attach_help, ptr_validate_state, layer_content_id, the verb helper, the message-handler injector) are internal/unexported, and re-adding the upload_file/upload_name copy keys doesn't touch exports — so devtools::document() isn't required unless roxygen comments on exported functions change. Still run it if any @param changes.
  • Don't accidentally restore the intentional removals. Phase 1.7's copy keys must not drag back the update_data_button / update_data_stale_class keys or the "Update data" button — those stay removed.
  • shinyjs. Not a current dependency; Phase 3 should use a hand-rolled addCustomMessageHandler rather than adding shinyjs to DESCRIPTION.

Verification gate (run after each phase, and once at the end)

  • devtools::test() — 0 failures; new tests present and passing.
  • devtools::test(filter = "<the file you touched>") during iteration.
  • devtools::check(document = FALSE, manual = FALSE, args = c("--as-cran", "--no-manual")) — 0 errors / 0 warnings (per project rule).
  • Manual smoke (per .claude/rules/testing.md manual coverage): launch ptr_app() with a formula using text/num/expr/var/upload placeholders and a ui_text override setting help/placeholder/empty_text/upload_file/upload_name — confirm they render; toggle a layer checkbox — confirm the panel greys out; trigger a plot error — confirm the canvas clears and the message reads "Plot error: …"; check a data-pipeline placeholder label reads "… in mutate()".
  • Re-render README.Rmd only if any example formula / copy default shown there changed.
  • Update dev/legacy-audit.html status notes for the items resolved (optional).
