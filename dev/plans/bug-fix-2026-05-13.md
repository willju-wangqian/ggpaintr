## BUG-A1 — aes `var` pickers stale after `pick_ds` switch (MEDIUM)

**Symptom.** App 1a/1b. With `pick_ds` starting on `mpg`, the three aes pickers (`ggplot_1_1_var_NA`, `ggplot_1_2_var_NA`, `ggplot_1_3_var_NA`) populate correctly on first render. Switch `pick_ds` → `diamonds`; wait for the producer-debounce. Mutate-stage and filter-stage `var` pickers and the `colvars` consumer all refresh to the diamonds columns. The three aes pickers stay with empty option lists indefinitely. Selecting fresh `colvars` then cascades to mutate/filter but still not to the aes pickers.

**Files almost certainly involved.**
- `R/paintr-server.R::ptr_setup_consumer_uis` (line ~772). The per-consumer `entry_reactive` already depends on `state$resolved_sources` (line ~822) and the upstream consumer/producer/source IDs. So the wiring framework is there. The question is whether aes consumers under a `ggplot()` call carry the right `upstream` chain at translate time.
- `R/paintr-translate.R` and/or `R/paintr-classify.R` — wherever the upstream chain for consumers nested inside `aes()` inside `ggplot()` gets built.
- `R/paintr-walk.R::find_consumer_ids_in_upstream` / `find_producer_ids_in_upstream` / `find_source_companion_ids_in_upstream` / `find_source_self_ids_in_upstream` — make sure they reach into the `aes()` call args, not just the top-level pipeline stages.

**Diagnostic to run first.** From a `testServer(ptr_server, ex1_formula, …)` session, after `session$setInputs(ggplot_1_pick_ds_NA = "diamonds")`:
1. Print the tree node for `ggplot_1_1_var_NA` and inspect its `$upstream`.
2. Print `find_source_self_ids_in_upstream(node$upstream)` — does it include `ggplot_1_pick_ds_NA`?
3. Print the same for the mutate-stage `var` picker (`ggplot_4_1_1_var_NA`), which we know does refresh.

The pickers that **do** refresh and the ones that **don't** share a tree but presumably differ in one of those three properties. The diff is the bug.

**Fix shape.** Either:
- the aes consumer's upstream chain is missing the `pick_ds` source ID — fix the tree builder so aes-arg consumers inherit the enclosing `ggplot()`'s data-source chain (the same way they already do for the data resolve path; otherwise the initial render wouldn't work either); or
- the upstream chain is correct but the `find_source_self_ids_in_upstream` walker stops short of the aes args. In that case extend the walker to recurse into call arguments of `aes`/`ggplot` calls.

Whichever it is, the fix must keep the **initial render** behaviour (which already works — aes pickers populate from the default `pick_ds = "mpg"` on first load), so the regression test below is required.

**Regression test (`tests/testthat/test-regression-bugs-2026-05-13.R`).**

```r
test_that("BUG-A1: aes var pickers refresh when pick_ds source changes", {
  withr::local_options(list(shiny.testmode = TRUE))
  # Use the same custom `pick_ds` source as feature-coverage-examples.R.
  ptr_define_placeholder_source(
    keyword       = "pick_ds",
    build_ui      = function(node, label = NULL, ...) {
      shiny::selectInput(node$id, label = label %||% "Dataset",
                         choices = c("mpg", "diamonds"))
    },
    resolve_data  = function(value, node, ...) {
      if (is.null(value) || !nzchar(value)) return(NULL)
      get(value, envir = asNamespace("ggplot2"))
    },
    resolve_expr  = function(value, node, ...) {
      if (is.null(value) || !nzchar(value)) return(NULL)
      rlang::sym(value)
    },
    copy_defaults = list(label = "Dataset for {param}")
  )
  withr::defer(ptr_clear_placeholder("pick_ds"))

  formula <- "pick_ds |> ggplot(aes(x = var, y = var))"
  shiny::testServer(function(input, output, session) {
    st <- ptr_server(input, output, session, formula)
    session$setInputs(ptr_update_plot = 1, .dummy = 1)
    # Initial: pick_ds default is "mpg" -- aes pickers see mpg cols.
    expect_setequal(
      runtime_consumer_entry(st, find_node_by_id(st$tree(), "ggplot_1_1_var_NA"),
                              snapshot = list())$cols,
      names(ggplot2::mpg)
    )
    # Switch source. Aes pickers must refresh.
    session$setInputs(ggplot_1_pick_ds_NA = "diamonds")
    expect_setequal(
      runtime_consumer_entry(st, find_node_by_id(st$tree(), "ggplot_1_1_var_NA"),
                              snapshot = list(ggplot_1_pick_ds_NA = "diamonds"))$cols,
      names(ggplot2::diamonds)
    )
  }, args = list())
})
```

If `find_node_by_id` doesn't exist as an exported helper, walk the tree inline with `find_nodes` + a predicate on `$id`.

**Iceberg check.** Before declaring done: file Step-3 from App 3's NEVER-EXERCISED walkthrough (which uses an `upload` source, not `pick_ds`). If aes pickers stay empty there too after a successful file upload, the fix wasn't deep enough — `pick_ds`-vs-`upload` source-type asymmetry was the same shape as BUG-1 deep in `c0d5467`.

## BUG-A2 — `ui_text$defaults$num$help` / `empty_text` ignored on shared widget (LOW)

**Symptom.** App 1a. With `ui_text$defaults$num = list(help = "Any number works here.", empty_text = "No number yet")`, the per-stage num widgets (head, filter) render with the override help text. The shared `num(shared = "lvl")` widget under "Shared controls" instead shows the alpha-specific default `"Enter a value between 0 and 1."` — i.e. the override never reaches it. `empty_text` override is also not visible.

**Files almost certainly involved.**
- `R/paintr-shared.R::shared_widget_label` (line ~167) — currently returns a label string (or NULL) but does **not** return a `help`/`empty_text`. The shared-widget UI builder must derive those four leaves through `ptr_resolve_ui_text` exactly like the per-stage builder does, but with `param = NULL` (multi-param) and `layer_name = NULL`. The current code only ever reads `copy_defaults$label`.
- `R/paintr-copy.R::ptr_resolve_ui_text` — confirm it accepts `param = NULL` and returns the `defaults$<keyword>` branch as the most-specific fallback. Spec from `dev/plans/api-audit.md`: signature reorder to `(component, keyword, param, layer_name, ui_text)` already landed.
- Whoever calls `shared_widget_label` and builds the shared widget UI (`paintr-shared.R` or `paintr-build-ui.R::invoke_build_ui` for a shared occurrence) — the call must thread `help`/`empty_text`/`placeholder` through to the placeholder's `build_ui` exactly like the non-shared path.

**Decision point.** Two acceptable resolutions:
1. **Apply the override.** Shared widget pulls `help`/`empty_text` from `ptr_resolve_ui_text("control", keyword = "num", param = NULL, …)`, which falls through `defaults$num`. Preferred — matches the docs intent and the per-stage behaviour.
2. **Document the omission.** Note explicitly that the shared widget intentionally uses per-param help when all occurrences share a more-specific param-bound rule; only the **label** is param-stripped. This is harder to defend (the user already chose to combine alpha + size into one widget — they don't want alpha's "0 and 1" help bleeding onto the size context).

Recommend (1).

**Regression test.**

```r
test_that("BUG-A2: defaults$num overrides reach the shared num widget", {
  formula <- paste0(
    "ggplot(mtcars, aes(x = mpg, y = hp)) + ",
    "geom_point(alpha = num(shared = 'lvl'), size = num(shared = 'lvl'))"
  )
  ui_text <- ptr_ui_text(list(defaults = list(num = list(
    help = "Any number works here.",
    empty_text = "No number yet"
  ))))
  copy <- ptr_resolve_ui_text("control",
                              keyword = "num",
                              param = NULL,        # shared widget: no single param
                              layer_name = NULL,
                              ui_text = ui_text)
  expect_equal(copy$help, "Any number works here.")
  expect_equal(copy$empty_text, "No number yet")

  # End-to-end through the shared-widget builder: scrape the rendered UI.
  shiny::testServer(function(input, output, session) {
    st <- ptr_server(input, output, session, formula, ui_text = ui_text)
    session$flushReact()
    ui <- output[["lvl_ui"]]   # or whichever output the shared widget mounts to
    expect_match(as.character(ui), "Any number works here.", fixed = TRUE)
  }, args = list())
})
```

(Confirm the shared-widget output id by inspecting `state$shared_resolutions` or the rendered DOM during a manual probe before finalising the test.)

## BUG-B1 — `ptr_app_bslib(title = …)` silently overridden by `ui_text$shell$title$label` (LOW)

**Symptom.** App 1b. `ptr_app_bslib(ex1_formula, …, title = "Feature-coverage demo (bslib)", ui_text = ex1_ui_text)` where `ex1_ui_text$shell$title$label = "Feature-coverage demo"`. Both `document.title` and the bslib navbar brand render `"Feature-coverage demo"`. The bslib-only `title` arg never appears.

**Files involved.**
- `R/paintr-app-bslib.R::ptr_app_bslib` — the `title` argument and how it threads into `ptr_app_bslib_components()` (or similar). Currently it's plumbed into `bslib::page_navbar(title = …)` but is also being overwritten downstream by the `ui_text$shell$title$label` read from `ptr_resolve_ui_text("title", …)`.

**Decision.** Make the explicit `title` argument win when it's passed. The current `ptr_app_bslib(title = …)` exists in the public signature as a bslib-only courtesy; users who pass it have explicitly asked for that string in the navbar brand and `<title>`. `ui_text$shell$title$label` keeps owning the default and the non-bslib path; bslib's `title` only overrides when supplied.

Concretely: in `ptr_app_bslib`, detect "user passed `title`" (default it to `NULL` if it isn't already, then test for `is.null(title)`), and when non-NULL, pass it straight through to `bslib::page_navbar(title = …)` and as the `<title>` without letting the `ui_text` resolution overwrite it. When `title` is NULL, fall through to `ptr_resolve_ui_text("title", …, ui_text = …)$label` as today.

Also check: the bug may be partly that the current default `title = "ggpaintr Plot Builder"` looks unset to the function but is actually passed — verify whether the bug is "explicit arg ignored" or "default-arg-equal-to-NULL test wrong", and adjust accordingly.

Add a short `@section Precedence:` to `?ptr_app_bslib` stating: "If both `title` and `ui_text$shell$title$label` are supplied, `title` wins."

Estimated blast radius is small — `title` is mentioned in `?ptr_app_bslib` and the workflow vignette; `dev/scripts/feature-coverage-examples.R` already exercises it.

**Regression test.**

```r
test_that("BUG-B1: explicit ptr_app_bslib(title=) wins over ui_text$shell$title", {
  formula <- "ggplot(mtcars) + geom_point()"
  ut <- ptr_ui_text(list(shell = list(title = list(label = "From ui_text"))))
  rendered <- as.character(ptr_app_bslib(formula, ui_text = ut,
                                         title = "From title arg"))
  expect_match(rendered, "From title arg", fixed = TRUE)
  expect_no_match(rendered, "From ui_text", fixed = TRUE)
})

test_that("BUG-B1: ptr_app_bslib falls back to ui_text$shell$title when title= absent", {
  formula <- "ggplot(mtcars) + geom_point()"
  ut <- ptr_ui_text(list(shell = list(title = list(label = "From ui_text"))))
  rendered <- as.character(ptr_app_bslib(formula, ui_text = ut))
  expect_match(rendered, "From ui_text", fixed = TRUE)
})
```
