ggpaintr vignette rewrite plan

Drafted 2026-05-13 on branch core-rewrite-impl · output of a /grill-me design session · status: plan locked, implementation pending

README has been rewritten in ggplot2 style (short Overview / Installation / Usage / More topics). This plan covers the vignette set that the README's "More topics" section points at, plus the man-page and NEWS.md hand-offs for content that doesn't belong in a vignette.

Locked design decisions

DecisionChoiceWhy
Vignette role vs. e-book Self-contained mini-tutorials Package must be usable without the (still-unwritten) e-book. Some overlap with the e-book is the accepted cost.
Intro / "Getting Started" vignette None — README is the only intro Every vignette is advanced / scoped to one user question. README's Usage block does the from-zero work.
Granularity 5 vignettes total (3 new + 2 existing) One per user-question cluster. Coarser would bury sections; finer would duplicate setup boilerplate.
Customization scope ui_text + custom placeholders Same audience ("change ggpaintr's defaults"). Plot-injection helpers (ptr_gg_extra, custom renderPlot) live in use-cases L3, not here.
L3 ("Level 3") meaning In-Shiny custom rendering via extract accessors L3 is NOT headless. It is the pattern of calling ptr_extract_plot() inside your own renderPlotly() / renderGirafe() / custom UI. All L3 examples are full running apps.
Headless workflow Internal-only, never appears in any vignette The headless runtime in R/paintr-headless.R is for the reactive runtime observer and dev-side regression tests. Public surface is real Shiny session only.
testServer in vignettes Banned The extract accessors do not actually render under testServer. Public surface is real Shiny session only; vignettes must reflect that.
Formula grammar reference home ?ptr_app man page; compact tour in use-cases L1 Reference content goes where R users reach (?fn). Vignette reproduces just enough table to teach in-line. Mirrors ggplot2's ?aes strategy.
Safety story home Dedicated ggpaintr-safety vignette "Can I share this publicly?" is a coherent user question cutting across L1/L2/L3. Folding it inside another vignette would bury it.
pkgdown nav order use-cases → customization → safety → gallery → llm User-journey order. Use-cases first (dominant landing question); safety after customization (most users hit it only when going public).

Vignette set

1. ggpaintr-use-cases new

User question: How do I use ggpaintr for my situation?

Opens with: compact formula tour (5 placeholders, one table, one worked example). Full grammar lives at ?ptr_app.

Three levels:

  • L1 — All-in-one entry point. ptr_app() / ptr_app_bslib() / ptr_app_grid(). The empty-call cleanup rule shown by example. Local data with non-syntactic columns via ptr_normalize_column_names().
  • L2 — Embed in your own Shiny app. Non-module path: ptr_controls_ui + ptr_outputs_ui + ptr_server. Module path: ptr_module_ui + ptr_module_server. Split outputs with ptr_register_plot / _code / _error. Optional ptr_init_state for advanced init.
  • L3 — Custom rendering via extract accessors. ptr_extract_plot(state) inside renderPlotly(), renderGirafe(), or any custom renderer. ptr_extract_code(state) for your own code panel. ptr_extract_error(state) for custom error UI. ptr_gg_extra(state, ...) for host-app layer injection. All examples are full running apps — no testServer.

2. ggpaintr-customization new

User question: How do I tailor ggpaintr's defaults?

  • Rewriting copy with ui_text. Six-section override catalog, merge precedence rules. Helpers: ptr_ui_text, ptr_resolve_ui_text.
  • Adding a new widget type. Hook contract across ptr_define_placeholder_value / _consumer / _source. Full worked example (absorbs gallery §3).
  • Sharing one widget across two plots. The shared-placeholder pattern (absorbs gallery §8).
  • Unregistering. ptr_clear_placeholder for cleanup between sessions / examples.

3. ggpaintr-safety new

User question: How do I share this with untrusted users?

  • expr_check flag — what it does, when (never) to turn it off.
  • The denylist boundary — ~151 entries, treated as complete. The AST walker is the actual safety mechanism (not the denylist alone).
  • Upload trust model — what's accepted, where normalization happens, what the host can / can't restrict.
  • Public-deployment recipe — shinyapps.io / Posit Connect checklist; what to lock down at each level.
  • Trade-off table: L1 vs L2 vs L3 deployment, default settings, additional surface a host introduces.

4. ggpaintr-gallery kept, surgery

User question: What does a real ggplot look like as a ggpaintr formula?

Stays as a recipe collection. Sections retained (renumber after deletions):

  • §4 — A realistic ggplot2 graphic, parameterized
  • §5 — Data-cleaning pipelines into ggplot()
  • §6 — Interactive output packages (plotly, ggiraph) — now the canonical L3 worked examples; cross-link from use-cases
  • §7 — ggplot2 extension packages (ggpcp, ggridges, ggrepel, ggalluvial, ggdist)

Surgery removes §3, §8, §9 (see "Gallery surgery checklist" below).

5. ggpaintr-llm kept, unchanged

User question: How do I plug an LLM into this?

No changes. Already covers ellmer integration, primer + tool split, provider/model swap, manual registration, token-free testing.

  • Customization's placeholder section gets a backlink from gallery's old §3 stub paragraph.

Risks & open questions

  • L3 ergonomics without testServer. Authors of host apps can't unit-test their L3 integrations through ggpaintr's accessors. This is now an accepted constraint (public surface = real Shiny session). If users push back, the response is "promote a slice of the internal headless API with a settled name."
  • Safety vignette content depth. The denylist + walker discussion is technical. Open: how deep to go on the AST walker without crossing into developer-notes territory.
  • Customization vignette length. ui_text + 3 placeholder constructors + shared-placeholder recipe is substantial. May need a clear TOC at the top to keep it scannable.
  • E-book hand-off. When the e-book lands, the vignettes' tutorial framing will overlap. Plan: keep vignettes as scoped how-tos; e-book owns linear pedagogy. Revisit if duplication becomes painful.
