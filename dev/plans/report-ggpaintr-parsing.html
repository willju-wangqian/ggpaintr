ggpaintr — Formula Parsing & Runtime

Technical reference for package developers · Source at R/paintr-translate.R, R/paintr-nodes.R, R/paintr-registry.R, R/paintr-server.R, and siblings.

Overview

ggpaintr converts a plain character string — called a formula — into a running Shiny application. The formula looks like a ggplot2 expression with special placeholder tokens (var, text, num, expr, upload) standing in for values the user will supply at runtime. The translation pipeline turns that string into a typed AST, assigns Shiny input IDs to each placeholder, builds the UI, and wires up a reactive server that substitutes live input values, prunes unset nodes, evaluates the resulting ggplot call, and renders the plot.

The pipeline has three major phases:

AParse-time (static): ptr_translate() → typed AST (ptr_root) → ID assignment → data classification → shared-placeholder binding. Runs once when the app boots.
BUI build: Walk the AST to emit Shiny widgets and layout tags. Each placeholder node has a build_ui hook in the registry.
CRuntime (reactive): On each Update-Plot click, snapshot live input values → substitute into tree → prune missing → evaluate → render plot + code + errors.

1 · Default Placeholders

The five built-in placeholder types are registered by ptr_register_builtins() R/paintr-builtins.R:220, which is called lazily on first registry access via ensure_registry_initialized() R/paintr-registry.R:8. Every built-in is registered with one of three roles — "value", "consumer", or "source" — described in §7.

2 · Input Formula Structure

A formula is a single character string containing a syntactically valid R expression. The only structural constraint imposed by ggpaintr (over and above R syntax) is that it must parse to exactly one expression, and that expression must be a ggplot2-style tree joined by +.

ggplot-only formula (no pipeline)

"ggplot(iris, aes(x = var, y = var, color = Species)) + geom_point(size = num)"

Each call separated by + at the top level is one layer in ggpaintr's model. ggplot() itself is the first layer; geom_point() is the second. Each layer may contain placeholder tokens anywhere in its argument list (or nested inside aes(), etc.).

Formula with a dplyr pipeline as the data source

"ggplot(iris |> filter(Sepal.Length > num), aes(x = var, y = var)) + geom_point()"

The first positional argument to ggplot() is the data. When that argument is itself a pipe chain, ggpaintr detects it via is_pipe_head() and wraps it in a ptr_pipeline node rather than a plain ptr_call. The pipeline is the data_arg of the enclosing ptr_layer.

Pipeline as a top-level layer prefix

"upload |> dplyr::filter(Sepal.Length > num) |> ggplot(aes(x = var, y = var)) + geom_point()"

Here the entire left-hand side of + is a pipe chain terminating in ggplot(). translate_layer() detects the outer pipe operator and dispatches to translate_piped_layer(), which separates the upstream stages (everything before the terminal call) from the terminal call (ggplot(...)). The upstream stages become a ptr_pipeline node stored as data_arg of the ptr_layer.

The + separator

ggplot2 overloads the + operator to add layers to a plot object. ggpaintr leverages exactly this structure: the parsed R AST of a multi-layer formula is a left-recursive binary tree of `+`(lhs, rhs) calls. split_top_plus() R/paintr-translate.R:89 flattens this tree into a flat list of sub-expressions by recursing into the left operand.

# R parse tree for:  A + B + C
#   `+`(`+`(A, B), C)
#
# split_top_plus returns: list(A, B, C)

split_top_plus <- function(expr) {
  if (is.call(expr) && identical(expr[[1L]], quote(`+`)) && length(expr) == 3L) {
    c(split_top_plus(expr[[2L]]), list(expr[[3L]]))
  } else {
    list(expr)
  }
}
R

The pipe operators supported are %>% (magrittr) and |> (native R 4.1+). The native pipe is rewritten to a sentinel token before parsing to work around the fact that |> desugars differently in R's AST than %>% — see §3.

Key constraint: ptr_translate() rejects any formula that parses to more than one top-level expression (i.e., semicolons or newline-separated statements). The + is the only supported layer separator. Assignment, loops, and multi-statement code are all invalid at the top level.

3 · Formula Parsing Mechanics

The entry point is ptr_translate(formula, expr_check, max_depth, ns_fn, annotate) R/paintr-translate.R:11. It produces a ptr_root AST node holding a flat list of layer nodes. When annotate = TRUE (the default), it also assigns Shiny IDs, classifies data lineage, and resolves shared-placeholder groups before returning.

1Pre-process native pipe — preprocess_native_pipe(formula) replaces every |> token with a sentinel string using R's own parse data (utils::getParseData), so the expression can be re-parsed with rlang::parse_exprs() and the sentinel is distinguishable from %>%.
2Parse to R expression — rlang::parse_exprs(preprocessed). Must yield exactly one expression; zero or multiple → abort.
3Depth guard — check_translate_depth(root_expr, max_depth) walks the raw AST and aborts if nesting exceeds max_depth (default 100), protecting against pathological inputs that would exhaust the stack.
4Split on + — split_top_plus(root_expr) returns a flat list of sub-expressions, one per ggplot layer.
5Translate each layer — lapply(layer_exprs, translate_layer). Each sub-expression becomes a typed AST node (§4, §5).
6Deduplicate layer names — dedupe_layer_names(): if two layers have the same function name (e.g., two geom_point calls), they are renamed geom_point and geom_point-2. This ensures unique Shiny input-ID prefixes.
7Assign active-input IDs — Every layer except ggplot gets an active_input_id set to <name>_checkbox. This is the Shiny input controlling whether the layer is currently enabled (the "toggle layer" checkbox in the sidebar).
8Wrap in ptr_root — ptr_root(layers, expr) creates the root node holding all layers and the original expression for source display.
9Safety check — ptr_validate_tree_safety(root, expr_check) walks every node looking for forbidden patterns (dangerous function calls, direct eval, etc.) in the static formula AST.
10Annotate (when annotate=TRUE) — Three passes: (a) ptr_assign_ids(root, ns_fn), (b) ptr_classify_data(root), (c) ptr_shared_bind(root).

Step 1 in detail: native pipe preprocessing

preprocess_native_pipe() R/paintr-translate.R:175 uses utils::getParseData(parse(text=formula, keep.source=TRUE)) to obtain the token table of the formula string, then searches for tokens with type "PIPE" (the native |>). It replaces each occurrence in-place with the sentinel string .ptr_pipe_native_sentinel (a package-level character constant set to a value that is not a valid user identifier). Replacements are applied in reverse source order (right-to-left, bottom-to-top) so that column offsets remain valid as prior tokens are replaced.

# Input:  "upload |> filter(x > num) |> ggplot(aes(y = var))"
# After:  "upload .ptr_pipe_native_sentinel filter(x > num) .ptr_pipe_native_sentinel ggplot(aes(y = var))"
# The sentinel is a valid symbol name, so rlang::parse_exprs() parses it as
# a binary infix call  `sentinel`(lhs, rhs)  -- structurally identical to %>%.
R

Steps 10a–10c: annotation passes

ptr_assign_ids — R/paintr-ids.R:13

Performs a pre-order rewrite of the tree using ptr_rewrite_pre(). For every placeholder node encountered it calls assign_id_to_placeholder(node, layer_name, path, param) R/paintr-ids.R:35, which sets node$id to:

node$id <- paste0(layer_name, "_", path_str, "_", node$keyword, "_", shared_part)
# e.g.: "ggplot_1_1_var_NA"  for the first var in ggplot(aes(x=var, ...))
#       "ggplot_1_1_var_xcol" for var(shared="xcol") at the same position
R

where path_str is the underscore-joined positional index path from the layer root (e.g., child 1 of child 1 = "1_1"), and shared_part is either the shared key or "NA".

For upload (source) placeholders, if the registry entry has a companion_id_fn, the companion id (ptr_upload_name_id(node$id)) is stored on node$companion_id.

ptr_classify_data — R/paintr-classify.R:5

Propagates upstream data context through the tree using classify_walk(). The context starts as NULL at the root. When a ptr_layer node has a non-NULL data_arg, the data-arg sub-tree becomes the context for that layer's children. The key action: every ptr_ph_data_consumer node receives node$upstream <- ctx_data — a reference to the sub-tree that will produce the upstream data frame at runtime. This upstream reference is later used by ptr_setup_consumer_uis() to determine which reactive sources to depend on when rebuilding the column-picker UI.

ptr_shared_bind — R/paintr-shared.R:5

Collects all placeholder nodes where node$shared is non-NULL, groups them by shared key, and calls canonical_shared_id(key) = paste0("shared_", key) to compute one canonical ID per key. It then rewrites every placeholder in a group to have node$id <- canonical_id. As a result, all occurrences of the same shared key point at the same Shiny input ID — a single widget drives all of them.

4 · Recognizing Function Calls, dplyr Verbs, Operators

ggpaintr does not maintain an explicit list of "dplyr verbs" during parsing. Classification at parse time is structural, not name-based. The distinction between a plain function call (→ ptr_call), a pipeline stage (→ element of ptr_pipeline$stages), and a ggplot layer (→ ptr_layer) is entirely determined by AST position and the presence of a pipe operator.

How translate_node dispatches

translate_node(expr) R/paintr-translate.R:290 is the recursive descent worker:

translate_node <- function(expr) {
  if (is.call(expr)) {
    head <- expr[[1L]]
    if (is_pipe_head(head))         return(translate_pipeline(expr))
    ph <- detect_placeholder(expr)
    if (!is.null(ph))               return(build_placeholder_node(expr, ph))
    if (identical(head, as.name("function")))
                                    return(ptr_closure(...))
    return(translate_call(expr))
  }
  if (is.symbol(expr)) {
    ph <- detect_placeholder(expr)
    if (!is.null(ph))               return(build_placeholder_node(expr, ph))
    return(ptr_literal(expr))
  }
  ptr_literal(expr)
}
R

Crucially, filter(x > num) inside a pipeline and theme(axis.title = element_text(size = num)) inside a ggplot layer are both translated by translate_call() — producing a ptr_call node. The fact that filter is a dplyr verb is irrelevant to the parse-time translation.

Pipe detection

is_pipe_head(x) R/paintr-translate.R:157 returns TRUE when x is a symbol whose character form is "%>%" or the sentinel string. No other operators trigger the pipeline path. This covers [, $, ::, arithmetic, and all other operators — these are translated as ptr_call nodes (generic function calls) with their arguments recursively translated.

The $ and [ operators

R parses df$col as `$`(df, col) and df[i, j] as `[`(df, i, j) — both are call expressions whose head is the operator symbol. Because neither is a pipe-head symbol, translate_node treats them as plain calls via translate_call(). The operator symbol becomes ptr_call$fun; its operands are recursively translated as ptr_call$args. No special-casing is needed.

The dplyr verb list at prune time

The only place where dplyr verb names matter is in empty-call pruning (§6). The function default_drop_when_empty() R/paintr-utils.R:174 returns a character vector of qualified names like "dplyr::filter", "dplyr::mutate", "dplyr::group_by", etc. A call that names one of these and has zero children after pruning is dropped. This list is not consulted during parsing.

Placeholder detection

detect_placeholder(expr) R/paintr-translate.R:358 checks whether the head symbol (or the bare symbol itself for bare keywords like var used without parentheses) is a registered keyword. It calls placeholder_keyword(expr) which checks against ptr_registry_keywords() — the live set of all registered keyword names. It does not hardcode the five built-in names; any custom-registered keyword is automatically recognized.

placeholder_keyword <- function(expr) {
  if (is.symbol(expr)) {
    nm <- as.character(expr)
    if (nm %in% ptr_registry_keywords()) return(nm)
    return(NULL)
  }
  if (is.call(expr) && is.symbol(expr[[1L]])) {
    nm <- as.character(expr[[1L]])
    if (nm %in% ptr_registry_keywords()) return(nm)
  }
  NULL
}
R

This means a call like var(shared = "x") is detected as a placeholder (keyword "var", shared value "x") before translate_call would process it. A bare symbol var (no parens) is also a valid placeholder — the empty-args form.

5 · Recognizing Layers vs Arguments

The distinction between a layer and an argument is decided at two levels: the top-level split on +, and the dispatch inside translate_layer().

Top-level: the + split

After split_top_plus(), each sub-expression becomes exactly one layer. There is no ambiguity at this point: everything at the top level of the + tree is a layer candidate, regardless of what function it calls. So theme(), labs(), and scale_color_brewer() all become layers — ggpaintr's model is structural, not semantic.

Layer-level: translate_layer dispatch

translate_layer(expr) R/paintr-translate.R:117 handles three cases:

  1. Placeholder: if detect_placeholder(expr) is non-NULL, the whole sub-expression is a placeholder layer (rare, but supported).
  2. Piped layer: if the head of the call is a pipe symbol (is_pipe_head(expr[[1L]])), dispatch to translate_piped_layer(). This handles the upload |> filter(...) |> ggplot(...) pattern.
  3. Plain layer: everything else → translate_plain_layer().

The data_arg concept

The key structural differentiation inside a layer is the data argument. extract_call_data_arg(expr) R/paintr-translate.R:246 applies a deliberate heuristic:

  • If the call has a named data = argument → that argument is the data_arg.
  • If the call head is the bare symbol ggplot and the first argument is positional (unnamed) → that first argument is the data_arg.
  • For any other call (geom_*, stat_*, etc.) — positional first arguments are not treated as data. The reason is that geom_*() and stat_*() take mapping as their first positional argument, so a positional first arg would misroute an aes() call into the data pipeline.
# ggplot(iris, aes(x = var))   → data_arg = ptr_literal(iris), children = [aes(...)]
# geom_point(data = df, ...)   → data_arg = ptr_literal(df),   children = [...]
# geom_point(aes(x = var))    → data_arg = NULL,               children = [aes(...)]
R

Piped layer decomposition

translate_piped_layer(expr) R/paintr-translate.R:201 receives the full pipe chain whose terminal call is the ggplot layer head (e.g., ggplot(...)). It:

  1. Calls collect_pipe_stages(expr, op) to walk the left-recursive pipe chain and collect all stage expressions in order.
  2. Separates the terminal stage from the upstream stages.
  3. Translates upstream stages individually via translate_node().
  4. Wraps the upstream stages in a ptr_pipeline node stored as data_arg.
  5. Translates the terminal call's children (excluding the implicit piped-in first argument) and creates the ptr_layer.

The layer's name comes from layer_call_name(terminal) — for a pipe-terminal expression the function looks at the RHS of the last pipe, which is the function being called.

layer_call_name

layer_call_name(expr) R/paintr-translate.R:133 extracts a string name from an expression for use as the layer's display/ID prefix. It handles: bare symbols, pkg::fn / pkg:::fn (returns just fn), parenthesized expressions (fn)(...) (recurses into parens), and pipe chains (recurses into the RHS terminal). For lambda heads or other non-symbol cases it falls back to rlang::expr_text(head), which can produce an unwieldy string — this is a known limitation documented in project memory.

6 · Trimming Empty Calls

Empty-call trimming happens at runtime, not at parse time. The function ptr_prune(node, safe_to_remove, is_standalone) R/paintr-prune.R:17 is called inside ptr_exec_headless() after placeholder substitution has replaced every unset placeholder with a ptr_missing sentinel.

Why it is needed

Consider filter(Sepal.Length > num) in a pipeline, where the user has not yet entered a value for num. After substitution, the call becomes effectively filter(ptr_missing). Evaluating that would produce a runtime error (filter does not know about ptr_missing). Pruning detects the empty state and removes the entire filter call from the pipeline, letting the data pass through unchanged — a natural default behavior.

The prune walk

The S3 dispatch table for prune_walk() covers every node class. Key behaviors:

Node classPrune behavior
ptr_ph_valueAlways returns ptr_missing(). An unsubstituted value placeholder cannot survive to eval.
ptr_ph_data_consumerAlways returns ptr_missing(). Same logic.
ptr_ph_data_sourceAlways returns ptr_missing().
ptr_callPrunes each arg. Drops args that become ptr_missing. Returns the pruned call (even if all args are dropped — a bare call with no args is still a valid call for many functions).
ptr_layerPrunes data_arg and all children. If both are empty/NULL and the layer name is in the remove_set and is_standalone(name) returns FALSE → returns NULL (layer is dropped from the tree).
ptr_pipelinePrunes each stage. Stages that become ptr_missing are dropped. If all stages vanish → ptr_missing(). If only one stage remains → unwrap the pipeline, return that stage directly.

The remove_set

default_drop_when_empty() R/paintr-utils.R:174 returns a union of qualified name lists from ggplot2, dplyr, tidyr, tibble, pillar, purrr, stringr, forcats, lubridate, and hms — covering all common tidyverse functions that should be silently dropped when they contain no resolved arguments. Examples: "dplyr::filter", "dplyr::mutate", "dplyr::group_by", "ggplot2::theme", "ggplot2::labs".

The caller can extend this with the safe_to_remove argument to ptr_app() / ptr_init_state().

The is_standalone predicate

ptr_default_is_standalone(name) R/paintr-prune.R:29 returns TRUE for names starting with "geom_" or "stat_". These layers are kept even when empty — a bare geom_point() with no arguments is a valid ggplot layer that uses inherited aesthetics from ggplot(). Non-standalone layers (e.g., theme() with all args gone) are dropped.

Name matching: bare vs qualified

name_matches_remove_set(bare, qualified, remove_set) R/paintr-prune.R:204 checks both the bare name ("filter") and the qualified name ("dplyr::filter") against the remove set. The qualified name is resolved at prune time via qualified_call_name() R/paintr-utils.R:180, which follows the search path to find the function's home namespace. This means a bare filter call that resolves to dplyr is correctly matched even without an explicit dplyr:: prefix in the formula.

8 · Custom Placeholder Compatibility

A custom placeholder is compatible with ggpaintr if and only if its hooks satisfy the following contracts at each phase of the pipeline.

At parse time (translate)

No special action is required from the author. detect_placeholder() / placeholder_keyword() check against the live registry, so any registered keyword is automatically recognized the next time ptr_translate() runs. The resulting node class is determined by the entry's role field — "value" → ptr_ph_value, "consumer" → ptr_ph_data_consumer, "source" → ptr_ph_data_source. The node carries node$keyword, node$expr, node$shared (NULL or a string), and after annotation, node$id and node$layer_name.

build_ui contract

The hook is called by invoke_build_ui() inside ptr_setup_consumer_uis() / build_ui_for(). It must return a valid shiny.tag or shiny.tag.list object. The inputId of all inputs the hook renders must be exactly node$id. If the hook renders a companion input (like the upload dataset-name field), its id must match node$companion_id, which is computed by companion_id_fn(node$id) and stored on the node during ID assignment.

resolve_expr contract

Called during substitute_walk() with the live value from the Shiny input snapshot and the node. Must return:

  • A language object (R call, symbol, or scalar) to be spliced into the plot expression — wrapped in ptr_literal(resolved) by the caller.
  • NULL to indicate "no value yet" → the slot becomes ptr_missing() and is pruned.
  • For the expr built-in specifically, returning an rlang language object causes the wrapper to emit a ptr_user_expr node instead of ptr_literal, which bypasses the safety denylist. Custom value placeholders should not return untrusted user-typed R expressions unless they perform their own safety validation.

resolve_data contract (source role only)

Called by ptr_setup_pipelines() when a source widget's input changes. Receives the raw Shiny file-input value (a list with name, size, type, datapath fields) and the node. Must return a data frame or NULL. Errors are caught by tryCatch in the observer and surfaced as resolve errors in the error panel.

validate_input contract (consumer role, optional)

Called during substitute_walk.ptr_ph_data_consumer() with the current value and upstream_cols (a character vector of column names from the resolved upstream data). Must return TRUE or NULL for "valid", or a character string error message for "invalid". Any other return type fails closed with a generic error.

ID conventions

The raw input ID on node$id is not namespaced. The server always reads inputs via input[[ns(raw_id)]] (where ns = state$server_ns_fn), and the UI must render the input with inputId = ns_fn(raw_id) (where ns_fn = state$ui_ns_fn). The invoke_build_ui() helper wraps the node's id correctly before calling build_ui, so the hook receives an already-namespaced id via node$id in the UI context. The raw id on the node in the static tree is always un-namespaced.

shared field

The node$shared field is NULL (not shared) or a string (shared key). After ptr_shared_bind(), all nodes sharing the same key have the same node$id. Custom placeholder hooks must not assume node$id is unique; it will be the canonical id paste0("shared_", key) for every node in a shared group.

9 · Constructing the Final Shiny App from the Formula

The entry point for a standalone app is ptr_app(formula, ...) R/paintr-app.R:108, which delegates to ptr_app_components() to build a ui and a server closure, then wraps them in shiny::shinyApp().

9.1 UI construction

ptr_build_app_ui(tree, ...) walks the annotated ptr_root to produce the sidebar layout. The critical inner call is build_ui_for(node, ...), an S3 generic dispatching on node class. For placeholder nodes it calls the registry's build_ui hook; for ptr_layer nodes it emits the layer's collapsible panel containing all its placeholder widgets.

Consumer placeholders (ptr_ph_data_consumer, i.e., var) receive a static shell only at UI build time — their actual column-picker content is rendered by a shiny::renderUI() output registered in the server. This is because the available columns are unknown at UI-build time (the data has not been uploaded yet).

9.2 Server bootstrap — ptr_server()

The server function calls ptr_server(input, output, session, formula, ...) R/paintr-server.R:300, which first calls ptr_init_state(formula, ...) to build a ptr_state object, then registers six groups of observers/outputs:

1ptr_setup_producer_inputs — wraps each value-placeholder input (text, num, expr) in a debounced reactive so rapid typing doesn't trigger expensive upstream-data recomputation on every keystroke.
2ptr_setup_pipelines — for each layer whose data_arg is an upload/source placeholder, registers an observe() that fires when the file input changes, calls entry$resolve_data(), and caches the resulting data frame in state$resolved_data[[layer_name]](df). For pipeline-head sources (not a bare data_arg), binds the resolved frame into state$eval_env under the dataset name.
3ptr_setup_stage_enabled — registers observers that read each layer's checkbox input and update state$stage_enabled().
4ptr_setup_runtime — the main reactive observer. Fires when the Update Plot button or the shared draw-all trigger fires (or when ptr_gg_extra() adds extras). Inside shiny::isolate(), snapshots all input values, resolves upstream column lists, calls ptr_exec_headless(), and stores the result in state$runtime(res).
5ptr_setup_consumer_uis — for each var (consumer) placeholder, registers a renderUI() output. The reactive reads structural dependencies (tree, stage_enabled, resolved data), resolves the upstream data frame, extracts column names, and calls invoke_build_ui() with those columns to produce the picker HTML.
6ptr_register_plot / ptr_register_error / ptr_register_code — bind output$ptr_plot, output$ptr_error, output$ptr_code as standard Shiny render functions reading state$runtime().

9.3 The runtime execution pipeline — ptr_exec_headless()

ptr_exec_headless() R/paintr-headless.R:21 is a pure function (no Shiny reactive dependencies) that takes the tree, input snapshot, eval environment, and stage-enabled map, and returns a result list with ok, plot, code_text, and error slots. Its steps:

Adisable_walk — marks layers inactive according to stage_enabled. prune_walk.ptr_layer returns NULL for inactive layers.
Bptr_substitute — walks the tree and replaces every placeholder node with either a ptr_literal (the resolved value), a ptr_user_expr (for expr placeholders), or ptr_missing() (input empty / NULL). Shared bindings are read from state$shared_bindings (a named list of reactives).
Cptr_prune — removes all ptr_missing nodes and drops empty verb calls from the remove_set (see §6).
Dinject_resolved_data_list — for bare-data-source layers, replaces the (already-pruned) data_arg with a ptr_literal wrapping the actual data frame object. This makes the eval step avoid re-running resolve_data; the code panel still shows the original source expression.
Eptr_assemble_plot_safe — calls ptr_eval(pruned), which converts the pruned tree to R language objects via layer_to_eval_expr() and evaluates each layer, assembling the result with Reduce(`+`, results).
FExtras — if ptr_gg_extra() has added extra ggplot objects, they are appended with Reduce(`+`, extras, res$plot).

9.4 layer_to_eval_expr and desugar_pipe_to_call

layer_to_eval_expr(layer) R/paintr-eval.R:26 converts a ptr_layer to a language object for eval(). If the layer has a piped data_arg, it calls pipeline_to_eval_expr(layer$data_arg) to build the pipeline expression and then desugar_pipe_to_call(upstream, rhs) to convert x |> f(a) into the call form f(x, a) that eval() can execute without the magrittr or native-pipe runtime. This desugaring happens at eval time, not during parsing.

10 · Shared Placeholders

What problem they solve

Without shared placeholders, every occurrence of a keyword in the formula gets its own independent widget. This is correct for most use cases but creates redundancy when the same conceptual value should drive multiple places in one plot (e.g., the same column used for both x and color), or the same value should drive multiple plots in a multi-instance app.

Shared placeholders collapse all occurrences of the same key to a single Shiny input widget. The user sees one control; all formula positions that reference that key track it in lockstep.

Annotation syntax

# Single shared column picker drives both x and color
ptr_app("ggplot(iris, aes(x = var(shared = 'col'), color = var(shared = 'col'))) + geom_point()")

# Shared size slider
ptr_app("ggplot(iris, aes(x = Sepal.Length, y = Sepal.Width)) + geom_point(size = num(shared = 'sz'))")
R

The only supported argument inside a placeholder call is shared = "<key>". Any other positional or named argument causes extract_shared() R/paintr-translate.R:380 to abort with a clear error message.

Parse-time processing

  1. extract_shared(expr, keyword) parses the shared argument from the call. Returns NULL for bare keywords (not shared) or the key string.
  2. The key is stored on the node as node$shared.
  3. After ID assignment, ptr_shared_bind(root) collects all nodes where node$shared is non-NULL, groups by key, and calls canonical_shared_id(key) = paste0("shared_", key). Every node in a group is rewritten to have node$id = canonical_id. For source placeholders, the companion id is also rewritten through companion_id_fn.

Runtime: how shared bindings reach the server

In ptr_make_app_server() R/paintr-app.R:160 (the single-instance path), the server closure builds a named list of reactives, one per shared key:

shared_reactives <- setNames(
  lapply(shared_keys, function(k) {
    canonical <- canonical_shared_id(k)          # "shared_k"
    shiny::reactive(input[[ns(canonical)]])
  }),
  shared_keys
)
R

This list is passed to ptr_server(..., shared = shared_reactives). In ptr_substitute(), when a placeholder node's node$id matches a key in ctx$shared, the value is read from ctx$shared[[key]]() (the reactive) rather than from the snapshot. This keeps the shared input decoupled from the snapshot-based individual inputs.

Single-instance vs multi-instance

The partition rule governs which surface renders the widget:

  • Single instance (ptr_app()): every shared key is formula-local. All shared widgets render inside the formula's inline shared section — a shiny::wellPanel() automatically appended to the sidebar by ptr_controls_panel(). No coordinator object is needed.
  • Multiple instances (ptr_module_server() + ptr_shared()): the coordinator's shared_partition() R/paintr-shared-coordinator.R:27 classifies each key: ≥2 formulas → panel key (rendered in the standalone ptr_shared_panel(obj) with global un-namespaced ids); exactly 1 formula → formula-local key (rendered inline in that formula's module). The ptr_shared_server(obj) returns a ptr_shared_state bundle threaded into each ptr_module_server(..., shared_state = sh).

Consumer placeholders (var) in shared context

A var(shared = "k") consumer does not render a static widget — it renders a column-picker via renderUI(). The ownership rule applies identically: if the key is formula-local, the owning module's ptr_setup_consumer_uis() + ptr_bind_local_shared_consumers() binds the picker. If the key is a panel key (≥2 formulas), the shared-panel server binds it using the representative node from the first formula that declared the consumer.

11 · Multiple ggpaintr Instances in One Shiny Session

When a page embeds two or more independent ggpaintr modules, each with its own formula, the Shiny input IDs for their respective placeholder widgets must not collide. ggpaintr achieves this entirely through Shiny's standard module namespacing mechanism.

The two namespace functions

Each module instance operates with two distinct namespace functions:

  • state$ui_ns_fn = session$ns — the module's namespace function. Applied to every inputId passed to renderUI() and every outputId in uiOutput(). Shiny's moduleServer() does not auto-namespace tag attributes emitted inside renderUI(), so these must be wrapped explicitly.
  • state$server_ns_fn = shiny::NS(NULL) (identity) — used for input[[id]], output[[id]], and updateXxx(session, id, ...) lookups. Under moduleServer(), Shiny's session already applies the module prefix to these lookups automatically, so the server-side namespace function is the identity.

At the top level (inside ptr_app(), no moduleServer()), both functions are shiny::NS(NULL) — the identity. This makes the behavior backward-compatible: all IDs remain as emitted by ptr_assign_ids().

ID generation — ptr_assign_ids

Raw placeholder IDs are generated by assign_id_to_placeholder() R/paintr-ids.R:35 as:

paste0(layer_name, "_", path_str, "_", keyword, "_", shared_or_NA)
# e.g. "ggplot_1_1_var_NA"
R

These raw IDs are layer-scoped, not globally unique. Two different formulas can both have ggplot_1_1_var_NA. Module namespacing makes them globally unique at the Shiny level: module A's input becomes A-ggplot_1_1_var_NA; module B's becomes B-ggplot_1_1_var_NA.

ptr_translate and ns_fn

ptr_translate(formula, ns_fn = shiny::NS(NULL)) accepts an ns_fn argument that is passed to ptr_assign_ids(). In the module path (ptr_module_server()), the pre-flight call to ptr_translate(formula, expr_check = FALSE) R/paintr-app.R:833 uses the default identity function. The actual per-instance tree used at runtime is built fresh inside ptr_server() → ptr_init_state() → ptr_translate() with the module's ns function. So the static tree used for preflight checks has raw ids, and the live tree used for widget rendering has namespaced ids.

Shared panel IDs (cross-formula keys)

Panel keys use global, un-namespaced IDs — shared_<key>. The ptr_shared_panel(obj) UI renders these widgets outside any module namespace. The ptr_shared_server(obj) runs at the top level, never inside a moduleServer(). The modules read these inputs via a reactive passed through the shared_state bundle, where the reactive is simply shiny::reactive(input[[canonical]]) — input here being the top-level server's input object, which sees the global ids directly.

Output IDs: ptr_plot, ptr_error, ptr_code

The output Shiny IDs ptr_plot, ptr_error, and ptr_code are fixed strings registered by ptr_register_plot() et al. via:

output[[state$server_ns_fn("ptr_plot")]] <- shiny::renderPlot(...)
R

Under moduleServer(), server_ns_fn is the identity, so output[["ptr_plot"]] is set — and moduleServer's session auto-prefixes this to A-ptr_plot for module A. The UI counterpart uses shiny::NS(id)("ptr_plot") = "A-ptr_plot" in ptr_ui_plot(id). Both sides agree on the namespaced ID without either needing to know the other's namespace string.

consumer output IDs

The dynamic column-picker output IDs are generated by consumer_output_id(raw_id) R/paintr-ids.R:68 as a function of the raw placeholder ID (typically by appending an _out suffix). The server registers output[[ns(consumer_output_id(raw_id))]]; the UI emits a uiOutput(ui_ns(consumer_output_id(raw_id))). Under module namespacing both sides apply their respective ns functions and remain in agreement.
