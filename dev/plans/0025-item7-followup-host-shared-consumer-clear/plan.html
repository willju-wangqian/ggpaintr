ADR 0025 #7 follow-up — close the shared-vs-single SOURCE asymmetry: port the rising-edge re-render + source_file_reset flag to the host source path

Status: Ready to implement (single-instance reference behaviour + host-path gap runtime-verified by differential AppDriver probes, 2026-05-29). Primary goal: the shared-coordinator host source path lacks the #7 machinery the single-instance source path has. Bring it to parity: clear the stale fileInput pill on the shortcut rising edge, add the host-scope source_file_reset flag, and feed file_reset= into the host resolve_upload_source (env-load gate + vacate-on-empty). Attended side effect: the downstream consumer picker must also clear/suppress correctly once the flag/vacate is in place (§5 S, §4.2). It is a consequence to get right, NOT the headline.

1. Context & baseline (verify before coding)

The asymmetry (this is where the session started — .scratch/ppupload-7-followup-shared-host-rerender.md). #7 made the source shortcut a framework-owned static textInput and re-renders only the source widget (the fileInput) on the shortcut's rising edge (empty→non-empty), clearing the stale filename pill once a typed shortcut takes over. The data side keys off a per-source state$source_file_reset flag (set on that rising edge, cleared on a genuine new upload) so the env-load gate AND vacate-on-empty work without depending on the unreliable server-side input[[src_id]] value. This landed for the single-instance / formula-local source path only:

  • ptr_setup_source_uis (R/paintr-server.R:2277-2426) — the rising-edge re-render (source_bump) + the state$source_file_reset[[raw_id]] flag + its two observers.
  • resolve_upload_source (R/paintr-server.R:1249-1382) — already accepts and honours file_reset (NULLs the file_info → env-load + vacate branches).

The gap. The multi-instance shared-coordinator host source path is a parallel implementation in ptr_setup_panel_sources (R/paintr-shared-ui.R:481-654). Its own renderUI (532-579) has no rising-edge re-render; it never creates/drives a source_file_reset flag; and it calls resolve_upload_source (601-616) without file_reset. So on the host path: typing a shortcut over an uploaded file does NOT visually clear the fileInput pill, and the data-side vacate-on-empty does not fire off the re-render signal. (The data gate's shortcut_active branch still forces the env-load, so a typed name is still used correctly — the asymmetry is the visual pill + the flag-driven vacate, not raw data selection.)

Why the host port is actually simpler than single-instance. In ptr_setup_source_uis the flag must be published on state$source_file_reset and read back in two other functions (the per-instance pipeline binders at R/paintr-server.R:1536,1614). In ptr_setup_panel_sources the renderUI, the flag, the shortcut, AND the resolve_upload_source call all live in the same local({...}) scope (lines 503-651) — so the flag is a local reactiveVal read directly at the resolve call. No cross-function plumbing.

The attended consumer side note (runtime-verified, secondary). While the flag/visual is fixed, the host downstream ppVar(shared=) picker must also clear/suppress correctly. Two differential probes (/tmp/ppupload7-probe shared vs /tmp/ppupload7-single single) showed: the host consumer binder runs at state=NULL (call site R/paintr-shared-ui.R:386), so (i) its identity snapshot is bound-name-blind, and (ii) a fired clear is reverted by the upload's trailing double-render. The vacate this plan adds (P3) makes slot(NULL) fire so the host picker re-renders empty (helps suppress), but the upload-over-typed consumer-clear needs the additional host-binder handling in §5 S. This is the side node — addressed, but downstream of the source fix.

Baseline. Branch ppUpload-bug at fe431c4; #7 single-instance source fix landed (8513dbc). Confirm the full serial gate is 0/0/0 before starting. devtools::check() has 15 PRE-EXISTING failures here (registry-split + pp-toggles, identical on baseline) — not a gate (memory rcmd-check-preexisting-failures).

2. Success Criteria

  1. (P1 — visual, PRIMARY) Pill clears on the host path. In a two-plot shared app: upload a file to #shared_ds, then type a non-empty shortcut into #shared_ds_shortcut → the fileInput no longer displays the uploaded filename (real browser, eyeball — the chromote/CDP harness cannot see the pill; see §7). Same observable as #7 Success Criterion 1, now on the shared host source widget.
  2. (P1) Typing is not disrupted and the clear is asymmetric: picking a file clears the textbox (mutex, kept) but does NOT re-render/wipe the just-uploaded file display (re-render fires on the shortcut rising edge ONLY).
  3. (P2 — flag) source_file_reset exists on the host path, keyed by the canonical source id (shared_ds): set TRUE on the shortcut rising edge, cleared FALSE on a genuine new upload. Verified by an instrumented probe / unit-style assertion on the resolver receiving file_reset=TRUE.
  4. (P3 — data) vacate-on-empty + env-load gate key off the flag. After a shortcut takes over (flag TRUE), clearing the shortcut textbox to empty vacates the host source binding (the picker/plots fall back to "no source"), and the env-load gate ignores the stale lingering fileInput value — both driven by file_reset, not the server-side input[[src_id]]. Verified via AppDriver (data observable).
  5. (S — attended consumer side) host ppVar(shared=) picker does not ride a stale column. Type an env dataset → select a column present in BOTH that dataset and the to-be-uploaded CSV → upload the CSV → the picker selection is cleared (not stale). And clearing the source (P3 vacate) leaves no stale pick.
  6. No regression to the single-instance source path (ptr_setup_source_uis, resolve_upload_source) or its verified clear behaviour. Full serial authoritative gate stays 0/0/0; the palmerpenguins shadow-bind test and the shared-source/consumer e2e cluster stay green.

3. Constraints

  • Mirror #7, do not re-invent. The host pieces must be byte-equivalent in shape to ptr_setup_source_uis's rising-edge block (same 400ms debounce window, same one-directional empty→non-empty edge, same file_reset_rv(FALSE) on a genuine new upload).
  • Rising-edge ONLY. A symmetric boolean dep would re-render on the falling edge (file-pick clears the textbox) and wipe the just-uploaded display — Success Criterion 2.
  • Key the flag by the canonical source id (shared_ds) — the same key panel_sources and the host_state use — NOT a bound_names layer-name key (memory bound-names-key-duality).
  • Keep the data gate intact. Do NOT change resolve_upload_source's logic, the auto-name binding, or the env-load path. P3 only passes the existing file_reset= argument the resolver already supports.
  • Keep the 400ms resolve-path debounce and the file→text mutex. Only ADD the rising-edge re-render + flag; do not remove the mutex (it is the file→text half, still needed).
  • Consumer side (S) is host-scoped. Any consumer-binder change lands in ptr_bind_shared_consumer_uis only; do NOT alter consumer_seed_decision's contract or the non-shared binder ptr_setup_consumer_uis (single-instance is verified-good).

5. Implementation steps

All PRIMARY edits land in the single local({...}) in ptr_setup_panel_sources (R/paintr-shared-ui.R:503-651), mirroring ptr_setup_source_uis.

  1. P1 — rising-edge re-render (pill clear). In the local() scope (after shortcut_input_id at line 527): add source_bump <- shiny::reactiveVal(0L); add source_bump() as a dep on the first line of the renderUI body (line 533). The textbox is already static (panel body emits it), so re-rendering touches only the fileInput.
  2. P2 — host source_file_reset flag. Add file_reset_rv <- shiny::reactiveVal(FALSE) in the same scope. When shortcut_input_id is non-NULL: a debounced (400ms) observeEvent on the shortcut that, on the empty→non-empty edge, bumps source_bump and sets file_reset_rv(TRUE) (tracking prev_shortcut); plus an observeEvent(input[[input_id]], ...) that sets file_reset_rv(FALSE) on a genuine new file pick. Byte-shape-equal to ptr_setup_source_uis lines 2318-2344. (Optionally also publish on a host-state slot keyed by canonical if a future reader needs it; not required since the resolve call is in-scope.)
  3. P3 — feed file_reset= into the host resolve call. At the resolve_upload_source(...) call (601-615) add file_reset = isTRUE(file_reset_rv()). The resolver's existing file_reset branch then NULLs the stale file_info → the env-load gate + the vacate-on-empty (A1) fall-through both fire off the re-render signal. Keep the mutex (645-647).
  4. S — attended consumer clear/suppress (host binder). In ptr_bind_shared_consumer_uis (R/paintr-server.R:2891-3256), make the host-scope (state=NULL) new-source clear survive the upload's trailing double-render: add a per-picker cleared_for_identity latch (next to last_src_identity/src_seen, ~2937) that holds selected=character(0) for an identity epoch until current is observed emptied, and fold a vacate edge (was-supplied → no cols) into the same clear trigger. Optionally make the host identity bound-name-aware via a backward-compatible bound_name_for= param on consumer_upstream_source_state fed from the host panel_owned_binding_name map (improves env→env rename detection). Do NOT touch consumer_seed_decision or the non-shared binder.
  5. Tests. New tests/testthat/test-shared-host-source-reset.R: P3 vacate + env-load gate (AppDriver data assertions), S-N1 consumer clear (overlapping column), S-N2 vacate-suppress, and a W1 single-instance guard. The visual pill (P1, P-N1) is asserted by an eyeball prototype run (harness cannot see it — §7), mirroring how #7 was validated.
  6. Docs. Update .scratch/ppupload-7-followup-shared-host-rerender.md: mark the host source flag + visual reset DONE; note the consumer-clear was handled alongside.

7. Why the harness can't prove P1 (read before claiming done)

The fileInput filename pill is rendered client-side by the browser's native file widget; chromote/CDP get_html/get_value cannot observe whether the pill text is shown (exactly as documented for #7's plan §6). P1 / P-N1 / P-N3 are therefore validated by an eyeball run of a throwaway prototype in a real browser, not by an AppDriver assertion. The flag (P2), the vacate + env-load gate (P3), and the consumer clear (S) ARE harness-observable via data/value assertions and carry the AppDriver regression. Do not claim P1 done off a green harness alone — run it and look (memory feedback-runtime-claim-must-run-app, feedback-visual-before-css-decision).

Independence clause. The integrator does not trust a status block: before treating this plan as done, the integrator independently re-runs the Definition of Done — Gate 3 (devtools::test() serial → 0/0/0) and Gate 0 (eyeball the host fileInput pill in a real browser, since the harness is blind to it) — and sees the expected output first-hand. A relevant-e2e-only pass never grounds a "done" claim.

Out-of-scope discipline (surface, don't absorb). Beyond the static §8 list: any out-of-scope issue discovered while implementing — e.g. the X1 cosmetic-pill facet on a different widget surface, a sibling untouched gate, an adjacent bug — is surfaced as a follow-up scratch note (append to .scratch/ppupload-7-followup-shared-host-rerender.md or a new .scratch/ entry) and NOT silently fixed, widened, or papered over.
