PLAN-02 — Option E: server-side bound_names reactiveVal

Source: ADR 0015 · Depends on: PLAN-01 (this supersedes its Option-A workaround tail) · Parallel group: G2 · Status: draft

Context

PLAN-01 landed the SC-1/SC-2/SC-3/SC-4 architecture from ADR-0015 §2 — drop the visibility-suspend gate, add a req(source_ready) guard inside entry_reactive, add the symmetric resolved_data dep loop to the shared site. During implementation, a second race surfaced in tests/testthat/test-adr12-bug-3a.R that the plan did not anticipate: the autoname observer's shiny::updateTextInput(session, comp_id, value = auto) at R/paintr-server.R:947 emits a client-roundtripped input update; in the next flush, the consumer's entry_reactive can fire reading the new companion text BEFORE the main upload observer's assign(binding_name, df, state$eval_env) has run. R's parent-chain scoping then resolves the substituted symbol (e.g. penguins) against datasets::penguins — wrong frame, wrong columns, polluted upstream cache.

PLAN-01 closed this second race tactically with a polling workaround inside entry_reactive: scan the companion text inputs in upstream_source_companion_ids, probe exists(nm, envir = state$eval_env, inherits = FALSE), and if the binding has not yet landed, shiny::invalidateLater(50); return(NULL). The cache was also bypassed for the consumer path (cache = NULL) so a fire inside the race window could not pollute it. Both are documented in the audit at dev/audit/audit-adr15-autoname-race-20260523-204227.html as Option A; both are workarounds that close the race by waiting it out rather than removing the race signal.

Option E (this plan): expose each source's settled binding name as a server-side shiny::reactiveVal, written by the source observer after the successful assign(). Consumers req() that reactiveVal instead of reading the client-driven companion text and probing eval_env. The signal now points at server-side state, so there is no race window to wait out. Restores state$upstream_cache, removes the magic invalidateLater(50), and makes the consumer read the correct signal — the same shape as the existing state$resolved_data / state$resolved_sources reactiveVals (signal written after the state it describes lands).

One sibling read of the same race shape lives in ptr_bind_shared_consumer_uis() (R/paintr-server.R:1770–1905): its renderUI body reads input[[ns(cmp)]] for each upstream_source_companion_ids entry into a snapshot, then resolves against state$eval_env. It already passes cache = NULL so cache pollution does not apply there, but the wrong-frame leak via R scoping can still produce a transient mis-populated picker. PLAN-02 fixes both call sites.

Governing principle

A reactive signal that describes server-side state MUST be written by the observer that produces that state, AFTER the state has landed. Any downstream req() on that signal must observe the state in its post-write configuration. The consumer-resolution path must never depend on a client-roundtripped input value as a proxy for "has the server-side eval_env binding been installed?" — that question must be asked of a server-side reactiveVal whose write is sequenced after the binding's assign() in the same observer body.

The ordering invariant (assign() before state$bound_names[[key]](name)) is load-bearing: if any future patch flips the order, the race silently returns. PLAN-02 enforces the invariant by encapsulating both writes behind a single private helper bind_source_value() — there is no public path that writes bound_names without first running assign(). Tests pin the helper's behavior and the absence of invalidateLater(50) in entry_reactive.

The BDD Then clauses below are contract. An implementation that lands the new helper but leaves the polling tail in entry_reactive "for safety", or that adds state$bound_names as a plain field that observers write directly (bypassing the helper), violates the principle and must fail the auditor's check.

Success Criteria

  • SC-1 — state$bound_names slot allocated. ptr_init_state() in R/paintr-server.R populates a new state$bound_names slot as a named list of shiny::reactiveVal(NULL), keyed exactly the same as state$resolved_data (by layer_name for bare-data-source layers) UNIONED with the keys of state$resolved_sources (by source_id for pipeline-head sources). Verified by grep -n "state\\\$bound_names\\|bound_names <-" R/paintr-server.R showing the allocation block adjacent to the existing resolved_data / resolved_sources allocation in ptr_init_state (currently lines ≈148–179). The slot is initialised with NULL values via shiny::reactiveVal(NULL) — same construction pattern as resolved_data.
  • SC-2 — bind_source_value() helper exists and is the ONLY writer of state$bound_names. A new private helper in R/paintr-server.R: 
    bind_source_value <- function(state, key, name, df, slot) {
  if (!is.null(df) && !is.null(name)) {
    assign(name, df, envir = state$eval_env)
    state$bound_names[[key]](name)
  }
  slot(df)
}
    Verified by: (a) grep -nE "state\\\$bound_names\\[\\[" R/paintr-server.R showing exactly TWO matches — the read inside ptr_setup_consumer_uis's entry_reactive body and the read inside ptr_bind_shared_consumer_uis's renderUI body (no other writes), and (b) grep -nE "state\\\$bound_names\\[\\[.*]]\\(" R/paintr-server.R showing exactly ONE write (the line inside bind_source_value itself). The helper is the choke point.
  • SC-3 — Both source observers in ptr_setup_pipelines() use the helper. The bare-data-source layer loop (around R/paintr-server.R:801–870) and the pipeline-head data-source loop (around R/paintr-server.R:887–928) BOTH replace the in-line if (!is.null(df) && !is.null(binding_name)) assign(binding_name, df, envir = state$eval_env); slot(df) sequence with a single bind_source_value(state, key, binding_name, df, slot) call. key is ln for the bare-data loop, sid for the pipeline-head loop. Verified by absence of any assign(.*, envir = state\\\$eval_env) hit OUTSIDE bind_source_value() in R/paintr-server.R (grep -nE "assign\\(.*envir = state\\\$eval_env\\)" R/paintr-server.R shows exactly one line — inside the helper).
  • SC-4 — Consumer entry: polling removed, req(bound_names) added, cache restored. Inside ptr_setup_consumer_uis()'s entry_reactive body (currently lines ≈1636–1745):
    • Removed: the entire .binding_pending block (currently lines ≈1689–1700), including the for (cmp in upstream_source_companion_ids) probe loop AND the if (.binding_pending) { shiny::invalidateLater(50); return(NULL) } tail. Verified by grep -n "invalidateLater\\|binding_pending" R/paintr-server.R returning zero matches inside ptr_setup_consumer_uis().
    • Added: immediately after the existing req(source_ready) block (which stays), a NEW guard if (!is.null(source_ready)) shiny::req(state$bound_names[[source_ready$id]]()). Reads from the same source_ready$id the existing block resolves; the bound_names slot key is identical to the resolved_data / resolved_sources key, so the lookup is direct.
    • Restored: the ptr_resolve_upstream call's cache = NULL argument is changed back to cache = state$upstream_cache. Verified by grep -nE "cache = NULL" R/paintr-server.R in ptr_setup_consumer_uis's body returning zero matches.
  • SC-5 — Shared consumer renderUI: same req(bound_names) guard. Inside ptr_bind_shared_consumer_uis()'s renderUI body (currently lines ≈1820–1860), immediately after the for (rv in state$resolved_data) rv() line, add a guard block that walks upstream_source_companion_ids + the upstream's source-self ids to discover which bound_names slots are upstream of this shared consumer, and shiny::req()s each one that exists in names(state$bound_names). Reads state$bound_names via the same key shape as the non-shared path (layer-name for bare-data, source-id for pipeline-head). The advisory branch for "Data source not yet provided" remains unchanged — req() halts before the advisory renders, so a still-pre-bind state still shows the advisory (because the renderUI body halts and the previous output stays in DOM, which on first fire IS the advisory; verified by the BDD scenario below). cache = NULL stays in this site (it never used the cache).
  • SC-6 — Regression fixture exists. Directory tests/testthat/fixtures/vignette-apps/adr15-option-e-race/ contains:
    • app.R — first line is pkgload::load_all(Sys.getenv("GGP_PKG"), quiet = TRUE, helpers = FALSE, attach_testthat = FALSE), then library(shiny), then a single ptr_app(ppUpload |> dplyr::filter(ppVar > ppNum) |> ggplot(aes(ppVar, ppVar)) + geom_point()). This is the exact formula from test-adr12-bug-3a.R that hit the race in PLAN-01's implementation cycle.

The uploaded file's column set is {x, y}; datasets::penguins's is disjoint. Any picker showing a column not in {x, y} proves the race fired.
  • SC-7 — Race-provable assertion. tests/testthat/test-adr15-option-e-race.R contains ONE test_that() block gated on skip_on_cran() + skip_if_not_installed("shinytest2") + skip_if_not_installed("chromote") + the source-root guard (skip_if(!file.exists(test_path("..","..","DESCRIPTION")))).   • Files this plan MUST NOT touch:
    • tests/testthat/test-adr12-bug-3a.R — the diagnostic regression that surfaced the race; SC-8 demands it still passes unchanged. Editing it to weaken assertions would silently restore the polling workaround.
    • tests/testthat/test-adr15-consumer-binding.R — PLAN-01's regression file; out of scope.
    • R/paintr-resolve.R, R/paintr-runtime.R — Option E is purely a UI-binding-and-source-write change; no resolver or runtime changes.
    • NAMESPACE, DESCRIPTION, man/ — no API changes (helper is internal, all touched functions are internal).
    • R/paintr-server.R body of ptr_bind_source_autoname() — Option E does NOT remove the autoname feature. The updateTextInput call stays; consumers just stop trusting it as a sync signal.
  • No scope creep. Do not refactor adjacent code. Do not extract additional helpers beyond bind_source_value(). Do not change the autoname mechanism. Do not touch Option Q (defensive eval-time free-variable check) — it is a separate, optional follow-on that PLAN-02 explicitly does not include.
  • SCOPE NARROWING FORBIDDEN. If SC-4 (remove polling) breaks any test, STOP and escalate. Do NOT re-add invalidateLater(50) "for safety". Do NOT skip the shared-site fix (SC-5). Do NOT keep cache = NULL "for now". The whole point of Option E is that the race is gone — re-introducing any workaround means the implementation does not match the principle.
  • Tree mutability assumption. state$tree() is initialised once in ptr_init_state() and never written after that (verified: grep -nE "state\\\$tree\\(.+\\)" R/paintr-server.R R/paintr-app.R shows only reads). Therefore state$bound_names needs no dynamic teardown — its keys are stable for the lifetime of the ptr_state. If a future change makes state$tree() mutable, bound_names will need parallel teardown alongside resolved_data / resolved_sources; this assumption must be re-verified.
  • Conventions: tidyverse style, 2-space indent. rlang::abort() if introducing errors (none expected). No new exports.
  • Serena / sub-agent constraints: when implementing in a worktree, sub-agents MUST use Edit/Write with absolute paths under the assigned worktree only. Do NOT use mcp__serena__replace_symbol_body from sub-agents — see project memory feedback-serena-project-root-doesnt-follow-worktree.

Proxy traps — do not be fooled (copied from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir(...)' with NO NOT_CRAN=true → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "PASS / SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) runs the check subprocess with NOT_CRAN=true; the e2e file cleanly SKIPs via the source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the new e2e file ERRORing, the guard regressed — harness defect, NOT product failure.
  • Only NOT_CRAN=true / devtools::test() RUNS the browser tests. That alone is the gate.

Independence clause

The implementer's "done" report is a claim, not proof. The auditor (and the post-merge auditor) re-runs the Definition-of-Done command and verifies expected output before any merge. Specifically: the auditor MUST run the gate from a clean devtools::load_all() in the worktree AND MUST verify by grep that:

  • No invalidateLater appears inside ptr_setup_consumer_uis() or ptr_bind_shared_consumer_uis().
  • No .binding_pending variable name appears anywhere in R/paintr-server.R.
  • The cache = ... argument inside entry_reactive's ptr_resolve_upstream call is state$upstream_cache, NOT NULL.
  • Exactly one line in R/paintr-server.R writes to state$bound_names[[...]](...) — the line inside bind_source_value().
  • Both source observers in ptr_setup_pipelines() end with a call to bind_source_value(state, ...) instead of inline assign() + slot().
