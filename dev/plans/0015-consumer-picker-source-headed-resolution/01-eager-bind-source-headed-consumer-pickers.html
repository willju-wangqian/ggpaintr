PLAN-01 — Eager-bind consumer pickers under source-headed upstream

Source: ADR 0015 · Depends on: none · Parallel group: G1 · Status: draft

Context

ADR 0015 locks Option B: a two-site fix that closes a UX defect where consumer pickers nested under a data-source placeholder (ppUpload, ppSample, etc.) silently fail to bind. The non-shared site's renderUI is suspended on visibility (waits for the user to click an inner Controls subtab); the shared site has a complete dep set for pipeline-head sources but is missing state$resolved_data for bare-data-source layers. Both manifest as "picker container in DOM but never populates" after the user uploads a file.

The instrumented runtime probe in ADR §1.5 proved the two mechanisms are distinct: the non-shared widget never fires its renderUI at all (suspended); the shared widget DOES fire but with stale eval_env because no dep latches the upload-completed signal. Both fixes are narrow; both must land together because they address the same UX symptom in symmetric code paths.

Concrete sites to edit (ADR §2.1, §2.2):

  • R/paintr-server.R:1685 — drop the pre-warm visibility-gate inside ptr_setup_consumer_uis(); add a shiny::req() guard inside the entry_reactive; drop the now-redundant subtab dep.
  • R/paintr-server.R:1771 — add one line to ptr_bind_shared_consumer_uis()'s renderUI body: for (rv in state$resolved_data) rv(), symmetric with the existing resolved_sources loop.

Governing principle

ADR 0015 commits the project to a uniform binding contract for consumer pickers: a consumer renderUI must bind at boot regardless of the inner-tab visibility of its host panel, and must remain reactive to every signal that can change its upstream's resolved data. The cache-pollution race that motivated fix-super1's visibility-gate is real but must be closed STRUCTURALLY (via shiny::req() on the appropriate source-ready reactive), NOT via suspension. The shared path must watch BOTH state$resolved_sources (pipeline-head sources) AND state$resolved_data (bare-data-source layers) — asymmetry between the two is the dep-set hole this plan closes.

An implementation that re-introduces a find_nodes(node$upstream, is_ptr_ph_data_source)-style guard, omits the req() guard (leaving the race open), or adds only the resolved_data loop to the shared site without also touching ptr_setup_consumer_uis() violates this principle. The two changes are symmetric halves of one architectural fix; landing only one half leaves either non-shared widgets or shared widgets stuck in the same failure state observed in the probes — just in disjoint user-facing scenarios.

Likewise, the BDD Then clauses below are contract. They name picker ids and DOM substrings that must appear without any subtab navigation. An implementation that lands the source change but writes tests that DO navigate to Controls before asserting (silently restoring the gate-as-precondition) violates the principle and will fail the auditor's check. STOP and escalate if the assertions as written cannot be made to pass; do not narrow them.

Definition of Done

Authoritative gate (copy verbatim from CLAUDE.md §Authoritative gate):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS >= 2355 (baseline 2352 + at least 3 new expect_* from the new test_that blocks; actual delta is the sum of expect_* calls in the three blocks).

Proxy traps — do not be fooled (copied from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir(...)' with NO NOT_CRAN=true → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "PASS / SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) runs the check subprocess with NOT_CRAN=true; the e2e file cleanly SKIPs via the source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the new e2e file ERRORing (pkgload "DESCRIPTION not found"), the guard regressed — harness defect, NOT product failure.
  • Only NOT_CRAN=true / devtools::test() RUNS the browser tests (source tree present → 0 SKIP). That alone is the gate.
  • e2e flake (intermittent, chromote wait_for_js timeout) is documented; retry the file in isolation before filing.
