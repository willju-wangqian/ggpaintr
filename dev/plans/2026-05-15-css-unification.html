Plan: unify CSS & asset injection across raw-Shiny entry points

Date: 2026-05-15 · Branch hint: feat/css-unification · Audit reference: dev/audit/2026-05-15-css-unification.html

One-line summary. Collapse five different hand-rolled asset bundles into a single internal helper ptr_assets(css = NULL); add the css = override portal to the one raw-Shiny entry point that lacks it (ptr_shared_ui); document the fifteen --ptr-* custom properties in a shared roxygen topic ?ptr_css. ptr_app_bslib is intentionally untouched — its substrate is different and the bslib theme story is deferred.

1. Problem statement

The audit at dev/audit/2026-05-15-css-unification.html §2 surfaced two concrete asset-side gaps:

  • ptr_shared_ui() self-wraps in .ptr-app but injects none of ptr_layer_assets() / ptr_ui_assets() / ptr_user_css_assets(). It renders correctly today only because it always shares a page with ptr_module_ui(); mounted standalone it would lose the disabled-stage cue, lose ggpaintr.css, and have no override portal.
  • ptr_outputs_ui() injects ptr_ui_assets() + ptr_user_css_assets() but skips ptr_layer_assets(). Latent foot-gun if mounted without ptr_controls_ui() on the same page — the class-toggle JS handler for .ptr-stage-disabled goes missing.

Both gaps share a root cause: each entry point hand-assembles its own asset bundle. Adding or changing an asset means editing every entry point, and drift between them is inevitable. This is the asset-injection analogue of the DOM-duplication problem that cefc9bf solved by introducing controllable_region().

3. Target architecture

3.1 The bundle

# R/paintr-build-ui.R

# Single asset bundle used by every raw-Shiny entry point. Emits, in order:
#   1. ptr_layer_assets() -- inline structural CSS (.ptr-stage-*) + the
#      ptr_set_class custom-message JS handler. Both must ride along inside
#      module embeddings because they are not file-scoped.
#   2. ptr_ui_assets()    -- <link> to ggpaintr.css (the cosmetic theme,
#      scoped under .ptr-app) + the code-mini-window JS.
#   3. ptr_user_css_assets(css) -- user override stylesheets, linked *after*
#      ggpaintr.css so equal-specificity rules win.
#
# Idempotent: every component is safe to emit more than once on a page
# (script handlers self-register; <link> tags dedupe at the browser;
# addResourcePath is idempotent for the same prefix). ptr_module_ui()
# nests ptr_controls_ui() + ptr_outputs_ui(), which together emit this
# bundle three times -- that is fine by construction.
#
# Not exported: ggpaintr users never need to inject assets manually.
# Note: ptr_app_bslib() deliberately bypasses this bundle (it must NOT
# link ggpaintr.css because its rules are gated on .ptr-app, which the
# bslib page chrome does not provide). The bslib path calls
# ptr_layer_assets() directly for structural stage CSS only.
ptr_assets <- function(css = NULL) {
  shiny::tagList(
    ptr_layer_assets(),
    ptr_ui_assets(),
    ptr_user_css_assets(css)
  )
}

3.4 ?ptr_css shared topic

New file: R/paintr-css-doc.R. Single roxygen block, no R object — a documentation-only topic.

#' CSS customisation hooks for ggpaintr apps
#'
#' Every raw-Shiny entry point (`[ptr_app()]`, `[ptr_app_grid()]`,
#' `[ptr_module_ui()]`, `[ptr_controls_ui()]`, `[ptr_outputs_ui()]`,
#' `[ptr_shared_ui()]`) accepts a `css =` argument: a character vector of
#' paths to `.css` files. Each path's parent directory is registered as a
#' Shiny resource under a hash-derived prefix, and one `<link>` tag is
#' emitted per file. User stylesheets are linked *after* `ggpaintr`'s
#' bundled stylesheet, so rules at equal specificity win.
#'
#' The bundled stylesheet exposes its theme as CSS custom properties
#' under the `.ptr-app` selector. Override any subset of these in a
#' user-supplied stylesheet to retheme without touching individual rules.
#'
#' @section Tokens:
#' \describe{
#'   \item{`--ptr-bg`}{App background. Default `#fafbfc`.}
#'   \item{`--ptr-surface`}{Card / sidebar background. Default `#ffffff`.}
#'   \item{`--ptr-border`}{Standard border colour. Default `#e7e9ee`.}
#'   \item{`--ptr-ink`}{Body text colour. Default `#1f2530`.}
#'   \item{`--ptr-muted`}{Muted / secondary text. Default `#6b7280`.}
#'   \item{`--ptr-accent`}{Primary accent (action button bg). Default `#c1372c`.}
#'   \item{`--ptr-accent-strong`}{Hover / active accent. Default `#9a2b22`.}
#'   \item{`--ptr-accent-weak`}{Faint accent tint. Default `rgba(193,55,44,0.13)`.}
#'   \item{`--ptr-accent-line`}{Accent border. Default `rgba(193,55,44,0.22)`.}
#'   \item{`--ptr-radius`}{Large radius (cards, sidebar). Default `10px`.}
#'   \item{`--ptr-radius-sm`}{Small radius (buttons, controls). Default `7px`.}
#'   \item{`--ptr-shadow`}{Card shadow. Default two-layer drop.}
#'   \item{`--ptr-shadow-lg`}{Floating-window shadow. Default two-layer drop.}
#'   \item{`--ptr-font`}{UI font stack. Default system-ui stack.}
#'   \item{`--ptr-mono`}{Mono font stack. Default system mono stack.}
#' }
#'
#' @section Specificity:
#' Some rules use a doubly-scoped selector (e.g. `.ptr-app .btn.action-button`).
#' To override these, match or exceed that specificity in your stylesheet
#' (either with the same selector, or with `!important`).
#'
#' @examples
#' \dontrun{
#'   # accent.css:
#'   #   .ptr-app { --ptr-accent: #2563eb; --ptr-accent-strong: #1e40af; }
#'   ptr_app("ggplot(mtcars, aes(var, var)) + geom_point()",
#'           css = "accent.css")
#' }
#'
#' @name ptr_css
#' @keywords internal
NULL

Every entry-point man page gains @seealso [ptr_css()].

5.1 Helper

render_ui <- function(ui) {
  # Render the tagList to a single HTML string for substring assertions.
  as.character(htmltools::renderTags(ui)$html)
}

linked_csses <- function(html) {
  # Pull every <link rel="stylesheet"> href from the rendered HTML.
  m <- regmatches(
    html,
    gregexpr('<link[^>]*href="([^"]+\\.css)"', html, perl = TRUE)
  )[[1L]]
  sub('.*href="([^"]+\\.css)".*', "\\1", m)
}

6. Acceptance criteria

  1. Wrap invariant. Test "every raw-Shiny entry point self-wraps in .ptr-app" passes.
  2. Override invariant. Test "every raw-Shiny entry point accepts css = path" and "user css links after ggpaintr.css" pass.
  3. Visual-cue invariant. Single-bundle invariant verified by "ptr_assets() emits assets in the documented order"; manual eyeball confirms the disabled cue still renders inside ptr_app.
  4. Documentation invariant. ?ptr_css renders; every entry-point man page contains a See also: ptr_css() line. Verified by devtools::check() emitting no missing-docs NOTE.
  5. Full suite green. devtools::test() passes; devtools::check() reports 0 errors / 0 warnings.

7. Risks & mitigations

RiskLikelihoodMitigation
Triple-injection of the bundle inside ptr_module_ui emits the same addResourcePath + <link> three times, producing a noisy <head>. certain Already tolerated today; addResourcePath is idempotent, browsers dedupe identical <link>, and the JS handler self-guards. Cost is HTML size. Acceptable.
A test elsewhere counts <link> tags and would now see a different count. low Pre-step-2 grep (§5.3). Update if found.
User-css ordering invariant accidentally regressed by a refactor that swaps args inside ptr_assets(). low Test "ptr_assets() emits assets in the documented order" locks it.
ptr_shared_ui mounted as a standalone fragment (post-step-3) now injects its own bundle and the grid page also injects the bundle, doubling the user-css link on the grid path. medium Per step 3 note: ptr_app_grid_components's nested ptr_shared_ui() call leaves css = NULL. Audit comment captures this. Verified by test (the grid path test should see exactly one user-css link, not two).
bslib path silently regresses because someone in the future routes it through ptr_assets() "for consistency". low Inline comment on ptr_assets() + on ptr_app_bslib documents that the bypass is deliberate. Audit §10 records the deferral.
