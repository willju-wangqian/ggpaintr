ggpaintr — shared placeholder UI for multi-instance embedding

Drafted 2026-05-14 on branch new-feature/shared-between-modules · output of a /grill-me design session · status: plan locked, implementation pending

Closes the third supported embedding mode: multiple ptr_module_ui() / ptr_module_server() pairs in one Shiny session. Shared placeholders work in ptr_app() and ptr_app_grid() today but have no widget UI in the split / multi-instance path.

Motivation

Shared placeholders today work inside ptr_app() (single instance) and ptr_app_grid() (N plots, one host server). The third supported use case — multiple ptr_module_ui() / ptr_module_server() pairs embedded in a user-authored Shiny app — has no place to render the shared widget panel. ptr_controls_ui() (split mode) calls ptr_controls_panel(..., render_shared_section = FALSE) at R/paintr-app.R:379, so any formula using shared = "..." in the embedded path silently loses its widget. This plan closes the gap with a new ptr_shared_ui() / ptr_shared_server() pair and recomposes ptr_module_ui() accordingly.

Success criteria

  • A user can write a Shiny app with two or more ptr_module_ui() instances that reference the same shared = "<key>", plus one ptr_shared_ui(formulas) and one ptr_shared_server(formulas) in the top-level server, and changing the shared widget updates every instance that consumes it.
  • The same pattern works with shared var() (data-consumer) and shared value placeholders (text, num, expr, upload).
  • When length(formulas) >= 2, ptr_shared_ui() renders a "Draw all" button that triggers every module's draw. When length(formulas) < 2, no button is rendered.
  • ptr_module_server(id, formula) aborts with a clear rlang::abort() message when the formula contains shared = "..." but shared_state = NULL.
  • ptr_module_server() emits a cli::cli_warn() once when a formula's shared keys are not all covered by names(shared_state$shared).
  • ptr_app_grid() public signature is unchanged; manual screenshots match pre-migration grid output pixel-for-pixel (or within accepted screenshot diff tolerance).
  • ptr_app(), ptr_app_components(), ptr_app_bslib(), ptr_make_app_server() are not modified. Existing tests pass without edit.
  • New testthat files: test-shared-ui.R, test-shared-server.R, test-module-server-shared.R. New manual demo: tests/manual/shared-multi-instance.R.
  • devtools::check() returns 0 errors, 0 warnings.

Constraints

  • Do not touch ptr_app() or its private call path (ptr_app_components, ptr_make_app_server, ptr_controls_panel's render_shared_section = TRUE branch). Two implementations of "render shared widgets" will coexist by design — the sidebar-embedded path stays as ptr_app()'s private mechanism; ptr_shared_ui() serves everyone else.
  • ptr_shared_ui() renders at NS(NULL). Shared inputIds remain flat (shared_<key>), matching today's canonical_shared_id(). One ptr_shared_ui() per page — no namespacing parameter.
  • ptr_shared_server() must read its session via shiny::getDefaultReactiveDomain(); no explicit input / output / session args. Abort if called outside a reactive domain.
  • ptr_shared_state is an S3 class with three public fields: shared, draw_trigger, shared_resolutions. ptr_module_server() validates inherits(x, "ptr_shared_state") via assertthat::assert_that().
  • ptr_shared_ui() and ptr_shared_server() both error when no formula contains a shared placeholder. Calling them is a declaration of intent.
  • The shared-consumer var() rendering uses ptr_bind_shared_consumer_uis() exactly as ptr_app_grid() does today; errors_rv routes to the new output$ptr_shared_errors uiOutput inside the panel.
  • Public inputIds owned by ptr_shared_ui(): ptr_shared_draw_all (button), ptr_shared_errors (output). The grid's current ptr_grid_draw_all becomes dead after PR-B and is removed.
  • ptr_module_ui() must remain wire-compatible with today: same DOM, same inputIds, same outputIds. Its rewrite is an internal composition wrapped in the existing sidebarLayout shell — observable behavior unchanged.
  • ptr_app_grid() keeps every public arg in its current signature; shared_ui = list(...) keeps its function(id) -> shiny.tag contract.

API surface

ptr_shared_ui(formulas, shared_ui = list(), ui_text = NULL,
              expr_check = TRUE, draw_all_label = "Draw all")

ptr_shared_server(formulas, envir = parent.frame(), expr_check = TRUE,
                  shared = list(), draw_trigger = NULL) # -> ptr_shared_state

ptr_module_server(id, formula, envir = parent.frame(), ...,
                  shared_state = NULL)

ptr_shared_state fields (public contract):

  • shared — named list of reactive()s, one per shared key, reading input[[<key>]].
  • draw_trigger — reactive on input$ptr_shared_draw_all, or NULL when length(formulas) < 2.
  • shared_resolutions — ptr_resolve_shared_consumers(trees) output, threaded into each ptr_module_server.

Override semantics on ptr_shared_server(): named entries in shared = list(...) replace the auto-derived reactive for that key; passing draw_trigger = my_reactive replaces the auto-derived trigger. Escape hatch for surgical control: skip shared_state entirely and pass shared = / draw_trigger = / shared_resolutions = directly to ptr_module_server() via ... — those args stay public.

DOM contract

ptr_shared_ui() returns a single wellPanel fragment, not a layout shell. The embedder owns the page layout and decides where the panel sits relative to the ptr_module_ui() instances. 

shiny::wellPanel(
  shiny::div(
    class = "ptr-shared-panel",
    shiny::tags$p(class = "ptr-shared-panel__title", "Shared controls"),
    shiny::tags$p(class = "ptr-shared-panel__hint",
                  "These widgets are linked across every plot below."),
    # one widget per shared key, in formula-discovery order
    <auto-built widget for key A>,
    <embedder-supplied builder output for key B>,
    ...,
    # only when length(formulas) >= 2:
    shiny::actionButton("ptr_shared_draw_all", draw_all_label),
    # always:
    shiny::uiOutput("ptr_shared_errors")
  )
)

DOM is identical to ptr_app_grid()'s current inline shared panel (R/paintr-app.R:588–600) plus the new error uiOutput. The asymmetry with ptr_module_ui() is deliberate:

FunctionReturnsWhy
ptr_module_ui() Full sidebarLayout shell wrapping controls + outputs. A single ggpaintr instance is self-contained chrome ("here is one plot, all in one box"). Embedders drop it into their page as a finished unit.
ptr_shared_ui() One bare wellPanel fragment. Page-level chrome. The embedder composes their own layout around N modules; ptr_shared_ui() is a tile they place — typically full-width above a fluidRow of modules, but free to live anywhere.

Implementation phases

PR-A — feature new

  1. Add R/paintr-shared-ui.R (or fold into R/paintr-app.R): ptr_shared_ui(), ptr_shared_server(), ptr_shared_state constructor + validator, print.ptr_shared_state method.
  2. Lift the embedder-builder branching (embedder_keys / auto_keys) out of ptr_app_grid_components() into ptr_shared_ui() internals. Leave grid using its inline path for now.
  3. Add shared_state arg to ptr_module_server(). On entry: validate, unpack into the existing shared / draw_trigger / shared_resolutions slots, abort on missing-state-with-shared-formula, warn on key-mismatch.
  4. Rewrite ptr_module_ui() to compose ptr_controls_ui() + ptr_outputs_ui() wrapped in the existing sidebarLayout shell. DOM and inputIds unchanged.
  5. Tests: test-shared-ui.R, test-shared-server.R, test-module-server-shared.R.
  6. Manual: tests/manual/shared-multi-instance.R — two ptr_module_ui + one ptr_shared_ui, exercising shared var + shared text.
  7. Docs: roxygen for the three new functions; cross-link from ptr_module_ui / ptr_module_server / ptr_controls_ui / ptr_outputs_ui @seealso blocks. Add a short vignette section under the embedding doc.

PR-B — grid migration refactor

  1. Rewrite ptr_app_grid_components() to call ptr_shared_ui(formulas = plots, shared_ui = shared_ui, ...) and ptr_shared_server(formulas = plots, envir = envir, expr_check = expr_check). Drop the inline shared panel, inline ptr_resolve_shared_consumers() / representative_nodes block, and inline ptr_bind_shared_consumer_uis() call.
  2. Pass each plot's shared_state to ptr_module_server().
  3. Remove ptr_grid_draw_all references; the new button id is ptr_shared_draw_all.
  4. Acceptance: existing ptr_app_grid() automated tests pass without edit; manual screenshots match pre-migration.

Risk register

RiskDescriptionMitigation
DOM drift in ptr_module_ui rewrite Today the function nests inside sidebarLayout(sidebarPanel(...), mainPanel(...)). A naïve composition could emit a flat tagList. Keep the sidebarLayout wrapper in the rewrite. DOM is preserved. Verified by a snapshot test on the rendered tag tree.
Asset double-injection Both ptr_controls_ui() and ptr_outputs_ui() inject ptr_ui_assets(). Composing them inside ptr_module_ui() produces two copies in the tag tree. Verify with htmltools::renderTags() snapshot in the manual test. Assets are idempotent (same href), so worst case is a cosmetic duplicate in the DOM.
Grid screenshot drift PR-B changes button id from ptr_grid_draw_all to ptr_shared_draw_all. Any external Selenium test referencing the old id breaks. Grep dev/scripts/ and tests/ for ptr_grid_draw_all before merging PR-B; update all hits in the same commit.
Override merge semantics User-supplied shared = list(x = my_reactive) must merge over auto-derived defaults rather than replacing the whole list. Test the merge explicitly with one overridden key and one auto-derived key in the same call.

Out of scope

  • Migrating ptr_app() to use ptr_shared_ui(). Tracked as a follow-up; involves moving the sidebar-embedded shared section to a top-level wellPanel, which is a visual change to the default app.
  • Multiple ptr_shared_ui() blocks on one page. Flat shared-key namespace forbids this today; revisit only if a real use case emerges.
  • Session-scoped error sink other than the ptr_shared_errors panel.
