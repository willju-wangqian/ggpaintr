ADR 0025 Item #7 — fileInput visual reset via static-textInput + rising-edge re-render

Status: Ready to implement (design validated by throwaway prototype, real browser, 2026-05-29). Scope: the ONE remaining deferred item of the ppUpload shortcut/upload campaign — make the fileInput's visual state clear when the shortcut takes over. Cosmetic only; the data gate is already correct and untouched.

1. Context & baseline (verify before coding)

  • The bug. After uploading a file then typing a non-empty shortcut, the data is correct (gate at resolve_upload_source ignores the lingering file and env-loads the typed name) but the fileInput widget still visibly shows the uploaded filename. ADR 0025 §2 (Q3-B mutex) says typing the shortcut should clear the sibling fileInput; the lingering display is a real deviation. The existing JS handler ptr_reset_file_input (inst/www/ggpaintr-layer.js:19-31) is buggy/ineffective and, by design, only works for fileInput — not for custom source widgets.
  • Validated design (prototype approach "a"). The shortcut textInput becomes framework-owned and static; only the source widget (fileInput for ppUpload) lives in a uiOutput that is re-rendered on a rising-edge "shortcut became non-empty" trigger. Re-rendering produces a fresh same-id fileInput with no filename baked in → the pill clears; the static textInput is never re-rendered → typing is undisturbed. Prototype + verdict: .scratch/ppupload-7-fileinput-reset-prototype/ (all three behaviors eyeballed YES in a real browser).
  • Baseline state. This plan assumes the ADR-0025 test-migration commit has landed (the 10 stale/regression failures resolved, including the super-2a app.R revert and the strengthened source-default boot oracle) and the full serial suite is 0/0/0. Confirm git log shows that commit and re-run the gate before starting, so #7's effects are attributable.
  • §7-A2 debounce analysis (already done). The 400ms shortcut debounce (R/paintr-server.R:1489-1494, 1565-1569) serves TWO roles: (i) feeding resolve_upload_source — keystroke-churn / "object not found" flicker reduction, independent of #7, must stay; (ii) gating the JS file-reset via shortcut_r passed into ptr_bind_source_mutex — the A2 shadow-bind race fix. #7 removes only role (ii). The same-id re-render never blanks input[[src_id]], so the A2 race cannot occur.

2. Success Criteria

  1. After uploading a file then typing a non-empty shortcut, the fileInput no longer displays the uploaded filename in a real browser (verified by eyeball — the chromote/CDP harness cannot see this; see §6).
  2. Typing the shortcut is not disrupted (no cursor jump / focus loss): the textInput is static, never inside the re-rendered uiOutput.
  3. Asymmetric: picking a file clears the textbox (kept updateTextInput) but does NOT re-render/wipe the just-uploaded file display (re-render fires on the shortcut rising edge only).
  4. The data stays the env-loaded frame (the resolve_upload_source gate remains the source of truth — unchanged).
  5. The behaviour generalises: a custom source registered with shortcut = TRUE gets the framework shortcut textInput + the re-render-on-shortcut clear for free (no fileInput-specific JS).
  6. The buggy ptr_reset_file_input JS handler is removed (not left dead).
  7. Full serial authoritative gate stays 0/0/0; the palmerpenguins shadow-bind test stays green.

3. Constraints

  • Cosmetic only. Do NOT change resolve_upload_source's gate, the auto-name binding, or the env-load path. The resolver stays the data source of truth.
  • Do not regress the F3/caveat-2/§3b fixes from commit 332f7b7, nor the super-2a app.R revert / source-default oracle from the test-migration commit.
  • Keep the 400ms resolve-path debounce. Remove only the JS reset + the text→file mutex half + the shortcut_r→mutex threading.
  • The shortcut textInput must be emitted as static UI in build_ui_for.ptr_ph_data_source (gated on entry$shortcut), never inside the source renderUI body. build_ui returns ONLY the source widget.
  • The re-render trigger must be rising-edge only (shortcut empty→non-empty). A symmetric boolean dep would re-render on the falling edge (file-pick clears the textbox) and wipe the just-uploaded file display — Success Criterion 3.
  • Do NOT rely on the chromote harness to "prove" the visual (CDP sets files invisibly to page JS). A real-browser eyeball is mandatory.
  • Prefer fixing/removing ptr_reset_file_input over adding a second mechanism. Do NOT add shinyjs as a dependency without asking.
  • R conventions: ptr_ prefix for exports only; rlang::abort/cli::*; run devtools::document() if any roxygen changes (none expected — internal functions).

4. Implementation steps (the 5 changes)

  1. R/paintr-builtins.R — ptr_builtin_upload_build_ui (≈229-251): return ONLY the fileInput (+ its attach_help). Remove the shortcut textInput from this hook. build_ui now produces just the source widget — the uniform contract for all sources.
  2. R/paintr-build-ui.R — build_ui_for.ptr_ph_data_source (≈73-88): when ptr_registry_lookup(node$keyword)$shortcut is TRUE, emit tagList(uiOutput(source_output_id(node$id)), textInput(ns_fn(node$shortcut_id), label/placeholder from name_copy, value = node$default %||% "")); else emit the bare uiOutput(...). The textInput is static here, never re-rendered. Resolve name_copy via the existing ui_text path (this hook already receives ui_text).
  3. R/paintr-server.R — ptr_setup_source_uis (≈2263-2366): the <id>_ui renderUI now renders ONLY build_ui(node) (the source widget), not the tagList. Add a per-source rising-edge re-render: a reactiveVal bumped by an observer on the (debounced) shortcut value when it transitions empty→non-empty, used as a renderUI dependency. Keep the existing boot-seed / has_rendered latch and outputOptions(suspendWhenHidden=FALSE).
  4. R/paintr-server.R — ptr_bind_source_mutex (≈1594-1645) + call sites (≈1519, 1587): reduce the mutex to the file→text direction only (updateTextInput(value="") on file pick — generic, kept). Delete the text→file half (the session$sendCustomMessage("ptr_reset_file_input", ...) observer) and the shortcut_r parameter. The rising-edge re-render (step 3) replaces it. Stop passing shortcut_r = at both call sites.
  5. inst/www/ggpaintr-layer.js (19-31): remove the now-dead ptr_reset_file_input custom-message handler.

5. BDD scenarios

Feature: fileInput visual reset when the shortcut takes over (ADR 0025 §2 Q3-B, item #7)

  Background:
    Given a ppUpload source rendered with a fileInput and a static shortcut textInput
    And the resolve_upload_source gate is unchanged (data correctness independent of this feature)

  Scenario: typing a shortcut clears the lingering fileInput display
    Given the user has uploaded "mt.csv" and the fileInput shows "mt.csv"
    When the user types a non-empty name into the shortcut textbox
    Then the source uiOutput re-renders a fresh fileInput
    And the displayed filename "mt.csv" is cleared
    And the data resolves to the env-loaded frame (not the upload)

  Scenario: typing is not disrupted by the re-render
    Given the shortcut textbox is empty
    When the user types characters into it
    Then the textInput retains focus and the typed value (it is static, never re-rendered)
    And the rising-edge re-render fires at most once (on the empty -> non-empty transition)

  Scenario: picking a file does NOT wipe its own just-uploaded display (asymmetric)
    Given the shortcut textbox holds a value
    When the user picks a file
    Then the shortcut textbox is cleared via updateTextInput
    And the fileInput continues to display the just-picked filename
    And no rising-edge re-render fires (the shortcut went non-empty -> empty, not empty -> non-empty)

  Scenario: a custom shortcut source inherits the clear with no fileInput-specific JS
    Given a custom source registered with shortcut = TRUE and a build_ui returning a non-file widget
    When the user types into its framework-provided shortcut textbox
    Then its source uiOutput re-renders to the widget's default state
    And no ptr_reset_file_input JS is involved

  Scenario: the palmerpenguins shadow-bind does not regress
    Given a single-instance ppUpload(shared='ds') |> ggplot(...) app on a search path carrying palmerpenguins::penguins
    When a file is uploaded (mutex clears the shortcut to "")
    Then resolve binds the uploaded frame under the canonical auto-name
    And no env-load of "penguins" occurs (input[[src_id]] is never blanked by a JS reset)

Proxy traps:

  • A raw testthat::test_dir()/test_file() with no NOT_CRAN=true → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate. Use devtools::test().

  • Registry env splits under raw test_dir (custom placeholders invisible to translate); devtools::test() heals it.
  • A residual flake under parallel load is not a product failure — re-confirm serially.
