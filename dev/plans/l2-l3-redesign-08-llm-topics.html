Step 08 — LLM topics rewrite (pass #LLM)

Phase 7 (parallel with Step 07). Large, separate. Target: inst/llm/topics/*, inst/llm/primer.md. Authority: ADR 0005 · CONTEXT.md.

Goal

Rewrite the LLM-facing topic corpus so generated guidance matches the shipped API. ptr_llm_* function signatures do not change; only emitted content.

Files

  • inst/llm/topics/level2_*.md, level3_*.md — re-derive from CONTEXT.md L2/L3.
  • level2_custom_ids.md — id table: keep ptr_plot/ptr_error/ptr_code; reflect bare pieces + combinators; the canonical moduleServer(id) ns story.
  • level3_custom_render.md — now the primary L3 render story (was L2).
  • formula_syntax.md, overview.md — shared section vs panel wording per the partition rule.
  • primer.md — top-level model summary.

Success criteria

  • No topic mentions a removed symbol (ptr_controls_ui, ptr_outputs_ui, ptr_ui_code_toggle, ptr_shared_ui) or the ptr_ui_plot flags or state$server_ns_fn as a user tool.
  • Topics state the convention (ptr_<x> L2 self-contained / ptr_ui_<x> L3 bare), the partition rule, single-vs-multi-instance, and the combinator recipe.
  • Custom render is presented as L3 with the safe moduleServer pattern; no headless/testServer promise.
  • ptr_llm_topics() enumerates the updated set with no dangling topic id; the corpus regenerates via the documented build with no error.

Constraints

  • Content-only; do not change ptr_llm_* code/signatures.
  • Stay consistent with Step 07 (vignette) — same examples, same names; these two must not diverge.
  • Do not introduce a "headless" or "extract-accessor level" framing.
  • Run after Steps 01–06 land (topics must describe shipped behavior).
