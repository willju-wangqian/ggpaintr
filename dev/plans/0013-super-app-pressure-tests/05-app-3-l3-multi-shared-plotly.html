PLAN-05 — Super-app 3: L3 multi-cell + shared + plotly

Source: ADR 0013 §"App 3 — L3 multi-cell + shared + plotly" + D1/D4/D5/D6 (G4+G6 row) + L1/L3/L4 (multi-instance & layout) · Depends on: 01 · Parallel group: G2 · Status: draft

Context

ADR 0013 §App-3 locks the slot picks: hand-rolled L3 (ptr_ui_page + ptr_shared_panel + 2× ptr_server) · per-cell mtcars literal · G4 pre-quoted via local rlang::expr var (PLUS G6 forwarded-symbol resolution) · custom plotly output on plot2 · default theme. One custom value placeholder ppRange registers locally (same shape as App-1's ppRange — name reuse is safe per ADR D5 because each fixture's child process is independent, project memory shinytest2-appdir-pkgload).

The shared coordinator carries one key "linked" that reaches plot1 (default ggplot output) and plot2 (plotly output) — the cross-output-backend propagation test that App-1 and App-2a cannot give. Shared widget id is "shared_linked" (canonical-shared-id, R/paintr-app.R:1322).

Important constraint from the ADR. ADR §App-3 documents that ptr_app_grid (R/paintr-app.R:1271) delegates to ptr_app_grid_components (R/paintr-app.R:1296) which asserts all(vapply(plots, rlang::is_string, ...)) at R/paintr-app.R:1309–1313 — grid takes string formulas only. To exercise G4 (pre-quoted via rlang::expr) alongside a custom plot output, App 3 hand-rolls the L3 surface instead of using ptr_app_grid. ptr_app_grid's pressure surface is covered by narrow fixtures (app-grid-shared-added, grid-shared-partition) and is out of scope here.

Unique pressure axes (ADR §App-3): (i) G4 + G6 capture — both formulas live in local rlang::expr(...) vars; ptr_server(formula_a, "plot1") uses enexpr capture; the captured symbol is then resolved via sys.frames fallback (G6); (ii) shared coordinator across hand-rolled L3 cells with two different output backends (default ggpaintr-managed and plotly) — project memory project-shared-section-binder documents the pre-2026-05-22 binder weakness here; (iii) custom plot output + B3 toggle (ptr_ui_toggle_code(ptr_ui_code("plot2")) attaches B3 next to a plotly output, not the default plot pane); (iv) F4 + F7 + F8 — state$runtime()$plot read inside renderPlotly; state$tree() available; state$runtime()$code_text drives the code panel.

Governing principle

ADR 0013 commits the suite to propagation assertions. App 3's distinguishing load: the shared widget's value MUST round-trip through the rendered code of both output cells, regardless of whether the cell's host is a default ggplot pane or a custom plotly pane. The B3 toggle MUST work next to a plotly output the same way it works next to a default pane — a regression where the toggle wiring is independent of the plot backend in code but accidentally coupled to it via shared state would surface here.

The G4 + G6 capture is also load-bearing: rlang::expr(...) bindings are a real user surface (the L3 vignette uses them), but no super-app before this one stresses them under L3. If the pre-quoted formula's placeholders are not discovered, the corresponding widgets do not appear on the page and the propagation assertions fail at set_sentinel — that failure mode is the auditor's evidence of the regression.

STOP-and-escalate on any structural infeasibility (e.g., hand-rolled L3 cannot register both cells against one ptr_shared_server). Do not silently fall back to ptr_app_grid — that defeats the G4 row entirely.

  • app.R shape. First line is the pkgload scaffolding. Body: library(shiny). Local ppRange registration. Two formulas as local bindings: formula_a <- rlang::expr(ggplot(mtcars, aes(x = ppVar(mpg), y = ppVar(wt), color = ppVar(cyl, shared = "linked"))) + geom_point(size = ppNum(2), alpha = ppNum(0.7)) + scale_x_continuous(limits = ppRange(c(10, 40))) + labs(title = ppText("Cell A: ggplot"))) and formula_b <- rlang::expr(ggplot(mtcars, aes(x = ppVar(hp), y = ppVar(qsec), color = ppVar(cyl, shared = "linked"))) + geom_point(size = ppNum(2)) + geom_smooth(method = ppText("lm"), linewidth = ppNum(1)) + labs(title = ppText("Cell B: plotly"))). A shared coordinator: shared <- ptr_shared(formulas = list(formula_a, formula_b), shared_ui = list(linked = function(id) selectInput(id, "Linked color", names(mtcars)))). UI built via ptr_ui_page(ptr_shared_panel(shared), shiny::fluidRow(...)) with two columns: column 1 holds plot1's ptr_ui_controls(formula_a, "plot1") + a default shiny::plotOutput(shiny::NS("plot1")("plot")) + ptr_ui_code("plot1"); column 2 holds plot2's ptr_ui_controls(formula_b, "plot2") + a plotly::plotlyOutput(shiny::NS("plot2")("custom_plot"), height = "500px") |> ptr_ui_toggle_code(ptr_ui_code("plot2")). Server: shared_state <- ptr_shared_server(shared); state_a <- ptr_server(formula_a, "plot1", shared_state = shared_state); state_b <- ptr_server(formula_b, "plot2", shared_state = shared_state); output[[shiny::NS("plot2")("custom_plot")]] <- plotly::renderPlotly({ res <- state_b$runtime(); shiny::req(isTRUE(res$ok), res$plot); plotly::ggplotly(res$plot) }). Final shiny::shinyApp(ui, server).

  • One test_that() block lands between PLAN-01's # >>> super-3 begin and # <<< super-3 end markers.
  • Shared "linked" key drives the rendered code of BOTH cells. The test sets the shared widget "shared_linked" to the unique choice "gear", draws plot1, draws plot2, then asserts in both cells' rendered code text that "gear" appears as a bare symbol in aes(... color = gear ...). The test sets "shared_linked" to "gear", draws plot2, toggles plot2's B3 to preserve (the toggle id is the one ptr_ui_toggle_code wires for plot2 — verified by inspecting the wrapped element's DOM), asserts the plot2 code text contains "ppVar("; toggles back to final, asserts the plot2 code text does NOT contain "ppVar(". Both transitions leave the plotly host with no error class.
  • G4 + G6 capture path is exercised end-to-end. The test asserts that all of plot1's expected widget input ids are present in the page (expect_dom_id per id, for each placeholder in formula_a) AND all of plot2's are present. If rlang::expr-captured formulas had their placeholders silently dropped, those input ids would not appear — failing the assertion.
  • ppRange (custom value placeholder) sentinel propagates in plot1. The test sets the ppRange slider (input id for plot1's scale_x_continuous(limits = ppRange(c(10, 40)))) to the literally-unique pair c(12, 38), draws plot1, asserts final-mode expect_sentinel_in_code(app, "plot1-<code id>", "c(12, 38)", "scale_x_continuous\\(limits\\s*=\\s*([^)]*)", "final"). In preserve mode, expect_sentinel_in_code(app, "plot1-<code id>", "c(12, 38)", "ppRange\\(([^)]*)\\)", "preserve").
  • No plot-error class on either cell, happy path. expect_no_plot_error(app, "plot1-plot") and an analogous assertion for plot2's plotly node (asserting "shiny-output-error" is absent from the plotlyOutput's HTML).
  • Authoritative gate returns FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + 1) with the browser stack and the plotly package available.

Definition of Done

Authoritative gate (copy verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + 1) when the browser stack + plotly are available.

Proxy traps — do not be fooled (verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate. Never claim the suite green, merge a branch, mark a step done, or clear context off the skipping/erroring harness.

This plan adds 1 new test_that() call at the super-3 anchor region. Gate must show PASS+1 with FAIL/ERROR still 0 and SKIP unchanged from pre-plan baseline (when plotly is available). If plotly is NOT installed on the gate-running machine, the test SKIPs cleanly via skip_if_not_installed("plotly") — that is correct behavior, but the orchestrator must run the gate on a machine where plotly is present to verify PASS+1.
