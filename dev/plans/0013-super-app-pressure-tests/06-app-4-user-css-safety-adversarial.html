PLAN-06 — Super-app 4: user_css + safety + adversarial

Source: ADR 0013 §"App 4 — user_css + safety + adversarial" + D1/D4/D5/D6 (G5 row)/D9 + J1/J3/J4 + K2/K3/K4 + I5 · Depends on: 01 · Parallel group: G2 · Status: draft

Context

ADR 0013 §App-4 locks the slot picks: ptr_app() + css · literal mtcars head · G5 string-builder force-eval (paste0 + sprintf, both inside the closed whitelist at R/paintr-app.R:260-262) · window code panel. One custom value placeholder ppColor registers locally (textInput wrapped in a styled <div class="ptr-super4-colorpicker">, validate_input rejects non-hex strings).

The roster: ppVar×3 · ppNum×4 · ppText×2 · ppExpr×1 · ppColor×1. Total widgets ≈ 11. Plus test-injected adversarial probes (NOT in app.R; the test sets bad values at runtime). One sibling user.css declares two CSS rules: a colorpicker border-color rule and a sentinel rule whose presence on the page proves user_css reached the DOM.

Unique pressure axes (ADR §App-4): (i) K2 + K3 coexistence — user_css + core htmlDependency assets coexist (project memory htmldependency-resource-prefix documents the explicit addResourcePath("ggpaintr", ...) at R/paintr-build-ui.R:591; the user_css's private resource prefix ggpaintr-user-<hash> is registered at R/paintr-build-ui.R:633); (ii) K4 user_css resource-path pruning — after the test session ends, no leak into subsequent sessions (the cleanup helper prune_dead_ggpaintr_resource_paths() documented in memory shinytest2-appdir-pkgload is exercised); (iii) G5 force-eval with closed-whitelist string builders; (iv) J1+J3 adversarial probe via ppExpr (test sets the widget to a string containing eval(parse(text = "1+1")) — the deny list at R/paintr-utils.R:448 + AST walker reject with the canonical literal "is not allowed in an `expr` input" at R/paintr-utils.R:609); (v) I5 inline validate_input rejection without crashing the runtime; (vi) custom widget styling reaches the page (user_css selector applies to the rendered widget).

K1 (bslib theming) is dropped from App 4's partition because ptr_app_bslib() is slated for removal (ADR §D1).

Governing principle

ADR 0013 commits the suite to propagation assertions. App 4's distinguishing load: the assertion contract must hold under a user-supplied stylesheet + core ggpaintr asset bundle coexisting on the same page. Safety regressions tend to live in eval-paths that look local (the denylist matches; the AST walker descends), so the adversarial probe MUST be the literal shape that historically bypassed — the project memory project-denylist-complete records that the walker's recursive descent is the primary safety mechanism, not the denylist alone. The test-injected probe MUST surface the rejection through the runtime panel with the canonical literal "is not allowed in an `expr` input" (R/paintr-utils.R:609), not just at parse time — that is the contract the deny list + walker promise jointly.

K4 user_css pruning is the only failure mode that bites later tests, not this one — a leak shows up as a downstream boot_vignette_app() failure with "Couldn't normalize path", order-dependently (memory htmldependency-resource-prefix). The plan's withr::defer + helper cleanup must close that asymmetry by construction; STOP-and-escalate if the implementer finds the helper does not run after AppDriver tear-down.

  • Fixture directory exists with three files. tests/testthat/fixtures/vignette-apps/super-4-user-css-safety-adversarial/app.R, tests/testthat/fixtures/vignette-apps/super-4-user-css-safety-adversarial/reference.R, tests/testthat/fixtures/vignette-apps/super-4-user-css-safety-adversarial/user.css. All tracked.
  • app.R shape. First line is the pkgload scaffolding. Body: ppColor registration via ptr_define_placeholder_value(keyword = "ppColor", build_ui = function(id, default = "#3366FF", label = "Color", ...) shiny::tags$div(class = "ptr-super4-colorpicker", shiny::textInput(id, label, value = default, placeholder = "#RRGGBB")), resolve_expr = function(value, ...) value, default_arg = ptr_default_string(), validate_input = function(value, ...) if (grepl("^#[0-9A-Fa-f]{6}$", value)) TRUE else "must be #RRGGBB hex"). Local string-builder fragments: y_arg <- "ppVar(wt)" and title_segment <- sprintf("Speed vs %s", "Weight"). Then ptr_app(paste0(...), css = "user.css"). The paste0 string body matches ADR §App-4 verbatim.
  • user.css shape. Two CSS rules:
    .ptr-super4-colorpicker input { border-color: #FF6633; }
    .ptr-super4-sentinel { color: #ABCDEF; }
    The second is the user_css-reached-the-page sentinel asserted by the test.
  • reference.R shape. The all-defaults equivalent ggplot2 expression with the paste0/sprintf resolved (so y_arg → wt, the sprintf → "Speed vs Weight"). Doc-only.
  • One test_that() block lands between PLAN-01's # >>> super-4 begin and # <<< super-4 end markers.
  • K2 + K3 coexistence — user_css + core htmlDependency assets both reach the page. The test asserts: (a) the user_css rule reached the page — verified by reading app$get_html("head") and checking it contains the literal substring ".ptr-super4-sentinel" (or alternatively a stylesheet link whose path begins with ggpaintr-user-, the resource-prefix produced by R/paintr-build-ui.R:633); (b) the core ggpaintr htmlDependency reached the page — verified by checking the head contains a stylesheet path beginning with /ggpaintr/ (the explicit addResourcePath-backed prefix at R/paintr-build-ui.R:591).
  • G5 force-eval propagation. The test sets the y-axis ppVar widget to a literally-unique column choice "qsec", draws, asserts final-mode expect_sentinel_in_code(app, code_output_id, "qsec", "aes\\([^)]*y\\s*=\\s*([^,)]*)", "final"). This proves the paste0-assembled formula's y_arg = "ppVar(wt)" was parsed into a real placeholder (not a literal string) — the widget exists and accepts a new value.
  • Custom widget container carries the styling class at the DOM AND the user_css rule applies. The test reads the ppColor widget's rendered HTML via app$get_html(<widget container selector>) and asserts the container's class attribute contains "ptr-super4-colorpicker". Additionally, the head HTML contains the literal CSS rule text ".ptr-super4-colorpicker" (proving the user_css selector for that class is wired up).
  • I5 inline validate_input rejection without crash. The test sets the ppColor widget to "notahex", draws, asserts: (a) the widget container shows the literal error string "must be #RRGGBB hex"; (b) expect_no_plot_error(app) still passes (the previous valid plot remains); (c) the code panel's previous sentinel content is preserved.
  • J1+J3 adversarial probe via ppExpr. The test sets the ppExpr input (the subtitle = ppExpr(nrow(mtcars)) placeholder) to the literally-unique adversarial string "eval(parse(text = \"1+1\"))", draws, asserts: (a) the inline error pane contains the canonical literal substring "is not allowed in an `expr` input" (from R/paintr-utils.R:609; the full template is "`<name>` is not allowed in an `expr` input (<where>)"); (b) the host output retains its prior plot (no crash); (c) expect_sentinel_nowhere(app, code_output_id, "eval(parse") succeeds (the adversarial text never reaches the rendered code).
  • K4 user_css pruning — no leak after teardown. The test scopes a withr::defer(prune_dead_ggpaintr_resource_paths(), envir = parent.frame()) at the top of the block (the helper is in helper-vignette-apps.R, exercised by other browser tests per project memory). After the test_that block completes, the test asserts (via a withr::defer-scoped post-check) that shiny::resourcePaths() contains no entry whose name begins with "ggpaintr-user-" AND whose target is a torn-down tempdir. Either the helper successfully pruned, or the user_css's resource path was registered to a persistent location that does not leak.
  • B3 toggle differential. Toggle preserve↔final once. In final mode the rendered code does not contain any of ppVar(, ppNum(, ppText(, ppExpr(, ppColor(; in preserve mode every one of those wrappers appears at least once. Host has no plot-error class in either mode.
  • Authoritative gate returns FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + 1) when the browser stack is available.

Definition of Done

Authoritative gate (copy verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + 1) when the browser stack is available.

Proxy traps — do not be fooled (verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate. Never claim the suite green, merge a branch, mark a step done, or clear context off the skipping/erroring harness.

This plan adds 1 new test_that() call at the super-4 anchor region. Gate must show PASS+1 with FAIL/ERROR still 0 and SKIP unchanged from pre-plan baseline.
