PLAN-01 — Test file scaffold + sentinel-propagation helpers

Source: ADR 0013 (D3 fixture-pair form + D4 probe-ledger assertion contract) · Depends on: none · Parallel group: G1 · Status: draft

Context

ADR 0013 D3 locks the single-file test layout: "tests/testthat/test-super-pressure.R (single file, all 5 apps inside as separate test_that() blocks)". ADR 0013 D4 locks the assertion contract: "For every placeholder in each super-app, the test sets a literally-unique sentinel value via app$set_inputs(<id> = <sentinel>, wait_ = FALSE), clicks the Update button, then asserts: Final-mode propagation … Preserve-mode round-trip … No error class on plot output."

This plan creates the enforcement surface that the five app plans (02–06) will fill in. Two artefacts: (a) the test file with five anchor-delimited empty regions, one per app, so concurrent worktrees never overlap; (b) a helper module exposing the sentinel-propagation primitives whose signatures force every caller to name the exact sentinel + the exact expected position. Any helper that lets a caller assert "some text" rather than "this specific sentinel at this specific position" is the failure shape recorded in project memory e2e-assertion-weakness-lens — every helper in this plan is shaped to make that shape impossible.

The plan produces zero test_that() blocks of its own. The gate is unaffected: no PASS added, no SKIP added, no test ID created. The five app plans then each insert exactly one test_that() block between their owned anchor markers.

Governing principle

ADR 0013 commits the suite to propagation assertions, not presence proxies. The motivating regression (the B3 toggle wiring, fixed in 2c504da) survived its full cycle of presence-style fixture assertions (expect_dom_id, expect_code_nonempty) because those helpers proved only that some code text reached the panel. Project memory e2e-assertion-weakness-lens distills the lesson: presence proves the surface; propagation proves the integration.

This plan's helper module is the binding mechanism for that principle. Every helper signature in helper-super-pressure.R takes an exact sentinel string and asserts the sentinel appears at an exact rendered position under an explicitly named mode (final or preserve). The helpers refuse to fall back to "code panel non-empty": a helper that did so would let an integration bug of the B3 shape survive again. The anchor markers in the test file are equally load-bearing — they make the suite's five apps independently editable, so a defect in one app's coverage does not pressure later plans to weaken the contract for the rest.

Strengthening, not weakening, is the only legal direction of change. If a future refactor wants to remove a propagation helper in favor of a presence helper "for speed", that is an ADR-0013 violation — flag it, do not absorb it.

Success Criteria

  • Test file exists with the five anchor regions. tests/testthat/test-super-pressure.R exists; grep -c "^# >>> super-" tests/testthat/test-super-pressure.R returns 5 and grep -c "^# <<< super-" tests/testthat/test-super-pressure.R returns 5. The marker pairs (in this exact order, with these exact strings) are: # >>> super-1 begin/# <<< super-1 end, # >>> super-2a begin/# <<< super-2a end, # >>> super-2b begin/# <<< super-2b end, # >>> super-3 begin/# <<< super-3 end, # >>> super-4 begin/# <<< super-4 end. Each region contains no test_that() calls — only whitespace and a single comment line stating which plan fills it.
  • Test file contains zero test_that() calls. grep -c "^test_that" tests/testthat/test-super-pressure.R returns 0. The gate's PASS/SKIP counts are unaffected by this plan.
  • Helper file exists and is loaded by testthat. tests/testthat/helper-super-pressure.R exists; testthat loads it automatically during test_dir() (no source() call needed in the test file).
  • Helper: boot_super_app(slug). Boots tests/testthat/fixtures/vignette-apps/<slug>/app.R via the same scaffolding as boot_vignette_app() (tests/testthat/helper-vignette-apps.R): skip_on_cran(), skip_if_not_installed("shinytest2"), skip_if_not_installed("chromote"), source-root guard, GGP_PKG env var, prune_dead_ggpaintr_resource_paths(), suppressWarnings(shinytest2::AppDriver$new(...)), withr::defer(app$stop()). Returns the AppDriver. Distinct name from boot_vignette_app() so the two helpers are independently evolvable.
  • Helper: expect_sentinel_in_code(app, code_output_id, sentinel, position_regex, mode). Reads the rendered code text via app$get_value(output = code_output_id), then asserts that regexpr(position_regex, code_text, perl = TRUE) matches AND the matched substring contains the literal sentinel. The mode argument is one of "final", "preserve" and is recorded in the assertion label (so a failure message names which mode failed). Any of code_text empty, sentinel missing entirely from code_text, position_regex matching but its capture not containing sentinel, fails the expectation with a label naming all four inputs.
  • Helper: expect_sentinel_nowhere(app, code_output_id, sentinel). Reads the rendered code text and asserts the literal sentinel does not appear anywhere in it (used for the inverse claim in final mode when only preserve mode should show a placeholder wrapper). fixed = TRUE.
  • Helper: set_sentinel(app, input_id, sentinel). Wraps app$set_inputs( = , wait_ = FALSE) with literal sentinel string. Identical semantics to the existing set_input() helper in helper-vignette-apps.R but renamed to make sentinel-shaped intent visible in test code.
  • Helper: draw_and_wait(app, button_id = "draw"). Clicks the named button and runs wait_for_idle(timeout = 25 * 1000). Default button id matches the project's existing draw-button convention in ptr_app() (so most callers pass no button_id argument).
  • Helper: toggle_code_mode(app, mode). Sets the B3 radio (input id "ptr_code_mode" as registered by ptr_register_code) to "final" or "preserve" using wait_ = FALSE, then app$wait_for_idle(). Aborts with stop("mode must be 'final' or 'preserve'") on any other value.
  • Helper: expect_no_plot_error(app, host_id = "ptr_plot"). Asserts the host output's outer HTML contains neither the class shiny-output-error nor the class ptr-alert--error. Default host id matches the project's existing ptr_app() host output id.
  • Anchor immutability guard. The file header of test-super-pressure.R contains a comment block that names every anchor pair, states the per-anchor plan ownership (super-1 → PLAN-02, super-2a → PLAN-03, super-2b → PLAN-04, super-3 → PLAN-05, super-4 → PLAN-06), and explicitly forbids later plans from removing or renaming anchor markers.
  • Authoritative gate returns FAIL 0 / ERROR 0 / SKIP 0 / PASS N_baseline (PASS count unchanged from baseline; this plan adds zero tests).

Constraints

  • Files this plan owns:
    • tests/testthat/test-super-pressure.R (NEW)
    • tests/testthat/helper-super-pressure.R (NEW)
  • Files this plan must NOT touch:
    • Any R/paintr-*.R source file. This plan is a test-infrastructure plan; no product change is allowed.
    • tests/testthat/helper-vignette-apps.R — even if the new helpers duplicate small fragments (e.g. set_input body), do not refactor the existing helper to share. Sharing helpers across two e2e suites couples them; the new helpers stay independent so super-pressure can evolve without disturbing the vignette-app suite.
    • Any existing tests/testthat/test-*.R or tests/testthat/fixtures/ file.
    • NAMESPACE, DESCRIPTION, anything under man/.
  • No new package dependencies. The helpers use only what shinytest2, testthat, withr, and shiny already provide. shinytest2 and chromote are referenced via skip_if_not_installed(), not library().
  • No production R-source side effects. The helper file does not call options(), Sys.setenv(), or modify .GlobalEnv at load time. All state is scoped per-test via withr::local_* inside the helpers, mirroring helper-vignette-apps.R.
  • SCOPE NARROWING FORBIDDEN. If implementing a Success Criterion or BDD Then as written breaks something (e.g. expect_sentinel_in_code reveals that app$get_value(output = code_output_id) returns NULL on the project's actual code-panel output id), STOP and escalate to the orchestrator. Do not narrow the helper to "asserts text not empty"; do not weaken the regex contract to substring inclusion. The whole point of these helpers is that they refuse to weaken.
  • Conventions to honor: project's CLAUDE.md and .claude/rules/testing.md. Two-space indent, snake_case helper names, testthat::expect_* wrappers, no get_values() (per memory shinytest2-appdir-pkgload).

BDD scenarios

Feature: Sentinel-propagation helpers refuse to weaken into presence proxies

Scenario: expect_sentinel_in_code fails on empty code text
  Given a stub app whose code-panel output value is the empty string ""
  When I call expect_sentinel_in_code(app, "ptr_code", "S_abc", "ppNum\\(([^)]*)\\)", "preserve")
  Then the testthat expectation fails
  And the failure label contains the four input strings "ptr_code", "S_abc", "ppNum", "preserve"

Scenario: expect_sentinel_in_code passes when sentinel sits inside the regex capture
  Given a stub app whose code-panel output value is "geom_point(size = ppNum(S_abc))"
  When I call expect_sentinel_in_code(app, "ptr_code", "S_abc", "ppNum\\(([^)]*)\\)", "preserve")
  Then the testthat expectation passes
  And the matched capture group contains the literal "S_abc"

Scenario: expect_sentinel_in_code fails when the regex matches but the capture does not contain the sentinel
  Given a stub app whose code-panel output value is "geom_point(size = ppNum(7))"
  When I call expect_sentinel_in_code(app, "ptr_code", "S_abc", "ppNum\\(([^)]*)\\)", "preserve")
  Then the testthat expectation fails
  And the failure label names the sentinel "S_abc" and the actual capture "7"

Scenario: expect_sentinel_nowhere passes on final-mode-stripped output
  Given a stub app whose code-panel output value is "geom_point(size = 7)"
  When I call expect_sentinel_nowhere(app, "ptr_code", "ppNum")
  Then the testthat expectation passes

Scenario: expect_sentinel_nowhere fails when the literal still appears
  Given a stub app whose code-panel output value is "geom_point(size = ppNum(7))"
  When I call expect_sentinel_nowhere(app, "ptr_code", "ppNum")
  Then the testthat expectation fails

Scenario: toggle_code_mode rejects unknown modes
  Given a stub app
  When I call toggle_code_mode(app, "raw")
  Then the call aborts with a message containing "mode must be 'final' or 'preserve'"

Scenario: anchor regions are present, ordered, and empty
  When I parse tests/testthat/test-super-pressure.R
  Then I find exactly five "# >>> super-N begin" / "# <<< super-N end" pairs
  And the pairs appear in this order: super-1, super-2a, super-2b, super-3, super-4
  And between each pair there is no test_that() call
  And no other content sits between any pair except a single TODO comment naming the plan that will fill it

Scenario: the suite gate is unaffected
  Given the baseline gate count before this plan: FAIL 0 / ERROR 0 / SKIP 0 / PASS N
  When I run the authoritative gate after this plan lands
  Then the gate reports FAIL 0 / ERROR 0 / SKIP 0 / PASS N
  And no new test ID is created

Definition of Done

Authoritative gate (copy verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N (N unchanged from baseline; this plan adds zero tests).

Proxy traps — do not be fooled (verbatim from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard (no DESCRIPTION in the .Rcheck sandbox). If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() runs the browser tests (source tree present → 0 SKIP). That alone is the gate. Never claim the suite green, merge a branch, mark a step done, or clear context off the skipping/erroring harness.

This plan adds 0 new test_that() calls. The PASS count must equal baseline; FAIL/ERROR/SKIP must equal 0. Any change from baseline means this plan introduced an unintended product-level effect (e.g. helper file was sourced eagerly and registered something at load time) — that is a regression of this plan, not a project defect.

Scope guard / ownership

This plan owns: tests/testthat/test-super-pressure.R, tests/testthat/helper-super-pressure.R. No generated files. NAMESPACE/man/ are untouched (no exported symbols added). Lockfiles, codegen output, OpenAPI clients — N/A.

Branch convention: this plan lives on branch 0013-test-file-scaffold-and-helpers in a worktree at .claude/worktrees/0013-super-app-pressure-tests/test-file-scaffold-and-helpers/. Merge into the orchestrator branch via --no-ff.

Independence clause

The implementer's "done" report is a claim, not proof. The auditor (and the post-merge auditor) re-runs the Definition-of-Done command, verifies FAIL 0 / ERROR 0 / SKIP 0 / PASS N_baseline, and independently verifies the five anchor pairs exist in the order locked above before any merge.

Baseline / drift reference

Known-good baseline: orchestrator branch HEAD at the commit where this plan's worktree was created (record the SHA in the worktree's first commit message). Pre-existing PASS count: read at worktree creation, written into the worktree commit message as baseline_pass=N. Any FAIL/ERROR/SKIP > 0 in baseline is named in dev/baselines/ if it exists; absent, baseline is taken as 0/0/0.

Out-of-scope handling

Findings outside this plan's owned files are surfaced in the audit report, never absorbed. If the implementer discovers (e.g.) that ptr_register_code's input id is not "ptr_code_mode" as documented but something else, that becomes a follow-up plan (a doc fix or a follow-up to ADR-0009), not a silent edit to R/paintr-server.R from this plan's branch.
