PLAN-07 — Unified validate_input(value, ctx) hook for value + consumer placeholders

Source: follow-up A from the 2026-05-23 ADR-0013 /exec-plan session (handoff document at /var/folders/.../handoff-XXXXXX.md). Depends on: — · Provides for: PLAN-03, PLAN-06 · Status: draft

Context

The 2026-05-23 /exec-plan on ADR-0013 merged PLAN-01, PLAN-02, PLAN-05 and left PLAN-03, PLAN-04, PLAN-06 BLOCKED. Two of those blockers (PLAN-03 + PLAN-06) share one root cause: ptr_define_placeholder_value() (R/paintr-registry.R:412-440) has no validate_input parameter. Only ptr_define_placeholder_consumer() (R/paintr-registry.R:508-541) supports the hook today.

A narrower fix — just bolt validate_input onto ptr_define_placeholder_value() with the consumer's existing function(value, upstream_cols) signature — was considered and rejected. Two problems with the narrow shape:

  1. upstream_cols is meaningless for a value placeholder. A value placeholder has no upstream column scope (data_aware = FALSE on its registry entry, R/paintr-registry.R:432). Passing an always-NULL second arg is dishonest and confuses the contract.
  2. The consumer's hook is itself too narrow. The substitute walker has the upstream data frame in scope (ctx at R/paintr-substitute.R:18-25) but only forwards a character vector of column names to the validator (R/paintr-substitute.R:124-127). Authors who want to validate by column type / range / uniqueness / NA-fraction can't — they're forced into build_ui-time choice filtering (loses the "inline error message" UX) or silent resolve_expr → NULL pruning.

This plan adopts a single unified hook signature for both roles: function(value, ctx), where ctx is a plain list with stable named fields. Each role populates the subset of fields that makes sense:

ctx fieldtypevalue roleconsumer rolemeaning
nodelist (ptr placeholder AST node)populatedpopulatedThe placeholder node itself; carries $id, $keyword, $args, etc.
keywordcharacter(1)populatedpopulatedConvenience alias for node$keyword.
upstream_colscharacter or NULLNULLpopulated when upstream resolved; else NULLUpstream column names; same value the consumer's build_ui receives as cols.
datadata.frame or NULLNULLpopulated when upstream resolved; else NULLUpstream data frame; same object the consumer's build_ui receives as data.

This closes both gaps in one move: PLAN-03 + PLAN-06 unblock because ptr_define_placeholder_value() finally has a hook; future data-aware consumer validators (column-type checks, range checks, …) work because ctx$data reaches the validator.

Governing principle

One signature, two roles, no shims. ggpaintr is pre-1.0 (DESCRIPTION version < 1.0.0); a hard cutover on the consumer's validate_input signature is acceptable. The migration cost is exactly one in-tree built-in (ppVar, R/paintr-builtins.R:201-209) plus the doc example in ptr_define_placeholder_consumer()'s roxygen (R/paintr-registry.R:502-505). Any out-of-tree custom placeholders that used the old signature will fail registration loudly via validate_hook() (R/paintr-registry.R:132-156) — that loud failure IS the correct migration UX, not a regression.

Dual-signature support (auto-detect formals at call time, branch between old and new) is explicitly rejected: it adds a permanent code path that is hard to reason about, hard to test exhaustively, and leaves the codebase carrying two contracts forever. Hard cutover with a clear error message is cleaner.

Success Criteria

  • Constructor: ptr_define_placeholder_value(). Gains a validate_input = NULL parameter (mirror the consumer's slot). When non-NULL, validates the hook via validate_hook(validate_input, "validate_input", c("value", "ctx")). Stored on the registry entry as entry$validate_input.
  • Constructor: ptr_define_placeholder_consumer(). The hook contract check changes from c("value", "upstream_cols") to c("value", "ctx") (one-line edit at R/paintr-registry.R:522). Storage and downstream call shape unchanged.
  • Substitute walker — value role. In substitute_walk.ptr_ph_value() (R/paintr-substitute.R:88-107), insert a validate_input call between the missing-value guard and the resolve_expr call, mirroring the consumer's pattern (R/paintr-substitute.R:124-137). Build the hook ctx as hook_ctx <- list(node = node, keyword = node$keyword, upstream_cols = NULL, data = NULL) (the local name hook_ctx avoids shadowing the walker's own ctx arg). On a single character return, rlang::abort(paste0("Invalid value for placeholder `", node$keyword, "`: ", ok)). TRUE/NULL accept.
  • Substitute walker — consumer role. Replace the existing call site (R/paintr-substitute.R:124-137): build hook_ctx <- list(node = node, keyword = node$keyword, upstream_cols = ctx$upstream_cols[[node$id %||% ""]], data = ctx$upstream_data[[node$id %||% ""]]) and pass that as the 2nd positional arg to entry$validate_input. The error-surfacing branch (rlang::abort on character return) is unchanged. The ctx$upstream_data slot is a NEW addition to the walker's incoming context — see the plumbing items below.
  • Walker ctx — new upstream_data slot. ptr_substitute() (R/paintr-substitute.R:16-27) gains a new parameter upstream_data = list(), defaulting to an empty named list. The constructed ctx at lines 20-25 gains a new field upstream_data = upstream_data. Shape parallels upstream_cols: a named list keyed by consumer node id, values are the resolved upstream data.frame (or absent when upstream has not resolved yet).
  • Server plumbing — expose the per-consumer data frames. Add a sibling wrapper to runtime_upstream_cols() (R/paintr-server.R:1331-1334): 
    runtime_upstream_data_frames <- function(state, snapshot = list()) {
  res <- runtime_upstream_data(state, snapshot)
  lapply(res, function(x) x$data)
}
    No new resolution work — runtime_upstream_data() already returns list(<id> = list(cols = ..., data = ...)) per consumer (R/paintr-server.R:1273-1325, especially line 1302 / line 1322: out[[c$id]] <- list(cols = names(df), data = df)). The new wrapper just extracts the data slot, symmetric to how runtime_upstream_cols() extracts cols.
  • Server call-site forwarding. At R/paintr-server.R:1148-1159 the existing ptr_exec_headless(..., upstream_cols = upstream_cols) call gains a sibling argument upstream_data = runtime_upstream_data_frames(state, snapshot). (Both wrappers share one underlying call — implementers may refactor to compute runtime_upstream_data(state, snapshot) ONCE and project to cols/data twice, instead of calling it twice. Either shape is acceptable.)
  • Headless plumbing. ptr_exec_headless() (R/paintr-headless.R:22-32 signature; line 39 body interior pass; line 128 recursive call) gains a new parameter upstream_data = NULL alongside the existing upstream_cols, plumbs it through identically. Note: ptr_exec_headless does NOT call ptr_substitute() directly — it calls ptr_complete_expr_safe() (R/paintr-safe.R:17+), which is where the real ptr_substitute() call lives. The new arg flows: ptr_exec_headless(upstream_data = X) → ptr_complete_expr_safe(upstream_data = X) → ptr_substitute(upstream_data = X).
  • Safe-substitute plumbing. ptr_complete_expr_safe() (R/paintr-safe.R:17 signature opens; line 22 final formal; line 31 forwards to ptr_substitute) gains the same upstream_data = list() parameter and forwards it to ptr_substitute().
  • Built-in migration: ppVar. ptr_builtin_var_validate_input() (R/paintr-builtins.R:201-209) migrates from function(value, upstream_cols) to function(value, ctx). Body reads ctx$upstream_cols. Behavior unchanged: length(value) != 1L → "Expected a single column; got <n>."; !value %in% ctx$upstream_cols → "Column `<col>` is not in the data."; otherwise TRUE.
  • Doc-example migration. The @examples block in ptr_define_placeholder_consumer()'s roxygen (R/paintr-registry.R:489-507) updates its validate_input = function(value, upstream_cols) ... to function(value, ctx) ... ctx$upstream_cols ....
  • Roxygen — consumer. The @param validate_input on ptr_define_placeholder_consumer() (R/paintr-registry.R:465-469) updates its argument-name from function(value, upstream_cols) to function(value, ctx), and gains a new sentence enumerating ctx fields (node, keyword, upstream_cols, data) and that ctx$upstream_cols / ctx$data may be NULL while upstream is pending.
  • Roxygen — value. New @param validate_input stanza on ptr_define_placeholder_value() mirroring the consumer's prose, but explicitly noting that ctx$upstream_cols and ctx$data are always NULL for the value role (value placeholders have no upstream column scope by definition).
  • devtools::document() clean. Regenerates man/ptr_define_placeholder_value.Rd and man/ptr_define_placeholder_consumer.Rd without warnings. NAMESPACE unchanged.
  • New unit-test coverage in tests/testthat/test-placeholder-registry.R (or a new test-validate-input-ctx.R). At least 6 new test_that blocks, each with a focused assertion:
    1. ptr_define_placeholder_value() with validate_input = function(value, ctx) TRUE registers successfully and the registry entry stores it.
    2. ptr_define_placeholder_value() with validate_input = function(value, upstream_cols) TRUE fails registration with the message from validate_hook() ("must accept argument(s): ctx").
    3. ptr_define_placeholder_consumer() with the new function(value, ctx) signature registers; with the legacy function(value, upstream_cols) signature fails registration loudly.
    4. Value-role substitute walk: registering a value placeholder whose validate_input returns the string "bad" for a specific input causes substitution of that input to abort with a message containing both the literal bad and the placeholder's keyword.
    5. Consumer-role substitute walk: a custom consumer whose validator returns a string when !is.numeric(ctx$data[[value]]) rejects a non-numeric column choice with that exact string when data is present in ctx. (This is the data-aware consumer-validation case that was previously impossible.)
    6. ppVar built-in regression: length-not-1 returns the documented string; unknown column returns the documented string; a valid column resolves to its rlang::sym(value) form.
    Use existing helpers in tests/testthat/helper-*.R where applicable; do not invent a new test-file location convention.
  • NEWS/changelog entry. If NEWS.md exists at the package root, add a "Breaking changes" bullet documenting the consumer validate_input signature change and the new value-role hook. If NEWS.md does not exist, no other file is touched in lieu — this constraint applies only when the file already exists.
  • Authoritative gate returns FAIL 0 / ERROR 0 / SKIP 0 / PASS (baseline + N_new) where N_new ≥ 6 (the new tests above; more is fine).

BDD scenarios

Feature: validate_input on value placeholders

Scenario: Value placeholder registers a validate_input hook
  Given a fresh placeholder registry
  When I call ptr_define_placeholder_value(
        keyword = "ppPositive",
        build_ui = function(node, ...) shiny::numericInput(node$id, NULL, 0),
        resolve_expr = function(value, node, ...) value,
        validate_input = function(value, ctx) if (value > 0) TRUE else "must be positive")
  Then registration succeeds without error
  And the registry entry for "ppPositive" has a non-null validate_input field

Scenario: Value placeholder rejects legacy (value, upstream_cols) signature
  Given a fresh placeholder registry
  When I call ptr_define_placeholder_value(
        keyword = "ppPositive",
        build_ui = ...,
        resolve_expr = ...,
        validate_input = function(value, upstream_cols) TRUE)
  Then registration fails with rlang::abort
  And the error message mentions both "validate_input" and "ctx"

Scenario: Invalid value surfaces as inline error, not silent prune
  Given a value placeholder "ppPositive" registered with validate_input rejecting <= 0
  And a Shiny app substituting ppPositive's value into a formula
  When the substitute walker visits the ppPositive node with value -1
  Then rlang::abort is raised
  And the message contains the literal string "must be positive"
  And the message contains the keyword "ppPositive"

Scenario: Value-role ctx omits upstream_cols and data
  Given a value placeholder whose validate_input is function(value, ctx) {
        stopifnot(is.null(ctx$upstream_cols), is.null(ctx$data)); TRUE }
  When the substitute walker visits that placeholder
  Then the stopifnot does not trigger
  And validation returns TRUE

Feature: validate_input on consumer placeholders via ctx

Scenario: Consumer validator reads ctx$data to check column type
  Given a consumer placeholder whose validate_input rejects when ctx$data[[value]] is non-numeric
  And upstream data is a data.frame with one numeric column "x" and one character column "g"
  When the consumer's value is set to "g"
  Then rlang::abort is raised with the validator's rejection message
  When the consumer's value is set to "x"
  Then validation accepts and resolve_expr is reached

Scenario: Consumer validator with legacy (value, upstream_cols) signature fails registration
  Given a fresh placeholder registry
  When I call ptr_define_placeholder_consumer with
        validate_input = function(value, upstream_cols) TRUE
  Then registration fails with rlang::abort
  And the error message mentions both "validate_input" and "ctx"

Scenario: Consumer ctx$upstream_cols matches what build_ui was passed as cols
  Given a consumer placeholder whose build_ui stashes its `cols` arg into a parent-process env
  And the same placeholder's validate_input stashes ctx$upstream_cols into the same env
  When both have run for the same node
  Then the two stashed character vectors are identical

Feature: ppVar built-in migrated to ctx signature, behavior unchanged

Scenario: ppVar rejects length != 1 with the documented message
  Given upstream data with columns c("mpg", "wt")
  When the ppVar node's value is c("mpg", "wt")
  Then validate_input returns the literal string "Expected a single column; got 2."

Scenario: ppVar rejects unknown column with the documented message
  Given upstream data with columns c("mpg", "wt")
  When the ppVar node's value is "hp"
  Then validate_input returns the literal string "Column `hp` is not in the data."

Scenario: ppVar accepts a valid column
  Given upstream data with columns c("mpg", "wt")
  When the ppVar node's value is "mpg"
  Then validate_input returns TRUE
  And the resolved expression equals rlang::sym("mpg")
