Scope

Amend a single BDD Then clause in PLAN-01's "ppUpload spec value is silently dropped (fileInput unsettable)" scenario (parent plan lines 123–131). The clause as written contradicts the scenario's own title and the parent plan's structural-asymmetry principle: it asks for an explicit cli::cli_inform announcing the dropped ppUpload value, while the title (and the only sensible user-facing contract) is silent. The post-merge implementation is silent; the BDD wording is the side that drifted, not the code.

What changes

The "silently dropped" scenario's third Then clause is deleted. No code change. No new test.

Scenario: ppUpload spec value is silently dropped (fileInput unsettable)
  Given a placeholder ppUpload with id "ggplot_1_1_ppUpload_NA"
    And spec = list(ggplot_1_1_ppUpload_NA = "ignored.csv",

    And no cli::cli_inform or cli::cli_warn names "ggplot_1_1_ppUpload_NA"

The new final clause makes the silence explicit so any future implementer can’t reintroduce a stray message without breaking a stated Then.

Rationale

  1. The scenario title is load-bearing. "silently dropped" is the contract. The original clause [2] ("a single cli::cli_inform names … as dropped") contradicts that. The contradiction almost certainly came from copy-paste from the next scenario in the parent plan ("spec entry for unknown id is dropped with single aggregate cli_inform" at parent line 133), which legitimately does want a message.
  2. A message here would be user-hostile. Programmatic spec= callers typically replay a saved app state. Their fileInput value is unavoidably non-round-trippable (Shiny’s fileInput takes its value from a browser-side upload event, not from updateXyzInput). Announcing this on every boot would be noise, not signal. The companion id ("…_name") does round-trip and carries the only piece a user can meaningfully reconstruct — that’s already covered by clauses [3] and [4].
  3. The code already implements the silent contract. Verified at R/paintr-server.R in this worktree:
    • The seed-write guard at L538–541 excludes row$keyword == "ppUpload" — the row is skipped via next with no side effect.
    • The cli_inform at L507–511 only fires for ids absent from state$input_spec$input_id. The ppUpload id is in input_spec, so it does not enter the dropped vector.
    • apply_spec_entry’s ppUpload branch at L677–682 is a silent return(TRUE) with no message.
    • The source_companion row for "…_name" routes through the second arm of write_seed at L540 and is type-normalized as character at L544–547. The companion textInput’s renderUI reads state$spec_seed[[bare_id]] as extra$selected.
