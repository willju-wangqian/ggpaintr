Scope

Two PLAN-01 BDD Then clauses drifted between plan-text and post-implementation reality. Both are test-honesty drift, not contract violations — the implementation is correct and the existing tests assert the architecturally-equivalent observable. The clauses are amended here so future audits do not re-flag them.

  1. M1 — session$input[[id]] assertions under testServer. PLAN-01 BDD clauses that say "session$input[[id]] equals X" were swapped to HTML-matcher assertions in the implementing tests (test-adr12b-spec-custom-keyword.R:91 for <option value="b" selected>; test-adr12b-renderui-emission.R:56–57 for the no-spec ppNum's id= + value="NA" matchers). Root cause: under shiny::testServer, MockShinySession does not auto-populate session$input[[id]] from a renderUI'd widget — only statically-mounted inputs flow back. The HTML matcher (e.g. <option value="b" selected>) is a literal architectural-propagation proof that the seed reached the build hook's selected formal, so the architectural contract is honored; only the BDD wording is stale. (The original audit cited :79 and :43, 76 for these matchers, but those line numbers are off — :79 is the state$spec_seed assertion, :43 is the testServer call, and :76 is a fixture formula. Lines above are re-verified at the post-merge HEAD efbf793.)
  2. L1 — state$spec_seed shape. PLAN-01 BDD says "state$spec_seed is an empty named list". The implementation made spec_seed a named environment, not a list, so writes from apply_spec_at_boot propagate by reference to the closure-captured state inside ptr_setup_value_uis / ptr_setup_source_uis / ptr_setup_consumer_uis. The "empty" intent is preserved (length(ls(state$spec_seed, all.names = TRUE)) == 0L). The tests assert on the new shape.

Neither change requires new test coverage; both are amendments to the BDD wording so the plan reads honestly against the merged implementation.

A future tidy-up could add a separate scenario exercising a non-trivial ppNum(default = 3.14), but it is not in scope for this addendum.

Rationale

  • M1 — HTML matcher is not a presence proxy. The literal substring <option value="b" selected> can only appear in the rendered HTML if the seed value reached the hook's selected formal and the hook chose to mark it selected. This is the same architectural contract as session$input[[id]] == "b", observed at a different layer. Project memory e2e-assertion-weakness-lens warns about presence proxies; the HTML literal is a propagation proof, not a presence proxy.
  • L1 — environment is the right call. The setup helpers (ptr_setup_value_uis etc.) capture state in a closure created at boot. If state$spec_seed were a list, writes from apply_spec_at_boot would modify a fresh copy and the renderUI helpers would read the original (empty) snapshot. An environment is the cheapest way to share state by reference across closures.
