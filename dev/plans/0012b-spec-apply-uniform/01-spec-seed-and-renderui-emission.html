PLAN-01 — state$spec_seed + uiOutput emission for value/source placeholders

Source: ADR 0012 (Bug B successor) · Depends on: none · Parallel group: G1 · Status: draft

Context

ADR 0012 §3.6 / PLAN-06 shipped spec= apply-at-boot via updateXyzInput dispatch keyed on placeholder keyword (apply_spec_entry at R/paintr-server.R:526-592). The registry contract for custom placeholders is a single build_ui = function(node, label, ...) hook — there is no update_value hook. So apply_spec_entry's keyword-dispatch is structurally exclusive to built-in keywords; a custom keyword falls through to FALSE at line 592 and the spec entry is silently aggregated into the "skipped invalid value" warning at end of session$onFlushed. This is Bug B from the 2026-05-22 handoff.

The fix routes every placeholder widget through the existing invoke_build_ui precedence logic (R/paintr-build-ui.R:721-733, which already seeds extra$selected from node$default when the hook accepts it) by rendering it server-side inside a renderUI whose extra$selected reads a new per-instance seed: state$spec_seed[[bare_id]], written once at boot by apply_spec_at_boot.

Step A semantics. The existing apply_spec_entry updateXyzInput dispatch path stays in place. apply_spec_at_boot writes the per-instance state$spec_seed before the deferred session$onFlushed dispatch, so the renderUI's first fire seeds the widget with the spec value via isolate(state$spec_seed[[bare]]); the subsequent updateXyzInput within onFlushed is redundant for built-in keywords (sets the same value) and skipped for custom keywords (path unchanged). End state for built-in keywords is unchanged; end state for custom keywords becomes "value applied" instead of "silently skipped" — this is the Bug B fix.

Governing principle

The registry contract for placeholders is a single hook — build_ui = function(node, label, ...). There is no symmetric update_value hook. Any spec-apply mechanism that dispatches on placeholder keyword and calls updateXyzInput is structurally exclusive to the built-in keywords that ship with ggpaintr: a custom placeholder keyword has no widget-type for which a built-in updateXyz exists, so the dispatch falls through to a silent no-op. The contract is uniform; the apply path must be uniform. The fix is to route every placeholder widget — built-in and custom — through the one hook that already exists (build_ui), seeding its extra$selected from a state-owned seed that apply_spec_at_boot writes once. An implementer reading only this plan must hold this principle: if you find yourself reaching for updateXyz in the spec path, you have re-introduced the structural asymmetry the plan exists to delete; STOP and escalate. Narrow scope to keep built-ins green at the cost of custom-keyword silence is the exact failure this plan forbids.

Authoritative gate (copied verbatim from CLAUDE.md):

Expected output: FAIL 0 / WARN 0 / SKIP 2 / PASS ≥ 2181 + N where N is the test-count delta from this plan's two new test files (estimate: N = 6–10 across the two files). Proxy traps — do not be fooled (copied from CLAUDE.md):

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN → skip_on_cran() fires → every shinytest2 browser test silently SKIPs. "green, SKIP n" off that is NOT the gate.
  • devtools::check(... --as-cran) does NOT make skip_on_cran() fire (devtools runs the check subprocess with NOT_CRAN=true). The browser fixtures instead cleanly SKIP via boot_vignette_app()'s source-root guard. If a check run shows the e2e file ERRORing (R CMD check found ERRORs, pkgload "DESCRIPTION not found"), that guard is missing/broken — a harness defect, not a product failure, and never the gate.
  • Only NOT_CRAN=true / devtools::test() RUNS the browser tests (source tree present → 0 SKIP for that file). That alone is the gate.
