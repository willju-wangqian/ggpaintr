PLAN-02 — Collapse apply_spec_entry to seed-only + revert test workarounds

Source: ADR 0012 (Bug B successor) · Depends on: 01 · Parallel group: G2 · Status: draft

Context

PLAN-01 added the state$spec_seed seed path and reshaped build_ui_for.ptr_ph_value / build_ui_for.ptr_ph_data_source to emit uiOutput containers rendered server-side. Both the new seed path AND the legacy updateXyzInput path now run, with the seed taking visual precedence (because renderUI fires before the deferred onFlushed callback) and the updateXyzInput calls being redundant for built-in keywords.

This plan deletes the redundant path so the codebase has one mechanism for spec-apply, restoring single-source-of-truth and removing the structural asymmetry between built-in and custom keywords. It also reverts the test workarounds in tests/testthat/test-adr12-spec-roundtrip.R that fell back to app$get_value(...) because shinyWidgets::updatePickerInput updates the widget client-side without re-rendering the server-side HTML — now that ptr_setup_consumer_uis renders the picker server-side via renderUI with selected = state$spec_seed[[bare]], the initial HTML carries <option ... selected> natively and app$get_html can assert it.

Governing principle

Same as PLAN-01: the registry contract is uniform (one build_ui hook), therefore the apply path must be uniform. PLAN-01 made the seed path WORK for custom keywords by adding it alongside the legacy path; PLAN-02 makes the seed path the ONLY path so the asymmetry cannot regress. A future contributor reading the post-PLAN-02 codebase must find no trace of "dispatch on keyword and call updateXyz" in the spec-apply flow. Pre-write normalization (numeric coercion, expression deparse) moves into the seed-write step inside apply_spec_at_boot / apply_spec_entry, not into renderUI consumers — so the "invalid value" warning still fires once at boot, never per-render. Mid-stream value validation in renderUI is a different anti-pattern (silent UI churn) and is forbidden.

Success Criteria

  • SC-1 (apply_spec_entry collapsed). After this plan, apply_spec_entry (R/paintr-server.R) contains zero calls to shiny::updateTextInput for the ppText branch, shiny::updateNumericInput, shiny::updateTextAreaInput, or shinyWidgets::updatePickerInput. Verify with this command, which scopes to the post-collapse apply_spec_entry body and counts the four placeholder-keyword update calls (must return 0): 
    Rscript -e 'src <- readLines("R/paintr-server.R"); start <- grep("^apply_spec_entry <- function", src); end_offset <- which(src[start:length(src)] == "}")[1] - 1; body <- src[start:(start + end_offset)]; placeholder_updates <- grepl("updateNumericInput|updateTextAreaInput|updatePickerInput", body) | (grepl("updateTextInput", body) & grepl("ppText", body, fixed = TRUE)); cat(sum(placeholder_updates), "\\n")'

Constraints

  • Files this plan owns:
    • R/paintr-server.R — collapse apply_spec_entry body to: (a) framework-internal cases (layer_checkbox, stage_enabled, source_companion) keep their updateCheckboxInput/updateTextInput calls; (b) placeholder cases (ppText, ppNum, ppExpr, ppVar) write the normalized value into state$spec_seed[[row$bare_id]] (pre-write normalization: ppNum coercion via suppressWarnings(as.numeric(.)[[1L]]) with FALSE on NA-from-non-NA, ppExpr language-to-text via paste(deparse(.), collapse="\n"), ppText character coercion); (c) ppUpload's fileInput dispatch stays a silent TRUE (unsettable), and the existing source_companion handling continues for the dataset-name field. Move the seed-write performed by PLAN-01 inside apply_spec_at_boot to the same place, so all spec normalization lives in one function.
    • tests/testthat/test-adr12-spec-roundtrip.R — revert app$get_value workarounds to app$get_html-based assertions where the widget is now server-rendered via PLAN-01's renderUI; retain app$get_value for the framework-internal geom_point_checkbox (assertion at current line 55) since it stays on updateCheckboxInput.
    • tests/testthat/fixtures/vignette-apps/adr12-spec-roundtrip/app.R — update the "EXCEPT" header comment block to reflect that the ppVar picker is now server-rendered (no more "updatePickerInput from inside the onFlushed callback hits a live widget" claim).
  • Files this plan must NOT touch:
    • Anything PLAN-01 owns (R/paintr-build-ui.R, tests/testthat/test-adr12b-*.R) — PLAN-01 must have merged first.
    • R/paintr-classify.R, R/paintr-walk.R, R/paintr-translate.R, R/paintr-shared.R, the consumer-colvars and adr12-bug-3a fixtures — owned by the parallel Bug A session.
    • R/paintr-placeholders.R — no registry contract change.
  • No new hook type. Do not add an update_value hook to the placeholder registry. The seed path makes one unnecessary; adding one would re-introduce two paths.
  • No mid-stream value validation in renderUI. All spec-value normalization happens once, at boot, before being written into state$spec_seed. The renderUI's role is purely to pass the seeded value through as extra$selected; it must not coerce, parse, or warn.
  • HTML-attribute assertion specificity. SC-3's app$get_html-based assertion must check for the actual attribute markup shinyWidgets / shiny render (verify by capturing a sample from a quick browser-fixture probe before writing the matcher). Do not assume a particular attribute spelling; capture the real output first. Project memory [[browser-probe-pitfalls]] warns about DOM-shape artifacts.
  • SCOPE NARROWING FORBIDDEN. If reverting the picker assertion to app$get_html reveals that shinyWidgets renders the selection in a way app$get_html cannot observe (e.g., the picker's display state is in a sibling JS-driven div), STOP and escalate to the orchestrator. Do NOT keep the app$get_value workaround and claim the plan done; that hides exactly the issue PLAN-02 exists to delete. Open a follow-up plan to investigate the shinyWidgets render path, do not absorb the workaround.
  • Conventions: as PLAN-01.

Definition of Done

Authoritative gate (copied verbatim from CLAUDE.md):

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / WARN 0 / SKIP 2 / PASS ≥ (post-PLAN-01 PASS count). SKIP count must remain exactly 2.

Proxy traps — do not be fooled:

  • A raw Rscript -e 'testthat::test_dir()/test_file()' with no NOT_CRAN silently SKIPs every shinytest2 browser test (skip_on_cran()). Never the gate.
  • devtools::check(... --as-cran) sets NOT_CRAN=true internally; the e2e file cleanly SKIPs via the source-root guard. Use only for the --as-cran NOTE check, never as the test gate.
  • The new SC-3 app$get_html assertion is a stricter check than the SC-2 app$get_value baseline. If SC-2 PASSes but SC-3 FAILs, the seed path is not actually producing server-rendered selection HTML — surface the divergence, do not weaken SC-3.
