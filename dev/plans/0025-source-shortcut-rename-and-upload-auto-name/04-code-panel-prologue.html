PLAN-04 — Code-panel prologue with reader-fn mirror

Source: ADR 0025 §4 · Depends on: 02 · Parallel group: G3 · Status: draft

Context

ADR 0025 §4 specifies that for every active upload-bound source, the code panel renders a prologue line above the formula:

upload_data1 <- read.csv("penguins.csv")  # uploaded as: penguins.csv
upload_data1 |> ggplot(aes(x = mpg)) + geom_point()

The prologue is aspirational, not byte-honest — ggpaintr read the file from a tempfile, not from the original path. The user is responsible for reproducing the binding themselves (running the prologue line in their own R session before invoking ptr_app(formula, spec = ...)). The reader-fn mirrors ggpaintr's actual choice based on file extension (the same dispatch table already in R/paintr-upload.R). Custom-source prologue hooks (prologue_emit_fn) are deferred per ADR scope cuts — current scope is built-in ppUpload only.

Governing principle

The code panel is the slot's authoritative source-of-truth surface — what the user sees there must let them reproduce the rendered plot in a vanilla R session. Without the prologue, an upload-bound formula reading ggplot_1_0_ppUpload_NA |> ggplot(...) in the code panel is unreproducible (no ggplot_1_0_ppUpload_NA in scope). With the prologue, the symbol is bound by a one-line precondition the user can copy verbatim. Naming the WHY: ADR 0025 §3 accepts the symbol-flip on upload only because §4 promises the new symbol will be made legible at a glance — the two decisions stand together.

Success Criteria

  • After uploading penguins.csv to a ppUpload() |> ggplot(...) app, the code panel's text starts with a prologue line of the form <node$auto_name> <- read.csv("penguins.csv") followed by a newline and the formula deparse. Verified by a shinytest2 case test-prologue-csv-upload.R that uploads a csv, clicks Update, and asserts grepl("^[A-Za-z0-9_]+ <- read\\.csv\\(\"penguins\\.csv\"\\)\\n", code_text).
  • Uploading foo.tsv emits <auto-name> <- read.delim("foo.tsv"). Uploading foo.xlsx emits <auto-name> <- readxl::read_excel("foo.xlsx"). Uploading foo.rds emits <auto-name> <- readRDS("foo.rds"). Verified by parameterized shinytest2 cases (one per ext) in test-prologue-reader-fn-mirror.R.
  • Without any active upload (the formula has no ppUpload(...), or upload-bound sources are vacated), the code panel emits NO prologue line — the rendered text is identical to today's behaviour. Verified by test-prologue-absent-without-upload.R.
  • With multiple active upload sources (e.g. a 2-formula ptr_app_grid each with its own ppUpload()), the code panel emits one prologue line per active source, in declaration order, separated by newlines. Verified by test-prologue-multi-source.R.
  • Custom sources registered with shortcut = TRUE but no prologue_emit_fn hook (i.e. all custom sources today) emit NO prologue line — the scope cut explicitly excludes them. Verified by test-prologue-custom-source-skip.R.
  • Full authoritative gate (Definition of Done) shows FAIL 0 / WARN 0 / SKIP 0 with FAIL count for pre-existing tests still 0.

Constraints

  • Files this plan owns:
    • R/paintr-server.R — add a prologue-emission helper (proposed: emit_upload_prologue(state)) that walks state$active_uploads (a new state slot populated by the upload observer at resolve time, storing per-key the tuple (auto_name, file_name, ext)) and emits one line per entry. Modify format_code_with_extras(res, extras_exprs) at line ~2728 to prepend the prologue text to base when non-empty. Modify resolve_upload_source() + try_bind_source_default_resolved() to register the (key → tuple) mapping under state$active_uploads[[key]] on a successful bind, and to clear the entry on a vacated bind (Plan 05 owns the vacate path but this plan adds the slot it writes/clears).
    • Reader-fn dispatch table — REUSE the existing extension→reader-name table at R/paintr-upload.R:240-245 (csv → "read.csv", tsv → "read.delim", rds → "readRDS", xlsx/xls → "readxl::read_excel") rather than introduce a parallel one. If the existing table is private inside R/paintr-upload.R, expose it via a small accessor (e.g. reader_fn_name_for_ext(ext)) returning NA_character_ for unknown extensions (in which case the prologue line is silently omitted for that entry). The accessor lives next to the existing table — DRY across reader-dispatch and prologue-emission.
    • R/paintr-extract.R (or wherever ptr_extract_code lives — if it's in R/paintr-server.R, edit there) — ensure the same prologue is included so ptr_extract_code() returns identical text to the on-screen panel.
    • New tests (shinytest2 fixtures + cases):
      • tests/testthat/test-prologue-csv-upload.R
      • tests/testthat/test-prologue-reader-fn-mirror.R (csv/tsv/xlsx/rds — 4 sub-tests)
      • tests/testthat/test-prologue-absent-without-upload.R
      • tests/testthat/test-prologue-multi-source.R
      • tests/testthat/test-prologue-custom-source-skip.R
      • tests/testthat/test-prologue-two-coordinators-shared-key.R — ADR worked example #2 end-to-end: two ptr_shared(..., id = ...) coordinators on one page sharing key 'ds', asserting distinct eval_env bindings (left_ds / right_ds), distinct code-panel substituted formulas, and distinct prologue lines.
      Each follows the project's app-dir + pkgload::load_all() pattern. Each starts with skip_on_cran(); skip_if_not_installed("shinytest2") + chromote skip. Fixture data files (a tiny csv/tsv/xlsx/rds) live under tests/testthat/fixtures/data/ — use the existing files there if available; otherwise create the minimum needed.
  • Files this plan must NOT touch:
    • R/paintr-upload.R reader dispatch (it already mirrors the four ext cases — this plan reads the same table, does not duplicate or rewrite it).
    • Auto-name derivation (Plan 02), UI mutex (Plan 03), vacate-on-empty + debounce (Plan 05), spec round-trip (Plan 06), docs (Plan 07).
    • Custom-source prologue_emit_fn hook — explicitly deferred by ADR scope cuts.
    • Generated man/*.Rd — integration step.
  • The prologue is plain text (not parsed/eval'd anywhere by ggpaintr). It exists for human reading + manual reproduction. Do NOT also eval it server-side as a "verification" step.
  • The aspirational nature of the original-filename literal is intentional. Do not try to substitute the tempfile path; the user-facing string must read like the actual file the user uploaded.
  • SCOPE NARROWING FORBIDDEN. If a registered file extension is missing from the reader-fn table, the prologue line for that entry is silently omitted (NOT replaced with a fake reader like read.csv); this is the documented default-skip behaviour.
  • Conventions: snake_case, 2-space indent.

BDD scenarios

Feature: Code-panel prologue for upload-bound sources (ADR 0025 §4)

Scenario: csv upload emits read.csv prologue with auto-name
  Given an app with `ppUpload() |> ggplot(aes(x = ppVar('mpg'))) + geom_point()`
   When the user uploads `tests/testthat/fixtures/data/mtcars.csv` and clicks Update
   Then the code panel's text begins with a line matching the regex `^[A-Za-z0-9_]+ <- read\.csv\("mtcars\.csv"\)\n`
    And the auto-name token on the LHS of the prologue equals node$auto_name (i.e. ends with the literal "_ppUpload_NA")
    And the next line is the substituted formula ` |> ggplot(aes(x = mpg)) + geom_point()`

Scenario: tsv upload emits read.delim prologue
  Given the same fixture renamed `mtcars.tsv`
   When the user uploads it and clicks Update
   Then the code panel's first line matches `^[A-Za-z0-9_]+ <- read\.delim\("mtcars\.tsv"\)\n`

Scenario: xlsx upload emits readxl::read_excel prologue
  Given an .xlsx fixture
   When the user uploads it and clicks Update
   Then the code panel's first line matches `^[A-Za-z0-9_]+ <- readxl::read_excel\("[^"]+\.xlsx"\)\n`

Scenario: rds upload emits readRDS prologue
  Given an .rds fixture
   When the user uploads it and clicks Update
   Then the code panel's first line matches `^[A-Za-z0-9_]+ <- readRDS\("[^"]+\.rds"\)\n`

Scenario: no prologue when no upload is active
  Given an app whose formula has no ppUpload call (e.g. `mtcars |> ggplot(aes(x = ppVar('mpg'))) + geom_point()`)
   When the app boots and the user clicks Update
   Then the code panel text does NOT contain " <- read.csv(" or any other reader-fn pattern
    And the text equals the substituted formula alone

Scenario: multi-source app emits one prologue line per active upload
  Given a `ptr_app_grid(list(f1, f2))` where f1 = `ppUpload() |> ggplot(...)` and f2 = `ppUpload() |> ggplot(...)`
   When both sources are populated via upload and Update is clicked
   Then the code panel contains exactly two prologue lines (one per active source)
    And the lines appear in declaration order (f1's first)
    And each line's LHS auto-name equals its node$auto_name

Scenario: ADR worked example #2 — two coordinators on one page bind distinct shared canonicals with no collision
  Given `obj_left  <- ptr_shared(list(f1_left,  f2_left),  id = "left")`
    And `obj_right <- ptr_shared(list(f1_right, f2_right), id = "right")`
    And both f1_left and f1_right contain `ppUpload(shared = 'ds') |> ggplot(aes(x = ppVar('mpg'))) + geom_point()`
    And a shinytest2 app embeds both coordinators on one page via `ptr_ui_shared_panel(obj_left)` + `ptr_ui_shared_panel(obj_right)`
   When the user uploads `mtcars.csv` to the left coordinator's shortcut and `penguins.csv` to the right and clicks each panel's Update
   Then the left  code panel text (read via `app$get_value(output = )`) matches the regex `^left_ds <- read\.csv\("mtcars\.csv"\)\nleft_ds \|> ggplot\(.*\)$` (prologue + substituted formula on the auto-name `left_ds`)
    And the right code panel text (read via `app$get_value(output = )`) matches the regex `^right_ds <- read\.csv\("penguins\.csv"\)\nright_ds \|> ggplot\(.*\)$`
    And neither code panel text contains the literal token of the other coordinator's auto-name (the left panel does NOT contain `right_ds`, and the right panel does NOT contain `left_ds`)
    And the rendered plot output on the left differs from the right (asserting via `app$get_html() != app$get_html()` — both non-empty)

Scenario: custom source without prologue_emit_fn hook emits no prologue
  Given a custom source ppDataset registered with shortcut = TRUE (no prologue_emit_fn)
   When the user picks a dataset via that source
   Then the code panel emits no prologue line for ppDataset
    And only the formula deparse is shown
