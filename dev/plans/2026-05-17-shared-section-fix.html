Plan — Shared section correctness, API, copy & empty-render

Date: 2026-05-17  |  Branch: vignette-review  |  Origin: grill-with-docs session (this conversation)  |  Governing decision: ADR 0005 + CONTEXT.md (shared section/panel partition; ownership clarifications added 2026-05-17)

Problem

A multi-instance app (the user's repro: two ptr_module_ui/ptr_module_server plots + a ptr_shared_panel(obj)) renders the standalone panel's value widget (sz) but the per-formula inline shared sections show no ax1/ax2 picker. A formula-local var(shared = "k") consumer key is emitted as an empty uiOutput that nothing ever fills.

Root cause (traced end-to-end, evidence in session)

  • B1 — no binder for formula-local shared consumers in the embed path. ptr_setup_consumer_uis() (R/paintr-server.R:916–919) deliberately nexts every c$shared consumer, delegating to the host binder ptr_bind_shared_consumer_uis(). ptr_make_app_server (single-instance) calls that binder explicitly (R/paintr-app.R:163–186, 212–223); ptr_module_server never does. So a formula-local consumer key's namespaced uiOutput (<id>-shared_<key>_ui) is never populated. Value keys (e.g. num(shared=)) work because they are static widgets self-bound by ptr_module_server's local_shared reactives.
  • B1b — ptr_shared_server is internally inconsistent. Its shared_reactives correctly restrict to obj$panel_keys, and its comment states formula-local keys are bound by each module — but its consumer block calls ptr_bind_shared_consumer_uis for all consumer_keys at global ns = identity ids (R/paintr-shared-ui.R ptr_shared_server body), i.e. it also (mis)binds formula-local consumers at ids no panel emits. ptr_app_grid's server is structurally identical (ptr_shared_server(obj) + per-plot ptr_module_server), so grid has the same defect.
  • B2 — ptr_module_ui has no shared= parameter (R/paintr-app.R:640); it hardcodes ptr_ui_controls(..., shared = NULL) so obj$panel_keys can never be excluded ⇒ panel keys also render inline (double-render). CONTEXT.md: "routing is governed solely by presence of shared=obj".
  • D5 — misleading copy. Inline hint "One value here is reused everywhere it is referenced" overclaims (the inline section is formula-local; "everywhere" implies cross-plot). Panel hint "linked across every plot below" hard-codes layout ("below") and overclaims ("every plot"; panel keys are the ≥2-formula subset). Both strings are hardcoded literals, not ui_text-resolved.
  • D6 — empty panel renders. shared_panel_body_tag() (R/paintr-shared-coordinator.R:165) always emits the wellPanel + title + hint + Draw-all + errors slot even when zero widgets render (e.g. all keys formula-local ⇒ panel_keys = character(0)): a confusing empty "Shared controls" box.

Verified baseline (do not regress)

Single-instance ptr_app() with var(shared=) is CORRECT today (empirically verified 2026-05-17): collect_shared_placeholders rep id is already canonical (shared_<key>), inline emits shared_ax1_ui (ns identity), ptr_make_app_server's host binder fills shared_ax1_ui → match. The fix must keep single-instance byte-stable.

Why prior tests missed it (coverage gap W1 must close)

The repro is the verbatim vignette chunk l2-shared-partition (vignettes/ggpaintr-use-cases.Rmd:265–295), yet the e2e suite never failed for three compounding reasons: (1) tests/testthat/test-e2e-vignette-examples-shinytest2.R:67–71 explicitly classifies l2-shared-partition as "structurally redundant … represented by … l2-shared" — but the l2-shared fixture's var(shared='metric') is referenced in both formulas (a panel key, the working path), whereas l2-shared-partition's var(shared='ax1') is formula-local (the binder-less path) — the "redundant" judgement collapsed the exact panel-vs-formula-local-consumer ownership axis the bug lives on; (2) the booted shared fixtures assert only structure (expect_dom_id, expect_rendered("#plot","ggplot"), expect_no_inline_error) and the bug is a blank uiOutput whose container still exists, so expect_dom_id passes; no test asserts a formula-local var(shared=) picker is populated; (3) knit-time only constructs shiny::shinyApp(), never runs the server's renderUI. W1 must close this — see W1 Success Criteria / DoD.

Decisions (locked this session)

#Decision
B1Extract one internal helper ptr_bind_local_shared_consumers(tree, output, input, ns, host_owned_keys = character(), …) that performs the assemble-representatives + ptr_bind_shared_consumer_uis call once. Both ptr_make_app_server and ptr_module_server call it. Single binder implementation; the duplicated ~20-line preamble is eliminated.
B1bUniform partition: ptr_shared_server's consumer-UI binding is restricted to panel_keys ∩ consumer_keys (same partition as shared_reactives). Host owns only panel keys; the owning instance owns every formula-local key. No double-bind.
B2ptr_module_ui gains shared = NULL, forwarded verbatim to ptr_ui_controls(shared = shared). Consistent with ptr_shared_panel(obj) / ptr_ui_controls(shared=). NULL ⇒ single-instance (render all keys); obj ⇒ exclude obj$panel_keys from the inline section.
D5Scoped copy + ui_text-overridable. Inline hint → "Drives every place this variable appears in this plot." Panel hint → "Drives every plot that uses it." Titles stay "Shared controls". New shell keys in ptr_default_ui_text(), resolved via ptr_resolve_ui_text("shell", …); overridable through ptr_shared(ui_text=) / ptr_module_ui(ui_text=).
D6shared_panel_body_tag() / ptr_shared_panel() return NULL when zero widgets render. Inline shared_section_tags() keeps its existing self-NULL. Losing the cross-module "Draw all" in the no-panel-keys case is accepted. ptr_shared_server(obj) must remain a safe no-op when panel_keys is empty and the ptr_shared_errors/ptr_shared_draw_all ids are absent.

No new ADR: all four implement the already-locked ADR 0005 / CONTEXT.md contract or are reversible low-lock-in choices. CONTEXT.md "Relationships" + "Flagged ambiguities" updated this session.

Build order

W2 → W1 → W4 → W3. W2 (mechanical) makes the inline exclude correct so W1 can be validated against the right widget set. W1 is the core fix. W4 depends on W1's partition being correct (empty-panel only well-defined once ownership is uniform). W3 is independent (copy) and lands last. Each item is independently testable and shippable.

Integration discipline scopeindependenceordering

Scope & ownership (item 5). Each work item owns exactly the files listed in its section. No W edits another W's files. Generated artefacts are integration-deferred: NAMESPACE and man/ are not merge-resolved — after all W merge, the integrator runs devtools::document() once on the integrated branch and commits the regenerated output. A W's own commit may include its devtools::document() output for local DoD, but conflicts in NAMESPACE/man/ at merge are resolved by regeneration, never by hand.

Ordering & revertable integration (item 9). Build order W2 → W1 → W4 → W3. Each W is implemented in its own git worktree off vignette-review (the base), independently testable. The user merges one W at a time, --no-ff, so any single W is revertable as one merge commit. W1 depends on W2 (correct exclude set) and W4 depends on W1 (uniform partition); a dependent W is rebased on the base only after its blocker has merged. W3 is independent and may merge in any slot.

Amendment (2026-05-17). Revertability is required at commit granularity, not worktree granularity: W1 stays solo (own worktree, own --no-ff merge, own gate run — high blast radius); W4 + W3 are batched into one worktree, one commit each, a single --no-ff merge and one authoritative-gate verification. Each W remains independently revertable (git revert <W3-sha> / <W4-sha>) and the build order is unchanged. This satisfies revertable integration at commit granularity; a separate per-unit worktree/merge is reserved for steps whose blast radius warrants it (W1).

Independence (item 6). Any "done / suite green / scope clean" from a W worktree session, sub-agent, or a prior turn is the reporter's claim, produced by the same fallible process — it is evidence for what to check, never proof. The integrator does not trust a status block: for every W merge the integrator independently re-runs the cross-cutting authoritative gate command (NOT_CRAN=true … test_dir) on the merged result and confirms FAIL 0 / ERROR 0 / SKIP 0 with the browser scenarios actually executed (not SKIPped — see proxy traps), plus checks the W's Success Criteria and Constraints against the diff, before accepting the merge.

Surface, don't absorb (item 8). Any defect, inconsistency, or scope question discovered while implementing a W that is not already covered by that W's Success Criteria is surfaced as a tracked follow-up (a new issue file under .scratch/<feature>/ per dev/agents/issue-tracker.md) and reported in the merge summary — it is never silently fixed inside the W (scope creep) nor papered over to make the gate pass. If a discovered defect blocks the W's DoD, stop and escalate; do not widen scope unilaterally.
