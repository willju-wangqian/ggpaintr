PLAN-04 — Host-side ptr_setup_panel_sources()

Source: ADR 0023 · Depends on: 01, 02 · Parallel group: G3 · Status: draft

Context

ADR 0023 §2 (Decision): "A new internal helper ptr_setup_panel_sources(obj, input, output, envir): for each panel-owned source key, assigns the renderUI for the uiOutput(shared_<key>_ui) container that shared_panel_body_tag() already emits; builds a reactive that watches the top-level input[[shared_<key>]] (fileInput) and the companion input[[shared_<key>_name]], runs the source registry's resolve_data hook, and falls back to try_bind_source_default() when the upload is empty and the source carries a default_arg resolvable through the host's envir parent chain; surfaces upload errors into the panel's ptr_shared_errors slot; the companion textInput (shared_<key>_name) is rendered once at panel scope."

ptr_shared_server() lives in R/paintr-shared-ui.R. The new helper runs inside that function, after the existing shared_reactives block, and populates the new panel_sources field that Plan 01 added to the bundle.

Governing principle

Ownership equals binding: whoever renders the surface is also the server binder. CONTEXT.md (locked 2026-05-17) names this for value and consumer roles; ADR 0023 extends it to the source role. The shared panel's shared_panel_body_tag() already emits the uiOutput(shared_<key>_ui) container for panel-owned source keys, so the panel server must be the binder for that renderUI — sending it to a per-instance binder would re-create the orphan-output / multi-writer problem the partition rule exists to prevent. The host's job is also just the panel keys; formula-local source keys stay on the per-instance path (decision §7 enforces this in ptr_setup_source_uis).

Success Criteria

  • A new internal helper ptr_setup_panel_sources(obj, input, output, envir) exists in R/paintr-shared-ui.R (file-scope, internal — no @export). It returns a named list of shiny::reactive values, one per panel-owned source key, keyed by canonical id (shared_<key>), suitable for assignment to the bundle's panel_sources field.
  • ptr_shared_server() calls ptr_setup_panel_sources(obj, input, output, envir) and passes the returned list to new_ptr_shared_state(..., panel_sources = ...). The single observable change to ptr_shared_server()'s return value is that state$panel_sources is now populated (was an empty list per Plan 01's default).
  • For each canonical id shared_<key> where key is in obj$panel_keys AND the corresponding placeholder is a data-source role (the panel-owned subset of obj$firsts$nodes whose nodes satisfy is_ptr_ph_data_source): the helper assigns output[[paste0("shared_", key, "_ui")]] <- shiny::renderUI({...}) using the same renderUI body shape as ptr_setup_source_uis() uses per-instance (the same invoke_build_ui call with the source registry's build_ui hook, registered upload file_copy / name_copy copy injection, and the same outputOptions(..., suspendWhenHidden = FALSE) mount).
  • The reactive value the helper returns for each panel-owned source id calls resolve_upload_source(input_slot, companion_slot, node, entry, envir, state, key, slot) (Plan 02's helper) with the top-level input slot names shared_<key> / shared_<key>_name (no namespacing — the panel uses global ids). For the host's purposes the state / key / slot args address a host-local reactiveVal the helper owns; the reactive's value tracks that reactiveVal.
  • Upload errors surfaced from resolve_upload_source() reach a host-owned error sink (a new internal ptr_shared_errors reactiveVal inside ptr_shared_server()) symmetrically with consumer-side errors. Verified by triggering an entry whose resolve_data throws and reading the error sink.
  • For a config with zero panel-owned source keys (the most common shape today), the helper returns list() and state$panel_sources remains an empty list — i.e., no observable change for existing tests.
  • All criteria verified by NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)' → FAIL 0 / ERROR 0 / SKIP 0 / PASS rises by ≥ 8 (the eight new testServer scenarios in tests/testthat/test-host-setup-panel-sources.R).

Constraints

  • Files this plan owns: R/paintr-shared-ui.R (the new helper + the ptr_shared_server() wiring of the new return-field), tests/testthat/test-host-setup-panel-sources.R.
  • Files this plan must NOT touch: R/paintr-server.R (Plans 02/05/06/07 own the per-instance side), R/paintr-shared-coordinator.R (Plan 03 owns the parser invariant).
  • The helper does NOT call ns(): panel ids are global / un-namespaced per CONTEXT.md ("the shared panel uses global, un-namespaced ids shared_<key>").
  • The helper must NOT re-implement upload-format detection / default-arg fallback / binding-name derivation — it calls resolve_upload_source() from Plan 02.
  • The shared_panel_body_tag() container for source-role panel keys MUST already exist at output[[shared_<key>_ui]]; if the implementer finds the container is not currently emitted for source-role keys, they pause and file a follow-up rather than modifying shared_panel_body_tag() in this plan.
  • SCOPE NARROWING FORBIDDEN — see the implementer/auditor prompts in /exec-plan.
  • Tidyverse style, snake_case, 2-space indent, rlang::abort() for errors.

BDD scenarios

Feature: ptr_shared_server populates panel_sources for panel-owned source keys

Scenario: Helper returns empty list when no panel-owned source keys exist
  Given obj <- ptr_shared(c(
    "ggplot(mtcars, aes(x = ppVar(shared='x'))) + geom_point()",
    "ggplot(mtcars, aes(x = ppVar(shared='x'))) + geom_col()"
  ))
  When testServer runs a server containing
       s <- ptr_shared_server(obj)
  Then s$panel_sources is identical to list()

Scenario: Helper returns a reactive per panel-owned ppUpload source key
  Given obj <- ptr_shared(c(
    "ggplot(ppUpload(shared='ds'), aes(x = ppVar(shared='col'))) + geom_point()",
    "ggplot(ppUpload(shared='ds'), aes(x = ppVar(shared='col'))) + geom_col()"
  ))
  When testServer runs s <- ptr_shared_server(obj)
  Then names(s$panel_sources) is c("shared_ds")
  And  shiny::is.reactive(s$panel_sources$shared_ds) is TRUE

Scenario: Container renderUI is assigned for the panel-owned source key
  Given the same obj as above
  And  the panel body emits uiOutput("shared_ds_ui")
  When testServer runs ptr_shared_server(obj)
  Then output$shared_ds_ui (after wait_for_idle) yields a renderUI body that
       is a tagList containing a Shiny fileInput element with id "shared_ds"

Scenario: Default-arg fallback primes panel_sources at boot
  Given df_main <- mtcars
  And  obj <- ptr_shared(c(
    "ggplot(ppUpload(df_main, shared='ds'), aes(x = ppVar(shared='col'), y = mpg)) + geom_point()",
    "ggplot(ppUpload(df_main, shared='ds'), aes(x = ppVar(shared='col'), y = hp)) + geom_col()"
  ))
  When testServer runs s <- ptr_shared_server(obj, envir = environment())
  And  the reactive context flushes
  Then shiny::isolate(s$panel_sources$shared_ds()) is identical to mtcars

Scenario: Upload error surfaces into the host's error sink
  Given obj has a panel-owned source with a registry entry whose resolve_data throws
  When the test sets input$shared_ds = list(datapath="bad", name="x.csv")
  And  the reactive context flushes
  Then the host's ptr_shared_errors slot contains a message naming the source key
       and the error text

Scenario: ADR worked example #6 — custom source keyword inherits panel-scope wiring
  Given a custom keyword registered via ptr_define_placeholder_source("ppDataChooser",
        build_ui = function(node, ...) shiny::selectInput(node$id, "Choose", c("mtcars","iris")),
        resolve_data = function(value, node, ...) get(value, envir = globalenv()))
  And  obj <- ptr_shared(c(
        "ggplot(ppDataChooser(shared='ds'), aes(x = ppVar(shared='col'))) + geom_point()",
        "ggplot(ppDataChooser(shared='ds'), aes(x = ppVar(shared='col'))) + geom_col()"))
  When testServer runs s <- ptr_shared_server(obj)
  And  input$shared_ds_name <- "mtcars"
  And  input$shared_ds <- "mtcars"
  And  the reactive context flushes
  Then shiny::isolate(s$panel_sources$shared_ds()) is identical to mtcars
       (the factored resolve_upload_source helper dispatches via the registry's resolve_data hook — no per-keyword wiring required)

Scenario: ADR worked example R3 — spec-seeding the fileInput with a data frame is silently skipped
  Given a panel-owned ppUpload(shared='ds') and spec = list(shared_ds = mtcars)
  When testServer boots the app with the spec
  And  the reactive context flushes
  Then no error is signalled (apply_spec_at_boot's ppUpload-role skip fires)
  And  shiny::isolate(input$shared_ds) is NULL (the fileInput value was not assigned)
  And  shiny::isolate(s$panel_sources$shared_ds()) is NULL
       (the placeholder-role row was dropped; only source_companion / shared__name rows are honored — per the path documented at R/paintr-server.R:574–575, 588–590)

Scenario: ADR worked example R4 — panel still emits the fileInput tag even when panel_sources is supplied
  Given my_df <- shiny::reactiveVal(mtcars)
  And  shared_state = list(panel_sources = list(shared_ds = my_df)) injected into ptr_shared_server
  When the host renderUI for output[[shared_ds_ui]] fires
  Then the rendered HTML contains an <input type="file"> element with id "shared_ds"
       (the escape-hatch reactive overrides the resolved value but does NOT replace the emitted widget shape)
