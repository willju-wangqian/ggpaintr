PLAN-02 — Extract resolve_upload_source() helper

Source: ADR 0023 · Depends on: none · Parallel group: G2 · Status: draft

Context

ADR 0023 §3 (Decision): "The csv/tsv/rds/xlsx detection, the resolve_data-hook dispatch, the default-arg fallback (eyeball-play try_bind_source_default, commit a3d1636), and the binding-name derivation today live inside ptr_setup_pipelines(). Extract them into an internal resolve_upload_source(input_slot, companion_slot, node, entry, envir) helper that returns the resolved df + binding name. Per-instance pipeline calls it with namespaced slots; host calls it with top-level slots. One resolver, two callers — any future format addition (Parquet, Arrow) is made once."

The current code in R/paintr-server.R has two near-identical observer bodies in ptr_setup_pipelines(): one at lines ~916–993 (bare-data-source layers) and one at lines ~1004–1064 (pipeline-head sources). Both compute the same file_info → df → binding_name shape. This plan extracts that mechanics into one helper. No behaviour change for the existing call sites; Plan 04 adds the third caller (host scope).

Governing principle

Upload-format support and the default-arg fallback are a single mechanism with multiple callers, not two parallel mechanisms that happen to look alike. The ADR's Consequences section calls this out: "Upload format support (csv/tsv/rds/xlsx and any future addition) and the default-arg fallback live in one function called from both scopes. Removes the divergence-risk vector that motivated Q4-I in the grilling session." An implementer tempted to leave the duplicated bodies in place "because they work" misses the WHY: the divergence-risk this refactor exists to eliminate is exactly the kind of bug that won't surface in tests until a future format is added in one place and not the other.

Success Criteria

  • A new internal helper resolve_upload_source(input_slot, companion_slot, node, entry, envir, state, key, slot) exists in R/paintr-server.R and returns invisible(NULL) after calling either bind_source_value() (when an upload is provided OR the default-arg fallback resolves) or slot(NULL) + set_resolve_error() (otherwise). The signature accepts the namespaced input/companion ids the caller already computed; the helper does NOT call ns() itself.
  • Both existing call sites inside ptr_setup_pipelines() (bare-data-source layer loop at line ~916 and pipeline-head source loop at line ~1004) call the helper instead of inlining the resolve mechanics. The two shiny::observe({ ... }) bodies shrink to roughly shiny::observe({ resolve_upload_source(src_id, comp_id, src, entry, state$eval_env, state, ln_or_sid, slot) }).
  • Behaviour is byte-equivalent at the boundary: the existing test suite (every test that exercises ptr_setup_pipelines, including test-shared-source-rendering.R, test-upload-companion-autofill.R, and every browser fixture that uploads a file) continues to PASS unchanged.
  • A new focused unit test (tests/testthat/test-resolve-upload-source-helper.R) exercises the helper directly with a stubbed registry entry: (a) file_info = NULL + node$default resolvable in envir → calls bind_source_value with the resolved df; (b) file_info populated → entry$resolve_data dispatched, bind_source_value called with returned df; (c) entry$resolve_data throws → set_resolve_error called with the message and slot(NULL); (d) invalid binding name (non-syntactic via make.names) → bind_source_value called with name = NULL.
  • All criteria verified by running NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)' and observing FAIL 0 / ERROR 0 / SKIP 0 / PASS rises by ≥ 4 (the four unit scenarios above) and existing PASS count is preserved exactly.

Constraints

  • Files this plan owns: R/paintr-server.R (only the source-resolution observer bodies in ptr_setup_pipelines() and the new helper definition), tests/testthat/test-resolve-upload-source-helper.R.
  • Files this plan must NOT touch: R/paintr-shared-ui.R (Plan 01, Plan 04), R/paintr-shared-coordinator.R (Plan 03), any vignette or fixture.
  • The helper is internal — not exported, no roxygen @export tag. A short header comment names the call sites.
  • Pure refactor: no new observable behaviour, no new logged messages, no new error class, no signature change visible to embedders. try_bind_source_default() stays as-is (the helper calls it directly).
  • SCOPE NARROWING FORBIDDEN — see the implementer/auditor prompts in /exec-plan.
  • Tidyverse style, snake_case, 2-space indent.

Feature: Single resolver for upload-source mechanics, two call sites

Scenario: Helper exists and is internal
  Given the package loaded via devtools::load_all
  When ggpaintr:::resolve_upload_source is queried
  Then it is a function whose formals match the names
       input_slot, companion_slot, node, entry, envir, state, key, slot

Scenario: NULL file_info with resolvable default-arg binds the resolved frame
  Given a stub state with eval_env containing df_main = mtcars
  And  a node with default = "df_main" and a registered upload entry
  And  input slots where input[[input_slot]] is NULL and input[[companion_slot]] is ""
  When resolve_upload_source(...) is called
  Then bind_source_value is called with name = "df_main" and df = mtcars
  And  the slot is set to mtcars

Scenario: Populated file_info dispatches entry$resolve_data and binds the result
  Given input[[input_slot]] = list(datapath = "<csv-file>", name = "penguins.csv")
  And  a registry entry whose resolve_data returns penguins
  And  input[[companion_slot]] = "penguins"
  When resolve_upload_source(...) is called
  Then bind_source_value is called with name = "penguins" and df = penguins
  And  set_resolve_error is called with NULL (cleared)

Scenario: entry$resolve_data error surfaces via set_resolve_error and clears the slot
  Given a registry entry whose resolve_data throws "boom"
  And  input[[input_slot]] = list(datapath = "<file>", name = "x.csv")
  When resolve_upload_source(...) is called
  Then set_resolve_error is called with message containing "boom"
  And  slot(NULL) is called

Scenario: Invalid binding name yields NULL name passed to bind_source_value
  Given input[[companion_slot]] = "1 not a name" (fails make.names)
  And  input[[input_slot]] = list(datapath = "<file>", name = "x.csv")
  And  entry$resolve_data returns mtcars
  When resolve_upload_source(...) is called
  Then bind_source_value is called with name = NULL and df = mtcars
