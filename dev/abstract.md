# ggpaintr — Academic abstract

*Draft intended for paper, vignette intro, JOSS submission, or talk abstract. Derived from `dev/high-level-picture.md`. Under 400 words.*

---

Building a Shiny application around a ggplot2 chart forces an author to describe the same chart three times: once as UI widgets in one file, once as server-side reactive plumbing in another, and once as the ggplot expression whose value slots must be hand-wired back to those widgets. The three copies drift, small edits ripple across files, and an otherwise one-line chart becomes an exercise in bookkeeping. This friction is why many R users fluent in ggplot2 never publish their analyses as interactive artifacts, and why Shiny developers pay the same boilerplate cost on every chart.

**ggpaintr** is an R package that collapses this authoring surface. Its design is a small compiler pipeline. The source language is a ggplot2-like formula string in which value slots may be replaced by placeholder keywords: `var` for a column picker, `text` for free text, `num` for a number, `expr` for an arbitrary R expression guarded by a denylist-protected AST walker, and `upload` for a runtime dataset. `ptr_parse_formula()` compiles this string into a single intermediate representation that every downstream subsystem consumes — the Shiny UI builder, the reactive runtime, the live generated-code pane, the `ui_text` localization merger, and user-defined placeholder hooks. Authors describe the chart once, inline, and ggpaintr derives the widgets, the reactive wiring, and the rendered plot from the shared representation.

The package exposes this pipeline at three integration levels. Level 1 — `ptr_app()` and `ptr_app_bslib()` — turns a formula into a running Shiny application in a single call, for teaching, demos, and ggplot2-fluent users who do not wish to write Shiny code. Level 2 provides embedding helpers (`ptr_input_ui()`, `ptr_output_ui()`, `ptr_server()`, the granular `ptr_register_*()` functions, and a four-layer `ui_text` override system) that let ggpaintr's widgets and runtime live inside an existing Shiny application under custom element ids and copy. Level 3 exposes a headless parse / exec / extract API (`ptr_exec()`, `ptr_extract_plot()`, `ptr_extract_code()`) that supports batch reporting, programmatic code generation, and user-owned `renderPlot()` blocks with no Shiny session. Extensibility is first-class: `ptr_define_placeholder()` lets teams register new widget types that participate in the same formula DSL as the built-ins.

ggpaintr is developer-oriented in its ecosystem positioning but serves two audiences — Shiny developers reducing per-chart boilerplate, and ggplot2 users seeking the shortest path from a chart expression to a shareable interactive artifact. Future work focuses on compatibility with adjacent publishing and runtime platforms, in particular Quarto and browser-only execution via `shinylive`.
