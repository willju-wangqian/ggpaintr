ADR 0016 — All super-app examples must be Path-B (naked-ggplot) evaluable

Status: Proposed. Implementation deferred — do not action until all ADR-0013 plans (including PLAN-03) have closed on post-add-expr. This ADR locks the principle; the patch is a follow-up.

Date: 2026-05-24

Relates to: ADR-0013 (super-app pressure-test suite), ADR-0009 (pp* placeholders and expression input).

Context

A ggpaintr formula that contains pp* placeholder calls can be evaluated by two distinct paths:

Path A — ptr_app(...) parse path.
The substitution layer walks the formula's AST, finds each pp*(...) node, registers it in the typed tree, and never invokes the runtime function(x, ...) x. Argument values are read from the call's AST shape, not from R-level evaluation. Missing arguments and unbound data symbols are tolerated — the parser only inspects structure.
Path B — naked ggplot evaluation.
The formula is evaluated by R as a plain expression. Each pp*(arg) executes its runtime form (typically function(x, ...) x) and returns its argument. ggplot composes the result. This is what happens when a user pastes the formula into the REPL or a script without a ptr_app() wrapper.

Path A is what the super-app fixture tests exercise. Path B is the user's discoverability surface: a reader who copy-pastes a super-app formula expects it to render as a plain ggplot in their own R session. That is one of ggpaintr's central value propositions — the same formula works inside ptr_app() and outside it.

Problem discovery (2026-05-24)

A scan of the super-app fixtures surfaced two Path-B breakers:

  • super-4 (merged at b9a189a) — tests/testthat/fixtures/vignette-apps/super-4-user-css-safety-adversarial/app.R:64 contains ppColor() with no argument. The comment at line 27 documents the intent: exercise build_ui's NULL-selected arrival path. Under Path A this is fine. Under Path B, ppColor() errors because the runtime form is function(x, ...) x and x has no default.
  • super-2a (PLAN-03, not yet merged) — the canonical formula contains ppMultiVar() with no argument. Same Path-B failure mode.

The other three super-apps (super-1, super-2b, super-3) pass every pp* call with an argument and are Path-B compatible modulo data-symbol binding.

Decision

Every super-app fixture's canonical formula MUST evaluate as plain ggplot under a Path-B harness. Concretely:

  1. The formula, when wrapped in a Path-B harness that
    • binds the data symbols the formula names (e.g. df_main, df_aux) to representative data frames, and
    • ensures every pp* the formula calls is in scope (built-ins via the package namespace; custom placeholders defined in the fixture file before the formula),
    MUST execute to completion and produce a valid ggplot object. No error, no missing-arg abort.
  2. No naked pp*() call (no argument) appears in the canonical formula. Every placeholder call carries a positional default value that is itself a valid ggplot input for the slot it occupies.
  3. If a fixture wants to test the NULL-arrival code path for a specific placeholder (as super-4 does for ppColor's build_ui), that test uses a separate side-formula that is explicitly NOT covered by the Path-B guarantee. The canonical formula stays Path-B clean.

This becomes a per-plan acceptance criterion: every plan that adds or modifies a super-app fixture under ADR-0013 must include a Path-B evaluation check in its Definition of Done.

Consequences

What this changes

  • super-4's canonical formula gets a positional default for ppColor (e.g. ppColor("#1f77b4") or a value the build_ui would naturally seed). The NULL-arrival code-path test moves to a side-formula or to a unit test of ppColor's build_ui directly.
  • super-2a's canonical formula gets a positional default for ppMultiVar when PLAN-03 lands. The default should be a value aes(group = ...) can consume — typically a column symbol or an interaction(...) expression.
  • A new helper tests/testthat/helper-path-b.R (or extension of helper-super-pressure.R) provides a single function that takes a fixture path, sources its formula, binds the named data symbols, and asserts the formula evaluates to a ggplot object. Each super-app test file calls this helper once per fixture.

What this enforces

  • Custom placeholders defined in fixtures (and, by extension, in user code) need a runtime form that handles missing arguments sensibly when the fixture's formula relies on a no-arg call. In practice this means either supplying a positional default in every formula site, or — as the registry-API follow-up — adding a default_value = argument to ptr_define_placeholder_value() / _consumer() / _source() so the auto-generated runtime fn carries that default. That registry-API decision is not locked here; see Open Questions.
  • The principle generalises beyond super-apps: a vignette example or a README snippet that pastes into a user's R session should also evaluate. ADR-0016 scopes the decision to super-apps only; extension to vignettes/README is left as an open question for a future ADR.

What this does NOT do

  • Does NOT require the runtime function(x, ...) x default to be changed package-wide. The change is per-fixture and per-call-site, not at the registry level.
  • Does NOT require any data-symbol binding outside the harness. Path B for a formula referring to df_main obviously needs df_main bound; the harness provides that.
  • Does NOT modify ADR-0013's already-merged scope. Plans 02, 04, 05, 06 stay as they landed. The Path-B follow-up patches them in a single later commit.

Implementation notes (deferred)

Do not action this ADR until ADR-0013 PLAN-03 lands on post-add-expr and the seven-plan bundle is fully closed. The follow-up patch should:

  1. Add tests/testthat/helper-path-b.R (or extend helper-super-pressure.R) with a expect_path_b_evaluable(fixture_path, bindings) helper that:
    • Reads the fixture's canonical formula via a source-grep of its app.R (the formula lives between the ptr_app( opener and its matching closer, modulo !!-spliced variables).
    • Binds the named data-symbol bindings in a fresh environment.
    • Calls eval(parse(text = formula)) in that environment.
    • Asserts the return is a ggplot S3 object.
  2. Patch super-4's canonical formula to give ppColor a positional default. Move the NULL-arrival test to a side-formula or a unit test of ppColor's build_ui.
  3. Patch super-2a's canonical formula (whichever shape PLAN-03 settles on) to give ppMultiVar a positional default.
  4. Wire the expect_path_b_evaluable helper into the existing super-pressure test_that blocks for super-1, super-2a, super-2b, super-3, super-4 — one assertion per fixture.
  5. Run the authoritative gate on the merge tip. Expect PASS to grow by 5 (one new assertion per super-app).

Open questions (for the follow-up patch session)

  1. Registry-level default_value = knob. Section "What this enforces" lists this as a possible API addition: ptr_define_placeholder_value(keyword = ..., default_value = "") auto-generates function(x = "", ...) x as the runtime form, removing the need to pass a default at every call site. Decision deferred to the follow-up session — implementer should weigh declarative-knob convenience vs the per-call-site explicitness that Path B already requires.
  2. Extension to vignettes / README. The Path-B principle arguably applies to any code example shipped with the package. Should the helper also run against vignette code chunks? Decision deferred; if yes, write ADR-0017 to extend.
  3. Treatment of !!-splice in canonical formulas. Several super-apps use !!formula1 to splice a stored formula expression into ptr_app(). The Path-B harness must resolve the splice before evaluation. Decision: the harness reads the underlying formula expression, not the post-splice text — the helper should accept a quoted-expression argument as well as a fixture-path argument.

Rejected alternatives

  • Change the default runtime form to function(x = NULL, ...) x package-wide. Would make every no-arg pp*() return NULL, which crashes ggplot deep in aes() evaluation with a much worse error than a missing-arg abort. Rejected — silent NULL propagation hides the problem instead of surfacing it.
  • Drop the Path-B principle and document that super-app formulas are ptr_app()-only. Rejected — it would amputate ggpaintr's discoverability story. The whole point of the placeholder DSL is that the same formula reads as ordinary ggplot.
  • Add Path-B coverage to every plan in ADR-0013 retroactively, blocking PLAN-03's merge. Rejected — it would re-open already-merged plans and stall PLAN-03. Better to patch all super-apps in one focused follow-up after ADR-0013 closes.

Acceptance criteria for the follow-up patch

  • Authoritative gate on the post-patch tip: FAIL 0 / WARN 0 / SKIP 0 / PASS N with N ≥ baseline + 5 (one Path-B assertion per super-app).
  • Every super-app fixture's canonical formula evaluates to a ggplot S3 object under the Path-B harness.
  • The follow-up patch does NOT modify already-merged ADR-0013 plans' BDDs or DoDs. The Path-B assertions are additional coverage.
  • The follow-up patch carries a /implementable PASS stamp.
