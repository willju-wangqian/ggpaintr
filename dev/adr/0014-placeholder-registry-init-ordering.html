ADR 0014 — Placeholder-registry init ordering for custom-placeholder constructors

Date: 2026-05-23 · Owner: ggpaintr · Status: DECIDED & APPLIED · Cross-refs: ADR 0013 (Follow-up C of the post-mortem) · supersedes the temporary workaround at tests/testthat/fixtures/vignette-apps/super-1-kitchen-sink/app.R:10–15

One-line summary. The three public placeholder constructors (ptr_define_placeholder_value, ptr_define_placeholder_consumer, ptr_define_placeholder_source) bypass the lazy registry init path, so registering a custom placeholder before any read from the registry silently locks out the five built-ins. Fix is one line per constructor: call ensure_registry_initialized() first.

1. Defect

The lazy init for .ptr_registry (R/paintr-registry.R:8–16) has two short-circuits:

ensure_registry_initialized <- function() {
  if (isTRUE(.ptr_registry_initialized$done)) return(invisible(NULL))
  .ptr_registry_initialized$done <- TRUE
  if (length(ls(.ptr_registry)) > 0L) return(invisible(NULL))   # ← the trap
  if (exists("ptr_register_builtins", mode = "function")) {
    ptr_register_builtins()
  }
  invisible(NULL)
}

The length(ls(.ptr_registry)) > 0L guard exists so that if some other code path has already pre-populated the registry, init will not re-register builtins on top. It assumes whoever populated the registry has already done init themselves.

All three public constructors (R/paintr-registry.R:412, 508, 623) skip init and go straight to ptr_registry_register(entry). Read-side helpers (ptr_registry_keywords, ptr_registry_lookup, ptr_clear_placeholder, see lines 47, 91, 99) do call ensure_registry_initialized(). That asymmetry is the bug.

Reproduction

devtools::load_all(".")
ptr_registry_clear()
ptr_define_placeholder_value(
  keyword = "ppFoo",
  build_ui = function(node, ...) shiny::textInput(node$id, "x"),
  resolve_expr = function(value, ...) value
)
ptr_registry_keywords()
# expected: c("ppVar","ppText","ppNum","ppExpr","ppUpload","ppFoo")
# observed: c("ppFoo")

Downstream symptom (observed in the super-1 fixture before the workaround at app.R:15 was added): ui_text$defaults validation aborts with "ui_text$defaults has unsupported entries: ppNum, ppText".

2. Decision — Shape A (conservative)

Add ensure_registry_initialized() as the first statement of each of the three constructors, before any validation. Leave the length(ls(.ptr_registry)) > 0L empty-guard intact as belt-and-suspenders.

Three one-line additions. No behavior change for any callsite that already reads the registry before registering a custom placeholder.

Rejected: Shape B (drop the empty-guard)

Once the constructors unconditionally init, the empty-check is redundant — the done flag covers every reachable path. Dropping it would be slightly cleaner.

Why not. Grep confirms only R/paintr-registry.R ever writes to .ptr_registry (no other source or test does), so Shape B would have been safe today. But the empty-guard is cheap and survives future refactors where a new write path might be added without going through init first. The cost of Shape A is one extra line check per registry read; the cost of Shape B breakage is silent double-registration of builtins. Conservative wins.

Implementation note: recursion guard in ptr_register_builtins()

Shape A introduces a subtle re-entry hazard. ptr_register_builtins() is implemented as a sequence of public-constructor calls (ptr_define_placeholder_value("ppText", …), etc.). After Shape A, each of those calls invokes ensure_registry_initialized(). On a freshly cleared registry (done=FALSE, length(ls)==0), the FIRST constructor re-enters ptr_register_builtins() from inside init, registers all five builtins, then returns to the outer call which still proceeds to ptr_registry_register(entry) on a now-populated env — emitting one "Overwriting placeholder registry entry" warning per builtin (5 per call). Test suites that clear-and-rebuild the registry produced 70 such warnings before this guard.

The fix is one line at the top of ptr_register_builtins():

ptr_register_builtins <- function() {
  .ptr_registry_initialized$done <- TRUE   # ← break the recursion
  text_fn <- ptr_define_placeholder_value(...)
  ...
}

Setting done=TRUE before any constructor fires ensures ensure_registry_initialized() short-circuits inside the recursive entry — the constructors then proceed straight to ptr_registry_register() on a still-empty env, registering each builtin exactly once with no warnings.

3. Consequences

  • Public API: no change. All three constructors keep the same signature, return value, and validation order. The added ensure_registry_initialized() call is idempotent (the done flag short-circuits on every subsequent call).
  • Test fixtures: the no-op ptr_registry_keywords() call at tests/testthat/fixtures/vignette-apps/super-1-kitchen-sink/app.R:10–15 is removed in the same change. It was load-bearing only because of this bug.
  • Regression coverage: a new tests/testthat/test-regression-bugs-2026-05-23.R pins the exact defect for each of the three constructors, an idempotency check (calling twice does not double-register builtins), and a no-recursion-warnings check that fails if the done=TRUE guard at the top of ptr_register_builtins() is removed.
  • Forward-compat: if a future contributor adds a fourth public registry writer, they must follow the same pattern. The empty-guard remains as a safety net but should not be relied on.
