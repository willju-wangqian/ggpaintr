ADR 0019 — C3c: Public staged pipeline (drop the 222 :::)

Status: Proposed. Promoted from ADR-0017 C3 grill session on 2026-05-24. Independent of ADR-0018; can land first.

Date: 2026-05-24

Parent: ADR-0017 (deepening opportunities catalog), candidate C3c.

Relates to: ADR-0018 (C3a — orthogonal; either may land first), ADR-0012 (canonical typed-tree decision).

Context

The translate→render pipeline has nine files (nodes, walk, translate, classify, prune, substitute, resolve, eval, render) and only two public entry points: ptr_translate(formula, ...) and ptr_eval / ptr_render.

Tests routinely need to assert that an intermediate-stage tree has a specific shape (e.g. "after ptr_prune, this disabled layer is gone"; "after ptr_substitute, this placeholder has been replaced with a literal"). Because no public path reaches the intermediate stages, tests reach in with ggpaintr:::.

Verified facts (2026-05-24)

  • 222 ggpaintr::: lines across 21 test files in tests/testthat/.
  • Files affected include test-adr12-consumer-uniformity.R, test-adr12-spec-roundtrip.R, test-classify-calls-lift.R, test-expression-mode.R, test-ppupload-name-e2e.R, test-ppupload-preserve-mode.R, test-prune-bare-pipe-stage.R, test-ptr-spec-emission.R, test-ptr-tree-structural-equal.R, test-regression-bugs-2026-05-12.R, test-regression-bugs-2026-05-13.R, test-rename-coverage.R, test-render-prefix-collapse.R, test-resugar-pipeline.R, test-rewrite-server-state.R, test-shared-server-partition.R, test-shared-stage-checkbox.R, test-ui-pieces.R, test-upload-unsupported-extension.R, test-validate-input-ctx.R, test-validate-input-retain.R.
  • The pass functions themselves (ptr_classify_data, ptr_prune, ptr_substitute, ptr_render) are unexported today — that's the only reason tests use :::. Their bodies are stable contracts in the project's internal vocabulary.

Problem statement

222 ::: calls is sustained, visible evidence that the public surface doesn't fit the project's testing reality. Beyond the aesthetic cost, every ::: use is a soft contract: changing an internal name silently breaks tests. The package gives no public way to ask "what tree does this formula produce after substitution?"

This is shallow-interface shape: the pipeline has a clearly-named intermediate-stage tree (the typed AST per ADR-0012), but no public way to get one at a chosen stage.

Decision (direction, not interface)

Add a single public staged-pipeline entry point — provisional name ptr_pipeline(formula, ..., stop_after = c("translate", "classify", "prune", "substitute", "render")) — that runs the pipeline up to the named stage and returns whatever that stage's natural output is (a typed tree for the intermediate stages; the rendered text for "render"). No behavioural change; pure redirection plus tests.

Migrate the 21 test files from ggpaintr:::ptr_X(tree) to ptr_pipeline(formula, stop_after = "X") where shape-equivalent. Where the test needs to inject a hand-built tree (rather than start from a formula), keep ::: — those are testing internal contracts, not pipeline behaviour, and that's a fair use.

Out of scope for this ADR: the exact return-type contract per stop_after value; whether stop_after = "translate" already exists as ptr_translate (it does — they may converge); whether to expose ptr_pipeline as a single-arg-defaults convenience (e.g. default stop_after = "render" ≡ ptr_render). These are interface design questions for the grilling session.

Consequences

What this buys

  • 222 ::: uses become ptr_pipeline(...) calls. The public surface and the test surface align. Internal renames stop silently breaking tests.
  • The intermediate-tree question ("what shape does this formula take at stage X?") gets a public, documentable answer. Useful for embedders too (an L3 user could inspect the resolved tree before letting ggpaintr render it).
  • The pipeline gains one named, documented entry point that exposes its full staged shape — better than the current two-entry-point façade that hides everything in between.

What this does NOT buy

  • No reduction in pass-file size or boilerplate (that is C3a / ADR-0018).
  • No type-level guarantee that a wrong-stage tree can't reach a wrong consumer (that is C3b).
  • No new behaviour — pure redirection layer.

Risks

  • Public-API commitment. Once ptr_pipeline is exported, its return-type contract per stop_after is load-bearing. If C3a or C3b later changes intermediate-stage shapes, this entry point's contract is the bottleneck.
  • Migration tedium. 222 sites across 21 files. Most are mechanical, but some tests reach for genuinely-internal helpers (not pass entry points), and those must stay on :::. Triage required.
  • Vignette / docs pressure. An exported entry point usually demands a roxygen page + example. That's small per CRAN's standards but non-zero.

Open questions for the grilling session

  1. Convergence with ptr_translate. stop_after = "translate" would do exactly what ptr_translate already does. Either deprecate ptr_translate in favour of ptr_pipeline, or keep both as documented synonyms. Pick.
  2. Return-type per stage. Always a tree for non-render stages? Or per-stage natural type (rendered text for "render", an R language object for "eval")?
  3. Stage names. Match the file names (classify, prune, substitute) or use more user-facing names (typed, active, resolved)? CONTEXT.md vocabulary should govern.
  4. Coverage of the 222 sites. A quick triage: which ::: calls are pass entry points (migratable) vs internal helpers (must stay :::)? Eyeball before locking the migration plan.
