ADR 0018 — C3a: Eliminate per-pass S3 walker boilerplate

Status: Proposed. Promoted from ADR-0017 C3 grill session on 2026-05-24. Direction sketched; interface design and migration plan deferred to a fresh-session grilling (handoff produced separately).

Date: 2026-05-24

Parent: ADR-0017 (deepening opportunities catalog), candidate C3a.

Relates to: ADR-0012 (tree is semantic, not syntactic — typed node classes locked).

Context

The ggpaintr formula pipeline runs four mutating passes over the typed AST: classify, prune, substitute, render. Each pass lives in its own file (R/paintr-classify.R, R/paintr-prune.R, R/paintr-substitute.R, R/paintr-render.R) and each rolls a complete S3 dispatch tree of 11–13 methods — one per node type (ptr_root, ptr_layer, ptr_pipeline, ptr_call, ptr_ph_value, ptr_ph_data_consumer, ptr_ph_data_source, ptr_user_expr, ptr_literal, ptr_closure, ptr_missing, default).

The ~80% common shape across these dispatch trees is the same "recurse into children, rebuild a new node from rewritten children" boilerplate. The ~20% variation is the per-pass per-node-type payload — the actual work each pass does at that node type.

Meanwhile, R/paintr-walk.R already provides generic walker primitives (ptr_walk, ptr_collect, ptr_rewrite_pre, ptr_cursor*) that are widely used elsewhere — build-ui.R (4 call-sites), server.R (4 call-sites), disable.R, ids.R, translate.R. The four mutating passes alone bypass these primitives in favour of open-coded dispatch trees.

Verified facts (2026-05-24)

  • Per-pass S3 method counts: classify = 9 methods; prune = 13; substitute = 13; render = 13 (+ a parallel stamp_current_pick_walk S3 tree of 10 methods in the same file).
  • Generic walker primitive usage in R/: ptr_walk = 6 call-sites; ptr_collect = 6 call-sites; ptr_rewrite_pre = 2 call-sites (disable.R, ids.R). The primitives ARE load-bearing — this ADR does NOT propose removing them.
  • Validators (ptr_assert_ids_assigned, ptr_assert_classified, ptr_assert_no_placeholders, ptr_assert_no_missing) are enforced in production: ptr_translate lines 106–107, ptr_eval lines 13–14. These ARE the canonical-form gates — already in place, not duplicated.
  • Sibling candidates: C3b (tree subtypes) and C3c (public staged pipeline) are tracked separately. C3c is promoted to its own ADR (0019); C3b needs an ADR-0012 conformity review before promoting.

Problem statement

The four mutating passes share a structural axis (per-node-type dispatch) and a payload axis (what to do at each node type), but the codebase only factors the latter. Adding a new node type — say ptr_some_new_thing — requires touching four pass files in lockstep, each time writing the same descend-on-children plumbing. Failure to update all four silently degrades passes (the default S3 method is invoked, often producing a structurally wrong tree without error).

This is shallow-module shape: each pass file contains ~80% boilerplate that has no per-pass meaning. The interface (a "pass") is large (13 method bodies) for the implementation it carries (one per-node-type payload table).

Decision (direction, not interface)

Introduce a single pass-driver primitive — provisional name ptr_pass(tree, handlers) — that owns descent over all node types. Per-pass code reduces to a handlers table (per-node-type payloads only). Each of classify.R / prune.R / substitute.R / render.R becomes the pass's handlers table plus an entry point that calls ptr_pass.

Out of scope for this ADR: the exact handlers contract (whether handlers see children pre- or post-descent; how to express "rebuild this node with these new children"; how passes that need cursor state — e.g. in_data_position — express that). These are interface design questions to be resolved in the grilling session.

Consequences

What this buys

  • Adding a new node type touches one place: the node constructor in paintr-nodes.R + one handler row per pass. Today: 5 files in lockstep, with a silent-failure mode via the default dispatch.
  • Each pass file shrinks substantially. Rough sketch: render.R's 47 functions / 636 lines could fall to ~25 functions / ~250 lines if the boilerplate goes; combined four-file reduction is on the order of 600–800 LOC.
  • Per-pass handler tables are easier to read as a unit — the "what does prune do at a layer?" answer is one cell in a table, not method-hunting across a file.

What this does NOT buy

  • No tree-subtype safety (that is C3b). After C3a, a wrong-stage tree still flows past a wrong consumer with a runtime assertion failure, not a type error.
  • No public test interface to intermediate stages (that is C3c).
  • No reduction in the number of pass files — same four, just thinner.

Risks

  • Subtle dispatch differences. Some passes may need pre-order vs post-order traversal, sibling state, or specific re-entry semantics. The pass-driver interface must expose those needs explicitly; ignoring them silently produces wrong-tree bugs.
  • S3 dispatch loss. If handlers is a named list keyed by class string rather than R's native S3, downstream extension by class is harder. The interface must either preserve method-dispatch semantics or document the trade-off.
  • Wide blast radius. Touches the four largest pipeline files. Must land behind the authoritative gate (NOT_CRAN=true Rscript -e 'testthat::test_dir(...)' → FAIL 0 / ERROR 0 / SKIP 0) with the existing 1583+ tests unchanged.
  • Boilerplate may not be boilerplate. If even 2 of the 4 passes need pass-specific behaviour at, say, the ptr_pipeline level (handling pipe ops vs nested calls differently), the "common" shape is smaller than it appears and the ROI shrinks. Worth quantifying during grilling.

Open questions for the grilling session

  1. Handler contract. Does a handler see (a) the node before descent, decide what to do; or (b) the node post-descent with already-rewritten children, just shape the output? Or both, depending on the pass?
  2. Cursor / context threading. disable.R uses ptr_rewrite_pre precisely because it needs the in_data_position cursor state. Can the four mutating passes be expressed without that cursor, or does ptr_pass need to expose it?
  3. Pre/post-order. Render is bottom-up by nature (children's text composes parent's text). Substitute and classify likely top-down. Can one driver express both, or do we need ptr_pass_pre + ptr_pass_post?
  4. The render.R twist. That file already houses TWO walker trees (render_walk + stamp_current_pick_walk). Does C3a collapse them both, or are they genuinely two passes that happen to colocate?
  5. Migration order. Migrating the simplest pass first (likely classify, 9 methods, 103 lines) lets the driver mature before tackling render. Or migrate render last to maximise leverage. Pick.
  6. Don't-break-it gate. Definition of Done is the authoritative gate from CLAUDE.md — FAIL 0 / ERROR 0 / SKIP 0. Pre-refactor: snapshot the pass-output trees for a representative set of formulas; post-refactor: byte-identical. Anything else is not done.

Definition of Done (for the eventual implementation plan)

Per CLAUDE.md authoritative gate:

NOT_CRAN=true Rscript -e 'suppressMessages(devtools::load_all(".")); testthat::test_dir("tests/testthat", reporter="progress", stop_on_failure=FALSE)'

Expected output: FAIL 0 / ERROR 0 / SKIP 0 / PASS N (current N = 1583+).

Plus: round-trip snapshot equivalence (pre-refactor pass outputs == post-refactor pass outputs) for a curated formula set spanning all node types and the super-app fixtures (per ADR-0013).
