The story, in plain language

ggpaintr lets a placeholder be shared — the same key (e.g. x) drives one widget that controls several spots, possibly across several plots. The framework must decide, for every shared key and every UI surface, a single yes/no question:

"Does THIS surface render and bind THIS shared key?"

That one question is currently answered three different, asymmetric ways depending on which embedding level you're in:

  1. Multi-formula (the coordinator). ptr_shared() builds a ptr_shared_spec object carrying panel_keys (keys referenced by ≥2 formulas → owned by the one shared panel) and local_keys_by_formula (per-formula inline keys). Downstream code reads these fields. Source: R/paintr-shared-coordinator.R — shared_partition() at line 28, panel_keys <- all_keys[formula_counts >= 2L] at line 42, the object assembled around lines 287–310.
  2. Single-formula L1. There is no coordinator (CONTEXT.md, locked: "single-instance has no coordinator — ever"). Instead a bare boolean render_shared_section = TRUE stands in for "this surface owns ALL shared keys." Source: R/paintr-app.R:169 (set TRUE for ptr_app), and the consuming branch shared_section <- if (isTRUE(render_shared_section)) around R/paintr-app.R:425 (also :760, :339 where it is FALSE / host_owned_keys = character(0)).
  3. L2 ptr_server. The complement is computed as a set difference: host_owned_keys = setdiff(consumer_keys, panel_keys) and passed into ptr_bind_local_shared_consumers(). Source: R/paintr-shared-ui.R:388; consumer R/paintr-server.R:3365 (ptr_bind_local_shared_consumers).

Same conceptual question, three signals that must be kept mutually consistent by hand. When they drift, the bug is "a shared widget renders but never binds" or "binds twice." This is not hypothetical: the 2026-05-22 partition-binder bug (project memory project-shared-section-binder) lived in channel #3 — a surface computed its host_owned_keys wrong and a formula-local var(shared=) went blank.

Why do this (motivation)

  • It is a recurring-bug class, not a tidiness itch. The ownership question has already produced at least one shipped defect; every new shared-key surface is a fresh chance to compute it wrong because there is no single function that answers it.
  • Locality. One place would answer "does surface S own key K?" for every surface. A new surface gets no "compute it correctly" step to get wrong.
  • Test leverage. "L2 omits panel-owned keys" / "single-formula owns all keys" become unit assertions against the view, not full-app boots.
  • Public-API-safe. This is an INTERNAL seam. No exported function changes signature or behavior — the lowest-risk category of the parked items, which is why ADR 0026 ranked it the strongest remaining candidate.

Proposed shape (direction, not locked)

Introduce an internal partition_view(part, surface_id) (name negotiable) that every channel queries instead of re-encoding ownership:

  • Multi-instance keeps building the coordinator object as today, but the consumers ask the view "owns(surface, key)?" rather than re-deriving from panel_keys.
  • Single-formula L1 constructs a trivial internal view that owns all keys (replacing the render_shared_section = TRUE boolean).
  • L2 server's host_owned_keys becomes a lookup against the view, not a separate setdiff the caller wires up.

Optionally fold in a validate_partition_invariant() fail-fast at ptr_shared() construction (merged from the dropped partition-rule-ownership-verification candidate).

Boundary / scope

  • Internal only. No public API change. ptr_shared/ptr_shared_panel/ptr_shared_server/ptr_app behaviour is frozen (CONTEXT.md locked).
  • Do not change the partition algorithm (shared_partition) — it is already centralised and correct; this candidate is about the consumption side (the three channels).
  • Re-grep first: ADR 0017 flagged the L2 host_owned_keys call-site count as UNVERIFIED. Count the real call sites before locking the view's interface.
  • Respect ADR 0006: at coordinator HOST scope, ptr_bind_shared_consumer_uis runs with state=NULL; the view must work in that mode (it should, since it queries the partition object, not state).
