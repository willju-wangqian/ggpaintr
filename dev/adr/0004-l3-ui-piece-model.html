ADR 0004 — L3 UI-piece model, single ptr_ui_page() shell, and the assets-internal supersession

Status: accepted · Date: 2026-05-16 · Supersedes: dev/plans/2026-05-15-css-unification.html §2 row "Bundle export: internal" (for the L3 path only) · Related: ADR 0002 (public-API tiering), ADR 0003 (controllable-region unification) · Implementation plan: dev/plans/2026-05-16-l3-piece-model.html

Context

The typed-AST rewrite's L3 narrative was redefined to "own every piece of ggpaintr's public UI": one exported builder per pane (ptr_ui_plot/ptr_ui_error/ptr_ui_code/ptr_ui_code_toggle/ptr_ui_controls/ptr_ui_header), composed by hand and wired with the existing server API. Field use of the first cut (commit 822ba35) exposed that (1) bare pieces under a non-Bootstrap root render the layer pickerInput unstyled; (2) the hand-written div(class="ptr-app") + ptr_ui_assets() ritual is a non-obvious footgun; (3) making each piece self-wrap is unsafe — .ptr-app carries min-height:100vh with no nested/sibling neutralization, so the canonical sibling layout would stack N viewport-heights; and (4) ptr_register_plot/error/code are exported but unusable without the internal-only runtime, i.e. dead, mis-documented surface.

Resolving (2)+(3) requires a single exported page shell that owns the one .ptr-app scope and the asset bundle, with bare pieces dropped inside it. That shell, plus the pre-existing need for an escape-hatch asset injector on non-fluidPage roots, conflicts with an already-implemented decision: the css-unification plan (in-tree, not ADR-formalized) §2 locked "assets bundle stays internal; users never need to inject ggpaintr assets manually; the public surface should shrink, not grow". The L3 "own every piece" model invalidates that decision's premise — hand-composition for non-Bootstrap-page roots is a legitimate manual-injection need that did not exist when only the self-wrapping bundle entry points did.

Decision

  • Add exported ptr_ui_page(..., page = shiny::fluidPage, css = NULL) as the single L3 shell: it is the Bootstrap page, owns the one .ptr-app div, and injects the asset bundle. Pieces stay bare (no per-piece .ptr-app/assets/css). Per-piece self-wrap is rejected (Defect 3); a .ptr-app "shell-vs-scope" CSS split is rejected (visual-regression risk to every existing app).
  • page= contract: any Bootstrap-3 page builder whose ... are tag children (fluidPage default, fixedPage, fillPage, bootstrapPage, basicPage). navbarPage (positional title + tabPanel children) and bslib/BS5 pages are out of contract; their supported path is the documented hand-decomposition recipe.
  • Supersede css-unification §2 for the L3 path: ptr_ui_assets() stays exported as the manual-injection escape hatch for out-of-contract roots. This is a conscious, recorded reversal of the "assets internal" lock — justified by the new L3 requirement, not drift. ptr_assets() remains the internal bundle name; the internal emitter renamed ptr_ui_assets→core_assets_tags (commit 822ba35) is retained; the css-unification plan's §3.1 prose is stale and carries a superseded banner (not rewritten — ADR/plan immutability).
  • Assets become a deduped htmltools::htmlDependency() so nesting shells/pieces injects one <head> copy; the ad-hoc JS registration guards are deleted.
  • Un-export ptr_register_plot/error/code (dead post-rewrite surface); delete the L2 "Splitting outputs further" subsection that documented an impossible use case.
  • ?ptr_css and every [ptr_css()] link are deliberate and stay. An earlier conclusion to "fix the dangling links" was wrong (it is a roxygen-only token-doc topic, R/paintr-css-doc.R). Recorded here, mirroring ADR 0002's "do not fix the missing function back" hazard.
  • All renames are deferred to the API-audit pass (dev/plans/api-audit.md). Doing them piecemeal pre-empts holistic tiering and double-churns just-rewritten docs.
Locked naming contracts (the audit must honor, not re-litigate):
  • ptr_module_ui/ptr_module_server are a deliberate sealed-bundle naming exception — not folded into the ptr_ui_*/ptr_* grammar.
  • ptr_ui_outputs ≡ ptr_ui_plot_code(error = TRUE) — an intentional discoverability alias (different default DOM), not a duplicate to be deduped away. Post-audit names: ptr_ui_plot_code (default error=FALSE), ptr_ui_outputs (error nested).
  • Post-audit server family: ptr_server, ptr_shared_resolve (from ptr_shared_server), ptr_module_server. UI family: ptr_ui_*.

Consequences

  • The public surface grows by ptr_ui_page() and retains ptr_ui_assets() — a deliberate, ADR-recorded exception to ADR 0002 / css-unification's "shrink, not grow" stance, scoped strictly to the L3 hand-composition requirement. It shrinks by three (ptr_register_*), net.
  • Breaking for anyone on this dev cycle's snapshot: ptr_register_* gone; the L3 hand-div.ptr-app recipe replaced by ptr_ui_page(). No deprecation (single user, pre-1.0).
  • "Exported but its required runtime companions are internal" remains a portal smell (ADR 0002) — ptr_register_* was a fresh instance and is removed accordingly.
  • css-unification's §2 assets-internal row is now historically inaccurate; it is superseded-in-part, banner-marked, not rewritten. Future readers must treat ADR 0004 as the live decision for assets exposure.
  • The API-audit pass owns all renaming, the long-term fate of ptr_ui_assets() exposure, and the §2.1/locked-contracts execution.
