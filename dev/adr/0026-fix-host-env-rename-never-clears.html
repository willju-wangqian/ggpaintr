ADR 0026 fix — host env→env shortcut rename never clears the consumer picker

Status: Open bug — fresh-session handoff. Source-confirmed this session (2026-05-30), NOT test-proven, NOT fixed. This is the immediate next item (the user chose to fix this before tackling the broader cross-file consolidation, ADR26-followup-cross-file). Context-rebuild doc; verify every file:line with find_symbol (paintr-server.R drifts).

Date: 2026-05-30. Relates to: ADR 0025 §3b (the consumer-clear-on-new-source contract), ADR 0024 (the shortcut is a data-loading entry point — typing an env name loads that frame), ADR 0023 + ADR 0006 (panel-owned sources; state=NULL at host scope).

The bug, in plain language

Setup: a shared data source whose widget is panel-owned (rendered once at host scope), used as an env shortcut — i.e. the user types a dataset name into the shortcut textbox and ggpaintr get()s that data.frame from the app environment (ADR 0024). A downstream consumer picker (ppVar) offers that frame's columns.

The user types ChickWeight, picks column weight. Then the user renames the shortcut to a different env dataset (e.g. PlantGrowth) that also happens to have a weight column. Per ADR 0025 §3b, a new user-supplied source is "new work" and the stale pick should clear. It does not. The picker keeps weight from the old frame, silently riding onto the new data.

This is an iceberg: the single-instance (non-host) path does NOT have this bug, and the prior ADR-0025-#7 latch work made the host UPLOAD case clear, so it looks fixed. Only the host env→env rename (no file upload at either end) is broken.

Exact mechanism (root cause)

The consumer's new-source clear is edge-triggered on an identity string built by consumer_upstream_source_state() (R/paintr-server.R:2575):

ids <- c(ids, paste0(sid, "#", dp, "#", bn))   # server.R:2621
# identity = paste(ids, collapse = "||")

where, per source node:

  • dp = the fileInput datapath (fv$datapath). For an env shortcut with no file upload, dp = "".
  • bn = the bound dataset name, read from state$bound_names[[sid]] (with a layer_name fallback) — server.R:2596–2615.

The clear fires in consumer_clear_for_new_source() (server.R:2651) via:

changed_since_last <- src_seen && !identical(identity, last_identity) && isTRUE(user_supplied)

At HOST scope the binder runs with state = NULL (ADR 0006 design; the host call is ptr_bind_shared_consumer_uis(...) with state omitted, and it threads that NULL straight into consumer_upstream_source_state(upstream_src_nodes, read_src_input, state, ...) at server.R:3281). So state$bound_names is NULL → bn = "" permanently.

For an env→env rename: dp = "" at both ends (no file) AND bn = "" (host scope). So identity = "shared_ds##" is constant across the rename → changed_since_last is always FALSE → the clear never fires. The ADR-0025-#7 cleared_for_identity latch (server.R:3300–3308) cannot help — it only re-asserts an existing clear edge across the upload double-render; the base clear never trips, so there is nothing to hold.

Why single-instance is immune: there state is real, so bn reads the live bound name, which flips when the rename binds a new frame → identity changes → clear fires.

Fix candidates (evaluate, don't assume)

The identity at host scope must change when the env name changes. Two candidate fixes; a fresh session must decide which is correct and prove it:

  1. Plan-of-record: host-aware bound name. Give consumer_upstream_source_state() an optional way to compute the bound name at host scope (e.g. a bound_name_for = function(node) ... param) that the host binder supplies, reading the panel binding via panel_owned_binding_name(node, entry, session, shortcut_value, df) / panel_source_canonical_name(node) (server.R:1101/:1105) instead of the NULL state$bound_names. Then bn reflects the renamed env name and identity flips. This is the fix the original ADR-0025-#7 plan named but skipped.
  2. Simpler candidate to evaluate: include the shortcut value in the identity. consumer_upstream_source_state() ALREADY computes sc (the trimmed shortcut textbox value) at server.R:2616–2619 but deliberately does not include it in the identity string (only sid#dp#bn). Adding sc would make an env rename flip the identity directly. Risk to investigate: there was presumably a reason sc was left out (it may cause spurious clears in some single-instance/spec-seed case, or double-count with bn when the live bind already captures it). Do not add it blindly — check why it's excluded and whether including it regresses the §3b "boot spec-seed must NOT clear" contract.

Whichever path: the change must be gated to host scope (is.null(state)) or otherwise proven not to alter single-instance identity, because single-instance is currently correct and must stay so.

Boundary / scope

  • Host scope only. Do not change single-instance behaviour (it is correct). Gate on is.null(state).
  • Do not break the §3b "boot data must seed" contract: boot env-default data (e.g. ppUpload(mtcars) where the shortcut == its default at boot) is neither user_supplied nor uploaded, so the default SHOULD seed — the fix must keep that (an env name that equals the default at boot is not "new work").
  • Do not regress the host UPLOAD clear (ADR-0025-#7) or the spec-seed-wins-at-boot path.
  • This is a bug fix, deliberately scoped narrower than the cross-file consolidation (ADR26-followup-cross-file); fix the bug first, consolidate later.

⚠ Test traps (these will give you a FALSE GREEN — read before writing the e2e)

  • Use a column present in BOTH frames (e.g. weight / body_mass_g). A selectInput/pickerInput silently drops a selection that is no longer a valid option — so a test using a column unique to the OLD frame "passes" because the option vanished, not because the clear fired. (verified finding overlapping-column-false-green-trap.)
  • The fileInput pill is harness-blind — but irrelevant here: this bug is env→env (no file), which is fully automatable (typing → re-render → env-load → picker populated/cleared is observable via get_value/get_html). No manual eyeball needed, unlike the #7 upload-pill case.
  • Boot an app DIRECTORY that pkgload::load_all()s (the tests/testthat/fixtures/vignette-apps/<slug>/app.R pattern, GGP_PKG env) — a parent-built shiny.appobj resolves to the stale installed ggpaintr and silently tests dead code (verified finding e2e-fixture-must-pkgload-source).
  • Consumer var pickers are suspended until their upstream has data AND the layer's Controls subtab is shown — set source first, then set_inputs(<layer>_subtab="Controls"), wait_for_idle(), then read the picker.
  • Run the gate serially — the seeding/derived-column browser cluster flakes under parallel load (verified parallel-gate-flakes-seeding-cluster); a residual parallel flake here is not a regression.
