ADR 0005 — L2/L3 redesign: ptr_<x> vs ptr_ui_<x> convention, shared coordinator, output combinators

Status: accepted · Date: 2026-05-16 · Supersedes: ADR 0004 in full (the L3 UI-piece model and all its locked naming contracts); the deferred-rename list in dev/plans/api-audit.md as it stood before this date; the ADR-0002 tiering rows for the shared/outputs/controls families. · Related: ADR 0002 (public-API tiering), ADR 0003 (controllable region) · Domain language: CONTEXT.md (authoritative; all sections marked LOCKED 2026-05-16) · Implementation plan: dev/plans/api-audit.md (rewritten this date)

Context

ADR 0004 defined L3 as "own every piece" with bare ptr_ui_* builders, a single ptr_ui_page() shell, and self-wrapping composites (ptr_controls_ui/ptr_outputs_ui/ptr_shared_ui) coexisting at L2. A grill-with-docs audit of that plan (2026-05-16) found the model under-specified and internally inconsistent on three independent axes:

  • Layer conflation. Every *_ui → ptr_ui_* rename in the old api-audit list silently moved a self-wrapping composite into the bare-piece namespace (ptr_controls_ui, ptr_outputs_ui, ptr_shared_ui all self-wrap .ptr-app+assets). The plan never distinguished the two layers.
  • Shared model was wrong. "Shared" was treated as inherently multi-formula; in fact a single formula can use shared keys (an inline section), and the multi-formula panel is a different surface with global ids and a top-level server. No code path partitioned keys by cross-formula occurrence — the desired routing did not exist.
  • Outputs were non-orthogonal. ptr_ui_plot carried error=/code_toggle= flags and a prebuilt ptr_ui_code_toggle composite duplicated them, contradicting "own every piece."

The shell-burden objection (multi-instance ⇒ multiple ptr_ui_page() calls; L2 users should not have to think about .ptr-app) made "unify on one shell" unacceptable. The conclusion was a structural redesign of the L2/L3 boundary, not a rename pass.

Decision

1. Naming convention (global)

ptr_<x> = L2 self-contained (owns its own .ptr-app + asset bundle; drop into the host app; no shell to remember). ptr_ui_<x> = L3 bare (emits only its widgets; the L3 user supplies their own Shiny / optional ptr_ui_page). This rule governs the entire API. The ptr_embed_* idea explored mid-audit is rejected.

2. Levels

  • L1 — ptr_app, ptr_app_grid, ptr_app_bslib (unchanged).
  • L2 — embed with the default layout. Entry: ptr_module_ui/ptr_module_server (the sealed-bundle exception keeps its established names; not read as generic ptr_<x>). Plus the multi-instance shared trio (below). The non-module split ptr_controls_ui/ptr_outputs_ui is removed.
  • L3 — own the layout / render path. Bare ptr_ui_* pieces + combinators; optional ptr_ui_page. Custom render is L3 (the enabling state is returned at the L2 server boundary, but using it to replace a pane is owning the render path). The server side is unchanged from L2; a piece with no UI is a no-op.

3. Shared model

Partition rule: a shared key referenced in exactly one formula renders in that formula's inline shared section (folded into ptr_ui_controls / the module sidebar, controls-namespaced, driven by that formula's own server). A key referenced in ≥2 formulas renders in the one standalone shared panel (global ids, top-level ptr_shared_server).

  • One instance ⇒ no coordinator; all shared keys auto-render inline. No panel, ever.
  • Multiple instances ⇒ the user builds a coordinator once: obj <- ptr_shared(formulas, shared_ui, ui_text, expr_check, draw_all_label) — a pure, non-reactive object that computes the partition and is the single source of truth. Consumed by exactly three functions, which take only obj: ptr_shared_panel(obj, css) (L2 self-contained panel), ptr_ui_shared_panel(obj) (L3 bare panel), ptr_shared_server(obj) (reactive bundle, unchanged downstream into ptr_module_server(shared_state=)).
  • The old ptr_module_ui(shared_panel=) boolean is removed; routing is governed solely by presence of shared=obj + the count-based partition.

4. Outputs — orthogonal pieces + combinators

ptr_ui_plot(id) / ptr_ui_error(id) / ptr_ui_code(id, style) are truly bare (no behavior flags). Behavior is added by nestable combinators: ptr_ui_inline_error(plot, error), ptr_ui_toggle_code(plotish, code). These are pure DOM-structure helpers with no server coupling — the bundled JS (inst/www/ggpaintr-ui.js) drives the toggle DOM-locally off .ptr-output/.ptr-code-toggle/.ptr-code-window classes. The default L2 layout (plot + inline error + toggle code) lives only inside the rebuilt ptr_module_ui.

5. ptr_module_ui rebuilt on the L3 API

ptr_module_ui/ptr_module_server keep their names and external contract (DOM ids ptr_plot/ptr_error/ptr_code preserved) but are reimplemented as an internal composition of bare pieces + their own self-contained shell + combinators + (single-formula) the auto inline shared section.

Superseded — do NOT carry forward (a future reader will find these in old docs):
  • ADR 0004's locked contracts in full: ptr_ui_outputs ≡ ptr_ui_plot_code(error=TRUE); the enumerated bare-piece set; ptr_shared_server → ptr_shared_resolve; ptr_ui_code_toggle → ptr_ui_plot_code. None of these survive. ptr_shared_server keeps its name.
  • All intermediate names from the 2026-05-16 audit conversation: ptr_embed_controls/outputs/shared, ptr_ui_inputs, include_error_panel, the "keep both layers / fix names" framing. Void — only the design in this ADR + CONTEXT.md stands.
  • project-headless-removed memory's "custom rendering demoted to an L2 subsection" clause. Only that clause — the no-headless / no-testServer-as-a-feature prohibition in that same memory still stands.

Rejected alternatives

  • Unify on one shell (L2 routed through ptr_ui_page): rejected — multi-instance ⇒ multiple page calls; imposes shell awareness on L2 users who should never see .ptr-app.
  • Per-key partition override (force a count-1 key into the panel within a multi-formula app): rejected — re-introduces a UI/server disagreement vector; the only lever is whether a coordinator exists.
  • Single-instance "promote to standalone panel for prominence": rejected — one instance is always inline; keeps code paths minimal.
  • Keep flags + add combinators, or keep the self-wrapping composite layer: rejected — the dual-path maintenance cost the audit set out to remove.
  • Expose state$server_ns_fn/ui_ns_fn as a public custom-output escape hatch: rejected — state is server-side only (unusable from UI); the canonical "ptr_server inside the user's own moduleServer(id)" pattern makes ns automatic and these unnecessary. They stay internal plumbing.

Consequences

  • This is a breaking, structural change with no deprecation (single user, pre-1.0). The public surface is re-shaped, not merely renamed.
  • The cross-formula partition algorithm is net-new — collect_shared_placeholders() is single-tree today; nothing counts cross-formula occurrence. New logic + dedicated tests.
  • UI↔server partition consistency is a correctness invariant: ptr_shared (UI) and ptr_shared_server (server) must derive identical partitions from the same obj; ptr_module_server's shared_state path must mirror only formula-local keys (the panel owns cross-formula keys via the top-level server).
  • Large downstream fallout, each its own implementation pass (enumerated in dev/plans/api-audit.md): the use-cases vignette, the LLM topics, README L-level summary, and the testthat suite (test-ui-pieces, test-rewrite-app, test-asset-bundle, test-user-css, shared tests, copy tests).
  • The "exported but its runtime companions are internal" portal smell (ADR 0002) is explicitly honored: server_ns_fn/ui_ns_fn remain non-public.
  • CLAUDE.md's "no separate CONTEXT.md" statement is now false and is corrected; CONTEXT.md is the authoritative domain language for this model.
