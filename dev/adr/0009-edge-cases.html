ADR 0009 — edge cases for review

Date: 2026-05-20 · Source ADR: dev/adr/0009-pp-placeholders-and-expression-input.html · Plans audited: dev/plans/placeholder-rename-expr-mode/

These are cases where ADR 0009 is silent or ambiguous — not cases already pinned down. Each lists candidate behaviors so you can pick one; my lean is marked "Recommended". After you decide each, I'll fold the resolutions into ADR 0009 (and propagate any plan changes), restamp, and re-audit.

Cases are grouped by area. Skim the area headers to triage which ones matter most.

A · Defaults & validators

A1Unary minus / scientific in numeric defaults

Examples: ppNum(-5), ppRange(c(-1, 1)), ppNum(-2.5e3).

In R's AST, -5 is NOT a literal — it's a call (`-`)(5). 2.5e3 IS a literal (numeric). -2.5e3 is again a call.

ADR is silent. The literal-only helpers as drafted would reject any unary-minus default, which is surprising.

Candidates:

A1.a (Recommended). Extend the numeric validators to accept unary - / + over a numeric literal: pattern-match (`-`)(n) and (`+`)(n) where n is a literal, return -n / +n. No eval; deterministic. ppNum(-5) and ppRange(c(-1, 1)) both work.
A1.b. Reject all non-literal numerics, including -5. User must use ppNum(5) for negatives via a wrapper or just live without negatives. Maximally pure; surprising.
A1.c. Eval the AST inside a safety-walked sandbox to allow arbitrary literal arithmetic (ppNum(1 + 1) too). Bigger surface, opens translate-time eval the ADR deliberately closed.

A2Non-literal numeric: ppNum(1 + 1)

Strictly: 1 + 1 is a call, not a literal. Literal-only validators reject.

ADR implies reject (literal-only), but doesn't say so explicitly.

A2.a (Recommended). Reject. Error message: "default for ppNum must be a numeric literal; got an expression." Documented in the helper's roxygen.
A2.b. Accept any constant-folded literal arithmetic by evaluating in an empty environment. Opens translate-time eval — same risk as A1.c.

A3Empty string default: ppText("")

An empty string is technically a string literal but conveys "no default."

ADR is silent.

A3.a (Recommended). Accept. Treat as a real default with value ""; the textarea boots empty. No semantic difference from omitting the arg, but doesn't surprise the author.
A3.b. Reject with "empty string is not a meaningful default; omit the positional instead." Tighter contract.

A4Backticked / whitespace symbols in ppVar

Example: ppVar(`my column`). as.character(quote(`my column`)) returns "my column".

ADR is silent; ptr_default_symbol_or_string() would accept this by mechanical pass-through.

A4.a (Recommended). Accept. ggplot already lets users write aes(x = `my col`); ppVar follows the same convention. The default stored on the node is the string "my column".
A4.b. Reject backticked names; require ppVar("my column") (string form) for non-syntactic column names. Tighter; loses parallelism with ggplot.

A5ppExpr(quote(x + 1)) — user pre-quotes the expression

A user trying to be careful writes ppExpr(quote(x + 1)). The validator's AST input is the call quote(x + 1), not x + 1.

ADR is silent; ppExpr's "store verbatim" rule would store the quote(x + 1) call as-is, which renders as quote(x + 1) in the textarea — confusing.

A5.a (Recommended). ppExpr's validator detects quote(...) / bquote(...) / expression(...) wrappers and unwraps once: the inner expression becomes the default. Documented as a usability ergonomic.
A5.b. Store verbatim and document the gotcha: "do not wrap in quote(); ppExpr already captures unevaluated."

B · Shared-key interactions

B1Different keywords sharing the same key

Example: aes(x = ppVar(mpg, shared = "k"), alpha = ppNum(0.5, shared = "k")) — same key on a column-name placeholder and a numeric placeholder.

ADR is silent; the runtime widget type collapses to the first occurrence's, but the registry-level type mismatch is a footgun.

B1.a (Recommended). Abort at translate-time with "shared key 'k' is used by incompatible placeholder roles (ppVar=consumer, ppNum=value)." Hard error; surfaces the bug at the earliest possible moment.
B1.b. Allow; first occurrence wins for widget kind; later occurrences are silently degraded. Maximally permissive; almost certainly hides bugs.
B1.c. Warn (not error) at translate-time. Half-measure.

B2Default conflict on a shared key — first has no default, later does

Example: aes(x = ppVar(shared = "k"), y = ppVar(mpg, shared = "k")).

ADR §8 says first-occurrence-wins silently. So the shared widget initializes with character(0) (no default) even though the second occurrence specifies mpg. This is consistent but possibly surprising; the second occurrence's intent is silently dropped.

B2.a (Recommended). Keep ADR §8 verbatim: first wins, no default → shared widget empty. Consistent with the label-conflict rule. Document the consequence in the helper roxygen.
B2.b. Promote: if first has no default but a later occurrence does, use the later one. "Most-specific-wins" semantics. Breaks the consistency of the first-occurrence-wins rule applied elsewhere (labels, widget config) — surprising in the opposite direction.
B2.c. Warn when defaults disagree among shared occurrences. Aligns with B1.c.

B3Same default repeated on a shared key

Example: aes(x = ppVar(mpg, shared = "k"), y = ppVar(mpg, shared = "k")) — identical defaults.

No conflict; first wins; everyone is happy. But: a redundancy you might want to warn about (the author is duplicating intent).

B3.a (Recommended). Silent. No warning. Redundancy is harmless and may be useful for documentation in long formulas.
B3.b. Lint at translate-time: "redundant default on shared key 'k' (occurrence 2)." Pedantic.

C · Default validity vs runtime data

C1ppVar(mpg) when the bound data doesn't have column mpg

The default "mpg" sits on the node. At UI boot, build_ui's intersect(selected, cols) silently drops it; the picker boots empty.

ADR is silent on whether to warn / abort / silently drop.

C1.a (Recommended). Silent drop (current behavior). The default is a best-effort hint. Users who care can check the data shape before booting. Matches today's intersect-and-drop behavior for prior input.
C1.b. Emit a one-shot UI hint ("default mpg not in data; picker reset.") in the placeholder's error/help slot. Helpful for demo authors; noisy for power users.
C1.c. Hard error at boot: "default mpg not in upstream data." Strict; breaks vignettes whose demo data evolves.

C2Default applies before async data arrives

Scenario: ppVar(mpg) in a formula whose pipeline starts with a ppUpload. At first render, columns are character(0). Does the picker show mpg selected (which will be dropped when columns resolve) or boot empty?

ADR is silent.

C2.a (Recommended). Seed with node$default regardless; intersect(default, cols) is empty until columns arrive, so the picker boots empty THEN promotes to mpg once columns include it. Same code path as for non-async data; one less special case.
C2.b. Hold the default in a "pending" state until cols are known, then apply. Adds a new reactive state to the build-ui orchestrator.

D · Plain-R behavior

D1Plain-R ppExpr(x + 1) outside aes()

The plain-R export is identity: ppExpr <- function(x, ...) x. Calling ppExpr(x + 1) outside aes() means evaluating x + 1 in the caller frame; if x is not bound, the user gets a confusing "object 'x' not found" error.

ADR §3 makes ppExpr identity for symmetry with ppVar/ppNum/ppText; the consequence is unspoken.

D1.a (Recommended). Accept the consequence. Document in ?ppExpr that plain-R use evaluates the arg in the caller frame; meaningful only inside aes-style NSE or when symbols are bound. Don't special-case.
D1.b. Make ppExpr's plain-R behavior return the language object instead of evaluating: ppExpr <- function(x) substitute(x). Asymmetric with the other identities; type-correct for "expression" placeholders. Surprising downstream — if a user writes ggplot(aes(x = ppExpr(y + 1))), tidy-eval expects a vector, gets language.
D1.c. Guard like ppUpload: error when called outside ptr_app(). Loses the "Example 2" symmetry that motivated the design.

D2Plain-R ppVar(mpg) as bare statement (not inside aes)

Example: result <- ppVar(mpg) at the top level of a script. Since identity, this evaluates mpg in the caller frame — error if not bound.

This is the same surface as D1, but for ppVar. Less surprising because the user explicitly invoked an "input widget" function outside a Shiny context.

D2.a (Recommended). Same as D1.a — document, don't special-case.
D2.b. Same guard pattern as ppUpload.

E · Expression-mode input

E1Pre-quoted expression: ptr_app(rlang::expr(ggplot(...)))

substitute(formula) captures the call rlang::expr(...), not its body. Naive deparse gives "rlang::expr(ggplot(...))" — won't parse as a ggplot formula.

ADR is silent. PLAN-04's dispatch handles strings + language + symbol-to-string indirection, but not pre-quoted wrappers.

E1.a (Recommended). ptr_capture_formula detects the wrapper calls rlang::expr(...), rlang::quo(...), quote(...), bquote(...) at the captured language root and unwraps once. Documented in ?ptr_app. Three wrappers cover the common cases.
E1.b. Don't special-case. Users who pre-quote get a parse error; they switch to passing the language object via a variable (f <- expr(...); ptr_app(f)) which works via PLAN-04's symbol-resolution branch.

E2Native pipe in expression mode

R's parser desugars mtcars |> ggplot(...) to ggplot(mtcars, ...) at parse time. Generated "preserve placeholders" code shows the desugared form, not the user-typed one. The user's |> is gone forever.

ADR §2 mentions this as a limitation but doesn't say whether the design wants to do anything about it.

E2.a (Recommended). Accept the loss; document the limitation prominently in ?ptr_app. Users who want |> preserved stay in string mode. (String mode's preprocess_native_pipe trick doesn't transfer to expression mode because we never see the source string.)
E2.b. Auto-suggest in error/warn messages: "in expression mode, native pipe is desugared; use string mode or %>% if you want |> in generated code." Helpful nudge.
E2.c. Refuse expression mode entirely when the source AST root resembles a piped chain (detect with a heuristic). Strict; breaks the symmetry the design was after.

F · Custom-placeholder authors

F1Custom keyword shadows a base R function

Example: ppSqrt <- ptr_define_placeholder_value(keyword = "sqrt", ...). Now sqrt(4) inside a formula is treated as a placeholder call. Likely catastrophic.

Or less dramatically: keyword = "c", keyword = "list", keyword = "do.call".

ADR is silent; the registry currently accepts any string as a keyword.

F1.a (Recommended). Reject at ptr_define_placeholder_*() registration time any keyword that resolves to a function in base, stats, utils, methods, graphics, grDevices, or ggplot2. Hard error with the conflicting name and the conflicting package. The pp-prefix convention makes this nearly automatic for users who follow it.
F1.b. Warn at registration: "keyword 'sqrt' shadows base::sqrt; this will conflict inside formulas." User chooses to proceed.
F1.c. Do nothing; trust the author. The footgun is real but rare since the pp-prefix convention is in place.

F2Custom validator that calls eval()

ADR says validators should be AST-only. There's no enforcement. A custom-placeholder author writes default_arg = function(x) eval(x). Translate-time evaluates arbitrary user formulas.

ADR is silent on enforcement.

F2.a (Recommended). Document the AST-only contract; don't enforce. The author is opting into the risk. Trusting the author keeps the registry surface simple and avoids ceremony around a rare misuse.
F2.b. Static check at registration: deparse(body(validator)) doesn't contain eval / parse / get / do.call. Heuristic; easy to fool; might still be worth it as a tripwire.
F2.c. Wrap every validator call in a sandbox (e.g. callr). Heavy; rules out legitimate use cases like accessing internal package helpers.

G · Code-gen / renderer

G1Preserve-mode render: bareword vs string for ppVar

The user-selected column is "mpg". In preserve mode, the renderer can emit ppVar(mpg) (bareword) or ppVar("mpg") (string). Both are valid input to ptr_app().

ADR is silent on the choice.

G1.a (Recommended). Bareword if the column name is a syntactic R name; quoted string otherwise (column names with spaces / dots-leading / starting with a digit). Maximally readable; falls back to string when needed.
G1.b. Always emit quoted string. Less ambiguous; reads less like ggplot.
G1.c. Always emit bareword; quote-with-backticks when needed. Closest to ggplot idiom; renderer needs to know when to use backticks.

G2Preserve-mode render for shared placeholders — do all occurrences emit the same default?

Example: shared coordinator picked mpg for shared key "k". In preserve mode, do we render ppVar(mpg, shared = "k") at every occurrence, or only the first?

ADR is silent; PLAN-05 doesn't specify.

G2.a (Recommended). Render ppVar(mpg, shared = "k") at every occurrence. The rendered formula round-trips to the same app: first-occurrence-wins still picks mpg on re-translation. Easy to implement (renderer doesn't track occurrence index).
G2.b. Render the default only at the first occurrence; later ones emit ppVar(shared = "k"). More faithful to the "first wins" intent; renderer must know it's not the first occurrence.

G3Preserve-mode for value placeholders whose current pick differs from the literal default

User authored ppNum(5); runtime picked 7.5. Preserve mode emits ppNum(7.5) (current_pick wins) or ppNum(5) (author default wins)?

PLAN-05 says "current pick supplied as the default" — i.e. current_pick wins. Confirm.

G3.a (Recommended). Emit current pick. The preserve-mode view is "the formula plus the current state", consistent across var/num/text/expr.
G3.b. Emit author default. The preserve-mode view is "the source formula as authored." Confusing because the visible default no longer matches the runtime state.

H · Migration / breaking-change peripherals

H1Shiny bookmark / saved-state URLs from before the rename

Today's apps generate Shiny input ids like layer1_var1. After rename, ids may change (e.g. layer1_ppVar1). Bookmarked URLs encoding old input ids would no longer restore correctly.

ADR is silent on bookmark compatibility.

H1.a (Recommended). Out of scope. ggpaintr's id-assignment is documented as internal and unstable. Bookmarks from old apps don't survive any internal rename; this is no different.
H1.b. Preserve the old id-suffix shape (e.g. ids still contain var not ppVar) to maintain bookmark stability. Couples the new public surface to an old internal naming choice. Probably not worth it.

H2User had a custom placeholder registered as "var" before the rename

Pre-ADR-0009, the built-in var blocked user-registered "var". Post-ADR-0009, "var" is no longer a built-in; the user's custom registration would now work. Surprise — the user's previously-shadowed code suddenly takes effect.

H2.a (Recommended). No action. Document in NEWS.md that the names var/num/text/expr/upload are no longer reserved by ggpaintr. Users get what they asked for; they were already aware they were trying to shadow a built-in.
H2.b. Reserve var/num/text/expr/upload as legacy-blocked keywords in the registry (registering one is an error with "this name was a previous built-in; use a non-conflicting name to avoid confusion"). Prevents silent re-purposing of an old name.

I · Validation messages

I1Multi-positional rejection wording

ppVar(mpg, disp) — two positionals, second one isn't named. PLAN-06 aborts with "Placeholder ppVar(...) accepts at most one positional argument (the default); got 2."

Is "got 2" sufficient, or do we want to name the offending positional explicitly?

I1.a (Recommended). Current wording is fine; concise and the user can re-read the line. Don't pretty-print extra context.
I1.b. Quote the offending args in the error: "got 2: mpg, disp." More helpful for long lines but verbose.

I2Unknown-named arg wording

ppVar(mpg, label = "X-axis") — label not declared in named_args. PLAN-06 aborts with "Placeholder ppVar(...) got unknown argument(s): label. Allowed: shared."

Does the message hint that custom named_args can add slots, or stay terse?

I2.a (Recommended). Terse; users learn the contract from ?ptr_define_placeholder_value. The error names the conflict and lists allowed args — enough to fix the immediate problem.
I2.b. Add a single-line hint: "Custom placeholders may declare extra named args via named_args; see ?ptr_define_placeholder_value." Helps newcomers; clutters the error for power users.

How to use this doc

Read each case. For each, tell me which option (a/b/c) you want, or describe a different resolution. I'll then:

  1. Fold the resolutions into ADR 0009 (one Edit per case, grouped into a single commit).
  2. Propagate any plan-level consequences (e.g. A1.a means ptr_default_numeric in PLAN-01 needs the unary-minus pattern; B1.a means PLAN-06's parser needs a per-shared-key role check).
  3. Restamp affected plans / ADR.
  4. Re-audit with subagents.
  5. Merge to all worktrees.

Skip-OK: cases where you accept the "Recommended" option as-is can be answered with a single "all rec" or "rec except X, Y, Z" — no need to enumerate every one.
