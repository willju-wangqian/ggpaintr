ADR 0026 followup — cross-file source/consumer parallel paths (#10)

Status: Parked — findings backup. NOT an active plan. This is a store of what we know about the two parallel source/consumer implementations, to be picked up after the narrower bug fix in ADR26-fix-host-env-rename-never-clears lands (the user's chosen sequencing — fix the live bug first, consolidate later, if at all). The synthesis demoted the full collapse to optional; this doc records why, and what is genuinely shared vs genuinely divergent, so a future investigation starts from truth, not from the over-stated raw catalog.

Date: 2026-05-30. Parent: ADR 0026 rank 10. Relates to: ADR 0023, ADR 0006, ADR 0025.

The story, in plain language

ggpaintr has two parallel implementations of the data-source → consumer machinery:

  • Single-instance / formula-local path — ptr_setup_source_uis() (R/paintr-server.R:2297) + ptr_setup_consumer_uis() (:2659) + the per-instance branch of ptr_bind_shared_consumer_uis() (:2948, state non-NULL). Always has a real state.
  • Shared-coordinator HOST path — ptr_setup_panel_sources() (R/paintr-shared-ui.R:482) + the host branch of ptr_bind_shared_consumer_uis() (state = NULL, ADR 0006). One widget at top level ("one widget, one owner").

They mirror each other's renderUI / resolve / mutex / seeding shape, but are distinct code. The documented consequence (verified finding two-parallel-source-handling-paths): a fix applied to one path does not propagate to the other and has to be hand-ported. The ADR-0025-#7 fileInput-reset work, and the host-env-rename bug (separate doc), are both symptoms of this split.

Crucial correction (don't trust the raw sweep here)

The automated sweep OVER-stated this duplication. Verified facts (2026-05-30):

  • The resolution CORE is already shared. resolve_upload_source() (R/paintr-server.R:1300) is called by BOTH paths — the host path (shared-ui.R) calls the same master impl. So upload resolution is NOT duplicated.
  • The spec-normalization IS duplicated — and this is the one the sweep's "no host twin" claim got WRONG (caught by the harvest verify pass): apply_spec_at_boot() (server.R:539) has a host twin apply_spec_at_boot_host() (shared-ui.R:722). These two are a genuine duplication candidate (the original "#10 Phase A").
  • The genuinely divergent copies are: (a) the two renderUI bodies, (b) the seeding-decision closures + latches (has_rendered, src_seen, last_src_identity, cleared_for_identity), (c) the spec-value normalization (the apply_spec_at_boot twin).

Why the full collapse is hard (and was demoted to optional)

The two paths' reactive dependencies genuinely differ, so a forced merge would couple them artificially:

  • Single-instance is state-backed: it reads/writes state$bound_names, state$resolved_sources, state$source_file_reset, a per-instance namespace.
  • Host is input-id-only / state=NULL (ADR 0006): it drives the picker off panel_sources[[sid]]() reactives + a synthetic host_env, reads the shortcut at session$rootScope()$input[[...]] (top-level, not namespaced), and has no real state.

This is the same divergence that makes the host-env-rename bug exist (bn="" at host). A single seam that pretends the two are interchangeable risks regressions in the most bug-prone area of the codebase. So: consolidate the genuinely-identical pieces (spec normalization first), do NOT force a unified source/consumer seam unless a later investigation proves the reactive shapes can be reconciled cleanly.

Phased approach (if pursued, after the bug fix)

  1. Phase A (the safe, real dedup): unify apply_spec_at_boot ↔ apply_spec_at_boot_host. Extract the pure spec-value normalization kernel both share; keep the per-scope dispatch (namespaced vs un-namespaced spec keys) as thin wrappers. ~40 LOC pure, ~10 LOC net. This is the one with the clearest motivation (the harvest-verified twin) and the lowest risk (pure normalization, not reactive wiring). Apply the same verify-first discipline: prove the two kernels are byte-equivalent in intent before merging; gate on the spec-roundtrip e2e (test-spec-roundtrip-*).
  2. Phase B/C (risky, optional): the renderUI bodies + seeding closures. Only after the host-env-rename fix (separate doc) settles the seeding semantics, and only if a focused investigation shows a shared seam that does not couple the state-backed and state=NULL reactive shapes. Wide-blast-radius shared-runtime change → full e2e at the merge boundary, never partial; never combine phases in one commit.

Boundary / scope

  • No public API change (internal runtime only).
  • Phase A only is "safe-ish"; the full seam-collapse is explicitly OPTIONAL and should not be forced.
  • Sequenced AFTER the host-env-rename fix. The bug fix may itself touch consumer_upstream_source_state at host scope; let that land and stabilise first.
  • Respect ADR 0006 (state=NULL at host) — any shared helper must work in both modes or be gated by is.null(state).
