ADR 0010 — ppUpload(name) name-default arg + identity-outside-app evaluation

Date proposed: 2026-05-21 · Promoted: 2026-05-21 · Owner: ggpaintr · Status: ACCEPTED — promoted from DEFERRED per /decision-to-plan session 2026-05-21; see Clarifications · Cross-refs: ADR 0009 (pp* grammar + expression-mode input) · Issue source: user-reported follow-up bug 2 from the ADR-0009 round-trip review (2026-05-21)

Follow-up note 2026-05-22. The preserve-mode render policy has changed since this ADR was drafted: an empty no-pick no-shared placeholder now renders as bare ppX (without parens) — see commit 8235d8a. Body references below to "ppUpload() emits as ppUpload() in preserve mode" describe the pre-change behaviour. The argued ADR 0010 design still holds (a ppUpload(penguins) arg form would still round-trip with parens via format_placeholder_value); only the empty-form description is stale. Re-read those claims as "pre-change baseline" when scoping the eventual implementation.

Context

ADR 0009 standardised the pp* placeholder grammar so that a positional arg to a placeholder declares its formula-time default — ppVar(mpg), ppNum(3), ppText("Hello"), ppExpr(mpg * 2). The default is validated by a default_arg closure declared on the registry entry (PLAN-01 / PLAN-06) and reaches the rendered widget's initial value via PLAN-07's invoke_build_ui seeding. The same default surfaces in preserve-mode code as ppX(value), matching what the user typed in the formula and what they currently see in the picker.

The ppUpload placeholder is the lone exception. It takes no positional default; the formula is always ppUpload() or the bareword ppUpload. At runtime the uploaded file is stored under some internal id (assigned by ggpaintr from filename / upload widget state). That internal id is not surfaced in the rendered code panel: in final mode, ppUpload is substituted with a literal referencing the resolved data, but in preserve mode the call still emits as ppUpload() — no display name available, no way for the user to choose one in the formula.

Two consequences fall out of this asymmetry:

  • Preserve mode hides the data identity. The user can read every other widget's current pick in preserve mode (ppVar(Sepal.Length)) but not the upload's. The code panel becomes copy-paste-ambiguous when the data source is a placeholder.
  • The formula cannot be evaluated outside ptr_app. ppVar(mpg), ppNum(3), etc. all evaluate as their default value when the runtime is the bare R interpreter — their identity-when-arg-present semantics let the formula stand alone for testing / docs / copy-paste-back. ppUpload currently raises (guard semantics from PLAN-08), so ppUpload |> filter(...) |> ggplot(...) is not runnable as plain R, even when the user knows which dataset they would test against.

Decision (proposed)

Adopt the ppUpload(name) shape — a single positional bareword or character arg declaring the display name the user expects the uploaded data to take in the rendered code. Status: deferred, not in scope this cycle.

ptr_app(
  ppUpload(penguins) |> dplyr::filter(species == "Adelie") |>
    ggplot(aes(x = ppVar, y = ppVar)) + geom_point()
)

Three behaviour changes:

  1. Registry slot. ppUpload's entry gains default_arg = ptr_default_symbol_or_string(), the same validator ppVar uses. Symbols (penguins) and string literals ("penguins") both accepted; the parser's extract_placeholder_args (PLAN-06) already routes positional args through this validator and stamps node$default.
  2. Final-mode substitution. Today substitute_walk.ptr_ph_data_source resolves the uploaded data and emits a literal referencing it. With ADR 0010, the substitution uses node$default as the display name when present, regardless of the file actually uploaded. Final-mode code becomes penguins |> filter(...) |> ggplot(...), stable across re-uploads.
  3. Preserve-mode rendering. render_placeholder_preserved dispatches ppUpload through format_placeholder_value with bareword formatting, identical to ppVar. ppUpload(penguins) in the formula round-trips back to ppUpload(penguins) in the code panel.
  4. Outside-app evaluation. ppUpload's runtime function (currently a guard that raises in non-app contexts) becomes function(x) x when called with an argument, while still raising when called with no argument (so accidental ppUpload() outside ptr_app still errors loudly rather than silently producing NULL). ppUpload(penguins) |> filter(...) |> ggplot(...) evaluates as plain R as long as the user has penguins in scope.

The formula language stays unchanged; this is purely an extension of ppUpload's argument grammar to align with the rest of the pp* family. No legacy form breaks (ppUpload() bareword still works, emits as ppUpload() in preserve mode, raises outside ptr_app).

Tradeoffs considered

ConcernAssessment
Alternative: render preserve-mode ppUpload using ggpaintr's internal data id.Rejected. The internal id is derived from filename / upload widget state, may not be a valid R identifier, may change on re-upload, and is not user-meaningful. The user typing ppUpload(penguins) declares stable intent.
Constant-fold of bareword args needs caller env.Already solved by ADR 0009 §4 (expression-mode capture). The caller env is threaded into ptr_app / ptr_server capture; ptr_default_symbol_or_string() accepts bare symbols and returns the symbol's character form without evaluation, so no data.frame lookup is required at translate time.
Guard-to-identity is a behaviour relaxation.The arg-bearing call ppUpload(x) becomes identity, but the no-arg ppUpload() still raises. Existing tests that exercise the guard against ppUpload() (no arg) remain green; only the new arg-bearing form is silently accepted.
Vignette / fixture migration burden.Optional — existing ppUpload usage continues to work. Migration happens incrementally as authors prefer the named form for documentation clarity. No atomic rename needed (unlike PLAN-08).
Multiple ppUpload instances in one formula with different names.Each instance has its own node id and its own widget. Different display names (ppUpload(left) + ppUpload(right)) render as themselves, no collision. Same as ppVar's multi-instance behaviour today.

Implementation sketch

Estimated ~30 minutes of work across three small touches:

  1. R/paintr-builtins.R — add default_arg = ptr_default_symbol_or_string() to the ppUpload registry entry. (Mirrors ppVar's declaration.)
  2. R/paintr-builtins.R — change the ppUpload runtime function from guard to function(x) if (missing(x)) rlang::abort("...") else x. The no-arg branch is still a guard; the arg branch returns its input unchanged.
  3. R/paintr-substitute.R — in substitute_walk.ptr_ph_data_source, when node$default is non-NULL, emit the substituted literal using node$default as the symbol name. (Preserves the runtime resolve path; only the rendered name changes.)
  4. R/paintr-render.R — render_placeholder_preserved already dispatches by keyword; add ppUpload to the bareword group alongside ppVar.

One new test file (tests/testthat/test-ppupload-name-arg.R) and one new e2e fixture (adr10-ppupload-name) suffice for coverage. No vignette rewrites required in the same change.

Clarifications (2026-05-21, post-promotion)

The original Decision section under-specified four points. The /decision-to-plan grilling resolved them as follows. These clarifications are the authoritative source for the implementation plans; the original Decision text remains as drafted but is read under these clarifications, not against them.

C1. The display-name mechanism is the existing companion textInput

The original §Decision-3 implied a new "bareword group" entry for ppUpload in format_placeholder_value and a new field for the display name. Source verification (2026-05-21) shows the mechanism already exists: ptr_builtin_upload_build_ui (R/paintr-builtins.R:218–240) already renders a companion textInput at node$companion_id with initial value = "", substitute_walk.ptr_ph_data_source (R/paintr-substitute.R:145–179) already reads ctx$snapshot[[node$companion_id]] and emits rlang::sym(name_value) in final mode, and format_placeholder_value (R/paintr-render.R:391–412) already lists "ppUpload" in bareword_kw. The work is therefore to seed that companion textInput with node$default at UI build time, not to invent a parallel mechanism. Preserve-mode rendering also depends on stamp_current_pick_walk.ptr_ph_data_source (R/paintr-render.R:358–364) reading the companion-id snapshot instead of the fileInput's value.

C2. Final-mode bind: via ADR-0009 captured caller env (no new injection plumbing)

When ppUpload(penguins) seeds the companion textInput with "penguins", substitute_walk.ptr_ph_data_source emits the symbol penguins into the final-mode AST. At live-eval time, the symbol is resolved through the Shiny session's eval env (state$eval_env = new.env(parent = envir), R/paintr-server.R:207), whose parent is the caller env captured by ptr_capture_formula (R/paintr-app.R:195). If the user has penguins bound in the calling env, the pipeline resolves and renders. If they upload a file, assign(binding_name, df, envir = state$eval_env) (R/paintr-server.R:504) shadows the parent-env binding, so upload override works correctly. No new env-threading code is required — the existing eval-env parent chain delivers auto-resolve for free.

C3. Auto-resolve in app context (semantic extension beyond "display name")

The original ADR framed ppUpload(penguins) as only a display-name declaration. Promoted scope: ppUpload(penguins) additionally means "treat penguins from the caller env as the default data; users can still re-upload to replace it." The user-facing semantics now match ppVar(mpg)'s default-but-overridable shape. No imperative pre-binding step is needed because (per C2) the eval env's parent chain delivers caller-env lookup automatically once the companion textInput is seeded.

C4. Outside-app eval: function(x) x identity; string passthrough accepted

The runtime ppUpload top-level function (R/paintr-builtins.R:365–367) changes from a guard that always raises to function(x, ...) if (missing(x)) rlang::abort(...) else x. The bare-form (ppUpload(), no positional arg) still raises with the existing message. The arg-bearing form is identity, so ppUpload(penguins) outside ptr_app returns the value of penguins from the caller env. String form (ppUpload("penguins")) is accepted and returns the string verbatim — symmetric with ppVar("mpg") outside app; the pipeline's next stage will surface a typing error if a string is not what the user meant.

C5. Legacy ppUpload() (no default) emission is unchanged

When node$default is NULL, the companion textInput starts with value = "" exactly as today; preserve-mode emits ppUpload(); final-mode emission depends on whether the user has typed a name into the companion input at runtime (same as today). ADR 0010 is strictly additive on the legacy path.

C6. node$default storage shape

ptr_default_symbol_or_string() (R/paintr-default-args.R:201–213) accepts a bare symbol or a string scalar and returns a single character scalar in both cases (the symbol is coerced via rlang::as_string()). Therefore node$default on ppUpload nodes is always either NULL (legacy) or a single character scalar (named form). This matches ppVar's node$default shape, so seeding shiny::textInput(value = node$default %||% "") works without coercion logic.
