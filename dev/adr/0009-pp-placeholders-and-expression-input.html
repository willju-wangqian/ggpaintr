ADR 0009 — pp-prefixed placeholders, expression-mode input, and registry-declared argument schema

Status: proposed — implementation pending · Date: 2026-05-20 · Supersedes: nothing (extends ADR 0006's placeholder surface) · Voids: nothing yet (will void the bare var/num/text/expr/upload keyword vocabulary on merge) · Related: ADR 0006 (single server entry; L3 = UI-only), ADR 0008 (placeholder freedom ceiling) · Domain language: CONTEXT.md (placeholder taxonomy unchanged; keyword renames are notational) · Implementation: none yet — this ADR records the locked design before any source edit.

Decision (proposed): Rename every placeholder keyword to a pp-prefixed camelCase form, add an expression-mode input alongside the existing string-mode, and move argument-schema enforcement from the parser into the placeholder registry so custom placeholders can opt into a positional default and named arguments through the same author API the built-ins use.

Context

Today's ptr_app() / ptr_server() accept a single string formula. Inside it, five keywords (var, num, text, expr, upload) mark placeholder slots; each accepts at most one named argument, shared, enforced by a single keyword-agnostic parser helper extract_shared() (R/paintr-translate.R:381). Anything else — positional args, other named args — is rejected at translate-time.

Three pressures push against this surface:

  1. Defaults at formula authoring time. The current contract gives no way to write "this picker should boot defaulted to mpg" in the formula itself. Vignette demos boot blank; users coding example apps must either edit the rendered UI or push state through a separate mechanism.
  2. Editor / IDE ergonomics for the formula. A string formula gives up syntax highlighting, paren-matching, and IDE completion; ggplot-like surfaces look more natural unquoted. A real-language formula is closer to ggplot and `aes()` idioms.
  3. Custom-placeholder authors are second-class. Built-in placeholders use a parser-internal "only shared allowed" rule that custom placeholders inherit by default — there is no way for a custom ppRange to declare "I accept c(0, 100) as a default and a step = named arg." Any extension that needs richer authoring syntax has to monkey-patch extract_shared, which the public API does not expose.

The bare keyword names (var, num) also collide with reader expectations: var() is stats::var() in base R; unprefixed names blend in with column names in aes() and obscure that they are placeholders. Renaming is a one-time cost; the resulting surface stays distinct and future-proof.

Decision

1. Keyword rename — hard, in one step

Old keywordNew keywordRole (unchanged)
varppVarconsumer
numppNumvalue
textppTextvalue
exprppExprvalue
uploadppUploadsource

The old names cease to be placeholders. A formula that uses var(mpg) or upload() errors at translate-time with a "renamed: use ppX" message. No deprecation cycle, no aliasing layer. The migration is mechanical and bounded; the long-term cost of dual vocabularies outweighs the one-time cost of renaming every vignette, test, and demo.

camelCase is chosen over pp_var / ph_var / var_ because, inside an aes() call surrounded by snake_case column names, the camelCase token pops visually as "not a column reference." The five symbols are the only camelCase exports in the package; the inconsistency cost with the surrounding ptr_* snake-case convention is paid in five places, not across the whole API.

2. Two input modes — string and unquoted expression

ptr_app() and ptr_server() dispatch on the captured argument at the public boundary (via substitute() / rlang::enexpr()):

  • String mode — ptr_app("ggplot(aes(x = ppVar(mpg))) + geom_point()"). Existing path. Goes through preprocess_native_pipe() (R/paintr-translate.R:176) so |> survives as a sentinel, then rlang::parse_exprs().
  • Expression mode — ptr_app(ggplot(aes(x = ppVar(mpg))) + geom_point()). The argument is captured unevaluated and joins the existing pipeline at ptr_translate()'s step 4 (split top-level + into per-layer expressions). The native-pipe preprocess is skipped — R's parser has already desugared |> at parse time, so the pipe surface is lost in expression mode. Users who want |> preserved in generated code must either stay in string mode or use %>%.

Both modes accept the new keyword vocabulary; both produce the same typed AST downstream. The translate-time pipeline is unchanged from step 4 onward.

3. Plain-R exports

For expression-mode use to be transparent — i.e. for

mtcars |> ggplot(aes(x = ppVar(mpg))) + geom_histogram()

to evaluate to the same histogram as the baseline outside ptr_app() — the placeholder keywords must be real R functions in scope. ggpaintr exports:

SymbolBodyWhy
ppVar / ppNum / ppText / ppExprfunction(x, ...) x (identity)Inside aes(), ggplot's tidy-eval will call the function with the data-context-evaluated arg; identity returns it unchanged.
ppUploadfunction(...) rlang::abort("ppUpload() is only meaningful inside ptr_app()")Upload has no plain-R semantic — its resolver expects a Shiny fileInput() value (documented at R/paintr-upload.R:59, consumed by the reader at :63), not a plain-R argument. Erroring is honest.

4. Registry-declared argument schema

Three new slots on the placeholder registry entry, set at ptr_define_placeholder_*() time:

SlotTypeDefaultMeaning
default_argNULL or validator functionNULLSchema for the (single) positional argument. NULL ⇒ positional rejected at translate-time. Function ⇒ called with the unevaluated AST of the positional; returns the canonical value (stored on node$default) or aborts with a keyword-specific message.
named_argsnamed list of validator functionslist()Schema for additional named arguments beyond shared. Empty ⇒ unknown named args rejected. Each validator receives the unevaluated AST and returns the canonical value; results land on node$named_args.
runtimefunctionidentity (for _value/_consumer); guard (for _source)The plain-R callable returned by ptr_define_placeholder_*(). Authors bind it to the keyword's name and @export. Override only when a custom plain-R body is genuinely needed.

Reserved name: shared may not appear in named_args. Enforced at ptr_define_placeholder_*() registration (hard error) and again at translate-time (defensively).

5. Default-validator helpers

Shipped with the package; symbol policy is documented per helper, not per parser branch:

HelperBareword symbolString literalNumeric literalOtherUsed by

ptr_default_string()rejectedacceptedrejectedrejectedppText

AST-only — no eval at translate-time. Helpers pattern-match the AST shape; they never call eval(). ppExpr defaults are kept as language objects and converted to text for the textarea, so user code is never executed at parse-time. The whole-formula safety walker remains the single eval-time gate.

Symbol policy lives in the helper, not the parser. For ppVar(mpg) and ppRange(my_range) the parser sees the same AST shape: a placeholder call with one symbol positional. Whether that symbol means "column name as a string" or "variable to dereference" is decided by the helper the keyword's definition selected. This mirrors ggplot's aes(): a verb decides what a bareword means.

The first cut is literal-only across the board, except ppVar. ppVar reads symbols as column-name strings (the established NSE convention every dplyr verb already uses); all other built-in helpers reject symbols. An eval-after-safety-walk variant (e.g. ptr_default_numeric_vector_eval(env)) is out of scope for this round — it can be added later as an additive change, because adopting it would require threading a caller environment through ptr_app() / ptr_server() / ptr_translate() and would open a translate-time eval surface the current design deliberately closes.

6. Parser change

extract_shared(expr, keyword) (R/paintr-translate.R:381) becomes extract_placeholder_args(expr, entry). It reads entry$default_arg and entry$named_args and applies the schema. Existing error texts ("positional args are not allowed", "got unknown argument(s)") survive as the no-schema-declared fallbacks. Validation failures produce keyword-specific messages supplied by the helpers.

Storage on the placeholder node:

node$default     # validated positional, or NULL
node$named_args  # named list of validated named args, or list()
node$shared      # existing shared-key string, or NULL

7. Define-helper surface returns the runtime callable

ptr_define_placeholder_value(), _consumer(), _source() currently end by calling ptr_registry_register(entry) (which returns invisible(entry) at R/paintr-registry.R:192) and the helpers themselves return that value invisibly. They will instead return the runtime function (function(x, ...) x for value/consumer; guard for source), so authors write:

#' @export
ppRange <- ptr_define_placeholder_value(
  keyword      = "ppRange",
  build_ui     = my_range_ui,
  resolve_expr = my_range_resolve,
  default_arg  = ptr_default_numeric_vector(length = 2L),
  named_args   = list(step = ptr_default_numeric())
)

One statement registers the placeholder, returns the runtime callable, and the author binds it under the right name. Built-ins use the same surface in paintr-builtins.R — no special-casing.

Why keyword = stays an explicit string. R does not give a function access to the LHS of the <- it is being assigned to: the RHS is fully evaluated before <- binds. Walking sys.calls() for an enclosing assignment works in simple REPL contexts but breaks inside lapply, local({...}), assign("X", ...), package-source files, and pipe contexts. R6::R6Class("Foo", ...) and setClass("Foo", ...) face the same constraint and both took the explicit-string path; this ADR follows that precedent. As a cheap safety net, the define helpers may do a best-effort sys.calls() walk and warn (not error) when the enclosing <- LHS differs from keyword =; silent in non-assignment contexts.

8. Default + shared, combinable

ppVar(mpg, shared = "axis") is legal. The positional default participates in shared-key resolution under the existing first-occurrence-in-formula-order rule (same rule that already governs label and widget-config conflicts among multiple shared = "k" occurrences). If two occurrences disagree on the default, the first wins silently; no translate-time abort.

9. Code-gen — two-version output, toggle in one panel

The code panel emits two strings:

VersionWhat it showsStatus
Final codeaes(x = mpg) — placeholder erased, current pick spliced in. Today's substitution behavior; valid plain R the user can copy and run with no ggpaintr dependency.default view
Formula with placeholdersaes(x = ppVar(mpg)) — placeholder preserved, current pick supplied as the default. Valid plain R because ppVar is the identity function (point 3); also valid input to ptr_app() in expression mode, round-tripping with the source formula.toggle

One panel, a toggle (radio or segmented control) between the two versions. The "formula with placeholders" view is the round-trippable form; the "final code" view is the publish-ready form.

Alternatives considered

A. Keep var/num/... and add ppVar/ppNum/... as aliases

Rejected: two vocabularies in the codebase forever (or through a deprecation cycle that drags out the migration). Adds confusion to docs and vignettes for no offsetting benefit; the rename is mechanical and bounded.

B. Snake-case pp_var or trailing-underscore var_

Rejected. pp_var matches tidyverse style but blends in with snake_case column names inside aes() — readers cannot see at a glance that a placeholder is present. var_ looks like a typo and gives no semantic prefix. camelCase pops; the style inconsistency cost is contained to five symbols.

C. Side-effecting register that captures the symbol name via substitute()

ptr_register_placeholder_value(ppRange, build_ui = ..., ...) would assign ppRange into the caller scope. Rejected: side-effecting assignment is a poor fit for package source files (where roxygen's @export scanner relies on the <- function pattern), and it removes the LHS as a visible declaration. The Option B return-value pattern keeps the <- idiom intact.

D. Eval-with-safety for non-literal defaults this round

Rejected for the first cut. Evaluating my_range inside ppRange(my_range) at translate-time requires threading a caller environment through ptr_app() → ptr_server() → ptr_translate() → the validator, plus running a safety walker on the AST before eval to keep the translate-time eval surface honest. Doable, but additive; landing it now would bloat the contract while we are still proving out the simpler literal-only schema. Documented as an out-of-scope follow-up.

E. Allow defaults but keep them in a separate defaults = list(...) argument to ptr_app()

Rejected: separates the default from the placeholder, requiring readers to cross-reference. The whole point of the in-formula default is co-locating it with the placeholder that uses it.

Consequences

Breaking changes (intentional)

  • Every formula in the codebase using var/num/text/expr/upload as a placeholder keyword must rename to the pp-prefixed form. Vignettes, fixture apps under tests/testthat/fixtures/vignette-apps/, the README, manual tests, and demo apps are all affected.
  • Custom-placeholder authors who registered under the old keywords must rename their registrations. The define helpers themselves keep their names (ptr_define_placeholder_value/_consumer/_source); only the example built-in keywords change.
  • Anyone calling var() / num() / etc. as plain R outside ptr_app() (rare; was never documented to work) will see a "function not found" error unless they switch to ppX.

Non-breaking additions

  • The default_arg and named_args slots are additive on the registry side; placeholders that omit them retain today's "only shared allowed" contract by default. Existing custom placeholders keep working under the rename without touching their definitions.
  • Expression-mode input is additive: string-mode formulas continue to work. The dispatch happens before any string-specific preprocessing.

Surface area gained

  • Authors of custom placeholders gain a real argument-schema language without parser-internal patches. ppRange(c(0, 100), step = 5)-style placeholders become possible without monkey-patching extract_shared.
  • Vignette and demo apps can boot with sensible non-empty initial state. Today's empty-picker UX, which forces users to choose a column before the first plot renders, is no longer mandatory.

Surface area gained that we deliberately do not use yet

  • Eval-with-safety variants of the default helpers are not shipped in this round. Adding them later does not require any further contract change — they slot into the existing default_arg field.
  • Rich inline error UX for default-validation failures is not in scope; translate-time abort with a keyword-specific message is sufficient as a first cut. Errors surface in the existing "could not parse formula" pathway.

Cross-cutting decision: prefer rlang over base R for AST work

Wherever the implementation has a choice between a base-R AST primitive and an rlang equivalent, prefer rlang:

UsePrefer (rlang)Avoid (base)
Capture an unevaluated argumentrlang::enexpr(x)substitute(x)
Inert quoted expressionrlang::expr(x + 1)quote(x + 1)
Evaluate a bare expression in a chosen envrlang::eval_bare(e, env)eval(e, env) (use rlang form when env is sealed)
Test AST shaperlang::is_symbol, rlang::is_call, rlang::is_syntactic_literal, rlang::is_string, rlang::call_name, rlang::as_stringdirect is.symbol, is.call, etc. (acceptable but less consistent)
Stringify language for displayrlang::expr_text(e)deparse(e)
Splicing with !!only via rlang::enexpr/rlang::expr/rlang::eval_tidynot supported in base

rlang is already in Imports; no new dependency. This is style guidance, applied in every plan that touches the AST or the placeholder pipeline.

Open follow-ups (out of scope this round)

  • Eval-with-safety variants of the default-arg helpers (e.g. ptr_default_numeric_vector_eval(env)) for placeholders that want to accept variable references like ppRange(my_range). Requires caller-env threading and a translate-time safety pass; additive when ready.
  • A linter / validator for the new author API (cross-check that keyword = matches the LHS at the package source level; best-effort sys.calls warning is the runtime fallback).
  • Surfacing the node$named_args contents in the code-gen "formula with placeholders" view round-trip (today's sketch handles node$default only; named args need an analogous renderer pass).
