ADR 0003 — Controllable-region UI abstraction

Status: Accepted · Date: 2026-05-15 · Plan: dev/plans/2026-05-14-controllable-region.html · Audit ref: dev/audit/2026-05-14-compatibility.html §3.3 (recommendation #1, partially closed — UI half).

Context

The "checkbox above a region greys out the region and removes its contribution at eval time" affordance was emitted by three separate sites:

  • build_pipeline_stage_ui() — per-module pipeline-stage block.
  • wrap_shared_widgets_with_stage_blocks() — module-side wrap of shared widgets that belong to an orphan stage.
  • ptr_shared_ui() — cross-module shared-orphan stage in the shared panel.

All three open-coded the same DOM tree (.ptr-stage wrapper, .ptr-stage-head with a checkboxInput(), .ptr-stage-fields body) and the same id convention (<sid>_stage_block). Two of them additionally emitted a bare .ptr-stage-fields "continuation" div under a seen/emitted guard.

The drift risk was concrete: the 2026-05-15 visual-cue change had to be made in CSS only because the three call sites already agreed on the class names — but a future edit that touches one site's wrapper class (e.g. adds a marker class for a new affordance) would silently diverge from the other two.

Decision

Introduce one internal file R/paintr-region.R with two helpers:

controllable_region(region_id, head_label, body, ns_fn = identity, default_on = TRUE)
controllable_region_continuation(body)

The three call sites above now build their wrappers through these helpers. The DOM contract — class names, id format (<ns>(<region_id>_stage_block)), checkbox inputId = <ns>(<region_id>) — lives in exactly one place.

Observer wiring is not part of this abstraction. The three existing observer setups (ptr_setup_stage_enabled, ptr_setup_shared_stage_enabled, and the in-line CSS-class observer in ptr_shared_server) stay separate because their reactive shapes are not symmetric: an observeEvent(input[[id]], …) binder cannot replace one whose trigger is an external reactive without obscuring real timing differences. See plan §2 callout and §10 follow-up.

Consequences

  • Single DOM emitter. Future "grey out X" affordances become a one-line caller of controllable_region(); they still write their own observer in whichever shape matches.
  • One DOM-id format change. The shared-panel orphan-stage block previously used <sid_input>_block; it now uses <sid_input>_stage_block like every other caller. The companion CSS-class observer in ptr_shared_server() was updated in the same commit. No public selector or test depended on the old suffix.
  • No public API change. Helpers are unexported and `ptr_`-unprefixed (internal-only convention).
  • Visual-cue contract unchanged. The disabled affordance keeps the 2026-05-15 strikethrough + muted labels + dashed border + faint tint combo, anchored to .ptr-stage-disabled .ptr-stage-head / .ptr-stage-fields / label. No opacity, filter, or transform may be reintroduced on any wrapper this helper emits — picker dropdowns are descendants and would otherwise be clipped by a stacking context.
  • Test guard against drift. tests/testthat/test-controllable-region.R asserts the continuation helper's class string equals the inner-fields class of the full helper, so a class rename in one helper without the other becomes a test failure.

Out of scope (deferred to plan §10)

  • Observer/state binder unification.
  • A region_is_disabled predicate (only one caller exists today).
  • Layer-level checkbox unification (ptr-layer-disabled path; different semantic).
  • Public "register your own region kind" API.

Verification

  • devtools::test() — full suite green (1250 PASS, 0 FAIL).
  • SC1 — exactly one shiny::div(class = "ptr-stage", …) emitter in R/ (the helper).
  • SC3 — three region kinds covered by test-controllable-region.R: pipeline-stage shape (CR.1, CR.2), continuation (CR.3), fabricated kind (CR.4).
