ADR 0022 — Spec panel replaces preserve-mode formula view

Status: Proposed. Drafted 2026-05-25.

Date: 2026-05-25

Relates to: ADR-0012 (spec= shape, apply-timing, panel-section emission), ADR-0020 and ADR-0021 (structural keywords as the trigger that exposed the misframing).

Implementation: dev/plans/0022-spec-panel-replaces-preserve-mode/ (TBD).

Context

The generated-code panel today exposes two views via a radio toggle (R/paintr-app.R:480-491): "Final code" emits the eval-mode substituted ggplot expression, "Show placeholders" emits a round-trip of the original formula with pp* tokens preserved plus a sparse ptr_spec <- list(...) block beneath (R/paintr-server.R:2253-2283). The formula-with-placeholders portion was intended as a reproducer: a non-owner of the app could copy it, paste it into ptr_app(), and recreate the running app.

That intent is incoherent under closer inspection. Two audiences interact with a ggpaintr app:

  • App owner — wrote the source ptr_app(formula, ui_text, css, expr_check, safe_to_remove, spec, ...). They already have the formula. They need a way to persist state (current widget values) across sessions or share a saved view with another owner.
  • App visitor — runs a deployed app, has no source. The plot-recipe escape hatch is the eval-mode substituted ggplot code; the visitor can take that away and use it without ggpaintr. They were never the audience for a formula reproducer.

The preserve-mode panel was attempting to serve a third audience that does not exist (a non-owner who wants to clone the app), at the cost of carrying a render-walker code path that cannot honestly reproduce the app for structural keywords. The two structural keywords (ppLayerOff, ppVerbSwitch — ADR-0020 / ADR-0021) never appear in the typed tree, so the preserve-mode renderer cannot emit them. A formula passed through preserve-mode loses (a) ppLayerOff's hide= default, (b) the entire ppVerbSwitch wrapper (which is the only way to create a stage checkbox on a placeholder-free verb — without the wrapper, the rebooted app has no such checkbox at all).

The right primitive for state persistence is spec: formula = app definition (owned by the owner, lives in source); spec = state of one running instance (deparseable, shareable, multiple per formula). Different states of the same plot are different specs against one formula. Different plots are different formulas (and ptr_app_grid() already covers the multi-plot case).

Decision

Soft-land preserve-mode out of the UI. Keep the implementation alive.

  1. Repurpose the radio's second choice. Rename the label "Show placeholders" → "Spec" and the input value "preserve" → "spec" in code_mode_toggle() (R/paintr-app.R:480-491).
  2. Trim the formula-render emission from the panel body. In R/paintr-server.R:2253-2283, drop the formula_text <- ptr_render(..., preserve_placeholders = TRUE) emission and the concatenation. The branch body becomes: 
    spec_text <- format_spec_for_panel(state$spec())
if (!nzchar(spec_text))
  "# No overrides yet — interact with the controls and click Update."
else spec_text
    The trimmed lines are commented out with a header pointing at this ADR; nothing in R/paintr-render.R is touched. ptr_render(..., preserve_placeholders = TRUE) remains a working internal function.
  3. Rewrite tests that drove the UI for preserve-mode round-trip assertions (tests/testthat/test-adr10-ppupload-name-browser.R, the relevant blocks of tests/testthat/test-super-pressure.R) as unit tests calling ptr_render(tree, preserve_placeholders = TRUE) directly. Coverage of the live render code is preserved; the UI proxy is dropped because the panel no longer surfaces this text.
  4. Update vignettes / feature-bank for the rename. Light copy pass — no behavioral change.

A future helper ptr_inject_spec(formula, spec, ui_text = NULL, css = NULL, ...) can collapse owner-side state into a literals-only formula for the rare "ship a frozen snapshot of my app to a non-owner without giving them the source" use case. That helper is out of scope for this ADR; the spec panel is sufficient for the in-scope audiences.

Consequences

  • The panel no longer claims to reproduce the formula. The owner is the formula's source of truth. The spec is a serializable snapshot of current widget values, copy-pasteable into a new ptr_app(formula, spec = ...) call by the same owner.
  • Structural-keyword honesty restored: the panel cannot lie about ppLayerOff / ppVerbSwitch because it no longer attempts to round-trip them.
  • The comparator-exclusion list (ADR-0021) and structural-keyword unwrap behavior (ADR-0020 / ADR-0021) are unaffected. Tree shape, id stamping, runtime evaluation are all unchanged.
  • Render-walker preserve-mode infrastructure (R/paintr-render.R) is dormant in the UI but alive in code and tests. A follow-up ADR may propose full removal once a deprecation window has elapsed.
  • The "Spec" label on the radio matches the spec = argument of ptr_app() 1:1, so panel and API share one term.
  • Empty-spec UX: the panel shows a single comment line ("# No overrides yet — interact with the controls and click Update.") instead of a blank box. Pre-Update users see something honest.

Alternatives considered

  • Teach the render walker to re-emit ppLayerOff / ppVerbSwitch from carrier-node metadata. Rejected: solves a problem the panel shouldn't be trying to solve at all (reproduction for non-owners). The audience analysis above settles the framing question, and Option 3 disappears with it.
  • Hard-delete preserve-mode rendering. Rejected as too aggressive for this change. The render code is small, intact, and may be repurposed (e.g. by ptr_inject_spec). Soft-landing preserves optionality at near-zero ongoing cost.
  • Keep preserve-mode and re-label it "approximate". Rejected: admits the dishonesty without removing it. Compounds rather than fixes.

Out of scope

  • ptr_inject_spec(formula, spec, ...) helper. Owner-side tool to bake state into a formula for snapshot distribution. Useful and probably worth building later; not required by this ADR.
  • Full removal of preserve-mode render code. Deferred. Revisit once the deprecation window has elapsed and no user has surfaced a need for the dormant path.
  • ui_text / css in the spec panel. They affect app shape but are owner-side boot config (not state). Owners hold them in source; no panel-side echo.
