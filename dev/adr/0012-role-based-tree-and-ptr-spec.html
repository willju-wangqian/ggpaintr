ADR 0012 — Role-based tree, terminal-grounded pipeline lift, and ptr_spec

Date: 2026-05-21 · Author: Claude Opus 4.7 (design session with user, post-ADR-0011 root-cause flip) · Status: DRAFT — design locked, implementation not started · Cross-refs: supersedes the diagnosis & fix shapes in ADR 0011 · vignette-review HEAD at draft time: ce298f1

One-line summary. The tree currently leaks surface syntax: |> vs %>% vs nested calls produce three structurally different trees for the same plot. Make the tree purely semantic — one canonical ptr_pipeline regardless of how the user wrote the chain — by lifting any nested-call data-arg whose innermost first-arg grounds to a known data source. Reuse the existing default_drop_when_empty verb list for stage elision. Add ptr_spec as a sibling artifact on ptr_state so preserve-mode code panels round-trip the user's picks (including stage-checkbox state) without forcing the wrong defaults on replay.

1. Context — why the current model is wrong

1.1 The subbed-1 vs subbed-2 probe

During the ADR-0011 debug session, dumping the post-substitute tree for two formulas that differ only in pipe syntax exposed the divergence in one comparison:

Inputdata_arg shape
ppUpload %>% filter(ppVar > ppNum) %>% mutate(new_var = ppVar + 1) %>% ggplot(...)ptr_pipeline with three explicit $stages: source penguins, filter(ppVar > ppNum), mutate(new_var = ppVar + 1)
ppUpload |> filter(ppVar > ppNum) |> mutate(new_var = ppVar + 1) |> ggplot(...)single nested ptr_call: mutate(filter(ppUpload, ppVar > ppNum), new_var = ppVar + 1) — no pipeline, no stages

The plots are semantically identical. The trees are not.

1.2 Root cause

R parses a |> f(b) as f(a, b) at parse time — the native pipe is purely syntactic sugar with no surviving AST marker. %>%, by contrast, is an ordinary infix function from magrittr whose call structure survives parsing intact. The current translator only constructs ptr_pipeline when it sees the %>% symbol (the is_pipe_head predicate in R/paintr-translate.R) or the sentinel injected by preprocess_native_pipe for string-input |> tokens (also in R/paintr-translate.R — locate by symbol name).

Crucially, preprocess_native_pipe only fires for string-input. In expression-input mode (R/paintr-app.R:184-279), the formula is captured via rlang::enexpr() then deparsed via rlang::expr_text() — and because the |> has already been desugared at parse time, expr_text returns the nested form. The preprocessor finds zero PIPE tokens and is a no-op.

3. Design details

3.1 The lift transform

Engine sketch (purely structural; the "should we lift here" decision is delegated to the terminal-grounding check at the call site):

resugar_pipeline <- function(expr) {
  stages <- list()
  cur <- expr
  while (is.call(cur) && length(cur) >= 2L && is.call(cur[[2L]])) {
    head      <- cur[[1L]]
    rest_args <- as.list(cur[-c(1L, 2L)])
    stages    <- c(list(as.call(c(list(head), rest_args))), stages)
    cur       <- cur[[2L]]
  }
  c(list(cur), stages)   # head = source; tail = stages in user-written order
}

This runs inside ptr_classify_calls at every site that holds a data-arg subtree (currently extract_call_data_arg for ggplot()'s data and the named data = arg). The result list is fed into the same ptr_pipeline constructor that today's translate_pipeline / collect_pipe_stages path uses for %>%, so downstream code (prune, render, runtime upstream resolution) sees one node shape.

3.2 Terminal-grounding check

The lift fires iff the deepest first-arg of the chain (after resugar_pipeline walks inward) is not itself a call — i.e., it must be a symbol, a literal, a ptr_ph_data_source, or any other non-call AST node.

This rule (verified 2026-05-21 against R/paintr-classify.R) is the structural form that supersedes an earlier three-category enumeration (placeholder source / data-frame-bound symbol / literal data frame). The earlier rule would have required ptr_classify_data to tag bare symbols and literals as data sources, which the current pass does not do. The non-call rule is purely structural: it requires no eval-env probe and naturally captures every intended terminal — a placeholder source is a non-call ptr_ph_data_source node; a bare data-name symbol like penguins is a non-call symbol; a literal value is a non-call literal.

Opaque terminals — any chain whose deepest first-arg is itself a call (make_data(), read_csv("foo.csv"), data.frame(x = 1)) — stay as a plain ptr_call: quiet failure, no warning. This matches the principle from §2: we don't pretend to recognize a pipeline when the source is a black-box call.

3.3 Stage elision

Unchanged from today's prune_walk.ptr_pipeline. The list default_drop_when_empty() is the contract; a stage call whose qualified head is in the list and whose non-data args have pruned to missing is dropped from the pipeline. The redesign's contribution is purely that every pipeline (regardless of surface form) hits this code path.

3.4 The + fence

The lift pass walks data-arg subtrees only. +-composed layers are already split into separate ptr_layer nodes by split_top_plus (in R/paintr-translate.R — called from ptr_translate's setup) before any per-layer translation runs. The lift can never walk through a + because it never sees one.

3.5 ptr_spec construction and shape

Ingredients already in hand:

  • Every placeholder-bearing node has a raw input id from ptr_assign_ids (in R/paintr-translate.R, called from the annotate block of ptr_translate).
  • Every stage has a stage_id for its enabled-checkbox (visible in the subbed-1 dump from §1.1: ggplot_2_stage_enabled).
  • The runtime freezes a snapshot of all input values on every Update click (res$snapshot, used by ADR-0009 preserve-mode).
  • Each placeholder hook declares its default in the registry.

Construction is a one-pass diff: ptr_spec = snapshot ∖ registry_defaults, keyed by fully-qualified Shiny input id. The result is added to state as state$spec (reactive over the runtime snapshot).

Code-panel emission, preserve mode, single self-contained call:

ptr_app(
  ppUpload |> filter(ppVar > ppNum) |> ggplot(aes(ppVar, ppVar)) + geom_point(),
  spec = ptr_spec
)

The preserve-mode code panel emits the formula on top, then — when the spec has at least one non-default entry — a blank line followed by a ptr_spec <- list(...) assignment:

ppUpload |> filter(ppVar > ppNum) |> ggplot(aes(ppVar, ppVar)) + geom_point()

ptr_spec <- list(
  `ggplot_2_stage_enabled` = FALSE,
  `ggplot_1_1_ppVar_NA`    = "bill_length_mm"
)

The panel does not wrap the formula in a ptr_app(...) call; the user composes the formula + spec into ptr_app(formula, spec = ptr_spec) themselves when they want to recreate the app. When the spec is empty (every value at default), the panel output is bit-identical to today's preserve-mode output — just the formula, no ptr_spec assignment. Spec encodes only deltas from default — readable, meaningful, small.

3.6 spec= apply at session boot

Inside ptr_server_internal, after the state is constructed and input bindings are wired:

  1. For each entry in spec, check whether the id matches this engine instance's namespace prefix. Skip if not.
  2. If the id is unknown to this engine instance (not present in the registry / no widget bound to it), silently drop with a single aggregate cli::cli_inform at the end listing the dropped ids.
  3. For each surviving entry, call the appropriate updateXyzInput at session boot via session$onFlushed (so widgets exist before we touch them).
  4. If the value is no longer a valid choice for the widget (e.g., a column that's gone from the user's data), drop with a cli::cli_warn. Let the registry default take over.

One-shot only. The spec is consulted at boot and then never again — no reactive watcher, no overriding subsequent user input.

3.7 Bug 3a fast-path deletion

The if (!is.null(state$resolved_data[[layer_name]])) { ... } block inside runtime_upstream_data (in R/paintr-server.R) — the per-layer fast-path — goes away. With the role-based tree, every consumer's c$upstream already encodes its true resolution point — for in-stage consumers, that subtree is the prior stage, not the full pipeline. The existing ptr_resolve_upstream(c$upstream, snapshot, ...) fallback handles every case correctly.

Regression-net: an e2e fixture per the §1.8 plan in ADR 0011 (adr12-filter-upstream): upload penguins.csv, assert the in-filter ppVar picker shows penguins' columns, assert the aes pickers still work whether or not filter has args.

5. Open questions carried into implementation

  1. Terminal-grounding precision — RESOLVED 2026-05-21. Verified that ptr_classify_data only tags ptr_ph_data_source nodes (no eval-env probing of bare symbols). The terminal-grounding rule was rewritten to "deepest first-arg is not itself a call" — a purely structural predicate that captures every intended terminal without requiring classify-data to do new work. See §3.2 for the rule and plan 02 SC for the implementer-facing form.
  2. Render-time pipe style preservation. If we want preserve-mode code panels to reflect the user's original pipe choice (|> vs %>%), the lift pass should record a per-pipeline render hint when the source string contains a literal pipe token. Out of scope for S1–S6; can be added later as a pure-cosmetic enrichment. Default for now: render as |> (R ≥ 4.1 universal).
  3. state$spec() at grid level — RESOLVED in S5. Each plot has its own ptr_state$spec(). S5 adds an internal helper ptr_spec_combine(specs) that unions per-plot specs into one flat list keyed by fully-qualified namespaced id. The grid wrapper (S6) uses this combiner to expose a single combined-flat spec for the grid, matching the spec= intake shape. A separate per-plot accessor (returning the list of per-plot specs un-combined) remains a future enhancement, not in S1–S6.

6. Relationship to ADR 0011

ADR 0011 documented the bug-3a / 3b investigation up to the (incorrect) UseMethod / load_all shadow-trap diagnosis. This ADR supersedes that diagnosis and proposes the structural fix. The two failed fix attempts in 0011 §2.6 and §2.7 should not be re-attempted — they were solving a non-existent problem.

0011's §1.5–§1.8 (Bug 3a hypothesis and fix shape) is partially correct: the per-layer fast-path is indeed the trip-wire, but the "is-inner" predicate proposed in §1.7 is no longer needed under this ADR — the structural pipeline lift makes every consumer's c$upstream resolve correctly without a per-layer override.

0011 stays as the historical investigation log; this ADR is the live design.
