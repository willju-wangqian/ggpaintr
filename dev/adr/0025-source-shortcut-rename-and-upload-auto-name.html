ADR 0025 — Source shortcut rename + upload auto-name + mutex-with-autoclear

Status: Accepted. Drafted 2026-05-26 from a /grill-with-docs session on branch shared-id-upgrade. Implemented (verified in source 2026-05-28: node$shortcut_id / node$auto_name in R/paintr-server.R + R/paintr-substitute.R; non-shared upload auto-name df_<hash(node$id)> per commit de1edd1; §3b consumer-seeding contract (ii) implemented and verified 2026-05-28).

Date: 2026-05-26

Relates to: ADR-0024 (promoted the companion from name override to data-loading entry point; this ADR completes that direction by renaming the surface and resolving the residual dual-role failure modes), ADR-0010 (ppUpload(name) positional arg form; preserved here as formula-shorthand for shortcut-textbox seed), ADR-0023 (panel-owned source binding via the coordinator's eval_env; the namespacing rule below extends to obj$id-prefixed auto-names), ADR-0015 (assign-before-signal invariant via bind_source_value; preserved unchanged), ADR-0006 (single public server entry; this ADR adds no new public server surface).

Supersedes: ADR-0024's Scope cuts bullets on F1 (companion-name binding sticky across an upload-clear) and on the dual-role conflict more broadly. ADR-0024 explicitly deferred those — this ADR resolves them by retiring the dual role rather than reconciling it.

Context

ADR 0024 promoted the source companion textInput (rendered beside ppUpload's fileInput) from name override for an uploaded frame to data-loading entry point. The promotion was a semantic widening of an already-keyword-agnostic mechanism: removing the is.null(node$default) guard at R/paintr-server.R:964 (try_bind_source_default) and R/paintr-server.R:1110 (try_bind_source_default_resolved) let a typed companion name bind a frame from state$eval_env's parent chain even without a default= on the source node.

Three failure modes survived the promotion, each empirically demonstrable (catalogued in dev/notes/placeholder-role-coverage2.html FINDING #8 follow-up):

  • F1 — Sticky-after-clear. A successful upload writes assign(name, df, envir = state$eval_env). Once the textbox is cleared (user deletion, or programmatic clear), the eval_env binding lingers because no path actively rm()s it; state$bound_names[[key]] stays non-NULL; downstream consumers keep reading the stale frame.
  • F2 — Code/plot mismatch. The user uploads penguins.csv but typed "mtcars" into the companion textInput → plot draws penguins (from the upload), code panel reads mtcars |> ggplot(...) (from the companion's name-override semantics). Two source-of-truth values for "what is this slot's data."
  • F3 — Eval-env pollution. Upload binds under a typed name that previously resolved to a different env-resident frame; the upload silently overwrites it inside state$eval_env. Same root cause as F1+F2 (uploads and the entry-point lookup share state$eval_env without provenance tracking).

F1, F2, F3 all share the same root: the companion textbox has two jobs — name override (for uploads) and env shortcut (for non-upload binds). Reconciling the two via origin tracking is possible but heavy; retiring one of the two is lighter. This ADR retires the name override role and keeps the env shortcut role, then derives the cascade of consequences (rename, auto-name generation for uploads, mutex between file and textbox, code-panel prologue, spec round-trip, vacate-on-empty).

Decision

1. Rename: companion_id_fn → shortcut (boolean); node$companion_id → node$shortcut_id

The public registry parameter on ptr_define_placeholder_source() changes from companion_id_fn = function(id) -> string (NULL default) to shortcut = FALSE (boolean default). Authors who want their custom source to opt in to env-shortcut behavior set shortcut = TRUE and render a shiny::textInput(inputId = node$shortcut_id, ...) inside their build_ui. Authors who don't opt in get a source that works like today's fileInput — no textbox, no env-shortcut path. The built-in ppUpload registers with shortcut = TRUE (preserves its current textbox).

Default-FALSE rationale (overrides initial inclination to default TRUE): auto-wrap at the framework level would force every custom source to carry a textbox the author may not want; author-emit at default-TRUE creates a silent-breakage trap (forget the line → textbox absent → no binding, no error). Default-FALSE makes opt-in explicit on both sides: the registry knob declares intent, and the build_ui realizes it.

The derivation rule for the id field on the node:

# R/paintr-ids.R::assign_id_to_placeholder
if (is_ptr_ph_data_source(node)) {
  entry <- ptr_registry_lookup(node$keyword)
  if (isTRUE(entry$shortcut)) {
    node$shortcut_id <- paste0(node$id, "_shortcut")
  }
}

The "_shortcut" suffix replaces today's "_name". Naming is consistent across the field, the suffix, and the public arg.

Reserved-name validator. The id derivation collides with another node$id only if some node's id ends literally in _shortcut. By id format <layer>_<path>_<keyword>_<shared-or-NA> (R/paintr-ids.R:42), this requires a user-supplied shared key of "shortcut". Translate-time validator rejects this with a clear error: "Shared key 'shortcut' is reserved by the source-shortcut id derivation. Pick a different shared key."

Hard rename, no lifecycle::deprecate_warn(). Package is 0.9.2, pre-1.0; ADR 0006 set the precedent (ptr_module_* → ptr_*, hard); the only in-tree consumer is ppUpload; no third-party callers identified. companion_id_fn arg is removed; node$companion_id field is removed; both internal call sites (R/paintr-ids.R:49-50, R/paintr-shared-ui.R:478-479) and the substitute walker (R/paintr-substitute.R:181-184) update to the new names.

2. UI-level mutex with soft-lock + auto-clear (Q3-B from grill)

The shortcut textbox and the fileInput are mutually exclusive: filling one auto-clears the other. There is no hard-greyed state — the act of using one cancels the other.

  • User types in the shortcut textbox ⇒ framework calls shiny::reset (or equivalent) on the sibling fileInput.
  • User picks a file ⇒ framework calls updateTextInput(value = "") on the sibling shortcut textbox.
  • Boot-time exception: ppUpload(penguins) (ADR 0010) seeds the shortcut textbox with "penguins" at boot. That boot seed does not count as a "first interaction" — the fileInput remains live; picking a file silently wipes the seeded textbox under the normal auto-clear rule.

Rationale. The two affordances feed the same source slot; without mutex, every resolution path must arbitrate precedence and every test must specify which input wins. The mutex collapses the matrix to "whichever was last touched is the live one." F1/F2 collision modes by construction cannot arise in this UI.

3. Auto-name for uploads = df_<hash(node$id)> (non-shared) / <canonical> (shared) / node$default (explicit)

When a file is uploaded and the shortcut textbox is empty, the upload's frame binds into state$eval_env under a system-generated symbol. The symbol is derived deterministically from the existing node$id infrastructure:

Source shape Auto-name (the symbol in eval_env) Lives in
ppUpload(shared='ds_a') with obj$id = NULL ds_a coordinator eval_env (panel) or owning instance eval_env (section)
ppUpload(shared='ds_a') with obj$id = "main" main_ds_a coordinator eval_env (panel) or owning instance eval_env (section)
ppUpload() non-shared, no explicit name df_<hash(node$id)>, where hash = substr(rlang::hash(node$id), 1, 6) (e.g. df_158206 for node$id = "ggplot_1_0_ppUpload_NA") owning instance eval_env
ppUpload(penguins) (ADR 0010) penguins (the node$default verbatim) owning instance eval_env

Uniqueness analysis. Verified against existing infrastructure:

  • Each per-instance state$eval_env is its own scope (created at R/paintr-server.R:222 as new.env(parent = envir); passed in via R/paintr-app.R:331, 1209). Two formulas in ptr_app_grid cannot collide because they have separate eval_envs.
  • Each coordinator's eval_env is its own scope (R/paintr-shared-ui.R:534 eval_env <- new.env(parent = envir)). Two ptr_shared(...) calls on one page produce two coordinators; same canonical in each lands in distinct eval_envs.
  • Per-instance eval_envs extend a coordinator's eval_env via parent chain (ADR 0023). Cross-coordinator lookup isn't a thing — each instance binds to exactly one coordinator via shared = obj.
  • Within one coordinator, canonicals are unique by construction.
  • Within one formula's tree, node$id is unique by assign_id_to_placeholder (R/paintr-ids.R:42).

Together: every auto-name is unique within its eval_env. No counter, no reserved-namespace beyond the shortcut validator above.

Symbol-flip on upload is accepted. Under ppUpload(penguins) (ADR 0010 form), the slot boots with textbox seeded "penguins" and binds caller-env penguins via env shortcut. If the user uploads a file, Q3-B's auto-clear empties the textbox, and the upload binds under the system auto-name df_<hash(node$id)> (e.g. df_158206) — deliberately decoupled from node$default so the user's env-shortcut symbol (penguins) does not leak onto uploaded data (the bug fixed in de1edd1). The code panel symbol flips from penguins to df_<hash(node$id)>. This is acceptable because the prologue line (Decision §4) makes the new symbol legible at a glance: df_158206 <- read.csv("uploaded.csv"). One mechanism (textbox seed → eval_env name); no special "reserved auto-name" sub-mechanism.

3b. Consumer pickers do not inherit the formula default for a NEW post-boot source — contract (ii)

Decision (locked 2026-05-28). A consumer placeholder (ppVar and friends) seeds its formula-side default only at boot, and only against the data that is present at boot. Once the app is running, any new data source — a file uploaded through ppUpload's fileInput (or a custom source's file-like mechanism), or the shortcut textbox renamed to a different dataset — clears the consumer picker. The picker comes up populated (the new frame's columns are offered) but with no selection; the user re-picks. Neither the formula default nor a stale prior pick carries onto the new data. Boot-time spec= seeding (ADR 0012) is the only thing that may override the default, and only at boot.

Why. An uploaded or swapped dataset is new work; its columns are not the parse-time context the formula author wrote the default (ppVar(mpg)) against, so auto-selecting that column would be a guess. By contrast, an env-default boot frame (e.g. ppUpload(mtcars) resolving mtcars via the shortcut at boot, per ADR 0010 + §3) is author-known data, so the formula default does seed at boot — that path must not regress.

Situation Consumer picker
Boot over an env-default frame (ppUpload(mtcars)), no spec= formula default seeds (mpg selected)
Boot with spec=list(<id> = "wt") wt (spec overrides the default, boot-only)
Post-boot: a file is uploaded cleared (unselected) — default does not carry
Post-boot: shortcut renamed to another dataset cleared (unselected)
After a clearing upload, the user picks a column that pick persists across re-renders of the same source

Mechanism (deterministic, not a reactive-flush race). The clear is edge-triggered on a change to the upstream source's identity, not a level check of "a file is present" (which would wipe the user's pick on every re-render and forbid picking). Identity = the fileInput datapath (changes on every upload, even same filename) + the bound dataset name (catches a shortcut rename), snapshotted by consumer_upstream_source_state(); the clear is gated to user-supplied data so the boot env-default settling never fires it. The decision is centralised in consumer_seed_decision(clear_for_new_source=) (R/paintr-server.R) and applied identically by both consumer binders — ptr_setup_consumer_uis() (non-shared) and ptr_bind_shared_consumer_uis() (shared) — so the two cannot drift.

History. Before this, the outcome held only by accident of reactive scheduling: the boot-injected default was echoed back as the input value, so a post-boot upload carried the default (e.g. mpg) onto the new frame. Commit de1edd1 (the §3 auto-name fix) flagged "make contract (ii) intentional rather than timing-incidental" as a follow-up; this section is that follow-up. Regression coverage: tests/testthat/test-spec-seed-boot-only.R (unit, consumer_seed_decision) and tests/testthat/test-adr25-upload-default-clear.R (e2e: boot-seed + upload-clear).

4. Code-panel prologue with original filename + reader-fn-mirror

For every active upload-bound source, the code panel renders a prologue line above the formula:

upload_data1 <- read.csv("penguins.csv")  # uploaded as: penguins.csv
upload_data1 |> ggplot(aes(x = mpg)) + geom_point()

The prologue is aspirational, not honest-to-the-byte: ggpaintr read the file from a tempfile, not from "penguins.csv" in the user's cwd. The user is responsible for reproducing the binding themselves (running the prologue in their own R session before running ggpaintr with the spec). This contract is the same as ADR 0010's ppUpload(penguins) — reproducibility-by-precondition, not reproducibility-by-self-contained-snippet.

The reader function in the prologue mirrors ggpaintr's actual choice based on file extension (read.csv for csv, readxl::read_excel for xlsx, etc.). Custom sources defined via ptr_define_placeholder_source can supply their own prologue line via a registry hook (deferred — current scope is built-in ppUpload only).

5. Eval_env contract preserved (minimum-change path)

Uploads bind into the existing state$eval_env via the existing bind_source_value writer (R/paintr-server.R:884-890), under the auto-name from Decision §3. The load-bearing comment at R/paintr-server.R:872-883 (assign-before-signal invariant) stays intact. No parallel state$upload_payload, no fresh child env, no dispatch fork. The substitute walker (R/paintr-substitute.R:179-213) reads ctx$snapshot[[node$shortcut_id]] first; on empty, falls back to node$auto_name (a new field stamped at translate time alongside node$id).

This is the minimum-change path verified against the source — every other candidate (parallel upload slot, child env) reintroduced the eval_env binding under a different name eventually, adding cost without paying it back.

6. Spec round-trip captures the auto-name

ptr_runtime_input_spec() dumps input[[shortcut_id]] values into spec[[shortcut_id]] as today. When the textbox is empty but state$bound_names[[key]]() holds an upload-derived name, the dump falls back to the bound name:

# R/paintr-input-spec.R, source-placeholder branch with `shortcut = TRUE`
val <- input[[shortcut_id]] %||% ""
if (!nzchar(val)) {
  bound <- state$bound_names[[key]]()
  if (!is.null(bound)) val <- bound
}
spec[[shortcut_id]] <- val

Reproducing a session = the consumer runs <auto-name> <- read.csv("...") in their R session, then ptr_app(formula, spec = list(...)). The spec seeds the textbox at boot with the auto-name; resolution finds <auto-name> via env shortcut. Equivalent to running ppUpload(<auto-name>) as the formula — the spec is the bridge between live upload state and reproducible env-resident state.

7. Vacate-on-empty + debounce (Q13 A1 + A2)

A1. When resolve_upload_source falls through to slot(NULL) without binding (file_info NULL and try_bind_source_default_resolved returned FALSE), actively vacate the slot:

# R/paintr-server.R::resolve_upload_source, no-file branch
prior_name <- shiny::isolate(state$bound_names[[key]]())
if (!is.null(prior_name) &&
    exists(prior_name, envir = state$eval_env, inherits = FALSE)) {
  rm(list = prior_name, envir = state$eval_env)
}
state$bound_names[[key]](NULL)
slot(NULL)

Without A1, the textbox-deletion sequence (upload → type → delete text) leaves state$bound_names[[key]] stale; consumer pickers stay populated with stale columns; ptr_runtime_input_spec() dumps the stale name. The plot widget itself preserves last-good on req-failure (independent of A1) — A1 fixes the system-state surfaces, not the plot's carry-over behavior.

A2. Debounce the shortcut textbox input read with a 300-500ms window (recommend 400ms) so intermediate keystrokes don't trigger env manipulation:

# at the observer's input-read site
shortcut_val <- shiny::reactive(input[[shortcut_id]]) %>% shiny::debounce(400)

Eliminates the "Object 'm' not found / Object 'mt' not found / ..." error-pane flicker as the user types a name; reduces rm/assign churn from one-per-keystroke to one-per-decision.

8. ADR 0010 reinterpretation (no superseding)

ppUpload(penguins) remains valid; ADR 0010's stated user-facing semantics ("default-but-overridable, with a meaningful display name") are preserved, but the mechanism is reinterpreted: under this ADR, ppUpload(penguins) is formula-shorthand for "seed the shortcut textbox with 'penguins' at boot." There is no separate name-reservation mechanism; the textbox seed is the only handle. Post-upload (auto-clear), the slot's binding name is the system auto-name (node$id), not penguins — accepted per Decision §3.

ADR 0010's node$default storage shape (single character scalar) is unchanged. Substitute walker behaviour for ppUpload(penguins) outside ptr_app (identity returning the value of penguins from the caller env) is unchanged.

Scope cuts

  • Vignette updates deferred. vignettes/ggpaintr-customization.Rmd:297, 502 reference node$companion_id and companion_id_fn. Update bundled with the implementation, not pre-emptively in this ADR.
  • NEWS.md entry deferred to the implementation PR.
  • CONTEXT.md amendment owed. Lines 48 ("exactly one per page" for shared panel) and 132 (dialogue reinforcing same) contradict the source code at R/paintr-shared-coordinator.R:220-234, 292 (which accepts id and namespaces every shared id). The contradiction predates this ADR; this ADR's auto-name namespacing rule (obj$id-prefixed shared canonicals) depends on multi-panel support being part of the documented surface. Amend CONTEXT.md in the same PR that implements this ADR.
  • Custom-source prologue hook deferred. Decision §4 mirrors the reader fn for built-in ppUpload only. A registry-level prologue_emit_fn(node, snapshot) -> string for third-party source types can land alongside the first such source.
  • No origin tracking. The handoff's old Option B (tag eval_env bindings with provenance) is rejected — Q3-B's mutex collapsed the "both file and shortcut active" case that origin tracking would have arbitrated. A1's vacate-on-empty handles the remaining staleness sources.
  • No "clear file" UI button. Vanilla Shiny's fileInput has no clear affordance and this ADR doesn't add one. Programmatic clear (via Q3-B auto-clear or spec replay) is the only way file_info goes from non-NULL to NULL.

Consequences

Positive

  • F1, F2, F3 retired by construction. F2 is dead because the textbox no longer overrides upload names. F3 is dead because uploads bind under system auto-names that cannot collide with user-typed shortcut names. F1's user-reachable surface (textbox-deletion-after-bind) is fixed by A1.
  • One mechanism, no special-cases. The textbox seed is the only handle on the source's binding name — whether typed by the user, set by ppUpload(name), applied via spec=, or fallen back to the system auto-name on upload.
  • Audience separation honoured. Group 1 (formula-authoring R users) lives in the textbox + env shortcut + ppUpload(name) path. Group 3 (non-R-using deployed-app consumers with their own data) lives in the fileInput + auto-name path. Mutex prevents either from accidentally invoking the other's mechanism.
  • Spec round-trip is meaningful for both groups. Group 3 can export spec from a session and hand it to a Group 1 user, who runs the prologue + ptr_app(formula, spec=) and reproduces the session exactly.
  • Minimum implementation surface. No new state slots; no parallel resolution pipeline; no novel env layer; uses existing bind_source_value / bound_names / eval_env contracts unchanged. Estimated ≤150 LOC across R/paintr-server.R, R/paintr-ids.R, R/paintr-substitute.R, R/paintr-registry.R, R/paintr-builtins.R, R/paintr-input-spec.R.

Risks & mitigations

  • Hard rename breaks third-party callers of companion_id_fn. No such callers are known. Mitigation: clear NEWS.md entry in the implementation PR; the only in-tree caller is the built-in ppUpload which moves to shortcut = TRUE.
  • Code-panel symbol flip on upload surprises users. Mitigated by the prologue line making the system auto-name legible at a glance. Recovery is one rename (the user adds ppUpload(my_name) to the formula or types my_name into the textbox).
  • Debounce window choice (400ms) may feel sluggish to fast typists or laggy on slow workstations. Mitigation: document the window; consider making it configurable via a package option (options(ggpaintr.shortcut_debounce = 400)) if real-world feedback warrants.
  • A1's rm() from eval_env interacts with caller-env shadowing. rm() removes only the local binding in state$eval_env; caller-env bindings reached via inherits = TRUE are untouched. This is the intended scope — A1 vacates only what ggpaintr itself put there.
  • Reserved-name validator may surprise embedders using shared = 'shortcut'. Mitigation: clear translate-time error message names the conflict and suggests renaming the shared key.
  • Multi-panel scenario is now load-bearing on the auto-name design. If the project decides to roll back multi-panel support (obj$id), the shared-key auto-name collision protection it provides disappears. Cross-ref this ADR before any such rollback.

Worked examples — what becomes possible

Example 1 — Custom source opts in to env-shortcut

A third-party author defines a custom source that loads frames from the embedder's environment via a typed name. With shortcut = TRUE, the registry stamps node$shortcut_id; the author renders a textInput bound to it inside their build_ui.

ptr_define_placeholder_source(
  keyword       = "ppDataset",
  role          = "source",
  shortcut      = TRUE,
  build_ui      = function(node, label = NULL, ...) {
    shiny::tagList(
      shiny::selectInput(node$id, label %||% "Dataset", choices = c("", "demo_a", "demo_b")),
      shiny::textInput(node$shortcut_id, "Or type env frame:", value = "")
    )
  },
  resolve_data  = function(value, node) get(value, envir = parent.frame()),
  resolve_expr  = function(value, node) rlang::sym(value)
)

ptr_app("ppDataset() |> ggplot(aes(x = ppVar('col'))) + geom_bar()")
# user types "mtcars" in the shortcut textbox -> binds caller-env mtcars,
# ppVar picker populates with names(mtcars), plot draws.

Example 2 — Multi-panel app with same shared canonical, no collision

Two coordinators on one page each declare a shared key ds; the obj$id prefix prevents the auto-name binding from colliding in either coordinator's eval_env or in the rendered code panels.

obj_left  <- ptr_shared(list(f1_left,  f2_left),  id = "left")
obj_right <- ptr_shared(list(f1_right, f2_right), id = "right")

# Both f1_left and f1_right contain `ppUpload(shared = 'ds') |> ggplot(...)`.
# When the user uploads to each panel's shortcut, state$eval_env binds the
# left frame under symbol `left_ds` and the right frame under symbol
# `right_ds`. Code panels read distinct symbols; no collision.

Example 3 — Spec round-trip reproduces an uploaded-data session

A Group-3 user uploads a file in a deployed app and exports the spec. A Group-1 user runs the prologue locally and reloads the spec; the textbox seed bridges the upload-only session to an env-resident reproduction.

# Original (Group 3) session: user uploaded penguins.csv into the empty
# textbox, so the frame bound under the auto-name df_<hash(node$id)> (§3).
# The shortcut INPUT id is <node$id>_shortcut; its captured VALUE is that
# auto-name. ptr_runtime_input_spec() dumps:
spec <- list(`ggplot_1_0_ppUpload_NA_shortcut` = "df_158206")

# Reproducer (Group 1) prepares caller env under the auto-name, then reloads:
df_158206 <- read.csv("penguins.csv")
ptr_app(
  "ppUpload() |> ggplot(aes(x = ppVar('species'))) + geom_bar()",
  spec = spec
)
# Boot: shortcut textbox seeded with "df_158206"; env shortcut
# resolves the caller-env frame; plot identical to original session.

Worked examples — what is rejected by design

Example 4 — Shared key 'shortcut' rejected at translate time

The reserved-name validator surfaces a translate-time error rather than letting the user discover a silent shortcut_id collision at runtime.

ptr_app("ppUpload(shared = 'shortcut') |> ggplot(aes(x = ppVar)) + geom_bar()")
# Error: Shared key 'shortcut' is reserved by the source-shortcut id
#   derivation. Pick a different shared key.

Example 5 — Concurrent upload + typed-text is structurally impossible

The Q3-B mutex makes the two affordances mutually exclusive at the UI level. Picking a file auto-clears the textbox; typing in the textbox auto-clears the file. There is no UI state with both active simultaneously, so the F2-shape code/plot mismatch cannot be expressed.

# Scenario the design rejects:
# (1) user uploads penguins.csv
# (2) user types "mtcars" in the shortcut textbox
# Without Q3-B: plot would draw penguins (from upload) while code panel
# reads `mtcars |> ggplot(...)` (F2 mismatch).
# With Q3-B: step (2) calls shiny::reset on the fileInput; the upload's
# file pill disappears. Only "mtcars" remains active. Plot draws mtcars;
# code panel reads `mtcars |> ggplot(...)`. No mismatch reachable.

Example 6 — Custom source with shortcut = TRUE but no textInput(node$shortcut_id) in build_ui

The registry stamps node$shortcut_id but the author's build_ui never binds an input at that id. No textbox renders; the env-shortcut path has no input to read. This is an author trap, not a translate-time error — the contract is "opt in on both sides" and silent breakage when one side opts out is the documented consequence of default-FALSE-author-emit (versus the rejected auto-wrap option).

ptr_define_placeholder_source(
  keyword  = "ppBroken",
  role     = "source",
  shortcut = TRUE,
  build_ui = function(node, label = NULL, ...) {
    # Author forgot to render shiny::textInput(node$shortcut_id, ...)
    shiny::selectInput(node$id, label %||% "Source", choices = c("", "a", "b"))
  },
  resolve_data = function(value, node) NULL,
  resolve_expr = function(value, node) rlang::sym(value)
)
# Boot: no shortcut textbox in the UI. User cannot type an env frame name.
# The env-shortcut behavior advertised by `shortcut = TRUE` is unreachable.
# No error; the source acts like `shortcut = FALSE` from the user's perspective.

Full authoritative gate post-merge: FAIL 0 / WARN 0 / SKIP 0 / PASS 3110.

Historical record (the bug as it existed pre-fix; preserved for audit-trail completeness):

A pre-existing bug in the ppUpload(<name>) boot path was discovered during the design grill but is not in this ADR's scope. Captured here so the next implementer is not surprised when hand-testing.

Symptom: with df_main <- mtcars in caller env, the app

ptr_app(
  ggplot(ppUpload(df_main), aes(x = ppVar(hp), y = mpg)) + geom_point()
)

boots with the ppVar(hp) column picker empty — even though the seeding mechanism at R/paintr-builtins.R:245 (value = node$default %||% "") and the boot-race fallback in try_bind_source_default / try_bind_source_default_resolved are both implemented (ADR 0010 / ADR 0024). Removing ppUpload (using ggplot(mtcars, …) directly) makes the picker populate correctly.

Why out of scope: the failure path is observer ordering between the source bind and the consumer picker's req(state$bound_names[[key]]) evaluation, OR the consumer picker is suspended at first paint per the .claude/rules/testing.md "var pickers are suspended" note.
