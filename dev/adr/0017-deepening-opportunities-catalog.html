ADR 0017 — Deepening opportunities catalog

Status: Proposal. Records candidates surfaced by a /improve-codebase-architecture sweep on 2026-05-24. No decision yet on which to action; each candidate is to be grilled and either promoted to its own ADR (with a concrete deepened design) or rejected with a recorded reason.

Date: 2026-05-24

Relates to: ADR-0005 (L2/L3 redesign — shared coordinator/partition implementation pending), ADR-0006 (single public server entry; L3 UI-only), ADR-0012 (tree is semantic, not syntactic), ADR-0014 (registry init ordering), CONTEXT.md (L1/L2/L3 embedding model and partition rule).

Context

An architectural sweep audited the 33 R/paintr-*.R files (~13 350 lines total) for shallow modules — modules whose interface is nearly as complex as their implementation, or whose split is purely lexical rather than structural. The sweep used Ousterhout's deletion test ("if I delete this module, does complexity vanish or concentrate?") and read against CONTEXT.md's locked L1/L2/L3 + shared-coordinator vocabulary.

This ADR catalogs the candidates so they can be discussed one at a time. It does not propose interfaces; each candidate that survives grilling will get its own ADR with the concrete deepened design.

Candidates

Ranked by "real bugs hide here," not by neatness.

C1 — Partition's ownership decision is communicated through three asymmetric channels

Files: R/paintr-shared-coordinator.R (algorithm), R/paintr-shared-ui.R + R/paintr-shared.R (state bundle, consumer rewrite), R/paintr-app.R (render_shared_section boolean, 6 sites), R/paintr-server.R (ptr_bind_local_shared_consumers's host_owned_keys argument), tests/testthat/test-shared-stage-checkbox.R (2 sites using the boolean).

Initial framing corrected after verification (2026-05-24): the partition algorithm (shared_partition()) is centralised — it is called exactly once, from inside ptr_shared(). is_shared_placeholder is a 3-line predicate (different question), and wrap_shared_widgets_with_stage_blocks receives orphan_stages from its caller without recomputing. The earlier "re-derived at every consumer" framing was wrong.

Real symptom: the partition's ownership decision ("does this surface render and bind this key?") is communicated to consumers through three different, asymmetric channels:

  1. Multi-instance: the rich coordinator obj carrying panel_keys + local_keys_by_formula flows downstream.
  2. Single-formula L1: no coordinator exists (CONTEXT.md, locked 2026-05-16: "single-instance has no coordinator — ever"); render_shared_section = TRUE boolean stands in for "this surface owns all shared keys."
  3. L2 ptr_server: ptr_bind_local_shared_consumers(host_owned_keys = …) — the complement of panel-owned keys, computed separately by whichever caller wires it up.

Same conceptual question, three different signals to keep consistent. The 2026-05-22 partition-binder bug (project memory project-shared-section-binder) lived in channel #3 — drift between the channels surfaced as a defect.

Deletion test: the partition algorithm itself is not duplicated and survives the test. What fails the test is the consumption side: removing any one channel forces re-encoding the ownership question in the remaining ones.

Deepening direction: introduce an internal partition_view(part, surface_id) abstraction (not public — user-facing API unchanged) that all three channels query. Multi-instance keeps building obj as today; single-instance gets a trivial obj constructed internally (owning all keys); L2 server's host_owned_keys becomes a lookup against the view, not a separate list.

Benefit (with "internal only" scope):

  • Locality of the ownership invariant. One place answers "does this surface own this key?" for every surface. The 2026-05-22 bug-shape becomes structurally hard to reintroduce — a new shared-key surface has no "compute it correctly" step to get wrong.
  • Test leverage. "L2 omits panel-owned keys" becomes a unit test against the partition view, not a full-app boot.
  • Smaller paintr-server.R. The setdiff dance in ptr_bind_local_shared_consumers moves into the view; binders read a ready list.

What it does not buy: no public API change; no reduction in file count; the algorithm was already centralised.

Cost estimate: ~150–250 LOC net. 6 boolean-→-view call-sites in paintr-app.R; ~3 server call-sites; 2 tests. ⚠ UNVERIFIED on the L2 server-side call-site count — needs a final grep before locking the design.

ADR conflict check: per discussion 2026-05-24, ADR-0005's pending shared-coordinator work is considered stale and out of scope; this candidate stands on its own merits.

Status: Deferred. Real but narrow. Worth it if new shared-key surfaces are expected or if test-time leverage on partition logic is wanted; skip if the 2026-05-22 fix is trusted to hold and no new surfaces are planned.

C2 — ptr_server_internal's invariants live in comments; pure-core extraction is partial

Files: R/paintr-server.R (orchestrator lines 726–785; 13 setup/register helpers), R/paintr-headless.R (existing pure-core ptr_exec_headless with reactive adapter ptr_setup_runtime), tests/testthat/test-runtime-headless.R (pure-path tests), tests/testthat/test-shared-stage-checkbox.R (one helper directly invoked).

Symptom (verified 2026-05-24): the orchestrator encodes four invariants as body comments, not as seam shape:

  1. shared_state overlay precedence ("explicit ... wins"), lines 736–745.
  2. Namespace auto-wire only inside moduleServer, lines 750–756.
  3. Value/source/consumer renderUI call order = registry-row order in ptr_runtime_input_spec(), lines 765–768.
  4. ptr_register_last_ok_cache MUST precede ptr_register_plot / _code (registration order = observer firing order), lines 775–779.

Silently reorder two lines and the first-flush spec seed or observer cache breaks; nothing structural objects.

Initial framing corrected after verification: the "every helper is single-use outlining" framing was too aggressive. Of 13 helpers: 1 already has the pure-core/reactive-adapter pattern (ptr_setup_runtime ⇄ ptr_exec_headless); 1 (ptr_setup_shared_stage_enabled) is directly invoked by a test and is a real seam; the rest are single-use. The "headless" name is also misleading post-rewrite — it is the pure runtime core, not an out-of-Shiny path.

Deepening splits into two distinct sub-candidates:

C2a — Extract pure cores from the 3 mixed helpers

Pull apply_spec_at_boot's validation, ptr_register_code's render logic, and ptr_setup_pipelines's binding_name derivation into a sibling module (extend paintr-headless.R after renaming to paintr-runtime-core.R, or new file). Each becomes pure helper + thin reactive adapter calling it.

  • Buys: ~150 LOC of unit-testable logic without Shiny; smaller paintr-server.R; the I/O surface becomes explicit per helper.
  • Cost: ~200 LOC moves + ~150 LOC of new pure tests.
  • Skip if: the current reactive tests are trusted to catch spec-seed / code-render regressions.

C2b — Group thin-wiring calls into named phases

Reorganise ptr_server_internal's body into 2–3 phase functions whose names encode the four invariants above. Helpers stay shaped as today; only the orchestrator changes.

  • Buys: the invariant comments become structural — reordering becomes obviously wrong, not subtly wrong.
  • Cost: ~30 LOC reorganisation, no behavioural change.
  • Worth doing regardless of C2a.

Status: Deferred. C2b is cheap and broadly applicable; promote whenever there's appetite for a low-risk structural pass. C2a is medium-effort / high-ROI only if the spec-seed / code-render paths are deemed fragile.

Naming side-effect (if either C2a or C2b promotes): rename paintr-headless.R → paintr-runtime-core.R (or similar) and update CONTEXT.md to document "runtime core" as the pure-functional substrate; "headless" name is a misleading legacy from the removed public path.

C3 — Translate→render pipeline has 11 files and no named canonical tree

Files: paintr-translate.R (810), paintr-walk.R (244), paintr-classify.R (103), paintr-nodes.R (150), paintr-prune.R (242), paintr-substitute.R (280), paintr-resolve.R (100), paintr-eval.R (144), paintr-region.R (103) → paintr-render.R (636).

Symptom: ADR-0012 locks "tree is semantic, not syntactic" — but the canonical form lives only in contributor heads. Each downstream pass writes its own walker. paintr-render.R is larger than paintr-translate.R because the resolved tree still carries surface detail the renderer must re-canonicalise. A "what does this placeholder look like at render time?" question crosses 4–6 files; tests reach with ::: to validate intermediate stages because there's no public path through the pipeline.

Deletion test: any single pass file deleted forces its logic to reappear inline in the next pass — but the need for so many passes is itself the question. If the resolved tree were named and documented, several passes might fold.

Deepening direction: introduce a named intermediate (provisional name: resolved tree) with a documented shape, produced once by translate+classify+resolve and consumed by everything downstream as a thin S3 dispatch. Render shrinks because it no longer reconstructs decisions.

ADR conflict check: directly extends ADR-0012's "one canonical tree" principle from parser surface syntax to downstream pipeline contract.

C4 — paintr-app.R conflates L1, L2, L3 surfaces and shared-binding plumbing

Files: R/paintr-app.R (1488), housing ptr_app / ptr_app_grid (L1), ptr_ui / ptr_server (L2), ptr_ui_plot / ptr_ui_error / ptr_ui_code / ptr_ui_controls / ptr_ui_inline_error / ptr_ui_toggle_code / ptr_ui_header / ptr_ui_page (L3 bare pieces + combinators), plus ptr_make_app_server and shared_section_tags (L1-only shared-binding plumbing).

Symptom: CONTEXT.md treats L1, L2, L3 as distinct seams — three audiences, three contracts. The file does not. A new contributor cannot tell from file shape what an L2 user is required to write versus what is L1-only convenience. ptr_make_app_server sits next to the L2 pair but only L1 uses it, blurring whether L2 embedders must call something similar.

Deletion test: the file does not meaningfully concentrate complexity — splitting it by audience would lose nothing and gain a per-seam interface a reader can hold in their head.

Deepening direction: split along the audience seam (working names: paintr-l1-app.R, paintr-l2-embed.R, paintr-l3-pieces.R) and make L2 self-sufficient for shared binding (or explicitly delegate to the coordinator), rather than relying on L1-only plumbing.

C5 — paintr-safe.R + paintr-safety.R: split topic, shared contract

Files: R/paintr-safe.R (125, ptr_complete_expr_safe, ptr_assemble_plot_safe, ptr_validate_plot_render_safe) and R/paintr-safety.R (90, ptr_validate_tree_safety + walker).

Symptom: two files split by when validation runs (parse-time vs render-time), not by what they protect. The actual invariant is one: "no untrusted expression ever reaches eval()." A denylist or walker change touches both files; a reader needs both open to answer any question about expr safety.

Deletion test: collapsing to one file does not regrow complexity.

Deepening direction: one expr-safety module with two named entry points — validate_tree (parse-time AST shape) and guard_eval (render-time tryCatch). Timing becomes a parameter, not a file boundary.

C6 — Placeholder-author contract is implicit and wide

Files: R/paintr-registry.R (731) + R/paintr-builtins.R (390); coupled to R/paintr-input-spec.R (442), R/paintr-default-args.R (293), R/paintr-checkbox-defaults.R (158).

Symptom: three exported entry points (ptr_define_placeholder_value/_consumer/_source) take 10+ arguments each (UI builder, resolver, named_args, default_arg, validators…). A custom-placeholder author must also know the NS/id convention, the first-flush ordering (encoded inside ptr_server_internal's comments — see C2), and the default-args/input-spec flow. The interface is wide; the contract is prose scattered across the registry source, the vignette, and ADR-0014.

Deletion test: shrinking the entry-point surface does not remove the contract — it makes it visible.

Deepening direction: introduce a placeholder spec object (constructor + validators) that bundles the contract, then ptr_register(spec) as the sole seam. Three wide entries collapse to one narrow one; the contract gains a name that can be documented in one place.

Honourable mentions

  • paintr-render.R size (636 lines, 47 functions) — likely an effect of C3 (resolved tree leaks detail); will shrink if C3 is taken.
  • Tests use ::: to reach internal pipeline stages — symptom of C3 not exposing the canonical tree, not an independent candidate.
  • Boolean render_shared_section flag — symptom of C1; replaced by the partition object once C1 is taken.

Decision

None yet — this is a proposal-status catalog. Each candidate will be grilled in turn. A candidate that survives gets its own ADR with a concrete deepened interface; a candidate that is rejected gets its rejection reason recorded inline in this ADR (so future architecture sweeps don't re-suggest it).

Process

  1. User picks a candidate to drill into.
  2. Grill the design — constraints, seam shape, what tests survive, what migrations the existing call-sites require.
  3. Update CONTEXT.md inline when terminology sharpens or new domain terms are introduced.
  4. Either: promote to its own ADR with locked decision + implementation plan; or: record rejection reason in this ADR's "Decisions" subsection below.

Decisions log

  • C2 — Deferred (2026-05-24). Split into C2a (extract pure cores from 3 mixed helpers; ~200 LOC of testable logic; medium-effort / high-ROI) and C2b (group orchestrator calls into named phases; ~30 LOC; low-effort / medium-ROI; worth doing regardless). Naming side-effect: rename paintr-headless.R → paintr-runtime-core.R if either lands. See C2 section above.
  • C3 — Promoted (2026-05-24). Verification corrected the framing on three counts (file count, render.R size, walker-primitive usage). Promoted as two independent ADRs:
    • ADR-0018 — C3a: eliminate per-pass S3 walker boilerplate. Status: Proposed, interface design deferred to fresh-session grilling (handoff produced 2026-05-24).
    • ADR-0019 — C3c: public staged pipeline (drop the 222 :::). Status: Proposed, independent of ADR-0018, can land first.
    C3b (tree subtypes) remains in this catalog, deferred pending an ADR-0012 conformity review.
