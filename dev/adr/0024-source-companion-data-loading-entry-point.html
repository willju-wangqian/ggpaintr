ADR 0024 — Source companion as data-loading entry point

Status: Accepted. Drafted & landed 2026-05-26. Authoritative gate: FAIL 0 / WARN 0 / SKIP 0 / PASS 3093 on the FINDING #8 + ADR 0024 commit on branch shared-id-upgrade.

Date: 2026-05-26

Relates to: ADR-0010 (introduced the companion textInput beside ppUpload's fileInput as a name override for the uploaded frame), ADR-0012 (added spec= + state$spec_seed; defined the source_companion role for spec dispatch), ADR-0015 (per-instance state$resolved_data / state$bound_names assign-before-signal invariant — the bind site this ADR widens), ADR-0023 (resolve_upload_source as the single resolver, with companion-driven name lookup at panel scope).

Implementation: in-flight on branch shared-id-upgrade alongside the FINDING #8 fix (dev/notes/placeholder-role-coverage2.html). Carries one source change (R/paintr-server.R — lift the is.null(node$default) guard in try_bind_source_default() and try_bind_source_default_resolved(), add structured error surface), one roxygen update (R/paintr-registry.R — promote the "Companion pattern" doc), and four shinytest2 cases. No plan directory — the change is one self-contained surface.

Context

The companion_id mechanism was introduced by ADR-0010 as a name override for an uploaded data frame: the user uploads a file, types an optional name into the sibling textInput (shared_<k>_name / <source-id>_name), and that name appears in the generated code panel and in state$eval_env as the symbol that substitute_walk.ptr_ph_data_source() emits. Without an upload, the companion was just a label, not a data source.

Three subsequent ADRs incrementally generalised the companion's role:

  • ADR-0012 made the companion a first-class spec target — spec = list(<companion-id> = "df_name") dispatches via apply_spec_entry()'s source_companion branch (R/paintr-server.R:691-695) to updateTextInput(). Spec carries a name into the companion at boot.
  • ADR-0015 added try_bind_source_default() — when no upload is present, look up the companion's value (or node$default on the boot race) in state$eval_env and bind it. This is the FIRST point where a typed name without an upload could actually bind a frame.
  • ADR-0023 made resolve_upload_source() the single resolver across per-instance and panel scopes, called from the bare-data-source-layer loop, pipeline-head loop, and panel-source observer. All three paths fall through to try_bind_source_default[_resolved]() when no upload is present.

But all three paths to a name-driven bind go through the same gate at R/paintr-server.R:945 / R/paintr-server.R:1069:

if (is.null(node$default)) return(FALSE)

This guard is consistent with the original ADR-0010 framing — the companion is a name override, so no default means nothing to override and nothing to bind. The downstream code already prefers comp_value over node$default when both are set:

binding_name <- if (has_companion) {
  nm <- comp_value
  # Boot race: the companion textInput is seeded with node$default by
  # `ptr_builtin_upload_build_ui()`, but `input[[comp_id]]` is NULL until
  # the widget registers. Fall back to `node$default` so the first
  # observer fire can bind immediately rather than waiting an extra
  # invalidation.
  if (!is.character(nm) || length(nm) != 1L || !nzchar(nm)) {
    nm <- node$default
  }
  if (is.character(nm) && length(nm) == 1L && nzchar(nm) &&
      make.names(nm) == nm) nm else NULL
} else if (...) { ... }

i.e., the machinery is keyword-agnostic and already routes through state$eval_env via get(binding_name, envir = state$eval_env, inherits = TRUE). The only thing the null-default guard does is bail before the companion is ever consulted.

FINDING #8 in dev/notes/placeholder-role-coverage2.html exposes two visible consequences of the guard:

  1. A user opening ptr_app("ppUpload() |> ggplot(aes(x = ppVar('col'))) + geom_bar()") and typing "mtcars" into the companion textInput sees no picker population and no plot — the typed name is rejected silently by the guard.
  2. A caller passing spec = list(<companion-id> = "mtcars") for a no-default= source gets the same silent failure (post the FINDING #8 spec-routing fix that landed the companion's value at boot).

Both failure modes are silent — there's no inline error, no log line, no visible signal that the companion was looked at and rejected.

Decision

Promote the source companion's contract from name override for an uploaded frame to data-loading entry point: when the companion holds a valid R name resolvable to a data.frame in state$eval_env's lookup chain, that frame binds — with or without node$default, with or without an upload. The companion is no longer a passive label; it is a typed-in shortcut for loading a frame from the embedder's environment.

This is a semantic promotion of an already-keyword-agnostic mechanism. Every piece of plumbing — companion id, spec_seed write/dispatch, state$eval_env lookup, state$bound_names assign-before-signal, generated-code-panel symbol substitution — is already in place. The change is the removal of one structural guard plus an honest error surface.

1. Lift the null-default guard (two sites)

At R/paintr-server.R:945 (try_bind_source_default(), used by the bare-data-source-layer loop) and R/paintr-server.R:1069 (try_bind_source_default_resolved(), used by resolve_upload_source() at the pipeline-head and panel loops):

# OLD
if (is.null(node$default)) return(FALSE)

# NEW
has_comp_value <- has_companion &&
                  is.character(comp_value) && length(comp_value) == 1L &&
                  nzchar(comp_value)
if (is.null(node$default) && !has_comp_value) return(FALSE)

(The variable names differ slightly between the two functions — the resolved variant already receives has_companion + comp_value; the unresolved one reads input[[comp_id]] itself and must compute the equivalent locally.)

The downstream binding-name derivation is unchanged: it already prefers the companion value, falls back to node$default when the companion is blank, and rejects names that fail make.names() validation.

2. Structured error surface on resolve failure

When binding_name is non-NULL but get(binding_name, envir = state$eval_env, inherits = TRUE) either errors or does not return a data.frame, currently both helpers return(FALSE) silently. The previous semantics ("no default = no bind") justified silent bail. The new semantics ("typed-in entry point") do not — the user typed a name and the binder rejected it; the user is owed a reason.

Both helpers gain one extra branch via set_resolve_error(state, key, msg) (R/paintr-server.R:847-855) before returning FALSE:

  • get() throws → "Object '<name>' not found in environment."
  • get() returns non-data.frame → "Object '<name>' is not a data frame."

The error is recorded keyed by the same resolution-stage id used elsewhere (layer name for bare-data-source, source-node id for pipeline-head + panel). The existing inline error pane (#ptr_error) reads state$resolve_errors() via ptr_register_error() and surfaces it without further plumbing. Successful binds clear the error via the existing set_resolve_error(state, key, NULL) call inside bind_source_value().

3. Document the promoted contract (roxygen only, vignette deferred)

R/paintr-registry.R — ptr_define_placeholder_source()'s @details "Companion pattern" section gets one new paragraph: when companion_id_fn is set, the companion is a data-loading entry point, not just a name override. A valid R-name typed (or seeded via spec=) into the companion binds the named data.frame from the embedder's envir — independent of any default= on the source node. Failures surface via the inline error pane.

The vignette is deferred. vignettes/ppupload.Rmd (or the relevant L2 vignette covering the upload widget) needs a companion-entry-point section, but it can wait until the first user-discoverable feature using a non-ppUpload source with a companion lands — there is no such custom source today, and adding a paragraph now without a reader artifact tied to it would compound the docs-test backlog.

Scope cuts

  • No companion-id read in renderUI bodies. The post-FINDING-#8-fix path lands the spec value into the companion textInput via the onFlushed updateTextInput dispatch (apply_spec_entry()'s source_companion branch). The renderUI bodies already correctly do not read state$spec_seed[[<companion-id>]] because there's nothing to read at first render — the companion is dispatch-driven, not renderUI-seeded. No new spec_seed read site is added; the seed slot becomes a reserved sentinel for the FINDING-#8 dispatch path, not a renderUI input.
  • No fileInput suppression. When the user fills only the companion and never uploads, the fileInput sits unused. This is the same shape as ppUpload(mtcars) today and is not a regression. A future ADR may decide whether to hide the fileInput when an env-loaded frame is bound, but doing so would couple the widget to the bind state and is out of scope here.
  • No widget-validation surface on the companion textInput. The error surface lives on the resolve path (set_resolve_error), not on the companion's own widget binding (no red-bordered shinyFeedback-style annotation). Promoting the inline-error pane to the authoritative surface mirrors how upload-side resolve errors already work.
  • No change to node$default semantics. When default= is set and the companion is blank, the boot-race fallback still binds node$default on the first observer fire. The change is additive — it widens the set of cases that bind, not the cases that don't.
  • Companion-name binding is sticky across an upload-clear. A successful upload writes the resolved frame into state$eval_env under the companion's typed name (bind_source_value at R/paintr-server.R:877: assign(name, df, envir = state$eval_env)). Once that assignment lands, the companion name resolves via env lookup — clearing JUST the fileInput (without also clearing the companion textInput) leaves the slot bound to the prior frame, because the entry-point lookup picks it up via get(name, envir, inherits = TRUE). Pre-ADR-0024 the no-default try_bind_source_default_resolved guard bailed immediately, so clearing the file alone cleared the slot. Post-ADR-0024 the user clears the slot by clearing BOTH inputs. This is consistent with the entry-point contract — the companion is the resolved name — and is captured by the updated tests/testthat/test-rewrite-pipeline-data-source.R assertion. Avoiding it would require either tracking which bindings came from upload (vs entry-point) so file-clear could selectively rm() the upload-side binding, or refusing to populate state$eval_env from uploads — both are larger surgery than this ADR justifies.
  • Self-invalidation risk in set_resolve_error. ADR 0024's structured error surface widens the set of contexts in which set_resolve_error is called from inside a reactive observer (previously the no-upload path only ever cleared, idempotent on an empty list). The helper's pre-existing cur <- state$resolve_errors() read created an undeclared reactive dependency on the resolve-errors slot itself — harmless when the only writes were idempotent clears, but a self-invalidating loop the moment a structured error is written from inside an observer that already reads resolve_errors via set_resolve_error itself. Fix: wrap the read in shiny::isolate() so set_resolve_error stays write-only from the caller's reactive-context perspective. Implemented at R/paintr-server.R:847-862.

Consequences

Positive

  • Eliminates a silent failure mode. The pre-decision behavior — typed companion name with no default= ⇒ no bind, no error — is gone; either it binds or it surfaces a structured error.
  • Companion is now a usable data shortcut. Users can bind in-environment frames without writing a default into the formula source. ptr_app("ppUpload() |> ...", spec = list(<comp> = "mtcars")) works end-to-end; a typed "mtcars" in the live app works end-to-end; both with the same error surface for typos.
  • The promotion is automatic for custom source keywords. Any source defined via ptr_define_placeholder_source(..., companion_id_fn = ...) whose build_ui renders an input at node$companion_id inherits the entry-point semantics — no per-keyword wiring.
  • The change is small and localised. Two conditional changes + two set_resolve_error calls + one roxygen paragraph + four tests. No coordinator, partition, or shared-section logic is touched.

Risks & mitigations

  • Error-message stability. The two new error strings become part of the user-visible surface. Pin them with regression tests so a future refactor doesn't drift them silently. (Test plan §C asserts on the substring "not found" or "not a data frame".)
  • Lookup environment surprises. get(..., inherits = TRUE) walks the parent chain — a user typing "plot" would resolve to graphics::plot (and then fail the "is.data.frame" check). The "not a data frame" branch covers this loudly. Worth noting in the roxygen update so embedders are aware their package's exports become reachable.
  • Compatibility. No existing test today depends on the null-default guard's silent-bail behavior (the bind never happens, so nothing asserts on the silent failure). All existing source-with-default tests are unaffected: the new guard is a strict superset of the old one (lifts only when the companion is non-empty).
  • Future spec_seed reader. If a future change adds a renderUI-side state$spec_seed[[<companion-id>]] reader (e.g. to honor the seed before the onFlushed flush), the FINDING #8 fix's seed-write-but-no-seeded-flag stub already provides the slot to read from. This ADR does not require it, but does not preclude it.
