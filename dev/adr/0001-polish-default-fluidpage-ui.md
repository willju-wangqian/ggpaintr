# Polish the default fluidPage UI with a bundled stylesheet instead of making bslib the default

The default app (`ptr_app()`) renders plain Shiny chrome that looks dated; `ptr_app_bslib()` already offers a nicer Bootstrap-5 layout, and `bslib` is already a (soft) package dependency. We decided to keep `ptr_app()` on `shiny::fluidPage` + `sidebarLayout` and modernize it with a bundled stylesheet (`inst/www/ggpaintr.css`, registered via `addResourcePath()`) plus a small inline JS helper (draggable code mini-window, clipboard copy, toggle), rather than promoting the bslib layout to be the default.

Why: `ptr_app()` is also the path embedded as a Shiny module (`ptr_module_ui()`) inside host apps, so its UI must stay framework-light and not impose a Bootstrap version or theme on the host. Keeping the bslib variant as an opt-in (`ptr_app_bslib()`) preserves that, while CSS-only polish on the default path gets the visual upgrade without widening the dependency/runtime surface. Cost accepted: two UI trees still coexist, and the polish is hand-rolled CSS rather than a theming framework.

Scope of the v1 polish: slim custom header bar (replacing `titlePanel`), neutral page background, sidebar styled as a sticky card, cardinal (`#c1372c`, sampled from the cybird mark) accent exposed only as `:root` CSS variables (no `ptr_options()` knob yet), plot in a card with a `</>` icon that toggles a drag-to-move (not yet resizable) code mini-window containing `verbatimTextOutput("ptr_code")` and a copy button, errors rendered as an alert box. The same asset helper must be wired into the grid (`ptr_app_grid`) and module (`ptr_module_ui`) entry paths.
