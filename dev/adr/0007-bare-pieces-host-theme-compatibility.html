ADR 0007 — Bare pieces vs. host-theme compatibility

Status: deferred — NO decision made · Date: 2026-05-19 · Supersedes: nothing · Voids: nothing · Related: ADR 0001 (polish default fluidPage UI), ADR 0005 (L2/L3 redesign — ptr_<x>/ptr_ui_<x> bare convention, "future ptr_options() knob" anticipated in ggpaintr.css header), ADR 0006 (single server entry; L3 = UI-side bare-piece composition) · Domain language: CONTEXT.md (no change) · Implementation: none — this ADR records an open design question and a recommendation only.

This ADR makes no decision. It captures a 2026-05-19 design discussion, the trade-offs, and a final recommendation, so the question is not re-discovered from scratch later. The maintainer has explicitly chosen to defer. Nothing in the codebase changes on the basis of this document. When the decision is taken, supersede this ADR (or add a "Decision" section and flip the status) — do not silently act on the recommendation below.

Context

While embedding ggpaintr's bare UI pieces (ptr_ui_controls(), ptr_ui_plot(), ptr_ui_toggle_code(), ptr_ui_code(), ptr_ui_inline_error()) into a bslib::page_sidebar() shell — without ptr_ui_assets() and without the .ptr-app wrapper — the components render visibly unstyled: giant native <h> headings, browser-default Copy/× buttons, a borderless card, a broken plot box. The maintainer expected "bare" to mean "a minimal-but-presentable default that coexists with / inherits the active Bootstrap/bslib theme." That expectation does not match the current design.

What "bare" actually means today

ggpaintr ships exactly one fixed, self-contained theme: inst/www/ggpaintr.css, loaded only by ptr_ui_assets() and entirely scoped under .ptr-app. Its header comment states the scoping exists for isolation — "embedding ggpaintr as a Shiny module never restyles the host page." Isolation is symmetric: the host's bslib/bootswatch theme also never reaches into ggpaintr. The single-piece builders emit no assets by design (per ptr_ui_assets() roxygen and inst/llm/topics/level3_layout.md:26 — "the pieces are deliberately bare"). So "bare" = structural HTML with zero intrinsic style, not "neutral self-styling default." There is no built-in tier between "raw fragments" and "ggpaintr's fixed cardinal theme."

The markup splits into two categories — this is the crux

CategoryElementsWhat "remove ggpaintr styling" yields
1. Standard Shiny/Bootstrap components pickerInput (.form-control/.selectize-input/.btn-default), actionButton (.btn .action-button), sidebarPanel/wellPanel (.well), plotOutput, verbatimTextOutput Genuine default Shiny/Bootstrap. ggpaintr.css only lightly overrides these (radius tweak, accent on .action-button, surface on .well). Strip it → real defaults → bslib / other packages style them for free. The maintainer's "fall back to default, stays compatible" reasoning holds here.
2. ggpaintr-invented elements .ptr-card, .ptr-card__head, .ptr-card__title, .ptr-card__body, .ptr-icon-btn, .ptr-copy-btn, .ptr-code-window (draggable floating panel), .ptr-alert, .ptr-app__header Raw unstyled HTML. Bootstrap/Shiny never defined these class names, so there is nothing to fall back to: "fall back" = zero matching rules = bare <div>/native <button>. This is exactly the unstyled screenshot. The "fall back to default" reasoning fails here.

The observed ugliness is dominated by Category 2 (the card chrome + icon buttons + code window built by ptr_ui_toggle_code()/ptr_ui_code()). The plotOutput itself is Category 1 and renders fine by default — the empty box is "no plot drawn yet," not a style break. Conclusion: a naive "disable the --ptr-* variables / disable ggpaintr.css" flag would deliver the desired result for Category 1 and the unstyled mess for Category 2, because Category 2 has no default state to revert to. Theme-neutrality for the whole surface is a markup question, not only a CSS question.

Options considered

Option A — Token aliasing onto Bootstrap variables (CSS-only)

Redefine ggpaintr's design tokens with Bootstrap variables as the value and the current cardinal palette as the fallback, e.g. --ptr-accent: var(--bs-primary, #c1372c), --ptr-bg: var(--bs-body-bg, #fafbfc), --ptr-ink: var(--bs-body-color, #1f2530), --ptr-border: var(--bs-border-color, #e7e9ee). No flag, no API surface, self-adapting: standalone ggpaintr is byte-identical (fallbacks fire); inside a bslib page the chrome auto-picks the bootswatch palette. ~15-line CSS change. This is almost certainly what ggpaintr.css:4-5 means by "a future ptr_options() knob is a one-line override."

Closes: the colour gap (Category 2 chrome coheres with the host palette). Does not close: structure — .ptr-card keeps ggpaintr geometry (radius/shadow/padding); there is no --bs-* a non-.card element auto-inherits shape from. Does not deliver "looks like default Shiny."

Option B — A plain / bare = TRUE rendering mode (markup change)

A mode on the piece builders / wrappers where "plain": (a) skips ptr_ui_assets() and the .ptr-app scope, and (b) substitutes standard Bootstrap components for the bespoke chrome — .ptr-card → wellPanel()/.card, icon buttons → actionButton(), the draggable code window → an inline verbatimTextOutput in a panel, .ptr-alert → Bootstrap .alert. Category 1 then needs nothing; Category 2 becomes real Bootstrap that bslib/other packages style for free. This is the only option that makes ggpaintr's whole output genuinely package-neutral.

Cost: a second rendering path to maintain and test, and "plain" loses identity features — the draggable floating code window has no standard equivalent and degrades to an inline panel. Touches the build-ui functions (ADR-locked UI layer), not just CSS.

Option C — Ship a minimal/reset stylesheet variant (CSS-only, opt-in)

An alternate stylesheet that gives the .ptr-* classes just enough neutral structure (border/padding via var(--bs-*)) to read as plain panels without imposing the cardinal theme. Middle ground: still ggpaintr-shaped, but visually quiet and host-coherent. Less faithful to "default Shiny" than B, cheaper than B, more than A.

Rejected outright

  • "Add a flag that just disables the --ptr-* variables and stays with default." Rejected as stated: false premise. Category 2 elements have no Bootstrap/Shiny default to revert to; disabling the tokens (or the whole stylesheet) deletes the theme without providing a substitute, reproducing the unstyled screenshot. The valid kernel of this idea survives as Option A (alias, don't delete) and Option B (provide a real standard-component substitute).

The tension is structural, not cosmetic: ggpaintr deliberately ships one opinionated, isolated theme, and the word "bare" advertises a theme-agnostic neutrality the pieces do not actually have. A buys host-colour coherence almost for free but cannot make ggpaintr's invented chrome look like native Shiny. B is the only thing that fully satisfies the original expectation, but it is a dual-path commitment that erodes ggpaintr's visual identity and edits the ADR-locked UI layer. C is the compromise nobody specifically asked for. None of these is forced now: the bare pieces remain fully usable today with ptr_ui_assets() + .ptr-app (ggpaintr's theme inside the host shell) — the gap is "host-themed/neutral output," a polish concern, not a correctness defect.

Final recommendation (advisory only — not adopted)

If/when this is taken up: do Option A first as a standalone, low-risk, no-API step — it is the explicitly-anticipated token-knob, closes the colour half of the gap for every embedding context, and is byte-identical standalone. Treat full structural neutrality (Option B) as a separate, later decision, adopted only if a concrete need for true package-neutral output (not just host-coherent colour) materialises; if it is adopted, scope it as an explicit opt-in mode and accept the inline-code-window degradation rather than trying to preserve the draggable widget in plain mode. Do not bundle A and B into one change. Option C only if a consumer specifically needs "neutral but ggpaintr-shaped" — otherwise it is maintenance for an unrequested middle.
