ADR 0028 — Plotly linked-selection helpers (ptr_ggplotly / ptr_plotly_selection)

Status: Locked decision (pending implementation). Routes to /decision-to-plan for the implementable plan; this doc records the decision + its verified baseline, it is not itself the plan.

Date: 2026-06-10 (grill-with-docs session). Companion artifacts: the new Plotly interop — linked selection and Draw sections of CONTEXT.md (same date); the runnable demo dev/scripts/plotly_linked_select.R. Respects: ADR 0006 (single public server entry; custom render = L3 UI-side, extracted from state) — this ADR adds helpers on top of that model and changes nothing in it.

The story, in plain language

The demo script dev/scripts/plotly_linked_select.R renders a ggpaintr formula through plotly::ggplotly(), lets the user box/lasso-select points, shows the selected rows in a table, and highlights them in a second plot. Everything in it is already possible at L3 — but three chunks of boilerplate are non-obvious and identical for every user who wants this: (1) tagging rows with a stable key so plotly events can name them, (2) wrapping ggplotly() with a source id, drag mode, and event registration, and (3) maintaining the selected-rows reactive across select/deselect/redraw. This ADR internalizes exactly those three chunks as a thin, opt-in helper layer — and makes the second plot expressible as a ggpaintr formula itself.

Decisions

  1. Scope: thin helper layer, nothing deeper. New file R/paintr-plotly.R; plotly stays in Suggests (already there) with a call-time guard, mirroring the optional-bslib pattern of ptr_app_bslib() (which guards via requireNamespace() + rlang::abort(), R/paintr-app-bslib.R:62-65; v1 may use rlang::check_installed() for the same effect — rlang is in Imports). Explicitly rejected for v1: a renderer = "plotly" backend on ptr_app()/ptr_ui(), and a state$selection() slot on ptr_state — both bake selection semantics into core before the helper layer proves them.
  2. API: two state-first helpers. 
    ptr_ggplotly(state, ..., source = NULL)            # inside renderPlotly(); returns a plain plotly object
ptr_plotly_selection(state, mode = c("rows", "flag"), source = NULL)   # returns a reactive
    ptr_ggplotly() reads state$runtime()$plot internally (with the req() guard), mints per-draw row keys, forwards ... to ggplotly(), sets dragmode = "select", and registers the plotly_selected/plotly_deselect events. State-first (not plot-first) matches the ADR-0006 canon "custom render extracts from the state" and lets the two helpers share a derived source id. The return value is an ordinary plotly object, so users keep piping plotly::layout() after it. source defaults to a string derived from the instance's namespace id (no collision between instances; override only to coordinate with hand-written plotly widgets).
  3. Selection semantics (CONTEXT.md, LOCKED). A selection refers to the drawn data — state$runtime()$plot$data of the draw that produced the widget — never to "the original object" (after a filter()/mutate() pipeline head or an upload swap there is no stable mapping back). Keys are minted per draw and are meaningless across draws, so the selection resets to empty on every draw. The helper returns rows, not bare indices (indices invite subsetting an object that no longer lines up). Two projections of the one selection: mode = "rows" (the selected slice; zero rows, same columns, when empty) and mode = "flag" (the full drawn data + logical .ptr_selected — the shape a highlight plot needs). Accepted limitation: only identity-stat, point-like layers carry keys; selecting on a geom_smooth/post-stat layer yields nothing.
  4. The second plot is a ggpaintr formula — powered by gate_draw = FALSE, not draw_trigger. The headline composition: 
    ptr_options(gate_draw = FALSE)                        # live mode
state1 <- ptr_server(f1, "p1")                        # plotly-rendered instance
sel    <- ptr_plotly_selection(state1, mode = "flag")
f2     <- rlang::expr(
  sel() |> ggplot(aes(x = ppVar(hp), y = ppVar(qsec), color = .ptr_selected)) + geom_point()
)
state2 <- ptr_server(f2, "p2")                        # redraws per brush, zero wiring
    A reactive pipeline head (sel() — a call; a bare reactive piped into ggplot() is a closure error) already boots, seeds pickers, and draws on the current runtime with zero core changes. Auto-redraw per brush comes from live mode: under gate_draw = FALSE the runtime body runs un-isolated (shiny::observe(run_render()), R/paintr-server.R:1970), so the formula's sel() read becomes an ordinary reactive dependency.
  5. Reserved channels (CONTEXT.md, LOCKED). The interop owns three names: the plotly key aesthetic (a user formula's own aes(key = ...) is overridden with a one-time warning — selection is meaningless under foreign keys, and the L3 escape hatch of calling ggplotly() directly stays open), .ptr_row (key column on the widget copy only; never visible in either projection), and .ptr_selected (the flag column; user-facing by design). The two columns are documented reserved names, silently overwritten on collision — overwrite is what keeps chained selection-fed instances correct (instance N+1's flag must replace the stale flag riding in from instance N's drawn data), and a per-draw warning would spam under live draws.
  6. Bookkeeping is server-side, per-session, keyed by instance id. Source string, selection state, and per-draw snapshot live in a store hung on session$userData — never a package-global environment (the placeholder registry is process-global because it is configuration; per-session reactive state in a global would leak across concurrent users). Two things cannot move into the store: the row key must physically travel to the browser and back inside the widget (a selection event is born client-side and carries only what the traces embed), and the flag must be a real column so it stays row-aligned through the second formula's own pipeline stages.
  7. Docs: roxygen only for v1. Full-loop @examples on both helpers; dev/scripts/plotly_linked_select.R remains the runnable demo. A dedicated ggpaintr-plotly vignette (with fixture mirror + build gate, matching the locked live-three pattern) is deferred, not rejected.
  8. Testing: three tiers. (1) Unit tests on pure cores — each helper is factored into a pure core + thin reactive shell (a design constraint, locked): key minting, key-override warning, both projections, collision overwrite, check_installed guard. (2) One e2e boot fixture (two instances + table; boots, widget renders, empty state, all-FALSE flags). (3) e2e selection round-trip by injecting the event at the Shiny input layer: plotly's R-side event_data() reads input[["plotly_selected-<source>"]] (the .clientValue- prefix is the client→server id, stripped before the R read — verified against the installed plotly source), settable via app$set_inputs(..., allow_no_input_binding_ = TRUE) with a crafted key payload — asserting table fill, flag-mode redraw, deselect-clears, redraw-resets. The exact input-id format is an internal plotly contract: verify against plotly source before committing, pin with a comment. If it proves unstable across plotly versions, downgrade tier 3 to driving the selection reactive directly — do not build mouse-drag or JS-emit simulation.

Verified baseline (probes run 2026-06-09/10 against pkgload::load_all("."), browser via shinytest2)

  1. The demo composition is encoded in dev/scripts/plotly_linked_select.R (structural; file verified). Its end-to-end boot, empty-state table, and key passthrough (32 unique keys, range 1–32, carried into the traces) were probe-verified at authoring time; the pinned regressions for the runtime claims are items 2–4 below, and the helper-level key contract lands with the tier-1 unit tests at implementation.
  2. A reactive pipeline head works today: ptr_server(rx() |> ggplot(aes(x = ppVar(wt), y = ppVar(mpg))) + geom_point(), "p") with rx <- reactiveVal(mtcars) in the host server boots, seeds both pickers (wt/mpg), and draws (regression: tests/testthat/test-runtime-gate.R::"reactive pipeline head boots and seeds pickers", fixture fixtures/vignette-apps/adr28-gate-counter-trigger).
  3. draw_trigger does NOT fire on data change — shrinking rx() with draw_trigger = rx produces no redraw (regression: tests/testthat/test-runtime-gate.R::"draw_trigger ignores non-counter values"). Mechanism: the gate is clicked(x) = is.numeric(x) && length(x) == 1L && x >= 1L applied to the trigger's value (R/paintr-server.R:1860, consumed at :1868) — a click-counter contract, not "any invalidation". The roxygen that claimed otherwise was doc drift; fixed in this change (see Consequences).
  4. gate_draw = FALSE closes the loop with no adapter — ptr_options(gate_draw = FALSE), no draw_trigger: shrinking rx() redraws the plot with no Update click (regression: tests/testthat/test-runtime-gate.R::"gate_draw=FALSE redraws on reactive pipeline-head change", fixture fixtures/vignette-apps/adr28-gate-live-reactive-head; this test doubles as the positive control proving the no-redraw assertion in item 3 has teeth). Mechanism: live mode runs the body un-isolated so every reactive read is a dependency (R/paintr-server.R:1957-1971).

Considered and rejected

  • ptr_draw_trigger(rx) counter adapter — proposed mid-grill to bridge draw_trigger's click-counter contract, then dropped: gate_draw = FALSE already exists, is the supported bypass, and the draw button is an ergonomic default rather than a core invariant (CONTEXT.md Draw term, amended 2026-06-10). No new trigger machinery.
  • Teaching the engine to auto-redraw on formula-data change under gate_draw = TRUE — breaks the batch-on-click promise everywhere (an upload finishing would cascade redraws in existing apps). Live mode already provides the behavior as an explicit opt-in.
  • Plot-first signatures (ptr_ggplotly(state$runtime()$plot)) — forces every user to re-write the pre-draw guard and decouples the two helpers' source ids.
  • (curveNumber, pointNumber) reverse mapping instead of the key channel — requires replicating ggplotly's trace-splitting logic; fragile across grouping/versions. The key aesthetic is plotly's documented contract.
  • Abort on user aes(key=) collision — louder than the failure deserves (formulas may come from templates or an LLM); warn-once + override keeps the render path alive. Respecting the user's key silently was also rejected: the selection helper would return garbage with no hint why.
  • Per-instance flag column names (.ptr_selected_p1) — breaks formula portability across instances; chaining is handled by overwrite instead.
  • Bare indices / persisting selection across draws / a third projection — see CONTEXT.md Avoid lists; all rejected during the grill.

Worked examples — what becomes possible

1. Linked box-select feeding a second formula plot

A plotly-rendered formula whose brush selection drives a second ggpaintr instance — highlight expressed in the formula itself, zero trigger wiring.

ptr_options(gate_draw = FALSE)                        # live mode

# server:
state1 <- ptr_server(f1, "p1")                        # f1: any ggpaintr formula
output$p1_plot <- plotly::renderPlotly(ptr_ggplotly(state1))

sel <- ptr_plotly_selection(state1, mode = "flag")    # drawn data + .ptr_selected
f2  <- rlang::expr(
  sel() |> ggplot(aes(x = ppVar(hp), y = ppVar(qsec), color = .ptr_selected)) +
    geom_point()
)
state2 <- ptr_server(f2, "p2")                        # redraws per brush

2. Selected-rows table off mode = "rows"

The selected slice of the drawn data rendered as an ordinary table; the zero-row-same-columns empty state makes the render unconditional.

sel_rows <- ptr_plotly_selection(state1, mode = "rows")
output$tbl <- renderTable(sel_rows())   # empty selection -> zero-row table, no req() dance

3. Composing extra plotly layout after the helper

ptr_ggplotly() returns a plain plotly object, so the user keeps piping plotly verbs after it.

output$p1_plot <- plotly::renderPlotly(
  ptr_ggplotly(state1, tooltip = "all") |>
    plotly::layout(legend = list(orientation = "h"))
)

Worked examples — what is rejected by design

1. Selecting on a stat-transformed layer

Only identity-stat, point-like layers carry row keys; brushing a smooth/post-stat layer yields nothing — by design, not a bug.

f <- rlang::expr(ggplot(mtcars, aes(ppVar(wt), ppVar(mpg))) + geom_smooth())
# brushing the smooth line -> ptr_plotly_selection() stays at its empty state

2. Persisting a selection across a redraw

Keys are per-draw; any new draw (picker change in live mode, Update click, upload swap) resets the selection to empty — carrying it across draws would silently mis-attribute rows.

# brush plot 1, then change a picker -> new draw -> new keys
sel_rows()   # zero-row slice again; the old selection is NOT carried over

3. Passing the selection reactive itself as draw_trigger

draw_trigger is a click-counter contract (numeric ≥ 1); a reactive carrying a data frame never fires it. Live mode is the supported route.

state2 <- ptr_server(f2, "p2", draw_trigger = sel)   # silently never redraws
# correct: ptr_options(gate_draw = FALSE) and let sel() be a live dependency

4. A bare reactive as the pipeline head

The formula head must call the reactive; piping the reactive object itself sends a closure into ggplot().

f2 <- rlang::expr(sel |> ggplot(aes(...)))    # error: cannot use a function as data
f2 <- rlang::expr(sel() |> ggplot(aes(...)))  # correct — note the parentheses

Consequences

  • Two doc fixes applied with this ADR: the draw_trigger roxygen now states the click-counter contract and points live-mode users to gate_draw = FALSE (R/paintr-server.R, ptr_init_state params); .claude/rules/testing.md's "only re-renders on the Update/Draw click" e2e rule is now qualified by gate_draw = TRUE, with the live-mode exception spelled out.
  • Pre-draw window: before instance 1's first draw, sel() has no drawn data and a selection-fed instance 2 shows its existing inline pre-draw state. Under live mode instance 1 boot-draws on the first flush, so the window is one flush — accepted, documented honestly in the roxygen.
  • In live mode, any picker change on the plotly-rendered instance triggers a draw, which mints new keys and resets the selection — consistent with the per-draw key semantics, worth one sentence in the roxygen.
  • gate_draw is a process-global option read at build time, so live mode applies to every instance in the app; a per-instance override is future work if a mixed app ever needs it.
  • Selection-fed formulas exercise the seeding/binding cluster (reactive head + picker seeding); per .claude/rules/testing.md blast-radius rules, runtime-adjacent changes during implementation widen the e2e set accordingly.
