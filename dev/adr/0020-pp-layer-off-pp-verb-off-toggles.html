ADR 0020 — ppLayerOff / ppVerbOff: source-mode checkbox toggles

Status: Proposed; partially superseded 2026-05-24 by ADR-0021. Drafted 2026-05-24. ADR-0021 supersedes this ADR on two specific points: (1) ppVerbOff is hard-removed in favor of ppVerbSwitch(.data, verb_expr, switch_on, label), which strictly subsumes it; (2) the Out-of-Scope follow-up note proposing a sibling-node ppControl and forbidding stage_id from the comparator-exclusion list is rejected — source-code inspection shows stage_id is a UI-routing key, not execution-shape, so it joins the exclusion list and the verb-side structural-equality invariant (SC7) is restored via wrapping, not siblings. The rest of this ADR (ppLayerOff, default_active / default_stage_enabled reader sites, checkbox_defaults = deprecation, precedence ladder) stands.

Date: 2026-05-24

Relates to: ADR-0009 (the pp* placeholder family + expression mode), ADR-0012 (canonical typed tree; tree is semantic, not syntactic; spec= shape and apply-timing).

Context

ggpaintr already lets the user write naked-ggplot code with empty pp* calls (ppVar(), ppNum(), ppText(), ppExpr()) — they evaluate to NULL outside ptr_app (per the recent 49246f9 fix), and inside ptr_app they become widget bind-points. The same naked-ggplot script becomes either a static plot or an interactive app, depending on which evaluator runs it.

Layer-on/off and stage-on/off toggles are the only first-class app controls that have no source-mode counterpart today:

  • Layer checkbox. Every non-ggplot layer gets an input <layer_name>_checkbox (R/paintr-translate.R:81-83), emitted into ptr_runtime_input_spec() with role "layer_checkbox" (R/paintr-input-spec.R:37-44). The boot value comes from checkbox_defaults = (a schema-side argument keyed by layer name) or, if absent, defaults to TRUE. There is no way to express "this layer should start unchecked" in the formula itself.
  • Stage-enabled checkbox. Every pipeline stage gets stage_id = <layer>_<path>_stage_enabled (R/paintr-disable.R:37-71), role "stage_enabled". The boot value is hardcoded TRUE in both the snapshot (R/paintr-headless.R:107-108) and the stage-head UI (R/paintr-region.R:36-50 via the default_on arg, hardcoded at all call sites). There is no schema-side equivalent of checkbox_defaults for stages, and no formula-side override at all.

The user can already drive both via spec = at session boot (apply_spec_at_boot() in R/paintr-server.R:477 dispatches updateCheckboxInput() for both roles), but that requires knowing the fully-qualified input id (geom_point_1_stage_enabled, namespaced for grid/embed apps) and writing it out by hand. It is a tool for round-trip restore, not for writing a formula.

Verified facts (2026-05-24)

  • The naked-ggplot prototype works as expected: ggplot(mtcars |> ppVerbOff(mutate(mpg = mpg + 100), FALSE) |> ppVerbOff(mutate(wt = wt + 1000), FALSE), aes(x = mpg, y = wt)) + ppLayerOff(geom_point(), FALSE) + ppLayerOff(geom_smooth()) renders one layer (the second ppLayerOff defaults hide = TRUE → NULL), with mpg[1] = 121 and wt[1] = 1002.62.
  • default_active already exists on every ptr_layer (R/paintr-nodes.R:15-20, init'd TRUE at R/paintr-translate.R:477,489) and is read at exactly one site: R/paintr-build-ui.R:144-145 → resolve_layer_default(layer_name, checkbox_defaults, node$default_active) at R/paintr-build-ui.R:531-539. The boot snapshot in R/paintr-headless.R:105-108 (layer_checkbox branch) does not currently read node$default_active — it instead consumes checkbox_defaults via a call to ptr_resolve_checkbox_defaults() at line 98 (which also transitively consults the global option via ptr_default_layer_state()). Once that resolver call is removed, the snapshot must learn to read node$default_active directly — drift hazard if only one side is taught the new field.
  • default_stage_enabled does not exist anywhere in the codebase. Two sites currently hardcode TRUE for stage-checkbox defaults: the snapshot branch above and controllable_region()'s default_on arg at the per-layer Data-panel stage call (R/paintr-build-ui.R:230-232, inside build_pipeline_stage_ui()) and the shared-stage variant (R/paintr-build-ui.R:486-488, inside wrap_shared_widgets_with_stage_blocks()).
  • Round-trip via state$spec emits on diff from the snapshot; once the snapshot honors the new defaults, spec = already works for both roles via the existing apply_spec_entry() dispatch (R/paintr-server.R:643-665).
  • No other tree consumer (paintr-prune.R, paintr-substitute.R, paintr-disable.R, paintr-input-spec.R, paintr-server.R) reads either default_active or the proposed default_stage_enabled — they read active (runtime, post-snapshot), active_input_id, and stage_id only. Adding the new field is read-side greenfield outside the two snapshot/UI sites enumerated above.
  • Layer-checkbox boot value already has three bands today (not two): per-call checkbox_defaults = list(<layer_name> = TRUE|FALSE) at R/paintr-checkbox-defaults.R:19-105 (note: both values are valid — the validator at R/paintr-checkbox-defaults.R:135-157 only requires logical, length>0, not NA; R/paintr-build-ui.R:535-537 uses isTRUE() on the entry); then the global option checkbox_default_all_other_layer (default TRUE) consulted via ptr_default_layer_state() at R/paintr-options.R:136-137 and used as fallback inside ptr_resolve_checkbox_defaults() at R/paintr-checkbox-defaults.R:22; then the implicit TRUE. Stage-checkbox boot value has no analogous global option — it is hardcoded TRUE in both reader sites.
  • The structural-equality comparator ptr_tree_structural_equal() at R/paintr-nodes.R:122-149 already excludes op and expr from its name-set walk (setdiff(names(a), c("op", "expr"))) — those fields capture surface notation (|> vs %>% vs nested-call; the cached original expr), not semantics. The new default_active / default_stage_enabled fields belong in the same exclusion category.
  • tests/testthat/test-adr12-consumer-uniformity.R:68 re-implements ptr_tree_structural_equal() inline (not a call to the real one). The inline helper is class-dispatched, not name-set-walking: for each node class it hand-picks the fields to compare (ptr_layer: name / data_arg / children; ptr_call: fun / args; etc.). It therefore already ignores default_active / default_stage_enabled by construction — no mirror change is needed. The right move is to switch the file to ggpaintr:::ptr_tree_structural_equal, which both gains tighter coverage (it compares the full slot set rather than a hand-picked subset) and clears a few ::: uses, aligning with ADR-0019's direction.

Problem statement

The naked-ggplot story stops short of toggles. Users who want a formula to express "this layer starts off" or "this pipeline stage starts off" today have to leave the formula and either set checkbox_defaults = (layer only, schema-side, keyed by name) or hand-write spec = entries (both roles, but verbose and bound to namespaced ids). The formula is no longer a single source of truth for what the user wants the app to look like at boot.

Decision (direction, not interface)

Add two new keywords to the pp* family that are structural unwrappers, not value placeholders:

ppLayerOff(layer_expr, hide = TRUE)            # used in layer position (top-level +)
ppVerbOff(.data, verb_expr, hide = TRUE)       # used as a data-pipeline stage

Out of source-mode (no ggpaintr evaluator on the stack) both behave exactly as their naked-R bodies imply: ppLayerOff returns NULL when hide, the layer expression otherwise; ppVerbOff re-routes the prior .data through the inner verb call when not hidden, returns .data unchanged when hidden.

Inside ggpaintr, both keywords are recognised by placeholder_keyword() (R/paintr-translate.R:594-605) but translated with special unwrap rules — they do not become ptr_ph_* nodes:

  • ppLayerOff(layer_expr, hide = L) in layer position → translate layer_expr as a normal layer (so active_input_id = "<name>_checkbox" is assigned by the existing path) and stamp default_active <- !isTRUE(L) on the resulting ptr_layer. In any other position → translate-time error.
  • ppVerbOff(.data, verb_expr, hide = L) as a pipeline stage → translate the inner verb_expr as the pipeline stage (so assign_stage_ids() tags it the usual <layer>_<path>_stage_enabled), stamp default_stage_enabled <- !isTRUE(L) on the resulting ptr_call. In any other position → translate-time error.
  • hide must be a static logical literal at translate time. Anything else is a translate-time error — it cannot be a default for a widget.

Two reader sites read the new defaults directly off the node (no checkbox_defaults consultation — that arg is deprecated per the next section):

  • ptr_default_snapshot() layer_checkbox branch (R/paintr-headless.R:105-108): use node$default_active %||% TRUE, looking the carrier ptr_layer up by layer_name. (The current call to ptr_resolve_checkbox_defaults() at line 98 is removed.)
  • ptr_default_snapshot() stage_enabled branch (R/paintr-headless.R:109-110): look up the stage carrier node by stage_id and use node$default_stage_enabled %||% TRUE.

Two UI sites:

  • Layer-checkbox initial value at R/paintr-build-ui.R:144-145 currently goes through resolve_layer_default(layer_name, checkbox_defaults, node$default_active). With checkbox_defaults = deprecated, this simplifies to isTRUE(node$default_active %||% TRUE); resolve_layer_default() at R/paintr-build-ui.R:531-539 can be removed (or inlined).
  • Stage-checkbox emission threads the carrier ptr_call's default_stage_enabled into controllable_region(..., default_on = ...) at two sites. Each site has a single-line plumbing change in the producer + a single-line addition at the call site:
    • Per-layer Data-panel: entry builder find_layer_placeholders_with_stage() at R/paintr-build-ui.R:303-355 already has the carrier ptr_call in scope as s (line 333) where it extracts verb; extend the entry shape from list(ph, stage_id, verb) to list(ph, stage_id, verb, default_stage_enabled). Consumer build_pipeline_stage_ui() at R/paintr-build-ui.R:230-232 then passes default_on = entries[[i]]$default_stage_enabled %||% TRUE to controllable_region() (all entries in one stage block carry the same value; reading the head entry is sufficient).
    • Shared-stage variant: producer collect_orphan_shared_stages() at R/paintr-build-ui.R:405-420 already has the carrier ptr_call in scope as n (line 408); extend the emitted record (lines 413-417) from list(stage_id, verb, shared_keys) to list(stage_id, verb, shared_keys, default_stage_enabled = n$default_stage_enabled). Consumer wrap_shared_widgets_with_stage_blocks() at R/paintr-build-ui.R:486-488 then passes default_on = st$default_stage_enabled %||% TRUE to controllable_region().

Nothing changes on the spec = side. apply_spec_at_boot() / apply_spec_entry() (R/paintr-server.R:643-665) already dispatches updateCheckboxInput for both layer_checkbox and stage_enabled roles. A user-supplied spec = entry overrides the hide =-derived default at flush time exactly as it does today for placeholder widgets.

Precedence (boot value of the checkbox)

Symmetric 3-band ladder for both roles, highest precedence first:

  1. spec = entry naming the input id (ad-hoc one-shot at boot — for round-trip restore or external boot configuration).
  2. Formula-side ppLayerOff(..., hide=) → default_active on the carrier ptr_layer; ppVerbOff(..., hide=) → default_stage_enabled on the carrier ptr_call. The canonical formula-side mechanism.
  3. Implicit TRUE.

This intentionally deprecates the per-call checkbox_defaults = argument and the global checkbox_default_all_other_layer option — see the next section.

Deprecation: checkbox_defaults = and the global option

Before this ADR, layer-checkbox boot state had three entry points: spec = (ad-hoc), per-call checkbox_defaults = (schema-side config), and the global option checkbox_default_all_other_layer (unscoped fallback). Adding ppLayerOff introduces a fourth. Four entry points for one bit of state is too many, and three of them are side-channels disconnected from the formula:

  • spec = is justified as the ad-hoc boot-state restore mechanism — it has a distinct purpose (round-trip, external configuration) and a distinct shape (id-keyed, namespaced).
  • ppLayerOff / ppVerbOff live inside the formula. They keep the naked-ggplot evaluation and the ptr_app evaluation consistent on what the boot state should be — the formula is the source of truth. This matches ADR-0009's "placeholder boot values live inside the call" principle (e.g. ppText(initial = "hi")), extended to toggles.
  • checkbox_defaults = and checkbox_default_all_other_layer are side-channels: they affect ptr_app only, and can silently drift from what the formula says. ppLayerOff strictly subsumes them — every checkbox_defaults shape has a mechanical per-layer ppLayerOff equivalent (direct entries → wrap that layer; group expansion → wrap each instance; logical-vector → wrap each instance with the matching hide=). Nothing is lost.

Therefore: deprecate both. Migration:

  • checkbox_defaults = list(geom_point = FALSE) → ggplot(...) + ppLayerOff(geom_point(), TRUE) + ...
  • checkbox_defaults = list(geom_point = c(TRUE, FALSE)) for two instances → geom_point() + ppLayerOff(geom_point(), TRUE) (or ppLayerOff(geom_point(), FALSE) + ppLayerOff(geom_point(), TRUE) if the user wants the explicit on-marker for symmetry).
  • ptr_options(checkbox_default_all_other_layer = FALSE) (opt-in mode) → wrap every default-off layer in ppLayerOff(..., TRUE). Verbose for many-layer apps, but explicit at the call site rather than hidden in global state.

Cost acknowledged: opt-in-mode users (many alternative layers, only a few on by default) trade one global setting for one wrapper per layer. This is real friction; the trade-off is that "what the app shows at boot" is now visible in the formula instead of split across global state and the formula.

Public surface affected (verified via grep + signature reads): the global option ptr_options(checkbox_default_all_other_layer = ...) at R/paintr-options.R:10-11, 34-43, 62, 135-137. The checkbox_defaults = argument is a named formal on: ptr_app (R/paintr-app.R:139), ptr_app_components (two variants at R/paintr-app.R:161, 299), ptr_app_grid (R/paintr-app.R:392), ptr_app_grid_components (R/paintr-app.R:749), ptr_app_bslib (R/paintr-app-bslib.R:61), ptr_init_state (R/paintr-server.R:89), ptr_run_formula (R/paintr-headless.R:121), ptr_default_snapshot (R/paintr-headless.R:86), and ptr_resolve_checkbox_defaults (R/paintr-checkbox-defaults.R:20). ptr_server forwards it via ... into ptr_init_state, not as a formal. The entire R/paintr-checkbox-defaults.R file (ptr_resolve_checkbox_defaults, two validators, plus ptr_strip_dedup_suffix — verified to have only one caller, also inside that file, so it dies with the file) is removed. Test files: tests/testthat/test-options.R (the half that covers checkbox_default_all_other_layer needs removal; the verbose half stays) and the 287-line tests/testthat/test-checkbox-defaults.R (the dedicated test for the file being removed) — needs to be deleted or rewritten against the formula-side ppLayerOff equivalent.

Removal cadence: straight remove in the same change. ggpaintr has no external users to deprecate-warn for — there is no production deployment surface to protect. Mechanical migration of any internal call sites (vignettes, tests, manual fixtures) happens in the same plan.

ADR-0012 invariant (tree is semantic, not syntactic)

The "syntactic equivalents produce the same canonical tree" rule holds only when hide = FALSE, i.e. when the user explicitly opted into the on-state:

  • mtcars |> ppVerbOff(mutate(x = 1), FALSE) ≡ mtcars |> mutate(x = 1) → trees identical (default_stage_enabled = TRUE on both, the on-by-default value).
  • ppLayerOff(geom_point(), FALSE) ≡ geom_point() → trees identical.

With hide = TRUE (the function's default), the carrier node still differs from the bare equivalent on the default_active / default_stage_enabled metadata — that's the requested signal — but the structural-equality comparator deliberately ignores it (see below). Functionally: trees compare equal for both values of hide, while downstream readers (snapshot, UI emitter) still see the metadata and seed the right boot value.

Structural-equality comparator

ptr_tree_structural_equal() at R/paintr-nodes.R:122-149 already excludes op and expr from the name set it walks — those fields capture how the user wrote it, not what the plot computes. Extend the exclusion to default_active and default_stage_enabled:

names_a <- setdiff(names(a), c("op", "expr", "default_active", "default_stage_enabled"))
names_b <- setdiff(names(b), c("op", "expr", "default_active", "default_stage_enabled"))

Justification: both fields encode a request about the UI widget's boot value, not anything the plot evaluation produces. Both are bool; both are user-flippable at runtime via the very checkbox they seed. The tree's shape and the plot it eventually renders are identical regardless of either field's value — exactly the same category as op (the pipe variant) and expr (the cached original R expression).

This decouples the comparator from the runtime: the comparator answers "do these two trees compute the same plot?" while the snapshot / UI emitter / spec-roundtrip layers read the metadata fields directly off the node when they need it. Tests that want to assert "translate stamped the right default" read node$default_active / node$default_stage_enabled directly rather than through the comparator — which is strictly better than overloading the comparator with two separate responsibilities.

Out of scope for this ADR

  • The exact spelling of error messages from translate-time validation (literal-only hide; wrong-position keyword).
  • The order in which ppLayerOff / ppVerbOff are introduced into the registry init pipeline (covered by ADR-0014).
  • Whether the same machinery should also let a user hide placeholder widgets (e.g. ppText(initial = "hi", visible = FALSE)). That is a separate UI-shape question; this ADR addresses only the binary on/off toggle that already exists.
  • Documentation of ppVerbOff's data-arg-position assumption is in-scope for the implementation (man page + use-cases vignette) but the exact wording is a writing decision, not an architecture decision.
  • Follow-up: ppControl placeholder for verb-stage structural equality (added 2026-05-24). During execution of plans 01/03 the team discovered that ppVerbOff(call, TRUE) and the bare call cannot be made structurally-equal via the comparator's exclusion list without giving up runtime semantics: Plan 01's stamp_default_stage_enabled_ids() stamps a stage_id on every wrapped ppVerbOff(hide=TRUE) stage (the disable mechanism needs a stable id to reference) but the bare-form stage never gets one (R/paintr-disable.R::assign_stage_ids gates on placeholder presence). stage_id is genuinely runtime-semantic — disable_walk reads it to drop stages — so it does not qualify for the comparator exclusion list. Plan 03 SC7 (verb-stage equality) was therefore dropped during execution; the layer-side SC6 still proves the principle. A future ADR will introduce a ppControl placeholder that expresses the verb-stage toggle through a shape that naturally round-trips through the comparator (e.g. by emitting a control-only sibling node rather than wrapping the verb call itself), restoring the verb-side structural-equality invariant without weakening either the disable mechanism or the comparator.

Consequences

What this buys

  • A naked-ggplot formula becomes a faithful expression of the app's boot state for every first-class control — not just placeholders. The "same formula in or out of ggpaintr" story extends to layer/stage toggles.
  • Zero new id schemes. All the existing layer_checkbox / stage_enabled input ids, widget rendering, runtime visitors (disable_walk, resolve_layer_default), and spec = round-trip continue to do exactly what they already do — the change is "teach two sites to honor a node-level default field". The dormant default_active field on ptr_layer finally gets a producer.
  • Composable with the existing layer-checkbox story: checkbox_defaults = (schema, by layer name) and ppLayerOff(hide=) (formula, per-instance) cover different ergonomic needs and stack via the documented precedence above.

What this does NOT buy

  • No new runtime power. Anything ppLayerOff / ppVerbOff can express is also expressible via spec = with the right id, or (for layer only) via checkbox_defaults =. This is ergonomic-symmetry work, not new behaviour.
  • No solution to "I want the checkbox to disappear entirely from the UI" — the toggle is still rendered; only its initial value is set.
  • No coverage for placeholder widgets' boot defaults via a new keyword — those are already covered by passing the literal argument inside the placeholder call itself (ppText(initial = "hi"), etc.).

Risks

  • Drift between the snapshot and the UI's initial checkbox value if only one of the two reader sites is updated. The "verified facts" section pinned both sites; the implementation plan must touch both in the same change, with a regression test asserting that ppLayerOff(..., TRUE) produces a layer whose UI checkbox value, snapshot entry, and post-flush state$layer_checkbox reactive all agree on FALSE.
  • Position discipline. A ppLayerOff() buried inside aes() or a ppVerbOff() outside a pipe chain has no useful semantics. Translate-time errors close that gap; without them, the user gets a silent no-op (the naked-R semantics apply at runtime, no toggle is registered, no widget appears).
  • ppVerbOff data-arg position. The unwrap assumes the verb call takes its data argument in position 1 (true for dplyr/tidyr verbs, magrittr/|> conventions). This must be documented in the man page and use-cases vignette; a verb that takes data elsewhere will not round-trip correctly through ppVerbOff.
  • Test-local re-implementation of the comparator. tests/testthat/test-adr12-consumer-uniformity.R:68 defines its own inline ptr_tree_structural_equal() (it does not call the real one). The inline helper is class-dispatched — it never reads default_active / default_stage_enabled, so the new fields don't break it. But it's also a weaker comparator than the real one (hand-picked fields per class vs full slot set). Switch the file to ggpaintr:::ptr_tree_structural_equal for stronger coverage and to clear a few ::: uses (aligns with ADR-0019).
