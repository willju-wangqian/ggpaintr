Read this whole document before attempting a fix. Bug 3b in particular has consumed multiple investigation cycles. Both naive fixes (one-line bare-name fallback, one-line namespace lookup) and the more-disciplined explicit-dispatch fix have been tried; the explicit-dispatch fix cleared bug 3b but regressed 49 tests. 1. Bug 3a — ppVar inside a pipe-stage call (e.g. filter(ppVar > ppNum)) has no column options

OPEN   needs live upload to repro

1.1 User report (verbatim, 2026-05-21)

library(dplyr)
ptr_app(
  ppUpload |> filter(ppVar > ppNum) |>
    ggplot(aes(ppVar, ppVar)) +
    geom_point()
)

# when filter is included (with the checkbox), the picker for ppVar in the
# filter doesn't work. There are no options for the ppVar picker inside the
# filter function.

User clarification: "the upstream data frame is already uploaded via ppUpload for filter. It seems that filter stage doesn't see it for some reason". So this is reproduced after the user has uploaded a file (e.g. penguins.csv) — ppUpload's resolve_expr has run, the data is available, the ppVar pickers in the downstream aes() populate correctly, but the ppVar nested inside filter(...) remains an empty selectInput.

1.2 The data-flow rule (user-stated, accepted)

Per the user, the semantic rule is: "for a data consumer, their current calling function should also be included as part of the upstream. Or its upstream starts from their calling function."

Concretely for the filter case:

  • The ppVar inside filter(ppVar > ppNum) is a child of the filter call.
  • filter's upstream is whatever feeds it in the pipeline — here, the resolved ppUpload.
  • Therefore the ppVar's cols = names of the resolved ppUpload data (the pre-filter columns).

This is the right semantic. The cols presented to the in-filter ppVar should be the cols of the data flowing INTO filter, not the cols of the data flowing OUT of the pipeline.

1.3 What the design already says

Source path inspected: R/paintr-server.R:866-921 runtime_upstream_data(state, snapshot).

The function walks every ptr_ph_data_consumer node, looks at three slots:

SlotUsed forNotes
c$sharedCross-layer shared bucketIf set and a host resolution exists in state$shared_resolutions, validate against that. Falls through otherwise.
c$layer_namePer-layer resolved data fast-pathIf state$resolved_data[[layer_name]]() is non-null, use those cols.
c$upstreamPer-position fallbackIf neither of the above hits, resolve c$upstream via ptr_resolve_upstream(c$upstream, snapshot, ...).

1.4 Empirical probe (no upload — static reproduction)

The translate output for the user's formula has three consumer nodes:

# Probe (post-translate, pre-substitute):
tree <- ptr_translate("ppUpload |> filter(ppVar > ppNum) |> ggplot(aes(ppVar, ppVar)) + geom_point()")
consumers <- ggpaintr:::find_nodes(tree, ggpaintr:::is_ptr_ph_data_consumer)

# consumer 1: id = "ggplot_2_1_1_ppVar_NA"    <- THE ONE INSIDE filter
#   layer_name = "ggplot"
#   upstream class = ptr_ph_data_source   (the ppUpload source alone)
#
# consumer 2: id = "ggplot_1_1_ppVar_NA"      <- ppVar in aes() x
#   layer_name = "ggplot"
#   upstream class = ptr_pipeline         (full ppUpload |> filter)
#
# consumer 3: id = "ggplot_1_2_ppVar_NA"      <- ppVar in aes() y
#   layer_name = "ggplot"
#   upstream class = ptr_pipeline         (full ppUpload |> filter)

Finding: The parser correctly sets c$upstream for the filter-internal consumer to just the ppUpload data source, not the full pipeline. The design does understand that the filter-internal consumer's upstream is the pre-filter data.

But all three consumers carry layer_name = "ggplot". So they all share the same state$resolved_data[["ggplot"]] fast-path. That fast-path's value is the post-pipeline resolved data (after filter applies), which is wrong for consumer 1. The per-layer fast-path fires before the per-position fallback (line 901-906 of runtime_upstream_data), so consumer 1's correct c$upstream resolution is never reached.

1.5 Hypothesis for what the user sees

When the user uploads:

  1. state$resolved_data[["ggplot"]] is populated with the post-filter data via an upload observer.
  2. If filter has no args yet (because ppVar/ppNum in filter are unset), filter is a no-op and the post-filter data equals the pre-filter data — consumers 2 and 3 populate fine, consumer 1 also populates from the same data.
  3. But once consumer 1's ppVar > ppNum arg becomes well-defined (the checkbox enables filter, ppVar inside filter has a value, ppNum has a value), filter actually filters, and state$resolved_data[["ggplot"]] shrinks to the filtered subset.
  4. Consumer 1 then sees its own filtered output's cols, not the pre-filter cols. If the filtered subset is empty (because filter args are still under construction), state$resolved_data[["ggplot"]] is empty → consumer 1 picker has no options.

This is consistent with the user's symptom: "no options for ppVar inside filter". The picker is bound to the post-filter resolved data, which is empty / wrong.

1.6 What was NOT done (out of scope tonight)

  • Live reproduction with an actual upload. Requires a shinytest2 fixture that uploads a real file then asserts consumer-1's picker options. The existing adr9-* e2e fixtures use built-in datasets only — none drive a file upload widget. Writing the upload-driving fixture is a follow-up task in itself (~30 min plus chromote setup).
  • Confirming the per-layer fast-path is the trip-wire. The 1.5 hypothesis is based on reading runtime_upstream_data's code path; not yet verified by tracing the actual state$resolved_data[["ggplot"]] contents at the moment of the user's bug.

1.7 Proposed fix shape (subject to live-upload verification)

In R/paintr-server.R's runtime_upstream_data, the per-layer fast-path should only fire for consumers whose c$upstream is the FULL layer pipeline (the aes-side consumers), not for consumers whose c$upstream is a narrower subtree (the pipe-stage-internal consumers).

Mechanical shape:

for (c in consumers) {
  if (is.null(c$id)) next
  if (!is.null(c$shared)) { ... existing shared path ... }

  # NEW: skip the per-layer fast-path when the consumer's c$upstream
  # is a strict subset of the layer's pipeline (i.e. it lives INSIDE a
  # pipe stage). The c$upstream subtree set by the parser already
  # captures the correct point of resolution for these consumers; the
  # per-layer resolved_data fast-path would over-shrink them to the
  # post-pipeline subset.
  layer_name <- c$layer_name
  is_inner   <- !inherits(c$upstream, "ptr_pipeline") &&
                !inherits(c$upstream, "ptr_layer")     # i.e. just a source
  if (!is_inner &&
      !is.null(layer_name) &&
      !is.null(state$resolved_data[[layer_name]])) {
    df <- state$resolved_data[[layer_name]]()
    if (!is.null(df)) {
      out[[c$id]] <- list(cols = names(df), data = df)
      next
    }
  }

  # ... existing per-position ptr_resolve_upstream fallback ...
}

The exact is_inner predicate needs verification — possibilities:

  • Use the class of c$upstream as above.
  • OR add a new slot to the consumer node at parse time: c$inside_pipe_stage = TRUE/FALSE, set by the parser. Cleaner contract.
  • OR compute the predicate from c$id shape (the in-filter consumer has id ggplot_2_1_1_ppVar_NA — a four-segment id while the aes consumers have three-segment ids). The extra segment is the call-depth marker; the parser already encodes this. A four-segment id means "nested inside a pipe-stage call".

Whichever predicate is chosen, the c$upstream per-position fallback already produces the right result — the fix is purely about not stealing the consumer with the per-layer fast-path.

2. Bug 3b — empty filter() not pruned out of pipeline (load_all S3-dispatch trap)

OPEN   two fix attempts reverted   diagnosis confirmed

2.1 User report (verbatim, 2026-05-21)

library(dplyr)
ptr_app(
  ppUpload |> filter(ppVar > ppNum) |>
    ggplot(aes(ppVar, ppVar)) +
    geom_point()
)

# when filter is included with no arguments, filter should be dropped from
# the expression tree. Why is it preserved in the code panel? I understand
# that filter(penguins) is what R sees. But we are working with the AST, we
# should be able to set the node for filter to NULL and get rid of it...
#
# Observed code-panel output:
#   ggplot(data = filter(penguins), aes(bill_length_mm, bill_depth_mm)) +
#     geom_point()

User's expectation: when filter(ppVar > ppNum)'s args all prune to missing, the entire filter stage should drop out of the pipeline.

2.2 Reproduction at the R level

NOT_CRAN=true Rscript -e '
suppressMessages(devtools::load_all("."))
library(dplyr)
tree   <- ptr_translate("ppUpload |> filter(ppVar > ppNum) |> ggplot(aes(ppVar, ppVar)) + geom_point()")
ctx    <- list(snapshot = list(), upstream_cols = list())
sub    <- ggpaintr:::substitute_walk(tree, ctx)
pruned <- ptr_prune(sub)
cat(ptr_render(pruned), "\n")
# OBSERVED:
#   filter() |>
#     ggplot() +
#     geom_point()
# EXPECTED:
#   ggplot() +
#     geom_point()    # or with ppUpload elided too

The empty filter() survives. With the qualified form dplyr::filter(...) (instead of the bare filter after library(dplyr)), the prune works correctly. So the bug is specific to bare-name dispatch under load_all.

2.3 First hypothesis (false-alarm): name-resolution gap

Initial guess: the remove-set contains "dplyr::filter" but name_matches_remove_set(bare="filter", qualified="filter", remove_set) never matches because qualified resolution didn't happen for the bare symbol.

Result: REJECTED. qualified_call_name(quote(filter)) in R/paintr-prune.R:179-199 already resolves bare symbols via get(nm, mode = "function"), follows environment(fn), and returns "dplyr::filter" when dplyr is attached. Probed:

cat("ggpaintr:::qualified_call_name(quote(filter)) ->",
    ggpaintr:::qualified_call_name(quote(filter)), "\n")
# dplyr::filter

So the name resolves correctly to the qualified form, which IS in default_drop_when_empty(). The user's correction on this point ("when ggpaintr reads filter, it should firstly check which package it is coming from, then compare with the list") is correct, and the design already does it. Bug 3b is downstream of name resolution.

2.4 Second hypothesis (confirmed root cause): UseMethod from package-env shadow

Probe sequence:

Step 1. Direct call to prune_walk on the filter stage alone does return ptr_missing: 
filter_stage <- sub$layers[[1]]$data_arg$stages[[2]]
direct <- ggpaintr:::prune_walk(filter_stage, remove_set = rs,
                                is_standalone = ggpaintr:::ptr_default_is_standalone)
class(direct)[1]   # => "ptr_missing"
Step 2. prune_walk(sub, ...) via namespace also produces the correct tree (filter stage dropped, layer1$data_arg becomes NULL): 
root_pruned <- ggpaintr:::prune_walk(sub, remove_set = rs, is_standalone = ggpaintr:::ptr_default_is_standalone)
class(root_pruned$layers[[1]]$data_arg)[1] %||% "NULL"   # => "NULL"
Step 3. A faithful replica of ptr_prune's body (same args, same internals, defined in globalenv()) ALSO produces the correct result: 
ptr_prune_replica <- function(node) {
  remove_set <- unique(c(ggpaintr:::default_drop_when_empty(),
                          ggpaintr:::validate_safe_to_remove(NULL)))
  is_standalone <- ggpaintr:::ptr_default_is_standalone
  ggpaintr:::check_prune_depth(node)
  ggpaintr:::prune_walk(node, remove_set = remove_set, is_standalone = is_standalone)
}
r1 <- ptr_prune_replica(sub)
class(r1$layers[[1]]$data_arg)[1] %||% "NULL"   # => "NULL"
Step 4. The REAL ptr_prune(sub) (defined in the package, accessed under devtools::load_all) returns the BROKEN tree: 
r2 <- ptr_prune(sub)
class(r2$layers[[1]]$data_arg)[1] %||% "NULL"   # => "ptr_call"  <- BUG

Steps 3 and 4 do exactly the same thing. Same internal calls. Same args. Same remove_set (confirmed via identical(), 369 entries). Different result.

2.5 The shadow trap

Confirmed via direct env inspection:

pw_pkg <- get("prune_walk", envir = as.environment("package:ggpaintr"))
pw_ns  <- get("prune_walk", envir = asNamespace("ggpaintr"))
environmentName(environment(pw_pkg))   # "package:ggpaintr"   <- SHADOW
environmentName(environment(pw_ns))    # "ggpaintr"           <- namespace
identical(pw_pkg, pw_ns)                # FALSE -- distinct closures

Calling the package-env shadow directly errors:

pw_pkg(sub, remove_set = rs, is_standalone = ggpaintr:::ptr_default_is_standalone)
# Error in UseMethod("prune_walk") :
#   no applicable method for 'prune_walk' applied to an object of class "c('ptr_root', 'ptr_node')"

So devtools::load_all creates a second copy of prune_walk in package:ggpaintr, whose closure env is the package env (not the namespace), and UseMethod from inside that shadow generic cannot find the methods registered in the namespace's S3 method table.

This is the same shape as the trap PLAN-01 hit with ptr_constant_fold_env — a generic / accessor lives in package:ggpaintr and its lookups don't reach the namespace's bindings. PLAN-01 fixed that case by routing through get("...", envir = asNamespace("ggpaintr")).

2.6 Fix attempt #1 — namespace lookup of the generic

Hypothesis: replace the bare prune_walk(...) call in ptr_prune with the namespace's version, so the generic's environment is the namespace, and UseMethod finds methods.

ptr_prune <- function(node, ...) {
  ...
  pw <- get("prune_walk", envir = asNamespace("ggpaintr"))
  pw(node, remove_set = remove_set, is_standalone = is_standalone)
}

Result: did NOT fix the bug. Even with pw being the namespace's prune_walk (verified: environmentName(environment(pw)) == "ggpaintr"), the output was still the broken tree (layer1$data_arg = ptr_call).

Surprise: a stand-alone ptr_prune2 defined in globalenv() using the same get(asNamespace) + pw(node, ...) shape DOES produce the correct tree:

ptr_prune2 <- function(node) {
  ggpaintr:::check_prune_depth(node)
  pw <- get("prune_walk", envir = asNamespace("ggpaintr"))
  rs <- unique(ggpaintr:::default_drop_when_empty())
  pw(node, remove_set = rs, is_standalone = ggpaintr:::ptr_default_is_standalone)
}
class(ptr_prune2(sub)$layers[[1]]$data_arg)[1] %||% "NULL"   # => "NULL"  CORRECT

So the bug is not just about the closure env of the generic being called. It depends on the closure env of the CALLER (ptr_prune itself). When ptr_prune lives in the package (under load_all, in package:ggpaintr), something in UseMethod's frame-walking sees the package-env context and fails to locate methods. The same call shape works when ptr_prune is in globalenv.

This is the deep trap. UseMethod's method lookup is documented as: (1) the generic's environment's .__S3MethodsTable__.; (2) walk parent.frames; (3) global env. In step (2), when the caller's enclosing env is package:ggpaintr (the shadow), the walk finds the package env's .__S3MethodsTable__. first — which under load_all is either empty or contains a stale subset of methods — and stops there.

2.7 Fix attempt #2 — explicit method dispatch

Hypothesis: skip UseMethod entirely. Every callsite I could grep for (R/paintr-safe.R:32, R/paintr-resolve.R:30, and ~15 test usages) passes the output of substitute_walk, which preserves ptr_root class (verified: inherits(substitute_walk(tree, ctx), "ptr_root") == TRUE for a translated formula). So in practice the entry node is always ptr_root; we can dispatch directly.

ptr_prune <- function(node, ...) {
  ...
  if (!inherits(node, "ptr_root")) {
    rlang::abort("`node` must be a `ptr_root` tree.")
  }
  m <- get("prune_walk.ptr_root", envir = asNamespace("ggpaintr"))
  m(node, remove_set = remove_set, is_standalone = is_standalone)
}

Result: FIXED bug 3b but REGRESSED a large number of unrelated tests. Final gate after this fix:

FAIL: 52   ERROR: 49   SKIP: 0   PASS: 1777   # (was 1982 baseline; -205 passing)

So 101 tests transitioned to failed-or-errored, and 205 fewer tests passed (some of the lost passes are expectations within a test_that block whose first expect_* raised an error).

I do not yet know the precise cause of this regression. Initial speculation was "the class assertion rejected non-root callers" — but the audit (§2.7 paragraph 1 above) shows every known caller passes ptr_root, so the assertion would not fire. Other speculation was "recursive calls inside the methods still hit the shadow trap" — plausible but not proven; prune_walk.ptr_root calls prune_walk(layer, ...) in a lexical scope (namespace) where bare prune_walk SHOULD resolve to the namespace generic. The third speculation was "S3 method bodies rely on UseMethod-set frame vars (.Generic, .Class, .Method) or expect to be invoked via NextMethod" — inspected, no NextMethod call in any prune_walk method, so this is unlikely too.

Action item for the next implementer: before trying Shape B (§2.10), reproduce Fix #2 and capture the FIRST failing test's traceback. The traceback will identify the actual recursion path that broke. Without that, all four proposed fix shapes are unverified.

2.9 What's known about the bug

QuestionAnswer
Does it reproduce under library(ggpaintr) from a real R CMD INSTALL?Not verified. Under a proper install the package env and namespace env share the same bindings; there is no shadow. So this may be a load_all-only artifact. Worth verifying by building the package with R CMD INSTALL ., then in a fresh R session: library(ggpaintr); library(dplyr); <repro>. If the empty filter prunes correctly there, the bug is dev-mode-only.
Why does ggpaintr:::prune_walk(sub, ...) called from globalenv work, but ptr_prune(sub) (which calls the same function) not work?UseMethod's method lookup uses the calling environment's enclosure chain in step (2). From globalenv, the walk doesn't hit any shadow. From inside ptr_prune living in package:ggpaintr, the walk hits the package env's S3 table (potentially incomplete) before reaching the namespace's.
How many internal callsites use bare prune_walk(...)?Many. R/paintr-prune.R alone has bare calls inside prune_walk.ptr_root, prune_walk.ptr_layer, prune_walk.ptr_call, prune_walk.ptr_pipeline, prune_walk.ptr_closure. Each recursion uses the bare form. A complete fix needs every one of these to dispatch through the namespace's S3 table, not the package env's shadow.

2.10 Proposed fix shapes (none verified)

Shape A — verify-then-no-op

If R CMD INSTALL doesn't reproduce the bug, document it as a load_all-only artifact and add a developer note: "if you reproduce filter() |> ggplot() in dev, run R CMD INSTALL . and confirm the bug doesn't repro under the installed package; the dev-mode artifact is benign because end users always run from an installed copy." Cheapest "fix" if applicable.

Shape B — convert prune_walk from S3 to explicit class-dispatch table

Replace the UseMethod-based generic with a function that switches on class(node)[1] and calls the matching helper by direct name (looked up via the namespace). No UseMethod, no caller-frame-dependent dispatch.

prune_walk <- function(node, remove_set, is_standalone) {
  cls <- class(node)[1]
  fn <- switch(cls,
    ptr_root             = prune_walk.ptr_root,
    ptr_layer            = prune_walk.ptr_layer,
    ptr_call             = prune_walk.ptr_call,
    ptr_pipeline         = prune_walk.ptr_pipeline,
    ptr_closure          = prune_walk.ptr_closure,
    ptr_user_expr        = prune_walk.ptr_user_expr,
    ptr_literal          = prune_walk.ptr_literal,
    ptr_missing          = prune_walk.ptr_missing,
    ptr_ph_value         = prune_walk.ptr_ph_value,
    ptr_ph_data_consumer = prune_walk.ptr_ph_data_consumer,
    ptr_ph_data_source   = prune_walk.ptr_ph_data_source,
    prune_walk.default)
  fn(node, remove_set = remove_set, is_standalone = is_standalone)
}

The method bodies stay unchanged but no longer need @export — they're called by direct name. The dispatch is now lexical (closure-captures the methods at namespace creation time), so the load_all package-env shadow has nothing to do with method lookup. Same closure env as the rest of paintr-prune.R, so the namespace is the resolution scope.

Risk: this is a stylistic departure from R's idiomatic S3 dispatch. May feel un-R-like. But it's a one-time restructuring and the method bodies are unchanged.

Shape C — keep S3 but force every call to qualify via namespace

Every bare prune_walk(...) in R/paintr-prune.R becomes get("prune_walk", envir = asNamespace("ggpaintr"))(...). Pollutes the methods but is mechanical. Need to verify whether UseMethod from inside such a "namespace-pinned" generic actually escapes the trap (Fix attempt #1 suggests it might not).

Shape D — restructure dispatch via fn_env() / topenv() tricks

Modify prune_walk to set its own caller frame before UseMethod dispatches. Possible via sys.function() + environment<- manipulation. Hacky and fragile.
