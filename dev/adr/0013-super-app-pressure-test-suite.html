ADR 0013 â€” Super-app pressure-test suite for placeholder Ă— host Ă— capture-mode interactions

Date proposed: 2026-05-23 Â· Owner: ggpaintr Â· Status: PROPOSED Â· Cross-refs: ADR 0009 (pp* grammar + expression-mode capture, MERGED 2026-05-21) Â· ADR 0010 (ppUpload(name), MERGED 2026-05-22) Â· ADR 0012 (role-based tree) Â· FEATURE-CHECKLIST (post-ADR-0009 inventory) Â· WORKFLOW (test workflow)

Context

Post ADR-0009 (expression-mode capture) and ADR-0010 (ppUpload(name)), ggpaintr's user-visible surface has grown along several axes that are individually unit-tested but never simultaneously exercised in one app. The risk that motivated this ADR is interaction bugs â€” defects that exist only when two or more features cooperate, of the kind that unit tests structurally cannot catch.

The single regression that motivated the FEATURE-CHECKLIST (the "code with placeholders" B3 toggle that was wired to a renderer but the renderer silently returned final-mode text; fixed in 2c504da) is the canonical example: every individual piece (toggle widget, preserve-mode renderer, code-panel reactive) was unit-covered, yet the integration between them silently failed. Presence-proxy assertions in narrow fixtures (expect_dom_id, expect_code_nonempty) hid it for ~one cycle. The lesson â€” recorded in project memory e2e-assertion-weakness-lens â€” is that propagation, not presence, is the assertion shape that catches integration regressions.

The existing test inventory has:

  • ~1,583 unit / integration tests under tests/testthat/test-*.R, each focused on a narrow seam (parser, registry, render, copy, runtime, etc.).
  • ~27 narrow e2e fixtures under tests/testthat/fixtures/vignette-apps/, each demonstrating one feature (a single ADR, a single bug). These are bound to vignette chunks and use the app-dir + pkgload pattern (project memory shinytest2-appdir-pkgload).
  • 5 user-facing vignettes under vignettes/ (customization, gallery, llm, safety, use-cases).

The structural gap: no fixture combines more than one or two features in the same formula, so the "two pieces both work, but their interaction silently corrupts state" failure mode has no scheduled gate-time exercise. A super-app pressure-test suite fills this gap.

Decisions

D1 â€” Partition principle: risk-seam by host Ă— data-head Ă— capture-path

The FEATURE-CHECKLIST's bottom-of-doc "super-test" suggestions are unstructured. We adopt a structured partition: each super-app picks one value of every mutually-exclusive slot, and crams the densest possible combination of compatible non-exclusive features inside. The exclusive slots â€” values a single formula can take exactly one of â€” are:

  • Embedding host: ptr_app() Â· ptr_app_grid() Â· hand-rolled L3 (ptr_ui_* + ptr_server()). (ptr_app_bslib() is slated for removal â€” not included in the partition.)
  • Data head: dataset literal (e.g. mtcars) Â· local-var dataset Â· ppUpload() at formula head Â· ppUpload() |> ... mid-pipeline Â· custom data source at head (D3 hook).
  • Formula-capture path: G1 string Â· G2 bare-expression Â· G3 !! splice Â· G4 pre-quoted via rlang::expr() Â· G5 string-builder force-eval (paste0 / sprintf).
  • Code-panel style: window Â· panel.

A single mega-app can hit one value per row only. The interaction bugs we most fear (B3 toggle Ă— E1 shared= in each host; mid-pipeline upload Ă— shared widget Ă— code-mode preserve; L3 custom output Ă— runtime error Ă— validate_input) live across host/data-head boundaries â€” so they require >1 app to cover. A suite of ~4â€“5, each maximally dense within its slot choices, covers them.

D2 â€” Suite size: 5 super-apps

Started at 4, expanded to 5 in response to a user-introduced refinement: App 2 should exercise all 3 custom-placeholder hook kinds functionally (not just register them). Since a custom D3 source claims the formula-head slot, and ppUpload-at-head is the existing canonical built-in source we also want pressured, App 2 splits into App 2a (built-in ppUpload-head) and App 2b (custom-D3-head). 5 apps is the upper edge of "a few" but each is independently debuggable when red. Maintainability cap: do not exceed 5 super-apps under this ADR; additional pressure points become narrow fixtures.

D3 â€” Artifact form: fixture-pair (manually runnable)

Each super-app ships as:

  • tests/testthat/fixtures/vignette-apps/super-N-<slug>/app.R â€” the live Shiny app. First line is the pkgload::load_all(Sys.getenv("GGP_PKG")) scaffolding (project memory shinytest2-appdir-pkgload); the rest is a regular Shiny app with natural labels and default values. Manually runnable via shiny::runApp("tests/testthat/fixtures/vignette-apps/super-N-<slug>").
  • tests/testthat/fixtures/vignette-apps/super-N-<slug>/reference.R â€” the equivalent ggplot2 expression at all-defaults, with no placeholders. Doc-only: not parsed by any test, not asserted against the substituted output. Its role is human eyeball comparison and copy-paste reuse.
  • tests/testthat/fixtures/vignette-apps/super-N-<slug>/*.csv (where relevant) â€” sample upload files, shipped sibling to app.R so the test can drive ppUpload widgets deterministically.
  • tests/testthat/test-super-pressure.R (single file, all 5 apps inside as separate test_that() blocks) â€” automated assertions via shinytest2 AppDriver.

Rationale: a manual playground that does not also run in the gate rots; pure automated fixtures that aren't manually launchable are debug-hostile. The fixture-pair pattern already in use by the narrow ADR fixtures gives both for free.

D4 â€” Assertion contract: probe-ledger with sentinels

For every placeholder in each super-app, the test sets a literally-unique sentinel value via app$set_inputs(<id> = <sentinel>, wait_ = FALSE), clicks the Update button, then asserts:

  • Final-mode propagation: the sentinel appears in the rendered code-panel text in its substituted form (e.g., ppNum sentinel 0.7314159 appears as the literal 0.7314159; ppVar sentinel "cyl" appears as the bare symbol cyl; ppExpr sentinel appears as the user-typed expression).
  • Preserve-mode round-trip: after toggling B3 to "Code with placeholders," the sentinel appears in the rendered text wrapped as ppX(<sentinel>) for value placeholders, as the bare symbol for var placeholders (consistent with how render_placeholder_preserved dispatches per keyword), with shared = "<key>" retained where applicable.
  • No error class on plot output: no .shiny-output-error on the plot pane.

This contract directly addresses the assertion-weakness lens from project memory e2e-assertion-weakness-lens â€” propagation, not presence, is what catches integration regressions. Every super-app's test includes at least one B3 toggle assertion, since that's the regression that motivated the FEATURE-CHECKLIST.

Rejected alternatives:

  • Snapshot HTML / code-text â€” high false-positive rate on incidental layout drift; tempts reviewers to rubber-stamp "update snapshot" PRs.
  • Targeted lightweight presence â€” the exact shape that hid B1 historically.
  • Plot-level rasterization via vdiffr â€” adds a test-time dependency, .svg snapshots checked into git, brittle to ggplot2 version drift.
  • AST-equal against reference.R â€” strict equality fails because of A10 silent-absence (empty ppText() drops out of labs() entirely; reference has subtitle = ""; substituted has no subtitle arg). Substring inclusion was considered, dropped in favor of sentinels which are stronger and don't depend on reference shape.

D7 â€” Multi-data-source via layer data =

Two super-apps demonstrate that a geom_* layer can carry its own placeholder-source data = distinct from the root data:

  • App 2a: root ppUpload(df_main) at formula head; geom_smooth(data = ppUpload(df_aux), aes(...), ...) at layer level. Two independent file widgets, two independent ppVar-picker scopes.
  • App 2b: root custom ppSample() at formula head; geom_rug(data = ppUpload(df_rug), aes(...)) at layer level. Tests ppUpload as a layer-data source (a path not covered by H3/H4 alone â€” those scope ppUpload at the formula head, not at layer-level data =).

Deliberately not tested: shared= spanning two different data-source scopes. That edge case (picker shows union of columns? intersection? first-data's? errors?) has under-specified semantics and deserves a focused narrow fixture rather than a super-app pile-on. Within each super-app, shared= stays inside the root-data scope only.

Coverage gaps deliberately left unfilled

The super-app suite is not intended to subsume all narrow fixtures. The following are not in scope and remain in narrow-fixture or unit-test land:

  • E3, E4 (shared label / default tiebreak across differing keys) â€” App 1 covers C6 (same-key tiebreak) but not the cross-key edge cases. Worth a small narrow fixture; cheap to add later.
  • E7 (mid-pipeline shared= on a data source) â€” shared= keys spanning data-pipeline stages. App 2a's shared= stays within the post-pipeline aes scope.
  • E8 (preserve-mode shared= round-trip) â€” App 1's preserve-mode assertion covers shared="grp" round-tripping in ptr_app(). The host-equivalent assertion under L3 is not in the super-app contract; it could be folded into App 3 cheaply if a regression appears.
  • D7 (keyword shadow rejection) â€” registering a keyword that exists in base R / utils / stats / dplyr / ggplot2 aborts. Adversarial registration cases. Would fit in App 4 or a separate narrow fixture; not currently included.
  • D8 (LHS-name vs keyword= drift warning) â€” best-effort warning when ppFoo <- ptr_define_placeholder_value(keyword = "bar"). Lexical-shadow path only; not exercisable from a single super-app's app.R because each app uses lexically-matching names.
  • F5 (ptr_extract_plot / _error / _code) â€” non-reactive helpers for download handlers and testServer. App 3's test could assert ptr_extract_code(state_b) returns final-mode text; recommended addition but not part of the locked contract.
  • G7, G8 (capture-time rejection of non-string non-call symbols, multi-expression formulas) â€” adversarial; would fit App 4's safety probes; not currently included.
  • I6, I7 (parse-time abort with malformed formula, empty/whitespace formula) â€” narrow probes, not super-app size.
  • J2 (default-arg constant-fold registry) â€” covered by existing unit tests; not surface-able through app.R.
  • J5 (keyword shadow-guard) â€” same as D7; adversarial registration.
  • J6 (parser uses no eval/parse/get on AST) â€” source-grep test exists; not testable from app.R.
  • cross-data-source shared= â€” shared= keys spanning two layers that consume different data sources. Under-specified semantics (which data's columns does the picker show?). Deserves a focused narrow fixture once semantics are decided.
  • ggiraph / DT custom outputs â€” App 3 covers plotly; ggiraph and DT use the same state$runtime()$plot path. Existing narrow fixtures cover them.
