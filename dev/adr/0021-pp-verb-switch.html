ADR 0021 — ppVerbSwitch: user-controlled UI checkbox for any pipeline verb

Status: Proposed. Drafted 2026-05-24.

Date: 2026-05-24

Relates to: ADR-0009 (the pp* placeholder family), ADR-0012 (canonical typed tree; spec= shape and apply-timing), ADR-0020 (parent — supersedes its Out-of-Scope follow-up note that named a sibling-node ppControl; this ADR records the user-adjudicated direction of wrapping instead of siblings, and supersedes ADR-0020's "stage_id is genuinely runtime-semantic" framing with the corrected source-code-evidenced framing "stage_id is a UI-routing key, not execution-shape").

Context

ADR-0020 shipped ppLayerOff and ppVerbOff, restoring the formula-as-source-of-truth principle for layer/stage default visibility. During execution, Plan-03 SC7 (the requirement that ppVerbOff(verb, TRUE) compare structurally-equal to bare verb) was dropped because the parent ADR's framing classified stage_id as "runtime-semantic" and therefore ineligible for the comparator-exclusion list. The parent's Out-of-Scope note proposed a future sibling-node shape (ppControl) as the resolution.

Two things are wrong with that proposal:

  1. The user-facing surface is wrong. Users write pipelines with wrapping syntax: mtcars |> ppVerbSwitch(mutate(x = x + 1)) |> ggplot(...). A sibling-node shape cannot be expressed in surface R without inventing a non-pipe operator. The parent's proposal would require either a tree-shape-only sibling (with the user still writing the wrapper) or a separate surface entirely — both costly.
  2. The "stage_id is runtime-semantic" framing overstates what stage_id does. Source-code inspection (verified facts below) shows stage_id is a UI-routing key: a stage with a stage_id but no UI checkbox bound (stage_enabled[[sid]] is NULL) behaves identically at runtime to a stage with no stage_id. stage_id only enables a UI-driven gate; it does not change runtime behavior on its own. Therefore it belongs in the same comparator-exclusion category as default_active / default_stage_enabled — UI metadata, not execution metadata.

This ADR adds the new keyword ppVerbSwitch, hard-removes the just-shipped ppVerbOff in favor of it (per user direction "no multiple surfaces doing the same thing"), and corrects the comparator-exclusion classification.

The capability gap that motivates the new keyword

Today, a pipeline verb stage gets a stage-enabled UI checkbox only if its subtree contains at least one pp* placeholder. The gate is in R/paintr-disable.R:20-22:

is_data_chain_call <- function(node) {
  is_ptr_call(node) && walk_has_placeholder(node)
}

So bare mutate(x = x + 1) in a pipeline gets no stage_id and no checkbox. The user has no formula-side way to say "give me a checkbox for this verb stage" without inventing a placeholder they don't need. ADR-0020's ppVerbOff closed half the gap (it forces a checkbox AND boots it off via Plan-01's stamp_default_stage_enabled_ids() workaround), but only when hide = TRUE. There is no formula-side way to say "give me a checkbox for this verb stage, default on" or "give me a checkbox with this custom label".

Verified facts (2026-05-24)

  • stage_id is a UI-routing key. The full chain: is_stage_disabled at R/paintr-disable.R:149-154 returns FALSE when sid is NULL or when stage_enabled[[sid]] is anything other than literal FALSE (isFALSE() rejects NULL); disable_walk.ptr_pipeline at R/paintr-disable.R:116-134 only drops stages when is_stage_disabled returns TRUE. Therefore a stage with a stamped stage_id but no bound reactive (the "unwired" case) behaves identically to a stage with no stage_id at all: both fall through is_stage_disabled → FALSE → stay. stage_id's only runtime effect is to enable a UI-driven drop; it does not change baseline execution. This contradicts ADR-0020's Out-of-Scope framing.
  • assign_stage_ids placeholder-gating at R/paintr-disable.R:55-72 uses the single predicate is_data_chain_call for both the pipeline-stage branch (line 62) and the lone-call branch (line 67). Extending that predicate is the natural extension point.
  • Plan-01 workaround. stamp_default_stage_enabled_ids() at R/paintr-translate.R:131-153 is a second-pass walk introduced specifically to mint stage_ids on ppVerbOff(..., TRUE)-wrapped stages that assign_stage_ids skipped. Once the gate is extended (see Decision below), this entire helper becomes redundant and is deleted.
  • Existing ppVerbOff unwrap. unwrap_pp_verb_off_stage at R/paintr-translate.R:832-853 translates the inner verb call via translate_node (line 844) and stamps inner_node$default_stage_enabled <- !isTRUE(hide). Raw mutate(x = x + 1) is a call object that fails is_ptr_node(), but post-translate_node it is a ptr_call (subclass of ptr_node) that passes is_ptr_call on line 845. The same machinery applies for ppVerbSwitch; no new typing concerns.
  • Current label rendering. build_pipeline_stage_ui at R/paintr-build-ui.R:230-243 auto-derives the stage checkbox label as shiny::tags$code(paste0(verb, "()")). There is no user-override path today. controllable_region at R/paintr-region.R:36-50 splices whatever head_label it receives into shiny::checkboxInput(label = ...) without further transformation.
  • Comparator exclusion list. ptr_tree_structural_equal at R/paintr-nodes.R:135-162 already excludes c("op", "expr", "default_active", "default_stage_enabled"). The pattern is established; extending it to stage_id, has_user_control, and stage_label is a one-line edit (line 142-143).
  • ppVerbSwitch(verb, switch_on=FALSE) ≡ ppVerbOff(verb, hide=TRUE). Naked-R: both return(.data) via symmetric guards (isTRUE(hide) vs isFALSE(switch_on)). Translate-time typed tree: both stamp default_stage_enabled = FALSE on a ptr_call obtained from the same translate_node path; the new field has_user_control = TRUE on the ppVerbSwitch-side is a no-op when default_stage_enabled = FALSE already fires the gate; both yield the same path-derived stage_id. Subsumption is full: ppVerbSwitch(verb, switch_on=FALSE, label="…") is a strict superset of ppVerbOff(verb); ppVerbSwitch(verb, switch_on=TRUE) is a new capability (UI checkbox, boot ON) that ppVerbOff cannot express; ppVerbOff(verb, hide=FALSE) is a no-op wrapper whose migration target is bare verb.

Decision

Introduce one new keyword:

ppVerbSwitch(.data, verb_expr, switch_on = TRUE, label = NULL)

Naked-R semantics. Identical shape to ppVerbOff: .data first, verb_expr captured via base-R substitute(), first-arg injection via as.call() + eval() in parent.frame(). switch_on = FALSE ⟹ return(.data) (skip the verb); switch_on = TRUE ⟹ apply the verb to .data. label is ignored outside ggpaintr (pure UI metadata).

Translate-time semantics (in ptr_app / ptr_server):

  • A new helper unwrap_pp_verb_switch_stage mirrors unwrap_pp_verb_off_stage: validates that switch_on is a literal scalar non-NA logical and label is a literal scalar character or NULL (both via new helpers validate_pp_verb_switch_switch_on and validate_pp_verb_switch_label, aborting with class "ptr_translate_error"); translates the inner verb call via translate_node; stamps three fields on the resulting ptr_call: has_user_control <- TRUE, default_stage_enabled <- switch_on, stage_label <- label.
  • Used outside a pipeline-stage position → translate-time error via the existing ptr_assert_no_surviving_structural_wrappers at R/paintr-translate.R:161-191. Extend the wrapper-name list from c("ppLayerOff", "ppVerbOff") to c("ppLayerOff", "ppVerbSwitch") and update the error-message branch to mention ppVerbSwitch's valid position.

Node-constructor change. Add two new slots to ptr_call in R/paintr-nodes.R: has_user_control = FALSE and stage_label = NULL. The existing default_stage_enabled = TRUE slot is unchanged.

Stage-id gate. Extend is_data_chain_call in R/paintr-disable.R:20-22 from one predicate to a disjunction:

is_data_chain_call <- function(node) {
  is_ptr_call(node) &&
    (walk_has_placeholder(node) || isTRUE(node$has_user_control))
}

Single source of truth for "should this stage have a UI checkbox?" Three triggers (placeholder presence, explicit has_user_control from ppVerbSwitch) collapse to one gate.

Delete stamp_default_stage_enabled_ids. The Plan-01 second-pass workaround (R/paintr-translate.R:131-153 + the call site at line 117) is no longer needed once the gate is extended: every stage that gets has_user_control = TRUE from ppVerbSwitch already gets a stage_id from assign_stage_ids via the extended gate. The lone caller at line 117 is removed in the same change.

UI emission. Thread stage_label through the same entries carrier in R/paintr-build-ui.R that already threads default_stage_enabled (line 240, 333, 451). build_pipeline_stage_ui at line 232-237 chooses between user-label and auto-label:

head_label <- if (!is.null(entries[[i]]$stage_label)) {
  entries[[i]]$stage_label              # user-provided plain text
} else if (has_verb) {
  shiny::tags$code(paste0(verb, "()"))  # existing auto-label
} else NULL

Mirrored in wrap_shared_widgets_with_stage_blocks at line 517-520.

Wrap-with-placeholder is allowed. ppVerbSwitch(mutate(x = ppVar(col)), label = "Transform x") is legal — the wrapper composes with placeholder-driven stage_id stamping; the user gets both their custom label and the inner placeholder widget. has_user_control = TRUE is idempotent w.r.t. placeholder presence in the gate logic.

Hard-remove ppVerbOff. Per user direction, no deprecation window. Removed in the same change: the ppVerbOff exported function in R/paintr-builtins.R:477-497 (and its roxygen @rdname pp_off_toggles entry); the ppVerbOff entry in ptr_builtin_keywords() at line 261; the ptr_register_structural_keyword("ppVerbOff") call at line 361; the unwrap_pp_verb_off_stage helper and its caller in build_pipeline_from_lift (replaced by the ppVerbSwitch equivalents); the ppVerbOff-specific branches in ptr_assert_no_surviving_structural_wrappers and the validate_pp_off_hide helper (replaced by the ppVerbSwitch-specific validators); the man page combined-Rd pp_off_toggles splits into one Rd for ppLayerOff alone (unchanged) and one new Rd for ppVerbSwitch; the use-cases vignette section "Layer and stage toggles — ppLayerOff and ppVerbOff" retitled to "Layer and stage toggles — ppLayerOff and ppVerbSwitch" with the ppVerbOff prose rewritten for ppVerbSwitch; the five test-pp-off-*.R files renamed to test-pp-toggles-*.R with the ppVerbOff halves rewritten as ppVerbSwitch coverage (the ppLayerOff halves stay verbatim).

ppLayerOff is unchanged. No parallel ppLayerSwitch. Layers already get a checkbox unconditionally, so the "force a checkbox to exist" half of ppVerbSwitch's value has no layer-side analogue. The "custom label" half is a real follow-up that this ADR explicitly does not address (out of scope below).

Precedence (boot value of the checkbox)

Same 3-band ladder as ADR-0020 § Precedence, with switch_on in place of hide (and the flag inverted):

  1. spec = entry naming the <stage_id>_enabled input id (ad-hoc one-shot at boot).
  2. Formula-side ppVerbSwitch(verb, switch_on = L) → default_stage_enabled = L on the carrier ptr_call.
  3. Implicit TRUE.

Structural-equality comparator (correction of ADR-0020 framing)

ADR-0020 added default_active and default_stage_enabled to the exclusion list. Its Out-of-Scope note explicitly preserved stage_id as not excluded, on the basis that stage_id is "runtime-semantic — disable_walk reads it to drop stages." Verified facts above show that framing is wrong: stage_id only enables a UI-driven gate; the gate's runtime effect (is_stage_disabled → drop) is identical between "no stage_id" and "stage_id with no bound reactive" — both fall through to FALSE.

Therefore extend the exclusion list at R/paintr-nodes.R:142-143:

names_a <- setdiff(names(a), c("op", "expr", "default_active",
                                "default_stage_enabled",
                                "stage_id", "has_user_control",
                                "stage_label"))
names_b <- setdiff(names(b), c("op", "expr", "default_active",
                                "default_stage_enabled",
                                "stage_id", "has_user_control",
                                "stage_label"))

Consequence (restores Plan-03 SC7 by side effect): ppVerbSwitch(mutate(x=1), switch_on=FALSE) ≡ bare mutate(x=1) structurally. The verb-side structural-equality invariant the parent ADR's note named is restored — through wrapping + corrected comparator-classification, not through siblings.

ADR-0012 invariant (tree is semantic, not syntactic)

Holds for all values of switch_on: surface forms data |> ppVerbSwitch(verb, switch_on=L), ppVerbSwitch(data, verb, switch_on=L), and data %>% ppVerbSwitch(verb, switch_on=L) all produce the same canonical typed tree (same stages, same flags). The carrier ptr_call still carries the has_user_control / stage_label / default_stage_enabled metadata so downstream readers (gate, snapshot, UI emitter) see what the user requested — but the comparator deliberately ignores it.

Out of scope for this ADR

  • The exact spelling of error messages from translate-time validation (literal-only switch_on / label; wrong-position keyword).
  • A parallel ppLayerSwitch(layer_expr, switch_on = TRUE, label = NULL). Layers already get a checkbox unconditionally, so the "force a checkbox" half doesn't apply; the "custom label" half is a real follow-up but not addressed here per user direction.
  • HTML or shiny::tag values for label =. label is a literal scalar character only; HTML/markup labels are a follow-up question if user demand arises.
  • Whether ppVerbSwitch wrapping a stage that already contains a placeholder should overwrite that placeholder's auto-derived label suffix (" in mutate()") or compose with it. The current decision: the stage-level stage_label overrides only the stage-block head label, not the per-placeholder suffix; if this proves confusing, a follow-up can rationalise.

Consequences

What this buys

  • The formula-as-source-of-truth principle extends to any verb-stage UI checkbox. Users can opt a bare verb stage into the UI without inventing a placeholder, and can give it a meaningful label.
  • ADR-0020's verb-side structural-equality invariant is restored as a side effect. ppVerbSwitch(verb, switch_on=FALSE) ≡ bare verb; the Plan-03 SC7 follow-up is closed.
  • One mental model for verb-stage UI presence: the gate is_data_chain_call answers "this stage has a UI checkbox" by checking placeholder presence OR has_user_control. No separate post-translate stamping pass; stamp_default_stage_enabled_ids is deleted.
  • One mental model for the pp*-toggles vocabulary: ppLayerOff(layer, hide=) for layers (which already have checkboxes), ppVerbSwitch(verb, switch_on=, label=) for verb stages (which don't). No ppVerbOff alias.
  • The stage_id-classification correction unblocks future work that wants to compare trees across UI-routing configurations (e.g., spec-roundtrip closed-loop tests, drift detection between equivalent formulations).

What this does NOT buy

  • No new runtime power beyond UI affordance. ppVerbSwitch doesn't change what verbs run when the checkbox is checked — it only decides whether a checkbox exists and what it boots to.
  • No layer-side label override. ppLayerOff remains label-less; layer-checkbox labels still come from layer name. Symmetric label work is a follow-up.

Risks

  • Hard-removal of ppVerbOff is a public-API break. Justified by user direction ("no one uses it") and by ppVerbOff having shipped only two days prior (2026-05-24, ADR-0020). Existing formulas using ppVerbOff must migrate to ppVerbSwitch(verb, switch_on=FALSE[, label=…]); ppVerbOff(verb, hide=FALSE) migrates to bare verb (no UI affordance, no behavioral change). The implementation plan must search-and-replace every internal call site (vignettes, tests, fixtures) in the same change.
  • The comparator-exclusion-list extension overrides ADR-0020's Out-of-Scope guidance. Future readers will see ADR-0020 say "do NOT add stage_id to the exclusion list" and ADR-0021 do exactly that. The justification (source-code evidence) is in §"Structural-equality comparator" above; ADR-0020's Status block will be amended in the same change to point to ADR-0021 as the corrected framing on this specific point.
  • The "wrap a placeholder-containing stage" composability case. ppVerbSwitch(mutate(x = ppVar(col)), label="X") produces a stage with both a user-label and inner-placeholder widgets. The label override path interacts with the existing per-placeholder label-suffix rule (build_pipeline_stage_ui at line 197-200). The implementation plan must add a regression test that asserts the stage-block head label = the user's label=, while the inner placeholder labels keep their existing rendering.
  • Test-rename + test-rewrite churn. The five test-pp-off-*.R files are renamed and partially rewritten. The plan must verify (a) the renamed files load via testthat::test_dir's ^test.*\.R$ glob, (b) the ppLayerOff halves still pass byte-for-byte, (c) the ppVerbSwitch-rewritten halves cover every scenario the ppVerbOff halves did (boot-off, spec-override, snapshot, browser-e2e, vignette-chunk).
  • Translate-time literal validation surface grows. Two new validators (validate_pp_verb_switch_switch_on, validate_pp_verb_switch_label) join validate_pp_off_hide (kept for ppLayerOff's hide). Three validators with similar shapes; opportunity for one parameterised helper, but only if the savings outweigh the indirection cost. Implementation plan decides.
