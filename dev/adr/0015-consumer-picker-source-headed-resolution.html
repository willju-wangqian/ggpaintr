ADR 0015 — Consumer picker binding under source-headed upstream

Date: 2026-05-23 · Owner: ggpaintr · Status: DECIDED · Cross-refs: ADR 0013 §App-2b (test that surfaced the defect) · supersedes the fix-super1 pre-warm exclusion at R/paintr-server.R:1685

One-line summary. Two parallel defects keep a consumer renderUI from binding once its upstream contains a data-source placeholder (ppUpload, ppSample, etc.): the non-shared site suspends the picker on visibility (waits for the user to navigate to the inner Controls subtab); the shared site has a complete dep set for pipeline-head sources but is missing state$resolved_data for bare-data-source layers. Fix is two narrow edits: drop the pre-warm gate + add a req() guard in ptr_setup_consumer_uis(), and add one dep line in ptr_bind_shared_consumer_uis().

1. Defect

1.1 Symptom (ADR 0013 §App-2b diagnostic record)

From the predecessor's commit a73ccca on branch 0013-app-2b-customsource-splice:

Layer-aes ppVar nested under a layer-data ppUpload — geom_rug(data = ppUpload(df_rug), aes(x = ppVar(mpg))) — emits its renderUI scaffold (geom_rug_1_1_ppVar_NA_ui DOM container is present) but the inner picker never binds (stays in recalculating). Preserve mode reveals the symptom: the captured code shows aes(x = ppVar) with the mpg default dropped.

The predecessor's hypothesis — "the consumer-discovery walker misses layer-aes ppVar under a layer-data ppX source" — was falsified at two diagnostic levels (see §1.2). The actual mechanism is runtime UI binding, not static classification.

1.2 Falsifying probes (both ran on this branch, base 4a23d39)

Static-tree probe (/tmp/probe-followup-b.R) — built the diagnostic formula via ptr_translate(formula_str) and inspected the classified tree. Found:

  • find_nodes(tree, is_ptr_ph_data_consumer) DOES find geom_rug_1_1_ppVar_NA.
  • classify_walk.ptr_layer at R/paintr-classify.R:39–54 classifies data_arg first and passes the classified data_arg as children_ctx to children, so the consumer's upstream IS correctly wired to the ppUpload source (id geom_rug_0_ppUpload_NA).
  • ptr_runtime_input_spec() DOES emit a row for the consumer (and a row for the ppUpload source + its companion).

Resolution-path probe (/tmp/probe-followup-b-resolve.R) — given snapshot = list(companion_id = "df_rug") and an eval_env with df_rug bound, ptr_resolve_upstream(consumer$upstream, …) returns the right data.frame with cols mpg, wt. With either input missing it returns NULL via the documented paths (ptr_missing() pruned, or silent tryCatch(eval, error = NULL)).

Therefore the static classifier, the walker, and the resolver are all correct. The runtime UI binding path is where the failure lives.

1.3 Mechanism A — non-shared consumer is suspended on visibility

ptr_setup_consumer_uis() at R/paintr-server.R:1685 (added by fix-super1) only pre-warms a consumer renderUI when its upstream contains no data source:

if (length(find_nodes(node$upstream, is_ptr_ph_data_source)) == 0L) {
  shiny::outputOptions(output, output_id, suspendWhenHidden = FALSE)
}

For geom_rug(data = ppUpload(df_rug), …), the consumer's upstream IS the ppUpload source → the condition is false → no pre-warm → the renderUI stays suspended (suspendWhenHidden = TRUE). The picker's DOM container is in the layer panel's static UI tree, but the picker itself never renders until the user activates the inner Controls subtab. The diagnostic super-2b test only passes because it explicitly calls set_sentinel(app, "ggplot_subtab", "Controls") before each root-aes pick; the layer-aes ppVar inside geom_rug is never given an analogous kick, so it stays unrendered — which the predecessor described as "stays in recalculating indefinitely".

The fix-super1 rationale (preserved in the same inline comment) is real but bounded:

Stay lazy when the upstream still contains a ptr_ph_data_source (e.g. ppUpload-headed pipelines): firing the renderUI eagerly there would race the upload observer's state$eval_env binding and would pollute state$upstream_cache with the wrong frame found by R's scoping fallback (datasets::penguins for <name> |> filter(...) before the uploaded df is bound).

The race is real if the renderUI calls ptr_resolve_upstream before the source has resolved. The proper guard is not suspension — it is an explicit req() on the appropriate source-ready signal (state$resolved_data[[ln]]() for bare-data-source layers, state$resolved_sources[[id]]() for pipeline-head sources). req(NULL) raises shiny.silent.error, which Shiny treats as "not ready yet" — the renderUI halts BEFORE ptr_resolve_upstream runs, so the cache is never touched, the scoping fallback never has a chance to find datasets::penguins, and the renderUI re-fires once the source resolves.

1.4 Mechanism B — shared consumer's dep set is missing resolved_data

ptr_bind_shared_consumer_uis() at R/paintr-server.R:1702–1830 has no outputOptions(suspendWhenHidden = FALSE) at all. Shared widgets live in always-visible host regions (sidebar / standalone ptr_shared_panel()), so the renderUI binds at boot regardless of visibility. The non-shared mechanism (§1.3) does not apply.

The shared renderUI's source-side dep set is:

for (rv in state$resolved_sources) rv()   # pipeline-head sources
for (cmp in upstream_source_companion_ids) { input[[ns(cmp)]] }
for (sid in upstream_source_self_ids)      { input[[ns(sid)]] }

It watches state$resolved_sources (allocated only for pipeline-head sources, per ptr_init_state at R/paintr-server.R:162–180) but never state$resolved_data (allocated only for bare-data-source layers, same block). For the combo geom_X(data = ppUpload(...), aes(... = ppVar(shared = "k"))):

  • The ppUpload source is a bare-layer source → it has a slot in state$resolved_data[[layer_name]] but NOT in state$resolved_sources[[source_id]].
  • The bare-layer upload observer (Loop 1 in ptr_setup_pipelines(), R/paintr-server.R:788–852) binds the resolved df into state$eval_env and bumps state$resolved_data[[ln]] — but never state$resolved_sources[[sid]] (no slot).
  • The shared renderUI has no dep on resolved_data. The only signals that fire it after upload are the companion text input and the source-self input — neither necessarily changes on upload (companion defaults from the formula's literal arg via ppUpload(df) — if the autobind would set the same value, no change event fires).

Empirical confirmation via an instrumented AppDriver probe in /tmp/probe-shared-upload/ (full trace in §1.5). Five different trigger sequences (upload alone; bump companion to df2; bump again to df3; click Update Plot) every time left the shared widget showing the "Data source not yet provided" advisory with zero picker options. The resolution$value upstream is correct, ptr_resolve_upstream with the right snapshot + eval_env returns the right df — the renderUI's body just never receives the right inputs at the right time because the dep set doesn't latch the upload-completed signal.

1.5 Instrumented trace (proof of mechanism B)

Probe replaced ptr_resolve_upstream in the fixture's app.R via assignInNamespace to log (snap, env_keys) on every call. Formula:

ptr_app(
  ggplot() +
    geom_point(data = ppUpload(df),
               aes(x = ppVar(shared = "v"),
                   y = ppVar(shared = "v")))
)

Trace (lines of form (snap, env_keys) -> result):

BOOT:
  (empty, empty)  -> NULL    # runtime_upstream_data initial pass
  (df,    empty)  -> NULL    # shared renderUI first fire; companion default "df"; eval_env empty
AFTER UPLOAD ALONE: (no new resolve calls — renderUI has no dep that fires)
AFTER COMPANION BUMP to "df2":
  (df2,   {df})   -> NULL    # renderUI re-fires from companion dep
                              # eval_env has 'df' (bound by an earlier observer fire)
                              # but snap is "df2" — eval(df2) finds nothing → NULL

The trace confirms:

  1. Upload alone triggers no shared renderUI re-fire (no dep latches the upload).
  2. When a companion change finally invalidates the renderUI, the snapshot's new companion value ("df2") leads the eval_env's old binding ("df") — the renderUI sees a stale eval_env in the same flush.

Adding for (rv in state$resolved_data) rv() to the renderUI dep block fixes both: the upload observer bumps resolved_data[[ln]] AFTER assigning to eval_env, so the resulting renderUI re-fire reads a settled eval_env. Symmetric with the existing resolved_sources dep that handles pipeline-head sources.

2. Decision — Option B (two-site fix)

Locked after grilling four candidate shapes and verifying empirically that one-site coverage is insufficient. Option B = the locked decision; A/C/D are recorded for completeness.

2.1 Change A — non-shared consumer pre-warm (ptr_setup_consumer_uis)

In R/paintr-server.R around the body of ptr_setup_consumer_uis() (lines 1564–1690):

  1. Drop the visibility-suspend gate at :1685. Replace 
    if (length(find_nodes(node$upstream, is_ptr_ph_data_source)) == 0L) {
  shiny::outputOptions(output, output_id, suspendWhenHidden = FALSE)
}
    with the unconditional call: 
    shiny::outputOptions(output, output_id, suspendWhenHidden = FALSE)
    matching the byte-stable pattern in ptr_setup_value_uis() and ptr_setup_source_uis().
  2. Add a req() guard at the top of entry_reactive. The guard must halt when the consumer's upstream depends on a data source that has not yet resolved. Walk node$upstream at setup time to discover whether the consumer is under a bare-data-source layer (node$layer_name appears in names(state$resolved_data)) or a pipeline-head source (an id appears in names(state$resolved_sources)); call the appropriate shiny::req(state$resolved_data[[ln]]()) / shiny::req(state$resolved_sources[[id]]()) as the first statement of the reactive body. When the upstream contains no source placeholder at all, no req() is needed (today's pre-warm-when-empty branch becomes the no-op default; consumer resolves immediately at boot).
  3. Drop the subtab refresh-hook dep. Remove the four lines: 
    subtab_id <- if (!is.null(node$layer_name)) {
  paste0(node$layer_name, "_subtab")
} else NULL
…
if (!is.null(subtab_id)) input[[ns(subtab_id)]]
    Verified redundant: D3 in lazy-consumer-resolve.md had the tab dep substituting for excluded producer-input deps; the implementation already includes producer-input deps directly (see upstream_producer_ids loop), so the tab dep is doubly redundant under eager pre-warm. Drops the R7 constraint on tabsetPanel(id=).

2.2 Change B — shared consumer dep set (ptr_bind_shared_consumer_uis)

In the renderUI body at R/paintr-server.R:1771, alongside the existing resolved_sources loop:

for (rv in state$resolved_sources) rv()
for (rv in state$resolved_data)    rv()    # NEW — symmetric with non-shared

One-line addition. No other code change to the shared path; the existing graceful-empty-state advisory branch handles the still-pre-resolution case correctly. No req() needed on the shared side — the advisory IS the equivalent UX.

2.3 Rejected alternatives

Option A (non-shared only). Was locked tentatively before the runtime probe; falsified by the instrumented AppDriver probe in §1.5 (shared widget never populates for the bare-data-source + shared combo across five trigger sequences). Documented as the trap: "shared widgets render at boot" was confused with "shared widgets refresh after upload" — the first is true, the second is not without the resolved_data dep.

Option C (full shared parity with req()). Treat the shared site exactly like the non-shared site (drop laziness, add req()). Rejected: the shared site is already not lazy on visibility — it has different mechanics — and changing its boot behavior would require redesigning the graceful-empty-state advisory branch. Option B closes the same hole with one line and zero UX change.

Option D (per-layer fast-path restoration). Restore the deleted-by-ADR-0012-§3.7 per-layer fast-path so the consumer reads state$resolved_data[[layer_name]] directly. Rejected: reverses an explicit ADR-0012 simplification and only addresses bare-data-source layers (pipeline-head sources keep the same suspension issue). Option B is uniform across both source shapes via the req() guard.

This test fails today (the gap the predecessor recorded) and must pass after Option B lands.

Add a new sibling super-pressure fixture tests/testthat/fixtures/vignette-apps/super-2c-shared-over-upload/ with:

4. Consequences

  • Public API: no change. ptr_setup_consumer_uis, ptr_bind_shared_consumer_uis, and ptr_setup_pipelines are all internal; no exported function signature changes.
  • Boot cost: every consumer renderUI evaluates once at boot. For typical ggpaintr formulas (placeholders per layer × layers ≲ 30) this is bounded and observable as a one-time cost. For consumers under an unresolved source, req() halts before ptr_resolve_upstream runs — no eval, no cache touch.
  • Cache pollution race (the fix-super1 concern): closed structurally. The renderUI body never calls ptr_resolve_upstream until the appropriate resolved_data / resolved_sources reactive is non-NULL — by which point the bare-source observer has already bound the resolved df into eval_env. The R-scoping-fallback path (datasets::penguins for <name> |> filter(...) before upload) cannot fire because the eval never runs.
  • D3 / R7 from lazy-consumer-resolve.md: the tab-activation dep is removed (D3's substitute-for-producer-deps role was already absorbed by the producer-input dep loop). R7's tabsetPanel-id constraint is relaxed — UI builders may keep or drop the id; nothing reads it server-side anymore.
  • Diagnostic branch 0013-app-2b-customsource-splice: the fixture files (app.R, sample_rug.csv) cherry-pick cleanly into followup-b as part of the regression-test step; the in-test gap comment is rewritten to a strict assertion. The branch itself can be deleted after merge.
  • Default-value flow: placeholder default_arg resolution is unaffected — it runs in substitute_walk at Draw time, not build_ui. Defaults still apply when the user hasn't picked a column.
  • Custom placeholders: the fix automatically propagates to any ptr_define_placeholder_consumer() custom consumer (same code path). VALUE and SOURCE roles are handled by different functions (ptr_setup_value_uis, ptr_setup_source_uis) that already pre-warm unconditionally — no change required.
  • Out of scope: behavioral changes to ptr_resolve_upstream, the typed-AST classifier, or the substitute walker. The fix is purely UI-binding plumbing in two functions.
