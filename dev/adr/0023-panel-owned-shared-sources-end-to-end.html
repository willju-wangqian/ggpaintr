ADR 0023 — Panel-owned shared sources end-to-end

Status: Accepted. Drafted 2026-05-25. Implemented (verified in source 2026-05-28: panel_sources bundle field + ptr_setup_panel_sources() / resolve_upload_source() in R/paintr-server.R; ADR §4/§5 PLAN-04/05 comments in place).

Date: 2026-05-25

Relates to: ADR-0005 (introduced ptr_shared / ptr_shared_panel / ptr_shared_server — never specified source-role behaviour), ADR-0006 (single public server entry; locked the partition-as-ownership rule for consumers/values — silent on sources), ADR-0015 (per-instance state$resolved_data / state$bound_names + req-gate — defines the per-instance picker→source dependency that this ADR generalises to host scope). Builds directly on the Cut 1 fix in commit d71893b (fix(shared): cut 1 — single-instance shared source renders + consumer reads canonical ids), which dropped the partition-blind skip in ptr_setup_source_uis() and fixed the rewrite-shared early-return.

Implementation: dev/plans/0023-panel-owned-shared-sources-end-to-end/ (TBD via /decision-to-plan).

Context

Two bugs surfaced against the new ptr_shared coordinator + ptr_app reduce to one root absence: source placeholders (ppUpload) were never wired through the shared/cross-formula path. ADR-0005/0006 (the L2/L3 redesign that introduced ptr_shared) discuss only value and consumer placeholders. Sources fell into the gap silently — partition logic gives them a canonical id and the panel calls build_ui_for, but every server-side wiring stopped short.

Bug 2 (single-instance) was fixed by Cut 1. Bug 1 (multi-instance, panel-owned shared source) remains and is the subject of this ADR:

panel <- ptr_shared(c(
  "ggplot(ppUpload(shared='ds'), aes(x = ppVar(shared='col'), y = ppVar)) + geom_point()",
  "ggplot(ppUpload(shared='ds'), aes(x = ppVar(shared='col'), y = ppVar)) + geom_col()"
))
# ui:     ptr_shared_panel(panel) +
#         ptr_ui(panel$formulas[[1]], "p1") + ptr_ui(panel$formulas[[2]], "p2")
# server: s <- ptr_shared_server(panel)
#         ptr_server(panel$formulas[[1]], "p1", shared_state = s)
#         ptr_server(panel$formulas[[2]], "p2", shared_state = s)

The upload fileInput appears at top-level #shared_ds after the Cut 1 fix would render it — but no host-side code resolves it, no per-instance pipeline knows to look at top-level rather than its own namespace, and no consumer-picker dep wakes when the upload lands. Three concerns:

ConcernPer-instance (works)Panel scope (missing)
Source widget renderUI (server-side filler)ptr_setup_source_uis()missing
Pipeline / resolve_data bindingptr_setup_pipelines()missing
Consumer-picker dep wiring downstreamptr_bind_shared_consumer_uis(... state = state)ptr_bind_local_shared_consumers(... state = NULL) — the dep block is gated on state non-NULL, so no resolved_data watch fires

Why the gap exists architecturally. state is created by ptr_init_state() inside moduleServer() for each ptr_server() call — per-formula by construction. ptr_shared_server() runs at the embedder's top level with no module wrapper, so no state exists to forward. ADR-0006 scoped the host server to shared widgets + cross-module signals, never per-formula state. Sources sit awkwardly in that scope: they need a server-side renderUI and a reactive resolved-df, but per-formula state is the wrong owner because the partition assigns them to the panel.

Constraints from the existing partition contract (CONTEXT.md, locked 2026-05-16):

  • Shared placeholders are role-agnostic — "every occurrence of the same key … is driven by one widget in lockstep" covers sources too.
  • Partition rule: key referenced in exactly one formula → that formula's inline section; ≥2 formulas → the standalone panel.
  • Ownership = whoever renders the surface is also the server binder: "the host server owns only panel keys; the owning instance binds every formula-local key" (resolved 2026-05-17). Today's source code does not honour this for the source role.
  • Shared panel uses global, un-namespaced ids (shared_<key>); cannot be namespaced or duplicated per module.

Decision

Extend the existing bundle channel between ptr_shared_server() and ptr_server() with a parallel source-side field. Per-instance pipelines learn one new branch: panel-owned source ids are substituted, not skipped. No new top-level abstraction; the partition is already at hand wherever it needs to be read.

  1. Bundle gains panel_sources. ptr_shared_state (returned by ptr_shared_server()) carries one new field: panel_sources, a named list of reactive data.frame values keyed by canonical shared id (shared_<key>), populated for exactly the panel-owned source keys (intersect of obj$panel_keys with source-role keys). The bundle stays "signals + payloads"; shared_resolutions is already a payload, so this is the same shape. Name kept: shared_state. No new top-level state object.
  2. Host-side resolver runs inside ptr_shared_server(). A new internal helper ptr_setup_panel_sources(obj, input, output, envir):
    • For each panel-owned source key, assigns the renderUI for the uiOutput(shared_<key>_ui) container that shared_panel_body_tag() already emits (the same renderUI body ptr_setup_source_uis assigns per-instance — extracted into a shared helper, see point 3).
    • Builds a reactive that watches the top-level input[[shared_<key>]] (fileInput) and the companion input[[shared_<key>_name]], runs the source registry's resolve_data hook, and falls back to try_bind_source_default() when the upload is empty and the source carries a default_arg resolvable through the host's envir parent chain.
    • Surfaces upload errors into the panel's ptr_shared_errors slot — symmetric with consumer-side errors. One widget = one error sink.
    • The companion textInput (shared_<key>_name) is rendered once at panel scope; all instances read the same name. Consistent with "one widget per shared key."
  3. Factor out the upload-resolution mechanics. The csv/tsv/rds/xlsx detection, the resolve_data-hook dispatch, the default-arg fallback (eyeball-play try_bind_source_default, commit a3d1636), and the binding-name derivation today live inside ptr_setup_pipelines(). Extract them into an internal resolve_upload_source(input_slot, companion_slot, node, entry, envir) helper that returns the resolved df + binding name. Per-instance pipeline calls it with namespaced slots; host calls it with top-level slots. One resolver, two callers — any future format addition (Parquet, Arrow) is made once.
  4. Per-instance pipelines substitute panel-owned source ids. In ptr_setup_pipelines() (R/paintr-server.R near line 1001), add a branch at the head of the per-source observer loop: 
    panel_owned <- sid %in% names(dots$panel_sources %||% list())
if (panel_owned) {
  # Bind from the host's resolved reactive instead of watching this
  # instance's own (non-existent) namespaced input. The panel resolver
  # populates `panel_sources[[sid]]` once per panel-owned key; this
  # observer takes a dep on it and mirrors the value into the
  # per-instance `state$resolved_sources[[sid]]` + `bound_names[[sid]]`
  # so downstream consumer reactives (`ptr_setup_consumer_uis` /
  # `ptr_bind_shared_consumer_uis`) see one unified resolved view
  # keyed identically.
  shiny::observe({
    df <- dots$panel_sources[[sid]]()
    if (is.null(df)) { slot(NULL); return(invisible()) }
    bind_source_value(state, sid, <binding_name>, df, slot)
  })
  return(invisible())  # do not also wire the input-watching observer
}
# else: existing per-instance path
    Downstream consumer code is unchanged — it reads state$resolved_sources and state$bound_names as today. The substitution is local to the source-observer loop; no other downstream code learns about panel ownership.
  5. Consumer-picker dep at host scope reads panel-owned sources. ptr_bind_shared_consumer_uis() (and ptr_bind_local_shared_consumers() when invoked at host scope by ptr_shared_server()) gains a panel_sources arg. When the consumer's resolved upstream source id sits in names(panel_sources), the renderUI takes a dep on panel_sources[[sid]]() instead of (or in addition to) state$resolved_sources / state$resolved_data. The existing if (!is.null(state)) gate (which today excludes the host path) is replaced with a panel_sources-aware path. One resolution code path, partition-routed. Cut 2 must preserve the has_rendered closure-flag pattern introduced by commit 401804c — first-render-omits-selected and subsequent-render-passes-current %||% character(0)-verbatim are independent invariants that Cut 2's panel_sources dep wiring must not regress; the panel_sources-aware path threads through the existing has_rendered closure, not around it.
  6. Parse-time invariant: panel-owned consumer requires panel-owned source. Inside ptr_shared() (the coordinator), after the partition is computed: for each panel-owned consumer key, every source it depends on (via the consumer's upstream pipeline) must also be panel-owned. If any source is formula-local, abort with a clear message naming the consumer key, the dependent source key, and the formulas where the asymmetry surfaces. Reasons: (a) a panel-owned consumer renders a single widget at top-level, so it cannot pick columns from two different formula-local datasets; (b) keeping it illegal preserves the "one widget, one source-of-truth" invariant; (c) cheap partition-time check; (d) no clean semantics for the cross-scope case (union? intersection? per-formula picker?), so disallowing is safer than picking arbitrarily.
  7. Partition-aware skip restored in ptr_setup_source_uis(). Cut 1 dropped the partition-blind skip entirely. With panel_sources now telling each ptr_server which ids the host owns, the skip is reintroduced in partition-aware form: if (s$id %in% names(dots$panel_sources %||% list())) next. The panel-side renderUI from point 2 owns those containers; the per-instance renderUI would otherwise write to orphan output slots (harmless today but a latent collision with future namespacing changes).

Consequences

  • Public surface change: ptr_shared_state gains a panel_sources field. Embedders who construct shared_state = manually (escape hatch) gain one new optional list entry. The bundle remains a forward-compatible plain list. validate_ptr_shared_state() grows one line.
  • Architectural framing intact: ADR-0006's "single public server entry" stands; the host doesn't gain its own engine, just a small resolver. The host still binds only panel keys; state remains per-instance.
  • Bundle channel grows, not the abstraction count: No new top-level state object, no new lookup chain, no eval-env parenting. panel_sources is a payload alongside shared / shared_resolutions. Reviewers can read the new field as one more line in the same shape they already know.
  • Single resolver: Upload format support (csv/tsv/rds/xlsx and any future addition) and the default-arg fallback live in one function called from both scopes. Removes the divergence-risk vector that motivated Q4-I in the grilling session.
  • CONTEXT.md correction: Line 125 ("the owning instance binds every formula-local key") becomes empirically true for every role (value/consumer/source) once Cut 2 lands. The clause is updated in the same change with role-explicit wording: "the owning instance binds every formula-local key — value, consumer, and source alike."
  • New parse-time error: "panel-owned consumer requires panel-owned source" — surfaces at ptr_shared() time, not at ptr_server() time, so the embedder sees the problem at coordinator construction (where they can act on it).
  • Out-of-scope deferral: the L1 grid path (ptr_app_grid()) internally constructs ptr_shared/ptr_shared_server in the same shape as the L2 host path, so Cut 2 propagates transparently to L1. No separate L1 work required.

Alternatives considered

  • Option B (Module-server-by-coincidence). Render panel sources inside the first ptr_server()'s moduleServer and re-expose them at top-level for the other instances. Rejected: breaks the "one widget, one owner" rule, makes order-of-ptr_server-call load-bearing (a property the call site doesn't visibly own), and disguises rather than solves the ownership question.
  • Option C (Panel moduleServer("shared", …)). Give the panel its own namespaced module so its outputs are at shared-…. Rejected: contradicts the locked CONTEXT.md claim that the shared panel uses global, un-namespaced ids (shared_<key>). Would force a documentation renegotiation and break embedders that hard-code top-level ids in custom CSS/JS.
  • Skip-only (Q3 Option I). Per-instance pipeline drops panel-owned source ids entirely, and the plot eval-env's parent lookup resolves them via a panel_eval_env on the bundle. Rejected: consumer pickers downstream then need a second code path ("is the column-source panel- or instance-owned? if panel, look there; if instance, look here"), splintering the consumer code. Substitute (Q3 Option II, chosen above) keeps state$resolved_sources as the single per-instance resolved-data answer.
  • Eval-env parenting (Q2 Option A original framing). Parent each ptr_server's state$eval_env from a host panel_eval_env. Rejected in favour of substitution because parenting introduces a new cross-server lexical edge (today there is no such edge) and doesn't propagate cleanly to the consumer-picker scope; substitution keeps the bundle as the single channel.
  • Narrow the contract — make ppUpload(shared=) a parse-time error. Rejected: contradicts three places in CONTEXT.md, partition rule loses generality, and the textbook multi-plot use case ("upload one dataset, drive two plots from it") becomes unsupported. Narrowing would shrink the feature for implementation-convenience reasons.
  • Cross-scope consumer with defined semantics (panel-owned consumer with formula-local sources, resolving via union / intersection / per-instance pickers). Rejected: no clean canonical semantics, and the "one widget, one source-of-truth" invariant is more valuable than the rare use case. Parse-time abort with a clear message is the right shape.

Worked examples — what becomes possible

All examples below assume ADR 0023 has landed (Cut 2). Each is runnable ggpaintr code; the marginalia explains why the example exercises a specific decision point above.

1. Two plots sharing one uploaded dataset (the textbook case)

The motivating Bug 1 reproducer. Source ds is referenced by both formulas → panel-owned. Consumer col is also referenced by both → panel-owned. One fileInput in the panel drives both plots; one column picker drives the x-axis on both.

library(ggpaintr); library(shiny)

panel <- ptr_shared(c(
  "ggplot(ppUpload(shared='ds'),
          aes(x = ppVar(shared='col'), y = ppVar(shared='col'))) +
     geom_point()",
  "ggplot(ppUpload(shared='ds'),
          aes(x = ppVar(shared='col'), y = ppVar(shared='col'))) +
     geom_col()"
))

shinyApp(
  ui = fluidPage(
    ptr_shared_panel(panel),    # one upload widget + one shared col picker
    fluidRow(
      column(6, ptr_ui(panel$formulas[[1]], "p1")),
      column(6, ptr_ui(panel$formulas[[2]], "p2"))
    )
  ),
  server = function(input, output, session) {
    s <- ptr_shared_server(panel)                       # populates panel_sources$shared_ds
    ptr_server(panel$formulas[[1]], "p1", shared_state = s)
    ptr_server(panel$formulas[[2]], "p2", shared_state = s)
  }
)

What the bundle carries: s$panel_sources$shared_ds is a reactive resolved df, populated by the host once the user uploads. Each ptr_server's pipeline substitutes state$resolved_sources$shared_ds from that bundle field instead of watching its own (non-existent) namespaced input — decision §4 above.

2. Same shape, with a default_arg so pickers populate at boot before any click

Couples ADR 0023's panel resolver with the eyeball-play try_bind_source_default fallback (commit a3d1636). At boot, the host resolver finds df_main via envir's parent chain and primes panel_sources$shared_ds with mtcars. Both column pickers populate immediately; no upload needed.

df_main <- mtcars

panel <- ptr_shared(c(
  "ggplot(ppUpload(df_main, shared='ds'),
          aes(x = ppVar(shared='col'), y = mpg)) + geom_point()",
  "ggplot(ppUpload(df_main, shared='ds'),
          aes(x = ppVar(shared='col'), y = hp))  + geom_col()"
))

# ... same UI / server wiring as example 1.

3. Mixed scope: panel-owned source, formula-local consumer

The source ds is referenced by both formulas → panel-owned (one widget in the panel). The consumers colA and colB are each used in only one formula → formula-local (each picker lives in its own instance's inline section). Per-instance pipeline substitutes the panel-resolved df into state$resolved_sources$shared_ds; the formula-local consumer reads from that state via the existing per-instance binder (decision §5 routes only the panel-owned consumer case to the host path — formula-local consumers stay on the per-instance path).

panel <- ptr_shared(c(
  "ggplot(ppUpload(shared='ds'),
          aes(x = ppVar(shared='colA'), y = mpg)) + geom_point()",
  "ggplot(ppUpload(shared='ds'),
          aes(x = ppVar(shared='colB'), y = hp))  + geom_col()"
))

# UI:    ptr_shared_panel(panel) +
#        ptr_ui(panel$formulas[[1]], 'p1') + ptr_ui(panel$formulas[[2]], 'p2')
# server: ptr_shared_server(panel)
#         ptr_server(panel$formulas[[i]], paste0('p', i), shared_state = s)

Panel holds just the upload (cross-formula); each instance's inline shared section holds its own column picker.

4. Single-instance shared source — already fixed by Cut 1 (included for the full shape)

One formula, no coordinator, every shared widget is owned by the single instance. ADR 0023 leaves this path untouched; panel_sources is absent from the bundle, the partition-aware skip at the per-instance source-UI loop (decision §7) is a no-op for this shape, and the existing per-instance pipeline path resolves the upload as it does today. This shape works on this branch's HEAD already — fixed by Cut 1 (commit d71893b), not pending Cut 2. Verified end-to-end: render the fileInput → user uploads a CSV → consumer picker populates with the uploaded df's columns.

ptr_app(
  ppUpload(shared = 'ds') |>
    ggplot(aes(x = ppVar(shared = 'a'), y = ppVar(shared = 'a'))) +
    geom_point()
)

5. Pipeline-head source with intermediate stages, shared across plots

The source is at the head of a |> chain that includes other stages. pipeline_source_ids (per R/paintr-server.R:166–170) catches it; the panel resolver populates panel_sources$shared_ds; per-instance pipelines substitute the resolved df into the chain head before evaluating downstream stages (dplyr::filter, mutate, etc.).

panel <- ptr_shared(c(
  "ppUpload(shared='ds') |> dplyr::filter(ppExpr(cyl > 4)) |>
     ggplot(aes(x = ppVar(shared='col'))) + geom_histogram()",
  "ppUpload(shared='ds') |> dplyr::mutate(z = ppExpr(mpg / wt)) |>
     ggplot(aes(x = ppVar(shared='col'), y = z)) + geom_point()"
))

6. Custom source keyword inherits panel-scope wiring for free

Because the factored resolve_upload_source() helper (decision §3) dispatches via the placeholder registry's resolve_data / companion_id_fn entries, any source keyword defined with ptr_define_placeholder_source() works at panel scope without further wiring.

ppDataChooser <- ptr_define_placeholder_source(
  keyword = "ppDataChooser",
  build_ui     = function(node, ...) shiny::selectInput(node$id, "Choose data", choices = c("mtcars","iris")),
  resolve_data = function(value, node, ...) get(value, envir = globalenv()),
  default_arg  = ptr_default_string()
)

panel <- ptr_shared(c(
  "ggplot(ppDataChooser(shared='ds'), aes(x = ppVar(shared='col'))) + geom_point()",
  "ggplot(ppDataChooser(shared='ds'), aes(x = ppVar(shared='col'))) + geom_col()"
))

7. Seed a panel-owned source via the companion id at boot

Spec routing for source placeholders is keyed off the companion input id (shared_<key>_name), not the fileInput id itself — the fileInput cannot accept a programmatic value (it lives in the browser's filesystem-picker). The companion path was landed by the existing apply_spec_at_boot() machinery (R/paintr-server.R:588–590: "placeholder rows (except ppUpload) and source_companion rows"); ADR 0023 does not introduce a new spec channel, it just makes that path reach panel-scope sources for free once panel_sources is wired.

# Owner-side: ship a saved view that boots straight into "df_main"
# without requiring the user to upload or type.
ptr_app(
  formula = ppUpload(shared='ds') |>
              ggplot(aes(x = ppVar(shared='col'))) + geom_point(),
  spec    = list(shared_ds_name = "df_main")   # seeds the companion textInput
)
# At boot: companion shows "df_main", `try_bind_source_default()` reads it,
# resolves `df_main` via state$eval_env's parent chain, primes
# state$resolved_sources, picker populates from `df_main`'s columns.

The same companion-id seed shape works for the panel scope: ptr_shared(...) + an embedder-supplied spec= through ptr_shared_server(spec = …) targets the top-level shared_<key>_name companion (one widget per panel-owned source key).

Worked examples — what is rejected by design

R1. Panel-owned consumer paired with formula-local source — parse-time abort

Decision §6 (parse-time invariant). Here col is referenced by both formulas → panel-owned; but each formula's source is a different bare-data dataset — formula-local. A single shared picker cannot list columns from two different datasets without an arbitrary choice (union? intersection? per-instance?). ADR 0023 rejects the configuration at ptr_shared() time:

ptr_shared(c(
  "ggplot(mtcars, aes(x = ppVar(shared='col'))) + geom_point()",
  "ggplot(iris,   aes(x = ppVar(shared='col'))) + geom_point()"
))
#> Error in `ptr_shared()`:
#> ! Shared consumer `col` is panel-owned (used in 2 formulas) but its
#>   upstream data sources are formula-local: `mtcars` (formula 1),
#>   `iris` (formula 2).
#> i Either annotate the sources as `shared = "..."` so they're also panel-owned,
#>   or drop `shared = "col"` so each plot gets its own picker.

R2. Different physical uploads for the same shared key — does not exist

Shared semantics is keyed to a canonical id, not to dataset value-equality. Two separate ppUpload(shared='ds') widgets do not exist in any configuration — by the rewrite, both occurrences of shared='ds' collapse to one widget with one canonical id. Users who upload "the same file" into two separate non-shared ppUpload() widgets will not get shared-consumer resolution; the user explicitly verified this when grilling Cut 1.

# Not a valid workaround for "shared dataset across plots":
ptr_app_grid(list(
  "ggplot(ppUpload(), aes(x = ppVar(shared='col'))) + geom_point()",  # one widget
  "ggplot(ppUpload(), aes(x = ppVar(shared='col'))) + geom_col()"     # another widget
))
# The two uploads are independent; the shared picker has no single source to
# read columns from -- it lands in the R1 case above and aborts at coordinator time.

R3. Spec-seeding the fileInput directly with a data frame — silently skipped (browser-driven)

The fileInput cannot be set programmatically — it carries a per-session tempfile path the browser owns. apply_spec_at_boot() explicitly skips ppUpload rows of role placeholder (R/paintr-server.R:574–575, 588–590) and the input-spec collector drops source_companion-classified rows from the companion-side round-trip (R/paintr-server.R:311–319) so the spec is honest about what survives. The supported seed path is the companion-id route shown in working example 6.

# Silently skipped at boot (not an error -- the spec is honest about
# what the browser cannot accept):
ptr_app(formula, spec = list(shared_ds = mtcars))

# DO this instead -- seed the companion, let the default-arg / eval_env
# fallback resolve the binding name to a frame:
df_main <- mtcars   # in the host script env
ptr_app(formula, spec = list(shared_ds_name = "df_main"))

R4. Programmatic upload through shared_state = escape hatch — same restriction

Embedders constructing shared_state manually can supply a panel_sources entry pointing at their own reactive — that's exactly how the bundle generalises. But the host's own renderUI for the panel widget will still emit a fileInput; the escape-hatch reactive simply overrides the resolved value. There is no path to replace the fileInput tag with something else without dropping the panel piece entirely and rendering your own widget.

# Works (manual override):
my_df <- shiny::reactiveVal(mtcars)
ptr_server(formula, "p1", shared_state = list(panel_sources = list(shared_ds = my_df)))

# Does NOT work (cannot replace the panel's emitted widget shape):
# -- if you want a custom widget, omit `ptr_shared_panel(panel)` from the UI
#    and render your own; supply `shared_state$panel_sources` manually.

R5. ptr_shared() with sources referenced by only one formula — they fall to inline sections, not the panel

Not strictly "rejected" but a frequent misconception. The partition rule applies to sources just like values/consumers: a source referenced by exactly one formula is formula-local, rendered in that instance's inline section, not in the panel. panel_sources for that key stays absent from the bundle.

panel <- ptr_shared(c(
  "ggplot(ppUpload(shared='ds_a'), aes(x = ppVar(mpg))) + geom_point()",  # only formula 1
  "ggplot(ppUpload(shared='ds_b'), aes(x = ppVar(mpg))) + geom_col()"     # only formula 2
))
# panel$panel_keys is empty for sources; ds_a renders inline under p1, ds_b under p2.
# `ptr_shared_panel(panel)` body is empty -- this is the "valid empty panel"
# case (CONTEXT.md flagged-ambiguity resolution, 2026-05-17).

Out of scope

  • L1 grid-specific wiring. ptr_app_grid() constructs the coordinator and the per-module servers in the same shape as L2; Cut 2 propagates transparently. If a grid-specific gotcha emerges during implementation, it joins the plan as a sub-item rather than a separate ADR.
  • Source widgets beyond ppUpload. Today's only source role placeholder is ppUpload. The factored resolve_upload_source() helper is keyword-aware (dispatches via registry), so a future custom source keyword (e.g. a ppDataChooser driven by a selectInput) inherits the panel-scope wiring for free. No work in this cut.
  • Panel-scope spec= seeding for panel-owned sources. Today ptr_app(spec = list(shared_<key> = …)) seeds shared value widgets at boot. Source widgets cannot be programmatically seeded with file content (fileInput is browser-driven); the companion shared_<key>_name textInput could be — but the seed shape needs its own design and is out of this cut. Deferred until a user surfaces a need.
  • Hard removal of the partition-blind skip story in test suites. Existing tests for the per-instance source path stay; the new panel-scope path gets its own browser fixture. No grand re-organisation.
  • ADR-0005 amendment. ADR-0005's silence on source-role behaviour is corrected by this ADR's existence. No formal amendment to 0005 is required; this ADR is referenced from CONTEXT.md's shared-coordinator section as the source-role addendum.
