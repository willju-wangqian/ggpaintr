ADR 0008 — No exprs placeholder; placeholders stay bounded

Status: rejected — decision made · Date: 2026-05-19 · Supersedes: nothing · Voids: nothing · Related: ADR 0005 / 0006 (L2/L3 redesign; single server entry) · Domain language: CONTEXT.md (no change) · Implementation: none — this ADR records a decision not to build a feature.

Decision: ggpaintr will not add an exprs (multi-line, structure-bearing, raw-code) placeholder. The existing expr placeholder — a single bounded R expression — stays as the maximum-freedom hole. This is a deliberate scope boundary, not a deferral.

Context

A 2026-05-19 discussion explored an exprs placeholder: a hole that takes multiple lines of arbitrary R and splices them into the formula at the token's structural position, so its content could add pipeline stages, add + layers, or supply the ggplot() base itself. Motivating sketches:

exprs |> ggplot(...)               # exprs = data-pipeline head
ggplot(...) + exprs + labs(...)    # exprs = several layers
exprs + labs(...)                  # exprs = pipeline + base + layers
mtcars |> exprs |> ggplot() + exprs # both, multiple holes

This started from a real bug report: a +-chain typed into the current expr placeholder (geom_line() + ggtitle("abc")) fails with "Cannot add <ggproto> objects together". That bug is real and tracked separately; it does not motivate exprs. Standalone geom_line() + ggtitle() is itself misuse of expr (an isolated ggproto chain with no plot base), so "generalize expr to absorb it" was explicitly rejected as the wrong fix.

Why exprs is the wrong direction

1. The reductio

Take exprs to its limit: one app, one exprs hole, user pastes arbitrary R, ggpaintr runs it. That is an R session wrapped in a Shiny app — a REPL with extra steps. It carries none of ggpaintr's value (a structured formula whose tokens become typed, validated, composable controls) and all of eval's liability. A feature whose extreme case is meaningless is a feature aimed in the wrong direction. ggpaintr's job is to turn a structured plot grammar into controls, not to host free-form code.

2. It breaks the typed-AST invariant

The post-rewrite pipeline parses the formula once into a typed tree; structure (root$layers, pipeline shape, data_arg) is fixed at translate time and placeholders are leaves within it. substitute_walk.ptr_root does a strict 1:1 in-place node replacement; substitute_walk.ptr_ph_value returns exactly one node; ptr_eval evaluates per layer and composes with Reduce(`+`). exprs would make structure depend on runtime input, forcing a pre-parse text-macro phase and a full re-translate — reintroducing exactly the parse-time/runtime entanglement the typed-AST rewrite removed.

3. Every realization needs unforced design forks

Forced questionWhy it is unforced today
Empty hole → ggplot() + + labs() invalidBounded placeholders prune cleanly; a structural hole needs bespoke dangling-operator pruning.
Placeholders nested inside exprs textWidgets are built before content exists → chicken-and-egg with no good v1 answer.
Position scope (pipeline vs. layer vs. aes() arg)Bounded placeholders have one well-defined slot; exprs has none.
Safety surfaceRaises the trust boundary to "arbitrary R structure," widening what P5 must screen for no product gain.

Multiple genuine forks with no obvious default is the signature of a feature that should not exist, not one that needs more design.

Decision

exprs is rejected. The freedom ceiling for a placeholder is the existing expr: a single, bounded R expression occupying one well-defined slot, screened by P5. Placeholders are inputs to a structured grammar; they are not a code-injection surface. Users who need arbitrary R should write the Shiny app directly (the L3 path, ADR 0006) — that is the supported escape hatch, and it is honest about being code.

Consequences

  • The "maximum freedom" principle is now recorded: ggpaintr deliberately does not maximize user freedom inside the formula DSL. Future "what if a placeholder could also do X structurally" proposals are answered by this ADR — route them to L3, not to a new placeholder.
  • The reported expr +-chain bug is unaffected by this decision and is handled on its own track (it is a misuse-reporting / error-message question, not a missing-feature one).
  • If this is ever revisited, supersede this ADR explicitly; do not add exprs by treating the rejection as a deferral.
Decision made (reject), 2026-05-19, at the maintainer's explicit request. Recorded so the question is not re-opened from scratch. Source claims (1:1 substitute_walk.ptr_root replacement, single-node ptr_ph_value return, per-layer ptr_eval + Reduce(`+`), expr single-expression contract) were verified against the source on 2026-05-19; re-verify before acting if revisiting.
