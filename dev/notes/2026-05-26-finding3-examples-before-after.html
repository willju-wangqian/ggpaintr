Correction (2026-05-26, this update) — example A's "always worked" claim was wrong. The original v1 of this doc framed Example A (non-shared ppUpload(df_main) + non-shared ppVar('hp')) as the negative-baseline that "always worked." A user pasted that exact code and reported the picker booting empty. Reproduction with shinytest2 confirmed: the picker's options list IS populated with names(df_main), but the default 'hp' never seeds the selection — the dropdown shows the placeholder "Choose one column" instead of pre-selecting hp. Cause: substitute_walk.ptr_ph_data_source() reads the upload companion's value from ctx$snapshot[[node$companion_id]], which is briefly NULL post-boot before the client reports the widget's value= back. The consumer's first entry_reactive() fire therefore returns NULL; the renderUI flipped has_rendered on that empty fire, locking the next valid fire into the seed %||% current %||% character(0) branch — extra$selected = character(0) suppressed invoke_build_ui()'s node$default injection (paintr-build-ui.R:842-855), so the default was permanently lost. Fix: gate the has_rendered <<- TRUE flip on length(cols) > 0L, so the first populated render is the one that flips the flag. Example A's verdict tables below are corrected; the diagnose loop that found this is a textbook "matrix-classification ≠ runtime pin" repeat of FINDING #8's v4→v7 escalation and the original FINDING #3 escalation.

Bug + fix in one paragraph

A non-shared ppVar(...) consumer downstream of a panel-owned ppUpload(shared = "k") booted with an empty picker and refused to draw. The consumer-side renderUI in ptr_setup_consumer_uis() (R/paintr-server.R) built its snapshot for ptr_resolve_upstream() by reading input[[ns(cmp)]] — per-instance namespaced. For panel-owned sources the companion textInput lives at HOST scope (un-namespaced; e.g. shared_ds_name with no p1- prefix), so the per-instance lookup is NULL. substitute_walk.ptr_ph_data_source() (R/paintr-substitute.R:184-188) therefore returns ptr_missing() → ptr_prune() strips the upstream → ptr_resolve_upstream() returns NULL → the picker dispatches with zero options. The fix pre-computes the set of upstream source companion/self ids that belong to a panel-owned source and routes those reads to session$rootScope()$input[[bare-id]] instead. Shared-section consumers (which use ptr_bind_shared_consumer_uis(), a different binder that never goes through ptr_substitute) were never affected, which is why the bug hid behind WE#3's mixed-scope coverage for so long.

Where it broke pre-fix. The consumer's first entry_reactive() fire returned NULL because substitute_walk.ptr_ph_data_source() read the companion's value from the input snapshot, which was briefly NULL post-boot. That fire dispatched a renderUI with cols = character(), and the post-fire has_rendered <<- TRUE flip locked the next valid fire (with proper cols) into the has_rendered=TRUE branch — selected_arg = seed %||% current %||% character(0) = character(0) — which set extra$selected = character(0) and suppressed invoke_build_ui()'s node$default injection (R/paintr-build-ui.R:842-855). Net: any ppVar('col') default was permanently lost as soon as the upstream included a ppUpload(df). Where it works post-fix. The has_rendered <<- TRUE flip is now gated on length(cols) > 0L (R/paintr-server.R, non-shared consumer renderUI). The first populated render is the one that flips the flag, so node$default still seeds via the has_rendered=FALSE branch on the first non-empty fire. Explicit user deselect semantics are unchanged: the deselect path writes character(0) into the input, so current = character(0), seed %||% current %||% character(0) = character(0) wins over default-injection — verified by the existing test-consumer-deselect-stays-empty.R and test-stale-pick-no-snap-to-default.R.

What changed in source

One self-contained edit inside ptr_setup_consumer_uis() in R/paintr-server.R:

  1. Just after the existing upstream_source_self_ids derivation, compute upstream_panel_companion_ids + upstream_panel_self_ids by walking node$upstream for source nodes whose $id is in names(state$panel_sources).
  2. In the two snapshot-building loops (companions, self-ids), route the read for any id in those sets to session$rootScope()$input[[bare-id]] instead of input[[ns(id)]].

For non-panel apps both sets are empty; the if-branch never fires; behaviour is bit-for-bit unchanged. This is why the fix is safe under the documented sampling-race flake on shared-state tests — the only behavioural delta the fix introduces is gated on state$panel_sources being non-empty.
