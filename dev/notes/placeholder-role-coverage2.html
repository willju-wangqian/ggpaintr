Method

The matrix below enumerates the 3 (scope) × 3 (role) × 2 (default) × 2 (spec= seeding) cells implied by ggpaintr's placeholder model. For each cell I trace the actual server-side binder code in R/paintr-server.R, R/paintr-shared-ui.R, R/paintr-shared-coordinator.R, and R/paintr-app.R; locate the default-handling site and the state$spec_seed read (if any); and grep tests/testthat/ for direct exercise of the cell. Read-only survey — no source files were edited.

Scope vocabulary (from CONTEXT.md lines 36-49 and ADR 0023):

  • non-shared — no shared = arg. Widget is per-instance, namespaced under <modId>-<auto-id>.
  • shared-section — shared = "k" but key appears in exactly one formula. shared_partition() assigns it to local_keys_by_formula; widget renders inside that formula's inline shared section, server-bound by that instance's own ptr_server() via the auto_bind_shared path (R/paintr-app.R:1117) plus ptr_bind_local_shared_consumers (R/paintr-app.R:1199).
  • shared-panel — shared = "k" AND key appears in ≥2 formulas. shared_partition() assigns to panel_keys; rendered once at host scope by shared_panel_body_tag() (R/paintr-shared-coordinator.R:316) under un-namespaced ids shared_<k>.

Role taxonomy: consumer (needs upstream df — ppVar built-in, is_ptr_ph_data_consumer node class), source (IS a df — ppUpload built-in, is_ptr_ph_data_source), value (scalar/expression no upstream — ppText / ppNum / ppExpr, is_ptr_ph_value). The five built-ins are ppText / ppNum / ppExpr / ppVar / ppUpload (registrations in R/paintr-builtins.R:337-381). Custom keywords are registered via ptr_define_placeholder_value / _consumer / _source.

Spec seeding dimension (4th axis, new in v2). The caller can pass spec = list(<id> = <value>) to ptr_app() / ptr_server(); apply_spec_at_boot() (R/paintr-server.R:511-657) filters entries by state$server_ns_fn(""), normalizes value shape per built-in keyword, and writes them to state$spec_seed[[bare_id]] synchronously before the first flush. Each renderUI body that participates in seeding reads shiny::isolate(state$spec_seed[[raw_id]]) on first fire and threads it into extra$selected via invoke_build_ui. Spec wins over registry/node$default at boot.

ppUpload + spec= routing (clarification, v3). The spec entry for a ppUpload source is NOT aimed at the fileInput (a fileInput cannot accept a programmatic value — see comment at R/paintr-server.R:582-584). It IS aimed at the bare-name companion textInput beside the file widget. apply_spec_at_boot() writes state$spec_seed[[<companion-id>]] for source_companion rows (R/paintr-server.R:597-599 + R/paintr-server.R:631), and the dispatch policy at R/paintr-server.R:669-674 defines source_companion → updateTextInput and keyword == "ppUpload" → silent skip (fileInput is not programmatically settable; its companion is reached via the source_companion branch). Downstream consumers resolve the data frame via the companion-named symbol bound into state$eval_env by the source observer's resolve_upload_source() / try_bind_source_default_resolved() chain (R/paintr-server.R:994-1084). The panel-side observer at R/paintr-shared-ui.R:541-554 reads input[[comp_id]] reactively inside a shiny::observe({...}), so the companion value (whether seeded by spec at boot or typed by the user later) re-invalidates the bind. So the design is: spec for ppUpload sources routes via the companion, and the panel-source binder consumes it reactively — no direct state$spec_seed read in ptr_setup_panel_sources() is needed. (See FINDING #8 for a v4 challenge to this conclusion.)

Source-companion values are coerced to character at the seed-write site (R/paintr-server.R:603-606): empty or NULL becomes "", otherwise as.character(value)[[1L]]. The companion id pattern for ppUpload is <source-id>_name (e.g. ggplot_1_ppUpload_NA_name; pinned in test-id-table.R rows; test-rewrite-input-spec.R:17 asserts upload_rows$role == c("placeholder", "source_companion")). For shared-section / shared-panel ppUpload(shared='ds'), the canonical id is shared_<k> and the companion id is shared_<k>_name (derived via entry$companion_id_fn in ptr_setup_panel_sources at R/paintr-shared-ui.R:478-480).

By-design notes (v3)

  • ppUpload + spec= routing is companion-aimed, not fileInput-aimed. The spec target for a ppUpload source (at any scope) is the bare-name companion textInput beside the file widget, NOT the fileInput itself. The fileInput cannot accept a programmatic value; this is recorded in source at R/paintr-server.R:582-584 and reinforced by the dispatch policy at R/paintr-server.R:669-674 (keyword == "ppUpload" → silent skip; source_companion → updateTextInput). apply_spec_at_boot() writes state$spec_seed[[<companion-id>]] for the source_companion row at R/paintr-server.R:597-599,631, with character coercion at L603-606. This is the documented API surface and is the SUFFICIENT spec= mechanism for source roles. Companion id pattern: <source-id>_name non-shared; shared_<k>_name shared (per entry$companion_id_fn at R/paintr-shared-ui.R:478-480).
