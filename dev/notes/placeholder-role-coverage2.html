Method

The matrix below enumerates the 3 (scope) × 3 (role) × 2 (default) × 2 (spec= seeding) cells implied by ggpaintr's placeholder model. For each cell I trace the actual server-side binder code in R/paintr-server.R, R/paintr-shared-ui.R, R/paintr-shared-coordinator.R, and R/paintr-app.R; locate the default-handling site and the state$spec_seed read (if any); and grep tests/testthat/ for direct exercise of the cell. Read-only survey — no source files were edited.

Scope vocabulary (from CONTEXT.md lines 36-49 and ADR 0023):

  • non-shared — no shared = arg. Widget is per-instance, namespaced under <modId>-<auto-id>.
  • shared-section — shared = "k" but key appears in exactly one formula. shared_partition() assigns it to local_keys_by_formula; widget renders inside that formula's inline shared section, server-bound by that instance's own ptr_server() via the auto_bind_shared path (R/paintr-app.R:1117) plus ptr_bind_local_shared_consumers (R/paintr-app.R:1199).
  • shared-panel — shared = "k" AND key appears in ≥2 formulas. shared_partition() assigns to panel_keys; rendered once at host scope by shared_panel_body_tag() (R/paintr-shared-coordinator.R:316) under un-namespaced ids shared_<k>.

Role taxonomy: consumer (needs upstream df — ppVar built-in, is_ptr_ph_data_consumer node class), source (IS a df — ppUpload built-in, is_ptr_ph_data_source), value (scalar/expression no upstream — ppText / ppNum / ppExpr, is_ptr_ph_value). The five built-ins are ppText / ppNum / ppExpr / ppVar / ppUpload (registrations in R/paintr-builtins.R:337-381). Custom keywords are registered via ptr_define_placeholder_value / _consumer / _source.

Spec seeding dimension (4th axis, new in v2). The caller can pass spec = list(<id> = <value>) to ptr_app() / ptr_server(); apply_spec_at_boot() (R/paintr-server.R:511-657) filters entries by state$server_ns_fn(""), normalizes value shape per built-in keyword, and writes them to state$spec_seed[[bare_id]] synchronously before the first flush. Each renderUI body that participates in seeding reads shiny::isolate(state$spec_seed[[raw_id]]) on first fire and threads it into extra$selected via invoke_build_ui. Spec wins over registry/node$default at boot.

ppUpload + spec= routing (clarification, v3). The spec entry for a ppUpload source is NOT aimed at the fileInput (a fileInput cannot accept a programmatic value — see comment at R/paintr-server.R:582-584). It IS aimed at the bare-name companion textInput beside the file widget. apply_spec_at_boot() writes state$spec_seed[[<companion-id>]] for source_companion rows (R/paintr-server.R:597-599 + R/paintr-server.R:631), and the dispatch policy at R/paintr-server.R:669-674 defines source_companion → updateTextInput and keyword == "ppUpload" → silent skip (fileInput is not programmatically settable; its companion is reached via the source_companion branch). Downstream consumers resolve the data frame via the companion-named symbol bound into state$eval_env by the source observer's resolve_upload_source() / try_bind_source_default_resolved() chain (R/paintr-server.R:994-1084). The panel-side observer at R/paintr-shared-ui.R:541-554 reads input[[comp_id]] reactively inside a shiny::observe({...}), so the companion value (whether seeded by spec at boot or typed by the user later) re-invalidates the bind. So the design is: spec for ppUpload sources routes via the companion, and the panel-source binder consumes it reactively — no direct state$spec_seed read in ptr_setup_panel_sources() is needed. (See FINDING #8 for a v4 challenge to this conclusion.)

Source-companion values are coerced to character at the seed-write site (R/paintr-server.R:603-606): empty or NULL becomes "", otherwise as.character(value)[[1L]]. The companion id pattern for ppUpload is <source-id>_name (e.g. ggplot_1_ppUpload_NA_name; pinned in test-id-table.R rows; test-rewrite-input-spec.R:17 asserts upload_rows$role == c("placeholder", "source_companion")). For shared-section / shared-panel ppUpload(shared='ds'), the canonical id is shared_<k> and the companion id is shared_<k>_name (derived via entry$companion_id_fn in ptr_setup_panel_sources at R/paintr-shared-ui.R:478-480).

Findings

  • FINDING #1 (minor, expanded in v2): host-scope shared-panel consumer cannot read spec-seed. ptr_bind_shared_consumer_uis() at R/paintr-server.R:2405-2407 reads seed <- if (!is.null(state)) shiny::isolate(state$spec_seed[[rep_node$id]]) else NULL. When invoked from ptr_shared_server() (host scope), state is NULL by design (ADR 0006). Consequence: an embedder who passes spec = list(shared_col = "mpg") on a per-instance ptr_server() won't seed the panel-owned consumer at the host slot — only at their own instance's namespaced slot (which then doesn't drive the panel widget because the panel widget reads from input[[shared_col]], not from input[[<mod>-shared_col]]). Workaround: rely on shared_widget_default() (first-occurrence default in the formula source). Fix sketch: at host scope, thread shared_state$spec_seed into the binder — or have ptr_make_app_server publish a host-level spec_seed reactive that the panel-consumer renderUI consults independently of state. Defect candidate.
  • FINDING #2 (architectural): shared-panel VALUES bypass the host renderUI path. There is no ptr_setup_panel_values() helper analogous to ptr_setup_panel_sources(). For a panel value key (e.g. ppNum(shared = "k") referenced in ≥2 formulas), shared_panel_body_tag() emits a static tag with the first-occurrence default baked in via shared_widget_default() (R/paintr-shared-coordinator.R:341-348). The host server's shared_reactives reads input[[shared_<k>]] at top-level scope (L290-309). Per-instance UI doesn't emit a placeholder tag for cross-formula shared values either. Net effect: the panel widget is a one-time static tag — it works for boot-time defaults and reactive reads, but cannot benefit from later renderUI features (label retitling, dynamic copy, spec-seed at host scope — see FINDING #7). Document in CONTEXT.md or build the missing ptr_setup_panel_values() for symmetry.
  • FINDING #3 (test gap): shared-panel source + non-shared consumer composite is not exercised directly. None of the panel-source tests pair the panel-owned ppUpload(shared='ds') with a downstream plain (non-shared) ppVar(...) consumer. WE#3 (test-shared-source-panel-multi-instance.R:260) is the closest but uses shared-section consumers. The substitution code path is exercised by test-pipeline-panel-source-substitution.R, but no end-to-end test asserts that a non-shared ppVar downstream of ppUpload(shared='ds') populates correctly when the panel resolver fires.
  • FINDING #4 (by-design, document): the parse-time panel-consumer/panel-source invariant. shared_assert_panel_consumer_sources() rejects two composites at ptr_shared() time: (a) panel-owned consumer with formula-local source, (b) panel-owned consumer with non-shared source. ADR 0023 §6 explains the rationale (one widget, one source-of-truth). Error class ptr_panel_consumer_source_mismatch exposes structured fields.
  • FINDING #5 (minor): single-formula ppUpload(shared='ds') renders. ADR 0023 Cut 1 (commit d71893b) dropped a partition-blind skip. The current ptr_setup_source_uis() skip at R/paintr-server.R:1872 is partition-aware (s$id %in% names(state$panel_sources %||% list())); single-instance has empty panel_sources so the source renders per-instance. Direct test: test-shared-source-rendering.R S-S.1. Worth keeping in the regression net — the easy regression is "make the skip partition-blind again".
  • FINDING #6 (reframed in v3): ptr_setup_panel_sources() does NOT read state$spec_seed, and that is by design. The panel-side renderer at R/paintr-shared-ui.R:486-522 reads node$default into extra_named$selected at L521-522 and does not consult a spec-seed map. The synthetic host_state built at L530-545 carries only eval_env / bound_names / resolve_errors. Previously (v2) this was flagged as a defect-candidate symmetry break. Reframed (v3): spec for ppUpload sources is routed via the companion textInput, which the source observer reads REACTIVELY (R/paintr-shared-ui.R:541-554). The companion value lands first (seed write at R/paintr-server.R:631); the observer invalidates; try_bind_source_default_resolved() binds the named frame. A direct state$spec_seed read in ptr_setup_panel_sources() would be redundant for ppUpload (the fileInput is non-programmatic) and is unnecessary for custom panel-source keywords that accept a programmatic value because the same updateTextInput / seed-read companion mechanism applies. Residual concern: a custom source keyword that lacks a companion AND accepts a programmatic value at its primary slot (i.e., uses neither file upload nor a companion textInput) would not see the panel-side seed. No such custom source exists in the built-ins; document if/when one is registered. v4 caveat: see FINDING #8 — the v3 reframing assumes the companion's initial render picks up the seeded value, which has not been empirically verified.
  • FINDING #7 (defect-candidate; unchanged from v2): shared-panel value path silently drops spec= entries. Direct corollary of FINDING #2 + the spec dimension. apply_spec_at_boot writes state$spec_seed[["shared_<k>"]] for a panel-owned value (no scope filter in the seed-write loop — only the ppUpload keyword exclusion). But the panel-slot widget is a static tag emitted by shared_panel_body_tag()'s else branch (R/paintr-shared-coordinator.R:341-348) which uses shared_widget_default() only — no renderUI, no state$spec_seed read. The per-instance shared-value loop's spec_seed read at R/paintr-server.R:1815 fires for namespaced uiOutputs that don't exist for panel-owned keys (the UI body skips host-owned keys). Net: an embedder passing spec = list(shared_lw = 2) for a ppNum(shared = "lw") that lives in ≥2 formulas (panel scope) sees the widget boot at its shared_widget_default() value, not 2. Fix sketch: a host-scope ptr_setup_panel_values() that registers a renderUI for each panel value key and reads state$spec_seed (parallel to the source-side fix in FINDING #1) — or, less invasively, convert the static tag emit into a host-scope updateXyzInput() call inside an onFlushed that consults state$spec_seed. Defect candidate.
  • RESOLVED FINDING #8 (v4 INVESTIGATE → v7 RESOLVED): spec→companion routing at boot was indeed broken; now fixed. The v4 prediction was correct: apply_spec_at_boot wrote state$spec_seed[[<source-companion-id>]] AND marked the row in seeded, but no reader existed for state$spec_seed[[<comp-id>]] anywhere in the server tree (the five seed reads at R/paintr-server.R:1745, 1815, 1881, 2107, 2406 all use raw_id = node$id — the source/value/consumer id, never the companion id; ptr_builtin_upload_build_ui at R/paintr-builtins.R:228-250 hard-codes the companion's value= to node$default %||% ""). Companion textInput first-rendered blank regardless of spec=. Fix A applied: exclude source_companion rows from seeded at R/paintr-server.R:631-643 so they fall through to the existing onFlushed updateTextInput dispatch in apply_spec_entry() at R/paintr-server.R:691-695. Regression test: tests/testthat/test-finding8-source-companion-spec.R + fixture tests/testthat/fixtures/vignette-apps/finding8-source-companion-spec/app.R. Bonus fixes discovered during the diagnose loop and landed alongside: (a) the is.null(node$default) guard in try_bind_source_default[_resolved] was blocking the downstream env-lookup path even when the companion (post-Fix-A) carried a valid name — lifted by ADR 0024 (companion as data-loading entry point) with a structured error surface via set_resolve_error; (b) latent self-invalidation in set_resolve_error — its cur <- state$resolve_errors() read created an undeclared reactive dependency on its own write target, masked pre-ADR-0024 by every caller writing only idempotent clears, surfaced the moment ADR 0024's structured-error write fired from inside the resolve observer; fixed by wrapping the read in shiny::isolate() at R/paintr-server.R:847-862. The three source×with-spec cells in the matrix stay at YES (the v3 reclassification was correct; v7 reclassification = "verified runtime-side, not just structurally"). FINDING #6's v3 reframing is preserved. Authoritative gate: PASS 3093 / FAIL 0 / SKIP 0 on the FINDING #8 + ADR 0024 commit.

By-design notes (v3)

  • ppUpload + spec= routing is companion-aimed, not fileInput-aimed. The spec target for a ppUpload source (at any scope) is the bare-name companion textInput beside the file widget, NOT the fileInput itself. The fileInput cannot accept a programmatic value; this is recorded in source at R/paintr-server.R:582-584 and reinforced by the dispatch policy at R/paintr-server.R:669-674 (keyword == "ppUpload" → silent skip; source_companion → updateTextInput). apply_spec_at_boot() writes state$spec_seed[[<companion-id>]] for the source_companion row at R/paintr-server.R:597-599,631, with character coercion at L603-606. Downstream consumers resolve the data frame via the companion-named symbol bound into state$eval_env by resolve_upload_source() / try_bind_source_default_resolved() (R/paintr-server.R:994-1084). The panel-source observer at R/paintr-shared-ui.R:528-554 reads input[[comp_id]] reactively, so the companion's seeded/typed value propagates to the bind without any direct state$spec_seed read in ptr_setup_panel_sources(). This is the documented API surface and is the SUFFICIENT spec= mechanism for source roles. Companion id pattern: <source-id>_name non-shared; shared_<k>_name shared (per entry$companion_id_fn at R/paintr-shared-ui.R:478-480). Verified (v7): FINDING #8's investigation confirmed the boot-render path was broken (companion textInput first-rendered blank despite a valid spec= entry) and the fix landed (see FINDING #8 entry above). The companion-routing design as described here now holds end-to-end with regression tests pinning both Fix A and the ADR 0024 entry-point lift.

FINDING #1 — host-scope shared-panel consumer can't read spec_seed [verdict: PARTIAL]

Expected: the panel-shared consumer picker #shared_col boots with "mpg" selected, because the embedder seeded it via spec = list(shared_col = "mpg").

Actual: picker boots from shared_widget_default() (first-occurrence default in the formula, or blank if no default); the spec entry is silently dropped at host scope.

library(ggpaintr)
library(shiny)

df_main <- mtcars

plots <- c(
  "ppUpload(df_main, shared = 'ds') |> ggplot(aes(x = ppVar(shared = 'col'), y = mpg)) + geom_point()",
  "ppUpload(df_main, shared = 'ds') |> ggplot(aes(x = ppVar(shared = 'col'), y = hp))  + geom_col()"
)
panel <- ptr_shared(plots)

ui <- fluidPage(
  ptr_shared_panel(panel),
  fluidRow(
    column(6, ptr_ui(panel$formulas[[1]], "p1", shared = panel)),
    column(6, ptr_ui(panel$formulas[[2]], "p2", shared = panel))
  )
)
server <- function(input, output, session) {
  s <- ptr_shared_server(panel, envir = environment())
  # `spec=` IS valid on the per-instance ptr_server() (signature at
  # R/paintr-app.R:1082-1083) -- NOT on ptr_shared_server() (signature at
  # R/paintr-shared-ui.R:222-225 has no spec= arg). Pass the host-scope panel
  # consumer seed on a per-instance ptr_server(), expecting it to propagate
  # to the host-scope panel widget:
  ptr_server(panel$formulas[[1]], "p1", shared_state = s,
             spec = list(shared_col = "mpg"))   # <-- silently dropped by host
  ptr_server(panel$formulas[[2]], "p2", shared_state = s)
}
shinyApp(ui, server)

Why it fails: the spec= arg IS valid on per-instance ptr_server() (signature: ptr_server <- function(formula, id = NULL, envir = parent.frame(), ..., shared_state = NULL, spec = NULL) at R/paintr-app.R:1082-1083). It is NOT a parameter of ptr_shared_server() (signature: ptr_shared_server <- function(obj, envir = parent.frame(), shared = list(), draw_trigger = NULL) at R/paintr-shared-ui.R:222-225). The real gap is on the read side at host scope: ptr_bind_shared_consumer_uis() at R/paintr-server.R:2405-2407 reads seed <- if (!is.null(state)) shiny::isolate(state$spec_seed[[rep_node$id]]) else NULL. At host scope state is NULL by design (ADR 0006); the seed branch returns NULL — so even though the per-instance ptr_server() wrote the seed into its OWN state$spec_seed, the host-scope renderUI for the panel widget never reads it. The panel widget binds to input[[shared_col]], not input[[<mod>-shared_col]], so the per-instance seed cannot drive it either.

Workaround today: rely on shared_widget_default() — put ppVar(default = "mpg", shared = "col") on the first occurrence in the formula source. This requires editing the formula, not the embedder's spec entry, defeating spec's purpose.
