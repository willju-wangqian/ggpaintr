Super-app eyeball & spec= round-trip checklist

The 10 super-app blocks in tests/testthat/test-super-pressure.R (5 supers × {with-default, no-default}) drive every built-in placeholder and seven custom-registered ones via shinytest2 + Chrome. They assert sentinel propagation, B3 toggle differential, picker scope discrimination, validate_input rejection, denylist rejection, resourcePath hygiene, and plotly-backend rendering. They do not compare pixels, drive plotly interactivity, exercise the real OS file picker, or close the loop on spec= round-trip from the code panel back into a fresh ptr_app() boot. This note is the human-side checklist for those gaps.

1. What the gate covers (recap)

Authoritative gate: NOT_CRAN=true Rscript -e 'devtools::load_all("."); testthat::test_dir(...)' (or devtools::test(), which sets NOT_CRAN=true itself). Expected: FAIL 0 / ERROR 0 / SKIP 0 on a machine with shinytest2 + chromote + Chrome. Under R CMD check the suite cleanly all-skips via the source-root guard.

Boot each super interactively. The playable copies live at dev/scripts/super-examples/<slug>/ — first line of each app.R is the only difference from the fixture (the pkgload::load_all(Sys.getenv("GGP_PKG")) scaffolding is replaced with a one-line comment, because in interactive play ggpaintr is already loaded).

devtools::load_all(".")
shiny::runApp("dev/scripts/super-examples/super-1-kitchen-sink")
# To run the no-default variant of any super:
shiny::runApp(shiny::shinyAppFile(

3. spec= round-trip (the missing closed loop)

Background. Preserve-mode code panel emits a trailing ptr_spec <- list(\`id\` = value, ...) beneath the formula text. ptr_app(formula, spec = ptr_spec) applies that list at session boot (via apply_spec_at_boot). Both halves are unit-tested independently — but no test in the tree currently closes the loop: takes the literal text the panel rendered, parses + evals it, and feeds the result into a fresh ptr_app() with an assertion that the second boot's widget snapshot equals the first. If the deparser and the apply-side ever drift on quoting / backtick / list-ordering / type rules, the unit tests stay green while the user-visible copy-paste workflow breaks silently.

For each super with non-trivial picks, run this six-step recipe:

  1. Boot via shiny::runApp("dev/scripts/super-examples/<slug>").
  2. Drive widgets to non-default values (one custom keyword + at least one built-in).
  3. Click Update.
  4. Toggle code mode to preserve.
  5. Copy the entire #ptr_code contents — both the formula block AND the trailing ptr_spec <- list(...).
  6. In a fresh R session, paste the text and run: 
    library(ggpaintr)  # or devtools::load_all("<repo>")
<the formula text from the panel, assigned to `f`>
<the ptr_spec block from the panel>
shiny::runApp(ptr_app(f, spec = ptr_spec))

Pass condition: the second boot's widgets show the same picks at first paint, the first draw shows the same plot — without clicking Update first. Fail conditions to watch for:

  • An entry in the spec is silently dropped (warning in console: Skipped N spec entries).
  • A custom-keyword spec entry doesn't seed its widget (build_ui hook may not be honoring selected).
  • Quoting drift between deparser and apply side: e.g. panel emits c('a', 'b') but apply rejects it as a non-named scalar.
  • Picker scope race: spec value names a column that hasn't been seeded yet (consumer pickers under suspended subtabs).
  • For super-2a / super-2b: ppUpload companion-name entries in the spec won't round-trip the underlying file — the user still has to re-upload. Confirm the bareword companion-name (df_main etc.) is at least emitted into the formula's data = slot in the panel text.

Recommended super order for round-trip (highest custom-keyword pressure first):

  1. super-2b — ppCoef + ppFactor + ppSample in one formula (three custom keywords at once).
  2. super-2a — ppPower + ppMultiVar; ppUpload companion-name behavior under round-trip is unique to this app.
  3. super-1 — ppRange + four built-ins; baseline-shape round-trip.
  4. super-4 — ppColor + ppExpr; user.css path is orthogonal but worth seeing the colored widget at second boot.
  5. super-3 — stretch. Two L3 cells under ptr_ui / ptr_server (no ptr_app_grid) means each cell emits its own ptr_spec block with namespaced ids (plot1-..., plot2-...). The combined-spec contract is ptr_spec_combine() + flat-spec dispatch; manually combining the two emitted blocks into one list() and passing it to both ptr_servers is the round-trip shape.

4. What to capture if anything fails

  • The exact input_id you set and the value you set it to (DOM ids are visible via browser dev-tools).
  • Screenshot of: #ptr_plot, #ptr_code contents, #ptr_error contents.
  • Console R messages from the browser tab and the parent R session.
  • For round-trip failures: the literal copied text (verbatim) and the first boot's widget snapshot you were trying to reproduce.

A round-trip failure is a real defect the gate cannot catch today; once you have one in hand, the test to add is straightforward — drive the same widget sequence under testServer, read output$ptr_code in preserve mode, parse the trailing ptr_spec block, feed it as spec= to a second ptr_server_internal boot, and assert the input map equals the first snapshot.
