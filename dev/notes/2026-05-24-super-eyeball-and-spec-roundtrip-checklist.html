Super-app eyeball checklist

Subjective-only human checks against the 5 super-app fixtures — the residue the automated gate provably cannot make. Each surviving recipe has a verified action sequence (renders cleanly through eval(formula) |> ggplot_build() |> ggplot_gtable()) and cites the gate assertion that proves its widget-by-ID pre-condition is reachable. Everything not on this list moved into the gate or was dropped.

Rule A — what this doc is. If a check can be expressed as a test assertion, it belongs in tests/testthat/, not here. This file lists only judgments that require a human's eyes: snappy / no flicker / synchronized / visually matches the rule / reads like a non-developer can understand. Spec round-trip is no longer in this doc — the closed loop runs in tests/testthat/test-spec-roundtrip-closed-loop.R.

2. Per-super eyeball checks

Boot each super interactively. Playable copies live at dev/scripts/super-examples/<slug>/ — first line of each app.R is the only difference from the fixture (the pkgload::load_all(Sys.getenv("GGP_PKG")) scaffolding is replaced by a one-line comment because ggpaintr is already loaded interactively).

# From the repo root:
devtools::load_all(".")
shiny::runApp("dev/scripts/super-examples/super-1-kitchen-sink")
# To run the no-default variant:
shiny::runApp(shiny::shinyAppFile(
  "dev/scripts/super-examples/super-1-kitchen-sink/app-no-default.R"))

super-1 kitchen-sink

Recipe A — bulk widget update → snappy redraw? (~2 min)
  1. Open the ggplot layer's Controls subtab. You'll see two ppExpr widgets: one for dplyr::filter(...) (expects a logical) and one for dplyr::mutate(adj = ...) (any numeric).
  2. Set the mutate ppExpr (the one feeding adj =) to mpg + wt. Leave the filter ppExpr at its default hp >= 75.
  3. Set aes(x) to disp, shared_grp to gear, labs(title) to S_T_2718, ppNum size to 0.7314159.
  4. Click Update. Plot redraws — does it feel snappy? Any visible blink / clear-then-redraw?
Trap if you stray. Setting the filter ppExpr to a non-logical (e.g. hp + 1) errors from dplyr::filter. Setting the mutate ppExpr to a value that pushes adj out of the formula's scale_y_continuous(limits = c(0, 50)) (e.g. hp + 1 → range 53–336) drops every row from geom_smooth, and loess on empty data hard-errors from seq_len(). The recipe above stays inside the limits and was verified through eval(formula) |> ggplot_build() |> ggplot_gtable().

Gate cite: tests/testthat/test-super-pressure.R:26-101 (super-1 with-default) drives the same widget IDs to the same values; expect_no_plot_error("ptr_plot") at L80 proves the host has no Shiny error class after draw.

Recipe B — rapid code-mode toggle → no flicker, no stale fragment? (~30 s)
  1. After Recipe A's draw, find the code-mode radio (final / preserve) above the code panel.
  2. Toggle final ↔ preserve ↔ final ↔ preserve three times as fast as you can click.
  3. Watch the code panel during the toggles — does it ever flash the wrong text, or leave a stale fragment from the previous mode? The text should switch atomically.

Gate cite: test-super-pressure.R:84 + L100 drive the toggle once and assert preserve-mode text shape; the gate does not stress-test rapid toggling.

super-2a upload-registry

Recipe A — real OS file picker round-trip (~3 min)
  1. Upload dev/scripts/super-examples/super-2a-upload-registry/sample_main.csv via the root ppUpload fileInput (the real OS picker dialog, not a path string).
  2. Upload sample_aux.csv via the geom_smooth fileInput.
  3. Click Update — plot renders without an error pane? (The gate uses shinytest2's fileInput shim, which sets datapath directly. This recipe exercises the actual browser file picker integration the shim simulates.)

Gate cite: test-super-pressure.R:161-339 drives identical widget IDs with shinytest2's shim; this eyeball covers only the human-driven picker piece.

Recipe B — ppPower out-of-range → error wording reads human? (~30 s)
  1. After Recipe A, find the geom_point alpha widget (ppPower, slider 0–1).
  2. Set it to 1.5 (out-of-range) via the slider's numeric input.
  3. Click Update. Read the message in the error pane out loud — does it sound like something a non-developer would understand? The plot should still show Recipe A's render (B3 prior-render retention).

Gate cite: test-super-pressure.R:307-326 pins the literal must be in [0,1] in #ptr_error. Eyeball judges readability, not the literal.

super-2b customsource-splice

Recipe A — ppSample iris→mtcars switch → no broken intermediate state? (~2 min)
  1. Boot the super (default: ppSample("iris"); root aes pickers seeded with Sepal.Length / Sepal.Width; shared_fac = Species).
  2. Switch the root ppSample from iris to mtcars. Do the aes(x) / aes(y) / shared_fac pickers flash a broken/empty intermediate during the switch, or do they re-bind cleanly?
  3. Set aes(x) = mpg, aes(y) = wt, shared_fac = cyl. Click Update. Plot should render with mtcars data (verified clean through eval+build).
  4. Switch back to iris, set aes(x) = Sepal.Length, aes(y) = Sepal.Width, shared_fac = Species, click Update. Same question about the reverse switch.

Gate cite: test-super-pressure.R:445-639 drives the same iris↔mtcars dance and asserts expect_no_plot_error + propagation; eyeball judges the transient state the gate cannot see.

Recipe B — geom_rug ppUpload → no stuck recalculating spinner? (~1 min)
  1. From a working state on iris (Recipe A's final step), upload sample_rug.csv via the geom_rug df_rug fileInput.
  2. The rug-layer pickers should populate from the CSV's columns (mpg, wt). Do you see a stuck recalculating spinner first, or do the pickers populate immediately? ADR-0015 / 6ee63d7 should mean no.

Gate cite: test-super-pressure.R:564-622 uploads sample_rug.csv and asserts the layer pickers do NOT offer iris columns (scope is df_rug, not the root). Eyeball judges spinner latency.

super-3 L3-multi-shared-plotly

Recipe A — shared key across ggplot ↔ plotly cells → synchronized? (~1 min)
  1. Boot the super. You see two cells: plot1 (ggplot) and plot2 (plotly).
  2. Drive the Linked color picker (shared coordinator panel) to am. Click Update on both cells.
  3. Both cells' point colors should change to color-by-am. Does the color update look synchronized across cells, or noticeably staggered?
Expected ggplot2 message. Cell B's geom_smooth emits a "colour aesthetic dropped during statistical transformation" warning when the linked variable is continuous numeric (am, gear, cyl, etc.). The smooth line is single-color; the points still color correctly. Harmless — if you want a perfectly clean console, the picker doesn't offer a factor option.

Gate cite: test-super-pressure.R:692-829 asserts expect_rendered + expect_no_plot_error on both #plot1-ptr_plot and the plotly cell; eyeball judges cross-backend timing.

Recipe B — plotly tooltips → content reasonable? (~30 s)
  1. After Recipe A's draw, hover over the plotly cell's points.
  2. Do the tooltips show the variables you expect (x, y, color)? Any garbled labels or unexpected fields? The gate only checks the plotly node exists, not what its tooltips say.

No gate cite — tooltip content is a plotly interactive feature outside ggpaintr's render path.

super-4 user-css adversarial

Recipe A — user.css applied → visual theme matches the rule? (~1 min)
  1. Open dev/scripts/super-examples/super-4-user-css-safety-adversarial/user.css in another window. Find the rule for .ptr-super4-colorpicker.
  2. Boot the super. Find the ppColor widget (geom_smooth, color text input).
  3. Does the widget actually look like the CSS rule says it should? The gate only confirms the link tag fetched and the wrapper class landed in the DOM; it cannot judge visual matching.

Gate cite: test-super-pressure.R:966-970 asserts the wrapper has class ptr-super4-colorpicker; eyeball judges visual outcome.

Recipe B — denylist eval(parse()) → error reads human? (~30 s)
  1. Find the ppExpr subtitle widget (labs).
  2. Type eval(parse(text="1+1")) and click Update.
  3. Read the error pane — does a non-developer user understand what's wrong and what to do?

Gate cite: test-super-pressure.R:1042 pins the literal is not allowed in an `expr` input. Eyeball judges readability.

Recipe C — ppColor non-hex → error reads human? (~30 s)
  1. Type notahex into the ppColor widget. Click Update.
  2. The must be #RRGGBB hex message — does a non-developer get it?

Gate cite: test-super-pressure.R:997-1016 pins the hex error literal. Eyeball judges readability.

  • Console R messages from the browser tab and the parent R session.
