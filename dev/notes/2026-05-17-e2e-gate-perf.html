e2e gate performance — serial Chromium boots

2026-05-17 · status: OUT OF SCOPE / tracked optimization · not part of any current plan · do not action without an explicit scoped task

Observation

The authoritative gate (NOT_CRAN=true … testthat::test_dir("tests/testthat")) takes ~80–86s wall. Per-file timing (from a JUnit/progress run, e.g. /tmp/iss02/run_16.log):

e2e-vignette-examples-shinytest2  [67.9s]
all other ~1,500 unit tests       ~14s combined

So ~83% of gate wall time is one file: tests/testthat/test-e2e-vignette-examples-shinytest2.R. Each test_that calls boot_vignette_app() (tests/testthat/helper-vignette-apps.R), which launches a fresh headless Chromium (chromote) + a Shiny subprocess that pkgload::load_all()s the whole package, per fixture, serially, for ~20+ fixtures. Chromium cold-boot + Shiny/app init dominates; the assertions are trivial.

Why this is not "just delete the slow tests"

The full e2e suite must remain mandatory at the merge / "done" gate. Skipping browser tests regularly is the exact proxy-trap that masked the W1 formula-local var(shared=) binder bug for an entire prior effort (see CLAUDE.md authoritative-gate block, corrected 2026-05-17, and project memory shinytest2-appdir-pkgload). "Slow" justifies keeping e2e out of the tight inner loop — never out of the gate.

Current (correct) cadence: inner loop = devtools::test(filter="<file>") (seconds); gate = full NOT_CRAN suite incl. e2e at merge/done only; flake-hunt loops = explicit one-off backgrounded diagnostics, never regular.

Proposed direction (preferred)

AppDriver / chromote reuse across fixtures. The serial per-fixture Chromium cold-boot is the dominant cost. A persistent chromote session / browser process reused across boot_vignette_app() calls (new tab or fresh AppDriver against a shared browser, app subprocess still per-fixture for isolation) should remove most of the ~68s. Net: a much cheaper gate with no reduction in coverage.

Alternative / complementary

  • Split e2e into its own target. Inner-loop command = test() minus the e2e file; gate command = full incl. e2e. Pure cadence change, zero risk, but doesn't speed the gate itself.
  • Parallelise fixture boots (testthat parallel / future) — bounded by Chromium/port resource contention; note this repo already has order-dependent shinytest2 isolation history (commit 77a927e; .scratch/shared-section-fix/issues/02), so parallelism must not reintroduce cross-test state bleed.

Risks / constraints for whoever picks this up

  • Browser/session reuse must preserve per-fixture isolation: the app subprocess pkgload::load_all()s dev source (see shinytest2-appdir-pkgload memory — stale system install hazard) and the resourcePath leak / prune_dead_ggpaintr_resource_paths() asymmetry (boot helper does not call it; test-shared-lockstep-shinytest2.R does — see issues/02 investigation notes) must be handled or the reuse will amplify, not fix, flakiness.
  • Keep the source-root skip guard (boot_vignette_app()) so devtools::check() stays honest (CLAUDE.md proxy-trap block).
  • Verify any reuse change against the JUnit-instrumented loop harness used for issues/02 (≥40× full suite, 0 faults) before trusting it.
