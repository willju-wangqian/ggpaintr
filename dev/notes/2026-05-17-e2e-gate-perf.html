e2e gate performance — two refuted levers; cost is the per-fixture shinytest2 boot handshake

2026-05-17, diagnosis corrected & second lever empirically refuted 2026-05-18 · status: OUT OF SCOPE / tracked; no viable speed lever found — see "Bottom line" · do not re-attempt the refuted levers without new evidence

Correction 1 (2026-05-18) — original Chromium-cold-boot diagnosis was wrong. The first version asserted the dominant cost was a "fresh headless Chromium cold-boot per fixture" and proposed "AppDriver / chromote reuse across fixtures". Both false: shinytest2 0.5.1 app_initialize_ L44 uses chromote::default_chromote_object()$new_session() — a process-wide singleton. Chrome cold-boots once for the whole file (~2.0s); identical() on the chromote object stayed TRUE across all boots. shinytest2 already does the proposed "reuse"; that direction wins ~0s. Struck-through below, not deleted (record of the error).
Correction 2 (2026-05-18) — the install-once lever (Correction 1's replacement) is ALSO refuted, end-to-end. Correction 1 re-diagnosed the cost as the per-fixture pkgload::load_all() (~1.8s ×18) and proposed installing the dev source once into a temp lib so each fixture library(ggpaintr)s it. That lever was implemented and measured against the real gate, and it does not work. Micro-benchmark confirmed the per-child package-load saving is real (load_all 1.64s → temp-lib library(ggpaintr) 0.58s, ≈1s/child). But an apples-to-apples A/B of the actual 18 boot_vignette_app() calls (Chrome warm, interaction waits excluded) showed: OLD load_all = 45.2s; NEW install-once = 47.9s — slightly worse. The package-load saving is off the critical path of AppDriver$new() (the child loads the package concurrently with chromote navigation + the Shiny-readiness handshake, which dominate and are unchanged), and the one-time R CMD INSTALL (~2.3s) makes the net negative. The implementation was reverted. Projected ≈19s gain → measured ≈0 (net −2.7s). Lesson: a real micro-benchmark saving is not a gate saving unless it is on the wall-clock critical path — verify end-to-end, never project from the micro-bench.

Observation

The authoritative gate (NOT_CRAN=true … testthat::test_dir("tests/testthat")) takes ~80–86s wall. Per-file: test-e2e-vignette-examples-shinytest2.R ≈ 68–70s (18 fixtures), all other ~1,500 unit tests ≈ 14s combined — ~83% of gate wall time is the one e2e file.

What actually costs the time (measured 2026-05-18)

  • Chrome: one ~2.0s cold-boot for the whole file (singleton; see Correction 1). ≈0 per fixture.
  • Per fixture (×18), ≈2.5s each ≈ 45s, the dominant term: AppDriver$new() spawns a fresh Shiny child process, opens a new chromote session, navigates it, and blocks on the Shiny-app-ready handshake + first render. This is structural shinytest2 per-app boot cost.
  • Child package load sits inside that ≈2.5s but not on its critical path — it overlaps chromote-side setup. Cutting it (Correction 2's lever) does not shorten the boot. Refuted.
  • Test interaction (set_input/draw/wait_for_idle, incl. the §5.1/§5.2 25s settle ceilings) accounts for the remainder of the ~68s beyond the ~45s of boots.

Refuted lever 1 — chromote reuse — superseded by Correction 1

"Chromium cold-boots per fixture; reuse the browser across fixtures." Chrome is a shared singleton; shinytest2 already reuses it. ~0s available.

Refuted lever 2 — install-once temp lib — superseded by Correction 2

"Per-fixture load_all dominates; install dev source once into a temp lib, fixtures library() it (≈19s win)." Implemented + A/B'd against the real gate: OLD 45.2s vs NEW 47.9s (net worse). Package-load is off the boot critical path. Reverted.

Why this is not "just delete the slow tests"

The full e2e suite must remain mandatory at the merge / "done" gate. Skipping browser tests regularly is the exact proxy-trap that masked the W1 formula-local var(shared=) binder bug for an entire prior effort (see CLAUDE.md authoritative-gate block, corrected 2026-05-17, and project memory shinytest2-appdir-pkgload). "Slow" justifies keeping e2e out of the tight inner loop — never out of the gate.

Current (correct) cadence: inner loop = devtools::test(filter="<file>") (seconds); gate = full NOT_CRAN suite incl. e2e at merge/done only; flake-hunt loops = explicit one-off backgrounded diagnostics, never regular.

Bottom line / remaining options

No package-load or browser-reuse lever shortens the gate — both plausible ones were implemented/checked and refuted. The ~45s is irreducible shinytest2 per-app boot×18 short of changing the harness shape. The only non-refuted directions are cadence, not speed:

  • Split e2e into its own target (recommended if anything). Inner-loop command = test() minus the e2e file (seconds); gate = full incl. e2e. Zero risk, zero behaviour change, real dev-loop relief; does not speed the gate itself. This is already the de-facto practice (CLAUDE.md cadence) — formalising it as a named target is the only low-risk improvement on the table.
  • Bounded parallel fixture boots (testthat parallel / future) — could cut gate wall time, but the gain is not ~45/N (see the measured spike below — the naive projection is refuted). testthat's parallelism unit is the test FILE (test_files_parallel), not test_that: all 18 e2e tests are in one file, so this requires splitting that file first — flipping Config/testthat/parallel alone buys nothing, and devtools::test() is the supported parallel entrypoint (a bare test_dir()+load_all() errors in worker bootstrap). Still elevated-risk: this repo has order-dependent shinytest2 isolation history (commit 77a927e; .scratch/shared-section-fix/issues/02). The process-global resourcePath-leak asymmetry — prerequisite (a) — was closed 2026-05-18: boot_vignette_app() now calls prune_dead_ggpaintr_resource_paths() (gate unchanged at PASS 1655, no-op for live paths). Remaining before any parallelism: (b) split the e2e file, (c) verify no flake regression on the now-concurrent suite via the JUnit-instrumented ≥40× loop (issues/02 harness). Deferred behind an explicit trigger (merge-gate wait becoming a concrete bottleneck), not a near-term task.
Recommendation: accept the ~68s e2e gate as the cost of real browser coverage; rely on the existing cadence (e2e out of the inner loop, mandatory at the gate). If the gate cost ever becomes intolerable, the only worthwhile work is the isolation hardening + bounded parallelism — a substantial scoped effort, not a quick win. Do not re-attempt chromote-reuse or install-once: both are empirically dead.

Parallelism spike — measured 2026-05-18 (break-even at small scale; ~45/N refuted)

Bounded throwaway spike: 6 cheap fixtures split into 2 files, run serial vs devtools::test() parallel with 2 workers (Config/testthat/parallel: true, Ncpus=2); DESCRIPTION + spike files reverted, nothing committed.

serial   (1 process, shared Chrome) : 18.15s
parallel (2 workers)               : 17.99s   <-- no speedup

It genuinely ran parallel (user 26s > real 18s = real CPU overlap), but the gain was entirely consumed by per-worker fixed cost ≈ ~9s: each testthat worker is a fresh R subprocess paying R startup + source package load + its own Chrome cold-boot — chromote::default_chromote_object() is a per-process singleton, so N workers = N Chromes, no sharing — + helper sourcing. At 3 fixtures/worker the fixed cost ≈ the concurrency gain ⇒ break-even.

This refutes the naive ~45/N projection. Per-worker fixed cost dominates at small scale — the same class of error as the install-once micro-bench (Correction 2): an extrapolation is not a measurement.

Projection only (NOT measured at scale, must be gate-verified before trusting): the ~9s fixed cost amortizes with more fixtures/worker. Full e2e ≈ 45s of serial boots; parallel wall ≈ 9s + (18/K)·2.5s → K=2 ~31s, K=3 ~24s, K=4 ~20s, vs serial ~45–68s. So a real ~2–3× merge-gate win is plausible at full scale only — but the spike sits at break-even, so this number is a projection from the derived per-worker fixed cost, not a result. The spike's deliverable was a ~20–24s K=4 target replacing the unfounded ~45/N — but that projection too was refuted by the full-scale measurement below (real K=4 ≈ 35s, ceiling ~1.4×). Read the next section as the authoritative result; this one is the intermediate step.

Not established by the spike: (1) isolation under concurrency — only ~1 clean parallel run (the repeat loop hit the bare-test_dir bootstrap error), n≈1 says nothing about a ~2.5% flake; (2) the full-scale number itself. Both require the file split + the issues/02 ≥40× JUnit loop on the concurrent suite before any trust.

Full-scale measurement — 2026-05-18 (the ceiling is ~1.4×, not ~2–3×)

Bounded throwaway: all 18 e2e fixtures (trivial boot + inherits(app,"AppDriver") assert — boots are the cost, assertions ~0), split K ways, serial vs devtools::test() parallel. DESCRIPTION + files reverted by an EXIT trap; nothing committed; tree verified clean.

serial   (1 process, shared Chrome) : 50.51s   [PASS 18]
parallel K=4 (Ncpus=4)             : 36.56s / 33.76s   (x2)
parallel K=6 (Ncpus=6)             : 41.55s            <-- WORSE than K=4
Full-scale parallel is a real but modest ~1.4× win (50.5s → ~35s on boots), not the ~2–3× / ~20s the projection promised. More workers is not better: K=6 (41.5s) is slower than K=4 (35s) — past K≈4 on this 8-core box, N concurrent Chrome+Shiny subprocess pairs contend for CPU/disk/memory faster than the shrinking per-worker fixture count saves. Shallow optimum around K=3–4, non-monotonic.

Third over-optimistic projection refuted in this note (≈45/N → ~20–24s → now measured ~1.4×). Each extrapolation overshot because per-worker fixed cost and inter-worker contention are larger than any isolated micro-derivation predicts. Same lesson as install-once (Correction 2) and the small spike: only the gate measurement counts; do not act on the projection.

Extrapolated to the real e2e file (~85s: these boots + the per-test set_input/draw/wait_for_idle incl. §5.1/§5.2 25s ceilings, which also parallelise per file): a similar ~1.4× ⇒ ~85s → roughly ~55–60s, not ~20s. A ~25–30s saving on a gate that runs at merge/done only — against the cost of splitting + maintaining the e2e file's verbatim-vignette structure, K× memory, non-obvious K tuning, and a mandatory ≥40× isolation re-trust on a now-nondeterministic suite.

Isolation: FAIL 0 across all parallel runs (2×K=4, 1×K=6) — but n≈3 is weak for a ~2.5% flake (P(0|unfixed) ≈ 0.9); this does not establish isolation safety. The issues/02 ≥40× JUnit loop remains mandatory before any trust.

Decision (reaffirmed, now evidence-backed): do not pursue parallelism now. The measured ceiling is ~1.4× (not transformative), non-monotonic in K, and carries real added fragility + maintenance for a merge-only gate. Keep deferred behind the explicit trigger. The concrete deliverable of this whole investigation: a measured ceiling replacing three successive over-optimistic projections — decisive input that says "not worth it" rather than a vague "maybe".
