Phased orchestration

Phase 1 — parallel implementation + per-worktree audit (plans 01–05)

  1. Spawn 5 implementer agents in parallel, one per worktree, in a single message with 5 Agent tool calls. Each agent gets the brief template below with its plan/worktree filled in. Model: opus, high effort.
  2. When each implementer reports done, immediately spawn an audit agent for that worktree (also opus, high effort). Audit briefs below. You may start audit-N while other implementers are still running — they don't share state.
  3. If implementer or auditor stops-and-reports (Success Criterion unmet, scope conflict, gate red they can't fix): surface to the user verbatim. Do NOT work around. Do NOT re-spawn with a "just do it" brief.
  4. Audit must independently re-run the authoritative gate. The implementer's "DoD met" is a claim, not proof (memory: verify-subagent-claims).

Phase 2 — sequential merges in plan order, with re-audit after each merge

Merge order: 01 → 02 → 03 → 04 → 05 → 06 → 07 → 08. (Even though 01-05 are parallel-safe in implementation, merge them sequentially to keep history reviewable.)

For each plan N in order:

  1. Pre-merge re-audit from the orchestrator: run the authoritative gate in the worktree adr9-N-…. Expect 0/0/0. If red, halt and surface to user.
  2. Merge worktree into vignette-review: git checkout vignette-review (you're already on it) and git merge adr9/NN-… --no-ff -m "merge: PLAN-NN <slug> (ADR 0009)". --no-ff keeps the per-plan merge commit visible in history.
  3. Post-merge re-audit: run the authoritative gate on vignette-review's new HEAD. Expect 0/0/0. If red — a regression has been introduced by the interaction between N and previously-merged plans — halt and surface to user; do NOT proceed to plan N+1.
  4. Spawn a focused audit sub-agent on the merged state with the post-merge brief below: verify no out-of-scope files were touched; verify the plan's "Files touched" matches the actual diff; verify no plan-file stamps drifted.
  5. If everything green: this is the orchestrator's commit point. The merge commit IS the commit (no separate commit needed — the merge created it). Confirm git log -1 shows the merge.
  6. Before starting plan N+1's implementer (if N+1 is sequential — 06, 07, 08): ff-merge vignette-review into the worktree branch adr9/(N+1) so the implementer sees the latest base. git -C /…/adr9-(N+1)-… merge vignette-review --ff-only. The worktree branch tip was previously at the pre-merge base; after this it has all of plan N's work + its own pending changes (none yet, since implementer hasn't started).

Phase 3 — atomic finale (plan 08)

Plan 08 is the breaking rename + UI toggle + vignette/test/fixture overhaul. After plans 01–07 are merged:

  1. ff-merge vignette-review into adr9/08-rename-atomic.
  2. Spawn the implementer for plan 08. This run will be the longest (touches the whole codebase).
  3. When implementer reports done, spawn the audit agent for 08.
  4. Pre-merge gate; merge into vignette-review with --no-ff; post-merge gate; final audit.
  5. Surface a final summary to user: 8 plans merged, gate green, list of files touched, anything outstanding.

What to do on failure

FailureAction
Implementer stops-and-reports (e.g. plan disagrees with source)Surface verbatim to user. Do NOT re-spawn. Wait for user steering.
Auditor reports DO NOT MERGESurface verbatim. Do NOT merge. User decides whether to re-spawn implementer with a corrective brief.
Pre-merge gate red in worktreeHalt. The implementer reported green but the gate is red. Re-run yourself; if confirmed red, the implementer's self-report was wrong. Surface to user.
Post-merge gate redRevert the merge (git reset --hard HEAD~1 is acceptable on vignette-review since no force-push to remote; but ask user first per CLAUDE.md destructive-action policy). Surface the regression.
Sub-agent budget exhausted / model rate-limitedStop. Surface to user with which plan is in-flight.

What this workflow does NOT do

  • Does NOT push to remote. The user merges to main or pushes when they're ready, not the orchestrator.
  • Does NOT delete worktrees. Keep them for the user to inspect post-merge.
  • Does NOT skip the gate to "save time". The NOT_CRAN gate is the only thing that exercises browser e2e; skipping it leaves a class of regressions undetected.
  • Does NOT bypass the pre-commit hook with --no-verify. Hook refusal is a signal worth investigating.
