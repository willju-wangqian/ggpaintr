3 — Blocked plans & recommended follow-ups

3a — PLAN-03 and PLAN-06 — API gap: validate_input on value placeholders

Root cause: ptr_define_placeholder_value (R/paintr-registry.R:413-441) has no validate_input parameter; only ptr_define_placeholder_consumer (R/paintr-registry.R:509-540) does. PLAN-03's ppPower and PLAN-06's ppColor both require validate_input on a value placeholder. ADR-0013's drift-audit cycle (during /decision-to-plan) caught 7 other API mismatches but missed this one. The implementer's report on each plan independently confirmed it; orchestrator independently re-verified by reading the source. Real API gap, not a misreading.

Recommended follow-up — single source plan:

  1. Extend ptr_define_placeholder_value() with validate_input = NULL, mirroring the consumer signature.
  2. Decide the validation contract for value placeholders: consumer's is function(value, upstream_cols). For value placeholders, upstream_cols is meaningless (no data context); consider function(value, node) or just function(value). Document explicitly which.
  3. Add if (!is.null(validate_input)) validate_hook(validate_input, "validate_input", <arg list>) in the constructor, store validate_input on the registry entry, wire the runtime invocation (search R/paintr-runtime.R / R/paintr-resolve.R for where consumer's validate_input is called).
  4. Regression: add a value-placeholder validate_input test to test-placeholder-registry.R (rejection path + inline-error surfacing).
  5. After that plan lands: rerun PLAN-03 and PLAN-06 verbatim. No further plan amendment needed.

3b — PLAN-04 — Product defect: layer-aes ppVar nested under layer-data ppUpload

Root cause: In a formula like geom_rug(data = ppUpload(df_rug), aes(x = ppVar(mpg))), the UI walker emits the DOM container geom_rug_1_1_ppVar_NA_ui but ptr_runtime_input_spec() never produces a row for it; the picker stays in recalculating indefinitely. Preserve mode reveals the captured shape is aes(x = ppVar) with the mpg default dropped — strongly suggests ptr_setup_consumer_uis()'s find_nodes(tree, is_ptr_ph_data_consumer) classifier misses the layer-aes ppVar nested under a layer-data ppUpload.

Recommended follow-up — investigation plan (source-side):

  1. Reproduce with the abandoned PLAN-04 commit: git checkout 0013-app-2b-customsource-splice && devtools::load_all() && shiny::runApp("tests/testthat/fixtures/vignette-apps/super-2b-customsource-splice"); observe the geom_rug picker in browser inspector.
  2. Read ptr_setup_consumer_uis() + is_ptr_ph_data_consumer in R/paintr-resolve.R (or wherever the classifier lives). Trace why the inner ppVar isn't classified as consumer-of-the-layer-data when its parent layer's data is itself a ppUpload.
  3. Likely fix: extend the consumer-discovery walker to descend into a layer's data = ppX(...) wrapper before classifying its aes() consumers — the data-source's columns become the consumer's column scope.
  4. Test: write a narrow fixture (one geom_rug with both data = ppUpload(...) + aes(x = ppVar(...))), assert the picker populates from the upload's columns.
  5. After the source plan lands: rerun PLAN-04 verbatim. The two BDD `Then` clauses the implementer narrowed will now succeed.

The abandoned commit a73ccca on branch 0013-app-2b-customsource-splice contains useful diagnostic detail (the inline comments naming the failure shape); don't delete it before the source plan lands. It's a record of the defect surface.
