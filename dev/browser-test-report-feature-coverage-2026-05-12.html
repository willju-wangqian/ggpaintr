BUG-1 — ptr_define_placeholder_source() without companion_id_fn bricks every downstream column picker

breaks App A & App B
Trigger
App A / App B (formula pick_ds |> head(num) |> dplyr::select(colvars) |> dplyr::mutate(metric = var + log(var)) |> dplyr::filter(var > num) |> ggplot(aes(x = var, y = var, color = var)) + …). pick_ds is defined via ptr_define_placeholder_source() with no companion_id_fn.
Observed
The console floods with this stack, repeated for output$ggplot_3_1_colvars_NA_ui, output$ggplot_4_1_1_var_NA_ui, output$ggplot_4_1_2_1_var_NA_ui, output$ggplot_5_1_1_var_NA_ui, plus once at observe [paintr-server.R#352] and once at observeEvent(input[[src_id]]) [paintr-server.R#386]: 
Warning: Error in map$get: key must be not be "" or NA
  144: map$get
  143: .values$get
  142: .subset2(x, "impl")$get
  141: [[.reactivevalues
  139: <reactive> [/Users/willju/Research/ggpaintr-rewrite/R/paintr-server.R#822]
  117: entry_reactive
  116: renderUI [/Users/willju/Research/ggpaintr-rewrite/R/paintr-server.R#829]
  115: func
   99: output$ggplot_3_1_colvars_NA_ui

Every <div id="…_ui"> for the pipeline consumer / aes var pickers stays in the recalculating CSS state forever; the Data sub-tab shows the select() enable checkbox with no widget under it; clicking "Update plot" silently does nothing (#ptr_code stays empty, no plot, no error).
Root cause
ptr_runtime_input_spec(ex1_tree) emits a role = source_companion row for pick_ds with input_id = NA and source_id = ggplot_1_pick_ds_NA. In ptr_setup_pipelines() (paintr-server.R ~L340-373): 
comp_id <- if (!is.null(node$companion_id)) ns(node$companion_id) else NULL
node$companion_id here is NA (not NULL), so the guard passes and a bogus namespaced id is built. The companion-autoname observer (L384-393) and the upstream-companion read in the consumer-UI reactive (paintr-server.R L821-822, for (cmp in upstream_source_companion_ids) val <- input[[ns(cmp)]]) then both hit input[[ns(NA)]] → the reactiveValues map rejects empty/NA keys. The !is.null() checks need to also reject NA and non-nzchar ids — or a source with no companion_id_fn shouldn't emit a source_companion node at all.
Expected
pick_ds resolves to a ggplot2 dataset, the consumer pickers populate with that dataset's columns, and switching pick_ds refreshes the downstream column choices.
Status
Reproduces on tip 119b8ab — log shows 6 map$get errors at the same four source locations (paintr-server.R:352, 386, 822, 829) on a fresh launch.

BUG-3 — ptr_app_grid() aborts when a shared_ui key is unused in the first plot formula

breaks App D
Trigger
App D — ptr_app_grid(plots = list(ex2_grid_a, ex2_grid_b), shared_ui = list(cv = …, pt = …), …). ex2_grid_a uses only colvars(shared = 'cv'); ex2_grid_b uses both colvars(shared = 'cv') and num(shared = 'pt').
Observed
Warning: Error in ptr_validate_shared_bindings: `shared` references key "pt" which is not used in any plot formula. Available formula keys: "cv".
  …
  57: ptr_validate_shared_bindings [/Users/willju/Research/ggpaintr-rewrite/R/paintr-utils.R#92]
  56: ptr_init_state [/Users/willju/Research/ggpaintr-rewrite/R/paintr-server.R#108]
  55: ptr_server [/Users/willju/Research/ggpaintr-rewrite/R/paintr-server.R#266]

  …
  40: server [/Users/willju/Research/ggpaintr-rewrite/R/paintr-app.R#628]
   1: shiny::runApp
Both plot panels render the title / layer picker / "Update plot" button, but plot_1-ptr_plot and plot_2-ptr_plot stay recalculating. "Render both" (works as a click target — wired correctly) and the per-plot "Update plot" buttons produce no plot, no code, no error.
Root cause
Validation runs per-plot-module against the full shared list, so initialising the plot_1 module (formula ex2_grid_a, only key cv) trips on the presence of pt. The diagnostic ("not used in any plot formula") is also factually wrong from the user's POV — pt is used by ex2_grid_b; the validator only saw one formula's keys. Validate shared against the union across plots, and let a shared widget unused by a given plot pass for that plot's module.
Status
Reproduces on tip 119b8ab — same stack, both plot outputs stuck recalculating.

BUG-2 — ui_text$shell$draw_button$label override is silently ignored

affects App A & App B
Trigger
App A / App B with ui_text = ex1_ui_text, which sets shell = list(… draw_button = list(label = "Render plot")).
Observed
Draw button reads "Update plot" (the package default). The sibling override shell$title$label = "Feature-coverage demo" does take effect (page header + document.title), so the override-parsing path works; only the draw-button leaf is dropped.
Root cause
paintr-app.R:191 reads 
label = shell_copy$update_plot_label %||% "Update plot"
But layer_panel_default_shell_copy() in paintr-build-ui.R:404-411 only populates data_subtab_label, controls_subtab_label, layer_checkbox_label, layer_picker_label — there is no update_plot_label key. The copy registry maps the override to c("shell", "draw_button"), i.e. shell_copy$draw_button$label, which the app never reads. The bslib shell shares the same path. (draw_all_label for the grid is a separate function arg and works — see App D.)
Expected
Draw button reads "Render plot".
Status
Reproduces on tip 119b8ab — document.getElementById('ptr_update_plot').innerText === "Update plot" in both App A and App B.
