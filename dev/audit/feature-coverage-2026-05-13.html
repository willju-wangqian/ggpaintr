BUG-A1 — Changing pick_ds at runtime leaves the aes var pickers stale (empty options)

App 1a, App 1b
App + step
  1. App launches with pick_ds defaulting to mpg. Aes x/y/color pickers (ggplot_1_1_var_NA, ggplot_1_2_var_NA, ggplot_1_3_var_NA) correctly list mpg columns.
  2. Switch the "Dataset for data" select from mpg to diamonds (via selectize.setValue / native change). Wait for the producer-debounce to settle.
  3. The mutate/filter var pickers (e.g. ggplot_4_1_1_var_NA, ggplot_5_1_1_var_NA) and the colvars consumer correctly refresh to diamonds columns. The three aes pickers under ggplot(aes(...)) remain with empty option lists indefinitely.
  4. Setting colvars to a subset of diamonds columns then propagates fresh column lists to mutate/filter pickers but still not to the aes pickers.
Observed
['ggplot_1_1_var_NA','ggplot_1_2_var_NA','ggplot_1_3_var_NA']
  -> opts: []          (after pick_ds -> diamonds)

['ggplot_4_1_1_var_NA','ggplot_5_1_1_var_NA']
  -> opts: ["carat","price","cut"]   (when colvars = [carat,price,cut])
Expected
Every consumer downstream of the pipeline (including the aes pickers under the ggplot() call) should refresh its option list when an upstream column-producing widget (here, the pick_ds source) changes value. The mutate/filter pickers correctly do; the aes pickers do not.
Hypothesis
The per-layer aes pickers may be wired to a one-shot column snapshot taken at session start (or only re-evaluated on a specific reactive trigger that pick_ds doesn't fire), rather than subscribing to the same upstream-cols reactive as the per-stage pickers. Worth comparing the ptr_setup_consumer_uis wiring of aes consumers against the stage consumers.

BUG-A2 — defaults$num$help override is ignored by the shared num widget

App 1a, App 1b
App + step
Load App 1a. Inspect the "Shared controls / Enter a number" widget at the top.
Observed
Help text reads "Enter a value between 0 and 1." (default copy specialised for the alpha use), even though ui_text$defaults$num$help is set to "Any number works here.". The per-stage num widgets (head, filter) do show the override text. The empty_text override "No number yet" is also not rendered on the shared widget.
Expected
Either the override applies (preferred), or the docs explicitly call out that shared-widget copy is computed from per-param defaults and not influenced by defaults$num.
Hypothesis
ptr_resolve_ui_text("control", keyword="num", …) likely walks the per-parameter defaults branch first for shared widgets and never reaches the defaults$num override. Audit the merge precedence in ptr_resolve_ui_text for the shared-widget code path.
