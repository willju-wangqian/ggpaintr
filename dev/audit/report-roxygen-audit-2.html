Issue Detail

Critical

ptr_server Critical @return — inaccurate R/paintr-server.R · line ~308
Doc states @return The `ptr_state` list, invisibly. but the function body ends with state (not invisible(state)). The return is visible. Callers who rely on the invisibility promise (e.g. piping into nothing in a server = function(...) { ptr_server(...) } context) will not be affected in practice, but the contract is wrong and the discrepancy between ptr_server (visible) and ptr_module_server (also returns state from moduleServer, effectively invisible) is undocumented.
ptr_define_placeholder_source Critical @return — missing R/paintr-registry.R · line ~354
The function has no @return tag at all. The sibling constructors ptr_define_placeholder_value and ptr_define_placeholder_consumer both document their return as "invisibly, the registry entry list" — this function returns the same thing (ptr_registry_register(entry) invisibly) but the omission leaves a blank in the rendered help page and breaks consistency for custom-source authors who consult all three reference pages.

ptr_server Major @param — incomplete R/paintr-server.R · line ~297
The ... param description says "Forwarded to ptr_init_state() (e.g. shared, draw_trigger, ui_text, checkbox_defaults, expr_check, safe_to_remove, ns)" but omits producer_debounce_ms, auto_bind_shared, shared_resolutions, shared_stage_enabled, and plots, all of which are valid in ptr_init_state(). Doc says @return A Shiny tag list. The implementation returns shiny::fluidPage(...) which is a shiny.tag (a div node), not a tagList. The distinction matters when callers check inherits(ui, "shiny.tag.list") vs "shiny.tag" or attempt to append children.

The [build_ui_for()] cross-reference links to a symbol that is only registered as S3method(build_ui_for, ...) — it is not export()ed in NAMESPACE. ptr_shared Major @return — incomplete slot list R/paintr-shared-coordinator.R · line ~87
@return says "Public fields: panel_keys (cross-formula keys), local_keys_by_formula (per-formula inline keys)." In practice the callers (ptr_shared_server, ptr_shared_panel, ptr_ui_shared_panel, ptr_module_server) also read formula_keys, formula_count, draw_all_label, trees, firsts, consumer_reps, and shared_ui. Advertising only two fields as "public" will confuse advanced embedders who read the return value directly.
ptr_shared_server Major @return + @param — partial/inaccurate R/paintr-shared-ui.R · line ~165
(a) @return mentions shared_stage_enabled in passing ("a named list of reactives, one per shared key, indicating whether that key's shared stage is active") but does not state when the list is non-empty (only when any formula contains a stage-enabled checkbox — an orphan stage). An empty list is the common case; users who receive an empty shared_stage_enabled and don't know why have no guidance here.

(b) @param shared says each named entry "replaces" the reactive for that key. The implementation shows the override only seeds the initial value — once the user picks something in the panel's picker, input[[cid]] wins (the override is only a fallback for the unset/empty input state). The word "replaces" overstates the override's permanence.
ptr_resolve_ui_text Major @param — incomplete value set R/paintr-copy.R · line ~688

Systematic Patterns

Pattern 1 — Broken [ptr_css()] cross-reference (6 functions)

ptr_css is a @keywords internal / NULL doc alias — no function body, not in export(). Yet ptr_app, ptr_app_grid, ptr_module_ui, ptr_ui_page, ptr_ui_assets, and ptr_shared_panel all include [ptr_css()] in their @seealso. These will render as broken hyperlinks in the pkgdown reference. Fix: change each to plain text "see ?ptr_css for CSS custom properties" or remove the link and inline the summary.

Pattern 2 — @seealso chains don't close the multi-instance API loop

ptr_shared() / ptr_shared_server() are absent from the seealso of all three app builders (ptr_app, ptr_app_grid, ptr_app_bslib) and from both panel functions. The multi-instance coordinator trio (ptr_shared → ptr_shared_panel / ptr_shared_server) forms a tight usage cluster — all six functions should cross-reference each other and the module pair that consumes the state.

Pattern 3 — Advanced ptr_init_state() params undiscoverable from ptr_server()

ptr_server()'s ... param lists only 7 of the 12 forwarded ptr_init_state() arguments. The omitted five (producer_debounce_ms, auto_bind_shared, shared_resolutions, shared_stage_enabled, plots) are the ones advanced embedders use for debounce tuning and shared-coordinator wiring. A user who never reads ptr_init_state()'s own help page has no signal these exist.

Pattern 4 — replace-vs-append semantics only in source comments, not public docs

ptr_gg_extra()'s replace-per-call behavior is documented in an inline source comment ("Replace-per-call: each invocation overwrites") but not in the roxygen block. Similarly, ptr_shared_server()'s shared override being a fallback (not a permanent replacement) is stated in a secondary clause. Both represent non-obvious divergences from what a reasonable caller would assume ("add extras" → accumulate; "override" → permanent). These should appear in the public docs, not just source comments.

Pattern 5 — L3 piece seealso chains are incomplete at the page-shell level

The L3 UI pieces (ptr_ui_plot, ptr_ui_error, ptr_ui_code, etc.) link to each other well, but ptr_ui_page() and ptr_ui_header() are inconsistently present in the chains. A user composing a hand-built L3 layout following the seealso trail from ptr_ui_controls() may not discover ptr_ui_page() — the shell that makes the pieces actually render correctly.
