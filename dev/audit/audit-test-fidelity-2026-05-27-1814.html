Methodology

For each test_that block, the classifier scans the body (comments stripped) for tier signals and applies the tier-break rule from the audit prompt:

  • L3 trigger: AppDriver$new / boot_vignette_app / boot_super_app / app_driver_from / fixtures/vignette-apps.
  • L2 trigger: testServer( / session$setInputs / local_mocked_bindings / isolate(reactive...) / ggpaintr::: / .render_widget_html / invoke_build_ui(.
  • STRONG L2 (subset of L2): block actually uses reactive-shortcut machinery — testServer, session$setInputs, local_mocked_bindings, reactiveVal(), shiny::isolate, .render_widget_html, ptr_init_state(, fake_state(, invoke_build_ui(, build_ui_for(, ptr_setup_*, runtime_*_entry, runtime_upstream_data, panel_owned_binding_name, try_bind_source_default_resolved, resolve_upload_source, vacate_source_binding, stamp_current_pick_walk, render_placeholder_preserved, ptr_app_components(, ptr_substitute(.
  • Dishonest STRONG L2: STRONG L2 and the test name contains an interaction-shaped word from the lexicon (click, select, pick, upload, switch, toggle, update, change, after, when, on, then, render, populate, bind, refresh, redraw, mutex, debounce, race, seed, emit, propagat, flush, react, observe, trigger, fire, wire, resolve, substitute).

The weak L2 residue (115 blocks) is L2-by-signal-pattern only because it calls a ggpaintr::: unexported helper that happens to be reactive-adjacent; those are pure helper unit tests in disguise (e.g. desugar_pipes_to_nested, resugar+rebuild, ptr_spec_combine) and not faithfulness offenders. They should arguably be re-classified L1 if the convention were inverted; the report leaves them L2 to stay faithful to the prompt's tier-break rule.

Anomalies

Mixed-tier files (intentional dual coverage). Four files combine L3 browser blocks with STRONG L2 unit blocks in the same file — the pattern is documented at the top of test-vacate-on-empty.R:

Two layers of coverage: (1) Unit test on the helper directly — proves the BDD Then clauses by driving the helper with a hand-built state. Avoids the shinytest2 wrapped-namespace fragility. (2) Browser-level observable test — proves the no-file branch reaches the helper AND the cleared state flows to user-visible surfaces.
  • test-adr0024-companion-entry-point.R — 4 L2 + 4 L3
  • test-adr10-ppupload-name-browser.R — 1 L2 + 1 L3
  • test-adr12-spec-roundtrip.R — 1 L1 + 5 L2 + 1 L3
  • test-vacate-on-empty.R — 2 L2 + 1 L3

This pattern is sound when the L3 test covers the integration and the L2 unit pins the contract independently. It becomes a defect only when the L3 sibling is missing — then the L2 is the only thing standing between the user and the bug, which is exactly the failure shape the prompt is hunting.

Helper-as-trojan. helper-rewrite.R::.render_widget_html drives ptr_server_internal through shiny::testServer and returns rendered HTML for a single output. Tests that look L1 (a one-line .render_widget_html(formula, "id") call) are actually L2 by tier-rule. The classifier handles this correctly via the .render_widget_html signal, but eyeballing the file would miss it. This helper is the single biggest L2 funnel in test-rewrite-*.R.

Files with zero L3 and ≥4 dishonest STRONG L2. Cleanest migration targets — high-risk fidelity, no existing fixture to lean on: test-ppupload-preserve-mode.R (7), test-host-setup-panel-sources.R (6), test-consumer-picker-panel-sources-dep.R (6), test-validate-input-retain.R (6), test-validate-input-ctx.R (5), test-resolve-upload-source-helper.R (5), test-vacate-on-empty-scope.R (5), test-shared-server.R (5).

Recommendations

(a) Migrate now — high user-impact, no L3 sibling exists

Prioritise the eight files listed in Anomalies / files with zero L3 and ≥4 dishonest STRONG L2. Each one targets a user-observable surface (consumer picker population, upload → binding, validator error display, vacate on empty) that has bitten this repo before (project memories e2e-assertion-weakness-lens, shared-section-binder). The pattern is mature: boot_vignette_app(slug) with a fixture under tests/testthat/fixtures/vignette-apps/<slug>/app.R whose body is verbatim-equivalent to the source it represents. Add one fixture per file (medium cost), assert with expect_picker_selected / expect_picker_populated / expect_sentinel_in_code — the helpers exist and are debugged.

(b) Leave alone

The weak L2 blocks (L2 by ggpaintr::: signal but no reactive-shortcut machinery) are pure helper unit tests — desugar_pipes_to_nested, resugar+rebuild, ptr_spec_combine, collect_orphan_shared_stages, the ppVerbSwitch UI SCs that call build_ui_for on a static tree. They assert pure structural invariants and would not benefit from a browser; their failure modes are caught at the AST level before reaching the reactive graph. Same goes for the four mixed-tier files: the L2 blocks there are deliberate contract pins that complement their L3 siblings.

The 82 L3 blocks are the gold standard — do not refactor away from them.

(c) L1 tests that may be too pure

The 774 L1 blocks are mostly parser / classifier / copy-rule / registry-construction tests — correct unit targets, no public-API integration risk. Two patterns worth a second look:

  • Parse-only "supported use case" tests (tests/testthat/fixtures/supported_use_cases) assert a translation succeeds but never round-trip through ptr_app_components → ptr_render. The spec-roundtrip-* family closes this gap symbolically, but a small L3 fixture per use case (5 cases, all enumerated in helper-fixtures.R) would catch the integration-vs-parse drift that bit BUG-A1 (browser-probe-pitfalls memory).
  • test-copy-rules.R, test-placeholder-registry.R, test-classify-calls-*.R assert merge / normalize / classification logic with hand-built inputs; if ptr_app() ever stopped calling those merges (a wiring regression), the L1 suite would stay green. A single L3 smoke test per file (custom ui_text override reaches the rendered widget label in a real browser session) would close this.
