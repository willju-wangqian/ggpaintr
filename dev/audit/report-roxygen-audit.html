Critical Issues

None — all previously flagged critical issues were false positives (see audit notes).

Major Issues

ptr_ui_page R/paintr-app.R Major Missing @return
@return is entirely absent. The Rd file has no \value{} section. ptr_ui_page() is a UI constructor — users cannot tell from the docs what it returns or how to embed it. All peer UI constructors document their return type (shiny.tag or shiny::tagList()).
ptr_shared_server R/paintr-shared-ui.R Major Accuracy — @return
@return lists 3 public fields; implementation has 4. Doc: "A ptr_shared_state S3 object with public fields shared, draw_trigger, shared_resolutions." The new_ptr_shared_state() constructor at line 22 also sets shared_stage_enabled, and validate_ptr_shared_state() reads and validates it. Callers who thread the bundle into ptr_module_server() depend on this field for stage-enabled toggles; omitting it from the docs makes the field effectively undiscoverable.
build_ui_for R/paintr-build-ui.R Major Accuracy — export vs internal
Marked @keywords internal yet exported in NAMESPACE. The roxygen block includes @keywords internal, which conventionally means "not for end users". However the function is listed in NAMESPACE as export(build_ui_for) and is explicitly called out in the ptr_ui_controls() description as an L3 user-facing API. The @keywords internal tag causes pkgdown to hide it from the reference index, and CRAN checks may warn. Either remove the tag (if the export is intentional) or unexport the function.
ptr_extract_plot / ptr_extract_error / ptr_extract_code R/paintr-server.R Major Undocumented reactive gotcha
Calling inside a reactive silently breaks dependency tracking — not documented. All three extractors wrap their read in shiny::isolate(state$runtime()), so they work correctly both inside and outside a reactive context. The undocumented trap is the opposite of what one might expect: calling them inside a reactive (e.g. inside observe() or reactive()) silently severs the reactive dependency — the caller will not re-run when state$runtime() updates. Users who expect live re-rendering based on these calls will be silently surprised. The @param or @details block should document this isolate-induced dependency breakage.
ptr_define_placeholder_source R/paintr-registry.R Major Missing @examples and @seealso
No @examples block; no @seealso cross-reference. ptr_define_placeholder_value and ptr_define_placeholder_consumer both have detailed @examples and reciprocal @seealso links. ptr_define_placeholder_source has neither, despite being the most complex of the three constructors (companion_id_fn, resolve_data, dual-input pattern). This asymmetry will hurt discoverability for custom source authors.

Patterns

Pattern 1: Missing @examples on almost all interactive/Shiny functions (24 of 37 exported)

Every function that builds a Shiny UI piece, runs a server, or launches an app lacks @examples. The four LLM helpers and the two copy-rule functions are the main exceptions. At minimum, L1 entry points (ptr_app, ptr_app_grid) and the module pair (ptr_module_ui, ptr_module_server) should have if (interactive()) { ... } examples consistent with ptr_app_bslib.

Pattern 2: ptr_css() cross-references resolve correctly — not a broken-link issue

ptr_app, ptr_app_grid, ptr_module_ui, ptr_ui_assets, and ptr_ui_page all link to [ptr_css()] in @seealso. ptr_css is a @name/NULL virtual Rd page defined in R/paintr-css-doc.R, tagged @keywords internal. It generates a valid Rd page and the cross-references resolve without a NOTE from R CMD check. No action needed on these links.

Pattern 3: @param depth inconsistency between entry points

ptr_app()'s @param expr_check is thorough (three modes, vignette link). ptr_app_bslib()'s is a one-liner stub. ptr_app_grid(), ptr_module_ui(), ptr_module_server(), and ptr_ui_controls() say only "Defaults to TRUE". Users of the non-primary entry points cannot discover the FALSE / list modes without reading ptr_app's docs. The full description (or a cross-reference like "See ptr_app()'s expr_check for the three modes") should appear consistently.

Pattern 4: All ptr_ui_* pieces missing @examples, hampering L3 discoverability

The entire L3 "own every UI piece" API (ptr_ui_plot, ptr_ui_error, ptr_ui_code, ptr_ui_inline_error, ptr_ui_toggle_code, ptr_ui_header, ptr_ui_controls, ptr_ui_page) has no @examples. Users discovering L3 via pkgdown or help() see descriptions but no skeleton showing how the pieces compose. A single multi-piece example shared via @rdname or placed in ptr_ui_page would close this gap efficiently.

Pattern 5: Shared API docs (ptr_shared / ptr_shared_panel / ptr_shared_server) have no examples

The multi-instance coordination API is the most complex part of the public surface and has no examples in any of its three main entry points. Given the required ordering (ptr_shared → UI placement → ptr_shared_server → ptr_module_server(shared_state=)), the absence of even a skeleton is a significant barrier to adoption.

Notes on Omissions from This Audit

  • S3 methods registered in NAMESPACE (build_ui_for.ptr_layer, prune_walk.*, etc.) are internal dispatch targets not documented as standalone help pages; they are excluded from this audit.
  • print.ptr_shared_state is in NAMESPACE as an S3method but has no standalone roxygen block — this is standard for print methods and not flagged.
  • ptr_css (the CSS token reference page) is a @name/NULL virtual Rd page in R/paintr-css-doc.R, tagged @keywords internal. It generates a valid Rd page; [ptr_css()] links from exported docs resolve correctly and produce no R CMD check NOTE.
