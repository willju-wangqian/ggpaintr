1. The lens (one-line restatement)

For each absorbed L2: if the orchestrator/runtime regressed only at THIS contract, would the browser DOM differ? If no → RETAIN as unit pin, do not migrate. If yes → MERGE. If yes-but-needs-extra-fixture → MERGE-deferred.

Two sub-questions sharpen the lens (from v6 §1):

  1. Does the assertion observe an internal-shape value (S3 class, list field name, NULL vs empty-list, named-list shape, object identity) that the rendered DOM does not reflect?
  2. Does the assertion's trigger come from an R-side mutation (e.g. ptr_gg_extra(), direct session$setInputs() of a non-DOM virtual input) that no DOM event can dispatch?

"Yes" to either → not browser-encodable as-is.

6. Decision the next step needs from the user

  1. RETAIN verdicts (server-state.R:67 and :120) — do you agree? v5 had them as MERGE; v7 demotes them on the same lens-failure pattern that produced J10's :153 RETAIN. If you'd rather absorb them anyway, the journey would need a custom-renderer fixture (D6-style) that exposes the runtime list shape via DOM — feasible but doubles fixture-build cost.
  2. Stage 6 (extras-change) — defer or build? The fixture is small (one button + one observeEvent) but is per-journey scaffolding rather than per-product code. v6 deferred D6 on the same trade.
  3. Stage 5 (tabset) — replace with active-pane-class observation, or build assignInNamespace instrumentation? Active-pane observation is simpler and tests the user-visible outcome; the namespace instrumentation tests the internal call shape (which is the L2's actual assertion). Recommendation: active-pane-class observation, plus a note in the journey file pointing back to this audit for the lens reasoning.

None of these decisions touch product code. They shape the Phase 2 journey file only. Once decisions land, the build is mechanical: write the fixture, write the test, run the gate, delete the absorbed L2s in the same commit.
