Findings

F-2  RETRACTED · FALSE POSITIVE   Upload-sourced code generation works correctly

This finding was wrong. Real-world verification (maintainer, browser, penguins.csv uploaded into the block-15j formula) shows the generated code is correct — the dataset name appears in both ggplot() and the geom_point() data argument:

ggplot(data = penguins, aes(x = bill_length_mm, y = bill_depth_mm)) +
  geom_point(data = penguins) +
  geom_smooth(method = "lm")

Root cause of the false positive (testing error, not a package defect): the original "repro" drove bare ptr_server() inside shiny::testServer with a hand-poked fileinfo input, no module ns, and a partial reactive flush. That harness does not reproduce ptr_app()'s upload → data-source → code-generation wiring (renderUI/observer cycle that substitutes the resolved object name into the code). The audit had already documented this exact limitation for 15j ("shared auto-binding is performed by ptr_app(), not ptr_server()") but failed to apply the same skepticism to the block-10 result, and instead promoted a harness artifact to a HIGH-severity source bug. Browser file upload being blocked by the automation environment is what prevented catching this directly.

Net: the upload path (blocks 10 & 15j) generates correct, copy-pasteable code. No source defect. No package change needed.

F-1  MED · SCRIPT-DOC   feature-sweep.R describes a non-existent "Update Data" button

Block 8 and the bottom checklist (lines ~553–555) instruct the tester to use an "Update Data button (G6 atomic snapshot)" on the per-layer Data sub-tab ("Edit a var to a non-existent column, click Update Data — the inline error appears and the cache is left untouched").

No such button exists. The Data sub-tab has only the per-stage checkbox + the stage's number/text input; the only action button anywhere is the global Update plot (ptr_update_plot). Confirmed in source: grep across R/ finds only ptr_update_plot (R/paintr-app.R:289, R/paintr-server.R:646); there is no update_data / ptr_update_data binding. The "G6" references in R/paintr-server.R are about terminal/dual upstream resolution, not a UI button.

Impact: the audit script misleads a tester into looking for UI that does not exist. Either G6 was consolidated into the single Update plot button (stale comment) or a feature was dropped — needs a maintainer call. Block 8 itself is fully functional: num threads through head(10), and the per-stage checkbox correctly drops the head() stage when unticked.

F-3  MED · SCRIPT-DOC   feature-sweep.R block 13 crashes — ptr_app_bslib() has no title=

Block 13 (lines ~228–233) calls:

ptr_app_bslib(
  "ggplot(...) + labs(title = text)",
  theme = bslib::bs_theme(version = 5, bootswatch = "minty"),
  title = "ggpaintr x bslib")

Running it verbatim aborts: Error: unused argument (title = "ggpaintr x bslib") — the app never launches. formals(ptr_app_bslib) = formula, envir, ui_text, checkbox_defaults, expr_check, safe_to_remove, theme; there is no title (nor does ptr_app() have one — it has css). Shell titles are set via ui_text = list(shell = list(title = list(label = …))).

Impact: the documented bslib example is broken. With the invalid argument removed, the bslib variant works correctly: themed page_sidebar navbar layout (bootswatch minty), ns-prefixed (ptr-) inputs, code ggplot(...) + labs(title = "bslib title") generated, plot renders.

F-17  LOW · SCRIPT-DOC   feature-sweep.R block 17 is incomplete — never triggers ptr_update_plot

Block 17 (the shiny::testServer accessor demo, lines ~538–548) sets only session$setInputs(.dummy = 1) then prints ptr_extract_code() / class(ptr_extract_plot()). Both come out empty / NULL, contradicting the section's intent ("Programmatic accessors … useful for tests").

This is not a code bug. state$runtime() is a reactiveVal(NULL) populated only after ptr_update_plot (or draw / extras) fires (R/paintr-server.R:184, 628, 646–649). The block never triggers it, so empty output is correct behaviour. Adding session$setInputs(ptr_update_plot=1); session$flushReact() to the same formula yields ggplot(data = mtcars, aes(x = mpg, y = hp)) + geom_point() and a ggplot object. The example as written simply doesn't demonstrate what its comment claims.

Testing limitation — browser file upload

The automation environment blocks Chrome DevTools DOM.setFileInputFiles ("Not allowed"), so the file picker in blocks 10 and 15j could not be driven through the browser. Both upload widgets render correctly in-browser (Browse + dataset-name companion + accepted-formats note: .csv,.tsv,.rds,.xlsx,.xls,.json; 15j shows one shared widget in the shared section feeding both layers). An attempt to validate block 10's data-flow headlessly via shiny::testServer + bare ptr_server produced a false positive (F-2, retracted): that harness does not reproduce ptr_app()'s upload → data-source → code-gen wiring, so it wrongly appeared to drop data =. Real-world browser upload (maintainer, penguins.csv) confirms the generated code is correct. Lesson: a bare-ptr_server testServer harness is not a faithful proxy for ptr_app() on the upload/data-source path (or the shared-binding path — already noted for 15j); when the browser path is blocked, an unreproduced testServer result must be flagged as inconclusive, not filed as a defect.

Methodology notes (for future sweeps)

  • Each block launched in its own fresh R process (process-global placeholder registry) on a fixed port; driven via Chrome MCP.
  • shinyWidgets pickerInput set via the underlying <select> + selectpicker('val',…); selectize inputs via el.selectize.setValue(…); text/number via value-set + input/change dispatch. These are far more reliable than coordinate-clicking the custom dropdowns.
  • #ptr_code and #ptr_error bind ~3–4 s after Update plot on error paths; a 2 s wait gives a false-empty read. Re-query after the wait — an early empty read is not a bug.
  • Console errors "A listener indicated an asynchronous response … message channel closed" are Chrome-extension noise, not app errors — baseline, ignored.
