Root cause: the autoname-vs-assign race blocking ADR-0015 PLAN-01

Generated 2026-05-23 · `followup-b` @ `5228646` · author: assistant (ADR-0015 PLAN-01 implementer)

This document exists because closing ADR-0015 PLAN-01's actual UX defect took far more than the plan's "two narrow edits to one file" suggested. The plan's mental model assumed shiny::req() alone would close the cache-pollution race. It doesn't. The real cause is a second flush-ordering race introduced by an in-tree shiny::updateTextInput() call inside ptr_bind_source_autoname(). The user flagged that update* calls were supposed to have been cleaned out previously, so this document lays the situation out for a clean decision.

1. What the plan asked for

ADR-0015 PLAN-01's SC-2 names the mechanism literally: at the top of each per-consumer entry_reactive, add shiny::req(state$resolved_sources[[id]]()) (or resolved_data[[layer_name]]() for bare-data layers). The plan's Governing principle writes:

The cache-pollution race that motivated fix-super1's visibility-gate is real but must be closed STRUCTURALLY (via shiny::req() on the appropriate source-ready reactive), NOT via suspension.

This is correct for one specific race: the OLD pre-warm-gate prevented entry_reactive from running at boot for source-headed consumers, so runtime_consumer_entry() never touched state$upstream_cache with a stale eval_env. req() achieves the same halt without suspending the renderUI's DOM container.

For the simple scenario PLAN-01's own fixture exercises (ppUpload(df) as layer-data, companion seeded with "df" at parse time), req() alone does close the race and the new test-adr15-consumer-binding.R file passes.

2. The race the plan didn't anticipate

The pre-existing regression test-adr12-bug-3a.R hits a different failure mode that req() alone does not fix. The fixture's formula:

ppUpload |> dplyr::filter(ppVar > ppNum) |> ggplot(aes(ppVar, ppVar)) + geom_point()

Notice ppUpload is bare (no default-arg / no companion seed). Its companion text input starts empty.

The race involves two pre-existing observers in R/paintr-server.R:

  • MAIN upload observer (ptr_setup_pipelines, line 875):
    1. Reads file_info <- input[[src_id]].
    2. Resolves df via entry$resolve_data(file_info, node).
    3. Reads nm <- input[[comp_id]] to derive binding_name.
    4. If df + binding_name both non-NULL: assign(binding_name, df, envir = state$eval_env).
    5. Calls slot(df) (sets state$resolved_sources[[id]]).
  • ptr_bind_source_autoname() (line 927): 
    shiny::observeEvent(input[[src_id]], {
  fi <- input[[src_id]]
  auto <- ptr_upload_autoname(input[[comp_id]],
                              if (!is.null(fi)) fi$name else NULL)
  if (!is.null(auto)) {
    shiny::updateTextInput(session, comp_id, value = auto)   ← THE TRIGGER 
  }
})

When the user uploads a file named penguins.csv with an empty companion:

  1. Cycle 1: input[[src_id]] changes.
    • MAIN obs fires (priority 0). Reads comp_id="". binding_name = NULL. assign skipped. slot(df1).
    • Autoname obs fires (priority 0). auto = "penguins". updateTextInput(session, comp_id, "penguins") — sends a message to the client.
    • entry_reactive invalidates (slot dep). RenderUI flush. Snapshot reads comp_id="". Substitute prunes. Resolve returns NULL. Empty picker rendered.
  2. Cycle 2: client echoes comp_id="penguins" back to server.
    • MAIN obs invalidates (depends on comp_id).
    • entry_reactive invalidates (snapshot reads input[[comp_id]] — a reactive dep).
    • Flush: both reactives invalidated by the same input change. Despite shiny's priority (observe@0 should run before renderUI@-1), the empirical trace shows entry_reactive firing while state$eval_env still lacks the penguins binding.
    • Substitute → penguins symbol. Eval against state$eval_env with no penguins binding → R's parent-chain scoping finds datasets::penguins (R 4.5+ ships a truncated-column penguins dataset baked in).
    • The wrong frame's cols are cached under deparse("penguins |> ...") in state$upstream_cache.
  3. When MAIN obs eventually does assign(penguins, df, eval_env) + slot(df2):
    • slot(df2) is dedup'd (identical(df1, df2) is TRUE because resolve_data reads the same file twice into structurally-equal frames). No further invalidation.
    • Even if entry_reactive re-fires, the cache returns the polluted truncated-penguins result.

The instrumented trace from /tmp/adr12-probe.log shows this directly:

The truncated-column bill_len / bill_dep / flipper_len / body_mass are the smoking gun: those are datasets::penguins's column names, not the uploaded CSV's. The in-aes ppVars get stuck on the cached polluted result; only the in-filter ppVar re-fires after the eval_env-bound state lands.

3. Why this took so long

The plan's diagnosis (ADR-0015 §1.5) named the cache-pollution race, but only described one of two interacting mechanisms. The plan author saw the OLD pre-warm-gate's behavior (no eager fire, no pollution) and assumed req() would architecturally replicate that property. It does — for upstream resolved-sources synchronization. It doesn't for the eval_env-binding-synchronization race that updateTextInput() opens.

The diagnostic cycle here:

  1. Implement SC-1..SC-4 verbatim. Gate runs: 3 failures in test-adr12-bug-3a.R, all "picker not populated".
  2. First wrong hypothesis: source_ready wasn't catching companion-bearing sources because the helper find_source_self_ids_in_upstream excludes nodes with companions. Fix: walk upstream with find_nodes(., is_ptr_ph_data_source) instead. Gate: 7 failures (added testServer-path regressions).
  3. Diagnosed the testServer regressions (test-rewrite-pipeline-data-source.R:132 + BUG-4): pre-existing contracts asserted "renderUI must not throw" — incompatible with req()'s silent-error semantics. Per user direction (Option A), updated those 2 test contracts.
  4. Down to 3 failures. Probed in testServer: works. Probed in shinytest2: fails with truncated penguins.
  5. Instrumented ptr_resolve_upstream via assignInNamespace in the fixture's app.R. Trace revealed evenv_has_penguins=FALSE at the polluting resolve.
  6. Tried isolate()-wrapping the companion read in snapshot. Removed the comp dep, but slot's identical-value dedup prevented the re-fire that would have caught the post-assign state. Made it worse.
  7. Settled on cache=NULL + invalidateLater(50) retry. Gate: 0/0/0/2357. This is the current committed state on followup-b@5228646.

What I didn't do until now: question the existence of the updateTextInput() call in the first place. The user's observation that update* calls are supposed to have been cleaned out is the key insight that makes a real fix obvious. Without that, the apparent options were all variations of "cope with the race"; with it, the option of "delete the race-causing call" comes back on the table.

5. Possible solutions

Each option is independent — they are not mutually exclusive but are listed in increasing scope. The current committed state is the workaround (Option A).

Option A — Keep the workaround (status quo: followup-b@5228646)

Mechanism: cache = NULL per-consumer resolve + invalidateLater(50) retry when eval_env binding hasn't landed yet. Gate is green (0/0/0/2357).

Pros: Already committed. Closes the user-visible UX defect. No new convention violations.

Cons:

  • Carries the architectural debt forward: the real cause (updateTextInput) remains.
  • 50ms polling is hacky and visible in any future race-condition postmortem.
  • Bypassing state$upstream_cache for the per-consumer path loses one layer of dedup (small perf hit; consumers each have distinct upstream subtrees anyway, so the dedup it provided was already minimal).
  • SC-2's literal "shiny::req()" mechanism is augmented beyond the plan as written.

Option B — Remove ptr_bind_source_autoname() entirely

Mechanism: Delete the function and its call sites. The user types the companion name themselves before resolution works.

Pros:

  • Closes the race at the root (no updateTextInput → no extra reactive cycle → no flush-order race).
  • The workaround in Option A becomes dead code (can revert invalidateLater + restore state$upstream_cache).
  • Restores the "no update*" convention for one of the three remaining call sites.
  • Smaller code surface.

Cons:

  • UX regression: companion no longer auto-fills from uploaded filename. Vignettes / docs that lean on this convenience would need to be updated.
  • Likely needs its own ADR ("remove autoname; companion is user-driven") to capture the rationale.

Option C — Replace updateTextInput with a renderUI-driven companion

Mechanism: Make the companion a renderUI output (not a static textInput). The renderUI's selected argument reads input[[src_id]] and computes the autoname server-side. No update* call. The companion is still user-editable.

Pros:

  • Preserves the autoname UX.
  • Removes the update* convention violation at line 936.
  • The new renderUI fires synchronously inside the server's reactive flush — no client roundtrip, no separate flush cycle, no race.

Cons:

  • Larger refactor: every consumer of the static textInput id changes to a uiOutput consumer.
  • User-typed-in-progress text gets clobbered every time renderUI re-fires (need to preserve current across re-fires, the same pattern ptr_setup_consumer_uis already uses for var pickers).
  • Needs care with shinyWidgets custom-rendered text inputs if they're in play.

Option D — Make MAIN observer higher-priority than the renderUIs

Mechanism: Add priority = 1 (or higher) to the MAIN upload observer in ptr_setup_pipelines. Forces MAIN obs to flush before any renderUI in the same cycle, eliminating the order race regardless of update*.

Pros: Single-line change. Doesn't remove the update* call but does close the race.

Cons:

  • Out of plan-01's scope (touches ptr_setup_pipelines, not ptr_setup_consumer_uis).
  • Relies on shiny's priority semantics (well-defined for observe(), less so across renderUI + observe interactions in chromote children).
  • Doesn't address the update* convention violation.
  • Subtle ordering change — any other observer at default priority that depends on the autoname-updated companion would now run after MAIN obs's assign, which may or may not be what they expect.

6. Recommendation

Option C if the project's long-term direction is "no update*": replace the static companion with a renderUI, killing the race at the root while preserving the autoname UX. Schedule it as ADR-0015 §3 (or a sibling ADR) with its own implementable plan.

Option B if the autoname UX is dispensable: simplest possible code, clearest convention compliance.

Either way, the workaround in Option A should be reverted in the same change so invalidateLater(50) + cache=NULL + the inlined-runtime_consumer_entry don't ossify as "the way ADR-0015 PLAN-01 works".

If neither B nor C is in scope right now, Option A's committed state is a known-good resting point — the user-visible UX defect is closed, the gate is 0/0/0/2357, and ADR-0013 PLAN-04 is unblocked. The debt is documented here for whichever future iteration takes the cleanup on.
