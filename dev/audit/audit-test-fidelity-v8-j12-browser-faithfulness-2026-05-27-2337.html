1. The lens (one-line restatement)

For each absorbed L2: if the orchestrator/runtime regressed only at THIS contract, would the browser DOM differ? If no → RETAIN as unit pin, do not migrate. If yes → MERGE. If yes-but-needs-extra-fixture → MERGE-deferred.

Two sub-questions sharpen the lens (from v6 §1 / v7 §1):

  1. Does the assertion observe an internal-shape value (S3 class, list field name, NULL vs empty-list, named-list shape, object identity) that the rendered DOM does not reflect?
  2. Does the assertion's trigger come from an R-side mutation (e.g. ptr_gg_extra(), direct session$setInputs() of a non-DOM virtual input) that no DOM event can dispatch?

"Yes" to either → not browser-encodable as-is.

7. Decision the next step needs from the user

  1. Negative-control test at test-adr12b-spec-custom-keyword.R:100 ("state$spec_seed is empty when no spec= is passed") — delete with the file, or RETAIN as unit pin? It is a pure internal-shape contract (empty env-map at boot when no spec=). DOM equivalent: nothing — no widget changes its rendered HTML based on whether state$spec_seed is empty vs non-empty as long as no value was seeded. Recommendation: RETAIN as unit pin — its purpose is "guard against future refactors that might inadvertently populate spec_seed from defaults", a regression class invisible to DOM. Same pattern as J10/J11 RETAINs.
  2. Fixture j12-renderui-emission combined formula — accept the 3-stage combination, or split into 3 separate fixtures? The combined formula ggplot(mtcars, aes(ppVar(mpg), ppNum)) + geom_text(aes(label=ppText), size=ppNum) exercises source ppVar + value ppNum + multi-keyword value rendering in one boot. Recommendation: combine — three fixtures with overlapping shapes is unnecessary scaffolding; the bare_ids stay distinct (ggplot_1_1_ppVar_NA, ggplot_1_2_ppNum_NA, geom_text_1_3_ppText_NA_ui, geom_text_2_ppNum_NA_ui) so assertions don't collide. Split into 3 only if a stage interferes with another (none expected — bindAll is whole-tree).
  3. Stage 3 — DOM observation of <option value="b" selected> on the selectInput's option HTML — confirm shape? Shiny's selectInput with selected = "b" renders the option list as <option value="a">a</option> <option value="b" selected>b</option> <option value="c">c</option> — the literal <option value="b" selected> appears verbatim in get_html("#ggplot_1_1_ppCustomChoice_NA"). If a future Shiny version changes this to selected="selected", the assertion would regress; the simpler/sturdier shape is expect_match(html, 'value="b"[^>]*selected', perl=TRUE). Recommendation: use the regex form for forward-compat.

None of these decisions touch product code. They shape the J12 journey + fixtures only. Once decisions land, the build is mechanical: write 3 fixtures, write the test, run the gate, delete the 7 absorbed L2 blocks in the same commit.
