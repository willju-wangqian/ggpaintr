CSS audit and unification proposals

2026-05-15 · static inspection · audit + proposals, no code edits this round

1. Summary

  • The bundled stylesheet (inst/www/ggpaintr.css) and the inline structural CSS (ptr_layer_assets()) only fire inside elements that carry the .ptr-app class. End users do not know .ptr-app exists, so an entry point that fails to self-wrap produces unstyled output and the user sees "ggpaintr looks different here".
  • Of the seven public UI entry points, all seven currently self-wrap in .ptr-app (bslib emits its own page chrome but inherits the .ptr-stage-* structural rules from ptr_layer_assets()). The drift is on the asset side, not the wrapper side: each entry point hand-assembles its own asset bundle, and the bundles do not match.
  • Concretely: ptr_shared_ui() wraps in .ptr-app but injects none of ptr_layer_assets() / ptr_ui_assets() / ptr_user_css_assets(). It works today only because it is always rendered next to ptr_module_ui() (which does inject them), so the page picks the assets up by accident. As a standalone fragment it would render unstyled.
  • The css = override portal is exposed by ptr_app, ptr_app_grid, ptr_module_ui, ptr_controls_ui, ptr_outputs_ui. It is not exposed by ptr_shared_ui; ptr_app_bslib uses bslib's theme = instead.
  • The leading proposal (#2) makes every raw-Shiny entry point pass through a single asset bundle and a uniform css = NULL arg. The supporting proposal (#1) folds the bundle assembly into one internal helper (ptr_assets()) so the matrix below collapses from "many partial calls" to "one call per entry point".

Key finding. The "users see inconsistent appearance across entry points" symptom maps to two distinct asset-side gaps, not to the wrap mechanism:
  1. ptr_shared_ui() injects no assets. It only renders correctly because it shares a page with ptr_module_ui(). As an independently-mounted fragment it would lose every cue.
  2. ptr_outputs_ui() skips ptr_layer_assets(). If a host page mounts ptr_outputs_ui() without also mounting ptr_controls_ui(), the class-toggle JS handler that adds .ptr-stage-disabled is missing. Not a current bug — but a latent foot-gun.

3. Asset inventory

3.1 The bundled stylesheet

  • inst/www/ggpaintr.css — 254 lines, every rule scoped under .ptr-app (file header comment, line 2). Served under the resource prefix ggpaintr/ (registered in ptr_ui_assets(), R/paintr-build-ui.R:570). Linked as <link href="ggpaintr/ggpaintr.css"> at R/paintr-build-ui.R:576.
  • Defines: CSS custom-property tokens (--ptr-bg, --ptr-surface, --ptr-border, --ptr-ink, --ptr-muted, --ptr-accent, --ptr-accent-strong, --ptr-accent-weak, --ptr-accent-line, --ptr-radius, --ptr-radius-sm, --ptr-shadow, --ptr-shadow-lg, --ptr-font, --ptr-mono); header bar; sidebar/wellPanel chrome; sticky-sidebar rule; shared-panel header copy; form control radius; primary action button; layer tabset; stage-fields border refinement; output cards; icon buttons; alert; code mini-window.
  • Every rule is gated on .ptr-app. Removing the wrap removes all cosmetic styling.

3.2 Inline style blocks emitted from R

  • ptr_layer_assets() (R/paintr-build-ui.R:521) — one <script> + one <style>. The script registers the ptr_set_class custom Shiny message handler (used by the server to toggle .ptr-stage-disabled / .ptr-layer-disabled). The style block declares:
    • .ptr-layer-disabled (opacity + pointer-events-none) — fires unscoped.
    • .ptr-stage-disabled .ptr-stage-head code / label / .ptr-stage-fields — the named visual cue. Fires unscoped.
    • .ptr-stage, .ptr-stage-head, .ptr-stage-fields, .ptr-stage-row — structural margin/padding/border. Fires unscoped.
    Important: these rules are not scoped under .ptr-app. They fire wherever they are injected, including inside bslib::page_sidebar output. This is intentional per the comment at :546: structural layout for pipeline stages must hold for every app shell, including bslib.
  • ggpaintr.css:50 refines .ptr-stage-fields border colour under .ptr-app — so colour drifts between bslib and raw-Shiny shells, structure does not.

3.3 Per-element style= attributes

None on a fast pass through R/paintr-*.R. No drive-by inline styles in the source.

3.4 User-override pipeline

  • ptr_user_css_assets(css) (R/paintr-build-ui.R:589) — takes a character vector of .css file paths. Validates that every path exists and ends in .css; aborts with a path-not-found / wrong-extension error otherwise. Each distinct parent directory is registered once under a hash-derived ggpaintr-user-<hash> resource prefix; the function emits one <link> per path. Returns NULL when css = NULL so callers can splice unconditionally.
  • Comment at :587: "Emitted after ggpaintr.css so user rules win on equal specificity." This invariant is preserved at every call site today.
  • The arg accepts paths only (string form would never have worked). Inline string CSS would require <style> blocks, which strict Content-Security-Policy headers reject; linked stylesheets served from addResourcePath pass CSP. Path-only is the right contract.

6. Override surface

  • css = arg. Path-only; accepts a character vector of .css file paths. Linked after ggpaintr.css so user rules win on equal specificity. Exposed by: ptr_app, ptr_app_grid, ptr_module_ui, ptr_controls_ui, ptr_outputs_ui. Not exposed by ptr_shared_ui or ptr_app_bslib.
  • CSS custom properties. The fifteen --ptr-* tokens listed in §3.1. These are the structured-override target — a user-supplied stylesheet can re-declare them inside a .ptr-app selector to retheme without touching individual rules. Not currently documented in user-facing roxygen.
  • bslib theme =. Only on ptr_app_bslib. Compiles a Sass-based theme; does not interact with ggpaintr.css (which fails to fire under bslib because there is no .ptr-app wrap).

7. Cascade & scoping risk

  • Nested .ptr-app (the module-inside-module case). ggpaintr.css:33 neutralises background and min-height on the inner wrap. Nothing else neutralises — so any rule on a child of .ptr-app applies once for each wrap. Audit found one consequence: .ptr-app form.well { position: sticky; ... } (:75) would apply to a nested form.well too, but in practice the grid app's wellPanel renders as div.well, not form.well, so the bug it protects against does not regress.
  • Stacking-context trap. Comment at R/paintr-build-ui.R:535 records that an earlier .ptr-stage-disabled { opacity: 0.5 } rule created a stacking context that clipped picker dropdowns. The current rule (text colour + border + faint background) avoids opacity / filter on stage ancestors. This is the only documented stacking-context interaction; no other rule in ggpaintr.css sets opacity, filter, or transform on an ancestor of a picker.
  • Host-page bleed. The .ptr-app scope is the protection. Anything inside the wrapper inherits ggpaintr styling; anything outside is untouched. Verified by reading ggpaintr.css top-to-bottom: every selector either starts with .ptr-app, is nested inside a .ptr-app … selector, or is one of the .ptr-stage-* structural rules that are intentionally unscoped (and only emit if R-side code adds the class). No bare element selectors, no global resets.
  • z-index. Two rules set z-index: .ptr-app form.well { z-index: 2 } (sidebar stacking context, :75) and .ptr-code-window { z-index: 1080 } (above Bootstrap modal range). No conflict with Bootstrap 3 stacking but worth re-checking under bslib / Bootstrap 5 if ptr_app_bslib ever inherits the code window.
  • Specificity. User-supplied css = files link after ggpaintr.css, so equal specificity yields to the user. If a user wants to override a doubly-scoped rule like .ptr-app .btn.action-button, they have to match the specificity (any equal-or-higher selector wins because of source order). Not unusual; worth one line in the docs.

8. Proposal #2 (leading): entry-point unification

Problem

The per-entry-point matrix in §2 shows that two raw-Shiny entry points (ptr_shared_ui, ptr_outputs_ui) are asset-incomplete in ways that today are masked by the fact that they always render next to a sibling entry point that fills in the gap. The masking is fragile: any future use case that mounts these fragments independently (custom layouts, alternate hosts, tests) will see broken styling. ptr_shared_ui additionally lacks the css = override portal.

Proposed change

  1. Every raw-Shiny entry point self-wraps in .ptr-app. Already true for all five. Make this an invariant that future entry points must preserve; cover it with a test.
  2. Every raw-Shiny entry point routes assets through a single ptr_assets(css = NULL) bundle. See Proposal #1.
  3. Every raw-Shiny entry point accepts css = NULL (path-only, same arg name). The two that don't today are ptr_shared_ui (add it) and the partial-asset-injection cases in the matrix (the new ptr_assets() bundle handles them by construction).
  4. Documentation. Add a shared roxygen topic (e.g. ?ptr_css) enumerating the fifteen --ptr-* custom properties as the structured-override target, plus a one-line specificity note. Every entry-point man page cross-references it via @seealso.

Files touched (proposal-time estimate)

  • R/paintr-build-ui.R — add ptr_assets() (Proposal #1).
  • R/paintr-app.R — replace per-entry-point asset assembly in ptr_build_app_ui, ptr_app_grid_components, ptr_module_ui, ptr_controls_ui, ptr_outputs_ui with single ptr_assets(css = css) call. ptr_outputs_ui gains ptr_layer_assets()'s class-toggle JS (via the bundle) so the standalone case no longer depends on a sibling.
  • R/paintr-shared-ui.R — ptr_shared_ui gains css = NULL arg; calls ptr_assets(css = css); closes the "shared-ui injects nothing" gap in the matrix.
  • man/ptr_css.Rd (new) — shared documentation topic.
  • Every entry-point man page — add @seealso ptr_css.

Risks

  • Idempotency of double-injected assets. When ptr_module_ui nests ptr_controls_ui + ptr_outputs_ui, the new bundle would be emitted three times on one page. The current comment at R/paintr-app.R:427 already states the assets are idempotent (linking the same href twice is harmless; addResourcePath is idempotent; the class-toggle script self-registers behind window.__ptr_set_class_registered). Verify in the implementation that this still holds for the consolidated bundle.
  • Source-order specificity invariant. User css = must continue to link after ggpaintr.css. Encode this ordering inside ptr_assets() and lock it with a test that inspects the rendered <head>.
  • ptr_app_bslib divergence stays. The bslib path will continue to lack ggpaintr.css cosmetic styling because it does not wrap in .ptr-app. Documented limitation; not in scope for this proposal. Deferred (§10).

Acceptance criteria

  1. Wrap invariant. Every raw-Shiny entry point produces an outermost tag carrying class = "ptr-app". Verified by mounting each entry point with a minimal fixture and asserting the rendered HTML.
  2. Override invariant. Every raw-Shiny entry point accepts css = NULL. Verified by signature inspection (formals()) plus a smoke test per entry point that passes a fixture .css and asserts the resulting <link> appears after the ggpaintr.css link in the rendered head.
  3. Visual-cue invariant. The .ptr-stage-disabled rule renders identically across the five raw-Shiny entry points. Verified by inspecting that each emits the same ptr_assets() bundle.
  4. Documentation invariant. ?ptr_css exists and enumerates the fifteen --ptr-* tokens with current default values; every raw-Shiny entry-point man page cross-references it.

9. Proposal #1 (supporting): ptr_assets() bundle

Problem

Today every entry point hand-assembles list(ptr_layer_assets(), ptr_ui_assets(), ptr_user_css_assets(css)) in some subset and order. The drift is visible in the matrix above: ptr_outputs_ui omits ptr_layer_assets(), ptr_shared_ui omits everything. Adding a new asset (e.g. a future icon-font link) would require editing every entry point. This is the same problem the controllable_region() helper solved for stage DOM in commit cefc9bf.

Proposed change

ptr_assets <- function(css = NULL) {
  shiny::tagList(
    ptr_layer_assets(),    # inline structural CSS + class-toggle JS
    ptr_ui_assets(),       # <link> to ggpaintr.css + code-window JS
    ptr_user_css_assets(css)  # user override (after ggpaintr.css)
  )
}
  • Internal — not exported. Per [[api-audit]] the goal is to shrink the public surface, not grow it.
  • Every entry point's asset-assembly collapses to ptr_assets(css = css).
  • The internal split between "rules in the file" (ggpaintr.css) and "rules in the inline shim" (ptr_layer_assets()) stays as a documented private choice: cosmetic + cacheable + CSP-friendly rules live in the file; structural rules that must hold inside any shell (including bslib) live in the inline shim. Comment block on ptr_assets() records the rationale.
  • Idempotent: linking the same href twice is harmless; addResourcePath is idempotent; class-toggle script self-registers behind window.__ptr_set_class_registered (R/paintr-build-ui.R:525).

Files touched

  • R/paintr-build-ui.R — add ptr_assets() next to the three existing helpers; keep them as internal building blocks (they are used independently by ptr_app_bslib for the layer-assets-only case).
  • R/paintr-app.R — replace five call sites.
  • R/paintr-shared-ui.R — add the call site (today: zero asset injection).

Risks

  • bslib path keeps using ptr_layer_assets() directly. Do not route ptr_app_bslib through ptr_assets() — it must not inject ggpaintr.css (no .ptr-app wrap; would link a stylesheet that fires nothing). The bslib path is the one entry point that still hand-picks its assets, deliberately. Document this exception inline.
  • Resource-path registration cost. ptr_ui_assets() calls shiny::addResourcePath("ggpaintr", …); doing it three times per page when nested is harmless but worth a comment.

Acceptance criteria

Covered by Proposal #2's acceptance criteria. No separate criteria — Proposal #1 is the mechanism, Proposal #2 is the contract.

10. Deferred

  • bslib theme story. ptr_app_bslib currently lives in a separate substrate: bslib emits its own page chrome, the .ptr-app wrap is absent, ggpaintr.css does not fire, and the override hook is bslib's theme = (not css =). Two questions deferred:
    1. Should ptr_app_bslib also wrap in .ptr-app so ggpaintr.css cosmetic rules apply? Would mean reconciling bslib tokens with our --ptr-* tokens (or letting them clash and documenting which wins).
    2. Should ptr_app_bslib grow a css = arg alongside theme =? Only useful if (1) lands.
    Both are bigger design decisions than the entry-point unification in Proposal #2. Defer to a future ADR.
  • Structured theme = list(accent = ...) arg on raw-Shiny entry points. Sugar over css = targeting the --ptr-* tokens. Reject for now (YAGNI; css = + documented tokens is already sufficient). Revisit if multiple users ask.
  • Dead-selector cleanup. .ptr-controls at ggpaintr.css:64 has no current emitter. Remove in the same PR that lands Proposal #1 as a drive-by tidy. Not load-bearing.
  • BEM-style consistency in stage classes. .ptr-stage-head uses single-hyphen; the rest of the codebase prefers __elem. Renaming is mechanical but breaks any user CSS that targets these. Not worth the churn unless a larger naming pass happens.

11. Acceptance criteria summary

  1. Wrap invariant on every raw-Shiny entry point (tested).
  2. Override invariant on every raw-Shiny entry point: css = NULL arg, path-only, linked after ggpaintr.css (tested).
  3. Visual-cue invariant: .ptr-stage-disabled renders identically across raw-Shiny entry points (verified by single-bundle invariant).
  4. ?ptr_css documents the fifteen --ptr-* tokens; every entry-point man page cross-references it.

12. Open questions

  • Needs runtime verification. Per the agreed methodology, every claim in this audit is static. The following are flagged for a runtime check before being escalated into ADR or plan work:
    • Whether ptr_shared_ui mounted as a standalone fragment (without a sibling ptr_module_ui on the page) actually loses the disabled-stage cue, or whether the class-toggle JS is somehow inherited.
    • Whether ptr_outputs_ui mounted standalone loses the class-toggle handler (almost certain given the static reading, but the matrix would benefit from a probe).
    • Whether ptr_app_bslib with a stage-bearing formula actually exhibits the disabled cue. The static reading says yes (structural rules in ptr_layer_assets() are unscoped) but the visual rendering inside bslib's grid system has not been eyeballed.
    • z-index interaction between .ptr-code-window (1080) and any bslib offcanvas / modal in ptr_app_bslib (only matters if the code window is ever wired into the bslib path).
    Per browser-probe-pitfalls (auto-memory): probe results need an R-side cross-check before any finding is filed.
  • Should ptr_assets() become ptr_app_assets()? Naming consistency with the existing ptr_app_header() pattern. Decide at implementation time; not load-bearing.
  • What happens to ptr_layer_assets() / ptr_ui_assets() / ptr_user_css_assets()? Keep them internal (still used independently by the bslib path), or fold them into ptr_assets() as private locals? Recommendation: keep them. They are independently useful for the bslib path and any future shell that needs partial asset injection.
